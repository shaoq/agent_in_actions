# Agent in Actions

面向 AI Agent 应用工程师的学习、实战与考核工程。

这个项目的目标是把 Agent 相关知识从零散概念整理成一套可训练、可复盘、可面试表达、可工程落地的能力体系。它不是资料仓库,也不是单纯的 demo 集合,而是围绕“成为能通过 Agent 应用工程师面试并胜任真实研发”的长期训练系统。

当前阶段:阶段 1 - 学习与知识框架搭建。

## 项目使命

本项目服务于一个明确目标:系统掌握 Agent 应用工程的核心技术,最终能够在公司面试中清楚证明自己的架构理解、工程实现、调试分析、安全治理和评测能力。

所有新增内容都必须回答至少一个问题:

1. 它是否帮助我更理解 Agent 系统如何工作?
2. 它是否帮助我把 Agent 能力落成可运行工程?
3. 它是否帮助我在面试中更清楚地表达取舍?
4. 它是否帮助我发现自己哪里还不会?

如果答案都是否,就不应该放进这个项目。

## 目录

- `docs/`: Agent 架构蓝图与专题文档。用于建立知识地图和长期参考。
- `labs/`: 动手实验和可运行 demo。用于把概念转成工程能力。
- `roadmap/`: 学习路线、阶段目标、里程碑和进度跟踪。
- `interview/`: 面试题、系统设计题、追问链和表达模板。
- `assessment/`: 考核任务、评分标准和阶段复盘。
- `design-system/`: Blueprint 技术文档视觉系统。

## 当前重点

1. 以 `docs/00-agent-architecture.html` 为总览,建立 Agent 应用工程的完整地图。
2. 按模块补齐专题文档:编排、工具调用、RAG、记忆、评测、治理、模型网关。
3. 为每个模块配套一个可运行实验和一组面试题。
4. 最终形成学习阶段和考核阶段两套资产。

## 学习闭环

每个主题都按同一闭环推进:

1. 知识输入:读文档、论文、官方说明或源码。
2. 架构抽象:整理这个主题在 Agent 系统中的位置、职责和边界。
3. 工程实验:做一个最小可运行 demo 或复现实战场景。
4. 失败分析:记录它会在哪里失败,如何观测,如何修复。
5. 面试表达:整理成 3 分钟解释、10 分钟系统设计和追问回答。
6. 考核验证:用题目或任务检查是否真正掌握。

不建议只阅读不产出。每次学习至少留下一个可复用资产:一页文档、一段实验代码、一组问答、一个复盘记录。

## 建议学习顺序

1. 编排控制层:状态、规划、反思、单 Agent 与多 Agent 协作。
2. 工具 / 动作层:tool calling、MCP、权限、失败重试、幂等。
3. 知识 / 检索:RAG、chunking、rerank、grounding、引用。
4. 记忆系统:短期记忆、长期记忆、用户偏好、经验沉淀。
5. 评测层:任务成功率、LLM-as-judge、回归集、安全评测。
6. 治理与可观测:trace、成本、限流、审计、prompt injection 防御。
7. 模型网关与基础设施:多模型路由、fallback、缓存、沙箱。

## 产出标准

新增专题文档应包含:

- 这个主题解决什么问题。
- 它在 9 层 Agent 架构中的位置。
- 核心概念和关键术语。
- 生产环境中的设计取舍。
- 常见失败模式。
- 面试高频追问。
- 后续实验入口。

新增实验应包含:

- 运行方式。
- 实验目标。
- 关键代码路径。
- 预期现象。
- 失败场景。
- 可扩展方向。

新增面试材料应包含:

- 问题。
- 短回答。
- 深入回答。
- 面试官可能追问。
- 工程案例表达。

## 防跑偏规则

- 不把本项目变成资料下载目录。
- 不把本项目变成纯前端展示项目。
- 不长期停留在概念总结,必须逐步补实验。
- 不为了追热点新增主题,除非能映射到架构、实验或面试能力。
- 不把无关业务文件、Excel、临时脚本放进根目录。
- 不把“模型 API 调用”误当成完整 Agent 工程能力。
