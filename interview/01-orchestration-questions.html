编排控制层面试题

使用方式

按理论优先顺序使用本题集:

  1. 完成编排五单元理论课程。
  2. 使用本题集做每单元的口述追问,此时不要求阅读实验代码。
  3. 通过编排无代码理论检查点。
  4. 再进入最小 Agent Loop 实验验证状态和 trace。
  5. 最后完成编排工程模块考核。

短回答用于建立定义,深入回答用于解释机制和取舍,追问用于暴露边界不稳定处。不要在第一次阅读时背诵标准答案。

1. Agent 编排层解决什么问题?

短回答

编排层是任务控制面。它维护运行状态,协调决策、动作和观察,并决定继续、完成、失败或转人工。模型提供判断能力,编排层保证任务按可控流程推进。

深入回答

普通模型调用只负责从输入生成输出,而 Agent 需要跨多个步骤与环境交互。编排层保存目标、步骤、观察、预算和状态,把当前上下文交给推理内核,校验结构化决策,触发动作,再把结果写回下一轮。生产环境还要处理超时、最大步数、checkpoint、权限门禁、失败恢复和 trace。它可以实现成 while loop、状态机、持久化图或任务队列,并不等同于某个框架。

可能追问

  1. 为什么不让 LLM 自己维护全部状态?
  2. 编排层和推理内核的边界在哪里?
  3. 哪些场景根本不应该使用 Agent loop?

工程表达

我会先说明状态字段和完成条件,再说明每个 phase 的输入输出与失败状态,最后讲为什么选择确定性工作流、ReAct 或 plan-and-execute,而不是只说使用了某个框架。

2. 一个最小单 Agent loop 包含哪些步骤?

短回答

最小链路是 decide -> act -> observe -> terminate。编排器先基于状态请求下一步决策,执行合法动作,把观察写回状态,再判断继续还是停止。

深入回答

开始时创建包含任务和预算的状态。每轮增加 step,记录 decision requested,让决策器返回 act 或 finish。act 需要校验动作与参数,执行后把 observation 写回;finish 保存最终结果并结束。任何阶段失败都应产生结构化错误。循环还必须有独立于模型的最大步数、时间或成本限制。

可能追问

  1. 为什么 observe 必须写回状态而不能只写日志?
  2. 如果动作不可逆,act 前需要增加什么步骤?
  3. finish 是否一定代表业务任务成功?

工程表达

可以结合实验中的 LoopState、DecisionProvider、ActionExecutor 和 TraceEvent 说明职责分离,并指出生产系统还需要 guardrail、授权和业务验收器。

3. 如何防止 Agent 无限循环?

短回答

不能只依赖模型主动结束。至少要设置最大步数、超时和成本预算,检测重复动作或无信息增益,达到阈值后失败、降级或转人工。

深入回答

终止条件分为正常完成、资源预算、失败阈值和外部中断。除了硬限制,还可以比较连续决策与观察,识别同一动作重复、状态没有变化或计划长期不收敛。高风险任务达到阈值后不应继续自动尝试。trace 必须记录终止原因,便于区分模型能力不足、工具无反馈和编排策略错误。

可能追问

  1. 最大步数应该如何设定?
  2. 如何定义“没有信息增益”?
  3. 转人工时需要携带哪些上下文?

工程表达

先用历史 trace 统计成功任务的步数分布设置初值,再按任务类型配置预算。转人工时提供目标、已执行动作、关键观察、失败原因和剩余风险。

4. Agent 状态、会话状态和长期记忆有什么区别?

短回答

Agent 运行状态服务于一次任务推进;会话状态服务于当前交互上下文;长期记忆沉淀跨会话可复用的用户偏好、事实或经验。它们可能使用相同存储,但语义和生命周期不同。

深入回答

运行状态包含当前节点、步骤、观察、预算和执行结果,需要 checkpoint 支持恢复。会话状态保存消息与渠道上下文。长期记忆经过写入策略和治理后跨任务读取。把三者混在一个 messages 数组中会导致上下文膨胀、记忆污染和恢复边界不清。

可能追问

  1. checkpoint 是否属于 Memory?
  2. 哪些状态必须持久化?
  3. 恢复旧 checkpoint 时如何处理代码版本变化?

工程表达

强调“存储介质相同不代表架构职责相同”。说明 schema version、checkpoint ID、状态迁移和敏感信息生命周期。

5. 动作失败后应该如何重试?

短回答

先分类失败是否可重试,再决定退避重试、重新决策、补偿或人工接管。涉及副作用的动作必须有幂等键,不能对所有异常无条件重试。

深入回答

超时和临时限流可能适合有限次数的指数退避;参数校验和权限错误应直接失败或重新决策;不可逆动作失败需要查询实际提交状态,避免“响应丢失但动作已成功”造成重复执行。编排状态要记录 attempt、action ID、错误类别和补偿状态。

可能追问

  1. 如何区分动作未执行和执行成功但响应超时?
  2. 重试应该在工具层还是编排层?
  3. 补偿事务失败怎么办?

工程表达

工具适配层负责单次调用的传输级重试,编排层负责业务级重新决策和跨步骤补偿。二者必须共享调用 ID 和可观察错误分类。

6. ReAct 和 plan-and-execute 如何选择?

短回答

路径依赖环境反馈、难以预先确定时选择 ReAct;步骤相对稳定、需要审批或并行时选择 plan-and-execute。固定流程优先普通工作流,不要默认使用自治循环。

深入回答

ReAct 每轮根据新观察决定下一步,适应性强,但调用次数、轨迹和成本难预测。Plan-and-execute 先形成计划再逐项执行,更易审计和并行,但计划可能过时,需要重规划触发器。很多生产系统是混合模式:外层确定性状态机控制阶段,局部节点使用 ReAct。

可能追问

  1. 如何判断计划已经过时?
  2. 混合模式的状态由谁维护?
  3. 是否应该保存模型的完整 reasoning trace?

工程表达

根据任务不确定性、动作风险、可审计性、延迟和成本选择。内部推理内容需考虑隐私与供应商策略,工程 trace 更应记录结构化决策理由和外部可验证证据。

7. 如何设计 Agent 的 trace?

短回答

trace 至少要能按 run 和 step 还原 decide、act、observe、terminate,记录状态变化、动作、结果、耗时、错误和预算,同时避免泄露敏感数据。

深入回答

每个 run 有稳定 ID,每步事件包含 phase、event type、时间、父子关系和必要 payload。模型调用记录模型、token、延迟和结构化输出;动作记录 action ID、参数摘要、授权结果、幂等键和结果;终止记录原因。PII、密钥和完整 reasoning 不应直接进入日志。trace schema 应可版本化并支持离线 eval 消费。

可能追问

  1. 日志、metric 和 trace 的区别是什么?
  2. 如何控制 trace 存储成本?
  3. 如何从 trace 判断是模型失败还是工具失败?

工程表达

使用 phase 和 error taxonomy 做第一层归因,再关联模型响应、validation error、工具状态码和状态 diff。采样不能丢失失败、高风险和人工接管轨迹。

8. 什么时候应该使用 Multi-Agent?

短回答

只有当任务存在明确专业分工、上下文隔离或可并行子任务,且收益大于通信、状态同步和评测成本时才使用。多数场景应先把单 Agent 的工具、指令和状态做好。

深入回答

Multi-Agent 可以隔离不同角色的上下文和权限,让独立子任务并行,或由评审角色检查产物。但它会引入任务分解质量、消息协议、共享状态、冲突仲裁、重复工作、token 成本和端到端归因问题。不能因为单 Agent 表现差就直接增加 Agent 数量,应先确认瓶颈是否真的来自上下文或能力分工。

可能追问

  1. Orchestrator-workers 与 peer-to-peer 有什么差异?
  2. 多 Agent 共享 Memory 会有什么风险?
  3. 如何评测一个子 Agent 对最终结果的贡献?

工程表达

先做单 Agent 基线,再用任务成功率、延迟、成本和失败归因比较 Multi-Agent 方案。若收益不能覆盖协调复杂度,保留单 Agent 或确定性 workflow。
