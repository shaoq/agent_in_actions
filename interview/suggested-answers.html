返回编排题库

ORC-01 · 编排层解决什么问题？

考察点：任务控制面与模型判断的边界。

30 秒回答：编排层维护运行状态并协调 decide、act、observe、terminate；模型只给候选判断，编排负责可控推进、预算、失败和人工接管。

3 分钟框架：先说明多步骤环境交互为何超出单次模型调用；再描述状态、阶段和终止条件；最后讲确定性门禁、checkpoint、trace 以及何时不用 Agent loop。

关键术语：control plane、state、phase、budget、termination。

工程证据：展示一次 run 的状态 schema、phase event 和终止原因。

方案取舍：路径稳定优先 workflow；环境反馈强才使用自治 loop。

常见错误：把编排等同某个框架，或让 LLM 自己保存全部状态。

追问提示：finish 只表示模型结束提议，仍需业务验收器确认成功。

ORC-02 · 最小 Agent loop

考察点：状态驱动的生命周期。

30 秒回答：创建带目标和预算的状态，每轮请求结构化决策，校验并执行动作，将 observation 写回，再由独立终止器决定继续、完成或失败。

3 分钟框架：定义 LoopState；描述 decide→validate→act→observe→terminate；为不可逆动作加入 authorize/approve；每步写 trace 和 checkpoint，恢复时依赖版本与幂等键。

关键术语：state machine、observation、checkpoint、idempotency。

工程证据：状态 diff、决策 schema、单元测试和恢复 trace。

方案取舍：内存 loop 简单但不可恢复；持久化图更可靠但复杂。

常见错误：只画 decide→act，不写 observation 和终止。

追问提示：observation 是下一轮决策输入，日志只是旁路证据。

ORC-03 · 防止循环不收敛

考察点：硬限制与收敛检测。

30 秒回答：不能等模型自觉停止；需要最大步数、时间和成本硬预算，并检测重复动作、状态无变化或无信息增益，达到阈值后降级、失败或转人工。

3 分钟框架：从历史成功步数设置任务型预算；对 action+args+observation 做重复指纹；定义正常、预算、失败阈值、取消四类终止；转人工携带目标、证据、已执行动作和剩余风险。

关键术语：step budget、deadline、stagnation、loop fingerprint。

工程证据：步数分布、重复率、termination_reason 指标。

方案取舍：阈值过紧损失复杂任务成功率，过松放大成本和风险。

常见错误：只设置 max_steps，不能识别低效但未超限的循环。

追问提示：信息增益可用新状态字段、证据覆盖或计划进度近似。

ORC-04 · 状态、会话与记忆

考察点：生命周期和语义 owner。

30 秒回答：运行状态服务一次任务推进，会话状态服务当前交互，长期记忆跨任务保存经治理的经历；可共用存储，但 schema、TTL、读写者和删除策略不同。

3 分钟框架：列举 run 的 step/预算/结果、session 的消息/渠道、memory 的偏好/经验；checkpoint 是运行状态快照而非长期记忆；恢复需校验 schema、代码和工具版本。

关键术语：run state、session、memory、schema version。

工程证据：三类数据模型、TTL 与迁移记录。

方案取舍：统一数据库降低运维成本，但必须保持逻辑边界。

常见错误：把 messages 数组当成全部状态和记忆。

追问提示：是否持久化取决于恢复、审计和副作用一致性要求。

ORC-05 · 动作失败恢复

考察点：失败分类和副作用。

30 秒回答：先区分可重试、不可重试和 effect unknown；传输层做有限退避，编排层做业务重决策或补偿，写动作必须透传幂等键并能查询 receipt。

3 分钟框架：分类超时/限流、校验、权限、业务拒绝；unknown 先查状态而非重放；记录 action_id、attempt、idempotency_key；不可恢复则保存证据转人工。

关键术语：error taxonomy、effect unknown、receipt、compensation。

工程证据：响应丢失故障注入和重复执行回归。

方案取舍：工具层适合传输重试，编排层拥有跨步骤业务语义。

常见错误：捕获所有异常后原样重试。

追问提示：补偿不是数据库回滚，也可能失败，需独立状态和人工兜底。

ORC-06 · 编排模式选择

考察点：不确定性与可控性取舍。

30 秒回答：固定路径优先 workflow；路径依赖观察时用 ReAct；步骤可预规划、需审批或并行时用 Plan-and-Execute；生产中常用外层状态机加局部自治。

3 分钟框架：按任务路径不确定性、动作风险、审计、延迟和成本建立矩阵；说明 ReAct 适应强但难预测，计划模式可审计但会过时；通过基线评测选择。

关键术语：ReAct、plan-and-execute、workflow、replan trigger。

工程证据：三方案成功率、步数、P95 和单位成本。

方案取舍：复杂模式提升适应性，也扩大状态和评测空间。

常见错误：默认自治程度越高越先进。

追问提示：计划过时可由关键前提失效、工具结果冲突或预算偏差触发。

ORC-07 · Trace 与 Checkpoint

考察点：调试证据与可恢复状态。

30 秒回答：trace 记录发生了什么，checkpoint 保存恢复所需状态；二者用 run/step/version 关联，恢复前校验代码、工具和授权，并依靠幂等键防止重复副作用。

3 分钟框架：trace 保存 phase、输入摘要、输出、延迟、错误和成本；checkpoint 保存节点、状态版本、待处理动作和 receipt；敏感字段脱敏；resume 使用 token、版本迁移和再授权。

关键术语：event trace、state snapshot、resume token、state version。

工程证据：中断恢复测试、state diff 和副作用去重记录。

方案取舍：频繁 checkpoint 恢复点细但存储与一致性成本高。

常见错误：认为日志足够恢复，或保存完整敏感 prompt。

追问提示：恢复前要重验审批有效期和外部资源版本。

ORC-08 · Multi-Agent 判定

考察点：分工收益是否覆盖协调成本。

30 秒回答：只有存在明确专业分工、上下文隔离或可并行子任务，且实测收益超过通信、共享状态、权限和评测复杂度时才用 Multi-Agent；先建立单 Agent 基线。

3 分钟框架：说明适用信号；比较 orchestrator-workers 与 peer 协作；定义消息协议、状态 owner、权限和仲裁；用成功率、延迟、成本及边际贡献做实验。

关键术语：task decomposition、protocol、shared state、marginal contribution。

工程证据：单 Agent 对照实验、子任务 trace 和成本拆分。

方案取舍：上下文隔离和并行换来归因、冲突和重复工作。

常见错误：单 Agent 效果差就增加 Agent 数量。

追问提示：共享记忆必须有 owner、权限和冲突策略，不能变成公共消息池。

REA · 推理内核

返回推理题库

REA-01 · 决策契约

考察点：推理、编排与执行边界。

30 秒回答：推理内核接收目标、受控上下文和可用能力，输出可校验的候选决策及外部证据；它不维护流程真相、不直接执行动作，也不直接调用厂商。

3 分钟框架：定义输入状态视图和 decision schema；说明编排校验、授权和执行；敏感或无关上下文最小化；模型访问统一走网关。

关键术语：decision contract、candidate action、state view、validation。

工程证据：decision JSON、校验失败 trace 与契约测试。

方案取舍：契约越强可控性越高，但表达空间更受限。

常见错误：把自然语言输出直接转为工具调用。

追问提示：状态真相由编排层持有，推理只看到当前授权视图。

REA-02 · 结构化输出

考察点：语法与语义可靠性。

30 秒回答：使用模型原生结构化输出或严格 schema，随后依次做解析、schema 校验和业务校验；失败按可修复性选择局部修复、有限重试或上报。

3 分钟框架：先约束 enum、必填和类型；再校验工具存在、参数范围和状态前提；记录 raw 摘要与 validation error；schema 版本随 run 持久化。

关键术语：JSON Schema、semantic validation、schema version。

工程证据：非法 JSON、未知 enum、危险参数测试集。

方案取舍：修复便宜但可能掩盖模型问题；重调更稳但增加成本。

常见错误：JSON 可解析就视为可执行。

追问提示：业务校验失败通常应重决策，而不是字符串修补。

REA-03 · 工具选择与越权

考察点：候选选择不等于授权。

30 秒回答：按当前任务和主体生成最小候选工具集，模型只提议工具和参数；确定性 schema、policy 和必要审批通过后，执行器才调用工具。

3 分钟框架：工具注册表按能力与风险筛选；校验参数与状态前提；policy 使用服务端身份；高风险动作绑定 action hash 审批；全链路记录 route 和 decision。

关键术语：tool allowlist、policy enforcement、least privilege。

工程证据：越权候选被拒 trace 与审批失效测试。

方案取舍：候选集小可控但可能降低覆盖率。

常见错误：把用户权限写进 prompt 并让模型判断。

追问提示：参数安全是业务 policy，不是 JSON 类型校验。

REA-04 · 推理模式选择

考察点：复杂推理是否有增益。

30 秒回答：简单任务先用直接生成基线；需要分解时引入计划，依赖环境反馈时逐步决策，需要高置信结果时才考虑多候选或评审，并用质量、延迟和成本实测。

3 分钟框架：按任务可分解性和验证性选择；反思只在有外部校验信号时有价值；多候选需要独立 grader；所有复杂度对照简单基线评测。

关键术语：decomposition、reflection、self-consistency、baseline。

工程证据：消融实验与错误类型变化。

方案取舍：复杂模式可能提升准确率，也放大延迟、成本和相关错误。

常见错误：把更长 reasoning 当成更可靠。

追问提示：反思没有新证据时可能只是重复合理化。

REA-05 · 不确定性处理

考察点：风险与证据充分度。

30 秒回答：不直接相信模型自报置信度，而是结合证据覆盖、校验结果和任务风险；低风险可给带限定的答案，高信息缺口先澄清或检索，高风险且证据不足则拒绝或转人工。

3 分钟框架：定义风险等级；建立可观测置信信号；选择澄清、检索、换模型、降级或拒绝；用校准集检验阈值与误拒率。

关键术语：calibration、evidence sufficiency、abstention。

工程证据：可靠性曲线、拒答精度和高风险切片。

方案取舍：更保守提升安全但降低完成率。

常见错误：把 0.9 的模型自信当 90% 正确率。

追问提示：澄清应针对会改变动作选择的最小缺口。

REA-06 · 决策失败恢复

考察点：按错误类型恢复。

30 秒回答：结构非法可约束重试，事实不足应检索或澄清，计划冲突要重规划，安全或权限失败不得靠重试绕过；所有策略都有尝试和成本上限。

3 分钟框架：建立 parse/schema/semantic/evidence/plan/policy taxonomy；选择修复、重采样、换模型、重规划或人工；清理导致重复失败的上下文；将失败轨迹加入 eval。

关键术语：failure taxonomy、retry budget、replan、fallback。

工程证据：各错误类别恢复率与额外成本。

方案取舍：换强模型可能提高恢复率，但不能修复错误状态或权限。

常见错误：对所有失败用同一 retry prompt。

追问提示：连续同类失败应停止并保存原始证据。

REA-07 · 非可信上下文

考察点：分层注入防御。

30 秒回答：外部文档和工具结果标记为数据而非指令，最小化可用工具与上下文；模型即使被操纵，确定性授权、审批和沙箱也必须阻止越权动作。

3 分钟框架：可信指令与非可信内容分区；检测和内容净化仅作信号；限制 tool set 和输出 schema；policy 绑定服务端身份；离线红队覆盖多轮绕过。

关键术语：instruction hierarchy、untrusted data、defense in depth。

工程证据：注入样例、policy deny、red-team 回归。

方案取舍：严格过滤可能损失文档信息，需用权限控制兜底。

常见错误：只增加一句“忽略恶意指令”。

追问提示：模型失守不应自动变成系统越权。

REA-08 · 模型网关边界

考察点：供应商解耦与运行治理。

30 秒回答：推理内核声明所需能力，模型网关负责供应商适配、路由、配额、fallback、成本和观测；直接调用 SDK 会让业务逻辑绑死模型并绕过统一治理。

3 分钟框架：定义逻辑请求和能力标签；网关映射厂商格式；记录 route reason、model version、token 与 latency；fallback 前校验能力等价和质量底线。

关键术语：capability contract、routing、provider adapter。

工程证据：同一决策契约跨两模型的兼容与回归。

方案取舍：统一抽象会隐藏部分厂商特性，需要受控扩展字段。

常见错误：把业务意图路由塞进模型网关。

追问提示：fallback 的结构兼容不代表语义等价。

TOOL-01 · Tool Contract

考察点：可理解、可校验、可演进的工具接口。

30 秒回答：除输入 schema 外，还要定义业务语义、输出与错误类型、副作用、权限、幂等、超时和版本；描述应帮助模型选对工具，但安全由服务端执行。

3 分钟框架：用稳定名称和明确字段约束；返回 typed result/error；声明 read/write 与风险等级；版本化契约并对旧 run 保持兼容；建立 contract test。

关键术语：schema、semantics、side effect、versioning。

工程证据：契约样例、错误枚举和兼容测试。

方案取舍：细粒度工具可控但增加选择数量。

常见错误：只有函数名和自由文本参数。

追问提示：错误要区分 retryable、denied、invalid 和 effect_unknown。

TOOL-02 · 调用生命周期

考察点：从候选动作到 observation。

30 秒回答：调用应经历 proposed、validated、authorized、executing、observed 或 failed；每步共享 action_id，结果规范化后写回状态，不能把 HTTP 200 直接当业务成功。

3 分钟框架：候选动作生成 ID；schema 与 policy 校验；执行器设置 deadline 和 idempotency；大结果存 artifact 只回摘要；业务验收器解释结果。

关键术语：action lifecycle、observation、receipt、artifact。

工程证据：状态迁移 trace 与超时测试。

方案取舍：事件更细便于归因，但增加存储和协议成本。

常见错误：工具异常直接拼成模型消息且丢失类型。

追问提示：敏感结果应最小化、脱敏并限制下游可见范围。

← 返回 TOOL-02 原题

TOOL-03 · 权限与审批

考察点：主体、范围和决定绑定。

30 秒回答：服务端解析真实主体和代理关系，policy 判断 scope；高风险动作再展示可理解摘要，审批凭据绑定主体、参数哈希、策略版本和有效期，执行前重新校验。

3 分钟框架：区分 authentication、authorization、consent 和 approval；按风险分级；checkpoint 保存待批动作；参数或身份变化使旧批准失效；所有决定进入 audit。

关键术语：subject、scope、policy decision、approval receipt。

工程证据：越权拒绝和参数漂移失效测试。

方案取舍：审批过多伤害体验，过少扩大不可逆风险。

常见错误：一个 consent=true 覆盖所有动作。

追问提示：恢复时必须检查 policy 和 expiry，而非盲目重放批准。

TOOL-04 · 沙箱隔离

考察点：执行隔离不等于授权。

30 秒回答：沙箱限制不可信代码或 UI 操作对文件、网络、凭据和资源的影响半径；它不能判断业务上是否允许退款或删除，业务授权仍由治理策略负责。

3 分钟框架：每 run/tenant 隔离 workspace；默认禁网并 allowlist；短期最小凭据；CPU/内存/时间配额；记录命令和 artifact；结束后销毁并监测逃逸。

关键术语：sandbox、network policy、ephemeral credential、quota。

工程证据：越界文件、禁网和租户串扰攻击测试。

方案取舍：隔离强度越高，启动延迟与成本越大。

RAG-04 · Rerank

30 秒回答：模型只能提出 memory candidate；系统根据类型、来源、置信、敏感性和同意做 policy 校验，必要时用户确认，再去重、处理冲突并提交版本化记忆。

3 分钟框架：candidate 提取；事实与推断分离；写入 policy 判断；冲突保留版本或优先权威来源；记录 source_run、subject、purpose、confidence 和 expiry。

关键术语：memory candidate、write policy、provenance、conflict resolution。

工程证据：候选拒绝、用户确认与冲突合并 trace。

方案取舍：自动写入流畅但污染风险高，确认写入更可信但打扰用户。

常见错误：把模型摘要直接 upsert。

追问提示：身份、健康和支付偏好等敏感事实需要更强确认与保留控制。

MEM-05 · 巩固与遗忘

考察点：过时、重复和派生关系。

30 秒回答：按稳定性、最近验证、使用价值和保留目的决定合并、降权、过期或删除；巩固摘要必须保留到原始经历的 provenance，删除要传播到派生记忆和缓存。

3 分钟框架：周期识别重复与冲突；事实类用 last_verified，情景类可衰减，法规要求决定 retention；使用 tombstone 防旧副本复活；验证不可恢复删除。

关键术语：consolidation、decay、TTL、tombstone。

工程证据：重复率、陈旧率和删除传播测试。

方案取舍：摘要降低成本但丢细节，保留原始事件增加治理负担。

常见错误：所有记忆统一按时间衰减。

追问提示：稳定偏好也可能被用户显式更新，不能永不失效。

MEM-06 · 隐私治理

考察点：同意、目的与主体控制。

3 分钟框架：consent receipt 记录版本和范围；敏感类型默认不长期保存；加密和访问审计；撤回触发索引、缓存、派生摘要删除；跨设备依赖服务端身份链。

常见错误：一次同意无限期覆盖所有目的。

追问提示：删除需要验证备份和派生数据的既定处理周期。

MEM-07 · 污染与投毒

常见错误：只在读取时做相似度排序。

GOV-01 · 在线治理与离线评测

考察点：实时控制与质量闭环。

30 秒回答：在线治理在请求路径做身份、授权、预算、guardrail 和 HITL，必须及时阻断风险；离线评测消费 trace 判断质量与安全趋势，再反馈数据、模型和编排。

3 分钟框架：用高风险动作说明 policy 在执行前生效；trace 记录决定与结果；离线 eval 重放切片、校准阈值并生成回归；发布后在线继续监控漂移。

关键术语：online control、offline eval、feedback loop。

工程证据：policy deny trace 与回归数据集 lineage。

方案取舍：同步控制增加延迟，但高风险不能等待离线发现。

常见错误：用每日评测替代实时授权。

追问提示：两者共享 run/step/error/policy version，不共享职责。

GOV-02 · Trace、Log、Metric

考察点：三类可观测信号。

30 秒回答：trace 用 run/span 还原因果链，log 记录离散事件细节，metric 聚合趋势和 SLO；三者用稳定 ID 关联，失败和高风险轨迹不能被普通采样丢掉。

3 分钟框架：模型、RAG、工具各建 span；事件含 phase、latency、token、error；metric 计算成功、P95、成本；控制高基数并对敏感 payload 摘要或脱敏。

关键术语：trace/span、structured log、metric、correlation ID。

工程证据：从一个 run 下钻到错误工具与状态 diff。

方案取舍：全量细节易诊断但成本和隐私压力大。

常见错误：有文本日志就声称“全链路可观测”。

追问提示：高基数 ID 放 trace/log，metric 标签只保留可控维度。

GOV-03 · 身份、授权与审计

考察点：代理执行可追责。

工程证据：授权拒绝、撤权和审计查询测试。

方案取舍：实时重验提高安全但增加策略依赖与延迟。

常见错误：日志只写“Agent 执行成功”。

追问提示：防篡改可用签名/不可变存储，删除要求需按法规设计分层数据。

GOV-04 · Prompt Injection 防御

考察点：纵深防御。

30 秒回答：把外部内容标成非可信数据，限制可见工具和敏感上下文；检测用于降风险，但确定性授权、参数校验、审批和沙箱必须保证模型被操纵也不能越权。

3 分钟框架：输入来源标记；指令与数据隔离；最小权限 tool set；输出与动作 policy；敏感动作 HITL；离线红队覆盖直接、间接、多轮和工具结果注入。

工程证据：恶意文档触发但 policy deny 的 trace。

方案取舍：检测阈值高会误阻正常内容，不能作为唯一边界。

常见错误：只靠 prompt 或黑名单。

追问提示：成功攻击还需在离线安全评测复现，并检查沙箱影响半径。

GOV-05 · 敏感 Trace

考察点：诊断价值与数据最小化。

常见错误：全量保存 prompt/response 且长期可查。

追问提示：采集前脱敏比查询时遮盖更能降低存储泄露风险。

GOV-06 · HITL 节点

考察点：风险分级与可恢复审批。

3 分钟框架：风险 policy；生成 action summary hash；保存 checkpoint；多渠道审批去重；参数漂移失效；超时取消或转人工；测批准延迟与误阻率。

常见错误：只问“确认吗”而不展示动作影响。

追问提示：HITL 是编排暂停状态，不是前端按钮直调工具。

常见错误：只统计月度账单，运行时没有预算。

常见错误：只改 system prompt 后恢复流量。

EVAL-01 · 为什么需要 Evals

追问提示：任务成功率要配合路径安全、成本和用户影响。

EVAL-02 · Outcome 与 Trajectory

考察点：正确结果与正确过程。

EVAL-03 · Grader 分工

考察点：可验证性与成本。

追问提示：同源模型可能共享盲点，应使用独立人评切片校准。

EVAL-04 · Judge 校准

考察点：评分可信度。

30 秒回答：先建立盲评人工 gold set，测 Judge 与人类的一致率和误差切片；随机交换候选顺序、隐藏模型身份、固定 rubric，并在 Judge 或 prompt 版本变化时重新校准。

方案取舍：pairwise 稳定但比较次数增长。

常见错误：Judge 能输出理由就认为客观。

追问提示：门禁需要误判成本分析，不只是平均相关系数。

常见错误：平均分提升就忽略高风险切片下降。

EVAL-07 · 线上反馈

EVAL-08 · 失败归因

考察点：端到端与组件证据。

30 秒回答：先用 trajectory trace 找到最早偏离：决策、检索证据、工具结果、状态写回或 policy；再用替换实验固定其他组件，证明改变哪一层能恢复结果。

3 分钟框架：阶段 taxonomy；中间 gold signal；重放时替换正确 evidence/tool output/model decision；比较 counterfactual；修复后跑全量和相邻 slice 回归。
