# Agent 应用工程师学习路线

本路线用于避免学习过程变成随机浏览。每一阶段都必须有明确目标、交付物和验收标准。

## 总原则

- 先理解系统边界,再深入单点技术。
- 先能讲清楚,再写 demo,最后做考核。
- 每个模块都要同时覆盖概念、工程、失败模式和面试表达。
- 不以“看过资料”为完成标准,只以“能解释、能实现、能排错、能面试表达”为完成标准。

## 阶段 1: 建立知识地图

目标: 能够清楚解释一个生产级 Agent 应用由哪些层组成,每层解决什么问题,边界在哪里。

交付物:

- Agent 系统架构总览。
- 每个核心组件的一页专题说明。
- 每个组件对应的高频面试问答。

验收标准:

- 能用 10 分钟讲清 Agent 架构全貌。
- 能解释记忆、RAG、工具、编排、评测之间的边界。
- 能指出常见错误设计,例如把 evals 放进实时请求链路,或让推理内核直连模型厂商。

建议专题顺序:

1. `00-agent-architecture`: 总览。
2. `01-orchestration`: 编排控制层。
3. `02-tool-calling`: 工具 / 动作层。
4. `03-rag`: 知识 / 检索。
5. `04-memory`: 记忆系统。
6. `05-evals`: 评测层。
7. `06-observability-governance`: 治理与可观测。
8. `07-model-gateway-infra`: 模型网关 + 基础设施。

## 阶段 2: 建立工程实践能力

目标: 能把核心概念落成可运行 demo,并能解释工程取舍。

交付物:

- 最小 Agent 编排实验。
- Tool calling / MCP 风格工具实验。
- RAG 检索实验。
- Memory 读写实验。
- Eval pipeline 实验。
- Trace 与失败分析实验。

验收标准:

- 每个实验有 README、运行方式、关键设计说明。
- 能说明为什么这么设计,以及替代方案的代价。
- 能根据失败案例定位问题属于模型、检索、工具、状态、权限还是评测。

建议实验顺序:

1. 最小 ReAct / plan-act-observe loop。
2. 带 schema 的 tool calling。
3. 工具权限、重试和幂等。
4. 简单 RAG pipeline。
5. 用户偏好记忆与任务记忆。
6. LLM-as-judge 评测。
7. trace 采集与失败复盘。
8. 成本控制和模型 fallback。

## 阶段 3: 面试表达与系统设计

目标: 把工程能力转化为面试中的结构化表达。

交付物:

- 高频问答。
- 系统设计题。
- 追问链。
- 项目复盘话术。

验收标准:

- 能回答“如何设计一个客服 Agent / 数据分析 Agent / 代码助手 Agent”。
- 能对追问给出清晰的分层回答。
- 能讲清成本、安全、可观测、评测与上线策略。

建议训练题:

1. 设计一个企业知识库客服 Agent。
2. 设计一个能调用内部系统的工单 Agent。
3. 设计一个数据分析 Agent。
4. 设计一个代码辅助 Agent。
5. 给定一段失败 trace,定位问题并给修复方案。

## 阶段 4: 考核与查漏补缺

目标: 用任务驱动方式验证能力闭环。

交付物:

- 限时系统设计题。
- 限时 debug 题。
- 端到端小项目。
- 自评和复盘记录。

验收标准:

- 设计能覆盖主流程、失败路径、权限、安全、成本和评测。
- 实现能运行,文档能解释,复盘能指出不足。

毕业标准:

- 能独立设计一个 Agent 应用的端到端架构。
- 能实现一个包含工具、RAG、记忆、评测、trace 的最小系统。
- 能解释为什么不用某些复杂方案。
- 能指出系统的主要风险和上线前门禁。
- 能把项目经验转成面试里的结构化回答。
