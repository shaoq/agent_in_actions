# 项目章程

## 一句话定义

`agent_in_actions` 是一个面向 Agent 应用工程师岗位的长期训练工程,用于把 Agent 知识转化为可解释、可实现、可调试、可评测、可面试表达的能力。

## 背景

项目使用者是一名正在系统学习 Agent 相关知识的研发工程师。学习目标不是泛泛了解 AI,而是掌握 Agent 应用工程中的核心技术点,并能在面试和真实工作中证明这些能力。

因此,本项目关注的是“Agent 应用工程能力”,不是纯模型训练、论文追踪、提示词收藏或工具列表堆砌。

## 北极星目标

最终达到以下状态:

- 能用清晰分层讲明一个生产级 Agent 应用如何设计。
- 能实现核心模块的最小可运行版本。
- 能通过 trace、日志和评测定位 Agent 失败原因。
- 能设计权限、安全、成本、评测和上线策略。
- 能在面试中讲出工程取舍,而不是只背概念。

## 能力地图

### 1. 架构能力

理解 Agent 系统的 9 层结构:

1. 交互与接入层
2. 编排控制层
3. 推理内核
4. 工具 / 动作层
5. 记忆系统
6. 知识 / 检索
7. 治理与可观测层
8. 评测层
9. 模型网关 + 基础设施

关键要求:

- 能说明每一层的职责。
- 能说明层与层之间的数据流。
- 能说明哪些能力是在线链路,哪些是离线旁路。
- 能指出常见错误架构。

### 2. 工程能力

能把核心概念落地成可运行实验:

- Agent loop
- planning
- tool calling
- MCP 风格工具注册
- RAG pipeline
- memory read / write
- eval pipeline
- trace and observability
- model routing
- sandbox and permission

关键要求:

- 能运行。
- 能解释。
- 能扩展。
- 能处理失败路径。

### 3. 调试能力

能判断一次 Agent 失败来自哪里:

- 需求理解错误
- prompt 或 system instruction 问题
- 模型能力不足
- 工具 schema 不清
- 工具执行失败
- 检索召回错误
- rerank 或上下文拼接问题
- 记忆污染
- 权限不足
- 状态机流转错误
- 评测标准不合理

### 4. 安全与治理能力

能设计基本上线保护:

- prompt injection 防御
- tool permission
- audit log
- rate limit
- budget control
- human in the loop
- PII and data boundary
- sandbox execution
- red team eval

### 5. 面试表达能力

能把上面的能力转成面试表达:

- 先讲问题定义。
- 再讲架构分层。
- 再讲关键链路。
- 再讲失败模式。
- 再讲治理与评测。
- 最后讲取舍和下一步。

## 内容边界

应该放入本项目:

- Agent 架构文档。
- Agent 专题学习笔记。
- Agent 实验代码。
- 面试题和回答。
- 系统设计题。
- 考核任务和评分标准。
- 学习复盘。

不应该放入本项目:

- 与 Agent 无关的业务表格。
- 临时下载文件。
- 纯新闻摘录。
- 未加工的文章收藏。
- 和学习目标无关的前端页面。
- 不可运行且无说明的代码片段。

## 决策原则

当不确定是否要新增内容时,按以下顺序判断:

1. 是否服务于 Agent 应用工程能力?
2. 是否能归入某个能力模块?
3. 是否能留下明确产物?
4. 是否能支持面试表达或工程实现?

只有满足这些条件,才值得进入项目。
