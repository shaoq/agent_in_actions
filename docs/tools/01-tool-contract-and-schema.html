01 工具契约与 Schema

学习目标

  • 把工具建模为稳定 capability，而不是一段 prompt 描述或任意函数。
  • 同时设计输入、输出、错误和副作用契约。
  • 区分模型侧结构化意图与执行侧权威请求。
  • 用版本化和兼容策略安全演进工具。

工具是受控能力契约

生产工具通常包含：稳定 capability_id、人类可读用途、输入/输出 schema、错误分类、风险与副作用声明、超时与幂等能力、所需权限、所有者和版本。模型看见的是裁剪后的能力描述；执行器使用的是注册表中的权威契约，不能信任模型回传的 endpoint、credential 或 risk level。

四类契约

契约必须说明典型缺陷
Input必填、类型、枚举、格式、上限、互斥条件、引用语义允许自由 SQL / shell / URL；金额单位不清
Output成功状态、业务字段、provenance、freshness、截断与分页只返回自然语言，无法确定性消费
Errorvalidation、authz、rate_limit、timeout、conflict、dependency、unknown_outcome所有异常压成 success:false 或一段文本
Effectread/write/external communication、可逆性、幂等支持、审批要求工具名字像查询，实际触发写入

示例契约

{
  "capability_id": "payments.refund.v2",
  "input": {
    "order_id": "ORD-42",
    "amount_minor": 1299,
    "currency": "CNY",
    "reason_code": "duplicate"
  },
  "execution": {
    "idempotency_key": "run-8:step-4:refund",
    "approval_ref": "approval-91",
    "contract_version": "2.3"
  }
}

敏感凭据、真实 endpoint、租户隔离信息应由执行侧根据可信身份绑定；不能让模型生成。领域约束（可退余额、币种一致、订单归属）必须由权威数据和确定性校验判断。

Schema 设计规则

  • 使用 additionalProperties: false 和明确枚举，减少隐式字段。
  • 用最小领域参数，不接受拼接后的危险指令；资源引用使用不可混淆的稳定 ID。
  • 把字符串长度、数组元素数、批量规模和数值范围写入约束，避免资源耗尽。
  • 输出 schema 包含 status、data、provenance 与 error 的互斥结构。
  • 危险动作的 capability 应与只读查询分开；不要用一个 mode 把权限差异藏在参数里。

版本与兼容

变化兼容性做法
新增 optional 输出字段通常向后兼容消费者忽略未知字段；记录契约版本
收紧输入枚举/必填可能破坏旧调用新 major capability 或迁移窗口
副作用或审批等级变化安全语义不兼容新版本并重新审核，禁止静默切换
下游 API 变化由 adapter 吸收时可保持上层稳定契约测试 + canary + rollback

工程要点

  • 建立 capability registry，记录 owner、版本、risk、permissions、SLO、schema 指纹和弃用日期。
  • 契约测试覆盖正常、边界、越权、超限、部分结果与下游错误。
  • 把模型可见描述与执行权威元数据分离，描述被 prompt injection 污染也不能改变执行策略。
  • 日志记录 capability/version/schema fingerprint，不记录明文 secret。

常见坑

  1. 把 SDK 函数签名直接当工具契约，遗漏业务语义和错误。
  2. 用自然语言 success message 代替结构化 observation。
  3. 一个万能工具包办查询和写入，最小权限无法成立。
  4. 无版本修改 schema，回放和故障归因失去依据。
  5. 把模型声明的 risk 或 user_id 当可信事实。

理论追问

请先脱稿回答，再展开参考答案核对边界、证据和取舍。

  1. 为什么工具 Schema 是能力契约，而不只是函数参数说明？

    查看参考答案

    它还要声明工具意图、前置条件、权限范围、副作用、错误类型、版本与输出证据，供模型选择、编排校验和调用方兼容。只描述参数类型会让同名能力混淆，也无法判断一次调用是否安全和成功。

  2. 参数通过类型校验后，还需要哪些校验？

    查看参考答案

    还需要语义与业务校验，例如金额范围、资源版本、字段组合、主体是否有权访问，以及动作是否需要审批。类型正确的 account_id 仍可能属于他人；Schema 校验不能替代授权和业务不变量。

  3. 新增必填字段时，为什么不能直接覆盖旧工具版本？

    查看参考答案

    已有 Prompt、客户端和在途 Run 可能仍生成旧参数，直接覆盖会造成不可恢复解析失败。应使用版本化契约、兼容默认值或迁移窗口，记录 schema version，并在回放集上验证旧调用和新调用。

  4. 工具错误为什么必须类型化而不是只返回一段文本？

    查看参考答案

    编排需要区分 invalid_input、unauthorized、conflict、rate_limited、temporary_failure 和 effect_unknown，才能决定修复、拒绝、退避、查询或转人工。自由文本难以稳定路由，也不利于指标聚合和回归。

  5. 一个“删除文件”工具把 recursive 默认为 true，有什么风险，如何修正？

    查看参考答案

    危险默认值会让缺少参数变成扩大副作用。应默认最小影响、要求显式目标和递归标志，限制允许目录，执行前展示审批摘要，并在沙箱内校验最终解析路径；高风险动作还应支持 dry-run 和审计 receipt。

面试表达

3 分钟结构

先定义工具是 capability contract；再讲 input、output、error、effect 四类契约；然后说明模型只产生 ActionIntent，执行侧从注册表解析权威契约并绑定身份凭据；最后补版本演进、契约测试、schema 指纹和越权/副作用防护。

学习产出

  1. 为退款能力写四类契约与 6 个类型化错误。
  2. 列出模型绝不能提供的 5 个执行字段。
  3. 设计一次不兼容 schema 迁移与回滚路径。
