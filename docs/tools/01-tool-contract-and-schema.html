01 工具契约与 Schema

学习目标

  • 把工具建模为稳定 capability，而不是一段 prompt 描述或任意函数。
  • 同时设计输入、输出、错误和副作用契约。
  • 区分模型侧结构化意图与执行侧权威请求。
  • 用版本化和兼容策略安全演进工具。

链路总览与现有讲解

工具契约与 Schema属于工具与动作层：输入是类型化 ToolRequest、actor、授权 / 审批引用和幂等信息，输出是ActionReceipt、Observation 或类型化 ToolError。学习时先沿现有图、表或主题章节理解链路，再用后续 stage 检查每个中间产物的不变量。

工具是受控能力契约

生产工具通常包含：稳定 capability_id、人类可读用途、输入/输出 schema、错误分类、风险与副作用声明、超时与幂等能力、所需权限、所有者和版本。模型看见的是裁剪后的能力描述；执行器使用的是注册表中的权威契约，不能信任模型回传的 endpoint、credential 或 risk level。

四类契约

契约必须说明典型缺陷
Input必填、类型、枚举、格式、上限、互斥条件、引用语义允许自由 SQL / shell / URL；金额单位不清
Output成功状态、业务字段、provenance、freshness、截断与分页只返回自然语言，无法确定性消费
Errorvalidation、authz、rate_limit、timeout、conflict、dependency、unknown_outcome所有异常压成 success:false 或一段文本
Effectread/write/external communication、可逆性、幂等支持、审批要求工具名字像查询，实际触发写入

示例契约

{
  "capability_id": "payments.refund.v2",
  "input": {
    "order_id": "ORD-42",
    "amount_minor": 1299,
    "currency": "CNY",
    "reason_code": "duplicate"
  },
  "execution": {
    "idempotency_key": "run-8:step-4:refund",
    "approval_ref": "approval-91",
    "contract_version": "2.3"
  }
}

敏感凭据、真实 endpoint、租户隔离信息应由执行侧根据可信身份绑定；不能让模型生成。领域约束（可退余额、币种一致、订单归属）必须由权威数据和确定性校验判断。

Schema 设计规则

  • 使用 additionalProperties: false 和明确枚举，减少隐式字段。
  • 用最小领域参数，不接受拼接后的危险指令；资源引用使用不可混淆的稳定 ID。
  • 把字符串长度、数组元素数、批量规模和数值范围写入约束，避免资源耗尽。
  • 输出 schema 包含 status、data、provenance 与 error 的互斥结构。
  • 危险动作的 capability 应与只读查询分开；不要用一个 mode 把权限差异藏在参数里。

版本与兼容

变化兼容性做法
新增 optional 输出字段通常向后兼容消费者忽略未知字段；记录契约版本
收紧输入枚举/必填可能破坏旧调用新 major capability 或迁移窗口
副作用或审批等级变化安全语义不兼容新版本并重新审核，禁止静默切换
下游 API 变化由 adapter 吸收时可保持上层稳定契约测试 + canary + rollback

核心环节：从输入到可验证输出

1. 工具 Schema

工具 Schema是生产者与消费者共同承认的版本化解释边界；结构合法不代表语义、授权或业务结果成立。

职责与 I/O
tool-owner 在“工具 Schema”环节接收 类型化 ToolRequest、actor、授权 / 审批引用和幂等信息，产出 ActionReceipt、Observation 或类型化 ToolError；输出必须能由下游按版本消费。
工作机制
显式 schema 定义字段、枚举、状态转移和错误联合，消费者先做语法校验，再按领域不变量做语义校验。本环节把 类型化 ToolRequest、actor、授权 / 审批引用和幂等信息 中与“工具 Schema”相关的事实转成 ActionReceipt、Observation 或类型化 ToolError 的一部分。
必须成立的约束
必填字段、版本兼容、非法状态、未知枚举和错误语义必须确定；不能用自由文本默认值掩盖缺失事实。本章共同约束是：调用契约、使用点授权和副作用协议必须同时成立；违反时必须产生可区分的拒绝、未知或升级结果。
难点与取舍
严格契约提升可恢复性但增加演进成本；兼容窗口由调用方升级节奏和旧数据回放覆盖决定。本层总体取舍是 通用接口与语义清晰度、自动重试与副作用风险，参数必须引用版本化数据、SLO 或风险基线。
边界与注意事项
契约约束信息形状和解释，不授予权限、不选择业务策略，也不证明下游验收成功。工具与动作层 的总边界是：不负责规划任务、授予权限或把技术成功定义成业务验收；错误会继续污染 RunState、后续 Decision、审计记录和最终业务状态。

3. 兼容性测试

兼容性测试必须先满足能力、风险和兼容硬约束，再比较质量、延迟与成本；名称或模型自报不能作为路由证据。

工作机制
能力画像裁剪不可用候选，版本化策略对剩余候选评分，adapter 规范输入输出并记录 route_reason 与替代项。本环节把 类型化 ToolRequest、actor、授权 / 审批引用和幂等信息 中与“兼容性测试”相关的事实转成 ActionReceipt、Observation 或类型化 ToolError 的一部分。

结构化输出、工具支持、上下文、区域和数据策略等硬约束不可被低价覆盖；fallback 必须兼容安全与语义。本章共同约束是：调用契约、使用点授权和副作用协议必须同时成立；违反时必须产生可区分的拒绝、未知或升级结果。

难点与取舍

冲突目标参数依据决策边界
通用接口与语义清晰度、自动重试与副作用风险使用版本化回放、失败切片、SLO 分位数和风险等级，不把教学值当生产默认。tool-owner只调整本层机制；跨层变化需要对应 owner 与独立验收。

失败传播与 first divergence

  1. 上游输入：先确认 类型化 ToolRequest、actor、授权 / 审批引用和幂等信息 的身份、作用域、时间与版本是否一致；输入错误不应在本层被“修正”为新事实。
  2. 首个偏离：按 工具 Schema → 语义契约 → 兼容性测试 → 错误分类 → 契约发布 比较预期与实际中间产物，第一个不满足概念不变量的 stage 才是本章候选根因。
  3. 下游影响：本层错误会改变 RunState、后续 Decision、审计记录和最终业务状态；最终症状可能在更晚阶段出现，但不能从症状直接倒推 Prompt。
  4. 归属边界：tool-owner修复本层契约或机制；若 first divergence 位于相邻层，保留证据并交给相应 owner，不跨层调参。

工程要点

  • 建立 capability registry，记录 owner、版本、risk、permissions、SLO、schema 指纹和弃用日期。
  • 契约测试覆盖正常、边界、越权、超限、部分结果与下游错误。
  • 把模型可见描述与执行权威元数据分离，描述被 prompt injection 污染也不能改变执行策略。
  • 日志记录 capability/version/schema fingerprint，不记录明文 secret。

常见坑

  1. 把 SDK 函数签名直接当工具契约，遗漏业务语义和错误。
  2. 用自然语言 success message 代替结构化 observation。
  3. 一个万能工具包办查询和写入，最小权限无法成立。
  4. 无版本修改 schema，回放和故障归因失去依据。
  5. 把模型声明的 risk 或 user_id 当可信事实。

理论追问

请先脱稿回答，再展开参考答案核对边界、证据和取舍。

  1. 为什么工具 Schema 是能力契约，而不只是函数参数说明？

    查看参考答案

    它还要声明工具意图、前置条件、权限范围、副作用、错误类型、版本与输出证据，供模型选择、编排校验和调用方兼容。只描述参数类型会让同名能力混淆，也无法判断一次调用是否安全和成功。

  2. 参数通过类型校验后，还需要哪些校验？

    查看参考答案

    还需要语义与业务校验，例如金额范围、资源版本、字段组合、主体是否有权访问，以及动作是否需要审批。类型正确的 account_id 仍可能属于他人；Schema 校验不能替代授权和业务不变量。

  3. 新增必填字段时，为什么不能直接覆盖旧工具版本？

    查看参考答案

    已有 Prompt、客户端和在途 Run 可能仍生成旧参数，直接覆盖会造成不可恢复解析失败。应使用版本化契约、兼容默认值或迁移窗口，记录 schema version，并在回放集上验证旧调用和新调用。

  4. 工具错误为什么必须类型化而不是只返回一段文本？

    查看参考答案

    编排需要区分 invalid_input、unauthorized、conflict、rate_limited、temporary_failure 和 effect_unknown，才能决定修复、拒绝、退避、查询或转人工。自由文本难以稳定路由，也不利于指标聚合和回归。

  5. 一个“删除文件”工具把 recursive 默认为 true，有什么风险，如何修正？

    查看参考答案

    危险默认值会让缺少参数变成扩大副作用。应默认最小影响、要求显式目标和递归标志，限制允许目录，执行前展示审批摘要，并在沙箱内校验最终解析路径；高风险动作还应支持 dry-run 和审计 receipt。

3 分钟结构

先定义工具是 capability contract；再讲 input、output、error、effect 四类契约；然后说明模型只产生 ActionIntent，执行侧从注册表解析权威契约并绑定身份凭据；最后补版本演进、契约测试、schema 指纹和越权/副作用防护。

日常实践关键点

核心判断

本章的日常工作不是记住名词，而是把“工具契约与 Schema”变成可执行控制环：先识别触发信号，再用版本化证据定位首个偏离点，由 tool-owner 执行动作，最后用成功、拒绝、降级三类结果验收。推荐顺序是 审阅工具 Schema → 补齐语义契约 → 执行兼容性测试 → 维护错误分类 → 门禁契约发布；任何参数都必须从真实 SLO、风险基线或失败样本推导。

深度实践卡

  • 1. 审阅工具 Schema
    触发信号
    当企业差旅 Agent 的“工具契约与 Schema”链路出现参数虽合法但语义错误、重复副作用、授权上下文丢失、Observation 被误判或适配器行为漂移，或变更直接触及“审阅工具 Schema”的输入、策略或 owner 时，必须启动这项检查。
    诊断顺序
    诊断“审阅工具 Schema”时先冻结 tool_version、schema、actor、授权、idempotency_key、调用状态与 observation。对照生产者与消费者的版本化结构，定位第一个字段缺失、枚举漂移或非法状态转移。
    执行动作
    落实“审阅工具 Schema”时，将输入、输出、错误、版本兼容和拒绝语义写入机器可校验契约，并为未知字段保留显式策略。由 tool-owner 记录执行前后的版本、决定与接管条件。
    验证方法
    验收“审阅工具 Schema”时，用契约测试、重复调用、拒绝路径、沙箱负测和 Trace 回放分别验证调用成功与业务成功；同时比较正确、拒绝和降级结果，阈值只从真实基线与风险容忍度推导。
    止损与取舍
    保留向后兼容读路径，写入回退到上一稳定版本，禁止静默猜测缺失字段；停用新工具版本或切为 dry-run，冻结副作用动作，依据 ledger 执行补偿或人工核对。对“审阅工具 Schema”先保证可恢复和可审计，再优化自动化率。
    面试记忆点
    围绕“审阅工具 Schema”准备一条“症状—首个偏离—动作—证据—止损”的真实或个人实验 Trace，并明确自己只承担的 tool-owner 职责。

  • 3. 执行兼容性测试
    触发信号
    当企业差旅 Agent 的“工具契约与 Schema”链路出现参数虽合法但语义错误、重复副作用、授权上下文丢失、Observation 被误判或适配器行为漂移，或变更直接触及“执行兼容性测试”的输入、策略或 owner 时，必须启动这项检查。
    诊断顺序
    诊断“执行兼容性测试”时先冻结 tool_version、schema、actor、授权、idempotency_key、调用状态与 observation。固定读写双方版本并回放旧样本，检查默认值、枚举、字段语义和双写 / 双读窗口是否造成不可逆解释差异。
    执行动作
    落实“执行兼容性测试”时，先定义兼容矩阵和迁移不变量，再按影子读取、双写核对、分批切换和旧版本退役的顺序发布。由 tool-owner 记录执行前后的版本、决定与接管条件。
    验证方法
    验收“执行兼容性测试”时，用契约测试、重复调用、拒绝路径、沙箱负测和 Trace 回放分别验证调用成功与业务成功；同时比较正确、拒绝和降级结果，阈值只从真实基线与风险容忍度推导。
    止损与取舍
    迁移窗口保留旧读路径与版本标记；发现语义漂移时停止新写入并从原始记录重建；停用新工具版本或切为 dry-run，冻结副作用动作，依据 ledger 执行补偿或人工核对。对“执行兼容性测试”先保证可恢复和可审计，再优化自动化率。
    面试记忆点
    围绕“执行兼容性测试”准备一条“症状—首个偏离—动作—证据—止损”的真实或个人实验 Trace，并明确自己只承担的 tool-owner 职责。
  • 4. 维护错误分类
    触发信号
    当企业差旅 Agent 的“工具契约与 Schema”链路出现参数虽合法但语义错误、重复副作用、授权上下文丢失、Observation 被误判或适配器行为漂移，或变更直接触及“维护错误分类”的输入、策略或 owner 时，必须启动这项检查。
    诊断顺序
    诊断“维护错误分类”时先冻结 tool_version、schema、actor、授权、idempotency_key、调用状态与 observation。比较候选能力、风险、延迟和成本约束，确认选择依据来自声明能力而非名称或模型自报。
    执行动作
    落实“维护错误分类”时，先裁剪不满足硬约束的候选，再对剩余候选打分，并把选择理由与替代项写入决策记录。由 tool-owner 记录执行前后的版本、决定与接管条件。
    验证方法
    验收“维护错误分类”时，用契约测试、重复调用、拒绝路径、沙箱负测和 Trace 回放分别验证调用成功与业务成功；同时比较正确、拒绝和降级结果，阈值只从真实基线与风险容忍度推导。
    止损与取舍
    路由信号不全时回到确定性默认路径；候选不可替代时明确失败而不是随意降级；停用新工具版本或切为 dry-run，冻结副作用动作，依据 ledger 执行补偿或人工核对。对“维护错误分类”先保证可恢复和可审计，再优化自动化率。
    面试记忆点
    围绕“维护错误分类”准备一条“症状—首个偏离—动作—证据—止损”的真实或个人实验 Trace，并明确自己只承担的 tool-owner 职责。
  • 5. 门禁契约发布
    触发信号
    当企业差旅 Agent 的“工具契约与 Schema”链路出现参数虽合法但语义错误、重复副作用、授权上下文丢失、Observation 被误判或适配器行为漂移，或变更直接触及“门禁契约发布”的输入、策略或 owner 时，必须启动这项检查。
    诊断顺序
    诊断“门禁契约发布”时先冻结 tool_version、schema、actor、授权、idempotency_key、调用状态与 observation。分别核对技术执行、策略合规和业务验收，避免把 HTTP 成功、结构合法或模型自报完成当成最终成功。
    执行动作
    落实“门禁契约发布”时，定义可机器判断的完成条件、hard fail、证据来源和验收 owner；证据不全时返回 unknown / pending 而不是 passed。由 tool-owner 记录执行前后的版本、决定与接管条件。
    验证方法
    验收“门禁契约发布”时，用契约测试、重复调用、拒绝路径、沙箱负测和 Trace 回放分别验证调用成功与业务成功；同时比较正确、拒绝和降级结果，阈值只从真实基线与风险容忍度推导。
    止损与取舍
    任一硬门禁失败就保留上一稳定版本；仅软指标退化时才允许受控试运行并持续观察；停用新工具版本或切为 dry-run，冻结副作用动作，依据 ledger 执行补偿或人工核对。对“门禁契约发布”先保证可恢复和可审计，再优化自动化率。
    面试记忆点
    围绕“门禁契约发布”准备一条“症状—首个偏离—动作—证据—止损”的真实或个人实验 Trace，并明确自己只承担的 tool-owner 职责。

排障顺序

  1. 冻结请求、策略、模型 / 工具、数据和部署版本，避免边复现边变化。
  2. 从最早可验证输入开始，逐段对照契约、决策、动作与验收事件；在首个偏离点停止跨层调参。
  3. 由 tool-owner 判断修复、降级、回滚或转人工，并记录未验证假设。
贯穿案例｜企业差旅 Agent：员工请求预订超预算航班时，只把制度条款当知识证据，把员工靠窗偏好当记忆，把预订动作交给工具层；本章只处理 工具契约与 Schema 的控制与证据。若发现参数虽合法但语义错误、重复副作用、授权上下文丢失、Observation 被误判或适配器行为漂移，先保留 trace_id 与版本，再在本层首个偏离点修复，避免把问题推给 Prompt。

应留证据

最小证据集应能回答谁在何时依据哪个版本、对什么输入做了什么决定，以及结果如何被验证。敏感值使用教学占位或受控引用，不把日志全文复制进面试材料。

代表性工程产物｜artifact-tool-01-contract-matrix
case: enterprise-travel-agent
component: tools
unit: 01
owner: tool-owner
input_version: teaching-v1
policy_or_contract_version: teaching-v3
decision: allow | deny | degrade | escalate
first_divergence: stage-name
evidence_refs: [trace-demo, eval-slice-demo]
validation: passed | failed | unknown
rollback_or_handoff: action-and-owner

以上字段是学习用数据形状，不是生产默认值。实际产物还应记录时间、租户 / 主体作用域、变更来源和保留策略，并能从 evidence_refs 回放本章结论。

实践型面试要点

表达时必须标记经历属于生产实战、个人实验或方案设计，只主张自己实际完成的职责，并用 Trace、评测或变更记录举证。企业差旅案例、示例版本和指标均为教学占位；没有验证线上规模、事故协作或长期效果时要主动说明。

实践面试 1

关联实践：审阅工具 Schema、补齐语义契约

常见问法

在生产或可复现实验中，你如何把“审阅工具 Schema”和“补齐语义契约”落到企业差旅 Agent 的 工具契约与 Schema 链路？

60–90 秒回答骨架

先交代 工具契约与 Schema 的业务风险和 tool-owner 边界；再说触发信号与首个偏离点；随后说明契约、动作和参数依据；最后给出验证、止损和复盘结果。回答要同时覆盖正常路径与拒绝 / 降级路径。

可信证据

围绕“审阅工具 Schema / 补齐语义契约”展示脱敏后的 trace_id、tool_version、schema、actor、授权、idempotency_key、调用状态与 observation、前后对照和失败切片；个人实验还应给出固定输入、复现步骤和观察结果，不虚构线上规模。

高频追问

为什么问题属于 工具与动作层 而不是相邻层？参数如何从 SLO 或风险基线推导？结果未知时怎样避免扩大副作用？tool-owner 不在线时，谁依据什么证据接管？

边界与取舍

“审阅工具 Schema / 补齐语义契约”会增加契约、存储或人工成本，只有错误成本与复现收益覆盖这些成本时才扩大自动化。未处理过该类生产事故时，应明确这是方案设计或个人实验结论。
