工具 / 动作层理论课程

工具 / 动作层（Tools / Actions）是 Agent 的执行适配面：接收已校验、已授权的调用请求，绑定凭据与运行约束，调用外部能力，再返回带来源、状态和错误类型的 observation。它不决定任务目标，不推进整个 run，也不把模型输出当权限凭证。

学习目标

  1. 设计可版本化的工具契约，区分 input、output、error 与 side-effect contract。
  2. 解释 prepare、authorize、dispatch、observe、commit 的调用生命周期与证据。
  3. 根据主体、资源、动作和上下文设计最小权限、审批与执行隔离。
  4. 按错误类别决定重试，使用幂等键、去重记录和补偿控制副作用。
  5. 辨析 Function Calling、MCP 与 Computer Use 的抽象层级和适用边界。

前置知识

  1. 理解九层 Agent 架构。
  2. 完成编排控制层课程，知道 run、step、pause/resume 和终止条件由谁拥有。
  3. 完成推理内核课程，理解 ActionIntent、schema 校验与能力选择。
先推理、后工具 · 推理内核回答“建议做什么”，工具层回答“如何把获准意图安全地做成，并留下什么证据”。两者之间必须经过编排推进与治理授权。

五个理论单元

顺序单元核心产出
01工具契约与 Schema四类契约、版本策略、类型化错误与组件边界
02调用生命周期与 Observation调用状态机、可观测字段与 observation 信任边界
03权限、审批与沙箱最小权限、TOCTOU 防护、审批绑定与隔离责任
04重试、幂等与副作用错误分类、幂等协议、未知结果与补偿设计
05Function Calling、MCP 与 Computer Use接口选择矩阵、信任边界与降级策略

统一组件边界

组件在动作链路中的职责不应交给工具层的事
推理内核从可见能力中提出结构化 ActionIntent工具层不规划下一步、不解释用户目标
编排控制校验步骤阶段，推进 run，暂停、恢复并写回 observation工具层不拥有全局状态机或终止条件
治理与可观测做策略判定、审批、审计和敏感数据规则工具层不自行放宽权限或覆盖 deny
工具 / 动作适配能力、执行请求、规范化结果与错误不直接访问模型厂商，不持有长期记忆
模型网关 + 基础设施提供模型访问、凭据托管、进程/网络/文件隔离等能力工具层声明和使用隔离策略，但底层容器不是工具业务逻辑

完成标准

  • 按顺序读完五个单元，并完成每页“学习产出”。
  • 能对退款、邮件发送或生产变更场景画出从 ActionIntent 到 observation 的链路。
  • 能解释“schema 合法、策略允许、执行成功、业务完成”为什么是四件不同的事。
  • 完成工具 / 动作层理论检查点：总分至少 80，且每项至少达到该项满分 60%。

检查点通过只证明 L2 理论边界与方案设计达标，不等于已经完成 L3 工具运行时实现。
