04 重试、幂等与副作用

学习目标

  • 按 transient、permanent、conflict、unknown outcome 分类错误。
  • 理解“使用幂等键”需要服务端去重协议与持久记录。
  • 区分 retry、status inquiry、resume、compensation 与 manual reconcile。
  • 为副作用定义 precondition、effect receipt 和恢复边界。

链路总览与现有讲解

重试、幂等与副作用属于工具与动作层：输入是类型化 ToolRequest、actor、授权 / 审批引用和幂等信息，输出是ActionReceipt、Observation 或类型化 ToolError。学习时先沿现有图、表或主题章节理解链路，再用后续 stage 检查每个中间产物的不变量。

重试不是默认恢复

错误分类矩阵

类别例子默认动作
Validation / Permanent缺字段、资源不存在、业务规则拒绝不重试；返回类型化错误，由编排澄清或改计划
Authorizationdeny、审批过期不绕过；重新授权或终止动作路径

Conflict版本/余额/状态已变化刷新权威状态并重新决策，不复用旧参数
Dependency outage持续不可用熔断、延迟恢复或人工出口，避免重试风暴

真正的幂等协议

幂等表示同一逻辑请求重复提交不会产生额外业务效果，不等于“HTTP 方法是 PUT”。可靠设计包含：

  1. 由编排/工具层基于 run + step + logical effect 生成稳定幂等键，重试不得换键。
  2. 服务端或可信 adapter 原子保存 key、规范化请求摘要、状态与 effect receipt。
  3. 相同 key + 相同摘要返回原结果；相同 key + 不同摘要必须 conflict。
  4. 记录的 TTL 覆盖客户端最大重试与恢复窗口；过期策略显式化。
  5. 并发首次请求需要原子 claim，避免两次都进入真实副作用。
idempotency_key = run-8:step-4:refund
request_digest = hash(order=ORD-42, amount=1299, currency=CNY)

NEW       -> atomically claim -> EXECUTING
EXECUTING -> duplicate waits/queries
SUCCEEDED -> return stored effect_receipt
FAILED_SAFE -> retry only if policy allows
UNKNOWN   -> reconcile before any new effect

副作用等级

类型示例控制
只读查询订单仍需授权、限流、隐私控制；通常可安全重试
可覆盖写更新草稿标题版本条件/ETag，避免覆盖并发修改

补偿不是回滚

分布式副作用通常无法 ACID 回滚。Compensation 是一个新的、可失败且需授权的业务动作，例如退款不是“撤销支付”，撤销邮件也无法让收件人忘记内容。因此要保存原 effect receipt、补偿原因、审批与新幂等键，并允许进入 manual reconcile。

重试预算

  • 同时受单调用次数、step 时长、run 总时长、成本和下游 rate limit 约束。
  • 指数退避加 jitter；尊重 Retry-After，避免所有 worker 同步重试。
  • 熔断器减少持续故障时的放大；half-open 探测与业务流量分开。
  • 推理修复、编排重试与工具网络重试分别计数，避免层层各重试 3 次造成指数放大。

核心环节：从输入到可验证输出

1. 按错误分类重试

按错误分类重试只有在主体、来源、时间、目的、置信和可撤回状态明确时才是可用记忆，不等于聊天文本或外部知识副本。

职责与 I/O
tool-owner 在“按错误分类重试”环节接收 类型化 ToolRequest、actor、授权 / 审批引用和幂等信息，产出 ActionReceipt、Observation 或类型化 ToolError；输出必须能由下游按版本消费。
工作机制
经历先成为 candidate，经验证与同意晋升；读取按作用域、时效、类型和置信排序，冲突保留版本而非覆盖历史。本环节把 类型化 ToolRequest、actor、授权 / 审批引用和幂等信息 中与“按错误分类重试”相关的事实转成 ActionReceipt、Observation 或类型化 ToolError 的一部分。
必须成立的约束
MemoryRecord 必须带 subject / tenant、provenance、consent、policy_version 与 TTL，删除要传播到索引、缓存和派生摘要。本章共同约束是：调用契约、使用点授权和副作用协议必须同时成立；违反时必须产生可区分的拒绝、未知或升级结果。
难点与取舍
更高召回能增强个性化，也会增加旧事实、投毒和隐私风险；读取与保留参数由时间切片评测和用户控制要求决定。本层总体取舍是 通用接口与语义清晰度、自动重试与副作用风险，参数必须引用版本化数据、SLO 或风险基线。
边界与注意事项
RunState 管当前执行，Session 管交互连续性，RAG 管外部资料；Memory 只沉淀经治理的经历。工具与动作层 的总边界是：不负责规划任务、授予权限或把技术成功定义成业务验收；错误会继续污染 RunState、后续 Decision、审计记录和最终业务状态。

2. 幂等键

幂等键只能把系统带回已知状态或安全终态；未知结果在查明副作用前不得被当成可重试失败。

职责与 I/O
tool-owner 在“幂等键”环节接收 类型化 ToolRequest、actor、授权 / 审批引用和幂等信息，产出 ActionReceipt、Observation 或类型化 ToolError；输出必须能由下游按版本消费。
工作机制
错误先分瞬时、永久、冲突与 unknown outcome；幂等键、ledger、checkpoint 和补偿记录共同决定 retry、query、resume 或人工核对。本环节把 类型化 ToolRequest、actor、授权 / 审批引用和幂等信息 中与“幂等键”相关的事实转成 ActionReceipt、Observation 或类型化 ToolError 的一部分。
必须成立的约束
尝试次数、累计 deadline、退避、副作用作用域和补偿 owner 都必须有界，幂等键需覆盖业务动作而非单次请求。本章共同约束是：调用契约、使用点授权和副作用协议必须同时成立；违反时必须产生可区分的拒绝、未知或升级结果。
难点与取舍
重试提高瞬时可用性也会放大负载与副作用；预算由下游容量、用户 deadline 和动作不可逆性决定。本层总体取舍是 通用接口与语义清晰度、自动重试与副作用风险，参数必须引用版本化数据、SLO 或风险基线。
边界与注意事项
补偿是新的业务动作而非时间回滚；网关 fallback 不应掩盖工具副作用或编排非法状态。工具与动作层 的总边界是：不负责规划任务、授予权限或把技术成功定义成业务验收；错误会继续污染 RunState、后续 Decision、审计记录和最终业务状态。

3. 标注副作用等级

标注副作用等级必须先满足能力、风险和兼容硬约束，再比较质量、延迟与成本；名称或模型自报不能作为路由证据。

结构化输出、工具支持、上下文、区域和数据策略等硬约束不可被低价覆盖；fallback 必须兼容安全与语义。本章共同约束是：调用契约、使用点授权和副作用协议必须同时成立；违反时必须产生可区分的拒绝、未知或升级结果。

网关选择模型与规范协议，不选择业务工具、不拥有推理意图，也不授予工具执行权限。工具与动作层 的总边界是：不负责规划任务、授予权限或把技术成功定义成业务验收；错误会继续污染 RunState、后续 Decision、审计记录和最终业务状态。

4. 补偿结果

补偿结果是生产者与消费者共同承认的版本化解释边界；结构合法不代表语义、授权或业务结果成立。

职责与 I/O
tool-owner 在“补偿结果”环节接收 类型化 ToolRequest、actor、授权 / 审批引用和幂等信息，产出 ActionReceipt、Observation 或类型化 ToolError；输出必须能由下游按版本消费。
工作机制
显式 schema 定义字段、枚举、状态转移和错误联合，消费者先做语法校验，再按领域不变量做语义校验。本环节把 类型化 ToolRequest、actor、授权 / 审批引用和幂等信息 中与“补偿结果”相关的事实转成 ActionReceipt、Observation 或类型化 ToolError 的一部分。
必须成立的约束
必填字段、版本兼容、非法状态、未知枚举和错误语义必须确定；不能用自由文本默认值掩盖缺失事实。本章共同约束是：调用契约、使用点授权和副作用协议必须同时成立；违反时必须产生可区分的拒绝、未知或升级结果。
难点与取舍
严格契约提升可恢复性但增加演进成本；兼容窗口由调用方升级节奏和旧数据回放覆盖决定。本层总体取舍是 通用接口与语义清晰度、自动重试与副作用风险，参数必须引用版本化数据、SLO 或风险基线。
边界与注意事项
契约约束信息形状和解释，不授予权限、不选择业务策略，也不证明下游验收成功。工具与动作层 的总边界是：不负责规划任务、授予权限或把技术成功定义成业务验收；错误会继续污染 RunState、后续 Decision、审计记录和最终业务状态。

难点与取舍

冲突目标参数依据决策边界
通用接口与语义清晰度、自动重试与副作用风险使用版本化回放、失败切片、SLO 分位数和风险等级，不把教学值当生产默认。tool-owner只调整本层机制；跨层变化需要对应 owner 与独立验收。

失败传播与 first divergence

  1. 上游输入：先确认 类型化 ToolRequest、actor、授权 / 审批引用和幂等信息 的身份、作用域、时间与版本是否一致；输入错误不应在本层被“修正”为新事实。
  2. 首个偏离：按 按错误分类重试 → 幂等键 → 标注副作用等级 → 补偿结果 → 重试预算 比较预期与实际中间产物，第一个不满足概念不变量的 stage 才是本章候选根因。
  3. 下游影响：本层错误会改变 RunState、后续 Decision、审计记录和最终业务状态；最终症状可能在更晚阶段出现，但不能从症状直接倒推 Prompt。
  4. 归属边界：tool-owner修复本层契约或机制；若 first divergence 位于相邻层，保留证据并交给相应 owner，不跨层调参。

工程要点

  • trace 记录 dispatch boundary、idempotency key 的安全摘要、attempt、result class 和 effect receipt。
  • 副作用前校验 precondition/version，副作用后先持久化 receipt 再对上层报告成功。
  • 批量动作按 item 返回结果；避免一个 timeout 隐藏部分成功。
  • 为 unknown outcome 建立运维队列和 SLA，而不是永久卡在“重试中”。

常见坑

  1. 每次重试生成新幂等键。
  2. 客户端带 key，但服务端不持久去重。
  3. timeout 立刻再次扣款/发送。
  4. 把补偿描述成绝对回滚，忽略新的失败与权限。
  5. 重试发生在多个层级，却没有统一预算。

理论追问

请先脱稿回答，再展开参考答案核对边界、证据和取舍。

  1. 哪些错误可以自动重试，哪些错误不应重试？

    查看参考答案

    短暂网络故障、明确未提交的限流或可重放只读请求可在预算内退避重试；参数错误、权限拒绝、业务冲突和未知副作用不能直接重试。先分类并确认副作用状态，再决定策略。

  2. 幂等键怎样设计才真正约束同一业务意图？

    查看参考答案

    键应由稳定业务意图生成，并绑定主体、目标资源、动作类型和关键参数，而不是每次尝试随机生成。服务端必须原子保存键与结果，相同键同参数返回原结果，不同参数复用同键应拒绝。

  3. 补偿为什么不等于数据库回滚？

    查看参考答案

    外部副作用可能已被用户看到、触发邮件或跨系统传播，补偿只是新的反向业务动作，可能失败且无法完全恢复原状态。必须建模补偿前置、权限、幂等、审计和残余风险。

  4. effect unknown 状态下为什么查询优先于重试？

    查看参考答案

    因为请求可能已在外部系统成功，重试会造成重复退款、工单或发信。先用 call_id、幂等键或业务唯一键查询真实状态；只有证据证明未提交，才允许再次执行。

  5. 重试预算应如何设置并进入 State？

    查看参考答案

    按工具风险、错误类型、SLO 和总任务预算设置最大次数、总耗时与成本，并记录 attempt、上次错误、退避时间和已消耗预算。达到软阈值可降级，达到硬阈值必须停止或转人工，不能由模型自行扩张。

超时后的处理

先确定是否已 dispatch；未派发的瞬时错误可退避重试。已派发但无响应属于 unknown outcome，先用稳定幂等键或业务查询接口核对 effect。只有确认安全且在统一预算内才重试；非幂等或不可逆动作进入补偿/人工核对，并保留 effect receipt 和审计链。

日常实践关键点

核心判断

本章的日常工作不是记住名词，而是把“重试、幂等与副作用”变成可执行控制环：先识别触发信号，再用版本化证据定位首个偏离点，由 tool-owner 执行动作，最后用成功、拒绝、降级三类结果验收。推荐顺序是 按错误分类重试 → 设计幂等键 → 标注副作用等级 → 记录补偿结果 → 限制重试预算；任何参数都必须从真实 SLO、风险基线或失败样本推导。

深度实践卡

  • 1. 按错误分类重试
    触发信号
    当企业差旅 Agent 的“重试、幂等与副作用”链路出现参数虽合法但语义错误、重复副作用、授权上下文丢失、Observation 被误判或适配器行为漂移，或变更直接触及“按错误分类重试”的输入、策略或 owner 时，必须启动这项检查。
    诊断顺序
    诊断“按错误分类重试”时先冻结 tool_version、schema、actor、授权、idempotency_key、调用状态与 observation。按 owner、生命周期、作用域、真相来源和更新方式逐项分类，并检查读取策略是否让低置信旧记录覆盖当前显式输入。
    执行动作
    落实“按错误分类重试”时，为 State、Session、Memory 与 Knowledge 设置不同 schema、TTL 和读取优先级；冲突时优先当前确认事实并返回来源。由 tool-owner 记录执行前后的版本、决定与接管条件。
    验证方法
    验收“按错误分类重试”时，用契约测试、重复调用、拒绝路径、沙箱负测和 Trace 回放分别验证调用成功与业务成功；同时比较正确、拒绝和降级结果，阈值只从真实基线与风险容忍度推导。
    止损与取舍
    分类或读取依据不明时不晋升长期记忆，关闭该类自动注入并通过消融确认影响；停用新工具版本或切为 dry-run，冻结副作用动作，依据 ledger 执行补偿或人工核对。对“按错误分类重试”先保证可恢复和可审计，再优化自动化率。
    面试记忆点
    围绕“按错误分类重试”准备一条“症状—首个偏离—动作—证据—止损”的真实或个人实验 Trace，并明确自己只承担的 tool-owner 职责。

排障顺序

  1. 冻结请求、策略、模型 / 工具、数据和部署版本，避免边复现边变化。
  2. 从最早可验证输入开始，逐段对照契约、决策、动作与验收事件；在首个偏离点停止跨层调参。
  3. 由 tool-owner 判断修复、降级、回滚或转人工，并记录未验证假设。
贯穿案例｜企业差旅 Agent：员工请求预订超预算航班时，只把制度条款当知识证据，把员工靠窗偏好当记忆，把预订动作交给工具层；本章只处理 重试、幂等与副作用 的控制与证据。若发现参数虽合法但语义错误、重复副作用、授权上下文丢失、Observation 被误判或适配器行为漂移，先保留 trace_id 与版本，再在本层首个偏离点修复，避免把问题推给 Prompt。

最小证据集应能回答谁在何时依据哪个版本、对什么输入做了什么决定，以及结果如何被验证。敏感值使用教学占位或受控引用，不把日志全文复制进面试材料。

实践型面试要点

表达时必须标记经历属于生产实战、个人实验或方案设计，只主张自己实际完成的职责，并用 Trace、评测或变更记录举证。企业差旅案例、示例版本和指标均为教学占位；没有验证线上规模、事故协作或长期效果时要主动说明。

实践面试 1

关联实践：按错误分类重试、设计幂等键

常见问法

在生产或可复现实验中，你如何把“按错误分类重试”和“设计幂等键”落到企业差旅 Agent 的 重试、幂等与副作用 链路？

60–90 秒回答骨架

先交代 重试、幂等与副作用 的业务风险和 tool-owner 边界；再说触发信号与首个偏离点；随后说明契约、动作和参数依据；最后给出验证、止损和复盘结果。回答要同时覆盖正常路径与拒绝 / 降级路径。

可信证据

围绕“按错误分类重试 / 设计幂等键”展示脱敏后的 trace_id、tool_version、schema、actor、授权、idempotency_key、调用状态与 observation、前后对照和失败切片；个人实验还应给出固定输入、复现步骤和观察结果，不虚构线上规模。

高频追问

为什么问题属于 工具与动作层 而不是相邻层？参数如何从 SLO 或风险基线推导？结果未知时怎样避免扩大副作用？tool-owner 不在线时，谁依据什么证据接管？

边界与取舍

“按错误分类重试 / 设计幂等键”会增加契约、存储或人工成本，只有错误成本与复现收益覆盖这些成本时才扩大自动化。未处理过该类生产事故时，应明确这是方案设计或个人实验结论。
