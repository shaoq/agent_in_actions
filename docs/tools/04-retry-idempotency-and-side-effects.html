04 重试、幂等与副作用

学习目标

  • 按 transient、permanent、conflict、unknown outcome 分类错误。
  • 理解“使用幂等键”需要服务端去重协议与持久记录。
  • 区分 retry、status inquiry、resume、compensation 与 manual reconcile。
  • 为副作用定义 precondition、effect receipt 和恢复边界。

重试不是默认恢复

错误分类矩阵

类别例子默认动作
Validation / Permanent缺字段、资源不存在、业务规则拒绝不重试；返回类型化错误，由编排澄清或改计划

Transient after dispatch下游处理后响应丢失未知结果；先按幂等键查询状态
Conflict版本/余额/状态已变化刷新权威状态并重新决策，不复用旧参数
Dependency outage持续不可用熔断、延迟恢复或人工出口，避免重试风暴

真正的幂等协议

幂等表示同一逻辑请求重复提交不会产生额外业务效果，不等于“HTTP 方法是 PUT”。可靠设计包含：

  1. 由编排/工具层基于 run + step + logical effect 生成稳定幂等键，重试不得换键。
  2. 服务端或可信 adapter 原子保存 key、规范化请求摘要、状态与 effect receipt。
  3. 相同 key + 相同摘要返回原结果；相同 key + 不同摘要必须 conflict。
  4. 记录的 TTL 覆盖客户端最大重试与恢复窗口；过期策略显式化。
  5. 并发首次请求需要原子 claim，避免两次都进入真实副作用。
idempotency_key = run-8:step-4:refund
request_digest = hash(order=ORD-42, amount=1299, currency=CNY)

NEW       -> atomically claim -> EXECUTING
EXECUTING -> duplicate waits/queries
SUCCEEDED -> return stored effect_receipt
FAILED_SAFE -> retry only if policy allows
UNKNOWN   -> reconcile before any new effect

副作用等级

类型示例控制
只读查询订单仍需授权、限流、隐私控制；通常可安全重试
可覆盖写更新草稿标题版本条件/ETag，避免覆盖并发修改

补偿不是回滚

分布式副作用通常无法 ACID 回滚。Compensation 是一个新的、可失败且需授权的业务动作，例如退款不是“撤销支付”，撤销邮件也无法让收件人忘记内容。因此要保存原 effect receipt、补偿原因、审批与新幂等键，并允许进入 manual reconcile。

重试预算

  • 同时受单调用次数、step 时长、run 总时长、成本和下游 rate limit 约束。
  • 指数退避加 jitter；尊重 Retry-After，避免所有 worker 同步重试。
  • 熔断器减少持续故障时的放大；half-open 探测与业务流量分开。
  • 推理修复、编排重试与工具网络重试分别计数，避免层层各重试 3 次造成指数放大。

工程要点

  • trace 记录 dispatch boundary、idempotency key 的安全摘要、attempt、result class 和 effect receipt。
  • 副作用前校验 precondition/version，副作用后先持久化 receipt 再对上层报告成功。
  • 批量动作按 item 返回结果；避免一个 timeout 隐藏部分成功。
  • 为 unknown outcome 建立运维队列和 SLA，而不是永久卡在“重试中”。

常见坑

  1. 每次重试生成新幂等键。
  2. 客户端带 key，但服务端不持久去重。
  3. timeout 立刻再次扣款/发送。
  4. 把补偿描述成绝对回滚，忽略新的失败与权限。
  5. 重试发生在多个层级，却没有统一预算。

理论追问

请先脱稿回答，再展开参考答案核对边界、证据和取舍。

  1. 哪些错误可以自动重试，哪些错误不应重试？

    查看参考答案

    短暂网络故障、明确未提交的限流或可重放只读请求可在预算内退避重试；参数错误、权限拒绝、业务冲突和未知副作用不能直接重试。先分类并确认副作用状态，再决定策略。

  2. 幂等键怎样设计才真正约束同一业务意图？

    查看参考答案

    键应由稳定业务意图生成，并绑定主体、目标资源、动作类型和关键参数，而不是每次尝试随机生成。服务端必须原子保存键与结果，相同键同参数返回原结果，不同参数复用同键应拒绝。

  3. 补偿为什么不等于数据库回滚？

    查看参考答案

    外部副作用可能已被用户看到、触发邮件或跨系统传播，补偿只是新的反向业务动作，可能失败且无法完全恢复原状态。必须建模补偿前置、权限、幂等、审计和残余风险。

  4. effect unknown 状态下为什么查询优先于重试？

    查看参考答案

    因为请求可能已在外部系统成功，重试会造成重复退款、工单或发信。先用 call_id、幂等键或业务唯一键查询真实状态；只有证据证明未提交，才允许再次执行。

  5. 重试预算应如何设置并进入 State？

    查看参考答案

    按工具风险、错误类型、SLO 和总任务预算设置最大次数、总耗时与成本，并记录 attempt、上次错误、退避时间和已消耗预算。达到软阈值可降级，达到硬阈值必须停止或转人工，不能由模型自行扩张。

面试表达

超时后的处理

先确定是否已 dispatch；未派发的瞬时错误可退避重试。已派发但无响应属于 unknown outcome，先用稳定幂等键或业务查询接口核对 effect。只有确认安全且在统一预算内才重试；非幂等或不可逆动作进入补偿/人工核对，并保留 effect receipt 和审计链。
