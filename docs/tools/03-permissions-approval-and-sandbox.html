03 权限、审批与沙箱

学习目标

  • 用 subject、action、resource、context 表达授权请求。
  • 设计与动作摘要强绑定、可过期、不可扩大范围的审批。
  • 区分治理策略判定、工具层执行约束与基础设施隔离。
  • 识别 prompt injection、confused deputy、TOCTOU 和凭据泄露。

链路总览与现有讲解

权限、审批与沙箱属于工具与动作层：输入是类型化 ToolRequest、actor、授权 / 审批引用和幂等信息，输出是ActionReceipt、Observation 或类型化 ToolError。学习时先沿现有图、表或主题章节理解链路，再用后续 stage 检查每个中间产物的不变量。

授权是独立确定性判定

模型或工具描述可以推荐动作，但不能授予权限。执行前应构造权威授权输入：

AuthorizationRequest {
  subject: user/service/tenant identity,
  action: capability_id + effect class,
  resource: canonical resource identifiers,
  context: run phase, risk, device, time, approval state,
  request_digest: hash(contract version + normalized arguments)
}

策略由治理层输出 allow、deny 或 require_approval；工具层强制执行该决定并使用短期、最小范围凭据。模型声称“用户已经同意”不构成审批证据。

最小权限的四个维度

维度约束反例
能力只授予当前 capability，读写拆分给 Agent 一个全能云管理员 token
资源限定租户、订单、目录、域名或数据行用用户输入直接选择任意路径/URL
时间短期 credential、deadline、单次或有限次数长期密钥写入 prompt 或环境日志
环境网络 allowlist、只读文件、CPU/内存/时长上限浏览器/代码工具继承宿主全部权限

审批是一份可校验契约

Approval 至少包含 approver identity、request digest、可读摘要、影响范围、有效期、策略版本和一次性使用约束。若参数、资源、contract version 或风险在等待期间改变，旧审批失效并重新判定。

  • 审批前：持久化 checkpoint，展示将做什么、对谁做、能否撤销和预期影响。
  • 审批中：run 进入 waiting，不占用工作线程，不重复调用。
  • 恢复时：重新认证主体、检查资源版本与策略，防止 TOCTOU。
  • 执行后：记录 effect receipt，向审批者/用户反馈结果或未知状态。

沙箱责任边界

位置职责
治理与可观测定义允许的网络、文件、命令、数据和审计策略，决定是否审批
工具 / 动作层声明所需能力，规范化输入，禁止危险拼接，向隔离运行时传最小请求
模型网关 + 基础设施实际提供容器/进程、文件系统、网络、secret broker、资源配额和销毁

“在 Docker 里运行”不自动安全。还需无特权用户、只读根文件系统、临时工作目录、网络默认拒绝、域名/IP 重绑定防护、syscall/资源限制、secret 按需注入和执行后销毁。

威胁与控制

威胁失效方式控制
间接 Prompt Injection网页诱导 Agent 泄密或执行危险工具外部文本标记不可信；权限与数据流策略不听从内容

核心环节：从输入到可验证输出

1. 收窄能力范围

收窄能力范围必须绑定当前 actor、资源、动作、目的和版本，并在真正使用能力或数据的时刻仍然有效。

职责与 I/O
tool-owner 在“收窄能力范围”环节接收 类型化 ToolRequest、actor、授权 / 审批引用和幂等信息，产出 ActionReceipt、Observation 或类型化 ToolError；输出必须能由下游按版本消费。
工作机制
身份链先把渠道主体映射为内部 actor，再由策略决策与短期能力令牌约束数据、工具、文件和网络使用。本环节把 类型化 ToolRequest、actor、授权 / 审批引用和幂等信息 中与“收窄能力范围”相关的事实转成 ActionReceipt、Observation 或类型化 ToolError 的一部分。
必须成立的约束
默认最小权限与 deny-by-default；撤回、角色变更、资源版本变化必须使缓存授权和派生访问失效。本章共同约束是：调用契约、使用点授权和副作用协议必须同时成立；违反时必须产生可区分的拒绝、未知或升级结果。
难点与取舍
频繁再授权更安全但增加延迟；缓存更快却可能陈旧，TTL 应由权限变化频率和风险窗口推导。本层总体取舍是 通用接口与语义清晰度、自动重试与副作用风险，参数必须引用版本化数据、SLO 或风险基线。
边界与注意事项
身份认证不等于授权，脱敏不等于匿名化，网络隔离也不能替代业务策略判断。工具与动作层 的总边界是：不负责规划任务、授予权限或把技术成功定义成业务验收；错误会继续污染 RunState、后续 Decision、审计记录和最终业务状态。

2. 策略预检

策略预检必须先满足能力、风险和兼容硬约束，再比较质量、延迟与成本；名称或模型自报不能作为路由证据。

tool-owner 在“策略预检”环节接收 类型化 ToolRequest、actor、授权 / 审批引用和幂等信息，产出 ActionReceipt、Observation 或类型化 ToolError；输出必须能由下游按版本消费。
工作机制
能力画像裁剪不可用候选，版本化策略对剩余候选评分，adapter 规范输入输出并记录 route_reason 与替代项。本环节把 类型化 ToolRequest、actor、授权 / 审批引用和幂等信息 中与“策略预检”相关的事实转成 ActionReceipt、Observation 或类型化 ToolError 的一部分。

4. 沙箱 Profile

沙箱 Profile必须让不可信输入即使成功诱导模型，也无法越过身份、能力、文件、网络或租户边界。

职责与 I/O
tool-owner 在“沙箱 Profile”环节接收 类型化 ToolRequest、actor、授权 / 审批引用和幂等信息，产出 ActionReceipt、Observation 或类型化 ToolError；输出必须能由下游按版本消费。
工作机制
信任标签、最小能力、短期凭证、进程 / 文件 / 网络隔离与输出检查构成分层防御，负测验证可利用链。本环节把 类型化 ToolRequest、actor、授权 / 审批引用和幂等信息 中与“沙箱 Profile”相关的事实转成 ActionReceipt、Observation 或类型化 ToolError 的一部分。
必须成立的约束
deny-by-default、凭证不进入模型上下文、租户资源不可共享、egress 采用 allowlist，PII 在离开受控域前处理。本章共同约束是：调用契约、使用点授权和副作用协议必须同时成立；违反时必须产生可区分的拒绝、未知或升级结果。
难点与取舍
更强隔离会降低复用和性能；profile 依据副作用、数据敏感度和攻击面分级，而不是所有任务使用同一沙箱。本层总体取舍是 通用接口与语义清晰度、自动重试与副作用风险，参数必须引用版本化数据、SLO 或风险基线。
边界与注意事项
Prompt guardrail 降低内容风险但不是安全边界；真正的执行隔离由基础设施提供，业务授权由治理提供。工具与动作层 的总边界是：不负责规划任务、授予权限或把技术成功定义成业务验收；错误会继续污染 RunState、后续 Decision、审计记录和最终业务状态。

难点与取舍

冲突目标参数依据决策边界
通用接口与语义清晰度、自动重试与副作用风险使用版本化回放、失败切片、SLO 分位数和风险等级，不把教学值当生产默认。tool-owner只调整本层机制；跨层变化需要对应 owner 与独立验收。
自动化收益 vs. 失败放大比较开启 / 关闭“收窄能力范围”与“安全负测”的消融结果、人工负载和恢复时间。证据不足返回 pending / unknown / escalate，不能靠更长 Prompt 隐藏边界。

失败传播与 first divergence

  1. 上游输入：先确认 类型化 ToolRequest、actor、授权 / 审批引用和幂等信息 的身份、作用域、时间与版本是否一致；输入错误不应在本层被“修正”为新事实。
  2. 首个偏离：按 收窄能力范围 → 策略预检 → 审批上下文 → 沙箱 Profile → 安全负测 比较预期与实际中间产物，第一个不满足概念不变量的 stage 才是本章候选根因。
  3. 下游影响：本层错误会改变 RunState、后续 Decision、审计记录和最终业务状态；最终症状可能在更晚阶段出现，但不能从症状直接倒推 Prompt。
  4. 归属边界：tool-owner修复本层契约或机制；若 first divergence 位于相邻层，保留证据并交给相应 owner，不跨层调参。

工程要点

  • 策略判定与执行日志共享不可伪造的 request digest 和 policy version。
  • 拒绝是正常类型化结果，不触发模型改写参数反复试探。
  • 对高风险动作默认 fail closed；观测类低风险能力可按明确策略降级。
  • 审批 UI 避免只显示“是否允许”；必须显示实际资源、金额、收件人和可逆性。

常见坑

  1. 让模型自己判断高风险并决定是否审批。
  2. 审批只绑定工具名，不绑定参数和资源。
  3. 恢复后直接执行，不重验策略和资源版本。
  4. 沙箱能联网任意地址并读取宿主凭据。
  5. 把治理决策逻辑复制到每个 adapter，产生不一致。

理论追问

请先脱稿回答，再展开参考答案核对边界、证据和取舍。

  1. 权限检查通过后，为什么仍然需要沙箱？

    查看参考答案

    权限回答主体是否可对资源执行动作，沙箱限制执行进程能访问的文件、网络、凭据、系统调用和资源。即使授权正确，工具漏洞、依赖攻击或错误参数仍可能越界；二者解决不同威胁。

  2. 一份可安全恢复的审批记录必须绑定哪些内容？

    查看参考答案

    应绑定主体与代理关系、具体动作、目标资源、参数摘要、金额或风险、策略版本、展示给审批人的证据、有效期和审批决定。恢复时任何关键参数或对象版本变化都必须重新授权。

  3. 最小权限除了“只能读”还应从哪些维度限制？

    查看参考答案

    还应限制资源范围、动作集合、字段或数据行、参数上限、时间窗口、调用次数、网络目标和凭据作用域。权限应面向一次委托和具体任务，而不是给 Agent 长期全局令牌。

  4. 如何防止审批后到执行前的 TOCTOU 问题？

    查看参考答案

    审批绑定资源版本与参数哈希，执行前重新读取对象状态并比较；策略、金额、收款方或权限变化时使审批失效。高风险动作还可使用短期 capability token 和事务性条件更新。

  5. 外部文档通过 Prompt Injection 诱导 Agent 调用发信工具时，防线如何分层？

    查看参考答案

    检索层标记不可信内容，推理只把它当数据；编排限制可选动作，治理层校验用户目标与授权，工具执行参数白名单，沙箱限制网络和凭据。单靠输入过滤不够，最终资源侧仍需最小权限。

面试表达

高风险工具如何设计

先用 subject/action/resource/context 做独立策略判定；需要审批时持久化 checkpoint，并让 approval 绑定规范化动作摘要和有效期；恢复后重验身份、策略与资源版本；最后由工具层使用基础设施提供的短期凭据和最小沙箱执行，记录 effect receipt。模型始终只是提议者。

日常实践关键点

核心判断

本章的日常工作不是记住名词，而是把“权限、审批与沙箱”变成可执行控制环：先识别触发信号，再用版本化证据定位首个偏离点，由 tool-owner 执行动作，最后用成功、拒绝、降级三类结果验收。推荐顺序是 收窄能力范围 → 执行策略预检 → 绑定审批上下文 → 选择沙箱 Profile → 运行安全负测；任何参数都必须从真实 SLO、风险基线或失败样本推导。
