02 调用生命周期与 Observation

学习目标

  • 解释 prepare、authorize、dispatch、observe、commit 各阶段。
  • 区分 transport success、tool success 与 business acceptance。
  • 设计能被推理与编排安全消费的类型化 observation。
  • 用 trace 识别超时、未知结果、部分结果和污染观察。

链路总览与现有讲解

调用生命周期与 Observation属于工具与动作层：输入是类型化 ToolRequest、actor、授权 / 审批引用和幂等信息，输出是ActionReceipt、Observation 或类型化 ToolError。学习时先沿现有图、表或主题章节理解链路，再用后续 stage 检查每个中间产物的不变量。

一次调用的状态机

Observation 不是任意文本

{
  "call_id": "call-31",
  "capability_id": "orders.read_status.v2",
  "status": "succeeded",
  "data": { "state": "shipped", "updated_at": "2026-07-17T08:30:00Z" },
  "provenance": { "system": "orders", "resource": "ORD-42", "observed_at": "..." },
  "effect_receipt": null,
  "attempt": 1,
  "redactions": []
}

Observation 必须保留“这是事实、工具错误、策略拒绝还是未知结果”的类型。外部系统返回的文本仍是不可信数据，可能携带 prompt injection；进入模型上下文前应按来源标注、长度限制和内容策略处理。

三种成功不能混淆

  • Transport success：HTTP/RPC 成功往返。
  • Tool success：响应符合契约且能力报告操作成功。
  • Business acceptance：编排层用业务条件确认目标达成，例如退款余额变化并取得凭证。

HTTP 200 可能携带业务错误；工具成功也可能只完成一个步骤。工具层返回证据，编排层决定 run 是否推进或完成。

核心环节：从输入到可验证输出

1. 调用状态机

调用状态机是生产者与消费者共同承认的版本化解释边界；结构合法不代表语义、授权或业务结果成立。

职责与 I/O
tool-owner 在“调用状态机”环节接收 类型化 ToolRequest、actor、授权 / 审批引用和幂等信息，产出 ActionReceipt、Observation 或类型化 ToolError；输出必须能由下游按版本消费。
工作机制
显式 schema 定义字段、枚举、状态转移和错误联合，消费者先做语法校验，再按领域不变量做语义校验。本环节把 类型化 ToolRequest、actor、授权 / 审批引用和幂等信息 中与“调用状态机”相关的事实转成 ActionReceipt、Observation 或类型化 ToolError 的一部分。
必须成立的约束
必填字段、版本兼容、非法状态、未知枚举和错误语义必须确定；不能用自由文本默认值掩盖缺失事实。本章共同约束是：调用契约、使用点授权和副作用协议必须同时成立；违反时必须产生可区分的拒绝、未知或升级结果。
难点与取舍
严格契约提升可恢复性但增加演进成本；兼容窗口由调用方升级节奏和旧数据回放覆盖决定。本层总体取舍是 通用接口与语义清晰度、自动重试与副作用风险，参数必须引用版本化数据、SLO 或风险基线。
边界与注意事项
契约约束信息形状和解释，不授予权限、不选择业务策略，也不证明下游验收成功。工具与动作层 的总边界是：不负责规划任务、授予权限或把技术成功定义成业务验收；错误会继续污染 RunState、后续 Decision、审计记录和最终业务状态。

2. 贯穿关联 ID

贯穿关联 ID必须能把结论追溯到同一请求的输入、版本、决定、动作和验收事件，而不是依赖不可检索的日志文本。

职责与 I/O
tool-owner 在“贯穿关联 ID”环节接收 类型化 ToolRequest、actor、授权 / 审批引用和幂等信息，产出 ActionReceipt、Observation 或类型化 ToolError；输出必须能由下游按版本消费。
工作机制
correlation_id / trace_id 串联 span 与结构化事件，引用保留原始证据位置、policy / model / tool 版本和 owner。本环节把 类型化 ToolRequest、actor、授权 / 审批引用和幂等信息 中与“贯穿关联 ID”相关的事实转成 ActionReceipt、Observation 或类型化 ToolError 的一部分。
必须成立的约束
时间、因果顺序、采样与脱敏策略必须明确；高基数字段不能无界进入 Metric，敏感原文不能直接写 Log。本章共同约束是：调用契约、使用点授权和副作用协议必须同时成立；违反时必须产生可区分的拒绝、未知或升级结果。
难点与取舍
全量遥测利于归因却增加成本与隐私面；采样按风险与失败类型分层，关键审计事件不得随机丢弃。本层总体取舍是 通用接口与语义清晰度、自动重试与副作用风险，参数必须引用版本化数据、SLO 或风险基线。
边界与注意事项
Trace 解释系统事件与可见证据，不保存模型私有推理；Metric 告警不能替代逐请求审计。工具与动作层 的总边界是：不负责规划任务、授予权限或把技术成功定义成业务验收；错误会继续污染 RunState、后续 Decision、审计记录和最终业务状态。

4. 三类成功

三类成功必须先满足能力、风险和兼容硬约束，再比较质量、延迟与成本；名称或模型自报不能作为路由证据。

职责与 I/O
tool-owner 在“三类成功”环节接收 类型化 ToolRequest、actor、授权 / 审批引用和幂等信息，产出 ActionReceipt、Observation 或类型化 ToolError；输出必须能由下游按版本消费。
工作机制
能力画像裁剪不可用候选，版本化策略对剩余候选评分，adapter 规范输入输出并记录 route_reason 与替代项。本环节把 类型化 ToolRequest、actor、授权 / 审批引用和幂等信息 中与“三类成功”相关的事实转成 ActionReceipt、Observation 或类型化 ToolError 的一部分。
必须成立的约束
结构化输出、工具支持、上下文、区域和数据策略等硬约束不可被低价覆盖；fallback 必须兼容安全与语义。本章共同约束是：调用契约、使用点授权和副作用协议必须同时成立；违反时必须产生可区分的拒绝、未知或升级结果。
难点与取舍
动态路由提升效率却增加不可复现性；先 shadow 再 canary，用关键切片与 SLO 决定放量和回退。本层总体取舍是 通用接口与语义清晰度、自动重试与副作用风险，参数必须引用版本化数据、SLO 或风险基线。
边界与注意事项
网关选择模型与规范协议，不选择业务工具、不拥有推理意图，也不授予工具执行权限。工具与动作层 的总边界是：不负责规划任务、授予权限或把技术成功定义成业务验收；错误会继续污染 RunState、后续 Decision、审计记录和最终业务状态。

难点与取舍

冲突目标参数依据决策边界
通用接口与语义清晰度、自动重试与副作用风险使用版本化回放、失败切片、SLO 分位数和风险等级，不把教学值当生产默认。tool-owner只调整本层机制；跨层变化需要对应 owner 与独立验收。
自动化收益 vs. 失败放大比较开启 / 关闭“调用状态机”与“调用 Trace”的消融结果、人工负载和恢复时间。证据不足返回 pending / unknown / escalate，不能靠更长 Prompt 隐藏边界。

失败传播与 first divergence

  1. 上游输入：先确认 类型化 ToolRequest、actor、授权 / 审批引用和幂等信息 的身份、作用域、时间与版本是否一致；输入错误不应在本层被“修正”为新事实。
  2. 首个偏离：按 调用状态机 → 贯穿关联 ID → Observation 契约 → 三类成功 → 调用 Trace 比较预期与实际中间产物，第一个不满足概念不变量的 stage 才是本章候选根因。
  3. 下游影响：本层错误会改变 RunState、后续 Decision、审计记录和最终业务状态；最终症状可能在更晚阶段出现，但不能从症状直接倒推 Prompt。
  4. 归属边界：tool-owner修复本层契约或机制；若 first divergence 位于相邻层，保留证据并交给相应 owner，不跨层调参。

概念—实践—证据回扣

核心区回答“是什么、为何成立、怎样传播”，实践区回答“何时触发、如何处置、怎样验收”。每个 stage 与 practice 使用稳定锚点双向连接：

  • 调用状态机的不变量或约束失效时，进入对应日常实践；用本页工程产物、Trace / 评测切片和拒绝 / 降级结果证明是否恢复。
  • 贯穿关联 ID的不变量或约束失效时，进入对应日常实践；用本页工程产物、Trace / 评测切片和拒绝 / 降级结果证明是否恢复。
  • Observation 契约的不变量或约束失效时，进入对应日常实践；用本页工程产物、Trace / 评测切片和拒绝 / 降级结果证明是否恢复。
  • 三类成功的不变量或约束失效时，进入对应日常实践；用本页工程产物、Trace / 评测切片和拒绝 / 降级结果证明是否恢复。
  • 调用 Trace的不变量或约束失效时，进入对应日常实践；用本页工程产物、Trace / 评测切片和拒绝 / 降级结果证明是否恢复。

工程要点

  • 使用 monotonic deadline 向下游传播剩余预算，避免每层重新获得完整 timeout。
  • 保存请求/响应摘要或受控对象引用；敏感 payload 脱敏，密钥永不入 trace。
  • 区分 timed_out_before_dispatch、timed_out_after_dispatch，后者可能是未知结果。
  • 流式/长任务返回 job handle，由编排安排轮询或 webhook 恢复，不在单次调用中无限阻塞。
  • 对 observation schema 做契约验证；坏响应不能伪装成普通事实进入推理。

常见坑

  1. 只记最终文本，丢失 call_id、attempt 与下游状态。
  2. 超时后一律判失败并重试，导致重复副作用。
  3. 把策略 deny 包装成“工具异常”，诱导模型绕过。
  4. 外部页面文本原样注入模型，形成间接 prompt injection。
  5. 工具报告 success 就结束 run，跳过业务验收。

理论追问

请先脱稿回答，再展开参考答案核对边界、证据和取舍。

  1. 一次有副作用工具调用为什么不能只有 success 和 failed 两种状态？

    查看参考答案

    提交后超时可能处于 effect_unknown，外部系统也可能接受请求但业务尚未完成。应表达 prepared、authorized、submitted、committed、rejected、unknown 等状态，避免把网络失败误判为业务未发生并盲目重试。

  2. Observation 为什么不能只是工具返回的自然语言？

    查看参考答案

    编排需要结构化字段判断下一步，包括 call_id、状态、错误类型、effect_id、资源版本、可重试性和证据引用。自然语言可作为说明，但不能替代机器可验证的业务结果和副作用 receipt。

  3. 技术调用成功、业务处理成功和 Agent 任务完成有什么区别？

    查看参考答案

    HTTP 200 只表示技术请求成功；业务成功要看工单、退款等外部状态满足规则；Agent 任务完成还要核对用户目标与全部约束。三者应由不同证据与验收器确认，不能互相推定。

  4. 工具响应超时但可能已创建工单时，编排应如何处理？

    查看参考答案

    将调用标为 effect_unknown，冻结相同副作用重放，用稳定 idempotency key 或 call_id 查询工单系统。确认已创建就回写 committed，确认未创建才重试；无法确认则进入对账或人工处理。

  5. 要还原一次工具调用，Trace 至少需要哪些字段？

    查看参考答案

    至少关联 run_id、step、tool 与 schema version、call_id、actor、授权决策、参数摘要、attempt、开始结束时间、结果状态、error taxonomy、effect_id 和 Observation。敏感参数需脱敏，但关联关系不能丢失。

面试表达

如何讲调用链

按 prepare、authorize、dispatch、observe、commit 五阶段展开，每阶段给出证据和失败类型；强调 transport/tool/business 三种成功；最后说明 observation 要带 provenance、类型化错误和关联 ID，未知结果必须先查询状态而不是盲目重试。

日常实践关键点

核心判断

本章的日常工作不是记住名词，而是把“调用生命周期与 Observation”变成可执行控制环：先识别触发信号，再用版本化证据定位首个偏离点，由 tool-owner 执行动作，最后用成功、拒绝、降级三类结果验收。推荐顺序是 追踪调用状态机 → 贯穿关联 ID → 固化 Observation 契约 → 分离三类成功 → 保留调用 Trace；任何参数都必须从真实 SLO、风险基线或失败样本推导。

深度实践卡

  • 1. 追踪调用状态机
    触发信号
    当企业差旅 Agent 的“调用生命周期与 Observation”链路出现参数虽合法但语义错误、重复副作用、授权上下文丢失、Observation 被误判或适配器行为漂移，或变更直接触及“追踪调用状态机”的输入、策略或 owner 时，必须启动这项检查。
    诊断顺序
    诊断“追踪调用状态机”时先冻结 tool_version、schema、actor、授权、idempotency_key、调用状态与 observation。对照生产者与消费者的版本化结构，定位第一个字段缺失、枚举漂移或非法状态转移。
    执行动作
    落实“追踪调用状态机”时，将输入、输出、错误、版本兼容和拒绝语义写入机器可校验契约，并为未知字段保留显式策略。由 tool-owner 记录执行前后的版本、决定与接管条件。
    验证方法
    验收“追踪调用状态机”时，用契约测试、重复调用、拒绝路径、沙箱负测和 Trace 回放分别验证调用成功与业务成功；同时比较正确、拒绝和降级结果，阈值只从真实基线与风险容忍度推导。
    止损与取舍
    保留向后兼容读路径，写入回退到上一稳定版本，禁止静默猜测缺失字段；停用新工具版本或切为 dry-run，冻结副作用动作，依据 ledger 执行补偿或人工核对。对“追踪调用状态机”先保证可恢复和可审计，再优化自动化率。
    面试记忆点
    围绕“追踪调用状态机”准备一条“症状—首个偏离—动作—证据—止损”的真实或个人实验 Trace，并明确自己只承担的 tool-owner 职责。

    诊断顺序
    诊断“分离三类成功”时先冻结 tool_version、schema、actor、授权、idempotency_key、调用状态与 observation。分别核对技术执行、策略合规和业务验收，避免把 HTTP 成功、结构合法或模型自报完成当成最终成功。
    执行动作
    落实“分离三类成功”时，定义可机器判断的完成条件、hard fail、证据来源和验收 owner；证据不全时返回 unknown / pending 而不是 passed。由 tool-owner 记录执行前后的版本、决定与接管条件。

    任一硬门禁失败就保留上一稳定版本；仅软指标退化时才允许受控试运行并持续观察；停用新工具版本或切为 dry-run，冻结副作用动作，依据 ledger 执行补偿或人工核对。对“分离三类成功”先保证可恢复和可审计，再优化自动化率。

排障顺序

  1. 冻结请求、策略、模型 / 工具、数据和部署版本，避免边复现边变化。
  2. 从最早可验证输入开始，逐段对照契约、决策、动作与验收事件；在首个偏离点停止跨层调参。
  3. 由 tool-owner 判断修复、降级、回滚或转人工，并记录未验证假设。
贯穿案例｜企业差旅 Agent：员工请求预订超预算航班时，只把制度条款当知识证据，把员工靠窗偏好当记忆，把预订动作交给工具层；本章只处理 调用生命周期与 Observation 的控制与证据。若发现参数虽合法但语义错误、重复副作用、授权上下文丢失、Observation 被误判或适配器行为漂移，先保留 trace_id 与版本，再在本层首个偏离点修复，避免把问题推给 Prompt。

应留证据

最小证据集应能回答谁在何时依据哪个版本、对什么输入做了什么决定，以及结果如何被验证。敏感值使用教学占位或受控引用，不把日志全文复制进面试材料。

代表性工程产物｜artifact-tool-02-call-trace
case: enterprise-travel-agent
component: tools
unit: 02
owner: tool-owner
input_version: teaching-v1
policy_or_contract_version: teaching-v3
decision: allow | deny | degrade | escalate
first_divergence: stage-name
evidence_refs: [trace-demo, eval-slice-demo]
validation: passed | failed | unknown
rollback_or_handoff: action-and-owner

以上字段是学习用数据形状，不是生产默认值。实际产物还应记录时间、租户 / 主体作用域、变更来源和保留策略，并能从 evidence_refs 回放本章结论。

实践型面试要点

表达时必须标记经历属于生产实战、个人实验或方案设计，只主张自己实际完成的职责，并用 Trace、评测或变更记录举证。企业差旅案例、示例版本和指标均为教学占位；没有验证线上规模、事故协作或长期效果时要主动说明。

实践面试 1

关联实践：追踪调用状态机、贯穿关联 ID

常见问法

在生产或可复现实验中，你如何把“追踪调用状态机”和“贯穿关联 ID”落到企业差旅 Agent 的 调用生命周期与 Observation 链路？

60–90 秒回答骨架

先交代 调用生命周期与 Observation 的业务风险和 tool-owner 边界；再说触发信号与首个偏离点；随后说明契约、动作和参数依据；最后给出验证、止损和复盘结果。回答要同时覆盖正常路径与拒绝 / 降级路径。

可信证据

围绕“追踪调用状态机 / 贯穿关联 ID”展示脱敏后的 trace_id、tool_version、schema、actor、授权、idempotency_key、调用状态与 observation、前后对照和失败切片；个人实验还应给出固定输入、复现步骤和观察结果，不虚构线上规模。

高频追问

为什么问题属于 工具与动作层 而不是相邻层？参数如何从 SLO 或风险基线推导？结果未知时怎样避免扩大副作用？tool-owner 不在线时，谁依据什么证据接管？

边界与取舍

“追踪调用状态机 / 贯穿关联 ID”会增加契约、存储或人工成本，只有错误成本与复现收益覆盖这些成本时才扩大自动化。未处理过该类生产事故时，应明确这是方案设计或个人实验结论。
