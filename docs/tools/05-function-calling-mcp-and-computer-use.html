05 Function Calling、MCP 与 Computer Use

学习目标

  • 区分模型结构化调用接口、上下文/工具协议与视觉交互执行方式。
  • 按稳定性、可观测性、权限与维护成本选择接口。
  • 为 MCP server 和 Computer Use 建立清晰信任边界。
  • 设计 API 优先、可降级但不静默扩大权限的策略。

链路总览与现有讲解

Function Calling、MCP 与 Computer Use属于工具与动作层：输入是类型化 ToolRequest、actor、授权 / 审批引用和幂等信息，输出是ActionReceipt、Observation 或类型化 ToolError。学习时先沿现有图、表或主题章节理解链路，再用后续 stage 检查每个中间产物的不变量。

三个概念不在同一层

方式主要解决优势关键风险
Function Calling让模型按 schema 返回函数/工具意图结构化、易校验、适配模型 API不自动保证选择正确、授权、执行或幂等
MCP标准化 client 与 server 间的工具、资源、提示等能力发现和调用连接解耦、可发现、生态复用server 是新信任边界；能力列表和返回内容仍不可信
Computer Use通过截图/DOM/输入事件操作没有合适 API 的界面覆盖遗留系统和人机界面脆弱、慢、难幂等、易受界面注入和误操作影响

Function Calling 的真实边界

Function Calling 改善候选输出的结构，但模型返回的 tool name/arguments 仍需：能力裁剪、schema/semantic 校验、治理授权、adapter 执行与 observation 验证。推理内核通过模型网关使用厂商结构化能力；工具层不应自己直接调用模型以“修复参数”。

MCP 的工程落点

MCP client/adapter 可以位于工具接入面，把远端 server 暴露的能力映射到内部 capability registry。接入时至少检查：

  • server 身份、传输与部署来源；生产环境不要自动信任任意本地/远端 server。
  • 发现的 tool schema 固化版本/指纹并经过审核；列表变化不应静默扩大 Agent 可见能力。
  • 把 MCP server 权限映射为内部 subject/resource policy，不能因“协议标准化”跳过授权。
  • 对返回 resources/content 标注 provenance 和 untrusted data，防间接 prompt injection。
  • 限制超时、响应大小、并发和网络；记录 server、tool、request/response 摘要及错误类型。

Computer Use 的受控循环

  1. 在隔离浏览器/桌面创建最小权限 session，限制域名、下载、剪贴板与凭据。
  2. 取得 observation（截图、可访问性树或 DOM 摘要），标记页面内容不可信。
  3. 模型提出单步 action；确定性校验坐标/元素、域名、动作类别和预算。
  4. 高风险动作（提交、购买、发送、删除、上传）在最后一步前预览并审批。
  5. 执行后重新观察，使用可见证据确认状态；不要仅凭“点击已发出”判成功。

尽可能使用语义元素和稳定 selector；坐标点击对布局、缩放和弹窗敏感。若站点提供可靠 API，优先 API，因为它更易契约化、幂等和审计。

选择矩阵

场景首选理由与防护
内部订单系统有稳定 APItyped API + Function Calling契约、权限、幂等和结果验证最清晰

统一 adapter 模型

Internal Capability Contract
  -> Local/API Adapter
  -> MCP Adapter
  -> Computer-Use Adapter

All adapters MUST enforce:
  validated request + authorization decision
  deadline + idempotency/effect policy
  isolation + typed observation + trace

内部契约保持稳定，接入方式是实现细节。但不同方式的可靠性并不相同，adapter 必须显式声明 risk、supports_idempotency、observation_strength 与 requires_human_confirmation。

核心环节：从输入到可验证输出

1. 评估接口适配性

评估接口适配性必须先满足能力、风险和兼容硬约束，再比较质量、延迟与成本；名称或模型自报不能作为路由证据。

职责与 I/O
tool-owner 在“评估接口适配性”环节接收 类型化 ToolRequest、actor、授权 / 审批引用和幂等信息，产出 ActionReceipt、Observation 或类型化 ToolError；输出必须能由下游按版本消费。
工作机制
能力画像裁剪不可用候选，版本化策略对剩余候选评分，adapter 规范输入输出并记录 route_reason 与替代项。本环节把 类型化 ToolRequest、actor、授权 / 审批引用和幂等信息 中与“评估接口适配性”相关的事实转成 ActionReceipt、Observation 或类型化 ToolError 的一部分。
必须成立的约束
结构化输出、工具支持、上下文、区域和数据策略等硬约束不可被低价覆盖；fallback 必须兼容安全与语义。本章共同约束是：调用契约、使用点授权和副作用协议必须同时成立；违反时必须产生可区分的拒绝、未知或升级结果。
难点与取舍
动态路由提升效率却增加不可复现性；先 shadow 再 canary，用关键切片与 SLO 决定放量和回退。本层总体取舍是 通用接口与语义清晰度、自动重试与副作用风险，参数必须引用版本化数据、SLO 或风险基线。
边界与注意事项
网关选择模型与规范协议，不选择业务工具、不拥有推理意图，也不授予工具执行权限。工具与动作层 的总边界是：不负责规划任务、授予权限或把技术成功定义成业务验收；错误会继续污染 RunState、后续 Decision、审计记录和最终业务状态。

2. Function Schema

Function Schema是生产者与消费者共同承认的版本化解释边界；结构合法不代表语义、授权或业务结果成立。

职责与 I/O
tool-owner 在“ Function Schema”环节接收 类型化 ToolRequest、actor、授权 / 审批引用和幂等信息，产出 ActionReceipt、Observation 或类型化 ToolError；输出必须能由下游按版本消费。
工作机制
显式 schema 定义字段、枚举、状态转移和错误联合，消费者先做语法校验，再按领域不变量做语义校验。本环节把 类型化 ToolRequest、actor、授权 / 审批引用和幂等信息 中与“ Function Schema”相关的事实转成 ActionReceipt、Observation 或类型化 ToolError 的一部分。
必须成立的约束
必填字段、版本兼容、非法状态、未知枚举和错误语义必须确定；不能用自由文本默认值掩盖缺失事实。本章共同约束是：调用契约、使用点授权和副作用协议必须同时成立；违反时必须产生可区分的拒绝、未知或升级结果。
难点与取舍
严格契约提升可恢复性但增加演进成本；兼容窗口由调用方升级节奏和旧数据回放覆盖决定。本层总体取舍是 通用接口与语义清晰度、自动重试与副作用风险，参数必须引用版本化数据、SLO 或风险基线。
边界与注意事项
契约约束信息形状和解释，不授予权限、不选择业务策略，也不证明下游验收成功。工具与动作层 的总边界是：不负责规划任务、授予权限或把技术成功定义成业务验收；错误会继续污染 RunState、后续 Decision、审计记录和最终业务状态。

4. Computer Use

Computer Use必须让不可信输入即使成功诱导模型，也无法越过身份、能力、文件、网络或租户边界。

职责与 I/O
tool-owner 在“ Computer Use”环节接收 类型化 ToolRequest、actor、授权 / 审批引用和幂等信息，产出 ActionReceipt、Observation 或类型化 ToolError；输出必须能由下游按版本消费。
工作机制
信任标签、最小能力、短期凭证、进程 / 文件 / 网络隔离与输出检查构成分层防御，负测验证可利用链。本环节把 类型化 ToolRequest、actor、授权 / 审批引用和幂等信息 中与“ Computer Use”相关的事实转成 ActionReceipt、Observation 或类型化 ToolError 的一部分。
必须成立的约束
deny-by-default、凭证不进入模型上下文、租户资源不可共享、egress 采用 allowlist，PII 在离开受控域前处理。本章共同约束是：调用契约、使用点授权和副作用协议必须同时成立；违反时必须产生可区分的拒绝、未知或升级结果。
难点与取舍
更强隔离会降低复用和性能；profile 依据副作用、数据敏感度和攻击面分级，而不是所有任务使用同一沙箱。本层总体取舍是 通用接口与语义清晰度、自动重试与副作用风险，参数必须引用版本化数据、SLO 或风险基线。
边界与注意事项
Prompt guardrail 降低内容风险但不是安全边界；真正的执行隔离由基础设施提供，业务授权由治理提供。工具与动作层 的总边界是：不负责规划任务、授予权限或把技术成功定义成业务验收；错误会继续污染 RunState、后续 Decision、审计记录和最终业务状态。

失败传播与 first divergence

  1. 上游输入：先确认 类型化 ToolRequest、actor、授权 / 审批引用和幂等信息 的身份、作用域、时间与版本是否一致；输入错误不应在本层被“修正”为新事实。
  2. 首个偏离：按 评估接口适配性 → Function Schema → MCP 信任边界 → Computer Use → 统一 Adapter 观测 比较预期与实际中间产物，第一个不满足概念不变量的 stage 才是本章候选根因。
  3. 下游影响：本层错误会改变 RunState、后续 Decision、审计记录和最终业务状态；最终症状可能在更晚阶段出现，但不能从症状直接倒推 Prompt。
  4. 归属边界：tool-owner修复本层契约或机制；若 first divergence 位于相邻层，保留证据并交给相应 owner，不跨层调参。

工程要点

  • 工具选择阶段只暴露当前任务需要的最小集合，避免 capability overload。
  • 接入方式变化需要回归契约、权限、错误映射和 trace，不只测 happy path。
  • 对 MCP/Computer Use 设独立 kill switch、并发与成本预算。
  • 降级必须语义等价且权限不扩大；否则返回不可用并交编排决定。

常见坑

  1. 把 Function Calling 当成执行器或授权系统。
  2. 连接 MCP server 后把所有工具自动暴露给模型。
  3. 网页提示“忽略之前要求”就改变策略或泄露数据。
  4. Computer Use 点击成功就判业务成功。
  5. API 不可用时静默切换到更高风险界面操作。

理论追问

请先脱稿回答，再展开参考答案核对边界、证据和取舍。

  1. Function Calling、MCP 和 Computer Use 为什么不是三个互斥替代方案？

    查看参考答案

    Function Calling 是模型输出结构化调用意图的机制；MCP 是应用发现和调用外部能力的协议；Computer Use 是通过界面感知与操作环境的一类高风险工具。它们处于不同层级，可以组合使用。

  2. Function Calling 已生成合法 JSON，系统还必须做什么？

    查看参考答案

    仍需工具存在性、版本、语义参数、授权、审批、预算和业务前置校验，再由编排调用适配器。模型输出只是候选 intent，不是可信 RPC，也不能证明执行成功。

  3. 接入一个 MCP Server 前应建立哪些信任边界？

    查看参考答案

    核验 server 身份与来源、工具清单和版本，限制传入上下文、凭据和网络，逐工具授权并记录调用；对返回内容按不可信数据处理。远程 server 还需超时、速率、隔离和供应链审计。

  4. 什么时候才值得使用 Computer Use？

    查看参考答案

    当目标系统没有稳定 API、任务确实依赖视觉界面且人工成本高于自动化风险时才考虑。优先 API 或专用工具；Computer Use 需隔离浏览器、允许站点和动作、截图证据、步骤预算及高风险点击审批。

  5. 如何用统一 adapter 观察三类工具接口？

    查看参考答案

    在上层统一映射 tool_id、schema version、call_id、actor、授权、输入摘要、状态、错误类型、effect_id 和 Observation，同时保留接口特有证据，如 MCP server、Function schema 或 Computer Use 截图。这样编排和评测不依赖具体协议。

面试表达

如何比较三者

Function Calling 是模型侧结构化意图接口；MCP 是 client/server 能力连接协议；Computer Use 是缺少合适 API 时的环境交互方式。三者都不能替代内部 capability contract、治理授权、工具运行时、幂等与 observation 验证。选型上 API 优先，MCP 用于标准连接，Computer Use 只做受限兜底。

日常实践关键点

核心判断

本章的日常工作不是记住名词，而是把“Function Calling、MCP 与 Computer Use”变成可执行控制环：先识别触发信号，再用版本化证据定位首个偏离点，由 tool-owner 执行动作，最后用成功、拒绝、降级三类结果验收。推荐顺序是 评估接口适配性 → 约束 Function Schema → 校验 MCP 信任边界 → 门禁 Computer Use → 统一 Adapter 观测；任何参数都必须从真实 SLO、风险基线或失败样本推导。

    触发信号

排障顺序

  1. 冻结请求、策略、模型 / 工具、数据和部署版本，避免边复现边变化。
  2. 从最早可验证输入开始，逐段对照契约、决策、动作与验收事件；在首个偏离点停止跨层调参。
  3. 由 tool-owner 判断修复、降级、回滚或转人工，并记录未验证假设。
贯穿案例｜企业差旅 Agent：员工请求预订超预算航班时，只把制度条款当知识证据，把员工靠窗偏好当记忆，把预订动作交给工具层；本章只处理 Function Calling、MCP 与 Computer Use 的控制与证据。若发现参数虽合法但语义错误、重复副作用、授权上下文丢失、Observation 被误判或适配器行为漂移，先保留 trace_id 与版本，再在本层首个偏离点修复，避免把问题推给 Prompt。
