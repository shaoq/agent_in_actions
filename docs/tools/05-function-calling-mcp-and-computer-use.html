05 Function Calling、MCP 与 Computer Use

学习目标

  • 区分模型结构化调用接口、上下文/工具协议与视觉交互执行方式。
  • 按稳定性、可观测性、权限与维护成本选择接口。
  • 为 MCP server 和 Computer Use 建立清晰信任边界。
  • 设计 API 优先、可降级但不静默扩大权限的策略。

三个概念不在同一层

方式主要解决优势关键风险
Function Calling让模型按 schema 返回函数/工具意图结构化、易校验、适配模型 API不自动保证选择正确、授权、执行或幂等
MCP标准化 client 与 server 间的工具、资源、提示等能力发现和调用连接解耦、可发现、生态复用server 是新信任边界；能力列表和返回内容仍不可信
Computer Use通过截图/DOM/输入事件操作没有合适 API 的界面覆盖遗留系统和人机界面脆弱、慢、难幂等、易受界面注入和误操作影响

Function Calling 的真实边界

Function Calling 改善候选输出的结构，但模型返回的 tool name/arguments 仍需：能力裁剪、schema/semantic 校验、治理授权、adapter 执行与 observation 验证。推理内核通过模型网关使用厂商结构化能力；工具层不应自己直接调用模型以“修复参数”。

MCP 的工程落点

MCP client/adapter 可以位于工具接入面，把远端 server 暴露的能力映射到内部 capability registry。接入时至少检查：

  • server 身份、传输与部署来源；生产环境不要自动信任任意本地/远端 server。
  • 发现的 tool schema 固化版本/指纹并经过审核；列表变化不应静默扩大 Agent 可见能力。
  • 把 MCP server 权限映射为内部 subject/resource policy，不能因“协议标准化”跳过授权。
  • 对返回 resources/content 标注 provenance 和 untrusted data，防间接 prompt injection。
  • 限制超时、响应大小、并发和网络；记录 server、tool、request/response 摘要及错误类型。

Computer Use 的受控循环

  1. 在隔离浏览器/桌面创建最小权限 session，限制域名、下载、剪贴板与凭据。
  2. 取得 observation（截图、可访问性树或 DOM 摘要），标记页面内容不可信。
  3. 模型提出单步 action；确定性校验坐标/元素、域名、动作类别和预算。
  4. 高风险动作（提交、购买、发送、删除、上传）在最后一步前预览并审批。
  5. 执行后重新观察，使用可见证据确认状态；不要仅凭“点击已发出”判成功。

尽可能使用语义元素和稳定 selector；坐标点击对布局、缩放和弹窗敏感。若站点提供可靠 API，优先 API，因为它更易契约化、幂等和审计。

选择矩阵

场景首选理由与防护
内部订单系统有稳定 APItyped API + Function Calling契约、权限、幂等和结果验证最清晰
多个 IDE/Agent 复用统一开发工具审核后的 MCP server标准发现与连接；仍需内部授权和沙箱
遗留后台只有网页受限 Computer Use域名隔离、逐步观察、提交前 HITL、低频使用
转账/生产发布专用 API / Workflow避免视觉自动化；强审批、预条件和 effect receipt
MCP server 异常fail closed 或经批准的等价 adapter不能静默切换到更高权限 Computer Use

统一 adapter 模型

Internal Capability Contract
  -> Local/API Adapter
  -> MCP Adapter
  -> Computer-Use Adapter

All adapters MUST enforce:
  validated request + authorization decision
  deadline + idempotency/effect policy
  isolation + typed observation + trace

内部契约保持稳定，接入方式是实现细节。但不同方式的可靠性并不相同，adapter 必须显式声明 risk、supports_idempotency、observation_strength 与 requires_human_confirmation。

工程要点

  • 工具选择阶段只暴露当前任务需要的最小集合，避免 capability overload。
  • 接入方式变化需要回归契约、权限、错误映射和 trace，不只测 happy path。
  • 对 MCP/Computer Use 设独立 kill switch、并发与成本预算。
  • 降级必须语义等价且权限不扩大；否则返回不可用并交编排决定。

常见坑

  1. 把 Function Calling 当成执行器或授权系统。
  2. 连接 MCP server 后把所有工具自动暴露给模型。
  3. 网页提示“忽略之前要求”就改变策略或泄露数据。
  4. Computer Use 点击成功就判业务成功。
  5. API 不可用时静默切换到更高风险界面操作。

面试表达

如何比较三者

Function Calling 是模型侧结构化意图接口；MCP 是 client/server 能力连接协议；Computer Use 是缺少合适 API 时的环境交互方式。三者都不能替代内部 capability contract、治理授权、工具运行时、幂等与 observation 验证。选型上 API 优先，MCP 用于标准连接，Computer Use 只做受限兜底。

学习产出

  1. 为同一 CRM 操作分别画出 API、MCP、Computer Use 信任边界。
  2. 列出 MCP server 接入审核的 8 个检查项。
  3. 设计 Computer Use 在“提交订单”前后的 HITL 与证据。
