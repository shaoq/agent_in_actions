03 澄清、HITL 与人工移交

学习目标

  • 按目标区分 clarification、approval/review 与 human handoff。
  • 设计从编排 checkpoint 到交互 prompt、人工决定和恢复的关联协议。
  • 生成最小必要、可验证、可回退的 handoff package，避免用户重复陈述。

链路总览与现有讲解

澄清、HITL 与人工移交属于交互与接入层：输入是渠道事件、媒体、session、actor、consent 和客户端能力，输出是规范化 RequestEnvelope、StreamEvent、HandoffPackage 与用户可见状态。学习时先沿现有图、表或主题章节理解链路，再用后续 stage 检查每个中间产物的不变量。

核心概念

模式要解决的问题交互层职责编排/治理职责
澄清 Clarification目标、对象、约束或关键事实不够明确展示单一问题、选项、默认值和为何需要识别缺口、暂停 run、校验答复后继续
HITL 审批/复核风险或策略要求人做受约束决定展示动作摘要、影响、证据、有效期和替代项策略触发、checkpoint、决定绑定、恢复时再授权
人工移交 Handoff机器无法/不应继续，需要转移服务责任说明状态、等待时间、提供退出路径、交付上下文确定移交原因和恢复点，保留机器侧 run 状态

澄清设计：用最小问题降低不确定性

  1. 指出缺口：对象、目标、约束、身份或风险中的哪一项不足。
  2. 一次只问能改变下一步的最小问题，优先给互斥选项、上下文默认值和“都不是”。
  3. 用 question_id、run_id、expected_schema、expires_at 关联答复；旧问题的迟到答复不得写进新阶段。
  4. 提供“跳过”“取消”“联系人工”和文本自由输入，不把按钮选项做成暗黑模式。
  5. 答复先规范化并交给编排校验；接入层不因某选项看似明确就执行动作。

HITL：把人的决定绑定到所见内容

一个可验证审批 receipt 至少包含 decision_id、subject/role、tenant、run/checkpoint、action_summary_hash、关键参数、影响范围、policy_version、decision、reason、created_at、expires_at。若恢复时金额、收件人、资源版本或权限变化，原批准失效并重新确认。

同意不等于审批，审批不等于执行 · 用户可能同意处理数据，但具体高风险动作仍需治理授权；人点击“批准”只产生决定证据，编排恢复并重新校验后，工具层才可能执行。

核心环节：从输入到可验证输出

1. 识别澄清触发

识别澄清触发必须以可观测信息增益和有界预算推进；没有新证据的重复思考不构成进展。

职责与 I/O
channel-owner 在“识别澄清触发”环节接收 渠道事件、媒体、session、actor、consent 和客户端能力，产出 规范化 RequestEnvelope、StreamEvent、HandoffPackage 与用户可见状态；输出必须能由下游按版本消费。
工作机制
入口条件选择最简单可行模式，每轮比较证据 / 状态变化，触线后返回 clarify、abstain、degrade 或 escalate。本环节把 渠道事件、媒体、session、actor、consent 和客户端能力 中与“识别澄清触发”相关的事实转成 规范化 RequestEnvelope、StreamEvent、HandoffPackage 与用户可见状态 的一部分。
必须成立的约束
最大步数、时间、成本、重规划次数和退出类型必须显式；自检不能无限递归，也不能绕过外部验证。本章共同约束是：事件关联、身份绑定、同意、顺序和可访问性必须端到端成立；违反时必须产生可区分的拒绝、未知或升级结果。
难点与取舍
更长推理可能提高复杂任务质量，也会累积延迟和错误；预算由任务价值、失败成本和消融结果决定。本层总体取舍是 交互流畅度与澄清 / 审批摩擦、实时性与可恢复性，参数必须引用版本化数据、SLO 或风险基线。
边界与注意事项
推理可以发现不确定性并提出澄清，编排负责实际暂停、预算执行和人工接管。交互与接入层 的总边界是：不做业务推理、流程控制或工具执行，只保证输入输出边界可信可用；错误会继续污染 编排入口、身份授权、用户理解和人工接管质量。

3. HITL 上下文

HITL 上下文必须把明确输入转换为版本化、可验证的中间产物，并让失败能够定位到所属 owner。

职责与 I/O
channel-owner 在“ HITL 上下文”环节接收 渠道事件、媒体、session、actor、consent 和客户端能力，产出 规范化 RequestEnvelope、StreamEvent、HandoffPackage 与用户可见状态；输出必须能由下游按版本消费。
工作机制
该环节通过显式数据结构和状态判断连接前后阶段，输出同时携带版本、决定和证据引用。本环节把 渠道事件、媒体、session、actor、consent 和客户端能力 中与“ HITL 上下文”相关的事实转成 规范化 RequestEnvelope、StreamEvent、HandoffPackage 与用户可见状态 的一部分。
必须成立的约束
身份、时序、作用域、预算和失败语义必须确定，字段缺失或结果未知不能被静默补齐。本章共同约束是：事件关联、身份绑定、同意、顺序和可访问性必须端到端成立；违反时必须产生可区分的拒绝、未知或升级结果。
难点与取舍
更多自动化会增加复杂度与误判面；参数需由真实 SLO、版本化数据和失败样本校准。本层总体取舍是 交互流畅度与澄清 / 审批摩擦、实时性与可恢复性，参数必须引用版本化数据、SLO 或风险基线。
边界与注意事项
该环节只拥有本层中间产物，不替代相邻层的推理、控制、授权、执行或验收职责。交互与接入层 的总边界是：不做业务推理、流程控制或工具执行，只保证输入输出边界可信可用；错误会继续污染 编排入口、身份授权、用户理解和人工接管质量。

4. Handoff Package

Handoff Package成立的前提是每个控制决定只有一个最终 owner，交接不转移未声明的权限与验收责任。

职责与 I/O
channel-owner 在“ Handoff Package”环节接收 渠道事件、媒体、session、actor、consent 和客户端能力，产出 规范化 RequestEnvelope、StreamEvent、HandoffPackage 与用户可见状态；输出必须能由下游按版本消费。
工作机制
控制点以类型化输入 / 输出连接 owner；交出的是受约束任务或事实，不是共享的隐式控制权。本环节把 渠道事件、媒体、session、actor、consent 和客户端能力 中与“ Handoff Package”相关的事实转成 规范化 RequestEnvelope、StreamEvent、HandoffPackage 与用户可见状态 的一部分。
必须成立的约束
决策、推进、动作、验收和事件升级必须分别可归属，交接双方不能同时写同一控制状态。本章共同约束是：事件关联、身份绑定、同意、顺序和可访问性必须端到端成立；违反时必须产生可区分的拒绝、未知或升级结果。
难点与取舍
拆分能提高并行度，却增加协调、上下文复制和冲突成本；是否拆分要由失败归因与吞吐基线决定。本层总体取舍是 交互流畅度与澄清 / 审批摩擦、实时性与可恢复性，参数必须引用版本化数据、SLO 或风险基线。
边界与注意事项
推理可以建议，工具可以执行，治理可以允许或拒绝，但只有编排 owner 推进状态，业务 owner 做最终验收。交互与接入层 的总边界是：不做业务推理、流程控制或工具执行，只保证输入输出边界可信可用；错误会继续污染 编排入口、身份授权、用户理解和人工接管质量。

5. 人工后恢复

人工后恢复只能把系统带回已知状态或安全终态；未知结果在查明副作用前不得被当成可重试失败。

职责与 I/O
channel-owner 在“人工后恢复”环节接收 渠道事件、媒体、session、actor、consent 和客户端能力，产出 规范化 RequestEnvelope、StreamEvent、HandoffPackage 与用户可见状态；输出必须能由下游按版本消费。
工作机制
错误先分瞬时、永久、冲突与 unknown outcome；幂等键、ledger、checkpoint 和补偿记录共同决定 retry、query、resume 或人工核对。本环节把 渠道事件、媒体、session、actor、consent 和客户端能力 中与“人工后恢复”相关的事实转成 规范化 RequestEnvelope、StreamEvent、HandoffPackage 与用户可见状态 的一部分。
必须成立的约束
尝试次数、累计 deadline、退避、副作用作用域和补偿 owner 都必须有界，幂等键需覆盖业务动作而非单次请求。本章共同约束是：事件关联、身份绑定、同意、顺序和可访问性必须端到端成立；违反时必须产生可区分的拒绝、未知或升级结果。
难点与取舍
重试提高瞬时可用性也会放大负载与副作用；预算由下游容量、用户 deadline 和动作不可逆性决定。本层总体取舍是 交互流畅度与澄清 / 审批摩擦、实时性与可恢复性，参数必须引用版本化数据、SLO 或风险基线。
边界与注意事项
补偿是新的业务动作而非时间回滚；网关 fallback 不应掩盖工具副作用或编排非法状态。交互与接入层 的总边界是：不做业务推理、流程控制或工具执行，只保证输入输出边界可信可用；错误会继续污染 编排入口、身份授权、用户理解和人工接管质量。

失败传播与 first divergence

  1. 上游输入：先确认 渠道事件、媒体、session、actor、consent 和客户端能力 的身份、作用域、时间与版本是否一致；输入错误不应在本层被“修正”为新事实。
  2. 首个偏离：按 识别澄清触发 → 最小问题 → HITL 上下文 → Handoff Package → 人工后恢复 比较预期与实际中间产物，第一个不满足概念不变量的 stage 才是本章候选根因。
  3. 下游影响：本层错误会改变 编排入口、身份授权、用户理解和人工接管质量；最终症状可能在更晚阶段出现，但不能从症状直接倒推 Prompt。
  4. 归属边界：channel-owner修复本层契约或机制；若 first divergence 位于相邻层，保留证据并交给相应 owner，不跨层调参。

核心区回答“是什么、为何成立、怎样传播”，实践区回答“何时触发、如何处置、怎样验收”。每个 stage 与 practice 使用稳定锚点双向连接：

工程要点

  • 双向状态同步：排队、坐席接受、用户离开、超时、解决和重新交回 Agent 都产生事件；客户端刷新后能恢复权威状态。
  • 避免双写：机器进入 handoff pending 后停止推进冲突动作；若并行提供自助路径，必须由编排协调版本和完成判定。
  • SLA 与退路：显示预计等待、通知方式、取消排队和异步回访；人工不可用时返回安全降级，不无限 spinner。
  • 人工输入不天然可信：坐席决定也要身份、角色、范围和审计；自由文本不得直接变成工具参数。
  • 隐私分层：按队列/角色裁剪 transcript 与附件，敏感字段默认遮盖，需要理由才能临时揭示。

常见坑

  • 循环澄清：反复问同一问题却没有新证据或退出条件，应升级或类型化失败。
  • “确认吗？”：没有动作对象、影响、费用与不可逆性，用户无法做知情决定。
  • 转人工即丢 run：坐席结束后无法回写结果，机器重复执行或用户重讲一遍。
  • 全量 transcript 打包：泄露无关个人数据，也掩盖真正待办。
  • 按钮直接调工具：绕过编排 checkpoint、重新授权和幂等控制。

理论追问

请先脱稿回答，再展开参考答案核对边界、证据和取舍。

  1. 什么时候应该向用户澄清，而不是让模型猜测？

    查看参考答案

    当缺失信息会改变动作、权限、成本或不可逆结果，且可用一个低成本问题显著降低不确定性时应澄清。对可安全默认的低风险格式细节可使用明确默认值；不能用猜测填补收款方、金额或目标资源。

  2. HITL 等待为什么必须是编排状态，而不只是 UI 弹窗？

    查看参考答案

    审批可能跨连接和长时间存在，需要持久化 paused 状态、动作参数、证据、有效期和恢复位置。UI 只是人的交互入口；若编排不知道在等待，重启或重连会丢失控制权或重复执行。

  3. 审批界面必须让人看到哪些内容？

    查看参考答案

    显示代表谁执行、具体动作、目标资源、关键参数、风险与副作用、证据来源、可选决定和有效期。批准记录绑定所见快照；只显示“Agent 请求继续”无法形成知情授权。

  4. 人工移交包应该包含什么，又不应该包含什么？

    查看参考答案

    包含用户目标、已验证身份、当前状态、关键证据、已尝试步骤、未决风险和建议下一步；不应倾倒完整隐藏 Prompt、无关历史或超范围敏感数据。包需可追溯并允许人工纠正。

  5. 为什么接入层不能在用户点击“批准”后直接调用业务工具？

    查看参考答案

    点击事件只是审批 Observation，必须回到编排校验有效期、参数和当前资源状态，再经过治理与工具执行。若 UI 直接调用，就形成第二个绕过 State、授权、幂等和 Trace 的编排器。

面试表达

澄清、HITL 和人工移交如何设计？

我按目的拆分：澄清补足决策信息，HITL 取得与动作摘要和策略版本绑定的人类决定，handoff 转移服务责任。编排在每种模式下持久化 checkpoint 并拥有恢复；接入层负责把问题、影响和状态以渠道可访问形式呈现，再把带 question/decision/resume id 的输入交回。移交包只含目标、已确认事实、未决问题、动作 receipt、未知副作用和最小身份范围，不复制全量历史。

日常实践关键点

核心判断

本章的日常工作不是记住名词，而是把“澄清、HITL 与人工移交”变成可执行控制环：先识别触发信号，再用版本化证据定位首个偏离点，由 channel-owner 执行动作，最后用成功、拒绝、降级三类结果验收。推荐顺序是 识别澄清触发 → 提出最小问题 → 固定 HITL 上下文 → 生成 Handoff Package → 处理人工后恢复；任何参数都必须从真实 SLO、风险基线或失败样本推导。

深度实践卡

  • 1. 识别澄清触发
    触发信号
    当企业差旅 Agent 的“澄清、HITL 与人工移交”链路出现重复事件、身份错绑、流式乱序、必要信息缺失、同意撤回未生效或用户无法安全恢复，或变更直接触及“识别澄清触发”的输入、策略或 owner 时，必须启动这项检查。
    诊断顺序

排障顺序

  1. 冻结请求、策略、模型 / 工具、数据和部署版本，避免边复现边变化。
  2. 从最早可验证输入开始，逐段对照契约、决策、动作与验收事件；在首个偏离点停止跨层调参。
  3. 由 channel-owner 判断修复、降级、回滚或转人工，并记录未验证假设。
贯穿案例｜企业差旅 Agent：员工请求预订超预算航班时，只把制度条款当知识证据，把员工靠窗偏好当记忆，把预订动作交给工具层；本章只处理 澄清、HITL 与人工移交 的控制与证据。若发现重复事件、身份错绑、流式乱序、必要信息缺失、同意撤回未生效或用户无法安全恢复，先保留 trace_id 与版本，再在本层首个偏离点修复，避免把问题推给 Prompt。
