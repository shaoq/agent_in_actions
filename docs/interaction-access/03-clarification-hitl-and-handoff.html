03 澄清、HITL 与人工移交

学习目标

  • 按目标区分 clarification、approval/review 与 human handoff。
  • 设计从编排 checkpoint 到交互 prompt、人工决定和恢复的关联协议。
  • 生成最小必要、可验证、可回退的 handoff package，避免用户重复陈述。

核心概念

模式要解决的问题交互层职责编排/治理职责
澄清 Clarification目标、对象、约束或关键事实不够明确展示单一问题、选项、默认值和为何需要识别缺口、暂停 run、校验答复后继续
HITL 审批/复核风险或策略要求人做受约束决定展示动作摘要、影响、证据、有效期和替代项策略触发、checkpoint、决定绑定、恢复时再授权
人工移交 Handoff机器无法/不应继续，需要转移服务责任说明状态、等待时间、提供退出路径、交付上下文确定移交原因和恢复点，保留机器侧 run 状态

澄清设计：用最小问题降低不确定性

  1. 指出缺口：对象、目标、约束、身份或风险中的哪一项不足。
  2. 一次只问能改变下一步的最小问题，优先给互斥选项、上下文默认值和“都不是”。
  3. 用 question_id、run_id、expected_schema、expires_at 关联答复；旧问题的迟到答复不得写进新阶段。
  4. 提供“跳过”“取消”“联系人工”和文本自由输入，不把按钮选项做成暗黑模式。
  5. 答复先规范化并交给编排校验；接入层不因某选项看似明确就执行动作。

HITL：把人的决定绑定到所见内容

一个可验证审批 receipt 至少包含 decision_id、subject/role、tenant、run/checkpoint、action_summary_hash、关键参数、影响范围、policy_version、decision、reason、created_at、expires_at。若恢复时金额、收件人、资源版本或权限变化，原批准失效并重新确认。

同意不等于审批，审批不等于执行 · 用户可能同意处理数据，但具体高风险动作仍需治理授权；人点击“批准”只产生决定证据，编排恢复并重新校验后，工具层才可能执行。

人工移交包

必须字段目的最小化要求
handoff_reason / priority解释为什么转人和 SLA使用类型化原因，不泄露内部 prompt
subject / channel / accessibility needs保持身份、语言和沟通方式仅提供坐席完成职责所需的 scope
goal / confirmed facts / unresolved questions避免用户重复陈述事实与模型推断分栏，附 provenance
actions / receipts / unknown effects防止重复执行明确已做、未做、结果未知和幂等键
checkpoint / resume_token / expiry机器与人工可安全接力令牌短期、单用、绑定主体与版本

工程要点

  • 双向状态同步：排队、坐席接受、用户离开、超时、解决和重新交回 Agent 都产生事件；客户端刷新后能恢复权威状态。
  • 避免双写：机器进入 handoff pending 后停止推进冲突动作；若并行提供自助路径，必须由编排协调版本和完成判定。
  • SLA 与退路：显示预计等待、通知方式、取消排队和异步回访；人工不可用时返回安全降级，不无限 spinner。
  • 人工输入不天然可信：坐席决定也要身份、角色、范围和审计；自由文本不得直接变成工具参数。
  • 隐私分层：按队列/角色裁剪 transcript 与附件，敏感字段默认遮盖，需要理由才能临时揭示。

常见坑

  • 循环澄清：反复问同一问题却没有新证据或退出条件，应升级或类型化失败。
  • “确认吗？”：没有动作对象、影响、费用与不可逆性，用户无法做知情决定。
  • 转人工即丢 run：坐席结束后无法回写结果，机器重复执行或用户重讲一遍。
  • 全量 transcript 打包：泄露无关个人数据，也掩盖真正待办。
  • 按钮直接调工具：绕过编排 checkpoint、重新授权和幂等控制。

面试表达

澄清、HITL 和人工移交如何设计？

我按目的拆分：澄清补足决策信息，HITL 取得与动作摘要和策略版本绑定的人类决定，handoff 转移服务责任。编排在每种模式下持久化 checkpoint 并拥有恢复；接入层负责把问题、影响和状态以渠道可访问形式呈现，再把带 question/decision/resume id 的输入交回。移交包只含目标、已确认事实、未决问题、动作 receipt、未知副作用和最小身份范围，不复制全量历史。
