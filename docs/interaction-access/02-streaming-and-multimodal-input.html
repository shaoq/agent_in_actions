02 流式响应与多模态输入

学习目标

  • 设计带 sequence、event type、resume cursor、取消和背压的流式交互协议。
  • 区分模型 token、编排状态、工具进度、引用和最终业务 receipt。
  • 为图片、音频、视频和文件建立上传、隔离、解析、授权、溯源与降级链路。

链路总览与现有讲解

流式响应与多模态输入属于交互与接入层：输入是渠道事件、媒体、session、actor、consent 和客户端能力，输出是规范化 RequestEnvelope、StreamEvent、HandoffPackage 与用户可见状态。学习时先沿现有图、表或主题章节理解链路，再用后续 stage 检查每个中间产物的不变量。

核心概念：流式的是事件，不只是文本

生产界面需要知道“正在排队”“需要澄清”“工具已派发”“结果仍未知”“生成完成但业务验收失败”。若协议只有 token 字符串，客户端只能猜状态，断线后也无法判断哪些内容已被用户看到。

事件类型关键字段客户端语义
run.accepted / progressrun_id、phase、sequence、trace_id任务已受理或阶段变化，不代表完成
content.deltamessage_id、part_id、delta、sequence可追加草稿；必须允许撤回或替换
citation / artifactsource_ref、access_scope、media_type按权限展示证据或附件
clarification / approvalinteraction_id、choices、expires_at需要用户输入，run 进入等待
errorcode、retryable、safe_message、support_ref类型化恢复或转人工，不泄露内部信息
message.completedmessage_id、finish_reason一条输出完成，不等于业务完成
run.completedoutcome、receipt_refs编排已完成业务验收并提交结果

可恢复流式契约

{
  "run_id": "run_31", "stream_id": "st_8", "sequence": 27,
  "event": "content.delta", "message_id": "m_4", "part_id": "p_text",
  "data": {"text": "订单已定位，正在核对取消规则…"},
  "replayable": true, "created_at": "2026-07-17T09:31:12Z"
}

客户端确认最高连续 sequence，断线重连提交 cursor。服务端重放必须保留同一 event id 和顺序；不可重放的敏感或超大事件返回 snapshot/ref。取消只表示用户发出 cancel intent，最终状态仍由编排返回 cancelled、completed 或 effect_unknown。

多模态输入流水线

  1. 入口限制：在上传前协商 MIME、大小、数量、时长与设备能力；使用短期 upload URL，不经应用进程缓冲大文件。
  2. 隔离与校验：以服务端嗅探结果校验真实类型，做恶意软件、压缩炸弹、宏、EXIF/PII 与内容策略检查。
  3. 主体与用途绑定：artifact_ref 绑定 tenant、subject、session/run、purpose、ACL、retention 与 checksum；引用不能单独充当访问权。
  4. 派生处理：OCR、ASR、缩略图、抽帧、结构解析等派生物保留 provenance、工具版本、时间段/坐标与置信度。
  5. 语义组装：接入层提交“用户文本 + artifact refs + 渠道能力”；由编排决定是否检索、调用视觉/语音能力或澄清。
  6. 删除传播：过期、撤回或删除时覆盖原件、派生物、缓存和索引；审计只保留必要元数据。
无障碍也是多模态契约 · 图片需要可编辑替代文本，音频需要 transcript/字幕，语音交互要提供文本等价路径；不能把“支持语音”当成只能说话和听声音。

核心环节：从输入到可验证输出

1. 定义事件协议

定义事件协议是生产者与消费者共同承认的版本化解释边界；结构合法不代表语义、授权或业务结果成立。

职责与 I/O
channel-owner 在“定义事件协议”环节接收 渠道事件、媒体、session、actor、consent 和客户端能力，产出 规范化 RequestEnvelope、StreamEvent、HandoffPackage 与用户可见状态；输出必须能由下游按版本消费。
工作机制
显式 schema 定义字段、枚举、状态转移和错误联合，消费者先做语法校验，再按领域不变量做语义校验。本环节把 渠道事件、媒体、session、actor、consent 和客户端能力 中与“定义事件协议”相关的事实转成 规范化 RequestEnvelope、StreamEvent、HandoffPackage 与用户可见状态 的一部分。
必须成立的约束
必填字段、版本兼容、非法状态、未知枚举和错误语义必须确定；不能用自由文本默认值掩盖缺失事实。本章共同约束是：事件关联、身份绑定、同意、顺序和可访问性必须端到端成立；违反时必须产生可区分的拒绝、未知或升级结果。
难点与取舍
严格契约提升可恢复性但增加演进成本；兼容窗口由调用方升级节奏和旧数据回放覆盖决定。本层总体取舍是 交互流畅度与澄清 / 审批摩擦、实时性与可恢复性，参数必须引用版本化数据、SLO 或风险基线。
边界与注意事项
契约约束信息形状和解释，不授予权限、不选择业务策略，也不证明下游验收成功。交互与接入层 的总边界是：不做业务推理、流程控制或工具执行，只保证输入输出边界可信可用；错误会继续污染 编排入口、身份授权、用户理解和人工接管质量。

2. 顺序与重连

顺序与重连必须让用户和服务端对事件顺序、已确认状态、局部失败与可恢复动作形成同一事实。

4. 媒体输入

媒体输入必须区分结构有效、动作完成、策略合规和业务验收；缺证据时只能是 pending 或 unknown。

职责与 I/O
channel-owner 在“媒体输入”环节接收 渠道事件、媒体、session、actor、consent 和客户端能力，产出 规范化 RequestEnvelope、StreamEvent、HandoffPackage 与用户可见状态；输出必须能由下游按版本消费。
工作机制
每类成功由不同 owner 和证据判定，hard fail 先于综合评分，发布 / 完成只消费版本化验收结果。本环节把 渠道事件、媒体、session、actor、consent 和客户端能力 中与“媒体输入”相关的事实转成 规范化 RequestEnvelope、StreamEvent、HandoffPackage 与用户可见状态 的一部分。
必须成立的约束
完成条件、拒绝条件、证据引用和验收 owner 必须可机器判断；模型自报或 HTTP 2xx 不能单独关单。本章共同约束是：事件关联、身份绑定、同意、顺序和可访问性必须端到端成立；违反时必须产生可区分的拒绝、未知或升级结果。
难点与取舍
严格门禁降低错误发布却可能延长流程；软指标观察窗口和 hard fail 范围由风险与历史波动确定。本层总体取舍是 交互流畅度与澄清 / 审批摩擦、实时性与可恢复性，参数必须引用版本化数据、SLO 或风险基线。
边界与注意事项
工具证明动作结果，治理证明允许，业务 owner 证明价值；编排只聚合这些证据推进终态。交互与接入层 的总边界是：不做业务推理、流程控制或工具执行，只保证输入输出边界可信可用；错误会继续污染 编排入口、身份授权、用户理解和人工接管质量。

失败传播与 first divergence

  1. 上游输入：先确认 渠道事件、媒体、session、actor、consent 和客户端能力 的身份、作用域、时间与版本是否一致；输入错误不应在本层被“修正”为新事实。
  2. 首个偏离：按 定义事件协议 → 顺序与重连 → 表达局部失败 → 媒体输入 → 观测流式链路 比较预期与实际中间产物，第一个不满足概念不变量的 stage 才是本章候选根因。
  3. 下游影响：本层错误会改变 编排入口、身份授权、用户理解和人工接管质量；最终症状可能在更晚阶段出现，但不能从症状直接倒推 Prompt。
  4. 归属边界：channel-owner修复本层契约或机制；若 first divergence 位于相邻层，保留证据并交给相应 owner，不跨层调参。

工程要点

  • 背压：限制每连接缓冲、合并高频 delta、优先保留状态/错误/完成事件；慢客户端不能拖垮生成或丢失终态。
  • 部分输出安全：敏感内容需先缓冲再审核；若边生成边展示，协议必须支持 content.replace/content.retract，并记录用户实际可见范围。
  • 顺序与并发：同一 message 的 part 内有序，不同 artifact 可并行；UI 按稳定 ID 更新，不依赖到达时刻猜关系。
  • 降级：视觉模型不可用时可请求用户补充文字；ASR 低置信度展示可校正 transcript；不静默假装已理解附件。
  • 可观测：记录 upload、scan、parse、TTFT、断线、重放、积压、取消和最终状态，不在 telemetry 保存原始敏感媒体。

常见坑

  • 把 HTTP 200 当完成：它可能只表示连接建立；业务动作仍在排队或结果未知。
  • 重连后重新生成：造成文本漂移或副作用重复；应重放事件或返回权威 snapshot。
  • 信任文件扩展名：伪装 MIME、宏和压缩炸弹会跨过边界。
  • 把 OCR 文本当用户原话：派生内容有版本、坐标和置信度，低置信度关键字段必须确认。
  • 隐藏错误只显示 spinner：用户不知道是否安全重试，也无法转人工。

理论追问

请先脱稿回答，再展开参考答案核对边界、证据和取舍。

  1. 流式响应为什么应建模为事件，而不只是连续 Token？

    查看参考答案

    生产流包含 started、delta、tool_status、citation、approval_required、completed 和 failed 等语义。事件带序号和 ID，客户端才能重连、去重和正确呈现状态；原始 Token 流无法表达暂停、工具进度或终态。

  2. 客户端断线重连时，如何避免重复显示或重复执行？

    查看参考答案

    客户端携带 last_event_id 恢复，服务端重放其后的不可变事件；输入消息使用稳定 idempotency key 去重。重连只恢复传输，不重新启动工具动作；编排 Run 与连接生命周期必须分离。

  3. 流式输出已经发送一半后发现安全问题怎么办？

    查看参考答案

    高风险内容应尽量在小缓冲区或结构化边界检查后再释放；一旦发现，立即停止流、发出明确 failed/retracted 事件并记录已暴露范围。不能假装未发生，必要时触发事件响应和用户通知。

  4. 图片、音频或附件进入模型前需要哪些信任处理？

    查看参考答案

    校验类型与大小、恶意文件、元数据和权限，做 OCR/转写时保留来源、时间与置信，并标记为不可信数据。敏感内容最小化后再进入推理，原文件访问使用短期受控引用。

  5. 慢客户端如何形成背压，系统应怎样处理？

    查看参考答案

    无限缓冲会耗尽内存并拉长连接。设置队列上限、心跳、合并低价值 delta、暂停上游或切换异步结果；关键终态和审批事件不得丢失，并记录 dropped/coalesced 指标。

面试表达

流式 Agent UI 怎样处理断线与部分结果？

我把输出建模成有类型、sequence 和稳定 id 的事件流，而不是 token 管道。客户端确认 cursor，重连时重放或获取 snapshot；进度、澄清、错误、消息完成和 run 完成分开。取消也是交互意图，最终状态由编排确认。多模态附件先经过隔离、类型校验、权限和派生处理，只把带 provenance 的 artifact ref 交给编排，因此接入层不会因收到文件就自行选择模型或执行任务。

日常实践关键点

核心判断

本章的日常工作不是记住名词，而是把“流式响应与多模态输入”变成可执行控制环：先识别触发信号，再用版本化证据定位首个偏离点，由 channel-owner 执行动作，最后用成功、拒绝、降级三类结果验收。推荐顺序是 定义事件协议 → 处理顺序与重连 → 表达局部失败 → 校验媒体输入 → 观测流式链路；任何参数都必须从真实 SLO、风险基线或失败样本推导。

排障顺序

  1. 冻结请求、策略、模型 / 工具、数据和部署版本，避免边复现边变化。
  2. 从最早可验证输入开始，逐段对照契约、决策、动作与验收事件；在首个偏离点停止跨层调参。
  3. 由 channel-owner 判断修复、降级、回滚或转人工，并记录未验证假设。
贯穿案例｜企业差旅 Agent：员工请求预订超预算航班时，只把制度条款当知识证据，把员工靠窗偏好当记忆，把预订动作交给工具层；本章只处理 流式响应与多模态输入 的控制与证据。若发现重复事件、身份错绑、流式乱序、必要信息缺失、同意撤回未生效或用户无法安全恢复，先保留 trace_id 与版本，再在本层首个偏离点修复，避免把问题推给 Prompt。
