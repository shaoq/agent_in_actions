04 同意、错误恢复与可访问性

学习目标

  • 把 consent 建模为有主体、目的、范围、版本、有效期和撤回路径的证据。
  • 用错误分类和权威状态设计安全的重试、取消、补偿与人工恢复体验。
  • 让键盘、读屏、低视力、听障、认知负荷和运动能力差异都能完成同一关键任务。

同意：可证明、可理解、可撤回

同意（consent）回答“用户是否理解并允许某种数据处理或交互用途”，不替代身份认证、资源授权、业务审批或工具执行。一个同意 receipt 应记录谁、同意了什么目的与数据类别、展示了哪个版本的说明、以何种交互作出、何时生效/过期、怎样撤回以及撤回传播到哪些系统。

设计要求可验证证据反例
目的与范围具体purpose、data categories、retention、recipients“同意所有条款并继续”
选择自由且对称accept/decline 同等可见，无预勾选拒绝藏在二级页面
版本与主体绑定notice_version、subject、tenant、locale、timestamp只保存布尔值 consent=true
可撤回与重新确认withdrawn_at、propagation status、material change trigger更新用途后沿用旧同意
最小必要按功能拆分可选目的不用个性化也强迫同意长期记忆

错误恢复：先说清事实，再提供安全动作

错误状态用户可见解释允许动作系统证据
input_invalid哪一项无法读取及格式要求原位修正、保留其他输入field、rule、safe detail
identity_required / forbidden需要重新验证或无权访问登录、切换主体、申请权限、退出assurance、policy decision、support ref
temporarily_unavailable服务暂不可用，尚未执行带退避重试、保存草稿、稍后通知dispatch=false、retry_after
effect_unknown请求可能已生效，不能直接再做一次查询状态、联系人工idempotency key、receipt/query ref
cancel_pending取消请求已收到，最终状态待确认查看状态、离开后通知run state、cancel event
partial_success哪些部分成功、哪些失败只重试安全部分、补偿或人工处理per-step outcome、receipts

错误文案应回答：发生了什么、用户数据/动作是否安全、系统做了什么、用户现在能做什么、如何获得支持。内部堆栈、模型 prompt、供应商响应和敏感资源标识只进入受控 trace，不直接暴露。

可访问性：关键状态不能只靠颜色或动画

  • 语义结构：原生 button、label、heading、table 与 landmark；自定义控件必须补齐名称、角色、状态和键盘行为。
  • 焦点管理：流式 delta 不抢焦点；澄清/审批 modal 打开时焦点进入，Escape 关闭后返回触发点；错误摘要可聚焦并链接到字段。
  • 状态播报：普通进度使用节制的 aria-live="polite"，阻断错误使用明确 alert；不要逐 token 让读屏反复朗读。
  • 多通道等价：音频有字幕/transcript，图片有替代文本，拖拽有按钮替代，语音输入可回看和修正。
  • 视觉与动作：可见焦点、高对比度、200% 缩放、44px 左右触控目标、减少动画偏好；状态同时用文字/图标表达。
  • 时间限制：审批或会话将过期前预警，可延长或保存；身份安全要求不允许延长时解释原因并保存非敏感草稿。
恢复路径必须可访问 · 如果错误提示能被读屏读到，但“重试/联系人工”只能鼠标点击，任务仍未可访问；如果 spinner 无限转且没有状态文字，所有用户都无法判断是否应重试。

工程要点

  1. 以统一 error code 驱动跨渠道一致语义，渠道只适配呈现；不要靠文案字符串判断重试。
  2. 重试按钮携带原 run/action ref，而不是重新提交自由文本；编排根据派发与 effect 状态决定重试、查询或拒绝。
  3. 页面刷新、设备切换或渠道降级后恢复权威 checkpoint，同时保留用户未提交的本地草稿并明确版本冲突。
  4. 对 consent、approval、cancel、retry 和 handoff 建立端到端键盘/读屏测试，记录 user-visible state 与 trace 的关联。
  5. 监测失败恢复率、重复提交、澄清放弃、转人工成功率和按辅助技术分层的完成率；差异是产品缺陷信号。

常见坑

  • 同意墙：拒绝就无法使用不相关核心功能，且用途模糊。
  • 错误后清空表单：用户重复劳动，辅助技术用户成本更高。
  • 所有失败都“重试”：effect unknown 下可能重复扣款或发信。
  • 自动重连隐藏状态：客户端看似恢复，实际 run 已过期或主体变更。
  • 只做颜色对比：缺少语义、焦点、键盘和状态播报，仍无法完成任务。
  • 自动聚焦新 token：打断读屏和键盘用户当前操作。

理论追问

请先脱稿回答，再展开参考答案核对边界、证据和取舍。

  1. 一份有效的用户同意记录应包含什么？

    查看参考答案

    包含主体、用途、数据类别、动作范围、展示文本版本、时间、渠道、有效期和撤回方式，并与具体委托关联。笼统接受服务条款不等于授权保存长期 Memory 或执行高风险动作。

  2. 错误提示为什么要区分“未执行”“可能已执行”和“已执行但验收失败”？

    查看参考答案

    这三种状态决定用户能否安全重试。模糊的“出错了”会诱发重复付款或工单；界面应基于编排和工具 Observation 说明已知事实、未知范围、系统正在查询还是需要人工对账。

  3. 用户刷新页面后，如何安全恢复审批或长任务？

    查看参考答案

    接入层用会话与 last_event_id 找回同一 Run，编排从持久 State 恢复，并重新验证身份、授权、审批有效期和资源版本。刷新只重建视图，不能创建新的副作用执行。

  4. 可访问性为什么不能只靠颜色表示运行状态？

    查看参考答案

    色觉差异、屏幕阅读器和弱网用户可能无法识别。状态需有文字、语义标签、ARIA live 策略和可聚焦控件；流式更新避免过度播报，审批与错误必须支持键盘操作和清晰焦点。

  5. 用户撤回同意时，进行中的 Run 和已保存数据如何处理？

    查看参考答案

    立即阻止依赖该同意的新动作，编排评估在途副作用并暂停或安全终止；按用途删除或限制已保存数据和 Memory，记录撤回与执行结果。法规要求保留的审计证据应最小化并说明依据。

面试表达

怎样设计可信、可恢复且可访问的 Agent 交互？

我把 consent 保存为绑定主体、目的、数据范围、notice 版本和撤回传播的 receipt，并明确它不替代授权或审批。错误协议区分未派发、明确失败、未知副作用、部分成功和取消等待，UI 只提供状态安全的动作。所有关键路径使用语义控件、稳定焦点、节制 live region 和多模态等价内容；刷新或重连从编排 checkpoint 恢复。接入层呈现与传递决定，但任务执行仍由编排、治理和工具链完成。
