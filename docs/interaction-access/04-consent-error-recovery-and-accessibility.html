04 同意、错误恢复与可访问性

学习目标

  • 把 consent 建模为有主体、目的、范围、版本、有效期和撤回路径的证据。
  • 用错误分类和权威状态设计安全的重试、取消、补偿与人工恢复体验。
  • 让键盘、读屏、低视力、听障、认知负荷和运动能力差异都能完成同一关键任务。

链路总览与现有讲解

同意、错误恢复与可访问性属于交互与接入层：输入是渠道事件、媒体、session、actor、consent 和客户端能力，输出是规范化 RequestEnvelope、StreamEvent、HandoffPackage 与用户可见状态。学习时先沿现有图、表或主题章节理解链路，再用后续 stage 检查每个中间产物的不变量。

同意：可证明、可理解、可撤回

同意（consent）回答“用户是否理解并允许某种数据处理或交互用途”，不替代身份认证、资源授权、业务审批或工具执行。一个同意 receipt 应记录谁、同意了什么目的与数据类别、展示了哪个版本的说明、以何种交互作出、何时生效/过期、怎样撤回以及撤回传播到哪些系统。

设计要求可验证证据反例
目的与范围具体purpose、data categories、retention、recipients“同意所有条款并继续”
选择自由且对称accept/decline 同等可见，无预勾选拒绝藏在二级页面
版本与主体绑定notice_version、subject、tenant、locale、timestamp只保存布尔值 consent=true
可撤回与重新确认withdrawn_at、propagation status、material change trigger更新用途后沿用旧同意

错误恢复：先说清事实，再提供安全动作

错误状态用户可见解释允许动作系统证据
input_invalid哪一项无法读取及格式要求原位修正、保留其他输入field、rule、safe detail
identity_required / forbidden需要重新验证或无权访问登录、切换主体、申请权限、退出assurance、policy decision、support ref
temporarily_unavailable服务暂不可用，尚未执行带退避重试、保存草稿、稍后通知dispatch=false、retry_after
effect_unknown请求可能已生效，不能直接再做一次查询状态、联系人工idempotency key、receipt/query ref
cancel_pending取消请求已收到，最终状态待确认查看状态、离开后通知run state、cancel event
partial_success哪些部分成功、哪些失败只重试安全部分、补偿或人工处理per-step outcome、receipts

错误文案应回答：发生了什么、用户数据/动作是否安全、系统做了什么、用户现在能做什么、如何获得支持。内部堆栈、模型 prompt、供应商响应和敏感资源标识只进入受控 trace，不直接暴露。

可访问性：关键状态不能只靠颜色或动画

  • 语义结构：原生 button、label、heading、table 与 landmark；自定义控件必须补齐名称、角色、状态和键盘行为。
  • 焦点管理：流式 delta 不抢焦点；澄清/审批 modal 打开时焦点进入，Escape 关闭后返回触发点；错误摘要可聚焦并链接到字段。
  • 状态播报：普通进度使用节制的 aria-live="polite"，阻断错误使用明确 alert；不要逐 token 让读屏反复朗读。
  • 多通道等价：音频有字幕/transcript，图片有替代文本，拖拽有按钮替代，语音输入可回看和修正。
  • 视觉与动作：可见焦点、高对比度、200% 缩放、44px 左右触控目标、减少动画偏好；状态同时用文字/图标表达。
  • 时间限制：审批或会话将过期前预警，可延长或保存；身份安全要求不允许延长时解释原因并保存非敏感草稿。
恢复路径必须可访问 · 如果错误提示能被读屏读到，但“重试/联系人工”只能鼠标点击，任务仍未可访问；如果 spinner 无限转且没有状态文字，所有用户都无法判断是否应重试。

核心环节：从输入到可验证输出

1. 用户同意

用户同意必须绑定当前 actor、资源、动作、目的和版本，并在真正使用能力或数据的时刻仍然有效。

职责与 I/O
channel-owner 在“用户同意”环节接收 渠道事件、媒体、session、actor、consent 和客户端能力，产出 规范化 RequestEnvelope、StreamEvent、HandoffPackage 与用户可见状态；输出必须能由下游按版本消费。
工作机制
身份链先把渠道主体映射为内部 actor，再由策略决策与短期能力令牌约束数据、工具、文件和网络使用。本环节把 渠道事件、媒体、session、actor、consent 和客户端能力 中与“用户同意”相关的事实转成 规范化 RequestEnvelope、StreamEvent、HandoffPackage 与用户可见状态 的一部分。
必须成立的约束
默认最小权限与 deny-by-default；撤回、角色变更、资源版本变化必须使缓存授权和派生访问失效。本章共同约束是：事件关联、身份绑定、同意、顺序和可访问性必须端到端成立；违反时必须产生可区分的拒绝、未知或升级结果。
难点与取舍
频繁再授权更安全但增加延迟；缓存更快却可能陈旧，TTL 应由权限变化频率和风险窗口推导。本层总体取舍是 交互流畅度与澄清 / 审批摩擦、实时性与可恢复性，参数必须引用版本化数据、SLO 或风险基线。
边界与注意事项
身份认证不等于授权，脱敏不等于匿名化，网络隔离也不能替代业务策略判断。交互与接入层 的总边界是：不做业务推理、流程控制或工具执行，只保证输入输出边界可信可用；错误会继续污染 编排入口、身份授权、用户理解和人工接管质量。

2. 传播同意撤回

传播同意撤回必须让用户和服务端对事件顺序、已确认状态、局部失败与可恢复动作形成同一事实。

职责与 I/O
channel-owner 在“传播同意撤回”环节接收 渠道事件、媒体、session、actor、consent 和客户端能力，产出 规范化 RequestEnvelope、StreamEvent、HandoffPackage 与用户可见状态；输出必须能由下游按版本消费。
工作机制
事件协议携带 event_id、sequence、kind、final 与 retryability；重连从确认点恢复，多模态先校验再规范化。本环节把 渠道事件、媒体、session、actor、consent 和客户端能力 中与“传播同意撤回”相关的事实转成 规范化 RequestEnvelope、StreamEvent、HandoffPackage 与用户可见状态 的一部分。
必须成立的约束
重复事件幂等、乱序可检测、同意撤回可传播，关键状态不能只靠颜色、声音或动画表达。本章共同约束是：事件关联、身份绑定、同意、顺序和可访问性必须端到端成立；违反时必须产生可区分的拒绝、未知或升级结果。
难点与取舍
更细事件提高反馈及时性却增加客户端状态复杂度；粒度由用户感知、重连成本和审计需求决定。本层总体取舍是 交互流畅度与澄清 / 审批摩擦、实时性与可恢复性，参数必须引用版本化数据、SLO 或风险基线。
边界与注意事项
接入层表达渠道事实与用户状态，不把局部 UI 完成当成业务完成，也不替编排决定下一步。交互与接入层 的总边界是：不做业务推理、流程控制或工具执行，只保证输入输出边界可信可用；错误会继续污染 编排入口、身份授权、用户理解和人工接管质量。

3. 错误契约

错误契约是生产者与消费者共同承认的版本化解释边界；结构合法不代表语义、授权或业务结果成立。

职责与 I/O
channel-owner 在“错误契约”环节接收 渠道事件、媒体、session、actor、consent 和客户端能力，产出 规范化 RequestEnvelope、StreamEvent、HandoffPackage 与用户可见状态；输出必须能由下游按版本消费。
工作机制
显式 schema 定义字段、枚举、状态转移和错误联合，消费者先做语法校验，再按领域不变量做语义校验。本环节把 渠道事件、媒体、session、actor、consent 和客户端能力 中与“错误契约”相关的事实转成 规范化 RequestEnvelope、StreamEvent、HandoffPackage 与用户可见状态 的一部分。
必须成立的约束
必填字段、版本兼容、非法状态、未知枚举和错误语义必须确定；不能用自由文本默认值掩盖缺失事实。本章共同约束是：事件关联、身份绑定、同意、顺序和可访问性必须端到端成立；违反时必须产生可区分的拒绝、未知或升级结果。
难点与取舍
严格契约提升可恢复性但增加演进成本；兼容窗口由调用方升级节奏和旧数据回放覆盖决定。本层总体取舍是 交互流畅度与澄清 / 审批摩擦、实时性与可恢复性，参数必须引用版本化数据、SLO 或风险基线。
边界与注意事项
契约约束信息形状和解释，不授予权限、不选择业务策略，也不证明下游验收成功。交互与接入层 的总边界是：不做业务推理、流程控制或工具执行，只保证输入输出边界可信可用；错误会继续污染 编排入口、身份授权、用户理解和人工接管质量。

4. 安全恢复

安全恢复只能把系统带回已知状态或安全终态；未知结果在查明副作用前不得被当成可重试失败。

职责与 I/O
channel-owner 在“安全恢复”环节接收 渠道事件、媒体、session、actor、consent 和客户端能力，产出 规范化 RequestEnvelope、StreamEvent、HandoffPackage 与用户可见状态；输出必须能由下游按版本消费。
工作机制
错误先分瞬时、永久、冲突与 unknown outcome；幂等键、ledger、checkpoint 和补偿记录共同决定 retry、query、resume 或人工核对。本环节把 渠道事件、媒体、session、actor、consent 和客户端能力 中与“安全恢复”相关的事实转成 规范化 RequestEnvelope、StreamEvent、HandoffPackage 与用户可见状态 的一部分。
必须成立的约束
尝试次数、累计 deadline、退避、副作用作用域和补偿 owner 都必须有界，幂等键需覆盖业务动作而非单次请求。本章共同约束是：事件关联、身份绑定、同意、顺序和可访问性必须端到端成立；违反时必须产生可区分的拒绝、未知或升级结果。
难点与取舍
重试提高瞬时可用性也会放大负载与副作用；预算由下游容量、用户 deadline 和动作不可逆性决定。本层总体取舍是 交互流畅度与澄清 / 审批摩擦、实时性与可恢复性，参数必须引用版本化数据、SLO 或风险基线。
边界与注意事项
补偿是新的业务动作而非时间回滚；网关 fallback 不应掩盖工具副作用或编排非法状态。交互与接入层 的总边界是：不做业务推理、流程控制或工具执行，只保证输入输出边界可信可用；错误会继续污染 编排入口、身份授权、用户理解和人工接管质量。

失败传播与 first divergence

  1. 上游输入：先确认 渠道事件、媒体、session、actor、consent 和客户端能力 的身份、作用域、时间与版本是否一致；输入错误不应在本层被“修正”为新事实。
  2. 首个偏离：按 用户同意 → 传播同意撤回 → 错误契约 → 安全恢复 → 可访问性检查 比较预期与实际中间产物，第一个不满足概念不变量的 stage 才是本章候选根因。
  3. 下游影响：本层错误会改变 编排入口、身份授权、用户理解和人工接管质量；最终症状可能在更晚阶段出现，但不能从症状直接倒推 Prompt。
  4. 归属边界：channel-owner修复本层契约或机制；若 first divergence 位于相邻层，保留证据并交给相应 owner，不跨层调参。

核心区回答“是什么、为何成立、怎样传播”，实践区回答“何时触发、如何处置、怎样验收”。每个 stage 与 practice 使用稳定锚点双向连接：

工程要点

  1. 以统一 error code 驱动跨渠道一致语义，渠道只适配呈现；不要靠文案字符串判断重试。
  2. 重试按钮携带原 run/action ref，而不是重新提交自由文本；编排根据派发与 effect 状态决定重试、查询或拒绝。
  3. 页面刷新、设备切换或渠道降级后恢复权威 checkpoint，同时保留用户未提交的本地草稿并明确版本冲突。
  4. 对 consent、approval、cancel、retry 和 handoff 建立端到端键盘/读屏测试，记录 user-visible state 与 trace 的关联。
  5. 监测失败恢复率、重复提交、澄清放弃、转人工成功率和按辅助技术分层的完成率；差异是产品缺陷信号。

常见坑

  • 同意墙：拒绝就无法使用不相关核心功能，且用途模糊。
  • 错误后清空表单：用户重复劳动，辅助技术用户成本更高。
  • 所有失败都“重试”：effect unknown 下可能重复扣款或发信。
  • 自动重连隐藏状态：客户端看似恢复，实际 run 已过期或主体变更。
  • 只做颜色对比：缺少语义、焦点、键盘和状态播报，仍无法完成任务。
  • 自动聚焦新 token：打断读屏和键盘用户当前操作。

    包含主体、用途、数据类别、动作范围、展示文本版本、时间、渠道、有效期和撤回方式，并与具体委托关联。笼统接受服务条款不等于授权保存长期 Memory 或执行高风险动作。

  2. 错误提示为什么要区分“未执行”“可能已执行”和“已执行但验收失败”？

    查看参考答案

    这三种状态决定用户能否安全重试。模糊的“出错了”会诱发重复付款或工单；界面应基于编排和工具 Observation 说明已知事实、未知范围、系统正在查询还是需要人工对账。

  3. 用户刷新页面后，如何安全恢复审批或长任务？

    查看参考答案

    接入层用会话与 last_event_id 找回同一 Run，编排从持久 State 恢复，并重新验证身份、授权、审批有效期和资源版本。刷新只重建视图，不能创建新的副作用执行。

  4. 可访问性为什么不能只靠颜色表示运行状态？

    查看参考答案

    色觉差异、屏幕阅读器和弱网用户可能无法识别。状态需有文字、语义标签、ARIA live 策略和可聚焦控件；流式更新避免过度播报，审批与错误必须支持键盘操作和清晰焦点。

  5. 用户撤回同意时，进行中的 Run 和已保存数据如何处理？

    查看参考答案

    立即阻止依赖该同意的新动作，编排评估在途副作用并暂停或安全终止；按用途删除或限制已保存数据和 Memory，记录撤回与执行结果。法规要求保留的审计证据应最小化并说明依据。

面试表达

怎样设计可信、可恢复且可访问的 Agent 交互？

我把 consent 保存为绑定主体、目的、数据范围、notice 版本和撤回传播的 receipt，并明确它不替代授权或审批。错误协议区分未派发、明确失败、未知副作用、部分成功和取消等待，UI 只提供状态安全的动作。所有关键路径使用语义控件、稳定焦点、节制 live region 和多模态等价内容；刷新或重连从编排 checkpoint 恢复。接入层呈现与传递决定，但任务执行仍由编排、治理和工具链完成。

日常实践关键点

核心判断

本章的日常工作不是记住名词，而是把“同意、错误恢复与可访问性”变成可执行控制环：先识别触发信号，再用版本化证据定位首个偏离点，由 channel-owner 执行动作，最后用成功、拒绝、降级三类结果验收。推荐顺序是 记录用户同意 → 传播同意撤回 → 返回错误契约 → 提供安全恢复 → 执行可访问性检查；任何参数都必须从真实 SLO、风险基线或失败样本推导。

深度实践卡

    触发信号
    当企业差旅 Agent 的“同意、错误恢复与可访问性”链路出现重复事件、身份错绑、流式乱序、必要信息缺失、同意撤回未生效或用户无法安全恢复，或变更直接触及“记录用户同意”的输入、策略或 owner 时，必须启动这项检查。
    诊断顺序
    诊断“记录用户同意”时先冻结 channel、event_id、session_id、actor、consent、媒体元数据与客户端能力。重点确认决策主体、资源、动作、目的和策略版本是否在“使用点”仍一致，不能只看入口鉴权。
    执行动作
    落实“记录用户同意”时，把允许、拒绝和需审批做成类型化策略结果，默认拒绝缺字段或版本不明的请求。由 channel-owner 记录执行前后的版本、决定与接管条件。
    验证方法
    验收“记录用户同意”时，执行重连、乱序、重复提交、撤回、无障碍与多渠道移交测试，并核对用户可见状态；同时比较正确、拒绝和降级结果，阈值只从真实基线与风险容忍度推导。

    一旦主体或目的无法证明，立即撤销能力令牌并转人工确认，不用事后过滤替代前置授权；停止自动推进，保留可恢复会话与用户确认点，必要时降级为同步文本或转人工。对“记录用户同意”先保证可恢复和可审计，再优化自动化率。

排障顺序

  1. 冻结请求、策略、模型 / 工具、数据和部署版本，避免边复现边变化。
  2. 从最早可验证输入开始，逐段对照契约、决策、动作与验收事件；在首个偏离点停止跨层调参。
  3. 由 channel-owner 判断修复、降级、回滚或转人工，并记录未验证假设。
贯穿案例｜企业差旅 Agent：员工请求预订超预算航班时，只把制度条款当知识证据，把员工靠窗偏好当记忆，把预订动作交给工具层；本章只处理 同意、错误恢复与可访问性 的控制与证据。若发现重复事件、身份错绑、流式乱序、必要信息缺失、同意撤回未生效或用户无法安全恢复，先保留 trace_id 与版本，再在本层首个偏离点修复，避免把问题推给 Prompt。
