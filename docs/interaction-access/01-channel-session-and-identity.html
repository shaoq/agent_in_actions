01 渠道、会话与身份

学习目标

  • 区分 transport connection、channel conversation、session、subject 与 run 的 owner 和生命周期。
  • 把渠道事件规范化为可版本化、可去重、可追踪的输入契约。
  • 解释 authentication、identity linkage、delegation 和 authorization 不是同一件事。

核心概念

接入系统最危险的捷径是用一个 session_id 同时代表连接、对话、用户和任务。WebSocket 重连不会创造新用户，用户切换账号不应继承旧主体，群聊中的 conversation 也不能证明当前发言者有权访问某个 run。

对象owner / 生命周期用途不可替代
connection_id渠道适配器 / 单次连接背压、心跳、重连游标不能证明用户身份
conversation_id渠道 / 对话容器私聊、群聊、话题线程不能等同一次 Agent 任务
session_id交互+编排 / 一段连续交互临时上下文、设备和能力不能自动成为长期记忆或权限
subject_id身份系统 / 已验证主体租户、用户、代理人、服务主体不能由显示名或模型猜测
run_id编排层 / 一次可恢复任务状态、步骤、预算、完成判定不能由接入层推进或复用

规范化输入契约

{
  "event_id": "evt_01J...",
  "occurred_at": "2026-07-17T09:30:00Z",
  "channel": {"kind": "im", "conversation_id": "c_42", "capabilities": ["text", "file", "buttons"]},
  "identity_context": {"tenant_id": "t_7", "subject_id": "u_9", "assurance": "mfa", "actor_type": "human"},
  "session_ref": {"session_id": "s_3", "run_id": null, "reply_to": "msg_8"},
  "intent_input": {"kind": "user_message", "text": "帮我取消明天的订单", "attachments": []},
  "locale": "zh-CN", "timezone": "Asia/Shanghai", "trace_id": "tr_a1",
  "schema_version": "1.2"
}

intent_input 是输入事实，不是最终执行意图。接入层可以做确定性结构化、编码标准化、附件引用和显式 UI intent 映射；语义歧义仍应交给编排/推理形成 clarify 或候选动作。

工程要点

  1. 适配器薄层：每个渠道只负责签名验证、事件解析、能力声明、速率/重放处理和统一输出，不复制业务流程。
  2. 身份连续性：账号绑定、设备切换、代理人操作和匿名转登录都要产生显式 linkage 事件；迁移 session 前重新检查租户和主体。
  3. 去重有边界：以 tenant + channel + event_id 去重；保留 payload hash 与处理状态。渠道重放可以幂等接收，但不能把已执行副作用重新触发。
  4. 最小上下文：向编排传递验证过的主体、作用域、身份强度和委托链；不把 access token、cookie 或原始 secret 放进 prompt 或 trace。
  5. 能力协商：声明是否支持 streaming、attachments、buttons、audio、edit、thread 和 receipts；降级必须保留语义而非静默丢字段。

失败定位矩阵

现象优先 owner关键证据
同一消息被处理两次渠道适配 / 编排幂等event_id、dedupe 状态、run 创建记录
群聊看到他人订单身份/授权，严重安全事件subject、tenant、resource scope、policy decision
换设备后任务丢失session/run 关联linkage event、run_id、checkpoint
按钮触发了错误动作接入契约 / 编排校验button value、schema version、action summary

常见坑

  • 相信客户端传来的 user_id：主体必须由服务端验证的 credential 或受控映射产生。
  • 把登录当授权：authentication 只证明是谁；资源与动作是否允许仍由治理策略判定。
  • 渠道内编排：在 Slack handler、Web controller 各写一套重试和任务推进，最终状态无法一致恢复。
  • 保存原始 token：扩大日志、prompt 与人工移交的泄露面。
  • conversation=session=memory：混淆生命周期，造成跨用户污染和无限保留。

理论追问

请先脱稿回答，再展开参考答案核对边界、证据和取舍。

  1. 交互与接入层为什么要把不同渠道规范化成统一事件？

    查看参考答案

    Web、IM、语音和 API 的消息、重试、附件与回执语义不同。接入层转换为带 event_id、session、actor、modality 和 correlation 的内部契约，使编排不依赖渠道细节，并能统一去重、恢复和审计。

  2. Session 为什么不等于长期 Memory？

    查看参考答案

    Session 维持一段交互的连接、消息顺序和临时上下文，通常随会话结束或短 TTL 清理；Memory 跨会话保存经过治理的经历。把 Session 全量沉淀会保存临时指令、敏感数据和错误陈述。

  3. 用户身份在进入 Agent 后应如何传递？

    查看参考答案

    使用可验证的 subject、tenant、认证强度、delegation 和同意信息作为结构化上下文，由治理和资源侧校验；不能只把用户名写进 Prompt。身份凭据本身应留在受控边界，Agent 获取的是最小声明和短期能力。

  4. 为什么每个渠道事件都需要稳定 event_id 和 correlation_id？

    查看参考答案

    断线重连、平台重投和异步回执会产生重复或乱序。稳定 ID 支持去重、把用户消息关联到同一 Run 和响应，并让 Trace 还原渠道、编排、工具与最终交付的因果关系。

  5. 从机器人转人工时，接入层和编排层分别负责什么？

    查看参考答案

    编排决定何时进入 handoff、冻结或结束 Run，并生成最小必要上下文；接入层把移交包、安全地路由到人工渠道，维护用户可见状态和回执。接入层不应私自继续业务流程。

面试表达

多渠道 Agent 如何保持身份与状态一致？

我先拆开连接、渠道对话、交互 session、已验证主体和编排 run。渠道适配器只把签名通过的事件规范化并附上 identity assurance、能力和 trace；编排拥有 run 与恢复。跨设备或匿名转登录通过显式 linkage 事件重绑，重新校验 tenant、subject 和授权。事件用渠道级 id 去重，但工具副作用仍按动作层幂等契约处理，因此接入层不会自行执行任务。
