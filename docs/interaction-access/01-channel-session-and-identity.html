01 渠道、会话与身份

学习目标

  • 区分 transport connection、channel conversation、session、subject 与 run 的 owner 和生命周期。
  • 把渠道事件规范化为可版本化、可去重、可追踪的输入契约。
  • 解释 authentication、identity linkage、delegation 和 authorization 不是同一件事。

链路总览与现有讲解

渠道、会话与身份属于交互与接入层：输入是渠道事件、媒体、session、actor、consent 和客户端能力，输出是规范化 RequestEnvelope、StreamEvent、HandoffPackage 与用户可见状态。学习时先沿现有图、表或主题章节理解链路，再用后续 stage 检查每个中间产物的不变量。

核心概念

接入系统最危险的捷径是用一个 session_id 同时代表连接、对话、用户和任务。WebSocket 重连不会创造新用户，用户切换账号不应继承旧主体，群聊中的 conversation 也不能证明当前发言者有权访问某个 run。

对象owner / 生命周期用途不可替代
connection_id渠道适配器 / 单次连接背压、心跳、重连游标不能证明用户身份
conversation_id渠道 / 对话容器私聊、群聊、话题线程不能等同一次 Agent 任务
session_id交互+编排 / 一段连续交互临时上下文、设备和能力不能自动成为长期记忆或权限
subject_id身份系统 / 已验证主体租户、用户、代理人、服务主体不能由显示名或模型猜测
run_id编排层 / 一次可恢复任务状态、步骤、预算、完成判定不能由接入层推进或复用

intent_input 是输入事实，不是最终执行意图。接入层可以做确定性结构化、编码标准化、附件引用和显式 UI intent 映射；语义歧义仍应交给编排/推理形成 clarify 或候选动作。

核心环节：从输入到可验证输出

1. 渠道 Envelope

渠道 Envelope是生产者与消费者共同承认的版本化解释边界；结构合法不代表语义、授权或业务结果成立。

职责与 I/O
channel-owner 在“渠道 Envelope”环节接收 渠道事件、媒体、session、actor、consent 和客户端能力，产出 规范化 RequestEnvelope、StreamEvent、HandoffPackage 与用户可见状态；输出必须能由下游按版本消费。
工作机制
显式 schema 定义字段、枚举、状态转移和错误联合，消费者先做语法校验，再按领域不变量做语义校验。本环节把 渠道事件、媒体、session、actor、consent 和客户端能力 中与“渠道 Envelope”相关的事实转成 规范化 RequestEnvelope、StreamEvent、HandoffPackage 与用户可见状态 的一部分。
必须成立的约束
必填字段、版本兼容、非法状态、未知枚举和错误语义必须确定；不能用自由文本默认值掩盖缺失事实。本章共同约束是：事件关联、身份绑定、同意、顺序和可访问性必须端到端成立；违反时必须产生可区分的拒绝、未知或升级结果。
难点与取舍
严格契约提升可恢复性但增加演进成本；兼容窗口由调用方升级节奏和旧数据回放覆盖决定。本层总体取舍是 交互流畅度与澄清 / 审批摩擦、实时性与可恢复性，参数必须引用版本化数据、SLO 或风险基线。
边界与注意事项
契约约束信息形状和解释，不授予权限、不选择业务策略，也不证明下游验收成功。交互与接入层 的总边界是：不做业务推理、流程控制或工具执行，只保证输入输出边界可信可用；错误会继续污染 编排入口、身份授权、用户理解和人工接管质量。

2. 关联会话

关联会话必须能把结论追溯到同一请求的输入、版本、决定、动作和验收事件，而不是依赖不可检索的日志文本。

职责与 I/O
channel-owner 在“关联会话”环节接收 渠道事件、媒体、session、actor、consent 和客户端能力，产出 规范化 RequestEnvelope、StreamEvent、HandoffPackage 与用户可见状态；输出必须能由下游按版本消费。
工作机制
correlation_id / trace_id 串联 span 与结构化事件，引用保留原始证据位置、policy / model / tool 版本和 owner。本环节把 渠道事件、媒体、session、actor、consent 和客户端能力 中与“关联会话”相关的事实转成 规范化 RequestEnvelope、StreamEvent、HandoffPackage 与用户可见状态 的一部分。
必须成立的约束
时间、因果顺序、采样与脱敏策略必须明确；高基数字段不能无界进入 Metric，敏感原文不能直接写 Log。本章共同约束是：事件关联、身份绑定、同意、顺序和可访问性必须端到端成立；违反时必须产生可区分的拒绝、未知或升级结果。
难点与取舍
全量遥测利于归因却增加成本与隐私面；采样按风险与失败类型分层，关键审计事件不得随机丢弃。本层总体取舍是 交互流畅度与澄清 / 审批摩擦、实时性与可恢复性，参数必须引用版本化数据、SLO 或风险基线。
边界与注意事项
Trace 解释系统事件与可见证据，不保存模型私有推理；Metric 告警不能替代逐请求审计。交互与接入层 的总边界是：不做业务推理、流程控制或工具执行，只保证输入输出边界可信可用；错误会继续污染 编排入口、身份授权、用户理解和人工接管质量。

3. 当前身份

当前身份必须绑定当前 actor、资源、动作、目的和版本，并在真正使用能力或数据的时刻仍然有效。

职责与 I/O
channel-owner 在“当前身份”环节接收 渠道事件、媒体、session、actor、consent 和客户端能力，产出 规范化 RequestEnvelope、StreamEvent、HandoffPackage 与用户可见状态；输出必须能由下游按版本消费。
工作机制
身份链先把渠道主体映射为内部 actor，再由策略决策与短期能力令牌约束数据、工具、文件和网络使用。本环节把 渠道事件、媒体、session、actor、consent 和客户端能力 中与“当前身份”相关的事实转成 规范化 RequestEnvelope、StreamEvent、HandoffPackage 与用户可见状态 的一部分。
必须成立的约束
默认最小权限与 deny-by-default；撤回、角色变更、资源版本变化必须使缓存授权和派生访问失效。本章共同约束是：事件关联、身份绑定、同意、顺序和可访问性必须端到端成立；违反时必须产生可区分的拒绝、未知或升级结果。
难点与取舍
频繁再授权更安全但增加延迟；缓存更快却可能陈旧，TTL 应由权限变化频率和风险窗口推导。本层总体取舍是 交互流畅度与澄清 / 审批摩擦、实时性与可恢复性，参数必须引用版本化数据、SLO 或风险基线。
边界与注意事项
身份认证不等于授权，脱敏不等于匿名化，网络隔离也不能替代业务策略判断。交互与接入层 的总边界是：不做业务推理、流程控制或工具执行，只保证输入输出边界可信可用；错误会继续污染 编排入口、身份授权、用户理解和人工接管质量。

4. 抑制重复事件

抑制重复事件是生产者与消费者共同承认的版本化解释边界；结构合法不代表语义、授权或业务结果成立。

5. 渠道移交

渠道移交成立的前提是每个控制决定只有一个最终 owner，交接不转移未声明的权限与验收责任。

职责与 I/O
channel-owner 在“渠道移交”环节接收 渠道事件、媒体、session、actor、consent 和客户端能力，产出 规范化 RequestEnvelope、StreamEvent、HandoffPackage 与用户可见状态；输出必须能由下游按版本消费。
工作机制
控制点以类型化输入 / 输出连接 owner；交出的是受约束任务或事实，不是共享的隐式控制权。本环节把 渠道事件、媒体、session、actor、consent 和客户端能力 中与“渠道移交”相关的事实转成 规范化 RequestEnvelope、StreamEvent、HandoffPackage 与用户可见状态 的一部分。
必须成立的约束
决策、推进、动作、验收和事件升级必须分别可归属，交接双方不能同时写同一控制状态。本章共同约束是：事件关联、身份绑定、同意、顺序和可访问性必须端到端成立；违反时必须产生可区分的拒绝、未知或升级结果。
难点与取舍
拆分能提高并行度，却增加协调、上下文复制和冲突成本；是否拆分要由失败归因与吞吐基线决定。本层总体取舍是 交互流畅度与澄清 / 审批摩擦、实时性与可恢复性，参数必须引用版本化数据、SLO 或风险基线。
边界与注意事项
推理可以建议，工具可以执行，治理可以允许或拒绝，但只有编排 owner 推进状态，业务 owner 做最终验收。交互与接入层 的总边界是：不做业务推理、流程控制或工具执行，只保证输入输出边界可信可用；错误会继续污染 编排入口、身份授权、用户理解和人工接管质量。

难点与取舍

冲突目标参数依据决策边界
交互流畅度与澄清 / 审批摩擦、实时性与可恢复性使用版本化回放、失败切片、SLO 分位数和风险等级，不把教学值当生产默认。channel-owner只调整本层机制；跨层变化需要对应 owner 与独立验收。

失败传播与 first divergence

  1. 上游输入：先确认 渠道事件、媒体、session、actor、consent 和客户端能力 的身份、作用域、时间与版本是否一致；输入错误不应在本层被“修正”为新事实。
  2. 首个偏离：按 渠道 Envelope → 关联会话 → 当前身份 → 抑制重复事件 → 渠道移交 比较预期与实际中间产物，第一个不满足概念不变量的 stage 才是本章候选根因。
  3. 下游影响：本层错误会改变 编排入口、身份授权、用户理解和人工接管质量；最终症状可能在更晚阶段出现，但不能从症状直接倒推 Prompt。
  4. 归属边界：channel-owner修复本层契约或机制；若 first divergence 位于相邻层，保留证据并交给相应 owner，不跨层调参。

工程要点

  1. 适配器薄层：每个渠道只负责签名验证、事件解析、能力声明、速率/重放处理和统一输出，不复制业务流程。
  2. 身份连续性：账号绑定、设备切换、代理人操作和匿名转登录都要产生显式 linkage 事件；迁移 session 前重新检查租户和主体。
  3. 去重有边界：以 tenant + channel + event_id 去重；保留 payload hash 与处理状态。渠道重放可以幂等接收，但不能把已执行副作用重新触发。
  4. 最小上下文：向编排传递验证过的主体、作用域、身份强度和委托链；不把 access token、cookie 或原始 secret 放进 prompt 或 trace。
  5. 能力协商：声明是否支持 streaming、attachments、buttons、audio、edit、thread 和 receipts；降级必须保留语义而非静默丢字段。

常见坑

  • 相信客户端传来的 user_id：主体必须由服务端验证的 credential 或受控映射产生。
  • 把登录当授权：authentication 只证明是谁；资源与动作是否允许仍由治理策略判定。
  • 渠道内编排：在 Slack handler、Web controller 各写一套重试和任务推进，最终状态无法一致恢复。
  • 保存原始 token：扩大日志、prompt 与人工移交的泄露面。
  • conversation=session=memory：混淆生命周期，造成跨用户污染和无限保留。

理论追问

请先脱稿回答，再展开参考答案核对边界、证据和取舍。

  1. 交互与接入层为什么要把不同渠道规范化成统一事件？

    查看参考答案

    Web、IM、语音和 API 的消息、重试、附件与回执语义不同。接入层转换为带 event_id、session、actor、modality 和 correlation 的内部契约，使编排不依赖渠道细节，并能统一去重、恢复和审计。

  2. Session 为什么不等于长期 Memory？

    查看参考答案

    Session 维持一段交互的连接、消息顺序和临时上下文，通常随会话结束或短 TTL 清理；Memory 跨会话保存经过治理的经历。把 Session 全量沉淀会保存临时指令、敏感数据和错误陈述。

  3. 用户身份在进入 Agent 后应如何传递？

    查看参考答案

    使用可验证的 subject、tenant、认证强度、delegation 和同意信息作为结构化上下文，由治理和资源侧校验；不能只把用户名写进 Prompt。身份凭据本身应留在受控边界，Agent 获取的是最小声明和短期能力。

  4. 为什么每个渠道事件都需要稳定 event_id 和 correlation_id？

    查看参考答案

    断线重连、平台重投和异步回执会产生重复或乱序。稳定 ID 支持去重、把用户消息关联到同一 Run 和响应，并让 Trace 还原渠道、编排、工具与最终交付的因果关系。

  5. 从机器人转人工时，接入层和编排层分别负责什么？

    查看参考答案

    编排决定何时进入 handoff、冻结或结束 Run，并生成最小必要上下文；接入层把移交包、安全地路由到人工渠道，维护用户可见状态和回执。接入层不应私自继续业务流程。

面试表达

多渠道 Agent 如何保持身份与状态一致？

我先拆开连接、渠道对话、交互 session、已验证主体和编排 run。渠道适配器只把签名通过的事件规范化并附上 identity assurance、能力和 trace；编排拥有 run 与恢复。跨设备或匿名转登录通过显式 linkage 事件重绑，重新校验 tenant、subject 和授权。事件用渠道级 id 去重，但工具副作用仍按动作层幂等契约处理，因此接入层不会自行执行任务。

日常实践关键点

核心判断

本章的日常工作不是记住名词，而是把“渠道、会话与身份”变成可执行控制环：先识别触发信号，再用版本化证据定位首个偏离点，由 channel-owner 执行动作，最后用成功、拒绝、降级三类结果验收。推荐顺序是 规范化渠道 Envelope → 关联会话 → 绑定当前身份 → 抑制重复事件 → 处理渠道移交；任何参数都必须从真实 SLO、风险基线或失败样本推导。

  • 1. 规范化渠道 Envelope
    触发信号

    诊断顺序

排障顺序

  1. 冻结请求、策略、模型 / 工具、数据和部署版本，避免边复现边变化。
  2. 从最早可验证输入开始，逐段对照契约、决策、动作与验收事件；在首个偏离点停止跨层调参。
  3. 由 channel-owner 判断修复、降级、回滚或转人工，并记录未验证假设。
贯穿案例｜企业差旅 Agent：员工请求预订超预算航班时，只把制度条款当知识证据，把员工靠窗偏好当记忆，把预订动作交给工具层；本章只处理 渠道、会话与身份 的控制与证据。若发现重复事件、身份错绑、流式乱序、必要信息缺失、同意撤回未生效或用户无法安全恢复，先保留 trace_id 与版本，再在本层首个偏离点修复，避免把问题推给 Prompt。
