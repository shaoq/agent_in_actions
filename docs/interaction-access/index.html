交互与接入层理论课程

交互与接入层（Interaction & Access）位于用户、渠道与 Agent 内部系统之间。它把 Web、App、IM、语音或人工坐席中的事件转换成统一输入包，维护渠道连接和会话关联，校验并传递身份上下文，再把编排产生的进度、澄清、审批、结果与移交事件适配回用户可理解的形式。

核心边界 · 交互层负责规范化意图、身份、会话和渠道能力，并交给编排层；它不选择执行计划、不直接调用工具，也不把“用户点了按钮”当成已完成授权或已执行动作。

学习目标

  1. 用 channel、conversation、session、subject 与 run_id 划清连接、身份和任务生命周期。
  2. 设计可恢复的流式协议与多模态输入流水线，区分 transport complete、generation complete 和 business complete。
  3. 把澄清、Human-in-the-loop（HITL）和人工移交建模为可暂停、可恢复、可审计的交互协议。
  4. 让同意可被证明、撤回和重新确认；用类型化错误、无障碍语义和降级路径保护所有用户。
  5. 能从交互 trace 判断故障属于渠道适配、身份、编排、治理还是下游执行。

前置知识

先完成九层架构，并至少理解编排职责与边界、状态/会话/记忆边界和身份、权限与审计。本课程把这些内部能力接到真实产品入口，不重复它们的完整实现。

组件边界与主数据流

相邻组件交互层交付或接收不得越界
编排控制层提交规范化意图、identity_context、session_ref、channel_capabilities；接收状态事件不推进 run、不决定重试/完成、不执行工具
治理与可观测请求身份校验、同意/审批策略和审计；传播 trace_id不自行授予业务权限，不把登录等同于动作授权
记忆系统提供主体与同意入口，展示查看/更正/删除操作不把 session transcript 自动晋升为长期记忆
工具 / 动作层只展示编排返回的动作摘要、进度与 receipt不从 UI 直接旁路调用工具
人工坐席传递最小必要上下文、待办、证据和 resume token不泄露全量历史，不丢失机器侧 checkpoint

学习路径与必须产出

顺序单元完成产出
01渠道、会话与身份标识符 owner 表与规范化输入契约
02流式响应与多模态输入事件协议、重连续传和多模态处理链
03澄清、HITL 与人工移交暂停/恢复协议与 handoff package
04同意、错误恢复与可访问性同意 receipt、错误矩阵与无障碍清单
门禁交互与接入层理论检查点五项场景证据与首次/复测记录

完成标准

  • 能从同一客服场景区分 channel_connection、conversation_id、session_id、subject_id 和 run_id。
  • 能解释断线重连、重复事件、部分输出、附件失败、澄清和人工移交如何保持状态一致。
  • 能给出同意范围、身份绑定、有效期、撤回与动作摘要证据，并说明交互层为何不能自行执行。
  • 完成理论检查点：总分至少 80/100，且 A–E 每项至少 12/20。

目录中的 available 只表示资产可用；只有真实作答、评分和复测证据才能更新个人掌握状态。
