01 目标、数据集与基线

学习目标

  • 把业务成功、用户价值、风险和成本转换为可测验收目标。
  • 组合生产、专家和合成样本，建立代表性与挑战性兼顾的目标数据集。
  • 通过冻结版本、数据切分和基线协议获得可重复比较。

核心概念

链路总览与现有讲解

目标、数据集与基线属于评测层：输入是版本化目标、dataset、Gold、实际 Outcome / Trajectory 与评分器，输出是分切片指标、失败归因、gate decision 和回归样本。学习时先沿现有图、表或主题章节理解链路，再用后续 stage 检查每个中间产物的不变量。

数据来源价值主要风险控制
生产 trace 抽样反映真实分布和长尾PII、反馈偏差、只含旧策略脱敏、权限、时间切分、来源标签
专家编写目标明确、可补关键边界覆盖窄、理想化双人复核、失败分类配额
合成与变异快速扩大组合与攻击面模板痕迹、分布失真人工抽检、与生产切片对照
事故与红队高价值高风险回归样本少、可能泄密最小披露、永久挑战集、严格访问

核心环节：从输入到可验证输出

1. 评测目标

评测目标必须固定评测要回答的问题、样本与正确性证据；没有可追溯 Gold 和切片的分数不能支持版本比较。

工作机制

必须成立的约束
目标、样本、Gold、基线和切片版本必须共同进入 EvalRun；训练泄漏、重复样本和未知来源要显式标记。本章共同约束是：样本、Gold、系统、Judge 与策略版本必须可比；违反时必须产生可区分的拒绝、未知或升级结果。
难点与取舍
更宽覆盖提高发现率却增加维护成本；样本数量由风险分层、置信区间和失败长尾决定，不追求无依据的大集合。本层总体取舍是 覆盖率与成本、评分一致性与人工深度、速度与发布风险，参数必须引用版本化数据、SLO 或风险基线。
边界与注意事项
数据集代表声明的流量与风险范围，不等于线上真实分布；Gold 也要允许版本修订和争议状态。评测层 的总边界是：离线评测消费 Trace 并反馈编排，不参与单次在线授权；错误会继续污染 发布门禁、路由 / 编排调整、回归集和线上观测重点。

失败传播与 first divergence

  1. 上游输入：先确认 版本化目标、dataset、Gold、实际 Outcome / Trajectory 与评分器 的身份、作用域、时间与版本是否一致；输入错误不应在本层被“修正”为新事实。
  2. 首个偏离：按 评测目标 → 数据集 → 复核 Gold → 可比基线 → 切片覆盖 比较预期与实际中间产物，第一个不满足概念不变量的 stage 才是本章候选根因。
  3. 下游影响：本层错误会改变 发布门禁、路由 / 编排调整、回归集和线上观测重点；最终症状可能在更晚阶段出现，但不能从症状直接倒推 Prompt。
  4. 归属边界：evaluation-owner修复本层契约或机制；若 first divergence 位于相邻层，保留证据并交给相应 owner，不跨层调参。

工程要点

  1. 先写 evaluation contract：任务目标、输入约束、可接受输出、禁止行为、成本/延迟预算和严重度分级。
  2. 建立 failure taxonomy 与 slice：意图、语言、租户、工具、数据新鲜度、风险级别和长链长度；总平均不能替代关键切片。
  3. 每个样本记录 dataset_version、source、consent/retention、expected outcome、可接受变体、rubric、risk 与 provenance。
  4. train/dev/test、回归集和隐藏挑战集按语义去重，避免同一对话改写泄漏；测试集不能用于 prompt 调参。
  5. 基线冻结模型、prompt、工具/schema、知识快照、随机参数和运行环境，报告重复运行均值、方差与失败样本，而不只报单点分数。

常见坑

  • 指标先于目标：追求“准确率”却没有定义业务验收与禁止行为。
  • 只用成功生产样本：选择偏差会漏掉放弃、拒绝和事故长尾。
  • 污染测试集：针对固定题目调 prompt 后的提升不能代表泛化。
  • 新旧系统环境不同：知识快照或工具数据变化会伪装成模型改进。

理论追问

请先脱稿回答，再展开参考答案核对边界、证据和取舍。

  1. 如何把“Agent 更好用”转成可评测目标？

    查看参考答案

    先拆成业务成功、安全、质量、延迟和成本目标，为每项定义可观察结果、门槛、hard fail 和关键切片。例如退款任务不只看回答满意度，还要看正确工单率、越权率、重复副作用和人工接管率。

  2. 一个有代表性的评测集为什么不能只包含正常成功样本？

    查看参考答案

    生产失败集中在边界、长尾、权限拒绝、无答案、工具故障、对抗输入和状态恢复。数据集应覆盖正常、边界、失败与安全切片，并记录来源、主体、环境和期望证据；否则平均分会高估真实能力。

  3. 建立 baseline 的意义是什么？

    查看参考答案

    Baseline 提供最简单可行方案的质量、成本和延迟参照，证明复杂 Agent、Multi-Agent 或新模型是否真的增益。没有 baseline，只能看到新版本绝对分数，无法判断增加的复杂度是否值得。

  4. 如何避免评测数据泄漏到 Prompt、模型选择或开发过程？

    查看参考答案

    区分开发集、校准集和保留测试集，限制访问并记录使用历史；对外部基准检查训练污染，最终门禁使用未参与调参的样本。线上回灌样本也要去重，防止同一问题同时出现在示例和测试。

  5. 为什么数据集、环境和评分器都必须版本化？

    查看参考答案

    结果只有在输入、工具环境、模型/Prompt、评分规则和依赖版本可追溯时才可比较。任何一项变化都可能造成分数漂移；报告需绑定 dataset、environment、runner 和 evaluator version，支持重放与回滚。

面试表达

如何为 Agent 建立评测集？

我先从产品目标、禁止行为和失败分类定义 evaluation contract，再按真实流量、专家边界、合成变异和事故红队四类来源分层取样。样本保留来源、风险、版本和 rubric，做语义去重与时间切分；测试集和隐藏挑战集不参与调参。基线冻结模型、prompt、工具、知识快照与随机性，最终按关键切片和不确定性报告。

日常实践关键点

核心判断

本章的日常工作不是记住名词，而是把“目标、数据集与基线”变成可执行控制环：先识别触发信号，再用版本化证据定位首个偏离点，由 evaluation-owner 执行动作，最后用成功、拒绝、降级三类结果验收。推荐顺序是 固化评测目标 → 版本化数据集 → 复核 Gold → 固定可比基线 → 检查切片覆盖；任何参数都必须从真实 SLO、风险基线或失败样本推导。

深度实践卡

  • 1. 固化评测目标
    触发信号

排障顺序

  1. 冻结请求、策略、模型 / 工具、数据和部署版本，避免边复现边变化。
  2. 从最早可验证输入开始，逐段对照契约、决策、动作与验收事件；在首个偏离点停止跨层调参。
  3. 由 evaluation-owner 判断修复、降级、回滚或转人工，并记录未验证假设。

实践型面试要点

表达时必须标记经历属于生产实战、个人实验或方案设计，只主张自己实际完成的职责，并用 Trace、评测或变更记录举证。企业差旅案例、示例版本和指标均为教学占位；没有验证线上规模、事故协作或长期效果时要主动说明。

常见问法

在生产或可复现实验中，你如何把“固化评测目标”和“版本化数据集”落到企业差旅 Agent 的 目标、数据集与基线 链路？

为什么问题属于 评测层 而不是相邻层？参数如何从 SLO 或风险基线推导？结果未知时怎样避免扩大副作用？evaluation-owner 不在线时，谁依据什么证据接管？
