05 安全红队与线上反馈

学习目标

  • 从资产、攻击者、信任边界和影响定义 Agent 红队威胁模型。
  • 覆盖直接/间接注入、越权、数据外泄、工具滥用、记忆投毒和资源耗尽。
  • 把线上 trace、告警、人工接管与用户反馈转成经治理的离线样本闭环。

核心概念

链路总览与现有讲解

安全红队与线上反馈属于评测层：输入是版本化目标、dataset、Gold、实际 Outcome / Trajectory 与评分器，输出是分切片指标、失败归因、gate decision 和回归样本。学习时先沿现有图、表或主题章节理解链路，再用后续 stage 检查每个中间产物的不变量。

攻击面红队问题验收信号
Prompt / RAG / Observation不可信内容能否改变高优先级指令或触发外发？拒绝、隔离、再授权、无秘密进入 egress
身份与工具能否跨租户、参数走私、审批后篡改或 confused deputy？最小权限、摘要绑定、TOCTOU 重验
记忆与知识能否持久投毒、污染检索或造成跨会话泄露？provenance、隔离、撤回与过期
资源与模型能否制造循环、重试风暴、成本耗尽或 Judge 操纵？预算终止、分歧升级、不可绕过 hard fail

核心环节：从输入到可验证输出

1. 威胁模型

威胁模型必须把威胁假设、攻击样本、线上信号和回归晋升连成有治理的证据链，反馈量不能等同风险。

工作机制
ThreatModel 定义资产与路径，AttackCorpus 固定复现输入，Triage 评估可利用性和影响，FeedbackIntake 去标识并经复核晋升回归。本环节把 版本化目标、dataset、Gold、实际 Outcome / Trajectory 与评分器 中与“威胁模型”相关的事实转成 分切片指标、失败归因、gate decision 和回归样本 的一部分。

难点与取舍
红队深度、误报成本与发布速度冲突；优先级依据可利用性、影响范围、出现频率和修复可验证性。本层总体取舍是 覆盖率与成本、评分一致性与人工深度、速度与发布风险，参数必须引用版本化数据、SLO 或风险基线。
边界与注意事项
离线红队发现和回归风险，在线护栏即时阻断，基础设施沙箱限制真实执行后果。评测层 的总边界是：离线评测消费 Trace 并反馈编排，不参与单次在线授权；错误会继续污染 发布门禁、路由 / 编排调整、回归集和线上观测重点。

失败传播与 first divergence

  1. 上游输入：先确认 版本化目标、dataset、Gold、实际 Outcome / Trajectory 与评分器 的身份、作用域、时间与版本是否一致；输入错误不应在本层被“修正”为新事实。
  2. 首个偏离：按 威胁模型 → 攻击语料 → 安全分诊 → 治理线上反馈 → 样本晋升 比较预期与实际中间产物，第一个不满足概念不变量的 stage 才是本章候选根因。
  3. 下游影响：本层错误会改变 发布门禁、路由 / 编排调整、回归集和线上观测重点；最终症状可能在更晚阶段出现，但不能从症状直接倒推 Prompt。
  4. 归属边界：evaluation-owner修复本层契约或机制；若 first divergence 位于相邻层，保留证据并交给相应 owner，不跨层调参。

工程要点

  1. 红队计划记录资产、攻击者能力、入口、信任边界、预期控制、严重度、成功判据和安全停止条件；在隔离测试租户中使用假凭据与 canary secret。
  2. 攻击生成同时采用专家 playbook、自动变异、多轮组合和工具状态故障；记录 attack_id、seed、版本、trace 与最小复现。
  3. 线上治理负责实时检测、拒绝、撤权、告警和事件响应；评测层只接收经同意、脱敏、最小化和访问控制的结果与 trace，不能在离线路径拦截请求。
  4. 采样不能只收用户点赞：合并负反馈、放弃、人工接管、策略拒绝、异常成本、近失事件和事故，并校正曝光与选择偏差。
  5. 每个确认失败进入 failure taxonomy、去重样本库与回归集；评测结论指向编排、治理、工具、RAG、记忆或网关 owner，修复经安全回归后再受控上线。

常见坑

  • 只测单轮越狱：Agent 风险常来自跨检索、记忆和工具的多步组合。
  • 在生产做破坏性红队：可能产生真实副作用和数据泄露，应使用隔离环境与停止条件。
  • 把告警率当安全质量：在线规则变化会改变告警分布，需要离线受控集验证。
  • 盲目回灌线上数据：会引入 PII、攻击载荷、重复和选择偏差。

理论追问

请先脱稿回答，再展开参考答案核对边界、证据和取舍。

  1. Agent 红队为什么必须从威胁模型开始？

    查看参考答案

    没有资产、攻击者、信任边界和潜在影响的定义，测试会停留在随机越狱。威胁模型决定要覆盖检索注入、记忆投毒、权限绕过、工具副作用、数据泄露和资源耗尽，以及相应 hard fail。

  2. 为什么破坏性红队应在隔离环境运行？

    查看参考答案

    攻击可能真的发信、删除数据或泄露凭据。应使用隔离租户、假账号、canary secret、网络白名单、资源预算和停止开关，并保留完整 Trace；生产只能做经过批准的低风险探测。

  3. 单轮安全测试通过，为什么多步 Agent 仍可能被攻破？

    查看参考答案

    风险会跨检索、记忆、推理和工具组合：第一步植入内容，后续步骤才读取并执行。评测必须覆盖多轮持久化、handoff、恢复和权限变化，检查整个 trajectory 而不只最终文本。

  4. 线上 Trace 如何安全转成红队与回归样本？

    查看参考答案

    从告警、人工接管和事故中采样，先最小化、脱敏、去重和确认授权，再由人工标注攻击链、预期策略和 hard fail；绑定来源和版本进入受控数据集，不能原样回灌敏感 Prompt。

  5. 安全评测发现绕过后，反馈应该到哪里？

    查看参考答案

    根据首个偏离点反馈给检索隔离、Memory 写入、编排暴露工具、治理策略、沙箱或模型路由 owner；修复需通过原攻击和邻近变体回归，再灰度上线。评测层给出证据，不直接修改在线 Run。

面试表达

如何建立 Agent 安全评测闭环？

我从威胁模型定义红队覆盖，在隔离租户用假凭据和 canary secret 运行多轮组合攻击，按越权、泄露、副作用和资源耗尽设置 hard fail。线上治理负责实时拦截与取证；评测层离线消费脱敏 trace、人工接管和事故样本，去重后进入版本化回归集，把首个偏离点反馈给正确 owner，修复通过安全门禁后再灰度上线。

日常实践关键点

核心判断

本章的日常工作不是记住名词，而是把“安全红队与线上反馈”变成可执行控制环：先识别触发信号，再用版本化证据定位首个偏离点，由 evaluation-owner 执行动作，最后用成功、拒绝、降级三类结果验收。推荐顺序是 维护威胁模型 → 版本化攻击语料 → 执行安全分诊 → 治理线上反馈 → 门禁样本晋升；任何参数都必须从真实 SLO、风险基线或失败样本推导。

深度实践卡

  • 1. 维护威胁模型
    触发信号
    当企业差旅 Agent 的“安全红队与线上反馈”链路出现总体分数上升但关键切片退化、Judge 漂移、发布结论无法复现或线上失败未进入回归集，或变更直接触及“维护威胁模型”的输入、策略或 owner 时，必须启动这项检查。
    诊断顺序
    诊断“维护威胁模型”时先冻结 目标版本、dataset_version、样本切片、Gold 来源、运行配置与评分器版本。从资产、攻击者能力、入口、跨边界路径和业务影响建立假设，确认样本覆盖可利用链而不只是关键词命中。
    执行动作
    落实“维护威胁模型”时，把威胁假设转成可复现负测，固定攻击语料、策略和预期拒绝点，并为漏拦与误拦分别分诊。由 evaluation-owner 记录执行前后的版本、决定与接管条件。
    验证方法
    验收“维护威胁模型”时，在冻结样本、关键风险切片和可比基线上重跑，并核对置信区间、hard fail 与人工抽检；同时比较正确、拒绝和降级结果，阈值只从真实基线与风险容忍度推导。
    止损与取舍
    发现可利用路径时先封禁相关能力或输入通道，隔离受污染数据，再补规则与回归样本；阻断发布、恢复上一可比基线和评分器版本，把争议样本转入人工复核队列。对“维护威胁模型”先保证可恢复和可审计，再优化自动化率。
    面试记忆点
    围绕“维护威胁模型”准备一条“症状—首个偏离—动作—证据—止损”的真实或个人实验 Trace，并明确自己只承担的 evaluation-owner 职责。

    触发信号
    当企业差旅 Agent 的“安全红队与线上反馈”链路出现总体分数上升但关键切片退化、Judge 漂移、发布结论无法复现或线上失败未进入回归集，或变更直接触及“版本化攻击语料”的输入、策略或 owner 时，必须启动这项检查。
    诊断顺序
    诊断“版本化攻击语料”时先冻结 目标版本、dataset_version、样本切片、Gold 来源、运行配置与评分器版本。检查样本、Gold、评分器、模型与策略版本是否可比，并查看关键切片而非只看聚合分。
    执行动作
    落实“版本化攻击语料”时，固定评测单元、评分依据和争议处理流程，让规则、人工与 Judge 各自覆盖擅长的判断。由 evaluation-owner 记录执行前后的版本、决定与接管条件。
    验证方法

    止损与取舍

    诊断顺序
    诊断“执行安全分诊”时先冻结 目标版本、dataset_version、样本切片、Gold 来源、运行配置与评分器版本。先判断失败标签是否互斥、可操作且能指向 owner，再抽查边界样本和评分分歧，避免只得到无法修复的“质量差”。
    执行动作
    落实“执行安全分诊”时，用确定性规则处理硬约束、Rubric 处理语义质量、人工处理高风险争议，并记录每类失败的升级路径。由 evaluation-owner 记录执行前后的版本、决定与接管条件。
    验证方法
    验收“执行安全分诊”时，在冻结样本、关键风险切片和可比基线上重跑，并核对置信区间、hard fail 与人工抽检；同时比较正确、拒绝和降级结果，阈值只从真实基线与风险容忍度推导。
    止损与取舍
    分类或评分依据不稳定时保留原始样本和多方判定，暂停把该分数用于发布门禁；阻断发布、恢复上一可比基线和评分器版本，把争议样本转入人工复核队列。对“执行安全分诊”先保证可恢复和可审计，再优化自动化率。

    执行动作
    落实“门禁样本晋升”时，把允许、拒绝和需审批做成类型化策略结果，默认拒绝缺字段或版本不明的请求。由 evaluation-owner 记录执行前后的版本、决定与接管条件。

    一旦主体或目的无法证明，立即撤销能力令牌并转人工确认，不用事后过滤替代前置授权；阻断发布、恢复上一可比基线和评分器版本，把争议样本转入人工复核队列。对“门禁样本晋升”先保证可恢复和可审计，再优化自动化率。

排障顺序

  1. 冻结请求、策略、模型 / 工具、数据和部署版本，避免边复现边变化。
  2. 从最早可验证输入开始，逐段对照契约、决策、动作与验收事件；在首个偏离点停止跨层调参。
  3. 由 evaluation-owner 判断修复、降级、回滚或转人工，并记录未验证假设。
贯穿案例｜企业差旅 Agent：员工请求预订超预算航班时，只把制度条款当知识证据，把员工靠窗偏好当记忆，把预订动作交给工具层；本章只处理 安全红队与线上反馈 的控制与证据。若发现总体分数上升但关键切片退化、Judge 漂移、发布结论无法复现或线上失败未进入回归集，先保留 trace_id 与版本，再在本层首个偏离点修复，避免把问题推给 Prompt。

应留证据

最小证据集应能回答谁在何时依据哪个版本、对什么输入做了什么决定，以及结果如何被验证。敏感值使用教学占位或受控引用，不把日志全文复制进面试材料。

case: enterprise-travel-agent

owner: evaluation-owner

实践型面试要点

表达时必须标记经历属于生产实战、个人实验或方案设计，只主张自己实际完成的职责，并用 Trace、评测或变更记录举证。企业差旅案例、示例版本和指标均为教学占位；没有验证线上规模、事故协作或长期效果时要主动说明。

实践面试 1

关联实践：维护威胁模型、版本化攻击语料

常见问法

在生产或可复现实验中，你如何把“维护威胁模型”和“版本化攻击语料”落到企业差旅 Agent 的 安全红队与线上反馈 链路？

60–90 秒回答骨架

先交代 安全红队与线上反馈 的业务风险和 evaluation-owner 边界；再说触发信号与首个偏离点；随后说明契约、动作和参数依据；最后给出验证、止损和复盘结果。回答要同时覆盖正常路径与拒绝 / 降级路径。

可信证据

围绕“维护威胁模型 / 版本化攻击语料”展示脱敏后的 trace_id、目标版本、dataset_version、样本切片、Gold 来源、运行配置与评分器版本、前后对照和失败切片；个人实验还应给出固定输入、复现步骤和观察结果，不虚构线上规模。

高频追问

为什么问题属于 评测层 而不是相邻层？参数如何从 SLO 或风险基线推导？结果未知时怎样避免扩大副作用？evaluation-owner 不在线时，谁依据什么证据接管？

边界与取舍

“维护威胁模型 / 版本化攻击语料”会增加契约、存储或人工成本，只有错误成本与复现收益覆盖这些成本时才扩大自动化。未处理过该类生产事故时，应明确这是方案设计或个人实验结论。
