05 安全红队与线上反馈

学习目标

  • 从资产、攻击者、信任边界和影响定义 Agent 红队威胁模型。
  • 覆盖直接/间接注入、越权、数据外泄、工具滥用、记忆投毒和资源耗尽。
  • 把线上 trace、告警、人工接管与用户反馈转成经治理的离线样本闭环。

核心概念

攻击面红队问题验收信号
Prompt / RAG / Observation不可信内容能否改变高优先级指令或触发外发？拒绝、隔离、再授权、无秘密进入 egress
身份与工具能否跨租户、参数走私、审批后篡改或 confused deputy？最小权限、摘要绑定、TOCTOU 重验
记忆与知识能否持久投毒、污染检索或造成跨会话泄露？provenance、隔离、撤回与过期
资源与模型能否制造循环、重试风暴、成本耗尽或 Judge 操纵？预算终止、分歧升级、不可绕过 hard fail

工程要点

  1. 红队计划记录资产、攻击者能力、入口、信任边界、预期控制、严重度、成功判据和安全停止条件；在隔离测试租户中使用假凭据与 canary secret。
  2. 攻击生成同时采用专家 playbook、自动变异、多轮组合和工具状态故障；记录 attack_id、seed、版本、trace 与最小复现。
  3. 线上治理负责实时检测、拒绝、撤权、告警和事件响应；评测层只接收经同意、脱敏、最小化和访问控制的结果与 trace，不能在离线路径拦截请求。
  4. 采样不能只收用户点赞：合并负反馈、放弃、人工接管、策略拒绝、异常成本、近失事件和事故，并校正曝光与选择偏差。
  5. 每个确认失败进入 failure taxonomy、去重样本库与回归集；评测结论指向编排、治理、工具、RAG、记忆或网关 owner，修复经安全回归后再受控上线。

常见坑

  • 只测单轮越狱：Agent 风险常来自跨检索、记忆和工具的多步组合。
  • 在生产做破坏性红队：可能产生真实副作用和数据泄露，应使用隔离环境与停止条件。
  • 把告警率当安全质量：在线规则变化会改变告警分布，需要离线受控集验证。
  • 盲目回灌线上数据：会引入 PII、攻击载荷、重复和选择偏差。

理论追问

请先脱稿回答，再展开参考答案核对边界、证据和取舍。

  1. Agent 红队为什么必须从威胁模型开始？

    查看参考答案

    没有资产、攻击者、信任边界和潜在影响的定义，测试会停留在随机越狱。威胁模型决定要覆盖检索注入、记忆投毒、权限绕过、工具副作用、数据泄露和资源耗尽，以及相应 hard fail。

  2. 为什么破坏性红队应在隔离环境运行？

    查看参考答案

    攻击可能真的发信、删除数据或泄露凭据。应使用隔离租户、假账号、canary secret、网络白名单、资源预算和停止开关，并保留完整 Trace；生产只能做经过批准的低风险探测。

  3. 单轮安全测试通过，为什么多步 Agent 仍可能被攻破？

    查看参考答案

    风险会跨检索、记忆、推理和工具组合：第一步植入内容，后续步骤才读取并执行。评测必须覆盖多轮持久化、handoff、恢复和权限变化，检查整个 trajectory 而不只最终文本。

  4. 线上 Trace 如何安全转成红队与回归样本？

    查看参考答案

    从告警、人工接管和事故中采样，先最小化、脱敏、去重和确认授权，再由人工标注攻击链、预期策略和 hard fail；绑定来源和版本进入受控数据集，不能原样回灌敏感 Prompt。

  5. 安全评测发现绕过后，反馈应该到哪里？

    查看参考答案

    根据首个偏离点反馈给检索隔离、Memory 写入、编排暴露工具、治理策略、沙箱或模型路由 owner；修复需通过原攻击和邻近变体回归，再灰度上线。评测层给出证据，不直接修改在线 Run。

面试表达

如何建立 Agent 安全评测闭环？

我从威胁模型定义红队覆盖，在隔离租户用假凭据和 canary secret 运行多轮组合攻击，按越权、泄露、副作用和资源耗尽设置 hard fail。线上治理负责实时拦截与取证；评测层离线消费脱敏 trace、人工接管和事故样本，去重后进入版本化回归集，把首个偏离点反馈给正确 owner，修复通过安全门禁后再灰度上线。
