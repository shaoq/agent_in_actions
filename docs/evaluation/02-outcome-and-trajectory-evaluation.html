02 结果评测与轨迹评测

学习目标

  • 区分 outcome、trajectory、process compliance 与 operational efficiency。
  • 从 trace 构造可重放、可归因的步骤级评分证据。
  • 把最终质量与路径安全组合成不会互相掩盖的指标体系。

核心概念

维度典型指标证据典型失败
结果正确性exact/semantic match、事实性、业务验收最终响应、外部系统状态、引用答案流畅但事实错
任务完成度约束满足、步骤覆盖、拒绝恰当性验收器、用户目标与终态部分完成却声明 finish
轨迹质量有效步骤率、工具选择、证据使用span、decision、tool call/observation碰巧答对但路径不可复现
安全与效率越权率、冗余调用、成本、延迟policy decision、token、时间、重试正确结果伴随未授权副作用

工程要点

  1. trace schema 至少关联 run_id、sample_id、step、state_version、model/prompt version、retrieval、decision、tool call、policy、observation、cost 与 terminal reason。
  2. 先定义关键事件与允许路径，不要求模型逐字复现“金色轨迹”；Agent 可有多条合法路径，评分应关注不变量和必要证据。
  3. 结果分与轨迹分分别报告，并为越权、泄露、错误副作用等设置 hard fail，禁止高结果分抵消安全失败。
  4. 失败归因采用“首个偏离点 + 传播链 + owner”：数据、检索、推理、编排、工具、治理或模型网关，不把一切归因于模型。
  5. 评测层读取治理与可观测层产出的脱敏 trace；它可建议调整路由、重试或完成判定，但由编排层实施策略。

常见坑

  • 只评最终文本：看不到不必要工具调用、越权尝试和错误补偿。
  • 金色轨迹过度约束：把合理替代路径误判为失败，抑制 Agent 自主性。
  • trace 不完整：缺少版本与 observation 时无法复现因果。
  • 平均分掩盖 hard fail：少量安全事故不能被大量简单样本稀释。

面试表达

为什么 Agent 需要轨迹评测？

因为 Agent 的价值和风险都发生在多步决策与动作中。最终结果可能碰巧正确，但中间存在未授权调用、错误检索或重试风暴。我会让离线评测消费结构化 trace，分别计算 outcome 与 trajectory 指标，以必要不变量而非唯一金色路径评分；安全失败设 hard fail，再把首个偏离点归因到正确组件并反馈编排策略。
