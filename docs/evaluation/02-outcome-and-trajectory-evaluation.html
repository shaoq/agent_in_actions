02 结果评测与轨迹评测

学习目标

  • 区分 outcome、trajectory、process compliance 与 operational efficiency。
  • 从 trace 构造可重放、可归因的步骤级评分证据。
  • 把最终质量与路径安全组合成不会互相掩盖的指标体系。

核心概念

链路总览与现有讲解

结果与轨迹评测属于评测层：输入是版本化目标、dataset、Gold、实际 Outcome / Trajectory 与评分器，输出是分切片指标、失败归因、gate decision 和回归样本。学习时先沿现有图、表或主题章节理解链路，再用后续 stage 检查每个中间产物的不变量。

维度典型指标证据典型失败
结果正确性exact/semantic match、事实性、业务验收最终响应、外部系统状态、引用答案流畅但事实错
任务完成度约束满足、步骤覆盖、拒绝恰当性验收器、用户目标与终态部分完成却声明 finish

1. 定义 Outcome 指标

定义 Outcome 指标必须区分最终 Outcome 与达到结果的 Trajectory，并把得分和失败归到可修复的具体步骤。

职责与 I/O
evaluation-owner 在“定义 Outcome 指标”环节接收 版本化目标、dataset、Gold、实际 Outcome / Trajectory 与评分器，产出 分切片指标、失败归因、gate decision 和回归样本；输出必须能由下游按版本消费。
工作机制
规范化事件形成 expected / actual trajectory，规则对齐关键状态与动作，归因器定位 first divergence，Rubric 为安全的部分进展给分。本环节把 版本化目标、dataset、Gold、实际 Outcome / Trajectory 与评分器 中与“定义 Outcome 指标”相关的事实转成 分切片指标、失败归因、gate decision 和回归样本 的一部分。
必须成立的约束
事件顺序、终态、工具副作用和验收必须独立；部分得分不能抵消 hard fail，失败标签需互斥且指向 owner。本章共同约束是：样本、Gold、系统、Judge 与策略版本必须可比；违反时必须产生可区分的拒绝、未知或升级结果。
难点与取舍
只看 Outcome 会漏掉侥幸成功，只看轨迹会惩罚合理替代路径；容许集合由业务等价性和人工复核校准。本层总体取舍是 覆盖率与成本、评分一致性与人工深度、速度与发布风险，参数必须引用版本化数据、SLO 或风险基线。
边界与注意事项
评测解释已有 Trace，不要求暴露模型私有思维，也不把编排差异一律归因给模型。评测层 的总边界是：离线评测消费 Trace 并反馈编排，不参与单次在线授权；错误会继续污染 发布门禁、路由 / 编排调整、回归集和线上观测重点。

失败传播与 first divergence

  1. 上游输入：先确认 版本化目标、dataset、Gold、实际 Outcome / Trajectory 与评分器 的身份、作用域、时间与版本是否一致；输入错误不应在本层被“修正”为新事实。
  2. 首个偏离：按 定义 Outcome 指标 → 规范轨迹事件 → 步骤归因 → 部分得分 → 失败分类 比较预期与实际中间产物，第一个不满足概念不变量的 stage 才是本章候选根因。
  3. 下游影响：本层错误会改变 发布门禁、路由 / 编排调整、回归集和线上观测重点；最终症状可能在更晚阶段出现，但不能从症状直接倒推 Prompt。
  4. 归属边界：evaluation-owner修复本层契约或机制；若 first divergence 位于相邻层，保留证据并交给相应 owner，不跨层调参。

工程要点

  1. trace schema 至少关联 run_id、sample_id、step、state_version、model/prompt version、retrieval、decision、tool call、policy、observation、cost 与 terminal reason。
  2. 先定义关键事件与允许路径，不要求模型逐字复现“金色轨迹”；Agent 可有多条合法路径，评分应关注不变量和必要证据。
  3. 结果分与轨迹分分别报告，并为越权、泄露、错误副作用等设置 hard fail，禁止高结果分抵消安全失败。
  4. 失败归因采用“首个偏离点 + 传播链 + owner”：数据、检索、推理、编排、工具、治理或模型网关，不把一切归因于模型。
  5. 评测层读取治理与可观测层产出的脱敏 trace；它可建议调整路由、重试或完成判定，但由编排层实施策略。

常见坑

  • 只评最终文本：看不到不必要工具调用、越权尝试和错误补偿。
  • 金色轨迹过度约束：把合理替代路径误判为失败，抑制 Agent 自主性。
  • trace 不完整：缺少版本与 observation 时无法复现因果。
  • 平均分掩盖 hard fail：少量安全事故不能被大量简单样本稀释。

理论追问

请先脱稿回答，再展开参考答案核对边界、证据和取舍。

  1. 为什么最终结果正确仍可能评测失败？

    查看参考答案

    Agent 可能碰巧答对，却经历越权工具调用、泄露数据、重复副作用或高成本循环。Outcome 衡量做成什么，Trajectory 衡量如何做成；安全和关键流程不变量应设 hard fail，不能被最终正确抵消。

  2. 为什么不应要求所有成功 Run 完全复制一条“黄金轨迹”？

    查看参考答案

    Agent 可能有多条合法路径，过度固定步骤会把合理探索误判为错误。应定义必须满足的不变量、禁止动作、必要证据和终态，让评测容许不同顺序，同时检查首个偏离点和效率。

  3. 哪些轨迹问题适合 hard fail？

    查看参考答案

    未授权访问、敏感数据泄露、错误不可逆副作用、绕过审批、伪造工具结果和把 effect unknown 当成功都应 hard fail。冗余步骤或较高成本可作为连续分数，但不能用平均质量掩盖安全事故。

  4. 如何用“首个偏离点”做失败归因？

    查看参考答案

    按 State、decision、retrieval、policy、tool、Observation 和 acceptance 的顺序与不变量比较，找到第一次出现错误输入、错误选择或缺失证据的位置，再追踪传播链。这样可把修复反馈给正确组件而不是一律改 Prompt。

  5. 支持轨迹评测的 Trace 最少要包含什么？

    查看参考答案

    需要 run/sample ID、step 与 state version、模型/Prompt、检索候选、结构化决策、policy decision、tool call/Observation、成本、终止原因和业务验收。敏感内容可脱敏，但版本和因果关联不能丢。

面试表达

为什么 Agent 需要轨迹评测？

因为 Agent 的价值和风险都发生在多步决策与动作中。最终结果可能碰巧正确，但中间存在未授权调用、错误检索或重试风暴。我会让离线评测消费结构化 trace，分别计算 outcome 与 trajectory 指标，以必要不变量而非唯一金色路径评分；安全失败设 hard fail，再把首个偏离点归因到正确组件并反馈编排策略。

日常实践关键点

核心判断

本章的日常工作不是记住名词，而是把“结果与轨迹评测”变成可执行控制环：先识别触发信号，再用版本化证据定位首个偏离点，由 evaluation-owner 执行动作，最后用成功、拒绝、降级三类结果验收。推荐顺序是 定义 Outcome 指标 → 规范轨迹事件 → 执行步骤归因 → 设计部分得分 → 维护失败分类；任何参数都必须从真实 SLO、风险基线或失败样本推导。

排障顺序

  1. 冻结请求、策略、模型 / 工具、数据和部署版本，避免边复现边变化。
  2. 从最早可验证输入开始，逐段对照契约、决策、动作与验收事件；在首个偏离点停止跨层调参。
  3. 由 evaluation-owner 判断修复、降级、回滚或转人工，并记录未验证假设。
