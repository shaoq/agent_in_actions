04 回归测试与上线门禁

学习目标

  • 控制模型、prompt、工具、知识、数据和随机性，进行公平版本比较。
  • 以效应大小、置信区间和关键切片定义分层门禁。
  • 连接离线门禁、灰度验证、在线监测与回滚。

核心概念

链路总览与现有讲解

回归测试与发布门禁属于评测层：输入是版本化目标、dataset、Gold、实际 Outcome / Trajectory 与评分器，输出是分切片指标、失败归因、gate decision 和回归样本。学习时先沿现有图、表或主题章节理解链路，再用后续 stage 检查每个中间产物的不变量。

门禁层例子决策
硬约束越权、泄露、错误副作用、审计缺失 = 0 容忍任一失败即阻断
关键切片高金额、多租户、长链、非中文、工具失败每个切片达到下限
总体质量任务成功、事实性、轨迹效率、成本、延迟无显著退化且目标指标提升
线上灰度SLO、人工接管、拒绝、事故与业务 KPI逐步放量或自动回滚

核心环节：从输入到可验证输出

1. 回归样本

回归样本必须把指标变化转成预先声明的发布、观察或回滚决定，关键 hard fail 不得被平均提升抵消。

职责与 I/O
evaluation-owner 在“回归样本”环节接收 版本化目标、dataset、Gold、实际 Outcome / Trajectory 与评分器，产出 分切片指标、失败归因、gate decision 和回归样本；输出必须能由下游按版本消费。
工作机制
RegressionSelection 选稳定集与风险集，Threshold 绑定基线波动，Shadow 收集无副作用对照，Canary 限流量验证并由 Gate 产出决定。本环节把 版本化目标、dataset、Gold、实际 Outcome / Trajectory 与评分器 中与“回归样本”相关的事实转成 分切片指标、失败归因、gate decision 和回归样本 的一部分。
必须成立的约束
候选与基线需同数据 / 策略条件；放量、观察窗口、退出阈值和回滚版本在发布前固定。本章共同约束是：样本、Gold、系统、Judge 与策略版本必须可比；违反时必须产生可区分的拒绝、未知或升级结果。
难点与取舍
门禁过严降低迭代速度，过松把风险推线上；阈值依据历史方差、样本量、业务损失和恢复时间校准。本层总体取舍是 覆盖率与成本、评分一致性与人工深度、速度与发布风险，参数必须引用版本化数据、SLO 或风险基线。
边界与注意事项
评测层给出 gate evidence，发布系统执行流量切换，线上治理仍逐请求判定风险。评测层 的总边界是：离线评测消费 Trace 并反馈编排，不参与单次在线授权；错误会继续污染 发布门禁、路由 / 编排调整、回归集和线上观测重点。

失败传播与 first divergence

  1. 上游输入：先确认 版本化目标、dataset、Gold、实际 Outcome / Trajectory 与评分器 的身份、作用域、时间与版本是否一致；输入错误不应在本层被“修正”为新事实。
  2. 首个偏离：按 回归样本 → 校准指标阈值 → Hard Fail → Shadow / Canary → 回滚决定 比较预期与实际中间产物，第一个不满足概念不变量的 stage 才是本章候选根因。
  3. 下游影响：本层错误会改变 发布门禁、路由 / 编排调整、回归集和线上观测重点；最终症状可能在更晚阶段出现，但不能从症状直接倒推 Prompt。
  4. 归属边界：evaluation-owner修复本层契约或机制；若 first divergence 位于相邻层，保留证据并交给相应 owner，不跨层调参。

工程要点

  1. 每次 run 保存 change_id、代码、模型、prompt、工具/schema、知识快照、dataset、judge 与环境版本，支持一键重放。
  2. 同一样本对候选与基线做 paired comparison；非确定系统重复运行，报告差值、效应大小、置信区间和样本量。
  3. 门禁写成机器可判定契约：hard fail、总体非劣界、目标提升、每个关键切片下限、成本/延迟预算与最大 Judge 分歧。
  4. 新增故障必须先转成去敏回归样本；修复某切片时同时运行全量集，防止局部优化破坏其他能力。
  5. 离线结论反馈编排选择候选策略；上线由编排/发布系统执行，小流量阶段由在线治理与可观测监测并触发回滚。

常见坑

  • 只比较平均值：随机波动或关键切片退化会被隐藏。
  • 候选和基线不是同一快照：环境漂移使结论失真。
  • 门槛随结果移动：先看分数再定标准会制造通过。
  • 离线通过直接全量：评测集永远不能完整覆盖真实分布。

理论追问

请先脱稿回答，再展开参考答案核对边界、证据和取舍。

  1. 新版本平均分提高，为什么仍可能不能发布？

    查看参考答案

    关键安全切片、长尾任务或高价值客户可能退化，平均值会被大量简单样本稀释。发布门禁应同时检查总体差异、置信区间、关键切片、hard fail、成本和延迟预算。

  2. 如何区分真实回归和随机波动？

    查看参考答案

    使用固定版本数据与环境，多次重复或配对比较，报告置信区间和效应大小；检查失败样本是否集中于稳定切片。门槛附近需要更多样本或人工复核，不能只比较一个小数点。

  3. 发布门禁应有哪些层级？

    查看参考答案

    先做 schema、单元与安全不变量等快速 hard gate，再跑代表性结果和轨迹回归，随后灰度监测线上 SLO 与切片；任一关键约束失败即阻断或回滚。不同风险变更可使用不同门槛。

  4. 灰度阶段发现人工接管率上升，应如何处理？

    查看参考答案

    先按任务、模型、工具和错误类型切片，判断是质量退化还是新策略更谨慎。若超过预设回滚阈值立即停止扩量，保留 Trace，修复后用新增样本复测；不能只看总体成功率仍高就忽略。

  5. 为什么 Prompt、模型、工具和编排变更都要走同一回归契约？

    查看参考答案

    它们都可能改变最终结果和轨迹，且失败会跨组件传播。统一 runner、数据、指标和门禁才能比较版本并正确归因；只给模型升级做评测会漏掉工具 schema 或重试策略引起的回归。

面试表达

如何设计 Agent 上线门禁？

我把门禁分成硬安全约束、关键切片、总体质量和线上灰度四层。候选与基线在同一版本化环境和数据上配对重复运行，报告差值与置信区间；越权等 hard fail 零容忍，每个高风险切片有独立下限，总体需满足非劣与成本预算。离线通过后只小流量上线，由在线监测决定放量或回滚。

日常实践关键点

核心判断

本章的日常工作不是记住名词，而是把“回归测试与发布门禁”变成可执行控制环：先识别触发信号，再用版本化证据定位首个偏离点，由 evaluation-owner 执行动作，最后用成功、拒绝、降级三类结果验收。推荐顺序是 选择回归样本 → 校准指标阈值 → 设置 Hard Fail → 执行 Shadow / Canary → 记录回滚决定；任何参数都必须从真实 SLO、风险基线或失败样本推导。

    执行动作
    落实“选择回归样本”时，把允许、拒绝和需审批做成类型化策略结果，默认拒绝缺字段或版本不明的请求。由 evaluation-owner 记录执行前后的版本、决定与接管条件。
    验证方法
    验收“选择回归样本”时，在冻结样本、关键风险切片和可比基线上重跑，并核对置信区间、hard fail 与人工抽检；同时比较正确、拒绝和降级结果，阈值只从真实基线与风险容忍度推导。
    止损与取舍
    一旦主体或目的无法证明，立即撤销能力令牌并转人工确认，不用事后过滤替代前置授权；阻断发布、恢复上一可比基线和评分器版本，把争议样本转入人工复核队列。对“选择回归样本”先保证可恢复和可审计，再优化自动化率。
    面试记忆点
    围绕“选择回归样本”准备一条“症状—首个偏离—动作—证据—止损”的真实或个人实验 Trace，并明确自己只承担的 evaluation-owner 职责。

    诊断顺序

    执行动作
    落实“校准指标阈值”时，从用户 SLO 和风险容忍度反推预算与阈值，记录样本窗口、分位数和参数依据。由 evaluation-owner 记录执行前后的版本、决定与接管条件。

    止损与取舍
    触线后按预定顺序限流、降级或拒绝，不能临时放宽安全与质量门槛换吞吐；阻断发布、恢复上一可比基线和评分器版本，把争议样本转入人工复核队列。对“校准指标阈值”先保证可恢复和可审计，再优化自动化率。

  • 3. 设置 Hard Fail
    触发信号

    诊断顺序
    诊断“设置 Hard Fail”时先冻结 目标版本、dataset_version、样本切片、Gold 来源、运行配置与评分器版本。分别核对技术执行、策略合规和业务验收，避免把 HTTP 成功、结构合法或模型自报完成当成最终成功。
    执行动作
    落实“设置 Hard Fail”时，定义可机器判断的完成条件、hard fail、证据来源和验收 owner；证据不全时返回 unknown / pending 而不是 passed。由 evaluation-owner 记录执行前后的版本、决定与接管条件。
    验证方法
    验收“设置 Hard Fail”时，在冻结样本、关键风险切片和可比基线上重跑，并核对置信区间、hard fail 与人工抽检；同时比较正确、拒绝和降级结果，阈值只从真实基线与风险容忍度推导。
    止损与取舍
    任一硬门禁失败就保留上一稳定版本；仅软指标退化时才允许受控试运行并持续观察；阻断发布、恢复上一可比基线和评分器版本，把争议样本转入人工复核队列。对“设置 Hard Fail”先保证可恢复和可审计，再优化自动化率。

    诊断顺序
    诊断“执行 Shadow / Canary”时先冻结 目标版本、dataset_version、样本切片、Gold 来源、运行配置与评分器版本。核对路由特征或实验分流是否稳定、可解释且未混入结果标签，按租户、风险和能力切片比较新旧路径。
    执行动作
    落实“执行 Shadow / Canary”时，先影子运行收集对照，再小流量 Canary；把硬能力约束放在打分前，并记录 route_reason、样本分配和退出阈值。由 evaluation-owner 记录执行前后的版本、决定与接管条件。

    关键切片或 hard fail 越界时立即归零新路径流量，恢复固定默认路由并保留对照 Trace；阻断发布、恢复上一可比基线和评分器版本，把争议样本转入人工复核队列。对“执行 Shadow / Canary”先保证可恢复和可审计，再优化自动化率。

    诊断顺序
    诊断“记录回滚决定”时先冻结 目标版本、dataset_version、样本切片、Gold 来源、运行配置与评分器版本。先区分可重试瞬时错误、确定性拒绝和结果未知，再检查副作用是否已有完成凭证。
    执行动作
    落实“记录回滚决定”时，以业务动作和作用域生成幂等键，设置尝试次数、累计 deadline、退避和补偿 owner。由 evaluation-owner 记录执行前后的版本、决定与接管条件。
    验证方法
    验收“记录回滚决定”时，在冻结样本、关键风险切片和可比基线上重跑，并核对置信区间、hard fail 与人工抽检；同时比较正确、拒绝和降级结果，阈值只从真实基线与风险容忍度推导。
    止损与取舍
    结果未知时先查 ledger 而不是盲目重放；无法补偿的动作必须停在人工验收点；阻断发布、恢复上一可比基线和评分器版本，把争议样本转入人工复核队列。对“记录回滚决定”先保证可恢复和可审计，再优化自动化率。
    面试记忆点
    围绕“记录回滚决定”准备一条“症状—首个偏离—动作—证据—止损”的真实或个人实验 Trace，并明确自己只承担的 evaluation-owner 职责。

排障顺序

  1. 冻结请求、策略、模型 / 工具、数据和部署版本，避免边复现边变化。
  2. 从最早可验证输入开始，逐段对照契约、决策、动作与验收事件；在首个偏离点停止跨层调参。
  3. 由 evaluation-owner 判断修复、降级、回滚或转人工，并记录未验证假设。
贯穿案例｜企业差旅 Agent：员工请求预订超预算航班时，只把制度条款当知识证据，把员工靠窗偏好当记忆，把预订动作交给工具层；本章只处理 回归测试与发布门禁 的控制与证据。若发现总体分数上升但关键切片退化、Judge 漂移、发布结论无法复现或线上失败未进入回归集，先保留 trace_id 与版本，再在本层首个偏离点修复，避免把问题推给 Prompt。

应留证据

最小证据集应能回答谁在何时依据哪个版本、对什么输入做了什么决定，以及结果如何被验证。敏感值使用教学占位或受控引用，不把日志全文复制进面试材料。

代表性工程产物｜artifact-eval-04-release-gate
case: enterprise-travel-agent
component: evaluation
unit: 04
owner: evaluation-owner
input_version: teaching-v1
policy_or_contract_version: teaching-v3
decision: allow | deny | degrade | escalate
first_divergence: stage-name
evidence_refs: [trace-demo, eval-slice-demo]
validation: passed | failed | unknown

以上字段是学习用数据形状，不是生产默认值。实际产物还应记录时间、租户 / 主体作用域、变更来源和保留策略，并能从 evidence_refs 回放本章结论。

实践型面试要点

表达时必须标记经历属于生产实战、个人实验或方案设计，只主张自己实际完成的职责，并用 Trace、评测或变更记录举证。企业差旅案例、示例版本和指标均为教学占位；没有验证线上规模、事故协作或长期效果时要主动说明。
