01 角色、职责与决策契约

学习目标

  • 用“输入—决策—输出—不变量”定义推理内核，而不是用“调用 LLM 的模块”定义它。
  • 区分候选决策、授权动作、执行结果和任务终态。
  • 设计可演进的 DecisionContext 与 Decision 判别联合。
  • 说明为什么推理内核必须经模型网关访问模型、由编排层驱动生命周期。

链路总览与现有讲解

角色、职责与决策契约属于推理内核：输入是最小 DecisionContext、证据引用、工具候选与风险约束，输出是类型化候选 Decision 或 ReasoningFailure。学习时先沿现有图、表或主题章节理解链路，再用后续 stage 检查每个中间产物的不变量。

核心概念：决策面而不是控制面

编排层问的是“当前 run 处于什么状态，下一轮是否允许发生”；推理内核回答的是“基于当前上下文，最合适的下一步意图是什么”。前者拥有控制权，后者提供受约束的决策候选。

一个可工程化的推理内核可以写成纯函数式契约：

decide(DecisionContext) -> Decision | ReasoningFailure

DecisionContext = 当前目标 + 已验证状态 + 观察摘要 + 可用能力 + 策略约束 + 决策预算
Decision        = act | respond | clarify | escalate | finish

“纯函数式”是语义目标，不要求实现完全无副作用；它要求相同证据和约束下的决策可被重放、比较和审计，外部动作不得藏在 decide 内部。

输入契约：只给决策所需的上下文

字段组内容工程约束反例
goal用户目标、业务成功条件、当前子目标区分原始请求和规范化目标；保留来源只有一段越来越长的 messages
state当前阶段、已完成步骤、剩余预算、等待条件由编排层提供只读快照并带 schema version允许模型自行宣布 run 已完成
observations工具、检索、记忆或用户返回的已验证观察带来源、时间、可信度和截断说明把工具错误伪装成正常文本
capabilities当前允许选择的能力名称、用途和参数摘要最小暴露；能力可见不等于动作已获授权把生产所有工具完整注入每次请求
constraints禁止项、风险级别、输出要求、引用要求机器可校验约束优先于自然语言提醒只写“请安全地处理”
budget剩余步骤、token、延迟、成本或候选数推理可据此选策略，编排负责真正计量和停止让模型自行维护真实预算

上下文不是越多越好。无关历史会放大延迟、成本、敏感信息暴露与注意力稀释。编排层应先做确定性筛选和压缩，并明确摘要丢失了什么。

输出契约：判别联合与共同字段

{
  "kind": "act | respond | clarify | escalate | finish",
  "decision_id": "stable-id",
  "summary": "面向审计的简短决策摘要",
  "evidence_refs": ["observation:search-7"],
  "confidence": "high | medium | low",
  "assumptions": ["客户身份已由接入层验证"],
  "payload": {}
}

共同字段用于回答“决定了什么、依据什么、还有什么假设”，但不应保存完整隐式思维过程。不同 kind 的 payload 必须互斥：

  • act：能力 ID、结构化参数、期望观察；只是动作意图，不是授权。
  • respond：可交付回答、引用需求、已知限制；不代表 run 一定结束。
  • clarify：阻塞缺口、最小必要问题、不同答案会改变什么。
  • escalate：无法安全自主处理的原因、需要的角色、已完成证据。
  • finish：候选结果、成功条件证据、未满足项；业务 acceptance 仍由编排或业务规则判断。

必须保持的不变量

  1. 决策不等于执行：act 不能绕过治理校验直接触发副作用。
  2. 声明完成不等于验收通过：finish 是候选终止意图，编排层依据业务 acceptance 决定终态。
  3. 上下文只读：推理内核不能直接修改 run state；状态改变由编排提交事件。
  4. 模型访问隔离：推理只表达所需能力、上下文窗和结构化输出要求，由模型网关选择具体模型。
  5. 失败类型化：解析错误、策略冲突、上下文不足和模型不可用不能都退化成“模型回答不好”。

核心环节：从输入到可验证输出

1. 决策上下文

决策上下文必须让同一版本输入得到可解释、可校验的候选分支，不能依赖模型私有上下文补全控制事实。

职责与 I/O
reasoning-owner 在“决策上下文”环节接收 最小 DecisionContext、证据引用、工具候选与风险约束，产出 类型化候选 Decision 或 ReasoningFailure；输出必须能由下游按版本消费。
工作机制
DecisionContext 只携带本步证据；判别联合把 answer、tool、clarify、abstain 等分支及共同引用显式化。本环节把 最小 DecisionContext、证据引用、工具候选与风险约束 中与“决策上下文”相关的事实转成 类型化候选 Decision 或 ReasoningFailure 的一部分。
必须成立的约束
每个分支必须互斥且字段完备，输入哈希、schema、模型与策略版本要能支持回放。本章共同约束是：结论必须受 schema、可见证据和风险约束限制；违反时必须产生可区分的拒绝、未知或升级结果。
难点与取舍
上下文过少会漏证据，过多会引入噪声与注入面；取舍由失败切片和消融实验而非 token 上限猜测决定。本层总体取舍是 上下文充分性与噪声、推理质量与延迟成本，参数必须引用版本化数据、SLO 或风险基线。
边界与注意事项
决策回放解释可见输入与输出，不记录或暴露模型私有思维过程，也不替代编排执行 Trace。推理内核 的总边界是：只提出候选决策，不拥有流程推进、授权或动作执行权；错误会继续污染 编排路由、工具选择、澄清和评测中的决策质量。

3. 决策证据

决策证据必须能把结论追溯到同一请求的输入、版本、决定、动作和验收事件，而不是依赖不可检索的日志文本。

职责与 I/O
reasoning-owner 在“决策证据”环节接收 最小 DecisionContext、证据引用、工具候选与风险约束，产出 类型化候选 Decision 或 ReasoningFailure；输出必须能由下游按版本消费。
工作机制
correlation_id / trace_id 串联 span 与结构化事件，引用保留原始证据位置、policy / model / tool 版本和 owner。本环节把 最小 DecisionContext、证据引用、工具候选与风险约束 中与“决策证据”相关的事实转成 类型化候选 Decision 或 ReasoningFailure 的一部分。
必须成立的约束
时间、因果顺序、采样与脱敏策略必须明确；高基数字段不能无界进入 Metric，敏感原文不能直接写 Log。本章共同约束是：结论必须受 schema、可见证据和风险约束限制；违反时必须产生可区分的拒绝、未知或升级结果。
难点与取舍
全量遥测利于归因却增加成本与隐私面；采样按风险与失败类型分层，关键审计事件不得随机丢弃。本层总体取舍是 上下文充分性与噪声、推理质量与延迟成本，参数必须引用版本化数据、SLO 或风险基线。
边界与注意事项
Trace 解释系统事件与可见证据，不保存模型私有推理；Metric 告警不能替代逐请求审计。推理内核 的总边界是：只提出候选决策，不拥有流程推进、授权或动作执行权；错误会继续污染 编排路由、工具选择、澄清和评测中的决策质量。

4. 分层校验决策

分层校验决策必须区分结构有效、动作完成、策略合规和业务验收；缺证据时只能是 pending 或 unknown。

职责与 I/O
reasoning-owner 在“分层校验决策”环节接收 最小 DecisionContext、证据引用、工具候选与风险约束，产出 类型化候选 Decision 或 ReasoningFailure；输出必须能由下游按版本消费。
工作机制
每类成功由不同 owner 和证据判定，hard fail 先于综合评分，发布 / 完成只消费版本化验收结果。本环节把 最小 DecisionContext、证据引用、工具候选与风险约束 中与“分层校验决策”相关的事实转成 类型化候选 Decision 或 ReasoningFailure 的一部分。
必须成立的约束
完成条件、拒绝条件、证据引用和验收 owner 必须可机器判断；模型自报或 HTTP 2xx 不能单独关单。本章共同约束是：结论必须受 schema、可见证据和风险约束限制；违反时必须产生可区分的拒绝、未知或升级结果。
难点与取舍
严格门禁降低错误发布却可能延长流程；软指标观察窗口和 hard fail 范围由风险与历史波动确定。本层总体取舍是 上下文充分性与噪声、推理质量与延迟成本，参数必须引用版本化数据、SLO 或风险基线。
边界与注意事项
工具证明动作结果，治理证明允许，业务 owner 证明价值；编排只聚合这些证据推进终态。推理内核 的总边界是：只提出候选决策，不拥有流程推进、授权或动作执行权；错误会继续污染 编排路由、工具选择、澄清和评测中的决策质量。

难点与取舍

冲突目标参数依据决策边界
上下文充分性与噪声、推理质量与延迟成本使用版本化回放、失败切片、SLO 分位数和风险等级，不把教学值当生产默认。reasoning-owner只调整本层机制；跨层变化需要对应 owner 与独立验收。
自动化收益 vs. 失败放大比较开启 / 关闭“决策上下文”与“决策回放”的消融结果、人工负载和恢复时间。证据不足返回 pending / unknown / escalate，不能靠更长 Prompt 隐藏边界。

失败传播与 first divergence

  1. 上游输入：先确认 最小 DecisionContext、证据引用、工具候选与风险约束 的身份、作用域、时间与版本是否一致；输入错误不应在本层被“修正”为新事实。
  2. 首个偏离：按 决策上下文 → Decision 判别联合 → 决策证据 → 分层校验决策 → 决策回放 比较预期与实际中间产物，第一个不满足概念不变量的 stage 才是本章候选根因。
  3. 下游影响：本层错误会改变 编排路由、工具选择、澄清和评测中的决策质量；最终症状可能在更晚阶段出现，但不能从症状直接倒推 Prompt。
  4. 归属边界：reasoning-owner修复本层契约或机制；若 first divergence 位于相邻层，保留证据并交给相应 owner，不跨层调参。

概念—实践—证据回扣

核心区回答“是什么、为何成立、怎样传播”，实践区回答“何时触发、如何处置、怎样验收”。每个 stage 与 practice 使用稳定锚点双向连接：

  • 决策上下文的不变量或约束失效时，进入对应日常实践；用本页工程产物、Trace / 评测切片和拒绝 / 降级结果证明是否恢复。
  • Decision 判别联合的不变量或约束失效时，进入对应日常实践；用本页工程产物、Trace / 评测切片和拒绝 / 降级结果证明是否恢复。
  • 决策证据的不变量或约束失效时，进入对应日常实践；用本页工程产物、Trace / 评测切片和拒绝 / 降级结果证明是否恢复。
  • 分层校验决策的不变量或约束失效时，进入对应日常实践；用本页工程产物、Trace / 评测切片和拒绝 / 降级结果证明是否恢复。
  • 决策回放的不变量或约束失效时，进入对应日常实践；用本页工程产物、Trace / 评测切片和拒绝 / 降级结果证明是否恢复。

工程要点

  • 为 DecisionContext 和 Decision 独立版本化；新增可选字段优先，破坏性变化需要迁移或双读。
  • 记录 decision_id、上下文指纹、契约版本、模型请求 ID 和证据引用，支持离线重放。
  • 将领域业务规则提取为确定性 validator；模型负责开放判断，程序负责可判定约束。
  • 对工具可见性做能力裁剪，推理只看到当前身份、阶段和策略允许考虑的候选集合。
  • 把拒绝、澄清、升级设计成一等输出，避免用空回答、异常或无限重试隐式表达。

  1. 为什么推理内核是“决策面”而不是“控制面”？

    查看参考答案

    推理内核根据目标、状态、Observation 和约束生成下一步意图，但不持有持久 Run State，也不决定状态迁移、权限、重试和终止。控制面属于编排层；这种分离使推理可替换、决策可校验、失败可归因。

  2. 交给推理内核的最小输入应包含什么，为什么不能直接塞入全部历史？

    查看参考答案

    最小输入应包含当前目标与约束、相关 State、经权限过滤的证据、可用动作契约和预算。全量历史会引入无关信息、敏感数据、过期事实和提示注入，还会增加成本；上下文应由编排按本步决策目的构造。

  3. “建议调用工具”和“已经执行工具”之间有哪些责任边界？

    查看参考答案

    推理输出只能是结构化 action proposal。编排随后校验 schema、业务前置、授权、审批和预算，再交给工具层及沙箱执行；工具 Observation 回写 State 后才形成事实。模型不能把建议或预测的结果当作已执行结果。

  4. 既要诊断决策错误，又不保存模型私有思维过程，Trace 应记录什么？

    查看参考答案

    记录决策输入的版本与引用、候选动作、最终结构化意图、置信或证据充分性、模型与 Prompt 版本、校验结果、延迟和成本即可。不要依赖或持久化私有 chain-of-thought；诊断需要可观察契约和外部证据。

  5. 模型返回 finish，但业务验收器失败时，下一步由谁决定？

    查看参考答案

    编排把验收失败作为 Observation 写回 State，并按错误类别决定补充证据、重试某一步、失败或转人工。推理内核可以提出修复候选，但无权覆盖验收事实；Trace 要区分 model_finish 与 business_accepted。

面试表达

30 秒回答

推理内核是 Agent 的决策面：它消费编排层准备的目标、状态、观察和约束，形成 act、respond、clarify、escalate 或 finish 等结构化意图。它不拥有 run 生命周期、不直接执行工具，也不直连模型厂商。生产设计重点是输入输出契约、证据引用、不确定性和失败类型，而不是把一次 LLM 调用当作整个 Agent。

追问：为什么同进程仍要分层？

进程是部署边界，职责是控制权边界。即使 decide 和 orchestrate 在一个服务里，也要保证模型结果不能直接改状态或执行副作用；这样才能独立测试、重放、替换模型并准确归因。

日常实践关键点

核心判断

本章的日常工作不是记住名词，而是把“角色、职责与决策契约”变成可执行控制环：先识别触发信号，再用版本化证据定位首个偏离点，由 reasoning-owner 执行动作，最后用成功、拒绝、降级三类结果验收。推荐顺序是 最小化决策上下文 → 维护 Decision 判别联合 → 绑定决策证据 → 分层校验决策 → 支持决策回放；任何参数都必须从真实 SLO、风险基线或失败样本推导。

  1. 冻结请求、策略、模型 / 工具、数据和部署版本，避免边复现边变化。

  2. 写出五类 Decision 的最小 payload，并列出至少六条可自动校验的不变量。
  3. 画出编排、推理、治理、工具和模型网关的边界，标注谁拥有状态改变。
