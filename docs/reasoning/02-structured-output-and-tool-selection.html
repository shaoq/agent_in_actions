02 结构化输出与工具选择

学习目标

  • 区分语法有效、schema 有效、语义有效和策略允许四层条件。
  • 把工具选择建模为受约束分类，而不是把工具描述全部塞进 prompt。
  • 设计有限修复、降级与类型化失败，避免无限“再试一次”。
  • 解释工具意图、授权、执行和 observation 的组件边界。

链路总览与现有讲解

结构化输出与工具选择属于推理内核：输入是最小 DecisionContext、证据引用、工具候选与风险约束，输出是类型化候选 Decision 或 ReasoningFailure。学习时先沿现有图、表或主题章节理解链路，再用后续 stage 检查每个中间产物的不变量。

为什么结构化输出仍然会失败

JSON 只是编码形式。一个响应可以是合法 JSON，却选择不存在的工具、漏掉业务必填字段、引用过期证据，甚至提出当前身份无权执行的动作。生产链路至少包含四层验证：

  1. 解析（parse）：字节能否解码为目标格式。
  2. 结构（schema）：kind、字段、类型、枚举和互斥关系是否合法。
  3. 语义（semantic）：参数是否符合领域规则，证据是否支持该选择，当前阶段是否允许。
  4. 策略（policy）：当前主体、风险、租户和审批状态是否允许考虑或执行该动作。
关键边界 · 推理内核可以减少不合理选择，并返回结构化 ActionIntent；治理与编排仍必须在执行前独立授权。模型输出永远不是权限凭证。

Decision schema 设计

优先使用判别联合，让每个 kind 只有自己的 payload。以下示例强调契约，不绑定具体 SDK：

{
  "kind": "act",
  "decision_id": "dec-1042",
  "summary": "需要查询订单当前状态",
  "evidence_refs": ["user-message:18", "identity-check:7"],
  "confidence": "high",
  "assumptions": [],
  "payload": {
    "capability_id": "orders.read_status.v2",
    "arguments": { "order_id": "ORD-42" },
    "expected_observation": "订单状态与最后更新时间"
  }
}

推荐约束：

  • additionalProperties: false，阻止模型悄悄创造未定义字段。
  • capability 使用稳定 ID 而非自然语言名字；版本变化可以显式迁移。
  • 参数不接受自由拼接的 shell、SQL 或 URL；使用最小领域字段。
  • evidence_refs 只能引用当前 DecisionContext 中存在的证据。
  • 敏感值和密钥由执行侧注入，不进入模型上下文或 ActionIntent。

工具选择是一个漏斗

阶段问题主要负责人输出
能力裁剪哪些能力与当前租户、身份、阶段相关？编排 + 治理可考虑的最小能力集合
意图匹配哪个能力最可能产生缺失观察？推理内核capability_id + 参数候选
结构校验字段、类型、枚举与条件约束是否合法？确定性 validatorvalidated intent / validation error
授权决策这个主体现在是否可执行？是否要审批？治理层allow / deny / require approval
真实执行如何超时、重试、幂等并产生类型化结果？工具 / 动作层observation / tool error

能力描述要写“何时使用、输入含义、返回证据、失败类型、禁止场景”，而不是营销式功能说明。相似工具需要互斥提示，例如 orders.read_status 只读，orders.refund 有副作用且需审批。

语义校验与业务约束

可判定规则应尽量放在确定性代码中。例如：

if decision.kind == "act":
  capability MUST be in visible_capabilities
  arguments MUST satisfy capability.input_schema
  evidence_refs MUST exist in current_context
  requested_effect MUST match current_run_phase

if decision.kind == "finish":
  acceptance_evidence MUST NOT be empty
  unresolved_blockers MUST be explicit

不要要求模型在自然语言里“保证订单金额小于 500 元”。让模型提供 amount 和证据引用，再由 validator 使用权威业务数据判断。

修复、重试与降级

失败类型是否适合修复修复输入停止条件
JSON 截断 / 类型错误通常可以一次受限修复validation error 的字段路径，不泄露额外敏感上下文最多 N 次或预算不足
未知 capability_id可以重新选择允许集合与最小差异重复同一非法选择即停止
缺少业务证据不应靠格式修复转为 clarify 或提出只读查询意图无法取得证据则 escalate
策略拒绝不得通过改写参数绕过只允许选择安全替代或转审批明确 deny 时终止该动作路径
模型网关不可用由网关做受控 fallback能力与 SLA 需求超时 / 成本 / 降级边界

修复请求应携带机器生成的错误摘要，而不是把整个失败对话再次堆入上下文。每次修复都记录 attempt、契约版本、错误码和候选响应指纹，便于发现循环。

结构化输出不解决什么

  • 不保证事实正确；需要权威观察、grounding 和引用验证。
  • 不保证动作获授权；需要治理层策略和审批。
  • 不保证执行幂等；需要工具层幂等键和副作用控制。
  • 不保证任务完成；需要编排层的业务 acceptance。
  • 不保证模型路由稳定；需要模型网关的版本、fallback 与观测。

核心环节：从输入到可验证输出

1. 治理 Schema 演进

治理 Schema 演进是生产者与消费者共同承认的版本化解释边界；结构合法不代表语义、授权或业务结果成立。

职责与 I/O
reasoning-owner 在“治理 Schema 演进”环节接收 最小 DecisionContext、证据引用、工具候选与风险约束，产出 类型化候选 Decision 或 ReasoningFailure；输出必须能由下游按版本消费。
工作机制
显式 schema 定义字段、枚举、状态转移和错误联合，消费者先做语法校验，再按领域不变量做语义校验。本环节把 最小 DecisionContext、证据引用、工具候选与风险约束 中与“治理 Schema 演进”相关的事实转成 类型化候选 Decision 或 ReasoningFailure 的一部分。
必须成立的约束
必填字段、版本兼容、非法状态、未知枚举和错误语义必须确定；不能用自由文本默认值掩盖缺失事实。本章共同约束是：结论必须受 schema、可见证据和风险约束限制；违反时必须产生可区分的拒绝、未知或升级结果。
难点与取舍
严格契约提升可恢复性但增加演进成本；兼容窗口由调用方升级节奏和旧数据回放覆盖决定。本层总体取舍是 上下文充分性与噪声、推理质量与延迟成本，参数必须引用版本化数据、SLO 或风险基线。
边界与注意事项
契约约束信息形状和解释，不授予权限、不选择业务策略，也不证明下游验收成功。推理内核 的总边界是：只提出候选决策，不拥有流程推进、授权或动作执行权；错误会继续污染 编排路由、工具选择、澄清和评测中的决策质量。

语法校验必须区分结构有效、动作完成、策略合规和业务验收；缺证据时只能是 pending 或 unknown。

职责与 I/O

4. 裁剪工具候选

裁剪工具候选必须先满足能力、风险和兼容硬约束，再比较质量、延迟与成本；名称或模型自报不能作为路由证据。

职责与 I/O
reasoning-owner 在“裁剪工具候选”环节接收 最小 DecisionContext、证据引用、工具候选与风险约束，产出 类型化候选 Decision 或 ReasoningFailure；输出必须能由下游按版本消费。
工作机制
能力画像裁剪不可用候选，版本化策略对剩余候选评分，adapter 规范输入输出并记录 route_reason 与替代项。本环节把 最小 DecisionContext、证据引用、工具候选与风险约束 中与“裁剪工具候选”相关的事实转成 类型化候选 Decision 或 ReasoningFailure 的一部分。
必须成立的约束
结构化输出、工具支持、上下文、区域和数据策略等硬约束不可被低价覆盖；fallback 必须兼容安全与语义。本章共同约束是：结论必须受 schema、可见证据和风险约束限制；违反时必须产生可区分的拒绝、未知或升级结果。

修复预算必须以版本化分布、业务切片和风险容忍度为依据；单个平均值或脱离基线的固定阈值不能代表系统边界。

失败传播与 first divergence

  1. 上游输入：先确认 最小 DecisionContext、证据引用、工具候选与风险约束 的身份、作用域、时间与版本是否一致；输入错误不应在本层被“修正”为新事实。
  2. 首个偏离：按 治理 Schema 演进 → 语法校验 → 语义校验 → 裁剪工具候选 → 修复预算 比较预期与实际中间产物，第一个不满足概念不变量的 stage 才是本章候选根因。
  3. 下游影响：本层错误会改变 编排路由、工具选择、澄清和评测中的决策质量；最终症状可能在更晚阶段出现，但不能从症状直接倒推 Prompt。
  4. 归属边界：reasoning-owner修复本层契约或机制；若 first divergence 位于相邻层，保留证据并交给相应 owner，不跨层调参。

概念—实践—证据回扣

核心区回答“是什么、为何成立、怎样传播”，实践区回答“何时触发、如何处置、怎样验收”。每个 stage 与 practice 使用稳定锚点双向连接：

  • 治理 Schema 演进的不变量或约束失效时，进入对应日常实践；用本页工程产物、Trace / 评测切片和拒绝 / 降级结果证明是否恢复。

  • 语义校验的不变量或约束失效时，进入对应日常实践；用本页工程产物、Trace / 评测切片和拒绝 / 降级结果证明是否恢复。

  • 修复预算的不变量或约束失效时，进入对应日常实践；用本页工程产物、Trace / 评测切片和拒绝 / 降级结果证明是否恢复。

工程要点

  • 同时保留原始模型响应的受控引用、解析结果和 validation errors；敏感内容按治理策略脱敏。
  • 把 schema conformance、非法工具率、修复成功率、重复决策率和澄清率作为可观测指标。
  • 构建 golden cases：合法 act、缺参、错工具、越权、证据不足、finish 过早、不可解析。
  • 模型或 prompt 升级时，用相同 DecisionContext 回放并比较结构化差异，而不只看最终文本。

常见坑

  1. 一个巨型 schema：所有 kind 字段都 optional，导致无法判断哪些组合有效。
  2. 工具描述越多越好：扩大混淆面、token 成本和越权暴露；应先裁剪。
  3. 解析失败就无限重试：可能是契约不兼容或输出截断，重试只烧预算。
  4. 模型参数直接执行：跳过语义校验、授权和幂等，是高风险副作用入口。
  5. 把拒绝当模型错误：策略 deny 是合法业务结果，不应让模型尝试规避。

理论追问

请先脱稿回答，再展开参考答案核对边界、证据和取舍。

  1. 为什么 JSON Schema 校验通过仍不代表决策正确？

    查看参考答案

    Schema 只证明结构和基本类型合法，不能证明工具适用、参数语义正确、资源存在、主体有权限或动作满足业务约束。结构校验后仍需语义校验、授权、前置状态检查和必要的人工审批。

  2. 工具选择为什么应设计成“能力过滤 → 候选判断 → 确定性校验”的漏斗？

    查看参考答案

    先按权限、环境、风险和输入可用性过滤不可调用工具，可缩小模型选择空间；模型只在合法候选中判断语义匹配；最后由代码校验参数与业务不变量。这样既保留语言理解，又避免模型绕过硬约束。

  3. 当两个工具描述高度相似时，应优先修改 Prompt 还是工具契约？

    查看参考答案

    先修工具契约：拆清职责、前置条件、参数、返回和反例，并用稳定 ID 与版本区分。仅靠 Prompt 补丁会随工具增多变脆；还应通过混淆矩阵和错误样本验证选择边界，必要时增加确定性路由。

  4. 结构化输出解析失败时，谁决定修复、重试或降级？

    查看参考答案

    解析器报告类型化错误，编排根据错误类别和预算决定一次受约束修复、切换模型、使用默认安全路径或转人工。推理内核不应无限自修；Trace 要保留原输出摘要、schema 版本、修复次数和终止原因。

  5. 模型选择了存在且参数合法的退款工具，但用户无权退款，应该在哪一步拦截？

    查看参考答案

    权限过滤最好在工具暴露前就排除该能力，动作执行前治理层和资源侧还要再次强制授权。Schema 合法不能替代授权；拒绝结果应作为 policy Observation 回写编排，而不是让模型尝试换参数绕过。

面试表达

3 分钟结构

先把输出设计为 act、respond、clarify、escalate、finish 的判别联合；再讲四层验证：解析、schema、语义、策略；然后讲工具选择漏斗：治理和编排先裁剪，推理选择，validator 校验，治理授权，工具层执行；最后补充有限修复、类型化失败、回放指标以及结构化输出仍不能替代事实、授权、幂等和业务验收。

追问：function calling 是否已经解决可靠性？

它改善了结构约束和调用接口，但不自动保证工具选择正确、参数符合业务语义、主体有权限、执行幂等或结果真实。仍需要能力裁剪、validator、授权、执行隔离和 acceptance。

日常实践关键点

核心判断

本章的日常工作不是记住名词，而是把“结构化输出与工具选择”变成可执行控制环：先识别触发信号，再用版本化证据定位首个偏离点，由 reasoning-owner 执行动作，最后用成功、拒绝、降级三类结果验收。推荐顺序是 治理 Schema 演进 → 执行语法校验 → 执行语义校验 → 裁剪工具候选 → 限制修复预算；任何参数都必须从真实 SLO、风险基线或失败样本推导。

围绕“裁剪工具候选 / 限制修复预算”展示脱敏后的 trace_id、decision_schema、可见证据、工具候选、风险约束、模型版本与决策 Trace、前后对照和失败切片；个人实验还应给出固定输入、复现步骤和观察结果，不虚构线上规模。

高频追问

为什么问题属于 推理内核 而不是相邻层？参数如何从 SLO 或风险基线推导？结果未知时怎样避免扩大副作用？reasoning-owner 不在线时，谁依据什么证据接管？

边界与取舍

“裁剪工具候选 / 限制修复预算”会增加契约、存储或人工成本，只有错误成本与复现收益覆盖这些成本时才扩大自动化。未处理过该类生产事故时，应明确这是方案设计或个人实验结论。

学习产出

  1. 为订单场景写一个三分支 Decision schema：查询、澄清、升级。
  2. 构造“JSON 合法但业务非法”的五个案例，并标注由哪层拦截。
  3. 设计最多两次修复的状态表，说明何时停止并返回类型化失败。
