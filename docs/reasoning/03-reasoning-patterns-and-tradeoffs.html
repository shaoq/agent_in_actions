03 推理模式及取舍

学习目标

  • 区分推理策略与编排拓扑：一个是如何形成决策，一个是谁推进步骤。
  • 用任务可枚举性、观察依赖、计划稳定性、风险、延迟和成本选择模式。
  • 为每个模式定义预算、停止条件、结构化产物和降级路径。
  • 识别过度推理、自检回声、计划漂移和候选投票等失败。

先区分两类“模式”

推理模式描述一次或一组决策如何形成，例如 Direct、Routing、Plan、ReAct、自检与候选比较。编排模式描述运行时如何安排步骤、状态和参与者，例如 Workflow、Router、Agent loop、Orchestrator-Workers。两者有关联但不相同：

  • 编排层可以在一个固定 Workflow 节点中调用 Direct 分类推理。
  • 有限 ReAct 需要编排层在每轮 act/observe 后重建上下文并执行停止规则。
  • Plan-and-Execute 的计划由推理内核提出，计划版本、步骤状态和恢复由编排层维护。
  • Multi-Agent 是协作拓扑，不等于“推理更深”；如果问题不需要独立上下文和并行专长，单 Agent 更简单。

模式选择的六个问题

  1. 路径可枚举吗？若业务步骤稳定，用确定性 Workflow；不要为“显得智能”增加开放推理。
  2. 证据已经齐全吗？齐全时 Direct 足够；缺外部事实时需要提出检索或工具意图。
  3. 计划能在执行前稳定吗？能则 Plan-and-Execute 便于审批；不能则有限观察循环更合适。
  4. 错误代价多高？高风险时增加确定性 validator、独立 evidence 和 HITL，而不是简单“多想几遍”。
  5. 延迟与成本预算多少？候选、反思和多轮调用必须证明边际收益。
  6. 如何判定进展？没有可观察的进展信号，就无法可靠停止循环或评价模式。

计划的工程契约

计划不是自然语言待办清单。可执行计划至少包含：

PlanStep {
  step_id, objective, prerequisites,
  required_capability, expected_evidence,
  acceptance_condition, risk_level,
  status  // 由编排层维护，不由模型直接写入
}

推理内核提出计划候选和 replan 理由；编排层保存 plan_version、逐步提交状态、执行授权、重试和恢复。观察与假设冲突时，应标记旧计划失效原因，而不是无痕改写历史。

有限 ReAct 的约束

  • 每轮只输出结构化 act / clarify / escalate / finish，不把隐式思维过程当作运行状态。
  • 编排层维护最大步数、总时长、成本、相同动作参数重复次数和连续无进展次数。
  • observation 必须带类型和 provenance；工具错误不是普通文本事实。
  • finish 需要引用 acceptance evidence；“我认为完成”不能越过业务验收。
  • 高风险动作在循环外设确定性门禁，必要时暂停并等待 HITL。

自检与多候选的真实边界

让同一个模型生成多个候选再投票，可能改善随机性问题，但候选往往共享同一上下文盲点。更可靠的顺序是：

  1. 先运行确定性 schema、业务规则和证据校验。
  2. 再用显式 rubric 比较少量候选，而不是让模型说“哪个更好”。
  3. 对高风险判断引入独立数据源、不同角色或人工，不把同源自检当独立证据。
  4. 用离线评测证明额外调用带来足够收益，再允许线上使用。

工程要点

  • 把策略名称、参数、候选数、计划版本和停止原因记录进 trace，支持按模式分桶评测。
  • 为每种模式定义基线：质量、延迟、token、工具调用数、澄清率、升级率与失败类型。
  • 允许按风险和难度动态路由，但路由规则必须版本化；模型不能自行无限提升预算。
  • 新模式上线采用离线回放、shadow、有限流量与回退门禁，不只依赖 demo。

常见坑

  1. 复杂就用 ReAct：很多复杂任务其实步骤稳定，Workflow 更可靠。
  2. 计划一经生成永不改变：外部观察变化后继续执行旧计划会制造错误副作用。
  3. 反思越多越准确：可能只增加延迟并强化同一错误前提。
  4. 把 Multi-Agent 当推理模式：协作成本、共享状态和归因可能超过收益。
  5. 只看最终答案：相同答案可能来自越权工具、错误证据或不可接受成本。

面试表达

30 秒回答

我按最小充分原则选择推理模式：路径可枚举就交给 Workflow；证据齐全的一步判断用 Direct；计划稳定且需审批时用 Plan-and-Execute；必须边观察边调整时才用有限 ReAct；高价值决策可增加显式 rubric 的候选比较。所有模式都要有结构化产物、预算、停止条件、失败出口和评测证据。

追问：怎么证明更深推理值得？

在代表性数据集上与简单基线比较任务成功率、轨迹合法率、严重错误、延迟和成本，并按难度与风险分桶。只有质量增益超过额外调用和失败面的代价，才对对应桶启用。

学习产出

  1. 为五个业务场景完成模式选择矩阵，每个场景拒绝至少两个替代方案。
  2. 为有限 ReAct 写出进展信号、四个停止条件和两个人工出口。
  3. 设计一次“Direct 基线 vs 复杂推理”的离线评测，明确指标和上线门槛。
