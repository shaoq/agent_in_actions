03 推理模式及取舍

学习目标

  • 区分推理策略与编排拓扑：一个是如何形成决策，一个是谁推进步骤。
  • 用任务可枚举性、观察依赖、计划稳定性、风险、延迟和成本选择模式。
  • 为每个模式定义预算、停止条件、结构化产物和降级路径。
  • 识别过度推理、自检回声、计划漂移和候选投票等失败。

链路总览与现有讲解

推理模式及取舍属于推理内核：输入是最小 DecisionContext、证据引用、工具候选与风险约束，输出是类型化候选 Decision 或 ReasoningFailure。学习时先沿现有图、表或主题章节理解链路，再用后续 stage 检查每个中间产物的不变量。

先区分两类“模式”

推理模式描述一次或一组决策如何形成，例如 Direct、Routing、Plan、ReAct、自检与候选比较。编排模式描述运行时如何安排步骤、状态和参与者，例如 Workflow、Router、Agent loop、Orchestrator-Workers。两者有关联但不相同：

  • 编排层可以在一个固定 Workflow 节点中调用 Direct 分类推理。
  • 有限 ReAct 需要编排层在每轮 act/observe 后重建上下文并执行停止规则。
  • Plan-and-Execute 的计划由推理内核提出，计划版本、步骤状态和恢复由编排层维护。
  • Multi-Agent 是协作拓扑，不等于“推理更深”；如果问题不需要独立上下文和并行专长，单 Agent 更简单。

模式选择的六个问题

  1. 路径可枚举吗？若业务步骤稳定，用确定性 Workflow；不要为“显得智能”增加开放推理。
  2. 证据已经齐全吗？齐全时 Direct 足够；缺外部事实时需要提出检索或工具意图。
  3. 计划能在执行前稳定吗？能则 Plan-and-Execute 便于审批；不能则有限观察循环更合适。
  4. 错误代价多高？高风险时增加确定性 validator、独立 evidence 和 HITL，而不是简单“多想几遍”。
  5. 延迟与成本预算多少？候选、反思和多轮调用必须证明边际收益。
  6. 如何判定进展？没有可观察的进展信号，就无法可靠停止循环或评价模式。

计划的工程契约

计划不是自然语言待办清单。可执行计划至少包含：

PlanStep {
  step_id, objective, prerequisites,
  required_capability, expected_evidence,
  acceptance_condition, risk_level,
  status  // 由编排层维护，不由模型直接写入
}

推理内核提出计划候选和 replan 理由；编排层保存 plan_version、逐步提交状态、执行授权、重试和恢复。观察与假设冲突时，应标记旧计划失效原因，而不是无痕改写历史。

有限 ReAct 的约束

  • 每轮只输出结构化 act / clarify / escalate / finish，不把隐式思维过程当作运行状态。
  • 编排层维护最大步数、总时长、成本、相同动作参数重复次数和连续无进展次数。
  • observation 必须带类型和 provenance；工具错误不是普通文本事实。
  • finish 需要引用 acceptance evidence；“我认为完成”不能越过业务验收。
  • 高风险动作在循环外设确定性门禁，必要时暂停并等待 HITL。

自检与多候选的真实边界

让同一个模型生成多个候选再投票，可能改善随机性问题，但候选往往共享同一上下文盲点。更可靠的顺序是：

  1. 先运行确定性 schema、业务规则和证据校验。
  2. 再用显式 rubric 比较少量候选，而不是让模型说“哪个更好”。
  3. 对高风险判断引入独立数据源、不同角色或人工，不把同源自检当独立证据。
  4. 用离线评测证明额外调用带来足够收益，再允许线上使用。

核心环节：从输入到可验证输出

1. 定义模式入口

定义模式入口必须先满足能力、风险和兼容硬约束，再比较质量、延迟与成本；名称或模型自报不能作为路由证据。

工作机制

结构化输出、工具支持、上下文、区域和数据策略等硬约束不可被低价覆盖；fallback 必须兼容安全与语义。本章共同约束是：结论必须受 schema、可见证据和风险约束限制；违反时必须产生可区分的拒绝、未知或升级结果。

边界与注意事项
网关选择模型与规范协议，不选择业务工具、不拥有推理意图，也不授予工具执行权限。推理内核 的总边界是：只提出候选决策，不拥有流程推进、授权或动作执行权；错误会继续污染 编排路由、工具选择、澄清和评测中的决策质量。

2. Plan 契约

Plan 契约是生产者与消费者共同承认的版本化解释边界；结构合法不代表语义、授权或业务结果成立。

职责与 I/O
reasoning-owner 在“ Plan 契约”环节接收 最小 DecisionContext、证据引用、工具候选与风险约束，产出 类型化候选 Decision 或 ReasoningFailure；输出必须能由下游按版本消费。
工作机制
显式 schema 定义字段、枚举、状态转移和错误联合，消费者先做语法校验，再按领域不变量做语义校验。本环节把 最小 DecisionContext、证据引用、工具候选与风险约束 中与“ Plan 契约”相关的事实转成 类型化候选 Decision 或 ReasoningFailure 的一部分。
必须成立的约束
必填字段、版本兼容、非法状态、未知枚举和错误语义必须确定；不能用自由文本默认值掩盖缺失事实。本章共同约束是：结论必须受 schema、可见证据和风险约束限制；违反时必须产生可区分的拒绝、未知或升级结果。
难点与取舍

边界与注意事项

3. ReAct

ReAct必须以可观测信息增益和有界预算推进；没有新证据的重复思考不构成进展。

失败传播与 first divergence

  1. 上游输入：先确认 最小 DecisionContext、证据引用、工具候选与风险约束 的身份、作用域、时间与版本是否一致；输入错误不应在本层被“修正”为新事实。
  2. 首个偏离：按 定义模式入口 → Plan 契约 → ReAct → 自检门禁 → 用消融选择模式 比较预期与实际中间产物，第一个不满足概念不变量的 stage 才是本章候选根因。
  3. 下游影响：本层错误会改变 编排路由、工具选择、澄清和评测中的决策质量；最终症状可能在更晚阶段出现，但不能从症状直接倒推 Prompt。
  4. 归属边界：reasoning-owner修复本层契约或机制；若 first divergence 位于相邻层，保留证据并交给相应 owner，不跨层调参。

概念—实践—证据回扣

核心区回答“是什么、为何成立、怎样传播”，实践区回答“何时触发、如何处置、怎样验收”。每个 stage 与 practice 使用稳定锚点双向连接：

  • 把策略名称、参数、候选数、计划版本和停止原因记录进 trace，支持按模式分桶评测。
  • 为每种模式定义基线：质量、延迟、token、工具调用数、澄清率、升级率与失败类型。
  • 允许按风险和难度动态路由，但路由规则必须版本化；模型不能自行无限提升预算。
  • 新模式上线采用离线回放、shadow、有限流量与回退门禁，不只依赖 demo。

常见坑

  1. 复杂就用 ReAct：很多复杂任务其实步骤稳定，Workflow 更可靠。
  2. 计划一经生成永不改变：外部观察变化后继续执行旧计划会制造错误副作用。
  3. 反思越多越准确：可能只增加延迟并强化同一错误前提。
  4. 把 Multi-Agent 当推理模式：协作成本、共享状态和归因可能超过收益。
  5. 只看最终答案：相同答案可能来自越权工具、错误证据或不可接受成本。

理论追问

请先脱稿回答，再展开参考答案核对边界、证据和取舍。

  1. 什么时候直接决策比 ReAct 或 Plan-and-Execute 更合理？

    查看参考答案

    当目标明确、证据已齐、只需一次低风险判断且输出可确定性校验时，直接决策延迟和成本最低。若环境反馈会改变下一步才需要 ReAct；若步骤依赖可提前分解并需要审批或并行，才考虑 Plan-and-Execute。

  2. ReAct 的“思考—动作—观察”循环必须加哪些确定性边界？

    查看参考答案

    至少包括最大步数与成本、工具白名单、参数和授权校验、重复动作与无信息增益检测、超时、人工升级及业务验收。循环适应性来自 Observation，不意味着模型可以自行决定无限重试或高风险动作。

  3. 计划什么时候算过期，谁触发重规划？

    查看参考答案

    依赖失败、外部对象版本变化、授权或审批失效、关键假设被 Observation 否定、预算变化时，计划即可能过期。编排根据结构化事件触发重规划，模型提出新计划；新旧差异和重新授权必须可审计。

  4. Self-consistency 或多候选投票为什么不能解决事实与权限问题？

    查看参考答案

    多个候选可能共享同一错误知识，投票只能反映模型分布而不是外部真值；权限更是确定性策略，不由多数意见决定。事实需检索或工具证据，权限需策略引擎，候选机制只用于受控的不确定判断。

  5. 高金额退款场景应如何组合确定性 Workflow 与推理模式？

    查看参考答案

    身份、资格、金额阈值、审批、幂等执行和验收放在确定性 Workflow；自然语言诉求理解、证据摘要或复杂原因调查可用受约束推理。编排明确边界，模型只能提出动作，高风险节点由规则和人工裁决。

我按最小充分原则选择推理模式：路径可枚举就交给 Workflow；证据齐全的一步判断用 Direct；计划稳定且需审批时用 Plan-and-Execute；必须边观察边调整时才用有限 ReAct；高价值决策可增加显式 rubric 的候选比较。所有模式都要有结构化产物、预算、停止条件、失败出口和评测证据。

追问：怎么证明更深推理值得？

在代表性数据集上与简单基线比较任务成功率、轨迹合法率、严重错误、延迟和成本，并按难度与风险分桶。只有质量增益超过额外调用和失败面的代价，才对对应桶启用。

核心判断

本章的日常工作不是记住名词，而是把“推理模式及取舍”变成可执行控制环：先识别触发信号，再用版本化证据定位首个偏离点，由 reasoning-owner 执行动作，最后用成功、拒绝、降级三类结果验收。推荐顺序是 定义模式入口 → 固化 Plan 契约 → 约束 ReAct → 设置自检门禁 → 用消融选择模式；任何参数都必须从真实 SLO、风险基线或失败样本推导。

先交代 推理模式及取舍 的业务风险和 reasoning-owner 边界；再说触发信号与首个偏离点；随后说明契约、动作和参数依据；最后给出验证、止损和复盘结果。回答要同时覆盖正常路径与拒绝 / 降级路径。

可信证据

围绕“设置自检门禁 / 用消融选择模式”展示脱敏后的 trace_id、decision_schema、可见证据、工具候选、风险约束、模型版本与决策 Trace、前后对照和失败切片；个人实验还应给出固定输入、复现步骤和观察结果，不虚构线上规模。

高频追问

为什么问题属于 推理内核 而不是相邻层？参数如何从 SLO 或风险基线推导？结果未知时怎样避免扩大副作用？reasoning-owner 不在线时，谁依据什么证据接管？

边界与取舍

“设置自检门禁 / 用消融选择模式”会增加契约、存储或人工成本，只有错误成本与复现收益覆盖这些成本时才扩大自动化。未处理过该类生产事故时，应明确这是方案设计或个人实验结论。

学习产出

  1. 为五个业务场景完成模式选择矩阵，每个场景拒绝至少两个替代方案。
  2. 为有限 ReAct 写出进展信号、四个停止条件和两个人工出口。
  3. 设计一次“Direct 基线 vs 复杂推理”的离线评测，明确指标和上线门槛。
