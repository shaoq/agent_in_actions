04 不确定性、失败与恢复

学习目标

  • 区分信息缺失、证据冲突、任务歧义、能力不足、策略阻塞和模型故障。
  • 把置信与证据充分性分开，设计 clarify、retrieve、repair、fallback 和 escalate 出口。
  • 基于 trace 做事实—假设—补证—修复—回归的分层归因。
  • 说明推理恢复与编排重试、网关 fallback、工具补偿和治理响应的边界。

链路总览与现有讲解

不确定性、失败与恢复属于推理内核：输入是最小 DecisionContext、证据引用、工具候选与风险约束，输出是类型化候选 Decision 或 ReasoningFailure。学习时先沿现有图、表或主题章节理解链路，再用后续 stage 检查每个中间产物的不变量。

不确定性不是一个分数

单个 confidence=0.82 很容易产生虚假精确感。工程上更重要的是说明“不确定在哪里、缺什么证据、什么动作能降低不确定性、错误代价如何”。

类型例子可降低方式安全出口
目标歧义“取消它”不知道指订单还是订阅最小必要澄清clarify，不猜测实体
事实缺失没有订单当前状态提出只读查询或检索意图act(read)；不可取得则 escalate
证据冲突CRM 与支付系统状态不同比较来源权威性、时间和一致性报告冲突并暂停高风险动作

证据充分性优先于自报置信

可交付决策应同时说明：

  • claim：准备采取或交付什么结论。
  • evidence_refs：哪些当前观察支持它，来源和时效如何。
  • assumptions：哪些条件尚未直接验证。
  • counterevidence：是否存在冲突证据或未解决阻塞。
  • impact：判断错误的损害与可逆性。
  • next_safe_step：证据不足时最小、可逆、可授权的下一步。

置信可作为路由信号，但阈值必须在代表性数据上校准。例如“high”不应只表示模型语气肯定，而应与错误率、覆盖率和风险桶对应。

恢复策略及所有权

  1. 澄清（clarify）：推理提出对决策有信息增益的最小问题；交互层呈现，编排等待并恢复。
  2. 补证（retrieve / inspect）：推理提出需要的观察；治理授权，工具或知识层执行，编排写回 observation。
  3. 结构修复（repair）：仅针对可修复的输出错误，携带字段路径，限制次数；不能用于绕过策略拒绝。
  4. 模型降级（fallback）：推理声明能力需求，模型网关选择兼容模型；若降级会破坏 schema 或质量门槛，应失败而不是静默继续。
  5. 重规划（replan）：推理基于新证据提出新计划和差异；编排保存版本并决定哪些已执行步骤需要核对。
  6. 转人工（escalate）：携带问题、已验证事实、冲突、已尝试动作和建议选项，避免让人工重新调查全部上下文。
副作用边界 · 已执行动作的撤销或补偿属于编排与工具层。推理可以提出补偿意图，但不能假设外部世界会随重新推理自动回滚。

建议记录结构化摘要、输入/契约版本、证据引用、模型请求关联 ID、校验结果、策略、预算与停止原因。不要默认记录密钥、完整 PII 或模型私有 chain-of-thought。调试需要的是可重放的控制证据，不是未经治理的内部文本。

安全与 Prompt Injection 的边界

检索文档或工具结果中的指令属于不可信数据，不能提升为系统约束。推理内核应保留 provenance、区分 data 与 instruction，并拒绝把外部文本当授权。但完整防线分布在：

  • 治理层：输入分类、策略、权限、审计和敏感数据处理。
  • 推理内核：遵循指令优先级，显式标记冲突与不可信证据。
  • 工具层与沙箱：即使推理受骗，也限制可执行能力和副作用。
  • 评测层：用注入和越权数据集持续回归。

核心环节：从输入到可验证输出

1. 证据充分性

证据充分性必须能把结论追溯到同一请求的输入、版本、决定、动作和验收事件，而不是依赖不可检索的日志文本。

职责与 I/O
reasoning-owner 在“证据充分性”环节接收 最小 DecisionContext、证据引用、工具候选与风险约束，产出 类型化候选 Decision 或 ReasoningFailure；输出必须能由下游按版本消费。
工作机制
correlation_id / trace_id 串联 span 与结构化事件，引用保留原始证据位置、policy / model / tool 版本和 owner。本环节把 最小 DecisionContext、证据引用、工具候选与风险约束 中与“证据充分性”相关的事实转成 类型化候选 Decision 或 ReasoningFailure 的一部分。
必须成立的约束
时间、因果顺序、采样与脱敏策略必须明确；高基数字段不能无界进入 Metric，敏感原文不能直接写 Log。本章共同约束是：结论必须受 schema、可见证据和风险约束限制；违反时必须产生可区分的拒绝、未知或升级结果。
难点与取舍
全量遥测利于归因却增加成本与隐私面；采样按风险与失败类型分层，关键审计事件不得随机丢弃。本层总体取舍是 上下文充分性与噪声、推理质量与延迟成本，参数必须引用版本化数据、SLO 或风险基线。
边界与注意事项
Trace 解释系统事件与可见证据，不保存模型私有推理；Metric 告警不能替代逐请求审计。推理内核 的总边界是：只提出候选决策，不拥有流程推进、授权或动作执行权；错误会继续污染 编排路由、工具选择、澄清和评测中的决策质量。

2. 类型化推理失败

类型化推理失败必须把明确输入转换为版本化、可验证的中间产物，并让失败能够定位到所属 owner。

职责与 I/O
reasoning-owner 在“类型化推理失败”环节接收 最小 DecisionContext、证据引用、工具候选与风险约束，产出 类型化候选 Decision 或 ReasoningFailure；输出必须能由下游按版本消费。
工作机制
该环节通过显式数据结构和状态判断连接前后阶段，输出同时携带版本、决定和证据引用。本环节把 最小 DecisionContext、证据引用、工具候选与风险约束 中与“类型化推理失败”相关的事实转成 类型化候选 Decision 或 ReasoningFailure 的一部分。
必须成立的约束
身份、时序、作用域、预算和失败语义必须确定，字段缺失或结果未知不能被静默补齐。本章共同约束是：结论必须受 schema、可见证据和风险约束限制；违反时必须产生可区分的拒绝、未知或升级结果。
难点与取舍
更多自动化会增加复杂度与误判面；参数需由真实 SLO、版本化数据和失败样本校准。本层总体取舍是 上下文充分性与噪声、推理质量与延迟成本，参数必须引用版本化数据、SLO 或风险基线。
边界与注意事项
该环节只拥有本层中间产物，不替代相邻层的推理、控制、授权、执行或验收职责。推理内核 的总边界是：只提出候选决策，不拥有流程推进、授权或动作执行权；错误会继续污染 编排路由、工具选择、澄清和评测中的决策质量。

3. 澄清或升级

澄清或升级必须以可观测信息增益和有界预算推进；没有新证据的重复思考不构成进展。

职责与 I/O
reasoning-owner 在“澄清或升级”环节接收 最小 DecisionContext、证据引用、工具候选与风险约束，产出 类型化候选 Decision 或 ReasoningFailure；输出必须能由下游按版本消费。
工作机制
入口条件选择最简单可行模式，每轮比较证据 / 状态变化，触线后返回 clarify、abstain、degrade 或 escalate。本环节把 最小 DecisionContext、证据引用、工具候选与风险约束 中与“澄清或升级”相关的事实转成 类型化候选 Decision 或 ReasoningFailure 的一部分。
必须成立的约束
最大步数、时间、成本、重规划次数和退出类型必须显式；自检不能无限递归，也不能绕过外部验证。本章共同约束是：结论必须受 schema、可见证据和风险约束限制；违反时必须产生可区分的拒绝、未知或升级结果。
难点与取舍
更长推理可能提高复杂任务质量，也会累积延迟和错误；预算由任务价值、失败成本和消融结果决定。本层总体取舍是 上下文充分性与噪声、推理质量与延迟成本，参数必须引用版本化数据、SLO 或风险基线。
边界与注意事项
推理可以发现不确定性并提出澄清，编排负责实际暂停、预算执行和人工接管。推理内核 的总边界是：只提出候选决策，不拥有流程推进、授权或动作执行权；错误会继续污染 编排路由、工具选择、澄清和评测中的决策质量。

4. 恢复 Owner

恢复 Owner成立的前提是每个控制决定只有一个最终 owner，交接不转移未声明的权限与验收责任。

职责与 I/O
reasoning-owner 在“恢复 Owner”环节接收 最小 DecisionContext、证据引用、工具候选与风险约束，产出 类型化候选 Decision 或 ReasoningFailure；输出必须能由下游按版本消费。
工作机制
控制点以类型化输入 / 输出连接 owner；交出的是受约束任务或事实，不是共享的隐式控制权。本环节把 最小 DecisionContext、证据引用、工具候选与风险约束 中与“恢复 Owner”相关的事实转成 类型化候选 Decision 或 ReasoningFailure 的一部分。
必须成立的约束
决策、推进、动作、验收和事件升级必须分别可归属，交接双方不能同时写同一控制状态。本章共同约束是：结论必须受 schema、可见证据和风险约束限制；违反时必须产生可区分的拒绝、未知或升级结果。
难点与取舍
拆分能提高并行度，却增加协调、上下文复制和冲突成本；是否拆分要由失败归因与吞吐基线决定。本层总体取舍是 上下文充分性与噪声、推理质量与延迟成本，参数必须引用版本化数据、SLO 或风险基线。
边界与注意事项
推理可以建议，工具可以执行，治理可以允许或拒绝，但只有编排 owner 推进状态，业务 owner 做最终验收。推理内核 的总边界是：只提出候选决策，不拥有流程推进、授权或动作执行权；错误会继续污染 编排路由、工具选择、澄清和评测中的决策质量。

失败传播与 first divergence

  1. 上游输入：先确认 最小 DecisionContext、证据引用、工具候选与风险约束 的身份、作用域、时间与版本是否一致；输入错误不应在本层被“修正”为新事实。
  2. 首个偏离：按 证据充分性 → 类型化推理失败 → 澄清或升级 → 恢复 Owner → 保存决策 Trace 比较预期与实际中间产物，第一个不满足概念不变量的 stage 才是本章候选根因。
  3. 下游影响：本层错误会改变 编排路由、工具选择、澄清和评测中的决策质量；最终症状可能在更晚阶段出现，但不能从症状直接倒推 Prompt。
  4. 归属边界：reasoning-owner修复本层契约或机制；若 first divergence 位于相邻层，保留证据并交给相应 owner，不跨层调参。

概念—实践—证据回扣

核心区回答“是什么、为何成立、怎样传播”，实践区回答“何时触发、如何处置、怎样验收”。每个 stage 与 practice 使用稳定锚点双向连接：

  • 证据充分性的不变量或约束失效时，进入对应日常实践；用本页工程产物、Trace / 评测切片和拒绝 / 降级结果证明是否恢复。

  • 澄清或升级的不变量或约束失效时，进入对应日常实践；用本页工程产物、Trace / 评测切片和拒绝 / 降级结果证明是否恢复。

工程要点

  • 为 clarification rate、abstention/escalation、证据覆盖、置信校准、非法决策和无进展循环建立分桶指标。
  • 恢复前核对 DecisionContext 版本、权限、外部状态和已发生副作用；不要只重放旧模型输出。
  • 把 retryable 与 non-retryable 写进类型，重试必须有新信息、退避或不同能力，否则只是重复。
  • 为失败出口定义用户可理解的反馈，不暴露内部敏感信息，也不伪装成成功。

常见坑

  1. “不确定就多调用几次”：没有新证据时，多次调用只改变采样。
  2. 把 confidence 当概率：未经校准的数字不能直接决定高风险动作。
  3. 任何错误都 fallback：策略拒绝、证据不足和工具副作用不能靠换模型解决。
  4. 恢复等于重跑：必须核对版本、权限和外部状态，避免重复副作用。
  5. 记录全部思维文本：增加隐私、合规和噪声风险，却未必提高可诊断性。

理论追问

请先脱稿回答，再展开参考答案核对边界、证据和取舍。

  1. 为什么模型自报 90% 置信度不能直接驱动高风险动作？

    查看参考答案

    自报置信度未必经过校准，会受措辞、模型和场景漂移影响。高风险动作应依据可验证证据充分性、规则校验、权限和审批；置信度最多作为路由信号，并需在独立数据集上校准。

  2. “证据不足”和“模型不会做”如何区分？

    查看参考答案

    先检查所需事实是否在受权上下文、RAG 或工具 Observation 中存在，再检查模型是否正确引用和使用。若输入缺证据，应检索、澄清或拒答；若证据齐全仍判断错，才归因推理能力、Prompt 或模型路由。

  3. 为什么失败分类必须先于重试？

    查看参考答案

    解析错误、权限拒绝、业务冲突、暂时网络故障和 effect unknown 的恢复策略完全不同。盲目重试会放大成本、绕过拒绝或重复副作用；编排应根据类型化错误决定修复、退避、查询、降级或转人工。

  4. 不确定性高时，澄清、检索、换模型和转人工应如何选择？

    查看参考答案

    缺少用户意图时澄清，缺外部事实时检索，能力或上下文窗口不足时才换模型；高风险、冲突无法消解或预算耗尽时转人工。选择应由缺口类型、风险、成本和可获得证据决定，而不是统一升级更强模型。

  5. 诊断推理失败时，Trace 为什么记录决策证据而不是完整思维链？

    查看参考答案

    工程诊断需要知道输入版本、证据引用、结构化候选、校验结果和后续 Observation，不需要暴露不可验证的私有思维过程。保存完整思维链增加隐私、安全和存储风险，也不能保证真实解释因果。

3 分钟结构

我先把不确定性分类为目标歧义、事实缺失、证据冲突、能力边界、策略阻塞和模型契约故障；再按类型选择澄清、补证、受限修复、网关 fallback、重规划或转人工。置信只做经过校准的路由信号，关键决策依赖 evidence refs、假设和影响。调试时严格分开 trace 事实与原因假设，并用补证和回归用例验证修复。

追问：如何避免模型“自信地错”？

不用语气判断可信度。要求引用当前上下文中的权威证据，确定性验证 schema 和业务规则，对置信做离线校准，高风险动作由独立策略与人工门禁控制，证据冲突时允许 abstain 或 escalate。

日常实践关键点

核心判断

本章的日常工作不是记住名词，而是把“不确定性、失败与恢复”变成可执行控制环：先识别触发信号，再用版本化证据定位首个偏离点，由 reasoning-owner 执行动作，最后用成功、拒绝、降级三类结果验收。推荐顺序是 判断证据充分性 → 类型化推理失败 → 选择澄清或升级 → 分配恢复 Owner → 保存决策 Trace；任何参数都必须从真实 SLO、风险基线或失败样本推导。

排障顺序

  1. 冻结请求、策略、模型 / 工具、数据和部署版本，避免边复现边变化。

关联实践：分配恢复 Owner、保存决策 Trace

常见问法

在生产或可复现实验中，你如何把“分配恢复 Owner”和“保存决策 Trace”落到企业差旅 Agent 的 不确定性、失败与恢复 链路？

60–90 秒回答骨架

先交代 不确定性、失败与恢复 的业务风险和 reasoning-owner 边界；再说触发信号与首个偏离点；随后说明契约、动作和参数依据；最后给出验证、止损和复盘结果。回答要同时覆盖正常路径与拒绝 / 降级路径。

可信证据

围绕“分配恢复 Owner / 保存决策 Trace”展示脱敏后的 trace_id、decision_schema、可见证据、工具候选、风险约束、模型版本与决策 Trace、前后对照和失败切片；个人实验还应给出固定输入、复现步骤和观察结果，不虚构线上规模。

高频追问

为什么问题属于 推理内核 而不是相邻层？参数如何从 SLO 或风险基线推导？结果未知时怎样避免扩大副作用？reasoning-owner 不在线时，谁依据什么证据接管？

边界与取舍

“分配恢复 Owner / 保存决策 Trace”会增加契约、存储或人工成本，只有错误成本与复现收益覆盖这些成本时才扩大自动化。未处理过该类生产事故时，应明确这是方案设计或个人实验结论。

学习产出

  1. 为退款调查场景写六类不确定性及各自安全出口。
  2. 分析一个重复工具调用 trace，分开事实、四个假设、补充字段、修复和回归。
  3. 设计一次暂停后恢复核对表，覆盖契约、权限、外部状态和副作用。
