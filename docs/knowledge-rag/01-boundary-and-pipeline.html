01 边界与端到端流水线

学习目标

  • 区分离线摄取链路与在线检索增强链路。
  • 为 query、retrieval result、evidence context 定义最小契约。
  • 说明知识层与编排、推理、治理、记忆和评测的边界。
  • 识别数据面、控制面和证据面的不同失败。

核心概念

对象最低字段用途
RetrievalRequestquery、subject、tenant、filters、deadline、trace_id约束查询与授权范围
EvidenceChunkcontent、source_id、version、locator、score、ACL支持声明、引用与审计
RetrievalTracerewrite、candidate ids、rank、latency、policy decision离线评测与失败归因

工程要点

  1. 离线链路负责 connect、parse、normalize、chunk、index、publish；在线链路负责 authorize、retrieve、rerank、pack、generate、verify。
  2. 证据内容与来源元数据必须一起流动，不能在拼 prompt 时丢失 source_id、版本和 ACL。
  3. 检索超时、空结果、权限拒绝和证据冲突应是类型化状态，由编排决定澄清、拒答或降级。
  4. 对每次请求记录索引版本与策略版本，确保线上回答可复现。

常见坑

  • 把 RAG 当一次向量查询：忽略摄取质量、权限、重排、引用和评测。
  • 让推理内核直连索引：绕过编排预算和治理策略，trace 也无法闭环。
  • 混淆记忆：会话偏好不是企业手册；外部文档也不应自动变成用户经历。
  • 空结果仍强答：正确行为通常是澄清、拒答或说明证据不足。

理论追问

请先脱稿回答，再展开参考答案核对边界、证据和取舍。

  1. RAG 检索结果为什么默认不应写入长期 Memory？

    查看参考答案

    RAG 结果是带来源、权限、版本和时效的外部知识，本次只应进入 Working Context；Memory 保存经过价值、主体、同意和生命周期治理的 Agent 经历。自动写入会把外部文档变成无来源的“个人事实”，造成污染和越权复用。

  2. RAG 的离线摄取流水线和在线查询流水线如何分工？

    查看参考答案

    离线负责来源接入、清洗、切分、元数据、ACL、索引和版本发布；在线负责身份过滤、查询理解、召回、重排、证据组装、生成与引用校验。二者通过索引版本和数据契约连接，不能在请求时临时摄取未治理资料。

  3. 访问控制为什么必须在检索阶段前置？

    查看参考答案

    若先召回后仅在生成时过滤，未授权片段已经进入重排、日志或模型上下文，仍可能泄露。应将租户、部门、主体和资源 ACL 纳入候选过滤，并在输出与审计处再次校验，形成纵深防御。

  4. 没有足够证据时，RAG 系统应返回什么？

    查看参考答案

    应返回结构化的 evidence_insufficient，包含查询、过滤条件、已查来源和缺口，而不是强迫模型补全。编排可据此澄清、扩大受权检索、调用其他数据源或安全拒答。

  5. 用户收到错误答案时，如何区分摄取、检索、重排和生成问题？

    查看参考答案

    查看索引版本和原文是否正确、相关文档是否进入候选、重排是否把正确证据排后、最终回答是否忠于选中证据。Trace 要保留 query、filters、candidate IDs/scores、selected chunks、引用和生成版本，找到首个偏离点。

面试表达

如何定义生产级 RAG？

我把 RAG 定义为一条证据供应链：离线把外部资料按版本、结构和 ACL 建成可回滚索引；在线以用户身份约束检索，经过召回、重排和上下文组装，把带来源的证据交给推理内核，再校验声明和引用。编排拥有调用时机与降级，治理拥有策略，评测消费 trace；记忆系统单独管理 Agent 经历。
