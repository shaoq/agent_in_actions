01 边界与端到端流水线

学习目标

  • 区分离线摄取链路与在线检索增强链路。
  • 为 query、retrieval result、evidence context 定义最小契约。
  • 说明知识层与编排、推理、治理、记忆和评测的边界。
  • 识别数据面、控制面和证据面的不同失败。

核心概念

链路总览：RAG 是版本化证据供应链

对象最低字段用途
RetrievalRequestquery、subject、tenant、filters、deadline、trace_id约束查询与授权范围
EvidenceChunkcontent、source_id、version、locator、score、ACL支持声明、引用与审计
RetrievalTracerewrite、candidate ids、rank、latency、policy decision离线评测与失败归因

离线知识供给把外部资料变成带来源、版本、权限和血缘的可检索派生数据；在线取证则在一次请求的身份、时间与预算内选择证据。两者通过 active index version 和 schema 相接，却有不同生命周期。数据面传递 query、chunk 与 claim，控制面决定路由、授权、deadline 和降级，证据面维持 source—locator—claim 的可追溯关系。共同不变量是：任何回答都能说明谁在何时，基于哪版允许证据，经哪些阶段得到什么状态。

逐环节拆解

01 · 离线知识供给

离线供给只发布可追溯、可撤销的知识版本，不把未经治理的资料直接变成在线事实。

职责与输入输出
知识层接收权威来源及变更事件，产出带 ACL、lineage 和质量报告的版本化索引；它不拥有在线回答决策。
工作机制
连接、解析、规范化、切分、特征生成后写入 shadow index，经门禁再原子发布 active version，旧版保留回放窗口。
必须成立的约束
source、document、chunk 与索引版本必须可关联，删除和撤权能传播；否则在线命中无法证明新鲜性与合法性。
关键难点
新鲜度、解析质量、构建成本和可回滚性相互牵制，发布频率应由来源风险与 freshness SLO 决定。
边界与注意事项
供给完成不等于问题可回答，也不负责 query rewrite；错误索引会让下游在错误语料上得到稳定但错误的结果。

02 · 编排与适用性路由

编排决定是否以及如何调用 RAG，并在端到端预算内解释每次跳过、降级或终止。

职责与输入输出
编排层接收用户意图、会话状态和风险标签，产出 RAG、Memory、工具或澄清分支及各阶段 deadline。
工作机制
路由先区分可更新外部知识、个人经历和实时真值，再按风险与剩余预算分配检索、重排和校验步骤。
必须成立的约束
路由不可绕过治理，子阶段预算不得突破请求 deadline；每个降级必须保留原因与未完成的证据义务。
关键难点
多检索提高覆盖却增加延迟和成本，过早跳过又会让模型凭参数记忆回答；阈值需按查询桶校准。
边界与注意事项
编排拥有流程，不拥有文档真值或权限策略；入口误路由后，扩大 Top-K 或修改 prompt 都不能修复边界错误。

03 · 请求契约

RetrievalRequest 要把自然语言意图变成不可静默丢失的身份、范围、时间、版本和预算约束。

职责与输入输出
接口接收原查询与调用上下文，产出带 subject、tenant、filters、as_of、deadline、trace_id 的类型化请求。
工作机制
把硬约束与可改写文本分开表示：地区、职级、否定、时间不得被 rewrite 覆盖，缺字段通过 schema 校验快速失败。
必须成立的约束
身份与租户来自可信会话而非用户文本，时间语义明确，filter 可审计；否则 query 会落入错误权限或版本空间。
关键难点
契约过窄无法表达复合问题，过宽又难兼容演进；应区分 required、optional 和可扩展字段并做版本协商。
边界与注意事项
请求契约描述检索意图，不证明资料存在，也不替代策略判定；字段缺失会污染后续候选和 Trace。

04 · 授权决策

授权由治理层基于可信主体和资源策略决定，模型与检索分数都不能扩大允许范围。

职责与输入输出
治理层接收 subject、tenant、action、resource scope、context 与 policy version，产出 allow/deny 和强制 filter。
工作机制
策略引擎把角色、属性、委托和时效组合为候选前过滤条件，并把 decision_id 交给检索与审计链。
必须成立的约束
策略输入不可由 prompt 篡改，过滤在召回前执行，缓存键包含权限范围；否则未授权内容可能进入模型和日志。
关键难点
动态撤权与低延迟存在张力，策略缓存需结合短 TTL、版本和主动失效，而非假设会话期间权限不变。
边界与注意事项
授权回答“能否看”，不回答“是否相关或真实”；误拒会制造空结果，误放则是安全事件而非普通召回误差。

05 · 检索与证据组装

检索产出带来源的候选证据而非答案，相关性只用于选择，不能替代真实性与支持关系。

职责与输入输出
知识层接收受权请求与 active index，产出排序候选及 EvidenceChunk 集，保留 source、version、locator、score 和 ACL。
工作机制
query 理解后由多路召回、融合、去重、重排与 context packing 逐级缩小候选，并记录纳入或移除原因。
必须成立的约束
证据元数据不能在拼装时脱落，最终 context 只含当前允许版本；空、冲突、过期结果必须成为显式状态。
关键难点
覆盖率、前排精度、多样性、延迟和 token 预算互相制约，K 与阈值应由真实查询分桶评测决定。
边界与注意事项
检索器不负责让模型忠实使用证据，也不能用高分证明政策有效；漏召正确证据会让生成优化失去前提。

06 · 生成与验证

生成只能在允许证据范围内组织声明，验证必须逐条证明支持、引用与输出策略，而非只看流畅度。

职责与输入输出
推理内核接收问题与 EvidenceChunk，产出 claims、答案草稿和 evidence ids；治理组件产出支持、引用和敏感性判定。
工作机制
回答拆为可核验 claim，再建立 claim—evidence 映射，检查冲突、locator 和政策，失败时删减、澄清或拒答。
必须成立的约束
模型不能创造来源、改变 ACL 或把文档指令提升为系统指令；关键 claim 无直接支持时不得表达为确定事实。
关键难点
完整回答与保守拒答之间需按风险校准，验证深度也消耗延迟和预算，高风险声明应采用更严格门槛。
边界与注意事项
生成正确不证明检索最优，Grounding 也不等于外部世界绝对真实；上游选错版本会忠实生成错误结论。

07 · 类型化结果与 Trace

最终结果必须同时表达业务状态与证据链，空结果、拒绝、超时、冲突和系统故障不可合并成一个“没答案”。

职责与输入输出
接口层汇总阶段状态，产出 evidence_found、insufficient_evidence、access_denied、timeout、conflict 或 error 及 Trace。
工作机制
状态机保留首个失败阶段、组件版本、deadline 消耗和 candidate/claim 决策，编排据此重试、澄清、降级或终止。
必须成立的约束
状态语义跨调用方稳定，Trace 脱敏但能回放，版本在请求开始时钉住；否则指标与事故归因会混淆。
关键难点
可观测性与隐私、存储成本之间需要取舍，采样不能丢掉高风险失败，字段演进也要保持历史可读。
边界与注意事项
Trace 是观测证据而非自动根因；它由治理和评测消费，但失败 owner 仍属于最先违反契约的具体层。

难点与取舍：边界、预算和证据义务

取舍不能牺牲的不变量决定依据
更宽召回 vs. 更低延迟权限过滤、版本和 locator 不因降级丢失查询风险、Recall 曲线、p95 与 token 预算
更完整验证 vs. 更快返回高风险 claim 必须有直接支持声明风险、错误成本和阶段 deadline
复现能力 vs. Trace 成本首个失败阶段与关键版本必须保留审计要求、采样策略与隐私等级

失败如何跨层传播

  1. 供给错误：旧制度被发布为 active，在线检索和生成可能都正确工作，却忠实输出过期结果。
  2. 控制错误：入口误路由或授权误放改变候选空间，后续分数与引用无法恢复已破坏的边界。
  3. 证据错误：locator、版本或 ACL 在组装中丢失，让答案看似合理但无法引用、撤权或审计。
  4. 状态错误：把 access_denied 当 empty 会诱发扩大检索或强答，掩盖 owner 并制造安全风险。

贯穿案例 01：请求、结果与 Trace

教学问题为“上海地区 P6 员工当前出差住宿上限是多少？”。主体 employee_42、租户 acme、地区 Shanghai、职级 P6、查询时间 2026-07-20T10:00+08:00 和制度版本约束都必须进入链路。以下字段只演示中间数据形状，不代表真实企业制度、通用参数或生产默认值。

RetrievalRequest {
  query: "出差住宿上限", subject: "employee_42", tenant: "acme",
  filters: { city: "Shanghai", job_level: "P6", effective_at: "2026-07-20" },
  index_version: "travel-policy-2026-07", deadline_ms: 900, trace_id: "rag-demo-01"
}
RetrievalResult { status: "evidence_found", evidence_ids: ["policy:v7:sec-4.2"], conflicts: [] }
Trace { route: "rag", policy: "allow:pd-81", stages: ["retrieve", "rerank", "verify"] }

如果主体无权访问，结果应是 access_denied；如果允许范围内没有适用版本，则是 insufficient_evidence，两者不能被空数组吞并。

工程要点

  1. 离线链路负责 connect、parse、normalize、chunk、index、publish；在线链路负责 authorize、retrieve、rerank、pack、generate、verify。
  2. 证据内容与来源元数据必须一起流动，不能在拼 prompt 时丢失 source_id、版本和 ACL。
  3. 检索超时、空结果、权限拒绝和证据冲突应是类型化状态，由编排决定澄清、拒答或降级。
  4. 对每次请求记录索引版本与策略版本，确保线上回答可复现。

常见坑

  • 把 RAG 当一次向量查询：忽略摄取质量、权限、重排、引用和评测。
  • 让推理内核直连索引：绕过编排预算和治理策略，trace 也无法闭环。
  • 混淆记忆：会话偏好不是企业手册；外部文档也不应自动变成用户经历。
  • 空结果仍强答：正确行为通常是澄清、拒答或说明证据不足。

理论追问

请先脱稿回答，再展开参考答案核对边界、证据和取舍。

  1. RAG 检索结果为什么默认不应写入长期 Memory？

    查看参考答案

    RAG 结果是带来源、权限、版本和时效的外部知识，本次只应进入 Working Context；Memory 保存经过价值、主体、同意和生命周期治理的 Agent 经历。自动写入会把外部文档变成无来源的“个人事实”，造成污染和越权复用。

  2. RAG 的离线摄取流水线和在线查询流水线如何分工？

    查看参考答案

    离线负责来源接入、清洗、切分、元数据、ACL、索引和版本发布；在线负责身份过滤、查询理解、召回、重排、证据组装、生成与引用校验。二者通过索引版本和数据契约连接，不能在请求时临时摄取未治理资料。

  3. 访问控制为什么必须在检索阶段前置？

    查看参考答案

    若先召回后仅在生成时过滤，未授权片段已经进入重排、日志或模型上下文，仍可能泄露。应将租户、部门、主体和资源 ACL 纳入候选过滤，并在输出与审计处再次校验，形成纵深防御。

  4. 没有足够证据时，RAG 系统应返回什么？

    查看参考答案

    应返回结构化的 evidence_insufficient，包含查询、过滤条件、已查来源和缺口，而不是强迫模型补全。编排可据此澄清、扩大受权检索、调用其他数据源或安全拒答。

  5. 用户收到错误答案时，如何区分摄取、检索、重排和生成问题？

    查看参考答案

    查看索引版本和原文是否正确、相关文档是否进入候选、重排是否把正确证据排后、最终回答是否忠于选中证据。Trace 要保留 query、filters、candidate IDs/scores、selected chunks、引用和生成版本，找到首个偏离点。

面试表达

如何定义生产级 RAG？

我把 RAG 定义为一条证据供应链：离线把外部资料按版本、结构和 ACL 建成可回滚索引；在线以用户身份约束检索，经过召回、重排和上下文组装，把带来源的证据交给推理内核，再校验声明和引用。编排拥有调用时机与降级，治理拥有策略，评测消费 trace；记忆系统单独管理 Agent 经历。

日常实践关键点

阅读清单前，先用“上海地区 P6 员工当前出差住宿上限”回忆本章的路由、请求契约、授权、证据、验证与类型化结果链；再追问身份、时间、版本或 deadline 哪条约束先失效，以及 Trace 能否证明它。

核心判断

先判断请求是否真的需要外部知识，再让身份、版本、预算和证据契约贯穿在线链路；生产级 RAG 的关键不是“检索到了文本”，而是每个阶段都有明确状态、owner、截止时间和可复现证据。

日常实践清单

  • 先做适用性路由：当问题依赖可更新、可引用的外部资料时进入 RAG；个人偏好交给 Memory，实时交易或账户真值走受控工具，避免把所有问题都转成相似度检索。

    执行动作

    验证标准
    四类请求做路由回归，确认账户真值不会误入相似度检索。
    止损与取舍

    上游收到空列表，无法判断该重试、拒答、提示无权限还是报警。

    迁移期保留旧适配器并监控未知状态，不以默认空结果换兼容。

    触发信号
    端到端 p95 越线，或上游取消后下游仍持续消耗资源。
    判断与定位
    从 Trace 比较各阶段耗时、排队时间和剩余预算。
    执行动作
    按 SLO 分配阶段预算，向下传绝对 deadline 与取消信号。
    验证标准
    注入慢依赖，确认状态可区分、调用被取消且总预算不越线。
    止损与取舍
    重排超时可退回首排候选，授权超时必须拒绝而不能放行。

    说明预算由阶段价值、尾延迟和失败安全边界共同决定。

    跨服务后地区、职级或时间约束消失，引用无法回到确定版本。
    判断与定位
    逐跳比对 request 与 evidence，找到字段被默认或丢弃的边界。
    执行动作
    把主体、约束、版本、locator、ACL 与 freshness 设为必填。
    验证标准
    缺字段快速失败，端到端能从答案引用反查原文快照。
    止损与取舍
    新旧 schema 双读只写新版，安全字段缺失时禁止宽松默认。
    面试转述
    用约束不能只存在 prompt 解释跨层契约和可复现性。
  • 钉住可复现版本：在请求开始时记录 corpus、index、retriever、reranker、prompt 和 policy 版本；灰度期间同一 Trace 不能悄悄跨版本，否则线上错答无法重放。
    触发信号
    同一请求重放候选不同，灰度问题也无法确定变化来源。
    判断与定位
    比对版本矩阵与别名解析时间，寻找请求中途切版环节。
    执行动作
    入口固定 corpus、index、retriever、prompt 与 policy 版本。
    验证标准
    保存输入与快照重放，候选和策略判定应在容差内一致。
    止损与取舍
    回滚只切新请求别名，存量请求保持固定版本避免链路撕裂。
    面试转述
    强调可观测不等于可复现，并讲清联合版本矩阵。
  • 明确失败 owner：知识层负责候选与证据契约，治理层负责策略判定，编排层负责重试和降级，推理层负责忠实使用证据；告警按首个失败阶段派发，避免统一归给模型团队。
    触发信号
    工单反复转派，或所有错答都被笼统归因给模型团队。
    判断与定位
    按 first divergence 判断未授权、未召回、未选中或未使用。
    执行动作
    为阶段状态建立 RACI、告警路由、升级时限与交接证据。
    验证标准
    演练五类失败，告警一次到达正确 owner 且证据足够。
    止损与取舍
    归因不清先由事件负责人止损，冻结多阶段并发调参。
    面试转述
    结合九层职责和首个偏离点回答谁负责、谁升级。
  • 用场景验收生产链路：发布前同时验证可回答、不可回答、权限拒绝、过期冲突和阶段超时，并在质量、p95 延迟、成本与 hard fail 均满足门禁后才放量。
    触发信号
    离线平均分提升，但权限、冲突或超时场景尚未覆盖。
    判断与定位
    检查验收矩阵是否只有 happy path，是否遗漏切片门禁。
    执行动作
    建立五类正负场景并绑定质量、延迟、成本和安全阈值。
    验证标准
    每类场景有预期状态，灰度无 hard fail 且指标在门限内。
    止损与取舍
    安全门禁失败立即回滚，普通波动先暂停扩量保留样本。
    面试转述
    按场景矩阵、指标门禁、灰度和自动止损讲验收流程。

排障顺序

收到错答或空答时，按“是否应该检索 → 当前主体是否被授权 → 正确资料是否进入召回候选 → 是否被融合与重排选中 → 是否进入最终上下文并被回答使用”逐步比对 Trace。找到第一个偏离点后只修改该阶段；若入口路由错误，就不要用扩大 Top-K 或改 prompt 掩盖边界问题。

应留证据

保留版本化 request/result schema 及契约测试、携带主体与过滤条件的阶段 Trace、各阶段 deadline 与 p50/p95 看板、类型化失败计数、版本矩阵、场景验收报告和灰度回滚记录。一次修复至少能回答“哪一版、哪一阶段、由谁处理、指标如何恢复”。

代表性工程产物：类型化结果与 deadline Trace
request_id=rag-7f21 | route=rag_policy | versions=travel-v42/idx-19/acl-8 | deadline_ms=total:1800,auth:100,retrieve:350,rerank:280 | result=insufficient_evidence | first_divergence=retrieve | constraints=region:上海,grade:P6

评审时同时核对固定版本、阶段剩余预算、约束保真和类型化状态；示例数值只展示字段关系，实际阈值必须由本系统 SLO 与基线校准。

日常面试要点

关联实践：适用性路由、类型化结果

高频问法

什么场景应该用 RAG，什么场景不该用；检索不到、无权限与系统超时如何区分？

60–90 秒骨架

先按外部知识、时效、引用和副作用判断数据源，再讲入口路由；最后用封闭状态枚举说明失败语义如何驱动不同编排分支。

必讲证据

展示路由决策表、request/result schema、状态分布与契约测试，说明不是靠 prompt 猜测应该走哪条链路。

常见追问

Memory 与 RAG 如何分界？实时账户真值为什么走工具？低置信路由如何降级？空结果能否重试？

回答边界

未做过生产路由时应明确是个人实验或方案设计；不要声称虚构的流量规模、误路由下降比例或线上收益。

关联实践：阶段 deadline、失败 owner

高频问法

如何把端到端延迟预算拆到 RAG 各阶段，并在错答或超时时快速找到责任边界？

60–90 秒骨架

从 SLO 和阶段价值分配绝对 deadline，向下传取消信号；再以 first divergence 和九层职责做归因、告警与升级。

必讲证据

给出阶段耗时 Trace、剩余预算、超时状态、RACI 和故障演练结果，证明降级与 owner 可以被验证。

常见追问

为什么不能平均切预算？授权超时能否降级？重排超时怎么办？多阶段同时异常时谁先接手？

回答边界

没有真实 SLO 数据时只解释方法和假设，不编造 p95、MTTR 或团队职责；参数需要通过线上基线校准。

关联实践：请求证据契约、版本固定、生产验收

高频问法

一条 RAG 链路上线前如何验收，怎样保证一次错答可以重放并安全回滚？

60–90 秒骨架

先固化请求与证据契约，再固定数据、检索、prompt 和策略版本；用正负场景矩阵、切片门禁、灰度和自动止损完成发布。

必讲证据

展示版本矩阵、可答与拒权用例、hard fail 门禁、回放记录和回滚 receipt，说明结论有可追溯产物。

常见追问

版本别名何时解析？跨版本请求如何处理？哪些指标必须 hard fail？回滚后如何确认恢复？

回答边界

只讲自己实际负责或验证过的环节；没有生产发布经历时标注方案设计，不能把模板门槛包装成公司指标。

经验真实性：回答时明确标注“生产实战、个人实验或方案设计”，只陈述自己实际负责且能出示证据的部分；未覆盖线上规模、事故、指标提升或业务收益时直接说明，不得用本页示例数值虚构项目经历。
