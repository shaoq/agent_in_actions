04 Grounding、引用与访问控制

学习目标

  • 区分 relevance、groundedness、citation correctness 与 completeness。
  • 把回答拆成可验证 claims，并绑定可定位 evidence。
  • 设计 pre-retrieval enforcement 与 post-generation verification。
  • 防止越权召回、引用泄露和检索内容中的 prompt injection。

核心概念

链路总览：可见、相关、支持与可追溯是不同证明

质量维度问题失败示例
Relevance证据与问题相关吗？召回同主题但不同年份制度
Groundedness声明能由证据推出吗？证据只说上限，回答擅自给出资格条件
Citation correctness引用真的支持对应声明吗？引用存在但定位到无关段落
Citation completeness所有可核验重要声明都有引用吗？金额有引用，例外条件无引用

Relevance 衡量证据与问题是否相关；Groundedness 衡量回答 claim 能否由给定证据推出；factual correctness 还要比较外部世界或权威真值，忠于过期文档的回答可以 grounded 却不正确。Citation correctness 检查某个 locator 是否支持绑定 claim，citation completeness 检查所有重要可核验 claim 是否都被覆盖。权限则更早回答“当前主体能否接触该证据”。这些维度不可互相替代，必须在不同接口留证。

逐环节拆解

01 · 策略执行

模型可以描述访问需求，但授权与强制过滤必须由可信策略组件执行，且模型永远不能扩大许可集合。

职责与输入输出
治理层接收可信 subject、tenant、resource、action、context 和 policy version，产出 allow/deny、filter 与 decision_id。
工作机制
策略把角色、属性、委托、地域和时效转换为服务端候选约束；检索服务只接受收窄，不接受客户端放宽。
必须成立的约束
身份不从 prompt 读取，授权先于召回，缓存键包含权限等价类和策略版本；默认拒绝缺失元数据的资源。
关键难点
细粒度策略提升安全却增加过滤与缓存复杂度，粒度取决于数据敏感性、撤权时效和查询性能。
边界与注意事项
Policy 证明可见性，不证明 relevance 或真实性；误放是安全事件，不能用生成后脱敏作为等价补救。

02 · 不可信证据边界

检索文本即使来自受权文档也只是数据，不能改变系统指令、权限、工具能力或输出策略。

职责与输入输出
上下文边界接收允许 EvidenceChunk，产出带来源标签、内容边界和风险标记的 untrusted evidence 区域。
工作机制
系统指令与文档内容分槽，检测指令型模式并标记；工具调用仍走独立 schema、授权与人工门禁。
必须成立的约束
文档不能提升自身优先级、读取秘密或构造未授权动作；来源可信不等于其中每句话都可执行。
关键难点
正常操作手册也包含命令式语言，检测需要兼顾召回可用性与注入风险，并用对抗样本而非关键词黑名单校准。
边界与注意事项
隔离降低间接 Prompt Injection 风险，不验证事实；若模型可直接调用工具，文本边界仍需动作层二次授权。

03 · Claim Decomposition

回答必须拆成可独立核验的最小重要声明，否则一个引用会掩盖其中未被支持的金额、条件或例外。

职责与输入输出
声明分析接收答案草稿，产出 claim id、文本、风险等级、可核验性及所需证据类型，供支持判定使用。
工作机制
按事实谓词拆分主体、属性、值、时间与条件，例如金额上限和“超标需审批”必须成为两个 claim。
必须成立的约束
高风险数字、日期、资格和否定不能藏在复合句中；不可核验建议与事实声明要区分标注。
关键难点
拆得过细增加验证成本，过粗产生部分支持假阳性；粒度由错误后果和引用呈现需求决定。
边界与注意事项
Decomposition 只定义验证单元，不判断支持关系；漏拆的 claim 会直接造成 completeness 指标虚高。

04 · 支持关系与 Grounding

Grounding 要证明每个重要 claim 可由允许证据推出，相关、同主题或高相似度都不等于支持。

职责与输入输出
支持判定接收 claims 与最终 evidence，产出 direct、partial、conflict、unsupported 及对应 evidence ids 和理由。
工作机制
比较 claim 的主体、数值、条件、时间与证据语义；多片段联合支持时明确组合关系，关键事实优先要求直接支持。
必须成立的约束
只能使用当前 context 中允许且版本明确的证据，partial 不可升级为 direct，unsupported 必须删减、澄清或拒答。
关键难点
蕴含判定受省略、表格和跨段关系影响；门槛由 claim 风险、人工标注和错误成本校准。
边界与注意事项
Groundedness 不证明源文档事实正确；忠于旧版制度仍可能事实错误，需由权威性与时效规则补充。

05 · 引用生成与校验

引用是 claim 到稳定来源位置的系统生成映射，不是模型写出的装饰链接，也不是 Grounding 的替代品。

职责与输入输出
引用组件接收 claim—evidence 映射，产出 source_id、document_version、locator、展示标签和打开时授权要求。
工作机制
模型只选择 evidence id，系统从索引元数据组装 URL/页码/段落并验证 locator 确实覆盖绑定 claim。
必须成立的约束
来源位置稳定且版本明确，用户打开引用时重新授权；无法定位或只支持相邻内容的引用不能通过。
关键难点
动态网页、表格和文档改版会让 locator 漂移，需保存版本化锚点并按前端可达性做集成测试。
边界与注意事项
correctness 看引用是否支持，completeness 看重要 claim 是否都有引用；有链接不等于两者任一成立。

06 · 冲突、权威性与时效

证据冲突必须依据适用范围、来源权威和生效时间裁决或披露，绝不能把最高相似度当政策优先级。

职责与输入输出
冲突解析接收支持同一 claim 的多版本证据，产出 selected、superseded、unresolved 和可审计裁决理由。
工作机制
先比较 subject/region/level 适用性，再比较有效期、来源层级和废止关系；无法决定时并列展示并澄清或转人工。
必须成立的约束
规则必须版本化且供应商中立，分数只用于相关性；不得默默丢弃冲突证据或凭模型常识覆盖权威资料。
关键难点
来源可能缺少明确 supersedes 关系，自动裁决与保守披露的界线应由业务风险和人工升级成本决定。
边界与注意事项
冲突处理不修复上游脏索引；旧版仍被召回应回到摄取 owner，当前请求只做安全处置。

07 · 输出复核与使用点撤权

权限在检索后仍可能变化，证据进入上下文、返回答案和打开引用的每个使用点都要守住当前授权。

职责与输入输出
输出治理接收 claims、citations、敏感标签和当前 policy version，产出 allow、redact、re-retrieve、deny 或人工升级。
工作机制
比较请求开始与使用点策略版本，复核 evidence 可见性、敏感输出和引用访问；撤权事件主动失效缓存。
必须成立的约束
二次复核只能收紧不能扩大，旧 Trace 与已生成 URL 不能绕过当前 policy；TTL 只是失效兜底。
关键难点
多次鉴权增加延迟，权限等价缓存又有撤权窗口；设计由数据敏感度、撤权 SLO 与一致性模型决定。
边界与注意事项
输出复核是纵深防御，不证明召回前越权无害；未授权文本已进模型或 Trace 时仍应按安全事件处置。

难点与取舍：可见性、支持度、权威性与时效

问题错误捷径正确决策依据
证据相关但版本旧按最高相似度回答适用范围、生效时间、supersedes 与权威等级
回答大部分被支持用一个引用覆盖整段claim 风险、direct/partial 状态与 completeness
权限可能动态变化只在会话开始授权撤权 SLO、策略版本和使用点风险

安全与证据失败传播

  1. 召回前策略误放会让未授权文本进入重排、模型和 Trace，输出脱敏不能撤销已经发生的数据接触。
  2. 不可信文档中的指令若越过数据边界，可能诱导模型泄密或调用工具，后果跨越知识层进入动作层。
  3. claim 未拆分会让部分支持伪装成完整支持，引用正确性与完整性随之同时失真。
  4. 冲突只按分数裁决会把旧政策当新政策；引用即便准确指向旧文档，也不能证明最终事实正确。

贯穿案例 04：Policy、Claim 与 Citation

对“上海地区 P6 员工当前出差住宿上限是多少？”，治理层先以 employee_42、acme 和 employees-cn 得到 policy decision，再只把允许的 v7 证据交给推理。示例金额使用 ¥X 占位，明确不是实际制度或生产默认值。

policy: { decision: "allow", filter: "tenant=acme AND acl=employees-cn", version: "p19" }
claims:
  c1: "上海 P6 当前住宿上限为 ¥X/晚" -> direct [v7/sec-4.2/row-sh-p6]
  c2: "超出上限一律不予报销"       -> unsupported []
citation: { claim: "c1", source: "travel-policy", version: "v7", locator: "p12/table2/r3/c4" }
verify: { c1: "pass", c2: "remove", output_policy: "allow" }

如果 v6 与 v7 冲突，应依据生效和废止关系裁决，而非比较相似度；如果用户打开引用时已撤权，则重新授权失败并拒绝展示原文。

Citation Correctness 不等于 Semantic Support

引用正确性至少包含四层：validity（引用目标存在且主体可访问）、correctness（locator 指向声称的版本与片段）、completeness（关键 claim 都有引用）、semantic support（证据实际蕴含该 claim）。一个 URL 格式正确、甚至文本相关，也可能不支持结论。应拆分 atomic claims，建立 claim—evidence edge，记录 support 类型（direct/partial/conflict/none）、版本、适用范围和 verifier 证据。

{
  "claim_id": "c3",
  "claim": "上海地区 P6 员工当前住宿上限为……",
  "evidence": [{
    "document_id": "doc_01J...",
    "version_id": "ver_...",
    "content_id": "tenant-cn:sha256:...",
    "locator": "page=12;table=3;row=P6",
    "support": "direct"
  }],
  "decision": "answer | qualify | abstain"
}

LLM-as-Judge 只能是受校准的证据之一

Judge 适合规模化筛查 claim 支持、答案相关性和引用覆盖，但会受提示、模型版本、位置、长度与自偏好影响。先用人工标注集校准一致性和误差切片，高风险 hard fail 仍由确定性规则或人工复核；保存 judge prompt/model/version 和原始理由。最终策略按风险组合：证据直接且授权有效则回答；部分支持则限定范围；冲突则并列来源与适用条件；无支持、过期或权限不确定则 abstain/转人工。

Context Builder—Prompt—CitationChecker 三方契约

Context Builder 为每个授权证据分配不可由模型伪造的 evidence_id，绑定 document/version/content/chunk、locator、ACL 和原文范围；Prompt 只允许引用这些 evidence_id，并要求输出 atomic claim→evidence_ids；CitationChecker 只验证 ID 在 allowlist、locator/版本可解析、编号格式和关键 claim 是否漏标。它不证明内容支持 claim，语义支持由独立 verifier 处理。

授权变化与流式输出提交边界

  1. 检索前以服务端 subject/tenant/policy 生成强制 filter，未授权内容不得进入候选、cache 或 trace 正文。
  2. Context 使用前和 citation 打开时再次鉴权；cache key、trace 脱敏、引用代理都包含授权 scope，防旁路泄露。
  3. 撤权事件先更新 read-time deny/tombstone 并取消进行中的 request，再异步清索引与缓存；清理回执未齐全时保持安全终止。
  4. 高风险流式回答先在服务端缓冲到 claim/citation/support 与使用点鉴权完成，再提交该段；不能先向用户发送、后发现 unsupported 或 revoked。

工程要点

  1. 以 subject、tenant、resource、action、context 形成策略输入，得到强制 filter；任何缓存键也必须包含权限范围。
  2. 证据使用稳定 source_id、document_version 与 locator；UI 引用应打开主体仍有权访问的目标。
  3. 检索内容属于不可信数据：与系统指令隔离，拒绝执行其中“忽略规则、调用工具、泄露秘密”等命令。
  4. 对重要回答做 claim decomposition；无支持、冲突或过期声明应删去、标注不确定或拒答。

常见坑

  • 检索全库后在生成阶段脱敏：敏感内容已进入模型上下文和 trace。
  • 把“附了链接”当 grounded：链接可能相关但不支持具体声明。
  • 缓存不区分用户和租户，造成跨主体证据复用。
  • 把文档中的指令视为可信 system prompt，触发间接 prompt injection。

理论追问

请先脱稿回答，再展开参考答案核对边界、证据和取舍。

  1. 回答带有引用链接，为什么仍可能没有 Grounding？

    查看参考答案

    引用可能与具体断言不对应、来自过期版本，或只在文档中出现相似词。Grounding 要求每个关键断言可由所引证据支持，并校验来源、版本、片段范围和蕴含关系；“有引用”只是格式，不是事实保证。

  2. 权限过滤应放在召回前还是生成后？

    查看参考答案

    必须在召回前基于身份和 ACL 缩小候选，避免未授权内容进入重排、模型和 Trace；生成后仍需做输出检查，防止上下文组合或模型记忆泄露。两处是纵深防御，不是二选一。

  3. 检索文档中的 Prompt Injection 应如何处理？

    查看参考答案

    把外部文档明确标记为不可信数据，禁止其改变系统指令或授权；抽取证据时保留来源并检测可疑指令，工具调用由编排和治理独立校验。不能让“文档说要发密钥”获得指令优先级。

  4. 证据之间冲突时，系统应直接选择分数最高的片段吗？

    查看参考答案

    不应。要比较来源权威性、生效时间、适用主体和版本，必要时同时展示冲突并澄清或转人工。检索分数表示相关性，不代表真实性或政策优先级；冲突处理规则应可审计。

  5. 多租户 RAG 如何证明没有跨租户泄露？

    查看参考答案

    索引和查询携带不可由模型覆盖的 tenant_id，服务端在候选生成前强制过滤，凭据和缓存按租户隔离。通过 canary 文档、负向权限用例、Trace 中的 policy decision 和定期越权评测提供证据。

面试表达

如何防止 RAG 越权与幻觉？

越权和幻觉要分层处理：先由治理层按主体与资源生成不可被模型覆盖的过滤条件，在检索前强制执行，并让 ACL 随 chunk 和缓存传播；再把证据作为不可信数据交给推理内核，要求声明绑定 source_id 与 locator，输出侧校验 groundedness、引用完整性和敏感信息。证据不足就拒答，引用链接也要二次鉴权。

日常实践关键点

阅读清单前，先用上海 P6 差旅回答回忆“策略过滤 → 不可信证据 → claim 拆分 → 支持判定 → 系统引用 → 冲突时效 → 使用点复核”；再分别证明当前主体能看、每条 claim 被支持、locator 正确且撤权后不可再用。

核心判断

先证明当前主体有权看证据，再证明证据确实支持回答中的具体 claim；权限与 Grounding 都必须由系统可审计地执行，不能依赖模型自觉过滤、自由生成链接或用相关性分数代替事实支持。

    核对 evidence ID、document_version、locator 与最终引用映射。
    执行动作
    模型只返回 evidence ID，由受控服务组装并校验引用目标。
    验证标准
    每条引用可定位到固定版本原文，伪造和失效 locator 被拒绝。
    止损与取舍
    定位失败就移除对应 claim 或拒答，不显示看似可信的链接。
    面试转述
    区分引用格式正确与引用内容真实、可定位、当前可访问。
  • 维护 claim—evidence 映射：把关键断言拆开并标记 direct、partial、conflict 或 unsupported；对金额、日期、资格等高风险 claim 要求直接支持，部分支持不能被整体流畅度掩盖。
    触发信号
    整段答案看似有引用，但金额、日期或资格条件并未被支持。
    判断与定位
    拆成最小可验证 claim，逐条标记 direct、partial、conflict。
    执行动作
    高风险 claim 要求直接证据，unsupported 删除或触发澄清。
    验证标准
    人工与自动检查均可复现每个关键 claim 的支持关系。
    止损与取舍
    完整性与流畅度冲突时优先忠实，明确披露证据不足。
    面试转述
    用逐 claim 验证说明 groundedness 不等于整段相似度。
  • 在召回前强制授权：用 subject、tenant、resource、action 与 context 生成服务端 filter 和 policy decision；模型、改写器和客户端参数都不能扩大允许资源范围。
    触发信号
    候选 Trace 出现跨租户资源，或权限过滤由客户端自由传入。
    判断与定位
    重放 subject、tenant、action、context 与 policy_version。
    执行动作
    策略服务生成服务端 filter，客户端条件只能进一步收窄。
    验证标准
    跨租户 canary 永不进入候选，篡改 filter 的请求被拒绝。
    止损与取舍
    策略不可用默认拒绝高敏请求，不以检索后过滤替代前置授权。
    面试转述
    强调召回到内存也可能构成泄露，权限必须先于检索。
  • 按权限范围设计缓存：缓存键包含主体或等价权限集合、tenant、policy_version 与资源版本；收到撤权事件时主动失效，并以短期 TTL 作为兜底而非唯一保证。
    触发信号
    用户撤权后仍命中旧答案，或不同权限主体共享同一缓存键。
    判断与定位
    检查键空间、policy_version、资源版本和失效事件 receipt。
    执行动作
    缓存按等价权限集合隔离，撤权事件主动失效并保留短 TTL。
    验证标准
    撤权负测在传播 SLO 内失效，跨主体请求不复用越权结果。
    止损与取舍
    失效链路异常先停高敏缓存，接受延迟上升以守住权限边界。
    面试转述
    说明 TTL 只是兜底，不能证明撤权即时生效。
  • 在使用点二次鉴权：证据进入上下文前复核可见性，用户打开引用时再次按当前身份授权；发现策略版本已变化就重新检索或拒绝，避免旧 Trace 和旧链接绕过撤权。
    触发信号
    检索和展示间隔较长，或旧会话中的引用在撤权后仍可打开。
    判断与定位
    比较检索、上下文组装、引用打开三个时点的策略版本。
    执行动作
    证据使用前和引用打开时按当前身份重新执行策略判定。
    验证标准
    在请求中途注入撤权，旧证据不会进入回答或被链接读取。
    止损与取舍
    策略变化则重新检索或拒绝，不能为了会话连续性沿用旧许可。
    面试转述
    用 TOCTOU 风险解释为何检索时授权一次还不够。
  • 隔离间接 Prompt Injection：把文档标为不可信数据，检测“忽略规则、调用工具、泄露秘密”等指令型内容；工具授权仍由编排与治理执行，检索文本不能提升自身优先级。
    触发信号
    检索文档含忽略规则、读取秘密或调用工具等指令文本。
    判断与定位
    区分业务事实和可执行指令，查看其是否影响工具或系统边界。
    执行动作
    证据标记为不可信数据，工具调用始终重新经过治理授权。
    验证标准
    注入样本不能改变系统指令、读取秘密或提升工具权限。
    止损与取舍
    检测不确定时隔离文档并降级回答，不自动执行其中动作。
    面试转述
    说明提示词提醒不是权限控制，安全边界必须在系统层。
  • 测试冲突与越权负例：证据冲突时按权威来源、生效时间和适用范围裁决或显式披露，不按最高相似度拍板；持续运行跨租户 canary、伪造引用、撤权后访问和注入样本。
    触发信号
    新旧制度同时命中，或安全变更后缺少持续越权负测。
    判断与定位
    比较来源权威性、生效时间、适用范围与 policy decision。
    执行动作
    按显式规则裁决或披露冲突，并持续运行 canary 与攻击样本。
    验证标准
    冲突样本状态稳定，跨租户、伪造引用和注入全部 hard fail。
    止损与取舍
    无法裁决就拒答或人工升级，不以相似度最高替代权威判断。
    面试转述
    用负测证明安全和冲突策略，而不是只描述 happy path。

排障顺序

遇到泄露或无依据断言时，先重放主体、租户和 policy_version，确认资源对当前身份是否可见；再检查候选证据的 ACL、版本和 locator 是否合法，然后逐条验证 claim 是否由所选证据支持，最后复核输出脱敏和引用打开时的二次授权。第一步已越权时立即按安全事件处置，不继续用生成侧过滤证明安全。

应留证据

保留不可由模型覆盖的 policy decision、候选与最终证据的 ACL 快照、claim—evidence 矩阵、引用生成记录、跨租户 canary 与安全负测结果、撤权事件和各缓存/索引失效 receipt、引用二次鉴权日志，以及冲突裁决和人工升级审计记录。

代表性工程产物：policy decision 与 claim—evidence 审计
subject=employee-17 | tenant=cn | action=read | policy=acl-8 | decision=allow | resource=travel-v7#table3-p6 | claim=上海P6住宿上限适用 | support=direct | citation_locator=page12/table3/rowP6 | reauth=allow | revocation_receipts=index:ok,cache:ok | canary_leak=0

该记录把授权、支持关系、引用定位和撤权验证串在一起；示例身份与结果均为教学占位，生产环境必须脱敏并限制审计日志访问。

参考知识回扣：逐 Claim 支持审计

上线抽样不要只看“有无引用”，而要逐 claim 核对引用有效性、版本、locator、语义支持和授权二次校验；分别统计 citation correctness 与 semantic support。面试时明确 Judge 的校准集、误差和人工兜底，避免把模型分数当事实裁决。

日常面试要点

关联实践：召回前授权、权限缓存、使用点鉴权

高频问法

多租户 RAG 如何防止越权，为什么检索前做过 ACL 过滤后仍要在使用点再次鉴权？

60–90 秒骨架

策略服务以主体和上下文生成不可扩大的 filter；缓存按权限集合隔离；上下文组装和引用打开时重鉴权以处理撤权与 TOCTOU。

必讲证据

展示 policy decision、跨租户 canary、缓存键、撤权 receipt 和中途撤权负测，证明边界覆盖完整时序。

常见追问

策略服务超时怎么办？TTL 能否保证撤权？共享缓存如何设计？召回后过滤是否足够？

回答边界

安全能力只能按真实测试范围陈述；未做渗透或生产多租户验证时，明确是方案设计而非合规结论。

关联实践：系统引用、claim—evidence

高频问法

怎样防止模型伪造引用，并判断一段回答中的每个关键断言是否真正有证据支持？

60–90 秒骨架

让模型只选择 evidence ID，由系统组装可定位引用；将回答拆为原子 claim，按 direct、partial、conflict、unsupported 验证。

必讲证据

给出 claim—evidence 矩阵、版本化 locator、失效引用负测和高风险 claim 的 hard fail 记录。

常见追问

引用正确等于答案正确吗？partial 如何处理？长答案如何拆 claim？原文更新后链接怎么办？

回答边界

自动 groundedness 不是绝对真值；应说明人工抽检与高风险规则，不虚构全自动零幻觉效果。

关联实践：注入隔离、冲突负测

高频问法

RAG 如何防御间接 Prompt Injection，检索到冲突的新旧制度时又依据什么裁决？

60–90 秒骨架

把检索文本视为不可信数据，工具权限由治理层独立判定；冲突按权威、生效时间、范围裁决，无法确定则披露或升级。

必讲证据

展示注入语料、工具越权负测、冲突样本、裁决审计与 canary 结果，说明规则持续进入发布门禁。

常见追问

仅靠 system prompt 可以吗？检测误报怎么办？最高相似度为何不可靠？何时需要人工升级？

回答边界

不承诺消除所有攻击；明确防御层、已测攻击面和残余风险，未实际红队测试时不能声称安全完备。

经验真实性：安全与引用问题必须注明是生产实战、个人实验还是方案设计；只陈述实际执行过的权限负测、撤权验证和 claim 审计，不能虚构泄露事故、攻击拦截率或合规认证。
