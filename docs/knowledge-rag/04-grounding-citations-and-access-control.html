04 Grounding、引用与访问控制

学习目标

  • 区分 relevance、groundedness、citation correctness 与 completeness。
  • 把回答拆成可验证 claims，并绑定可定位 evidence。
  • 设计 pre-retrieval enforcement 与 post-generation verification。
  • 防止越权召回、引用泄露和检索内容中的 prompt injection。

核心概念

质量维度问题失败示例
Relevance证据与问题相关吗？召回同主题但不同年份制度
Groundedness声明能由证据推出吗？证据只说上限，回答擅自给出资格条件
Citation correctness引用真的支持对应声明吗？引用存在但定位到无关段落
Citation completeness所有可核验重要声明都有引用吗？金额有引用，例外条件无引用

工程要点

  1. 以 subject、tenant、resource、action、context 形成策略输入，得到强制 filter；任何缓存键也必须包含权限范围。
  2. 证据使用稳定 source_id、document_version 与 locator；UI 引用应打开主体仍有权访问的目标。
  3. 检索内容属于不可信数据：与系统指令隔离，拒绝执行其中“忽略规则、调用工具、泄露秘密”等命令。
  4. 对重要回答做 claim decomposition；无支持、冲突或过期声明应删去、标注不确定或拒答。

常见坑

  • 检索全库后在生成阶段脱敏：敏感内容已进入模型上下文和 trace。
  • 把“附了链接”当 grounded：链接可能相关但不支持具体声明。
  • 缓存不区分用户和租户，造成跨主体证据复用。
  • 把文档中的指令视为可信 system prompt，触发间接 prompt injection。

理论追问

请先脱稿回答，再展开参考答案核对边界、证据和取舍。

  1. 回答带有引用链接，为什么仍可能没有 Grounding？

    查看参考答案

    引用可能与具体断言不对应、来自过期版本，或只在文档中出现相似词。Grounding 要求每个关键断言可由所引证据支持，并校验来源、版本、片段范围和蕴含关系；“有引用”只是格式，不是事实保证。

  2. 权限过滤应放在召回前还是生成后？

    查看参考答案

    必须在召回前基于身份和 ACL 缩小候选，避免未授权内容进入重排、模型和 Trace；生成后仍需做输出检查，防止上下文组合或模型记忆泄露。两处是纵深防御，不是二选一。

  3. 检索文档中的 Prompt Injection 应如何处理？

    查看参考答案

    把外部文档明确标记为不可信数据，禁止其改变系统指令或授权；抽取证据时保留来源并检测可疑指令，工具调用由编排和治理独立校验。不能让“文档说要发密钥”获得指令优先级。

  4. 证据之间冲突时，系统应直接选择分数最高的片段吗？

    查看参考答案

    不应。要比较来源权威性、生效时间、适用主体和版本，必要时同时展示冲突并澄清或转人工。检索分数表示相关性，不代表真实性或政策优先级；冲突处理规则应可审计。

  5. 多租户 RAG 如何证明没有跨租户泄露？

    查看参考答案

    索引和查询携带不可由模型覆盖的 tenant_id，服务端在候选生成前强制过滤，凭据和缓存按租户隔离。通过 canary 文档、负向权限用例、Trace 中的 policy decision 和定期越权评测提供证据。

面试表达

如何防止 RAG 越权与幻觉？

越权和幻觉要分层处理：先由治理层按主体与资源生成不可被模型覆盖的过滤条件，在检索前强制执行，并让 ACL 随 chunk 和缓存传播；再把证据作为不可信数据交给推理内核，要求声明绑定 source_id 与 locator，输出侧校验 groundedness、引用完整性和敏感信息。证据不足就拒答，引用链接也要二次鉴权。
