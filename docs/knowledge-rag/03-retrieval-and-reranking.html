03 检索与重排

学习目标

  • 区分 lexical、dense、metadata 与 graph retrieval 的信号。
  • 解释 query rewrite、hybrid fusion、dedup、rerank 与 diversity 的顺序。
  • 根据 recall、precision、latency 和成本设定候选漏斗。
  • 为低置信、冲突和无结果设计受控分支。

核心概念

链路总览：候选漏斗中的目标逐级变化

阶段主要目标典型指标关键 trace
Query保留意图、补充可检索表达rewrite consistency原查询、改写、过滤条件
Recall候选覆盖Recall@K、MRR各检索器候选与原始分数
Rerank有限预算内提高精度nDCG@K、Precision@K重排前后次序、模型版本
Pack形成可读、去重、多样的证据context relevance、token cost截断、去重和最终 chunk ids

检索不是一次相似度排序，而是受身份、时间和结构约束的候选漏斗。Query 阶段保护意图；召回阶段优先让正确证据进入候选；融合把不可比分数空间转成共同次序；重排在小集合上提高前排质量；packing 才决定模型实际看见什么。每一步都可能让 Gold Evidence 消失，因此必须记录候选级输入、输出和移除原因，而不能只保存最终 Top-K。

逐环节拆解

01 · Query 理解

Query 理解先分离用户想问的语义与不可改写的硬约束，任何扩展都不能改变主体、否定、实体和时间。

职责与输入输出
查询层接收原问题和可信请求上下文，产出意图、实体、关键词、语言、query type 及锁定的 metadata filters。
工作机制
通过规则、词典或模型抽取“住宿上限”、上海、P6、当前等成分，并把文本信号与权限 filter 分开编码。
必须成立的约束
subject、tenant 和 ACL 不从自然语言推断；日期、否定与编号必须保真，否则候选空间从入口就发生漂移。
关键难点
短查询歧义大，过度结构化可能误锁条件；解析策略需按查询桶的下游 Recall 与约束保持率校准。
边界与注意事项
Query 理解描述需求，不决定资料真假或授权；抽取错误会同时污染多路检索，难以由后续融合恢复。

02 · 改写与分解

改写增加可检索表达但保持问题等价，分解只拆依赖关系，不得悄悄补入原问题没有的前提。

职责与输入输出
改写器接收结构化 query，产出同义表达、关键词变体或带依赖的子查询，并保留原查询与 diff。
工作机制
为语义检索补充等价说法，为词法检索保留制度名、城市和职级；复合问题可先查适用政策再查对应标准。
必须成立的约束
硬约束以不可覆盖字段传递，改写数量受预算控制；每个子查询能追溯回原意图和合并规则。
关键难点
扩展提高覆盖也引入语义漂移和延迟，是否改写与展开几路应由 rewrite consistency 与端到端收益决定。
边界与注意事项
改写不是回答，也不能替代 metadata filter；把“当前”丢掉会召回历史制度并传播为时效冲突。

03 · 多路候选召回

不同检索信号覆盖不同失败模式，多路召回追求候选覆盖，但每一路都必须执行同一授权与版本边界。

职责与输入输出
召回器接收改写 query、filters 和 index version，分别产出带 channel、rank、raw score 的候选列表。
工作机制
lexical 匹配编号和精确术语，dense 覆盖同义语义，metadata 执行地区/时间等结构过滤，graph 连接显式实体关系。
必须成立的约束
所有通道共享稳定 chunk_id 与权限过滤，raw score 保留原空间含义；未授权候选不得先召回再删除。
关键难点
更多通道提高 Recall 也增加重复、延迟和维护成本，通道选择与 K 应按实体型、语义型等查询桶决定。
边界与注意事项
metadata 过滤不是相关性排序，graph 也不自动保证事实；某通道无增益时应有消融证据再移除。

04 · Fusion 融合

Fusion 合并排名信号，但不能证明证据真实，也不能替代对 query—document 交互的 Rerank。

职责与输入输出
融合器接收多路候选及各自 rank/score，产出统一候选次序和每一路的贡献说明，供去重与重排消费。
工作机制
RRF 常用 score(d)=Σ 1/(k+rank_i(d))，通过名次而非 raw score 合并；k 控制头部名次差异的影响。
必须成立的约束
缺席与低排名语义要明确，稳定 ID 用于合并；不同模型 raw score 不得未经校准直接相加。
关键难点
通道权重与 RRF k 会改变覆盖和前排，参数必须在目标查询、版本和 latency 预算上评测，不能照搬常数。
边界与注意事项
score 是算法输出，rank 是相对次序，二者通常都不是概率；融合高位只说明多路一致或某路靠前。

05 · 去重与多样性控制

去重压缩相同证据，多样性保留互补证据；两者都不能把独立来源的冲突误删成“一致”。

职责与输入输出
候选整形接收融合列表与 lineage，产出去除精确重复、标记近重复且控制来源/章节多样性的候选集。
工作机制
先按稳定 chunk/content hash 合并，再用文档与父子关系聚类；在预算内限制同一来源的相似块占比。
必须成立的约束
去重保留 canonical、来源集合和被合并 ID；不同版本、权威来源或适用范围不可只因文本相似被吞掉。
关键难点
过强去重可能漏掉例外条款，过弱则浪费 token 并制造虚假多证据；阈值按任务和 lineage 调整。
边界与注意事项
多样性不等于相关性，也不替代冲突裁决；错误合并会让下游看不到真实的版本或政策分歧。

06 · Rerank 重排

Rerank 在有限候选上估计 query—evidence 相对适配度，提高前排精度，但其分数不是事实概率。

职责与输入输出
重排器接收 query 与候选文本/元数据，产出新次序、模型版本、score 和截断信息，供阈值与 packing 使用。
工作机制
通过 query—document 联合交互捕捉局部相关性，通常只处理召回后的较小集合；可加入领域特征或分桶模型。
必须成立的约束
输入候选不能越权，文本截断策略固定且可追踪；跨模型或版本的 score 不可直接比较或共用阈值。
关键难点
候选数、模型能力、延迟和成本相互制约，nDCG 提升也需确认没有损害关键 Gold Evidence 的保留。
边界与注意事项
Rerank 无法召回上游遗漏的证据，也不验证真实性；位置偏差可能把长文或关键例外压低。

07 · 阈值、拒答与受控降级

阈值把连续相关性信号转成业务动作，必须按分桶校准并输出显式状态，而不是伪装成通用置信概率。

职责与输入输出
决策层接收排序、分数、冲突、新鲜度和剩余预算，产出继续、扩展检索、澄清、拒答或受控降级。
工作机制
在标注校准集上寻找风险可接受的边界，并按查询类型、来源和模型版本维护阈值及灰区策略。
必须成立的约束
阈值绑定 score 定义和版本，高风险缺证或冲突不能被低风险平均值覆盖；降级不得绕过权限与引用义务。
关键难点
覆盖率与误答率此消彼长，分布漂移会让旧阈值失效，因此需持续观察拒答、纠错和切片质量。
边界与注意事项
阈值不是事实边界，低分也可能因索引缺陷；它触发动作但不自动解释根因，需候选 Trace 支撑。

08 · Context Packing

最终上下文是受 token、结构和证据义务约束的可读集合，进入候选不等于最终被模型看到。

职责与输入输出
组装器接收重排候选与预算，产出带 evidence id、来源边界、顺序和截断记录的最终 context pack。
工作机制
按关键证据覆盖、去重和多样性选块，必要时扩展父块或邻接片段，并显式标记每个来源的开始与结束。
必须成立的约束
不能截断金额与适用条件的组合，source/locator/ACL 必须随文本保留，文档内容作为不可信数据隔离。
关键难点
更多上下文提高覆盖也增加噪声、成本和位置效应；预算由任务风险、窗口和端到端质量曲线决定。
边界与注意事项
Packing 不负责生成或 Grounding 校验；若 Gold Evidence 在此被移除，最终错答看似像模型问题实为组装失败。

难点与取舍：覆盖、精度、多样性、延迟与 token

阶段增加预算的收益代价与决定依据
多路 Recall提升候选覆盖，降低单信号盲区重复与延迟；按 query bucket 的 Recall@K 决定
Rerank提高前排相对次序模型成本；按 nDCG、关键证据保留和 p95 决定
Context容纳互补与完整证据噪声和 token；按 grounded task quality 决定

指标边界：Recall@K 只问 Gold Evidence 是否进入前 K，不关心具体次序；MRR 只关注第一个相关结果的位置，忽略其后证据；nDCG 可表达多级相关性与整体次序，却依赖可靠 relevance label。三者都不能证明权限正确、证据真实或生成忠实。

候选漏斗的失败传播

  1. Query 改写丢失“当前”或 P6，所有通道会在错误约束下返回看似相关的历史制度。
  2. Gold Evidence 未入召回候选，Fusion 与 Rerank 无法创造它；扩大重排模型不能修复 corpus 或 recall 缺口。
  3. 直接相加 raw score 让某通道量纲支配次序，正确候选可能在 Fusion 被系统性压低。
  4. 去重或 packing 移除金额的适用条件，模型最终只看见半条事实，错误会被误归给 Generation。

贯穿案例 03：候选、融合与 Context

同一教学问题携带可信 filters：city=Shanghai、level=P6、effective_at=2026-07-20。Query 改写为“上海 P6 国内出差 酒店 住宿标准 上限”，但 filters 不允许被改写器覆盖。以下名次和分数只说明中间形状，不是生产默认阈值。

lexical: [v7/sec-4.2 rank=1, v6/sec-4.1 rank=2]
dense:  [v7/sec-4.2 rank=2, faq/travel rank=1]
filter:  v6 removed(reason=expired)
RRF:     v7/sec-4.2 rank=1; faq/travel rank=2
rerank:  v7/sec-4.2 score=0.73 rank=1
context: [v7/sec-4.2/heading, v7/sec-4.2/row-sh-p6, v7/sec-4.2/exception]

RRF 的 0.73 类分数不能解读成“73% 真实”。最终 pack 同时保留标题、上海 P6 行和例外条款，才构成下一章可用于 claim 支持的完整证据。

概念—实践—证据回扣

概念与约束失效信号对应实践验证证据
改写丢实体、否定、时间或身份约束保护 Query 硬约束原查询—改写 diff 与保持率
单一通道在特定查询桶持续漏召按查询类型选择信号通道级 Recall 与消融报告
raw score 量纲支配融合次序使用可解释的融合通道 rank 与融合贡献
旧阈值跨版本造成误答或过度拒答按分桶校准阈值校准曲线与拒答切片
重复块挤占互补证据或例外被误删控制去重与多样性合并 lineage 与最终 chunk ids
候选与 token 预算失控拖垮 p95管理候选漏斗预算各阶段 K、token、延迟和收益
Gold Evidence 消失却无法定位阶段保存候选级 Trace融合与重排前后完整次序

工程要点

  1. 访问过滤应作为强约束进入候选生成，而不是召回后仅靠 prompt 告诉模型忽略。
  2. hybrid retrieval 可用 RRF 等 rank fusion 合并不同分数空间，避免直接相加不可比的 raw score。
  3. 对实体编号、错误码和专有词保留关键词检索；对改写和语义同义表达使用 dense retrieval。
  4. 阈值由校准集决定；低分、冲突或证据过时应返回显式 insufficient_evidence。

常见坑

  • 只调 Top-K，不看候选来自哪一路和相关文档是否在语料中。
  • 把 reranker 分数当概率，跨模型或跨版本使用同一阈值。
  • 查询改写丢失日期、否定、实体或租户约束。
  • 塞入大量相似 chunk，既浪费 token 又形成虚假“多证据”印象。

理论追问

请先脱稿回答，再展开参考答案核对边界、证据和取舍。

  1. 为什么召回率和精确率不能只优化一个？

    查看参考答案

    只追求召回会把大量噪声送入模型，增加成本和错误引用；只追求精确可能漏掉必要证据。通常先用宽召回保证候选覆盖，再由重排和证据预算提升前排质量，并按任务风险设置不同权衡。

  2. 混合检索什么时候优于纯向量检索？

    查看参考答案

    专有名词、编号、错误码和精确短语适合关键词检索，语义改写和同义表达适合向量检索。混合检索在企业资料中通常更稳，但需要分数归一、去重和融合评测，不能简单相加。

  3. top-k 应如何确定，而不是统一设为固定值？

    查看参考答案

    根据查询类型、文档粒度、重排能力、上下文预算和关键证据覆盖率调节。应在验证集上观察 recall@k、噪声率、延迟和最终任务质量，并允许高风险缺证场景扩大检索或拒答。

  4. 重排模型可能引入哪些新失败？

    查看参考答案

    它可能受查询偏差、文档长度、位置、领域外语料或 Prompt Injection 影响，把正确候选压低。必须保留初始召回与重排前后分数，设置候选下限，并用专门的重排数据集和切片评估。

  5. 检索结果看似相关但答案仍错，Trace 应先检查什么？

    查看参考答案

    先确认相关文档是否被召回且权限正确，再看选中的 chunk 是否包含回答所需完整事实、重排位置与最终上下文是否截断。之后才检查生成是否忽略或曲解证据，避免把所有失败归因给模型。

面试表达

为什么需要 hybrid retrieval 和 rerank？

关键词检索擅长精确术语，向量检索擅长语义改写，两者合并提高候选覆盖；cross-encoder 或 LLM reranker 再利用 query—document 交互提高前列精度。我的漏斗会保留每路候选、融合次序和重排版本，用 Recall@K 与 nDCG@K 分开优化，并在证据不足时拒答，而不是让更大的 K 掩盖语料或权限问题。

日常实践关键点

阅读清单前，先用差旅制度查询回忆“硬约束 → 改写 → 多路候选 → Fusion → 去重 → Rerank → 阈值 → Context”；再沿候选 Trace 判断 v7 的上海 P6 证据第一次在哪一步消失，以及该步的分数究竟表示什么。

核心判断

检索调优必须观察正确证据在候选漏斗中的去向，而不是只看最终回答或单一分数；每次改写、融合、重排和上下文组装都要保留输入输出，并由真实查询分桶决定参数与阈值。

日常实践清单

  • 保护 Query 硬约束：改写前后自动比较实体、编号、日期、否定词、租户与资源范围；发现约束丢失或含义扩大时回退原查询，并把差异送入改写回归集。
    触发信号
    改写后召回变多，但地区、职级、编号或否定条件消失。
    判断与定位
    结构化抽取改写前后实体、范围、时间与否定词并做 diff。
    执行动作
    把硬约束独立传给 filter，语义改写只处理可扩展表达。
    验证标准
    约束保真测试通过，负例不会因改写扩大允许资源范围。
    止损与取舍
    发现语义漂移立即回退原查询，召回损失优先于范围越界。
    面试转述
    用 Query 改写不是自由生成解释约束保护与回退条件。
  • 按查询类型选择信号：错误码、产品编号和精确短语优先保留 lexical 通道，同义表达使用 dense 通道，时间与组织范围交给 metadata filter；不要让单一路径承担全部召回。
    触发信号
    编号查询被语义近邻淹没，或同义问法在关键词通道漏召。
    判断与定位
    按查询桶分别观察 lexical、dense 与 filter 的 gold 命中。
    执行动作
    为查询类型配置通道、候选预算和不可绕过的 metadata filter。
    验证标准
    各桶 Recall@K 达标且跨租户、过期版本不会进入候选。
    止损与取舍
    新增通道成本超限时缩减候选，不直接删除唯一有效信号。
    面试转述
    用错误码与同义表达对比说明 Hybrid 的互补性。
  • 使用可解释的融合：不同检索器 raw score 不可比时采用 RRF 等 rank fusion，并记录每路 rank、融合贡献和缺席原因，避免分数缩放变化悄悄改变排序。
    触发信号
    某通道分数尺度变化后排序突变，且无法解释候选贡献。
    判断与定位
    核对每路 rank、缺席原因、去重键和融合前后的候选次序。
    执行动作
    采用 RRF 等 rank fusion，并逐候选记录各通道贡献。
    验证标准
    手算小样本与系统结果一致，单通道缺失不会静默清空结果。
    止损与取舍
    融合退化可切回上版权重，保留原始候选便于离线复算。
    面试转述
    先说明 raw score 不可直接比较，再解释 RRF 贡献证据。
  • 按分桶校准阈值：围绕查询类型、风险、语言和 reranker 版本，用版本化样本观察 precision、recall 与拒答代价；模型升级后重新校准，不把旧阈值当成概率常数。
    触发信号
    同一阈值在语言、风险或查询类型之间表现明显分化。
    判断与定位
    绘制各桶 score 分布、precision/recall 与拒答代价曲线。
    执行动作
    按风险和查询桶版本化阈值，并绑定 reranker 版本。
    验证标准
    独立校准集与回归集均满足误答、漏答和拒答门槛。
    止损与取舍
    高风险桶阈值偏保守，模型换版后回退旧组合并重校准。
    面试转述
    强调 score 不是概率，阈值来自代价函数和数据分布。
  • 控制去重与多样性：按 source、document_version 和内容近重复聚类，限制同一文档相邻 chunk 垄断上下文；需要多来源佐证时显式设置来源多样性，而非把重复片段当多份证据。
    触发信号
    Top-K 被同文档相邻片段占满，其他权威来源没有上下文预算。
    判断与定位
    按来源、版本与内容指纹聚类，观察重复候选的预算占比。
    执行动作
    限制单文档配额，近重复折叠并为必要来源设置多样性目标。
    验证标准
    证据覆盖提升且关键定义未被过度去重，多来源确为独立佐证。
    止损与取舍
    多样性不得牺牲相关性底线；高风险事实仍优先权威来源。
    面试转述
    区分重复片段与独立证据，说明 context 是稀缺预算。
  • 管理候选漏斗预算：把召回候选数、rerank 数、context token、阶段延迟和调用成本一起调节；高风险查询可增加验证，低风险查询优先控制预算，证据不足时选择拒答而非无限扩 K。
    触发信号
    扩大 K 后成本和延迟上升，答案质量却无稳定增益。
    判断与定位
    画出召回、融合、重排、组装各漏斗的保留率与边际收益。
    执行动作
    联合调候选数、rerank 数、token、deadline 和风险级别。
    验证标准
    质量、p95、token 与单位请求成本同时落在门限内。
    止损与取舍
    证据不足选择拒答，不以无限扩 K 掩盖语料或路由问题。
    面试转述
    用边际收益曲线回答 Top-K 与 rerank 数如何选择。
  • 保存候选级 Trace：为每个 chunk 记录通道、filter、原始 rank、融合 rank、rerank score、最终是否入选及截断原因，让“正确证据在哪一步消失”可以直接回答。
    触发信号
    gold 文档存在却未进上下文，团队只能反复猜测检索参数。
    判断与定位
    逐候选查看通道、过滤、融合、重排和截断的状态变化。
    执行动作
    记录稳定 chunk ID、各阶段 rank/score、入选与淘汰原因。
    验证标准
    任一失败样本可直接指出 gold 首次消失阶段并离线复算。
    止损与取舍
    敏感文本不写 Trace，只存受控 locator、特征与版本标识。
    面试转述
    用候选漏斗回答如何定位召回、融合还是重排问题。

排障顺序

错答时先分别查看 lexical、dense、metadata 或其他通道的原始候选，确认 gold evidence 是否出现；再检查 fusion 是否合并和去重正确、rerank 是否把它压低、context pack 是否因预算或重复而移除。只在确认第一处偏离后调整相应参数，并用相同查询桶复测，避免同时改写、改 K 和改阈值导致无法归因。

应留证据

保留原查询与改写 diff、各路 candidate trace、filter 与查询桶、融合和重排前后次序、gold evidence 命中位置、Recall@K、MRR 或 nDCG、去重率、上下文 token、阶段 p50/p95 延迟、成本，以及每次参数变更的评测版本和回滚基线。

代表性工程产物：候选级漏斗 Trace
query_bucket=policy_exact | chunk=travel-v7#table3-p6 | lexical_rank=2 | dense_rank=11 | rrf=0.0307 | fused_rank=3 | rerank_score=0.81 | rerank_rank=1 | selected=true | context_tokens=286 | constraints=region:上海,grade:P6 | drop_reason=null

同一记录保留各路贡献、过滤条件和最终去留，可用来定位 gold evidence 首次丢失阶段；排名、阈值与 token 仅为结构示例，不能直接作为生产默认值。

日常面试要点

关联实践：Query 约束、多路信号、可解释融合

高频问法

为什么使用 Hybrid Search，Query 改写和 RRF 分别解决什么问题，又如何避免约束漂移？

60–90 秒骨架

先按查询类型说明 lexical、dense、metadata 的互补，再把硬约束独立传递；最后用 rank fusion 解决异构分数不可比并保留贡献。

必讲证据

展示改写 diff、各路 gold 命中、候选贡献与融合前后排序，证明增加通道确实修复具体查询桶。

常见追问

RRF 常数如何选？某路缺席怎么办？filter 放在哪一层？什么时候不需要 dense 检索？

回答边界

不要把 Hybrid 描述成必然更好；没有线上 A/B 时明确使用离线实验数据及其流量代表性限制。

关联实践：阈值校准、漏斗预算

高频问法

Top-K、rerank 数量和拒答阈值如何确定，怎样平衡召回、延迟、成本与高风险误答？

60–90 秒骨架

按查询桶观察漏斗边际收益和 score 分布，以误答、漏答、拒答代价定阈值；再将质量与 p95、token、成本放入联合门禁。

必讲证据

给出分桶 PR 曲线、候选保留率、延迟成本曲线和拒答样本，说明参数与 reranker 版本共同管理。

常见追问

score 是概率吗？模型升级为什么重校准？高风险桶如何处理？无限增加 K 有什么问题？

回答边界

没有业务损失函数时只能提出校准方案，不可编造最优阈值、成本下降或生产 SLO 达成情况。

关联实践：去重多样性、候选 Trace

高频问法

正确证据明明在知识库却没进入最终上下文，你会怎样定位是召回、融合、重排还是组装问题？

60–90 秒骨架

从各原始通道找 gold evidence，沿 fusion、rerank、dedup、context pack 逐层对齐，锁定首次消失位置后只改该阶段。

必讲证据

展示候选级 Trace、淘汰原因、重复聚类、token 截断位置和单阶段复测结果，避免凭最终答案猜原因。

常见追问

如何定义 gold？去重会误删吗？敏感候选怎样记录？多个阶段都异常时先修哪个？

回答边界

只能陈述 Trace 覆盖到的阶段；若未记录原始候选，应承认证据不足并先补观测，不能倒推确定归因。

经验真实性：回答检索调优时明确哪些是生产实战、个人实验或方案设计；只引用真实评测集、Trace 与参数对比，不虚构流量分布、召回提升、成本下降或线上故障结论。
