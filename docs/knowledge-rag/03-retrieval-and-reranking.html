03 检索与重排

学习目标

  • 区分 lexical、dense、metadata 与 graph retrieval 的信号。
  • 解释 query rewrite、hybrid fusion、dedup、rerank 与 diversity 的顺序。
  • 根据 recall、precision、latency 和成本设定候选漏斗。
  • 为低置信、冲突和无结果设计受控分支。

核心概念

阶段主要目标典型指标关键 trace
Query保留意图、补充可检索表达rewrite consistency原查询、改写、过滤条件
Recall候选覆盖Recall@K、MRR各检索器候选与原始分数
Rerank有限预算内提高精度nDCG@K、Precision@K重排前后次序、模型版本
Pack形成可读、去重、多样的证据context relevance、token cost截断、去重和最终 chunk ids

工程要点

  1. 访问过滤应作为强约束进入候选生成，而不是召回后仅靠 prompt 告诉模型忽略。
  2. hybrid retrieval 可用 RRF 等 rank fusion 合并不同分数空间，避免直接相加不可比的 raw score。
  3. 对实体编号、错误码和专有词保留关键词检索；对改写和语义同义表达使用 dense retrieval。
  4. 阈值由校准集决定；低分、冲突或证据过时应返回显式 insufficient_evidence。

常见坑

  • 只调 Top-K，不看候选来自哪一路和相关文档是否在语料中。
  • 把 reranker 分数当概率，跨模型或跨版本使用同一阈值。
  • 查询改写丢失日期、否定、实体或租户约束。
  • 塞入大量相似 chunk，既浪费 token 又形成虚假“多证据”印象。

理论追问

请先脱稿回答，再展开参考答案核对边界、证据和取舍。

  1. 为什么召回率和精确率不能只优化一个？

    查看参考答案

    只追求召回会把大量噪声送入模型，增加成本和错误引用；只追求精确可能漏掉必要证据。通常先用宽召回保证候选覆盖，再由重排和证据预算提升前排质量，并按任务风险设置不同权衡。

  2. 混合检索什么时候优于纯向量检索？

    查看参考答案

    专有名词、编号、错误码和精确短语适合关键词检索，语义改写和同义表达适合向量检索。混合检索在企业资料中通常更稳，但需要分数归一、去重和融合评测，不能简单相加。

  3. top-k 应如何确定，而不是统一设为固定值？

    查看参考答案

    根据查询类型、文档粒度、重排能力、上下文预算和关键证据覆盖率调节。应在验证集上观察 recall@k、噪声率、延迟和最终任务质量，并允许高风险缺证场景扩大检索或拒答。

  4. 重排模型可能引入哪些新失败？

    查看参考答案

    它可能受查询偏差、文档长度、位置、领域外语料或 Prompt Injection 影响，把正确候选压低。必须保留初始召回与重排前后分数，设置候选下限，并用专门的重排数据集和切片评估。

  5. 检索结果看似相关但答案仍错，Trace 应先检查什么？

    查看参考答案

    先确认相关文档是否被召回且权限正确，再看选中的 chunk 是否包含回答所需完整事实、重排位置与最终上下文是否截断。之后才检查生成是否忽略或曲解证据，避免把所有失败归因给模型。

面试表达

为什么需要 hybrid retrieval 和 rerank？

关键词检索擅长精确术语，向量检索擅长语义改写，两者合并提高候选覆盖；cross-encoder 或 LLM reranker 再利用 query—document 交互提高前列精度。我的漏斗会保留每路候选、融合次序和重排版本，用 Recall@K 与 nDCG@K 分开优化，并在证据不足时拒答，而不是让更大的 K 掩盖语料或权限问题。
