02 摄取、切分与索引

学习目标

  • 设计增量、幂等、可追溯的摄取管道。
  • 按文档结构和问题粒度选择 chunk 策略。
  • 让权限、版本、血缘和删除语义进入索引。
  • 用 shadow index 与质量门禁实现可回滚发布。

核心概念

链路总览：索引是可重建的派生数据

切分策略适用风险保护措施
固定 token + overlap结构弱、基线验证语义截断、重复候选标题继承、去重、边界窗口
结构感知Markdown、手册、法规超长章节层级元数据 + 二次切分
语义切分主题跳转明显的长文成本高、不可稳定复现固定模型版本和阈值
父子索引局部召回、整体阅读上下文膨胀子块检索、父块按预算扩展

权威来源是事实主本，解析文本、chunk、Embedding 和倒排项都是可以按版本重建的派生物。摄取链路必须同时管理内容生命周期与权限生命周期：新增、更新、撤回和撤权都要沿 lineage 到达每一种索引和缓存。发布不是“写完即生效”，而是构建 shadow version、验证覆盖与安全、原子切换 alias，并保留旧版用于回滚和历史回答复现。

逐环节拆解

01 · 来源快照

每次摄取先冻结可寻址的权威来源快照，后续派生物才能被复现、比较和撤销。

职责与输入输出
连接器接收来源事件、凭据和游标，产出不可变 source snapshot、内容 hash、抓取时间与来源版本。
工作机制
以增量游标发现变化，下载原始字节并计算 hash；事件与快照分离保存，使同一输入可重新运行 parser。
必须成立的约束
快照必须关联 source_id、权限范围和 observed_at，不能只保存最新正文；否则历史索引和删除原因不可证明。
关键难点
来源 API 的更新语义、限流和迟到事件不同，轮询频率需由 freshness SLO、来源风险和成本共同决定。
边界与注意事项
快照忠实于来源但不保证解析正确，也不等于业务有效版本；漏采会形成 corpus 缺口并在检索阶段表现为零召回。

02 · 解析与规范化

解析要保留可回答问题所需的结构与定位，不是把所有格式压成一串看似干净的纯文本。

职责与输入输出
解析器接收原始快照与 MIME 信息，产出段落、标题路径、页码、表格单元、阅读顺序、语言和解析质量信号。
工作机制
按格式选择原生解析或 OCR，规范字符与布局，同时保留结构节点及到原文坐标的映射，异常进入隔离队列。
必须成立的约束
规范化不能改变金额、日期、否定和表格对应关系；每段文本能回到原快照 locator，否则引用与抽检失效。
关键难点
扫描件、跨页表格、多栏布局和代码对阅读顺序敏感，parser 选择与置信阈值应按文档类型分桶评测。
边界与注意事项
解析恢复结构但不决定 chunk 粒度，也不判断政策真假；乱码或表头丢失会被 Embedding 忠实编码并放大。

03 · 结构元数据与 ACL

内容、结构、适用范围和 ACL 必须在切分前绑定，并随每个派生 chunk 一起传播。

职责与输入输出
元数据阶段接收解析树与来源策略，产出标题层级、生效时间、地区、职级、权威等级、tenant 和资源 ACL。
工作机制
继承文档级属性并允许章节级覆盖，把可过滤字段标准化；策略引用保留 policy source 与 version 而非只存布尔值。
必须成立的约束
ACL 默认拒绝且不可为空，适用范围不能由正文自由猜测；权限变更必须有事件可驱动后续失效。
关键难点
静态文档 ACL 与动态主体属性需在查询时结合，元数据模式必须兼顾过滤性能、演进和来源差异。
边界与注意事项
摄取保存资源约束，不替代在线策略决策；ACL 缺失会让索引无法安全前置过滤，属于数据治理事故。

04 · 稳定 ID、幂等与去重

同一来源事件重放必须得到同一逻辑结果，而内容变化又必须产生可区分、可追溯的新版本。

职责与输入输出
ID 层接收 source identity、版本与规范化内容，产出 document_id、chunk_id、content_hash 和 ingestion_run_id。
工作机制
稳定业务键定位实体，版本标识变化，hash 判断精确重复；upsert 与事件去重表保证 at-least-once 消费可重放。
必须成立的约束
ID 不依赖向量库随机键，幂等键覆盖来源事件；近重复不能只按相似度删除，需保留权威性和有效期判断。
关键难点
模板页脚、镜像来源和轻微修订使去重边界模糊，canonical 选择应由来源权威与时间规则决定。
边界与注意事项
去重减少重复派生物，不负责解决语义冲突；ID 漂移会制造幽灵旧块，使更新、删除和评测对不上。

05 · Chunk 语义设计

Chunk 是最小可检索证据单元，应同时保留可命中性、可理解性和可引用边界。

职责与输入输出
切分器接收结构化文档与目标问题分布，产出带父子关系、标题路径、offset、版本和 ACL 的 EvidenceChunk。
工作机制
优先在章节、段落、表格或代码边界切分，超长节点二次切分；可用子块召回后按预算扩展父块或邻接块。
必须成立的约束
金额与适用条件、表头与数据行、定义与例外不能被无标记拆散；重叠块要保留去重所需关系。
关键难点
小块易命中但语境不足，大块完整却噪声和 token 成本高，粒度必须用真实 query 的证据完整率校准。
边界与注意事项
chunk size 不是跨文档通用常数，语义切分也不自动更优；坏切分会让正确文档存在却没有完整证据单元。

06 · 稀疏、稠密特征与索引

索引保存互补的可检索信号及其版本，不存在一种特征能覆盖精确术语、语义表达和结构过滤。

职责与输入输出
特征层接收 chunk 与 metadata，产出 lexical terms、dense vector、结构键及对应 model/analyzer/index version。
工作机制
倒排索引保留词项与字段，向量索引保存 Embedding 邻域，metadata 建过滤路径；各路共享稳定 chunk_id。
必须成立的约束
Embedding、分词器和距离度量版本明确，向量维度不可混写，ACL 在所有检索通道都可执行。
关键难点
召回质量、构建延迟、容量与查询延迟相互制约，索引参数需在目标查询与数据规模上版本化评测。
边界与注意事项
Embedding 相似不等于业务相关或真实，特征层也不负责融合排序；原地升级会破坏对比和回滚。

07 · 质量门禁与发布

只有通过覆盖、可检索性、安全与性能门禁的完整索引版本，才可被原子切换为在线 active。

职责与输入输出
发布器接收 shadow index、质量报告和回归集，产出 promote、hold 或 rollback 决策与 active alias receipt。
工作机制
先做解析抽检、reconciliation、golden retrieval、越权和删除测试，再以原子 alias 切换并灰度观察。
必须成立的约束
发布单元必须完整且不可部分可见，门禁绑定 corpus/index 版本；hard fail 不能被平均质量提升抵消。
关键难点
更严格门禁降低风险却延长 freshness lag，阈值应按来源风险、流量切片与回滚能力确定。
边界与注意事项
索引门禁证明供给可用，不证明最终答案正确；跳过 shadow 会让失败难以隔离且回滚成为再次重建。

08 · 血缘、删除与撤权传播

能写入的每一种派生物都必须能按 source 或权限事件定位、失效并给出传播完成证据。

职责与输入输出
生命周期控制器接收删除、撤回和 ACL 变化，沿 source→document→chunk→index/cache 血缘产出 tombstone 与 receipt。
工作机制
先阻止新查询读到目标，再异步清理稀疏、稠密和缓存副本；周期 reconciliation 检出遗漏和孤儿记录。
必须成立的约束
传播有明确 SLO、幂等事件和完成清单，历史 Trace 只保留合规审计所需脱敏引用，不能成为旁路副本。
关键难点
多索引最终一致性与立即撤权冲突，需要读时拦截、主动失效和后台清理组合，而非只依赖 TTL。
边界与注意事项
逻辑不可见与物理删除是两个阶段；任一通道仍可召回即未完成，不能以主向量库已删宣告成功。

难点与取舍：结构、粒度、特征和发布

决策主要张力依据
Parser / OCR结构恢复质量 vs. 成本与吞吐格式分桶、关键字段保真率、异常率
Chunk 粒度局部命中 vs. 完整语境与重复噪声真实问题、证据完整率、context 成本
特征组合精确词项 vs. 语义覆盖与容量查询桶 Recall、延迟和权限过滤能力
发布频率新鲜度 vs. 门禁时间和回滚风险来源风险、freshness SLO、hard fail

失败如何沿派生链传播

  1. 来源漏采使正确制度不在 corpus，后续索引与召回再优也只能返回空或旧资料。
  2. 解析丢表头会把“城市—职级—金额”关系拆坏，chunk 和 Embedding 随后稳定保存错误结构。
  3. ID 或 lineage 漂移让新旧块共存，造成冲突、重复候选以及删除无法闭环。
  4. ACL 漏传会扩张在线候选空间；删除只完成一个索引则在其他通道形成可再次泄露的幽灵证据。

贯穿案例 02：从快照到 active index

仍使用“上海地区 P6 员工当前出差住宿上限是多少？”。教学来源快照 travel-policy.pdf@v7 在 2026-07-18 被观察到，解析器保留“4.2 国内住宿标准”标题、表头、上海行、P6 列和页码；适用主体、地区、生效时间与 ACL 随 chunk 传播。示例仅用于说明数据形状，不代表真实制度或生产默认参数。

snapshot: { source_id: "travel-policy", version: "v7", hash: "sha256:demo" }
chunk: { id: "travel-policy:v7:sec-4.2:row-sh-p6", locator: "p12/table2/r3/c4",
         scope: { city: "Shanghai", level: "P6", effective_from: "2026-07-01" },
         acl: "employees-cn", parent: "sec-4.2" }
index: { version: "travel-policy-2026-07", state: "active", previous: "2026-06" }

若解析抽检发现表头与数据行错位，该版本应停在 shadow；若 v7 被撤回，则 tombstone 必须使稀疏、稠密、缓存和 active alias 都不再返回该 chunk。

稳定文档身份、版本与内容寻址

推荐把身份拆成四层：source_system_id 标识连接器或权威源；source_object_id 取自源系统稳定主键；二者映射为内部稳定 document_id；每次可审计更新生成 version_id；规范化内容计算 content_hash，并与业务隔离域组成 content_id = namespace + content_hash。因此参考实现中的 namespace+hash 与这里的方案相似，但它只解决不可变内容对象，不承担“同一业务文档跨版本”的逻辑身份。

{
  "source_system_id": "policy-cms",
  "source_object_id": "doc-8421",
  "document_id": "doc_01J...",
  "version_id": "ver_2026-07-23T08:30Z",
  "content_hash": "sha256:...",
  "content_id": "tenant-cn:sha256:...",
  "locator": {"section_path":["差旅","住宿"],"page":12},
  "chunk_id": "doc_01J...:ver_...:sec_..."
}

约束：URL、文件路径和标题会变，只能是 locator；hash 会随内容变，只能识别内容；若源系统无稳定主键，应由 document registry 维护 canonical locator、别名与人工合并/拆分记录，禁止每次摄取重新推导身份。

摄取注册表、状态机与可恢复执行

每个 source/document/version 在 registry 中记录期望状态和实际状态。一次摄取至少经历 discovered → snapshotted → parsed → normalized → chunked → indexed → validated → published；失败进入可重试或隔离态，而不是留下“向量已写、稀疏索引未写”的半成品。步骤以 document/version 和 pipeline_version 作为幂等键，写入 staging index，通过 coverage、ACL、重复率、golden query 和删除负测后再原子切换 alias。

故障恢复动作完成证据
解析器超时/坏页保留原始快照，按错误类别重试或转 HITL失败页、parser_version、人工结论
部分索引写入重跑幂等 stage 或丢弃 staging，不污染 active各索引 receipt 与对账 diff
删除/撤权tombstone 高优先级传播至 sparse、dense、cache、citation artifact传播延迟、残留扫描、消费端回执
模型/切分升级shadow build + 固定集回归 + alias 原子切换版本矩阵、质量/延迟差异、回滚点

按文档结构路由解析器，并把低置信送入 HITL

解析不是“PDF 转文本”。先识别文件类型、扫描/OCR、版面、多栏、表格、代码、图片、语言和加密状态，再路由到专用 parser。输出应保留 heading path、页码/坐标、表格行列、列表层级、阅读顺序与解析置信度。低置信表格、关键字段缺失、页数突变或乱码率异常进入 Human-in-the-loop 队列；人工修订要形成可复用规则和标注集，而非静默改正文。

Parent-Child / AutoMerging：细召回与完整上下文分工

索引小的 child chunk 以提高命中粒度，但向生成侧提供带标题、定义和约束的 parent context。多个相邻 child 命中同一父节点且覆盖度达到经验证条件时，可 AutoMerging 成父块；否则保留分散证据，避免把无关段落一起送入上下文。parent/child ID、相邻关系、locator、ACL 和版本必须同源，合并后仍要重新执行 context budget、去重与授权校验。

索引工程：ANN、BM25、过滤与缓存各司其职

Dense ANN 索引以近似搜索换取吞吐，需理解 recall—latency—memory 的曲线；BM25/倒排索引保留编号、专名和精确短语；metadata/ACL filter 是强约束，不应交给相关性模型猜。缓存至少区分 embedding、query result、document object 和 final response：键必须包含 tenant/subject scope、corpus/index/policy/model 版本，撤权与删除要主动失效。缓存命中率不能以复用过期或越权结果为代价。

无上游主键时，首次发现由 registry 分配 document_id 并保存 canonical locator 与 alias。后续只有“旧对象消失 + 新 locator 内容/结构/时间窗口唯一匹配”才建议移动；多个候选、分拆/合并或 ACL 冲突进入 ambiguity quarantine，由人工确认。系统不得靠标题或 hash 静默合并业务对象。

Registry、事件、Ledger 与 Run Manifest 数据形状

registry: {document_id, source_system_id, source_object_id, aliases[], active_version_id, state, acl_fingerprint}
source_event: {event_id, source_cursor, operation, source_object_id, observed_at, payload_hash}
action_ledger: {idempotency_key, document_id, version_id, step, attempt, before_state, after_state, status}
run_manifest: {run_id, pipeline_version, source_snapshot, expected_documents, started_at, completed_at}
index_receipts: {chunk_id, sparse_receipt, dense_receipt, metadata_receipt, cache_invalidation_receipt}

Action Ledger/WAL 先记录意图，再写派生存储，最后汇总 receipt；重放相同幂等键只补齐缺失动作。run manifest 让批次覆盖率、孤儿和半写入可计算，registry 则是逻辑状态真相，不把任一向量库当作唯一真相。

恢复、不一致收敛与升级回滚

部分写入时保持旧 active，失败 version 标记 failed/quarantined；重试从 ledger 找缺失 receipt，而非重复创建 chunk。读路径先查 registry/tombstone，阻断已撤权内容，再异步清理副本。定时 reconciliation 比较 source snapshot、registry、sparse、dense、parent KV 与 cache，检测 orphan chunk、缺索引和 superseded 残留；修复后再次运行权限与 gold query。

Embedding/Chunker 升级使用 shadow index 和独立 version namespace，蓝绿对比后切 active alias，并保留满足业务恢复时间目标的回滚窗口。备份至少保存权威源快照/游标、registry、pipeline 配置、chunk 产物或可重建种子；RPO 由允许丢失的源事件窗口决定，RTO 由恢复 registry、重放 ledger、重建索引和切换 alias 的测量结果决定，不能只写“有备份”。

统一中间模型、解析质量与分块选择

Parser 输出统一 Document → Section → Block：Section 保存 heading tree 与 reading_order；Block 类型覆盖 paragraph/list/table/code/formula/image，保存 page、bbox、row/column、language、locator 和 confidence。ParserRouter 先诊断 scanned、multi-column、table-heavy、encrypted 等特征，再选择有限降级链；页数突变、关键表格缺列、阅读顺序异常、乱码或低置信进入隔离，关键制度字段强制 HITL。人工修正同时回写 corrected artifact、reason code 和 parser 回归样本。

分块适用输入典型失败
固定无结构短文本基线切断定义/表格
递归有自然分隔符的长文结构语义仍可能丢失
结构感知标题层级清晰文档依赖解析质量
语义主题边界不规则文本模型漂移、成本增加
父子细检索、完整回答父块噪声和版本同步
句子窗口局部事实与邻域解释跨远距离约束不足
格式感知表格、代码、公式专用 parser/renderer 复杂

父块存 KV/object store，child 以同一 chunk_id 对齐 BM25 与 dense index；AutoMerging 依据同父命中覆盖和预算触发，父子更新/删除共享 version 与 tombstone，禁止新 child 回填旧 parent。

ANN、Embedding 与稀疏索引的工程边界

索引Recall/延迟内存/构建采用依据
Flat精确基线、规模大时慢结构开销低、查询计算高小规模或 ANN Recall 对照
HNSW高召回低延迟，可调搜索深度图结构内存与构建较高读多、内存可接受
IVF/PQ压缩与探针数影响 Recall省内存，需训练/重建大规模且可接受近似损失

Embedding worker 进行受限批处理、限流、指数退避和断点进度；缓存键包含 normalized_content_hash、model_version、preprocess_version，漂移或模型升级走 shadow 蓝绿重建。BM25 的 index/query analyzer 必须 parity；中文需验证分词、词典、数字/型号和同义词策略。sparse/dense/parent KV 共享 canonical chunk_id，通过 receipts 和 reconciliation 对账。VectorStore 抽象只暴露 upsert/delete/query/filter/health 能力与契约，不抹平不同引擎的过滤一致性、事务、距离分数和可观测差异。

工程要点

  1. 为 source、document、chunk 建立稳定 ID；内容 hash 用于幂等去重，版本号用于复现。
  2. 解析时保留标题路径、页码、表格行列、语言、时间和权威等级；不要只保留纯文本。
  3. ACL、tenant、保密级别在 chunk 层可过滤，删除或撤权必须传播至稀疏、稠密与缓存索引。
  4. 新索引先跑 coverage、解析成功率、重复率、检索 golden set 和越权测试，再原子切换 alias。

常见坑

  • 按统一字符数切所有资料，导致代码、表格和条款失去结构。
  • 更新只追加不删除，让撤回文档继续被召回。
  • 先向量化再补权限，最终无法在检索前可靠过滤。
  • Embedding 模型升级原地覆盖，无法对比、回滚或复现旧回答。

理论追问

请先脱稿回答，再展开参考答案核对边界、证据和取舍。

  1. Chunk 越小是否召回越精准？

    查看参考答案

    不一定。小 chunk 可能提高局部词匹配，却丢失定义、表格或跨段约束；大 chunk 保留语境但增加噪声和成本。应按文档结构、问题粒度和模型窗口设计，并用真实查询评估召回、引用完整性和答案质量。

  2. 为什么每个索引片段必须保留来源与版本元数据？

    查看参考答案

    回答需要追溯到文档、章节、生效时间和索引版本，权限过滤与删除也依赖这些字段。没有 provenance，系统无法解释引用、处理过期冲突、回滚索引或证明某次回答使用了哪版制度。

  3. 文档更新时，如何避免新旧片段同时命中？

    查看参考答案

    为文档建立稳定 source_id、内容版本和有效期，摄取采用幂等 upsert；新版本发布后原子切换 active index 或标记旧版本失效。查询只读取允许版本，并通过删除或 tombstone 清理旧片段。

  4. 为什么 ACL 既要在摄取时保存，也要在查询时执行？

    查看参考答案

    摄取时保存资源级权限和租户归属，才能构建可过滤索引；查询时结合当前主体、委托和策略版本做动态判定。只在摄取时判断会忽略权限变化，只在查询后过滤则可能让未授权内容进入候选链路。

  5. 如何验证切分与索引变更没有破坏知识可用性？

    查看参考答案

    维护覆盖定义、表格、跨段关系、长文和权限边界的版本化查询集，对比 recall@k、关键证据完整率、重复率、索引延迟与引用正确性；还要做删除、权限变更和旧版本不再可检索的测试。

面试表达

如何选择 chunk size？

我不会从固定数字出发，而从目标问题粒度、文档结构、检索器输入限制和生成上下文预算出发。先用结构感知基线，再以 chunk-level recall、重复率、完整性和端到端 groundedness 做离线对比；同时保留标题路径、版本与 ACL，并通过父子索引解决“细粒度召回、完整上下文回答”的矛盾。

日常实践关键点

阅读清单前，先用差旅制度 v7 回忆“来源快照 → 解析结构 → 元数据与 ACL → 稳定 ID → chunk → 多路特征 → shadow 门禁 → 删除传播”；再判断上海 P6 对应表格关系在哪一步可能被破坏，以及什么证据能复现。

核心判断

检索质量异常时先查看解析后的真实 chunk，而不是先调 Embedding 或 Top-K；摄取链路必须做到结构可见、写入幂等、血缘可追、删除可证明、索引可版本化发布和回滚。

排障顺序

出现“文档明明存在却检索不到”时，先打开权威原文与解析产物，确认目标文字、标题路径、表格关系和 ACL 是否保留；再沿 lineage 检查 chunk、内容 hash、document_version、tombstone 与 active index 是否一致，最后才比较 Embedding、切分参数和索引查询。若真实 chunk 已错误，任何检索调参都只是放大脏数据。

应留证据

保留按文档类型分桶的解析报告和失败样本、source—document—chunk lineage、摄取幂等日志、源与索引 diff、删除或撤权 receipt、freshness 与重复率看板、版本化 golden set 结果、shadow index 对比及 alias 切换和回滚记录。

代表性工程产物：lineage、撤权传播与 shadow diff

该记录能从命中反查来源，也能证明删除传播和升级比较；示例指标不是生产门槛，真实项目应附分桶样本、基线区间和审批记录。
