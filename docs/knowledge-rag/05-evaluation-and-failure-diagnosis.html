05 评测与失败诊断

学习目标

  • 建立 corpus、ingestion、retrieval、generation、citation 与 policy 分层指标。
  • 构建含可回答、不可回答、冲突、过期和越权样本的评测集。
  • 使用 trace 与 ablation 做阶段归因。
  • 把线上反馈转成版本化回归门禁，而非只改 prompt。

核心概念

链路总览：评测要比较每个阶段的 Expected 与 Actual

层级指标 / 测试所需证据
Corpus / Ingestioncoverage、freshness、parse success、ACL integritysource snapshot、document/chunk lineage
Retrieval / RerankRecall@K、MRR、nDCG、filter leakage、latency候选 ids、分数、重排前后次序
Generationanswer correctness、groundedness、abstention最终 context、claims、模型与 prompt 版本
Citation / Securitycorrectness、completeness、unauthorized disclosure、injection resistancesource locators、policy decision、redaction

最终答案是多阶段函数的末端输出，同一个错答可能由 corpus 缺失、解析损坏、索引未发布、授权误杀、召回漏失、重排压低、context 截断、生成曲解、引用错位或策略误判造成。可诊断评测必须为每条样本定义可执行的期望证据链、固定组件版本，并保存实际 Trace。First Divergence 是 expected 与 actual 第一次分叉的位置，它把“哪里出现症状”转成“哪个 owner 最先违反契约”。

逐环节拆解

01 · 评测单元与 Gold

Gold 不是一句参考答案，而是一组带主体、时间、可回答性、允许证据与禁止证据的可执行判定条件。

职责与输入输出
评测设计接收真实任务与风险模型，产出 query、subject、as_of、answerability、Gold Evidence、Gold Answer 边界和 forbidden evidence。
工作机制
Gold Evidence 标出必须或可接受的 source/version/locator；Gold Answer 描述允许语义，不要求逐字匹配；不可回答样本定义拒答期望。
必须成立的约束
证据必须对该主体合法且在评测时间有效，forbidden evidence 明确越权与过期来源；标注变更要版本化审计。
关键难点
答案可能多样、证据可能等价，单一字符串会误判；标注粒度由风险、任务可判定性和人工成本决定。
边界与注意事项
Gold Answer 衡量任务结果，Gold Evidence 衡量证据链；有正确答案文本不代表检索或权限过程正确。

02 · 版本快照

没有 corpus 到 policy 的完整版本矩阵，分数变化无法归因，线上失败也无法可靠重放。

职责与输入输出
评测运行器冻结 dataset、corpus、parser、chunker、index、retriever、reranker、prompt、model 和 policy 版本。
工作机制
每个 run 生成不可变 manifest，输入输出与 trace_id 关联；对照实验只改变声明中的目标变量。
必须成立的约束
同一次运行不能跨 active index，外部 judge 与随机参数也需记录；缺关键版本的样本只可观察，不可下归因结论。
关键难点
完全固定提高复现性却可能偏离线上动态环境，需要同时维护离线快照与受控线上 replay/shadow。
边界与注意事项
版本快照证明比较条件，不证明系统质量；版本标签若只是“latest”，重跑同名实验仍可能得到不同链路。

03 · Corpus 与 Ingestion 指标

上游评测先证明正确资料存在、有效、解析完整且进入 active index，再讨论在线检索参数。

职责与输入输出
数据评测接收来源清单、快照、解析树与索引 manifest，产出 coverage、freshness、parse success、ACL integrity 和 reconciliation 差异。
工作机制
按 Gold locator 验证原文存在，再检查解析后的字段、chunk lineage、active version 与删除/撤权状态。
必须成立的约束
指标按文档类型和权限切片，成功率不能掩盖关键表格损坏；coverage 定义必须与任务所需证据绑定。
关键难点
整体解析成功不等于关键事实保真，需结合结构抽检与 query-driven probes，在成本和缺陷发现率间取舍。
边界与注意事项
Corpus 指标不证明检索会命中，但失败时已足够把 owner 定位上游；扩大 K 不能创造缺失或损坏的证据。

04 · Retrieval 与 Ranking 指标

检索指标只衡量 Gold Evidence 在候选漏斗中的存在与次序，不能证明权限、Grounding 或最终答案正确。

职责与输入输出
排名评测接收 Gold Evidence 和各阶段候选，产出 Recall@K、MRR、nDCG、Precision、filter leakage 与 latency。
工作机制
Recall@K 看是否进入窗口，MRR 看首个相关项，nDCG 比较多级相关性的次序折损；分别对 recall、fusion、rerank 计算。
必须成立的约束
relevance label、K、查询桶和版本固定，授权外证据即使相关也算 leakage 而非正例；候选去重规则保持一致。
关键难点
单一指标会鼓励错误优化，例如 MRR 忽略多证据完整性；组合指标和门槛由任务证据形态与成本决定。
边界与注意事项
高 Recall 只说明候选中存在正确证据，不保证 packing 保留或模型使用；分数下降也可能来自 Gold 过期。

05 · Generation、Citation 与 Policy 指标

答案、支持、引用和安全必须分别计量，任何平均质量提升都不能抵消越权或关键无支持声明。

职责与输入输出
末端评测接收最终 context、claims、citations 和 policy decision，产出 correctness、groundedness、citation、abstention 与安全结果。
工作机制
规则验证数值与 locator，模型 judge 评语义但需校准；claim 级支持计算 completeness，负例测试越权、注入与拒答。
必须成立的约束
judge 版本与 rubric 固定，高风险项有人审或确定性核验；cross-tenant、伪造引用等 hard fail 独立于平均分。
关键难点
流畅度易误导 judge，开放式答案又难规则化，应按可判定字段、claim 支持与人工抽检组合证据。
边界与注意事项
末端指标描述症状，不能单独定位阶段；正确拒答可能降低回答率却提高风险质量，需结合 answerability 解读。

06 · First Divergence

优先修复 expected 与 actual 首次分叉的阶段，因为其后异常通常是上游错误的传播结果。

职责与输入输出
诊断器接收版本化 Gold 链和实际 Trace，产出首个偏离阶段、违反的不变量、owner 与可证伪的根因假设。
工作机制
依次比较 source、parse、chunk、index、policy、candidate、rank、context、claim 和 citation，首个不一致处停止泛化归因。
必须成立的约束
Expected 必须足够精确且 Trace 未丢阶段；相关只是时间先后，不等于因果，仍需复现实验验证假设。
关键难点
多个并发缺陷可能产生同一症状，第一偏离点缩小搜索空间但不保证唯一根因，需要 oracle 或 ablation 复证。
边界与注意事项
First Divergence 是诊断坐标，不是自动修复建议；如果 Gold 本身错误，应回到评测数据 owner 而非改生产系统。

07 · Oracle 与 Ablation

Oracle 用理想上游输入测试下游上限，Ablation 一次替换一个变量；两者用于隔离因果而非证明线上收益。

职责与输入输出
实验器接收失败样本与版本矩阵，产出 oracle evidence、单阶段替换实验、质量差异和是否支持根因假设。
工作机制
把 Gold Evidence 直接送给 generation 测试下游；或只替换 parser、retriever、reranker、prompt 中一个组件做配对比较。
必须成立的约束
除目标变量外其他输入、版本和随机条件保持可比，样本切片一致；一次改多阶段无法解释增益来源。
关键难点
Oracle 输入可能比线上 context 更干净而高估上限，实验结论需再用真实漏斗 shadow 验证外部有效性。
边界与注意事项
Oracle 成功说明下游有能力，不代表上游改动容易；Ablation 发现关联后仍需评估延迟、成本和安全副作用。

08 · 切片、发布门禁与反馈治理

发布决策看真实高风险切片和 hard fail，线上反馈只有经脱敏、去重与人工确认后才能成为新 Gold。

职责与输入输出
评测层汇总租户、语言、文档类型、意图、风险和长尾来源切片，产出门禁、灰度、回滚与数据集更新决策。
工作机制
离线回归后做 shadow，再分阶段灰度；观察 hard fail、质量、p95 和成本，线上失败经审核进入版本化回归集。
必须成立的约束
高风险切片退化不可被总体平均抵消，回滚阈值预先定义；用户点击和旧回答不能未经核实充当真值。
关键难点
切片过细样本不足，过粗掩盖局部退化；门槛需结合置信区间、业务损失、流量和回滚能力决定。
边界与注意事项
评测消费 Trace 并反馈编排层，不接管在线路由；反馈闭环若自举错误答案，会让回归集固化生产偏差。

难点与取舍：质量、风险、成本与可复现性

选择收益必须同时观察
更大评测集覆盖更多长尾与切片标注质量、运行成本和版本维护
更严格门禁降低高风险回归freshness、发布速度与回滚能力
更多线上反馈贴近真实分布隐私、选择偏差、人工确认与 Gold 污染

相关失败与首个偏离点

  1. Gold 只含答案文本，检索到 forbidden evidence 仍可能被判正确，安全错误从评测入口被漏掉。
  2. 版本矩阵不完整时，分数变化可能来自 corpus、模型或策略漂移，任何单点根因都缺乏证据。
  3. Corpus 缺失会表现为 Recall 低、Context 空和最终拒答；最后一个症状不应让修复落到 prompt。
  4. 只看平均分会掩盖某租户越权或某文档类型解析崩溃，错误发布后再由线上反馈放大样本偏差。

贯穿案例 05：Gold、版本与第一偏离点

同一差旅问题的评测单元固定主体、上海、P6、as_of=2026-07-20 和允许来源 v7；v6 与其他租户制度属于 forbidden evidence。示例金额继续使用 ¥X，不代表真实政策或固定阈值。

Gold: { answerable: true, evidence: ["v7/sec-4.2/row-sh-p6"],
        answer: "¥X/晚（含适用条件）", forbidden: ["v6/*", "tenant-other/*"] }
versions: { corpus: "c17", parser: "p8", index: "2026-07", retriever: "r12",
            reranker: "rr5", prompt: "g9", model: "m4", policy: "p19" }
expected: source ✓ -> parse ✓ -> index ✓ -> TopK ✓ -> context ✓ -> claim ✓
actual:   source ✓ -> parse ✓ -> index ✓ -> TopK ✗
first_divergence: retrieval-ranking

Oracle 将 Gold Evidence 直接交给 generation 后回答与引用均通过，支持“生成有能力、召回链路先失败”的假设；随后只替换 retriever 做 ablation，并用同一版本矩阵复测。

指标树、版本矩阵与实验闭环

评测必须从数据供应链到用户结果分层：摄取看 source coverage、解析成功/置信、ACL 完整、freshness 与 deletion lag；检索看 Recall@K、MRR/nDCG、过滤误杀和来源多样性；上下文看 gold coverage、噪声、截断与 token；生成看 claim support、citation validity/correctness/completeness、abstention；系统看 P50/P95/P99、成本、错误率与安全 hard fail。每条结果绑定 corpus、parser、chunker、embedding、index、retriever、reranker、prompt、model、judge 和 policy 版本。

实验以失败假设开始：固定数据集和版本，只改变一个变量；先看组件指标是否按预期变化，再看端到端收益与切片副作用。用 oracle evidence 判断生成上限，用 ablation 移除 rewrite/rerank/cache 判断真实贡献；报告样本量、置信区间或重复运行波动，避免用一次平均分宣称改进。

用质量—延迟—成本 Pareto 与 Hard Fail 决定发布

不存在脱离场景的统一 top-k、chunk size 或阈值。为不同风险切片画质量、P95 延迟和单位请求成本的 Pareto frontier：被其他方案三项同时压制的配置淘汰；候选方案再依据业务 SLO 选择。跨租户泄露、撤权后仍命中、关键 claim 无支持、伪造引用和注入成功属于 hard fail，不能被平均分抵消。发布采用 shadow/canary，观察线上分布漂移和人工纠错，异常时回滚到固定版本矩阵。

指标测什么，以及不能推出什么

指标测量对象不能单独推出
Recall@Kgold evidence 是否进入前 K前排干净、答案正确
Precision@K前 K 相关比例关键证据没有漏
MRR首个相关证据排名多个必要证据完整
nDCG分级相关性的排序质量语义支持或授权正确
ANN RecallANN 相对 Flat/精确近邻的损失embedding 语义质量
Groundedness / Faithfulnessclaim 是否由 context 支持来源真实、最新、已授权
Citation 指标validity/correctness/completeness/support四者可互相替代

规则适合可确定的版本、ACL、ID、格式和 hard fail；人工标注定义 Gold、answerability 和争议仲裁；LLM Judge/NLI 用于规模化语义筛查但必须版本化与校准。RAGAS 是可选工具映射，不是指标语义的权威源。

评测集与单变量实验模板

样本来自产品需求、专家编写、线上失败/低置信/人工纠错和对抗生成；入库前脱敏去重，由双人或仲裁标注 query、subject/time、answerability、acceptable claims、Gold Evidence、forbidden evidence 与风险级别。保留未参与调参的 hold-out，并按来源、文档类型、语言、时效、权限、可答性、长尾和真实流量频率切片；报告标注一致性与分歧原因。

experiment:
  hypothesis: "结构感知 chunk 改善表格问题 gold coverage"
  fixed: corpus/parser/embedding/index/retriever/reranker/prompt/model/policy
  grid: chunk_size × top_k × top_n × context_budget
  one_change: chunker_version
  measures: Recall@K, nDCG, claim_support, citation, abstention, p95, tokens, unit_cost
  slices: table / current-version / ACL-negative / unanswerable
  hard_fail: tenant_leak | revoked_hit | fabricated_citation | unsupported_high_risk_claim
  decision: Pareto candidate / reject / canary / rollback

Embedding、Reranker、Prompt 分别做单变量 ablation；oracle retrieval 将 Gold 直接放入 context 测生成上限，oracle generation/人工答案帮助识别检索上限。Pareto 只保留质量、P95、token/单位成本、拒答率之间不被同时支配的配置，再按业务风险选型；shadow/canary 监控分布漂移，触发预设条件即回滚固定版本矩阵。

工程要点

  1. 评测样本包含 query、subject、time、gold evidence、acceptable answer、forbidden evidence 和 answerability。
  2. 固定 corpus、index、retriever、reranker、prompt、model 与 policy 版本，变更其中一项时用同一 golden set 对比。
  3. 用 oracle retrieval 隔离 generation：若给定正确证据仍答错，优先查生成；若正确证据未入 Top-K，查检索上游。
  4. 线上记录 p50/p95 延迟、空结果率、拒答率、点击与人工纠错；高风险回归必须阻断发布。

常见坑

  • 只用 LLM-as-judge 看最终答案，不保存检索候选和版本。
  • 评测集全是可回答问题，无法验证拒答与越权保护。
  • 以平均分掩盖某租户、语言或文档类型的严重退化。
  • 检索变差就扩大 K，让成本和噪声上升而根因仍在。

理论追问

请先脱稿回答，再展开参考答案核对边界、证据和取舍。

  1. 为什么只评最终回答无法定位 RAG 失败？

    查看参考答案

    最终错误可能来自原文缺失、摄取过期、召回漏失、重排错误、上下文截断或生成不忠实。必须分别评估数据、retrieval、reranking、grounding 和任务结果，并用 Trace 找到首个偏离点。

  2. RAG 评测集应包含哪些类型的样本？

    查看参考答案

    除了常规问答，还应包含无答案、同义改写、精确编号、跨片段、时效冲突、权限拒绝、对抗注入和长尾来源。每条样本要有允许来源、关键证据、主体权限和拒答期望，而不只是一个参考文本。

  3. 检索指标高但任务成功率低，下一步怎么诊断？

    查看参考答案

    检查 top-k 中的证据是否完整、重排和上下文组装是否丢失关键片段、生成是否引用错位，以及业务问题是否需要工具而非文档。高 recall 不保证证据可用，也不保证模型忠实利用。

  4. 线上 Trace 如何进入离线 RAG 回归集？

    查看参考答案

    先按失败、低置信、人工纠正和高价值切片采样，脱敏并去重，保留查询、主体权限、索引版本、证据和结果；再由人工确认期望与合法来源后版本化入库，不能把未核实线上回答直接当金标。

  5. 哪些 RAG 失败应设置 hard fail？

    查看参考答案

    跨租户或跨权限泄露、伪造引用、引用不支持关键断言、把恶意文档指令当系统指令，以及在必须有证据的高风险问题上编造答案，都应 hard fail，不能被平均相关性或流畅度抵消。

面试表达

RAG 回答错误如何定位？

我从 gold evidence 反向检查：语料是否存在且新鲜、解析和 chunk 是否保留、索引版本是否包含、授权过滤是否误杀、Top-K 是否召回、rerank 是否保留、context packing 是否截断、声明是否由证据支持、引用是否指向正确版本。通过 oracle evidence 和阶段 ablation 隔离生成问题，再把复现样本加入版本化回归集。

日常实践关键点

阅读清单前，先用差旅样本回忆“可执行 Gold → 完整版本快照 → Corpus/检索/生成分层指标 → First Divergence → Oracle/Ablation → 切片门禁”；再说明 v7 证据未入 Top-K 时，为何不能先改 prompt。

核心判断

最终答案分数只能说明症状，日常诊断要从版本化 Gold Evidence 反向寻找 Corpus、Ingestion、Retrieval、Rerank、Context、Generation、Citation 或 Policy 的第一个偏离点，并让每次发布都可比较、可止损、可回滚。

日常实践清单

  • 定义可执行的 Gold Evidence：每条样本保存 query、subject、时间、answerability、允许和禁止来源、关键 locator 与可接受答案边界；只写参考回答无法判断检索在哪一步失败。
    触发信号
    评测只有参考答案，无法判断候选、引用或拒答是否正确。
    判断与定位
    检查样本是否包含主体、时间、answerability 与证据 locator。
    执行动作
    标注允许和禁止来源、关键 claim 及可接受回答边界。
    验证标准
    不同评审者能据此一致判断召回、支持和拒答结果。
    止损与取舍
    歧义样本先隔离复审，不让错误 Gold 驱动系统调参。
    面试转述
    强调评测单元是 query、身份、时间与证据，而非单一答案。
  • 固定复现版本快照：评测结果绑定 corpus、parser、chunker、index、retriever、reranker、prompt、model 和 policy 版本；缺少任一关键版本的线上样本不直接作为回归结论。
    触发信号
    同一评测多次结果不同，或线上失败无法在离线环境重现。
    判断与定位
    逐项核对数据、解析、切分、索引、模型、prompt 和策略版本。
    执行动作
    评测 run 固定完整版本矩阵、输入快照与随机性配置。
    验证标准
    同快照重复执行结果稳定，差异能归因到显式变更组件。
    止损与取舍
    版本不全的线上样本仅作线索，不直接进入发布回归结论。
    面试转述
    说明 RAG 复现必须固定整条链路，不能只记录模型版本。
  • 按真实流量切片：除总体分外，持续观察租户、语言、文档类型、查询意图、风险等级和长尾来源；某个高价值切片退化时不能被平均分提升抵消。
    触发信号
    总体指标上涨，但特定租户、语言或高风险意图投诉增加。
    判断与定位
    按真实流量权重切片，对比基线、样本量与置信区间。
    执行动作
    维护租户、语言、类型、风险和长尾来源的固定切片看板。
    验证标准
    关键切片均过门禁，平均分不能抵消高价值切片退化。
    止损与取舍
    样本过少先延长观察或人工复核，高风险退化直接停灰度。
    面试转述
    用平均数掩盖长尾问题解释为什么评测必须切片。
  • 寻找第一偏离点：把 expected 与 actual 的原文、chunk、候选、排序、上下文、claim 和 citation 逐层对齐；首个错误阶段才是优先修复对象，后续异常通常只是传播结果。
    触发信号
    错答同时伴随低召回、差引用和不忠实生成，多团队并发调参。
    判断与定位
    逐层对齐 expected/actual，记录首个不一致的组件和版本。
    执行动作
    只修首个偏离阶段，冻结其余变量并保存前后 Trace。
    验证标准
    该阶段恢复后下游传播异常同步消失，且同桶回归不退化。
    止损与取舍
    无法定位先补观测，不用同时改 K、prompt 和模型碰运气。
    面试转述
    按原文到引用的顺序演示如何系统化诊断 RAG 错答。
  • 使用 oracle 与 ablation：把正确证据直接交给生成以隔离生成能力，再一次只替换 parser、retriever、reranker 或 prompt；若同时修改多阶段，就无法证明收益来自哪里。
    触发信号
    无法判断错误来自检索还是生成，或一版同时修改多个组件。
    判断与定位
    先把 Gold Evidence 直接输入生成，观察回答与引用是否恢复。
    执行动作
    一次只替换 parser、retriever、reranker 或 prompt 做消融。
    验证标准
    对照组、样本版本和指标一致，收益能归因到单一变化。
    止损与取舍
    Oracle 仍失败就停止检索调参，优先修生成或验证契约。
    面试转述
    用控制变量和 oracle 上界回答如何证明优化有效。
  • 设置 hard fail 与发布门禁：跨权限泄露、伪造引用、关键 claim 无支持、恶意文档改变指令和高风险无证据强答必须阻断发布，不能由相关性或流畅度平均分抵消。
    触发信号
    平均质量变好，但出现任何越权、伪造引用或高风险强答。
    判断与定位
    将安全和事实硬失败与普通相关性指标分开统计和审阅。
    执行动作
    把越权、注入、无支持 claim 等注册为零容忍发布门禁。
    验证标准
    故意注入 hard fail 样本可自动阻断，审批不能被平均分绕过。
    止损与取舍
    灰度中命中立即自动回滚并保全 Trace，再进入安全复盘。
    面试转述
    区分可权衡质量指标与绝不能被平均的安全底线。
  • 治理灰度与线上回灌：先做 shadow 对比，再按风险和租户灰度并监控回滚信号；线上失败、低置信和人工纠正样本需脱敏、去重、人工确认 Gold 后才进入版本化回归集。
    触发信号
    离线集与真实流量脱节，线上失败样本却未经治理直接回灌。
    判断与定位
    核对采样理由、隐私字段、重复簇、人工 Gold 与数据版本。
    执行动作
    shadow 后按风险灰度，失败样本脱敏去重并人工确认入库。
    验证标准
    回灌样本可追溯且不含敏感信息，新增回归能复现线上失败。
    止损与取舍
    隐私或标签不确定就隔离，不以快速扩充数据集牺牲可信度。
    面试转述
    讲清反馈闭环不是自动学习，而是受审计的数据治理流程。

排障顺序

从 Gold Evidence 反向比对：Corpus 是否存在且新鲜 → 解析和 chunk 是否保留完整事实 → active index 与 ACL 是否包含 → 各路候选是否召回 → fusion 与 rerank 是否保留 → context pack 是否截断 → generation 是否忠实形成 claim → citation 与 policy 是否正确。记录第一个 expected/actual 分叉，再用 oracle 或单阶段 ablation 复证。

应留证据

保留评测集和 Gold Evidence 版本、全组件版本矩阵、逐阶段 expected/actual Trace、Recall@K 与 nDCG、groundedness、citation correctness、拒答与 hard fail 结果、各流量切片看板、shadow/灰度报告、发布门禁、回滚原因和线上样本入库审计。

日常面试要点

关联实践：Gold Evidence、版本快照

高频问法

如何构建一套可执行、可复现的 RAG 评测集，为什么只有参考答案远远不够？

60–90 秒骨架

评测单元包含 query、身份、时间、answerability、允许与禁止证据及回答边界；每次 run 固定全链路版本矩阵和输入快照。

必讲证据

展示 Gold Evidence schema、标注一致性、版本矩阵、重复运行差异和歧义样本复审记录。

常见追问

Gold 从哪里来？文档更新后怎么办？线上样本可直接入库吗？如何处理多种正确答案？

回答边界

清楚说明样本量、来源和覆盖不足；未做专家标注时不能声称评测集代表全部生产流量或绝对正确。

关联实践：第一偏离点、Oracle / Ablation

高频问法

线上出现 RAG 错答时如何定位根因，并用实验而不是直觉证明修复来自哪个阶段？

60–90 秒骨架

按原文、chunk、候选、排序、上下文、claim、citation 对齐首个偏离点；随后用 Gold Evidence oracle 和单变量 ablation 复证。

必讲证据

给出 expected/actual Trace、first_divergence 字段、oracle 对照、单阶段 diff 与相同切片回归结果。

常见追问

多个阶段都错先修哪个？Oracle 也失败说明什么？没有候选 Trace 怎么办？如何避免过拟合单例？

回答边界

观测缺失时应说“当前无法确定”，先补证据；不要把相关性当因果，也不编造事故 MTTR 或优化收益。

关联实践：流量切片、发布门禁、线上反馈

高频问法

RAG 版本如何从离线评测走到灰度发布，哪些指标必须 hard fail，线上反馈又怎样进入回归集？

60–90 秒骨架

先 shadow，再按风险和租户灰度；切片质量与安全 hard fail 共同门禁，命中回滚信号即止损；反馈脱敏去重并人工确认 Gold。

必讲证据

展示切片看板、越权与伪造引用门禁、灰度决策、回滚 receipt 和线上样本入库审计链。

常见追问

平均分上涨能否放量？小切片怎么判断？用户点赞能当标签吗？何时触发自动回滚？

回答边界

没有生产灰度经历时按方案设计回答，不编造发布规模、事故数量、转化收益或“零回滚”记录。

经验真实性：评测面试回答必须区分生产实战、个人实验和方案设计；只引用实际拥有的 Gold、版本快照、对照实验和灰度记录，未知样本覆盖、指标提升和业务收益要明确未知，不得虚构。
