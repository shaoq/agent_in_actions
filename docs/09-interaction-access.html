UNIT.02

架构位置与边界

职责边界 · 交互层拥有协议、身份上下文、会话呈现和用户体验，不拥有任务执行生命周期；它把规范化意图交给编排，并呈现编排事件。HITL 的界面在交互层，暂停与恢复状态由编排拥有。
契约面组件级定义
输入渠道事件、actor、session、内容、多模态附件、locale、consent
规范化输出intent、identity context、channel capabilities、correlation ID
编排返回progress、clarification、result、error、approval、handoff 事件
用户呈现流式文本、结构化卡片、可访问替代、恢复与撤销入口

UNIT.03

模式与取舍

MODE 1

同步对话

短任务和快速反馈；简单但易受超时限制。

MODE 2

流式事件

长生成与进度；体验好但需顺序、重连和去重。

MODE 3

澄清循环

信息不足或高风险；降低误执行但增加轮次。

MODE 4

人工移交

需要判断、授权或情绪处理；需最小充分上下文。

UNIT.05

生产工程约束

  • 输入大小、类型和附件在边缘校验并恶意内容隔离。
  • 任何审批或恢复动作绑定主体、动作摘要、版本和过期时间。
  • 错误信息对用户可理解，对工程侧保留关联 ID。
  • 支持键盘、屏幕阅读器、字幕、对比度和非流式降级。
UNIT.06

面试表达

30 秒回答

交互与接入层是协议和信任边界：它规范化渠道、身份、会话、同意和多模态输入，将请求交给编排，再把进度、澄清、结果或人工移交适配回用户；它不是隐藏在 UI 里的第二个编排器。

3 分钟回答结构

从渠道和身份规范化开始，说明流式事件、重连和多模态能力；再讲澄清与 HITL 中 UI 和编排状态的分工；最后落到同意、错误恢复、移交最小化和无障碍，并用事件 ID 与关联 ID 支撑诊断。

表达检查 · 不要只报框架或产品名；必须讲清输入输出、owner、证据、失败归因和为什么选择该模式。
