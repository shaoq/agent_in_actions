学习契约

Understand

理解

能用自己的话解释编排层为什么存在、如何推进任务，以及它和推理内核、工具、记忆、治理层的边界。

Build

实现

能实现一个有显式状态、终止保护、失败状态和 trace 的最小单 Agent loop。

Debug

调试

能从 trace 判断失败发生在 decide、act、observe 还是 terminate，并提出针对性修复。

Explain

表达

能在 30 秒说明定义，在 3 分钟讲清数据流、失败模式与方案取舍，并应对追问。

范围边界 · 本单元只实现单 Agent 编排。Multi-Agent 只学习何时值得使用；tool schema、MCP、权限、重试和幂等进入后续工具调用单元。

显式状态模型

字段作用错误做法实验对应
task保存本次运行目标，作为所有决策的稳定输入。只存在最初 prompt，后续步骤无法核对目标。LoopState.task
step / max_steps提供循环预算和确定性停止条件。只期待模型“自己知道何时停”。step、max_steps
observations保存动作反馈，使下一步能根据环境更新。工具结果只打印日志，没有进入状态。Observation 列表
status区分 running、completed、max_steps 与 failed。所有退出都返回空字符串或异常。RunStatus
trace记录每步 phase、event type 与关键 payload。只有自然语言日志，无法聚合和回放。TraceEvent 列表
状态不是记忆 · LoopState 是一次运行的控制状态；跨会话保存的用户偏好或历史经验属于 Memory。持久化 checkpoint 可以保存运行状态，但“存在哪里”和“语义属于哪一层”是两个问题。

FIG.06

编排模式取舍

ReAct

边观察边决策

适合路径难以预先确定、环境反馈会改变下一步的任务。优点是适应性强；代价是调用次数、轨迹漂移和终止控制更难预测。

Plan And Execute

先规划再执行

适合步骤依赖清楚、可并行或需要提前审批的任务。优点是可审计；代价是计划可能很快过时，需要重规划策略。

Deterministic Workflow

固定工作流

步骤稳定、合规要求高时优先使用普通代码或状态机，只在判断节点调用模型。不要为了“Agent 化”牺牲可预测性。

Multi-Agent

多角色协作

只有当上下文隔离、并行分工或独立专业能力带来的收益大于协调成本时才采用。否则先提升单 Agent 的工具、指令和状态设计。

FIG.07

面试表达

30 秒回答

编排层负责推进 Agent 任务，而不是提供模型智能。它维护显式状态，把当前上下文交给推理内核产生下一步决策，再执行动作、记录观察并判断继续、完成、失败或转人工。生产设计的重点是终止条件、状态持久化、失败恢复、权限边界和 trace，而不只是写一个 while loop。

3 分钟回答结构

先定义：编排是任务控制面。再讲链路：input → state → decide → act → observe → terminate。再讲边界：模型提出决策，工具负责动作，治理层校验，编排层维护控制状态。再讲可靠性：最大步数、超时、幂等、checkpoint、人工接管和结构化 trace。最后讲取舍：固定流程优先用确定性工作流；未知路径用 ReAct；步骤稳定且可审批时用 plan-and-execute；Multi-Agent 必须证明分工收益。

工程案例表达 · 不要只说“我用了 LangGraph”。应说明状态有哪些字段、哪个事件触发路由、如何防止无限循环、动作失败是否重试、trace 如何定位问题，以及为何选择该编排方式。
