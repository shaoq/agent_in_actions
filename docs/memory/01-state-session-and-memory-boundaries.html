01 状态、会话与记忆边界

学习目标

  • 用 owner、生命周期、作用域和恢复语义区分 state、session 与 memory。
  • 解释短期上下文窗口为什么不等于持久记忆。
  • 定义从会话事实到长期记忆的显式晋升门禁。

核心概念

链路总览与现有讲解

状态、会话与记忆边界属于记忆系统：输入是带主体、来源、目的、同意和时间的经历候选，输出是类型化、版本化、可撤回的 MemoryRecord 或拒绝结果。学习时先沿现有图、表或主题章节理解链路，再用后续 stage 检查每个中间产物的不变量。

对象owner / 生命周期典型内容恢复语义
Run state编排层 / 单次 run当前 step、工具结果、重试计数checkpoint 后精确恢复
Session context接入+编排 / 一次会话近轮消息、临时引用、UI 状态按 session 隔离，过期即清理
Long-term memory记忆系统 / 跨会话确认偏好、情景、稳定事实、程序版本化、可撤回、可审计
External knowledge知识 / RAG / 资料生命周期制度、手册、代码文档按 source 版本与 ACL 更新

核心环节：从输入到可验证输出

1. 分类 State、Session 与 Memory

分类 State、Session 与 Memory决定记录的 owner、生命周期、读取方式和更新规则；类型是行为契约，不是便于展示的标签。

职责与 I/O
memory-owner 在“分类 State、Session 与 Memory”环节接收 带主体、来源、目的、同意和时间的经历候选，产出 类型化、版本化、可撤回的 MemoryRecord 或拒绝结果；输出必须能由下游按版本消费。
工作机制
先按真相来源与未来用途区分 RunState、Session、Memory、Knowledge，再在 Memory 内区分工作、情景、语义、偏好与程序记录。本环节把 带主体、来源、目的、同意和时间的经历候选 中与“分类 State、Session 与 Memory”相关的事实转成 类型化、版本化、可撤回的 MemoryRecord 或拒绝结果 的一部分。
必须成立的约束
分类必须在写入前确定并进入 schema；同一事实不能靠复制到多个类型解决读取问题。本章共同约束是：主体、目的、来源、置信、TTL 和同意必须可证明；违反时必须产生可区分的拒绝、未知或升级结果。
难点与取舍
粒度过粗会让检索与治理混乱，过细会提高迁移成本；类型集合由真实读写路径和失败样本稳定后再扩展。本层总体取舍是 个性化收益与隐私、召回率与错误长期化风险，参数必须引用版本化数据、SLO 或风险基线。
边界与注意事项
会话文本只是来源，外部制度属于 RAG；错误类型不能用更高相似度掩盖。记忆系统 的总边界是：保存 Agent 经历而非外部制度；不替代 RunState、Session 或 RAG；错误会继续污染 后续上下文注入、个性化 Decision 和治理审计。

2. Memory Candidate

Memory Candidate阶段产生的是带来源和治理状态的候选事实，未经验证、同意与目的检查不得成为长期可读记忆。

职责与 I/O
memory-owner 在“ Memory Candidate”环节接收 带主体、来源、目的、同意和时间的经历候选，产出 类型化、版本化、可撤回的 MemoryRecord 或拒绝结果；输出必须能由下游按版本消费。
工作机制
候选从显式用户陈述或已验收事件提取，附 source_event、subject、purpose、confidence、consent 与敏感类别后进入 accept / reject / confirm。本环节把 带主体、来源、目的、同意和时间的经历候选 中与“ Memory Candidate”相关的事实转成 类型化、版本化、可撤回的 MemoryRecord 或拒绝结果 的一部分。
必须成立的约束
来源必须可回放，敏感字段按目的最小化；模型摘要不能提升证据等级，同意范围不能由相似语义推断。本章共同约束是：主体、目的、来源、置信、TTL 和同意必须可证明；违反时必须产生可区分的拒绝、未知或升级结果。
难点与取舍
自动写入提高覆盖却会长期化误解；确认频率依据错误成本、稳定性和用户摩擦分级。本层总体取舍是 个性化收益与隐私、召回率与错误长期化风险，参数必须引用版本化数据、SLO 或风险基线。
边界与注意事项
候选不是 RunState 的 checkpoint，也不是把 RAG 文档复制到用户名下。记忆系统 的总边界是：保存 Agent 经历而非外部制度；不替代 RunState、Session 或 RAG；错误会继续污染 后续上下文注入、个性化 Decision 和治理审计。

3. 晋升门禁

晋升门禁只有在当前主体、目的、时效和冲突优先级同时满足时才能影响本次 Decision；“被召回”不等于“应注入”。

职责与 I/O
memory-owner 在“晋升门禁”环节接收 带主体、来源、目的、同意和时间的经历候选，产出 类型化、版本化、可撤回的 MemoryRecord 或拒绝结果；输出必须能由下游按版本消费。
工作机制
读取先做 tenant / subject / purpose 过滤，再按类型、时间、置信和任务相关性排序；晋升与注入分别保留 PolicyDecision。本环节把 带主体、来源、目的、同意和时间的经历候选 中与“晋升门禁”相关的事实转成 类型化、版本化、可撤回的 MemoryRecord 或拒绝结果 的一部分。
必须成立的约束
当前显式输入优先于旧偏好，撤回或过期记录不可命中；返回内容必须带 memory_id、来源、版本和置信。本章共同约束是：主体、目的、来源、置信、TTL 和同意必须可证明；违反时必须产生可区分的拒绝、未知或升级结果。
难点与取舍
召回越宽个性化越强但污染面越大；top-k、TTL 和最低置信由时间切片与关闭 Memory 的消融结果校准。本层总体取舍是 个性化收益与隐私、召回率与错误长期化风险，参数必须引用版本化数据、SLO 或风险基线。
边界与注意事项
Memory 提供候选上下文，推理决定如何引用，编排决定何时读取；不能用向量相似度恢复控制状态。记忆系统 的总边界是：保存 Agent 经历而非外部制度；不替代 RunState、Session 或 RAG；错误会继续污染 后续上下文注入、个性化 Decision 和治理审计。

5. 用消融验证边界

用消融验证边界必须让记录的变更、衰减、迁移、访问和删除都可证明到最终派生副本；不可见不等于已删除。

职责与 I/O
memory-owner 在“用消融验证边界”环节接收 带主体、来源、目的、同意和时间的经历候选，产出 类型化、版本化、可撤回的 MemoryRecord 或拒绝结果；输出必须能由下游按版本消费。
工作机制
版本图记录 supersede、migrate、expire、delete 与派生索引 / 缓存，主体请求和治理作业沿 lineage 传播并产生 receipt。本环节把 带主体、来源、目的、同意和时间的经历候选 中与“用消融验证边界”相关的事实转成 类型化、版本化、可撤回的 MemoryRecord 或拒绝结果 的一部分。
必须成立的约束
保留期、删除 SLA、合法留存、迁移版本和审计访问必须显式；删除后重建与备份恢复也不能让记录重新可读。本章共同约束是：主体、目的、来源、置信、TTL 和同意必须可证明；违反时必须产生可区分的拒绝、未知或升级结果。
难点与取舍
更长保留利于个性化与归因，却增加隐私和陈旧风险；期限由用途、法规、用户预期和实测价值共同决定。本层总体取舍是 个性化收益与隐私、召回率与错误长期化风险，参数必须引用版本化数据、SLO 或风险基线。
边界与注意事项
Memory owner 证明记录生命周期，平台 owner 证明物理存储传播，治理 owner 处理主体权利与审计。记忆系统 的总边界是：保存 Agent 经历而非外部制度；不替代 RunState、Session 或 RAG；错误会继续污染 后续上下文注入、个性化 Decision 和治理审计。

失败传播与 first divergence

  1. 上游输入：先确认 带主体、来源、目的、同意和时间的经历候选 的身份、作用域、时间与版本是否一致；输入错误不应在本层被“修正”为新事实。
  2. 首个偏离：按 分类 State、Session 与 Memory → Memory Candidate → 晋升门禁 → 按主体与目的读取 → 用消融验证边界 比较预期与实际中间产物，第一个不满足概念不变量的 stage 才是本章候选根因。
  3. 下游影响：本层错误会改变 后续上下文注入、个性化 Decision 和治理审计；最终症状可能在更晚阶段出现，但不能从症状直接倒推 Prompt。
  4. 归属边界：memory-owner修复本层契约或机制；若 first divergence 位于相邻层，保留证据并交给相应 owner，不跨层调参。

工程要点

  1. 所有对象都带 tenant_id、subject_id、purpose、created_at、expires_at；长期记忆还要带 provenance、consent、confidence 与 policy_version。
  2. 会话只产生 memory candidate；确定性规则或受控评估决定接受、拒绝、合并或等待确认。
  3. 读取长期记忆时返回“内容 + 来源 + 时间 + 置信度 + 可见范围”，由编排注入上下文。
  4. run state 的一致性由编排 checkpoint 解决；不要拿向量相似度恢复关键控制状态。

常见坑

  • 保存全部对话：放大隐私、成本和投毒面，也没有形成可用记忆。
  • 跨用户复用：相似 query 不能证明相同 subject 或授权范围。
  • 让 LLM 决定真相：摘要只是候选，必须保留来源与验证状态。
  • 把手册存为“语义记忆”：外部资料仍应走知识 / RAG 的来源与 ACL 链路。

理论追问

请先脱稿回答，再展开参考答案核对边界、证据和取舍。

  1. Run State、Session 和长期 Memory 的生命周期分别是什么？

    查看参考答案

    Run State 服务一次执行，包含步骤、Observation、审批和终态；Session 维持一段交互的渠道与对话连续性；长期 Memory 跨会话保存经过治理、未来仍有价值的经历。三者可共用存储基础，但必须有类型、权限和 TTL 边界。

  2. 为什么外部制度文档不应直接存入用户 Memory？

    查看参考答案

    制度属于 Knowledge/RAG，具有来源、版本、ACL 和组织级生命周期；写进用户 Memory 会失去权威来源并在制度更新后残留旧事实。Memory 可以记录“用户曾处理过某类制度问题”的经历，但外部条款仍应实时检索。

  3. Working Memory 应由推理内核还是编排层持有？

    查看参考答案

    其工程载体应属于编排维护的当前 State，推理内核只消费本步需要的工作上下文并返回候选更新。若让模型私有上下文成为唯一工作记忆，系统就无法恢复、审计或判断哪些内容是事实。

  4. 为什么不能把全部聊天历史自动当作 Memory？

    查看参考答案

    聊天中包含临时诉求、错误陈述、敏感数据和对抗指令，且多数内容没有未来价值。写入前要做主体确认、价值判断、来源与置信、同意、去重、冲突和保留期限治理。

  5. Agent 因旧偏好给出错误建议时，如何证明问题来自 Memory 而非 RAG？

    查看参考答案

    Trace 要区分 memory read 与 retrieval 事件，记录记录 ID、主体、版本、来源、分数和写入时间。回放时分别移除 Memory 与 RAG 证据做消融，找到首个错误上下文，并检查旧偏好是否覆盖了更新事实。

面试表达

状态、会话和记忆有什么区别？

我从生命周期和 owner 切分：编排状态服务一次 run 的确定性推进；会话上下文服务一次交互会话；长期记忆跨会话沉淀经验证的经历。会话内容只有经过同意、作用域、置信度与保留策略门禁后才能晋升。外部制度和手册属于知识 / RAG，不因检索技术相似就归为记忆。

日常实践关键点

核心判断

本章的日常工作不是记住名词，而是把“状态、会话与记忆边界”变成可执行控制环：先识别触发信号，再用版本化证据定位首个偏离点，由 memory-owner 执行动作，最后用成功、拒绝、降级三类结果验收。推荐顺序是 分类 State、Session 与 Memory → 生成 Memory Candidate → 执行晋升门禁 → 按主体与目的读取 → 用消融验证边界；任何参数都必须从真实 SLO、风险基线或失败样本推导。

深度实践卡

  • 1. 分类 State、Session 与 Memory
    触发信号
    当企业差旅 Agent 的“状态、会话与记忆边界”链路出现旧偏好覆盖新事实、跨主体命中、错误长期化或删除后仍可读取，或变更直接触及“分类 State、Session 与 Memory”的输入、策略或 owner 时，必须启动这项检查。
    诊断顺序
    诊断“分类 State、Session 与 Memory”时先冻结 subject_id、purpose、来源、同意状态、置信度、记录版本与读写 Trace。按 owner、生命周期、作用域、真相来源和更新方式逐项分类，并检查读取策略是否让低置信旧记录覆盖当前显式输入。
    执行动作
    落实“分类 State、Session 与 Memory”时，为 State、Session、Memory 与 Knowledge 设置不同 schema、TTL 和读取优先级；冲突时优先当前确认事实并返回来源。由 memory-owner 记录执行前后的版本、决定与接管条件。
    验证方法
    验收“分类 State、Session 与 Memory”时，用时间切片回放、跨主体负测、删除传播测试和关闭 Memory 的消融实验复核；同时比较正确、拒绝和降级结果，阈值只从真实基线与风险容忍度推导。
    止损与取舍
    分类或读取依据不明时不晋升长期记忆，关闭该类自动注入并通过消融确认影响；停止晋升或读取可疑记录，恢复上一策略版本，并保留受影响 memory_id 清单。对“分类 State、Session 与 Memory”先保证可恢复和可审计，再优化自动化率。
    面试记忆点
    围绕“分类 State、Session 与 Memory”准备一条“症状—首个偏离—动作—证据—止损”的真实或个人实验 Trace，并明确自己只承担的 memory-owner 职责。
  • 2. 生成 Memory Candidate
    触发信号
    当企业差旅 Agent 的“状态、会话与记忆边界”链路出现旧偏好覆盖新事实、跨主体命中、错误长期化或删除后仍可读取，或变更直接触及“生成 Memory Candidate”的输入、策略或 owner 时，必须启动这项检查。
    诊断顺序
    诊断“生成 Memory Candidate”时先冻结 subject_id、purpose、来源、同意状态、置信度、记录版本与读写 Trace。核对主体、来源、时间、置信度、同意与当前事实，确认错误发生在候选、晋升还是读取排序。
    执行动作
    落实“生成 Memory Candidate”时，把写入当受治理的状态变更：先候选、再验证、后晋升；读取时保留来源并允许拒绝注入。由 memory-owner 记录执行前后的版本、决定与接管条件。
    验证方法
    验收“生成 Memory Candidate”时，用时间切片回放、跨主体负测、删除传播测试和关闭 Memory 的消融实验复核；同时比较正确、拒绝和降级结果，阈值只从真实基线与风险容忍度推导。
    止损与取舍
    隔离可疑记录、回退上一版本并传播删除；不要用新的摘要覆盖而留下旧记录可见；停止晋升或读取可疑记录，恢复上一策略版本，并保留受影响 memory_id 清单。对“生成 Memory Candidate”先保证可恢复和可审计，再优化自动化率。
    面试记忆点
    围绕“生成 Memory Candidate”准备一条“症状—首个偏离—动作—证据—止损”的真实或个人实验 Trace，并明确自己只承担的 memory-owner 职责。
  • 3. 执行晋升门禁
    触发信号
    当企业差旅 Agent 的“状态、会话与记忆边界”链路出现旧偏好覆盖新事实、跨主体命中、错误长期化或删除后仍可读取，或变更直接触及“执行晋升门禁”的输入、策略或 owner 时，必须启动这项检查。
    诊断顺序
    诊断“执行晋升门禁”时先冻结 subject_id、purpose、来源、同意状态、置信度、记录版本与读写 Trace。分别核对技术执行、策略合规和业务验收，避免把 HTTP 成功、结构合法或模型自报完成当成最终成功。
    执行动作
    落实“执行晋升门禁”时，定义可机器判断的完成条件、hard fail、证据来源和验收 owner；证据不全时返回 unknown / pending 而不是 passed。由 memory-owner 记录执行前后的版本、决定与接管条件。
    验证方法
    验收“执行晋升门禁”时，用时间切片回放、跨主体负测、删除传播测试和关闭 Memory 的消融实验复核；同时比较正确、拒绝和降级结果，阈值只从真实基线与风险容忍度推导。
    止损与取舍
    任一硬门禁失败就保留上一稳定版本；仅软指标退化时才允许受控试运行并持续观察；停止晋升或读取可疑记录，恢复上一策略版本，并保留受影响 memory_id 清单。对“执行晋升门禁”先保证可恢复和可审计，再优化自动化率。
    面试记忆点
    围绕“执行晋升门禁”准备一条“症状—首个偏离—动作—证据—止损”的真实或个人实验 Trace，并明确自己只承担的 memory-owner 职责。
  • 4. 按主体与目的读取
    触发信号
    当企业差旅 Agent 的“状态、会话与记忆边界”链路出现旧偏好覆盖新事实、跨主体命中、错误长期化或删除后仍可读取，或变更直接触及“按主体与目的读取”的输入、策略或 owner 时，必须启动这项检查。
    诊断顺序
    诊断“按主体与目的读取”时先冻结 subject_id、purpose、来源、同意状态、置信度、记录版本与读写 Trace。重点确认决策主体、资源、动作、目的和策略版本是否在“使用点”仍一致，不能只看入口鉴权。
    执行动作
    落实“按主体与目的读取”时，把允许、拒绝和需审批做成类型化策略结果，默认拒绝缺字段或版本不明的请求。由 memory-owner 记录执行前后的版本、决定与接管条件。
    验证方法
    验收“按主体与目的读取”时，用时间切片回放、跨主体负测、删除传播测试和关闭 Memory 的消融实验复核；同时比较正确、拒绝和降级结果，阈值只从真实基线与风险容忍度推导。
    止损与取舍
    一旦主体或目的无法证明，立即撤销能力令牌并转人工确认，不用事后过滤替代前置授权；停止晋升或读取可疑记录，恢复上一策略版本，并保留受影响 memory_id 清单。对“按主体与目的读取”先保证可恢复和可审计，再优化自动化率。

  • 5. 用消融验证边界
    触发信号
    当企业差旅 Agent 的“状态、会话与记忆边界”链路出现旧偏好覆盖新事实、跨主体命中、错误长期化或删除后仍可读取，或变更直接触及“用消融验证边界”的输入、策略或 owner 时，必须启动这项检查。
    诊断顺序
    诊断“用消融验证边界”时先冻结 subject_id、purpose、来源、同意状态、置信度、记录版本与读写 Trace。检查样本、Gold、评分器、模型与策略版本是否可比，并查看关键切片而非只看聚合分。
    执行动作
    落实“用消融验证边界”时，固定评测单元、评分依据和争议处理流程，让规则、人工与 Judge 各自覆盖擅长的判断。由 memory-owner 记录执行前后的版本、决定与接管条件。
    验证方法
    验收“用消融验证边界”时，用时间切片回放、跨主体负测、删除传播测试和关闭 Memory 的消融实验复核；同时比较正确、拒绝和降级结果，阈值只从真实基线与风险容忍度推导。
    止损与取舍
    结论不可复现或评分分歧越界时冻结发布，把样本送人工仲裁并更新失败分类；停止晋升或读取可疑记录，恢复上一策略版本，并保留受影响 memory_id 清单。对“用消融验证边界”先保证可恢复和可审计，再优化自动化率。
    面试记忆点
    围绕“用消融验证边界”准备一条“症状—首个偏离—动作—证据—止损”的真实或个人实验 Trace，并明确自己只承担的 memory-owner 职责。

排障顺序

  1. 冻结请求、策略、模型 / 工具、数据和部署版本，避免边复现边变化。
  2. 从最早可验证输入开始，逐段对照契约、决策、动作与验收事件；在首个偏离点停止跨层调参。
  3. 由 memory-owner 判断修复、降级、回滚或转人工，并记录未验证假设。
贯穿案例｜企业差旅 Agent：员工请求预订超预算航班时，只把制度条款当知识证据，把员工靠窗偏好当记忆，把预订动作交给工具层；本章只处理 状态、会话与记忆边界 的控制与证据。若发现旧偏好覆盖新事实、跨主体命中、错误长期化或删除后仍可读取，先保留 trace_id 与版本，再在本层首个偏离点修复，避免把问题推给 Prompt。

应留证据

最小证据集应能回答谁在何时依据哪个版本、对什么输入做了什么决定，以及结果如何被验证。敏感值使用教学占位或受控引用，不把日志全文复制进面试材料。

实践型面试要点

表达时必须标记经历属于生产实战、个人实验或方案设计，只主张自己实际完成的职责，并用 Trace、评测或变更记录举证。企业差旅案例、示例版本和指标均为教学占位；没有验证线上规模、事故协作或长期效果时要主动说明。
