01 状态、会话与记忆边界

学习目标

  • 用 owner、生命周期、作用域和恢复语义区分 state、session 与 memory。
  • 解释短期上下文窗口为什么不等于持久记忆。
  • 定义从会话事实到长期记忆的显式晋升门禁。

核心概念

对象owner / 生命周期典型内容恢复语义
Run state编排层 / 单次 run当前 step、工具结果、重试计数checkpoint 后精确恢复
Session context接入+编排 / 一次会话近轮消息、临时引用、UI 状态按 session 隔离，过期即清理
Long-term memory记忆系统 / 跨会话确认偏好、情景、稳定事实、程序版本化、可撤回、可审计
External knowledge知识 / RAG / 资料生命周期制度、手册、代码文档按 source 版本与 ACL 更新

工程要点

  1. 所有对象都带 tenant_id、subject_id、purpose、created_at、expires_at；长期记忆还要带 provenance、consent、confidence 与 policy_version。
  2. 会话只产生 memory candidate；确定性规则或受控评估决定接受、拒绝、合并或等待确认。
  3. 读取长期记忆时返回“内容 + 来源 + 时间 + 置信度 + 可见范围”，由编排注入上下文。
  4. run state 的一致性由编排 checkpoint 解决；不要拿向量相似度恢复关键控制状态。

常见坑

  • 保存全部对话：放大隐私、成本和投毒面，也没有形成可用记忆。
  • 跨用户复用：相似 query 不能证明相同 subject 或授权范围。
  • 让 LLM 决定真相：摘要只是候选，必须保留来源与验证状态。
  • 把手册存为“语义记忆”：外部资料仍应走知识 / RAG 的来源与 ACL 链路。

理论追问

请先脱稿回答，再展开参考答案核对边界、证据和取舍。

  1. Run State、Session 和长期 Memory 的生命周期分别是什么？

    查看参考答案

    Run State 服务一次执行，包含步骤、Observation、审批和终态；Session 维持一段交互的渠道与对话连续性；长期 Memory 跨会话保存经过治理、未来仍有价值的经历。三者可共用存储基础，但必须有类型、权限和 TTL 边界。

  2. 为什么外部制度文档不应直接存入用户 Memory？

    查看参考答案

    制度属于 Knowledge/RAG，具有来源、版本、ACL 和组织级生命周期；写进用户 Memory 会失去权威来源并在制度更新后残留旧事实。Memory 可以记录“用户曾处理过某类制度问题”的经历，但外部条款仍应实时检索。

  3. Working Memory 应由推理内核还是编排层持有？

    查看参考答案

    其工程载体应属于编排维护的当前 State，推理内核只消费本步需要的工作上下文并返回候选更新。若让模型私有上下文成为唯一工作记忆，系统就无法恢复、审计或判断哪些内容是事实。

  4. 为什么不能把全部聊天历史自动当作 Memory？

    查看参考答案

    聊天中包含临时诉求、错误陈述、敏感数据和对抗指令，且多数内容没有未来价值。写入前要做主体确认、价值判断、来源与置信、同意、去重、冲突和保留期限治理。

  5. Agent 因旧偏好给出错误建议时，如何证明问题来自 Memory 而非 RAG？

    查看参考答案

    Trace 要区分 memory read 与 retrieval 事件，记录记录 ID、主体、版本、来源、分数和写入时间。回放时分别移除 Memory 与 RAG 证据做消融，找到首个错误上下文，并检查旧偏好是否覆盖了更新事实。

面试表达

状态、会话和记忆有什么区别？

我从生命周期和 owner 切分：编排状态服务一次 run 的确定性推进；会话上下文服务一次交互会话；长期记忆跨会话沉淀经验证的经历。会话内容只有经过同意、作用域、置信度与保留策略门禁后才能晋升。外部制度和手册属于知识 / RAG，不因检索技术相似就归为记忆。
