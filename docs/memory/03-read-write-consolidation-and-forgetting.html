03 读取、写入、整合与遗忘

学习目标

  • 设计候选写入、验证、去重、冲突整合和发布门禁。
  • 组合 scope、recency、relevance、confidence 与 risk 读取记忆。
  • 区分衰减、归档、逻辑删除和物理删除。

核心概念

链路总览与现有讲解

读取、写入、整合与遗忘属于记忆系统：输入是带主体、来源、目的、同意和时间的经历候选，输出是类型化、版本化、可撤回的 MemoryRecord 或拒绝结果。学习时先沿现有图、表或主题章节理解链路，再用后续 stage 检查每个中间产物的不变量。

核心环节：从输入到可验证输出

1. 校准记忆读取排序

校准记忆读取排序只有在当前主体、目的、时效和冲突优先级同时满足时才能影响本次 Decision；“被召回”不等于“应注入”。

职责与 I/O
memory-owner 在“校准记忆读取排序”环节接收 带主体、来源、目的、同意和时间的经历候选，产出 类型化、版本化、可撤回的 MemoryRecord 或拒绝结果；输出必须能由下游按版本消费。
工作机制
读取先做 tenant / subject / purpose 过滤，再按类型、时间、置信和任务相关性排序；晋升与注入分别保留 PolicyDecision。本环节把 带主体、来源、目的、同意和时间的经历候选 中与“校准记忆读取排序”相关的事实转成 类型化、版本化、可撤回的 MemoryRecord 或拒绝结果 的一部分。
必须成立的约束
当前显式输入优先于旧偏好，撤回或过期记录不可命中；返回内容必须带 memory_id、来源、版本和置信。本章共同约束是：主体、目的、来源、置信、TTL 和同意必须可证明；违反时必须产生可区分的拒绝、未知或升级结果。
难点与取舍
召回越宽个性化越强但污染面越大；top-k、TTL 和最低置信由时间切片与关闭 Memory 的消融结果校准。本层总体取舍是 个性化收益与隐私、召回率与错误长期化风险，参数必须引用版本化数据、SLO 或风险基线。
边界与注意事项
Memory 提供候选上下文，推理决定如何引用，编排决定何时读取；不能用向量相似度恢复控制状态。记忆系统 的总边界是：保存 Agent 经历而非外部制度；不替代 RunState、Session 或 RAG；错误会继续污染 后续上下文注入、个性化 Decision 和治理审计。

2. 写入候选

写入候选阶段产生的是带来源和治理状态的候选事实，未经验证、同意与目的检查不得成为长期可读记忆。

职责与 I/O
memory-owner 在“写入候选”环节接收 带主体、来源、目的、同意和时间的经历候选，产出 类型化、版本化、可撤回的 MemoryRecord 或拒绝结果；输出必须能由下游按版本消费。
工作机制
候选从显式用户陈述或已验收事件提取，附 source_event、subject、purpose、confidence、consent 与敏感类别后进入 accept / reject / confirm。本环节把 带主体、来源、目的、同意和时间的经历候选 中与“写入候选”相关的事实转成 类型化、版本化、可撤回的 MemoryRecord 或拒绝结果 的一部分。
必须成立的约束
来源必须可回放，敏感字段按目的最小化；模型摘要不能提升证据等级，同意范围不能由相似语义推断。本章共同约束是：主体、目的、来源、置信、TTL 和同意必须可证明；违反时必须产生可区分的拒绝、未知或升级结果。
难点与取舍
自动写入提高覆盖却会长期化误解；确认频率依据错误成本、稳定性和用户摩擦分级。本层总体取舍是 个性化收益与隐私、召回率与错误长期化风险，参数必须引用版本化数据、SLO 或风险基线。
边界与注意事项
候选不是 RunState 的 checkpoint，也不是把 RAG 文档复制到用户名下。记忆系统 的总边界是：保存 Agent 经历而非外部制度；不替代 RunState、Session 或 RAG；错误会继续污染 后续上下文注入、个性化 Decision 和治理审计。

3. 去重并整合记忆

去重并整合记忆必须保留事实的来源与时间关系，使合并后的记录仍可解释冲突和回退，不能用新摘要抹掉历史。

职责与 I/O
memory-owner 在“去重并整合记忆”环节接收 带主体、来源、目的、同意和时间的经历候选，产出 类型化、版本化、可撤回的 MemoryRecord 或拒绝结果；输出必须能由下游按版本消费。
工作机制
稳定键与语义候选先发现重复，再按来源权威、时间和确认状态执行 link、merge、supersede 或 quarantine。本环节把 带主体、来源、目的、同意和时间的经历候选 中与“去重并整合记忆”相关的事实转成 类型化、版本化、可撤回的 MemoryRecord 或拒绝结果 的一部分。
必须成立的约束
冲突双方、裁决理由与派生关系必须保留；不可信指令、越权主体和异常写入模式进入隔离而非自动合并。本章共同约束是：主体、目的、来源、置信、TTL 和同意必须可证明；违反时必须产生可区分的拒绝、未知或升级结果。
难点与取舍
激进整合降低存储和噪声却可能丢失时序差异；阈值由人工复核准确率与下游错误切片决定。本层总体取舍是 个性化收益与隐私、召回率与错误长期化风险，参数必须引用版本化数据、SLO 或风险基线。
边界与注意事项
整合改变 MemoryRecord 关系，不修改外部权威知识，也不替治理层完成安全事件处置。记忆系统 的总边界是：保存 Agent 经历而非外部制度；不替代 RunState、Session 或 RAG；错误会继续污染 后续上下文注入、个性化 Decision 和治理审计。

失败传播与 first divergence

  1. 上游输入：先确认 带主体、来源、目的、同意和时间的经历候选 的身份、作用域、时间与版本是否一致；输入错误不应在本层被“修正”为新事实。
  2. 首个偏离：按 校准记忆读取排序 → 写入候选 → 去重并整合记忆 → 衰减与遗忘 → 证明删除传播 比较预期与实际中间产物，第一个不满足概念不变量的 stage 才是本章候选根因。
  3. 下游影响：本层错误会改变 后续上下文注入、个性化 Decision 和治理审计；最终症状可能在更晚阶段出现，但不能从症状直接倒推 Prompt。
  4. 归属边界：memory-owner修复本层契约或机制；若 first divergence 位于相邻层，保留证据并交给相应 owner，不跨层调参。

工程要点

  1. 写：先做 sensitive classification、consent、novelty、confidence、risk 和 duplicate 检查；高风险事实要求用户确认。
  2. 整合：保留旧版本和支持事件，不可覆盖式“最后写入胜出”；冲突要显式标记 pending、confirmed 或 revoked。
  3. 读：先按 tenant、subject、purpose、consent 强过滤，再按 relevance、recency、confidence、utility 排序，返回有限预算。
  4. 用：模型看到记忆仍需标注不确定性；会改变权限、付款或医疗决策的记忆必须二次核验。
  5. 忘：TTL 衰减影响召回权重；撤回与删除要传播到主存、索引、缓存、摘要和备份保留策略，并留下不含原文的审计 tombstone。

常见坑

  • 每轮都写：产生重复、噪声和自我强化回路。
  • 只按相似度读：相关不等于被授权、最新或正确。
  • 摘要覆盖原事件：失去 provenance，无法纠错或删除派生事实。
  • 逻辑删除即完成：缓存、向量索引和派生摘要仍可能泄露。

  1. 什么信息值得写入长期 Memory？

    查看参考答案

    应同时满足未来复用价值、主体明确、来源可追溯、同意与权限允许、时效可管理，并通过去重和冲突检查。临时任务细节、未经证实陈述、敏感密钥和外部制度原文通常不应写入。

  2. Memory 检索为什么不能只按向量相似度排序？

    查看参考答案

    还要考虑主体与权限、记忆类型、时效、置信、来源、使用场景和负反馈。高相似但过期或属于其他用户的记录不能进入上下文；检索结果应作为候选，由编排与当前指令校验。

  3. 整合多条记忆时如何处理冲突？

    查看参考答案

    保留原始记录与 lineage，根据来源权威、时间、用户确认和作用域生成新版本；无法消解时同时保留冲突并降低置信，必要时向用户澄清。不能静默覆盖，否则无法撤销和解释。

  4. 遗忘为什么不仅是设置 TTL？

    查看参考答案

    还包括按用途删除、用户撤回、纠错、衰减、容量淘汰和法规保留例外。删除需要覆盖索引、缓存、派生摘要和备份策略，并提供可验证 tombstone；TTL 只是时间触发机制。

  5. 如何评估 Memory 是否真正提升 Agent？

    查看参考答案

    在同一任务集对比启用与禁用 Memory 的成功率、澄清次数、成本和延迟，同时报告误用率、过期率、跨主体泄露、写入准确率和删除可验证性。命中率高不代表使用正确。

我把 transcript 只当候选来源。写入经过同意、敏感性、置信度、重复和冲突门禁；整合时保留版本与支持事件；读取先做 subject/purpose 授权，再做相关性与新鲜度排序；遗忘既包括效用衰减，也包括撤回后的跨存储删除传播。每次读写都带 policy_version 和 trace_id。

日常实践关键点

核心判断

本章的日常工作不是记住名词，而是把“读取、写入、整合与遗忘”变成可执行控制环：先识别触发信号，再用版本化证据定位首个偏离点，由 memory-owner 执行动作，最后用成功、拒绝、降级三类结果验收。推荐顺序是 校准记忆读取排序 → 验证写入候选 → 去重并整合记忆 → 执行衰减与遗忘 → 证明删除传播；任何参数都必须从真实 SLO、风险基线或失败样本推导。

深度实践卡

  • 1. 校准记忆读取排序
    触发信号
    当企业差旅 Agent 的“读取、写入、整合与遗忘”链路出现旧偏好覆盖新事实、跨主体命中、错误长期化或删除后仍可读取，或变更直接触及“校准记忆读取排序”的输入、策略或 owner 时，必须启动这项检查。
    诊断顺序
    诊断“校准记忆读取排序”时先冻结 subject_id、purpose、来源、同意状态、置信度、记录版本与读写 Trace。核对主体、来源、时间、置信度、同意与当前事实，确认错误发生在候选、晋升还是读取排序。
    执行动作

排障顺序

  1. 冻结请求、策略、模型 / 工具、数据和部署版本，避免边复现边变化。
  2. 从最早可验证输入开始，逐段对照契约、决策、动作与验收事件；在首个偏离点停止跨层调参。
  3. 由 memory-owner 判断修复、降级、回滚或转人工，并记录未验证假设。
