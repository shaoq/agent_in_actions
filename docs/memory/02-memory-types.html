02 工作、情景、语义与程序记忆

学习目标

  • 按用途选择 working、episodic、semantic、procedural memory。
  • 为每类记忆定义来源、索引、冲突与失效策略。
  • 识别类型转换中的证据损失和过度概括。

核心概念

链路总览与现有讲解

记忆类型属于记忆系统：输入是带主体、来源、目的、同意和时间的经历候选，输出是类型化、版本化、可撤回的 MemoryRecord 或拒绝结果。学习时先沿现有图、表或主题章节理解链路，再用后续 stage 检查每个中间产物的不变量。

类型回答的问题结构 / 读取主要风险
工作记忆当前任务还要保留什么？短期 scratchpad、按 task_id上下文溢出、控制状态混入
情景记忆何时发生过什么？事件+时间线、按时间/实体片段失真、错误归因
语义记忆关于该用户有哪些稳定事实？声明+来源集合、按实体/属性过时、冲突、过度概括
程序记忆过去怎样做有效？步骤/策略+适用条件复制偶然成功、权限漂移

核心环节：从输入到可验证输出

1. 记忆类型

记忆类型决定记录的 owner、生命周期、读取方式和更新规则；类型是行为契约，不是便于展示的标签。

职责与 I/O
memory-owner 在“记忆类型”环节接收 带主体、来源、目的、同意和时间的经历候选，产出 类型化、版本化、可撤回的 MemoryRecord 或拒绝结果；输出必须能由下游按版本消费。
工作机制
先按真相来源与未来用途区分 RunState、Session、Memory、Knowledge，再在 Memory 内区分工作、情景、语义、偏好与程序记录。本环节把 带主体、来源、目的、同意和时间的经历候选 中与“记忆类型”相关的事实转成 类型化、版本化、可撤回的 MemoryRecord 或拒绝结果 的一部分。
必须成立的约束
分类必须在写入前确定并进入 schema；同一事实不能靠复制到多个类型解决读取问题。本章共同约束是：主体、目的、来源、置信、TTL 和同意必须可证明；违反时必须产生可区分的拒绝、未知或升级结果。
难点与取舍
粒度过粗会让检索与治理混乱，过细会提高迁移成本；类型集合由真实读写路径和失败样本稳定后再扩展。本层总体取舍是 个性化收益与隐私、召回率与错误长期化风险，参数必须引用版本化数据、SLO 或风险基线。
边界与注意事项
会话文本只是来源，外部制度属于 RAG；错误类型不能用更高相似度掩盖。记忆系统 的总边界是：保存 Agent 经历而非外部制度；不替代 RunState、Session 或 RAG；错误会继续污染 后续上下文注入、个性化 Decision 和治理审计。

2. 来源与置信

来源与置信阶段产生的是带来源和治理状态的候选事实，未经验证、同意与目的检查不得成为长期可读记忆。

职责与 I/O
memory-owner 在“来源与置信”环节接收 带主体、来源、目的、同意和时间的经历候选，产出 类型化、版本化、可撤回的 MemoryRecord 或拒绝结果；输出必须能由下游按版本消费。
工作机制
候选从显式用户陈述或已验收事件提取，附 source_event、subject、purpose、confidence、consent 与敏感类别后进入 accept / reject / confirm。本环节把 带主体、来源、目的、同意和时间的经历候选 中与“来源与置信”相关的事实转成 类型化、版本化、可撤回的 MemoryRecord 或拒绝结果 的一部分。
必须成立的约束
来源必须可回放，敏感字段按目的最小化；模型摘要不能提升证据等级，同意范围不能由相似语义推断。本章共同约束是：主体、目的、来源、置信、TTL 和同意必须可证明；违反时必须产生可区分的拒绝、未知或升级结果。
难点与取舍
自动写入提高覆盖却会长期化误解；确认频率依据错误成本、稳定性和用户摩擦分级。本层总体取舍是 个性化收益与隐私、召回率与错误长期化风险，参数必须引用版本化数据、SLO 或风险基线。
边界与注意事项
候选不是 RunState 的 checkpoint，也不是把 RAG 文档复制到用户名下。记忆系统 的总边界是：保存 Agent 经历而非外部制度；不替代 RunState、Session 或 RAG；错误会继续污染 后续上下文注入、个性化 Decision 和治理审计。

3. 读取策略

读取策略只有在当前主体、目的、时效和冲突优先级同时满足时才能影响本次 Decision；“被召回”不等于“应注入”。

职责与 I/O
memory-owner 在“读取策略”环节接收 带主体、来源、目的、同意和时间的经历候选，产出 类型化、版本化、可撤回的 MemoryRecord 或拒绝结果；输出必须能由下游按版本消费。
工作机制
读取先做 tenant / subject / purpose 过滤，再按类型、时间、置信和任务相关性排序；晋升与注入分别保留 PolicyDecision。本环节把 带主体、来源、目的、同意和时间的经历候选 中与“读取策略”相关的事实转成 类型化、版本化、可撤回的 MemoryRecord 或拒绝结果 的一部分。
必须成立的约束
当前显式输入优先于旧偏好，撤回或过期记录不可命中；返回内容必须带 memory_id、来源、版本和置信。本章共同约束是：主体、目的、来源、置信、TTL 和同意必须可证明；违反时必须产生可区分的拒绝、未知或升级结果。
难点与取舍
召回越宽个性化越强但污染面越大；top-k、TTL 和最低置信由时间切片与关闭 Memory 的消融结果校准。本层总体取舍是 个性化收益与隐私、召回率与错误长期化风险，参数必须引用版本化数据、SLO 或风险基线。
边界与注意事项
Memory 提供候选上下文，推理决定如何引用，编排决定何时读取；不能用向量相似度恢复控制状态。记忆系统 的总边界是：保存 Agent 经历而非外部制度；不替代 RunState、Session 或 RAG；错误会继续污染 后续上下文注入、个性化 Decision 和治理审计。

4. 记忆冲突

记忆冲突必须保留事实的来源与时间关系，使合并后的记录仍可解释冲突和回退，不能用新摘要抹掉历史。

职责与 I/O
memory-owner 在“记忆冲突”环节接收 带主体、来源、目的、同意和时间的经历候选，产出 类型化、版本化、可撤回的 MemoryRecord 或拒绝结果；输出必须能由下游按版本消费。
工作机制
稳定键与语义候选先发现重复，再按来源权威、时间和确认状态执行 link、merge、supersede 或 quarantine。本环节把 带主体、来源、目的、同意和时间的经历候选 中与“记忆冲突”相关的事实转成 类型化、版本化、可撤回的 MemoryRecord 或拒绝结果 的一部分。
必须成立的约束
冲突双方、裁决理由与派生关系必须保留；不可信指令、越权主体和异常写入模式进入隔离而非自动合并。本章共同约束是：主体、目的、来源、置信、TTL 和同意必须可证明；违反时必须产生可区分的拒绝、未知或升级结果。
难点与取舍
激进整合降低存储和噪声却可能丢失时序差异；阈值由人工复核准确率与下游错误切片决定。本层总体取舍是 个性化收益与隐私、召回率与错误长期化风险，参数必须引用版本化数据、SLO 或风险基线。
边界与注意事项
整合改变 MemoryRecord 关系，不修改外部权威知识，也不替治理层完成安全事件处置。记忆系统 的总边界是：保存 Agent 经历而非外部制度；不替代 RunState、Session 或 RAG；错误会继续污染 后续上下文注入、个性化 Decision 和治理审计。

5. 迁移错误类型

迁移错误类型必须让记录的变更、衰减、迁移、访问和删除都可证明到最终派生副本；不可见不等于已删除。

职责与 I/O
memory-owner 在“迁移错误类型”环节接收 带主体、来源、目的、同意和时间的经历候选，产出 类型化、版本化、可撤回的 MemoryRecord 或拒绝结果；输出必须能由下游按版本消费。
工作机制
版本图记录 supersede、migrate、expire、delete 与派生索引 / 缓存，主体请求和治理作业沿 lineage 传播并产生 receipt。本环节把 带主体、来源、目的、同意和时间的经历候选 中与“迁移错误类型”相关的事实转成 类型化、版本化、可撤回的 MemoryRecord 或拒绝结果 的一部分。
必须成立的约束
保留期、删除 SLA、合法留存、迁移版本和审计访问必须显式；删除后重建与备份恢复也不能让记录重新可读。本章共同约束是：主体、目的、来源、置信、TTL 和同意必须可证明；违反时必须产生可区分的拒绝、未知或升级结果。
难点与取舍
更长保留利于个性化与归因，却增加隐私和陈旧风险；期限由用途、法规、用户预期和实测价值共同决定。本层总体取舍是 个性化收益与隐私、召回率与错误长期化风险，参数必须引用版本化数据、SLO 或风险基线。
边界与注意事项
Memory owner 证明记录生命周期，平台 owner 证明物理存储传播，治理 owner 处理主体权利与审计。记忆系统 的总边界是：保存 Agent 经历而非外部制度；不替代 RunState、Session 或 RAG；错误会继续污染 后续上下文注入、个性化 Decision 和治理审计。

失败传播与 first divergence

  1. 上游输入：先确认 带主体、来源、目的、同意和时间的经历候选 的身份、作用域、时间与版本是否一致；输入错误不应在本层被“修正”为新事实。
  2. 首个偏离：按 记忆类型 → 来源与置信 → 读取策略 → 记忆冲突 → 迁移错误类型 比较预期与实际中间产物，第一个不满足概念不变量的 stage 才是本章候选根因。
  3. 下游影响：本层错误会改变 后续上下文注入、个性化 Decision 和治理审计；最终症状可能在更晚阶段出现，但不能从症状直接倒推 Prompt。
  4. 归属边界：memory-owner修复本层契约或机制；若 first divergence 位于相邻层，保留证据并交给相应 owner，不跨层调参。

工程要点

  1. 情景记录 immutable event；语义事实用 claim_id 和 supporting_episode_ids 建立可追溯派生。
  2. 程序记忆必须携带 precondition、tool/schema version、success metric 和 approval scope，环境变化后重新验证。
  3. 同一属性允许并存候选版本，通过 recency、source authority、confirmation 和 conflict policy 解析。
  4. 类型选择先看错误成本：高风险偏好可要求显式确认，低风险展示偏好可采用软读取。

常见坑

  • 把所有向量记录叫语义记忆：embedding 是索引手段，不是认知类型。
  • 把一次事件概括成稳定事实：“今天喝茶”不等于“永远偏好茶”。
  • 程序记忆即自动执行：复用策略仍要经过当前权限、参数和风险校验。
  • 把公共事实塞进个人 semantic memory：公共外部事实属于知识 / RAG。

理论追问

请先脱稿回答，再展开参考答案核对边界、证据和取舍。

  1. 工作、情景、语义和程序记忆如何区分？

    查看参考答案

    工作记忆是当前任务临时状态；情景记忆记录发生过的事件；语义记忆保存从经历中治理出的稳定事实或偏好；程序记忆保存可复用做法。分类依据是内容语义和生命周期，不只是不同数据库表。

  2. “用户喜欢中文回答”属于哪类记忆，为什么仍需作用域？

    查看参考答案

    它通常是用户级语义偏好，但必须绑定用户、场景、来源、置信和可撤回性。用户在某次会议要求英文不代表全局偏好改变；读取时还要考虑当前明确指令优先于历史记忆。

  3. 从多次成功轨迹提炼程序记忆时，为什么不能直接复制完整步骤？

    查看参考答案

    成功可能依赖旧工具、特定环境或偶然路径。应抽取稳定策略、适用条件、版本和失败边界，再通过评测验证；程序记忆是候选经验，不应绕过编排、权限和当前环境检查。

  4. 模型从对话中抽取的“事实”为什么不等于语义记忆真值？

    查看参考答案

    抽取可能误解否定、主体或时间，也可能受到注入。写入前需要来源引用、主体确认或外部验证、冲突检测和置信门槛；高风险事实应保留候选状态而非直接成为可信记忆。

  5. 同一内容可能属于多种记忆时如何选择？

    查看参考答案

    先问未来使用目的和生命周期：仅当前推理使用则留在工作状态；需要回顾事件则存情景；稳定偏好或事实经验证后存语义；可复用流程经评测后存程序。允许从情景整合为语义，但保留 lineage。

面试表达

为什么要区分四类记忆？

因为四类对象的生命周期与正确性标准不同。工作记忆追求任务连续性；情景记忆保留可追溯事件；语义记忆整合稳定声明；程序记忆复用经验证的做法。我会分别定义 schema、索引、冲突和过期策略，并要求所有晋升保留 provenance。

日常实践关键点

核心判断

本章的日常工作不是记住名词，而是把“记忆类型”变成可执行控制环：先识别触发信号，再用版本化证据定位首个偏离点，由 memory-owner 执行动作，最后用成功、拒绝、降级三类结果验收。推荐顺序是 选择记忆类型 → 记录来源与置信 → 制定读取策略 → 处理记忆冲突 → 迁移错误类型；任何参数都必须从真实 SLO、风险基线或失败样本推导。

深度实践卡

  • 1. 选择记忆类型
    触发信号
    当企业差旅 Agent 的“记忆类型”链路出现旧偏好覆盖新事实、跨主体命中、错误长期化或删除后仍可读取，或变更直接触及“选择记忆类型”的输入、策略或 owner 时，必须启动这项检查。
    诊断顺序
    诊断“选择记忆类型”时先冻结 subject_id、purpose、来源、同意状态、置信度、记录版本与读写 Trace。比较候选能力、风险、延迟和成本约束，确认选择依据来自声明能力而非名称或模型自报。
    执行动作
    落实“选择记忆类型”时，先裁剪不满足硬约束的候选，再对剩余候选打分，并把选择理由与替代项写入决策记录。由 memory-owner 记录执行前后的版本、决定与接管条件。
    验证方法
    验收“选择记忆类型”时，用时间切片回放、跨主体负测、删除传播测试和关闭 Memory 的消融实验复核；同时比较正确、拒绝和降级结果，阈值只从真实基线与风险容忍度推导。
    止损与取舍

    落实“处理记忆冲突”时，把写入当受治理的状态变更：先候选、再验证、后晋升；读取时保留来源并允许拒绝注入。由 memory-owner 记录执行前后的版本、决定与接管条件。

    隔离可疑记录、回退上一版本并传播删除；不要用新的摘要覆盖而留下旧记录可见；停止晋升或读取可疑记录，恢复上一策略版本，并保留受影响 memory_id 清单。对“处理记忆冲突”先保证可恢复和可审计，再优化自动化率。

    诊断顺序
    诊断“迁移错误类型”时先冻结 subject_id、purpose、来源、同意状态、置信度、记录版本与读写 Trace。固定读写双方版本并回放旧样本，检查默认值、枚举、字段语义和双写 / 双读窗口是否造成不可逆解释差异。

    落实“迁移错误类型”时，先定义兼容矩阵和迁移不变量，再按影子读取、双写核对、分批切换和旧版本退役的顺序发布。由 memory-owner 记录执行前后的版本、决定与接管条件。
    验证方法
    验收“迁移错误类型”时，用时间切片回放、跨主体负测、删除传播测试和关闭 Memory 的消融实验复核；同时比较正确、拒绝和降级结果，阈值只从真实基线与风险容忍度推导。
    止损与取舍
    迁移窗口保留旧读路径与版本标记；发现语义漂移时停止新写入并从原始记录重建；停止晋升或读取可疑记录，恢复上一策略版本，并保留受影响 memory_id 清单。对“迁移错误类型”先保证可恢复和可审计，再优化自动化率。
    面试记忆点
    围绕“迁移错误类型”准备一条“症状—首个偏离—动作—证据—止损”的真实或个人实验 Trace，并明确自己只承担的 memory-owner 职责。

排障顺序

  1. 冻结请求、策略、模型 / 工具、数据和部署版本，避免边复现边变化。
  2. 从最早可验证输入开始，逐段对照契约、决策、动作与验收事件；在首个偏离点停止跨层调参。
  3. 由 memory-owner 判断修复、降级、回滚或转人工，并记录未验证假设。
贯穿案例｜企业差旅 Agent：员工请求预订超预算航班时，只把制度条款当知识证据，把员工靠窗偏好当记忆，把预订动作交给工具层；本章只处理 记忆类型 的控制与证据。若发现旧偏好覆盖新事实、跨主体命中、错误长期化或删除后仍可读取，先保留 trace_id 与版本，再在本层首个偏离点修复，避免把问题推给 Prompt。
