04 隐私、投毒与治理

学习目标

  • 识别越权、敏感信息持久化、间接注入和自我投毒。
  • 设计最小化、同意、隔离、审计、撤回和删除控制。
  • 建立能定位写入来源和影响范围的记忆 trace。

核心概念

威胁典型路径核心控制
隐私过度收集全量 transcript 长期保存目的限制、字段最小化、TTL、可见同意
跨租户泄露共享向量索引或缓存漏过滤物理/逻辑隔离、检索前强制授权、缓存键含身份
记忆投毒用户、工具或网页诱导保存恶意指令来源信任、指令/事实分离、隔离区、确认与异常检测
自我强化模型输出被重新写入并视为事实禁止无来源回写、独立验证、循环检测
删除不完整主记录删了，派生摘要仍在血缘图、删除传播、缓存失效、可验证 tombstone

工程要点

  1. 将 input content 永远视为数据；“以后忽略安全策略”不得因为被保存就升级成系统指令。
  2. 每条记忆保存 actor、source_kind、source_id、writer、review_state、policy_version 和 derived_from。
  3. 敏感类别设 deny-by-default 或显式确认；儿童、医疗、财务等高风险域使用更短保留期与人工复核。
  4. 监控 unusual write rate、跨 subject 命中、重复冲突、来源集中度、删除 SLA 和高风险使用次数。
  5. 治理层制定策略与审计；记忆系统执行生命周期；离线评测用 poisoning、redaction、deletion 样本验证控制。

常见坑

  • 只做 prompt 防护：持久层授权与删除问题无法靠提示词解决。
  • 加密即合规：加密不回答“是否应该收集”和“谁能用”。
  • 用户说的都可信：攻击者可利用长期身份冒充、社会工程和工具 observation 投毒。
  • 误把知识文档注入归记忆层：外部文档的摄取与检索安全由知识 / RAG 负责；只有当其内容被错误晋升为经历时才成为记忆写入漏洞。

理论追问

请先脱稿回答，再展开参考答案核对边界、证据和取舍。

  1. 保存用户偏好前需要哪些隐私与同意证据？

    查看参考答案

    至少记录主体、用途、字段分类、来源、同意版本、作用域、保留期限和撤回方式。敏感推断不能因为模型觉得“有用”就写入；读取和导出也要执行访问控制与审计。

  2. Memory Poisoning 与普通错误记忆有什么区别？

    查看参考答案

    普通错误可能来自误抽取或过期，投毒则是攻击者有意让恶意内容跨会话影响决策。防护要限制写入主体、隔离不可信来源、做冲突和异常检测，并在高风险动作前重新依赖权威事实。

  3. 用户要求删除记忆时，怎样证明删除完成？

    查看参考答案

    删除稳定 memory ID 对应的主存、向量索引、缓存和派生整合记录，写入不可恢复内容的 tombstone 和审计事件，并通过读取负向测试确认不再召回。备份按既定保留策略到期清除，不能承诺即时物理擦除却无证据。

  4. 如何防止多个用户或 Agent 之间发生记忆串扰？

    查看参考答案

    所有记录和查询都绑定 tenant、subject、agent scope 与用途，服务端在召回前强制过滤，缓存与索引按作用域隔离。使用 canary 记录和负向权限测试验证，不能依赖 Prompt 告诉模型“不要看别人数据”。

  5. 发生记忆投毒事故时，Trace 需要保留哪些证据？

    查看参考答案

    需要写入事件、actor、来源内容摘要、策略与模型版本、整合 lineage、后续读取记录、受影响 run 和动作。先冻结相关写入与高风险读取，再定位传播范围、撤销记录、构建回归样本并验证修复。

面试表达

如何防记忆投毒？

我把写入视为高风险副作用：不可信输入先进入候选隔离区，保留来源与 actor；通过敏感性、授权、事实/指令分类、置信度和异常检测后才发布。读取仍按 subject 和 purpose 强制过滤，高风险使用二次确认。模型输出不能无来源回写，删除依赖血缘传播并用安全回归样本验证。
