04 隐私、投毒与治理

学习目标

  • 识别越权、敏感信息持久化、间接注入和自我投毒。
  • 设计最小化、同意、隔离、审计、撤回和删除控制。
  • 建立能定位写入来源和影响范围的记忆 trace。

核心概念

链路总览与现有讲解

隐私、投毒与治理属于记忆系统：输入是带主体、来源、目的、同意和时间的经历候选，输出是类型化、版本化、可撤回的 MemoryRecord 或拒绝结果。学习时先沿现有图、表或主题章节理解链路，再用后续 stage 检查每个中间产物的不变量。

威胁典型路径核心控制
隐私过度收集全量 transcript 长期保存目的限制、字段最小化、TTL、可见同意
跨租户泄露共享向量索引或缓存漏过滤物理/逻辑隔离、检索前强制授权、缓存键含身份
记忆投毒用户、工具或网页诱导保存恶意指令来源信任、指令/事实分离、隔离区、确认与异常检测
自我强化模型输出被重新写入并视为事实禁止无来源回写、独立验证、循环检测
删除不完整主记录删了，派生摘要仍在血缘图、删除传播、缓存失效、可验证 tombstone

核心环节：从输入到可验证输出

1. 同意与目的

同意与目的阶段产生的是带来源和治理状态的候选事实，未经验证、同意与目的检查不得成为长期可读记忆。

职责与 I/O
memory-owner 在“同意与目的”环节接收 带主体、来源、目的、同意和时间的经历候选，产出 类型化、版本化、可撤回的 MemoryRecord 或拒绝结果；输出必须能由下游按版本消费。
工作机制
候选从显式用户陈述或已验收事件提取，附 source_event、subject、purpose、confidence、consent 与敏感类别后进入 accept / reject / confirm。本环节把 带主体、来源、目的、同意和时间的经历候选 中与“同意与目的”相关的事实转成 类型化、版本化、可撤回的 MemoryRecord 或拒绝结果 的一部分。
必须成立的约束
来源必须可回放，敏感字段按目的最小化；模型摘要不能提升证据等级，同意范围不能由相似语义推断。本章共同约束是：主体、目的、来源、置信、TTL 和同意必须可证明；违反时必须产生可区分的拒绝、未知或升级结果。
难点与取舍
自动写入提高覆盖却会长期化误解；确认频率依据错误成本、稳定性和用户摩擦分级。本层总体取舍是 个性化收益与隐私、召回率与错误长期化风险，参数必须引用版本化数据、SLO 或风险基线。
边界与注意事项
候选不是 RunState 的 checkpoint，也不是把 RAG 文档复制到用户名下。记忆系统 的总边界是：保存 Agent 经历而非外部制度；不替代 RunState、Session 或 RAG；错误会继续污染 后续上下文注入、个性化 Decision 和治理审计。

3. 记忆投毒

记忆投毒必须保留事实的来源与时间关系，使合并后的记录仍可解释冲突和回退，不能用新摘要抹掉历史。

职责与 I/O
memory-owner 在“记忆投毒”环节接收 带主体、来源、目的、同意和时间的经历候选，产出 类型化、版本化、可撤回的 MemoryRecord 或拒绝结果；输出必须能由下游按版本消费。
工作机制
稳定键与语义候选先发现重复，再按来源权威、时间和确认状态执行 link、merge、supersede 或 quarantine。本环节把 带主体、来源、目的、同意和时间的经历候选 中与“记忆投毒”相关的事实转成 类型化、版本化、可撤回的 MemoryRecord 或拒绝结果 的一部分。
必须成立的约束
冲突双方、裁决理由与派生关系必须保留；不可信指令、越权主体和异常写入模式进入隔离而非自动合并。本章共同约束是：主体、目的、来源、置信、TTL 和同意必须可证明；违反时必须产生可区分的拒绝、未知或升级结果。
难点与取舍
激进整合降低存储和噪声却可能丢失时序差异；阈值由人工复核准确率与下游错误切片决定。本层总体取舍是 个性化收益与隐私、召回率与错误长期化风险，参数必须引用版本化数据、SLO 或风险基线。
边界与注意事项
整合改变 MemoryRecord 关系，不修改外部权威知识，也不替治理层完成安全事件处置。记忆系统 的总边界是：保存 Agent 经历而非外部制度；不替代 RunState、Session 或 RAG；错误会继续污染 后续上下文注入、个性化 Decision 和治理审计。

4. 主体访问与纠正

主体访问与纠正必须让记录的变更、衰减、迁移、访问和删除都可证明到最终派生副本；不可见不等于已删除。

职责与 I/O
memory-owner 在“主体访问与纠正”环节接收 带主体、来源、目的、同意和时间的经历候选，产出 类型化、版本化、可撤回的 MemoryRecord 或拒绝结果；输出必须能由下游按版本消费。
工作机制
版本图记录 supersede、migrate、expire、delete 与派生索引 / 缓存，主体请求和治理作业沿 lineage 传播并产生 receipt。本环节把 带主体、来源、目的、同意和时间的经历候选 中与“主体访问与纠正”相关的事实转成 类型化、版本化、可撤回的 MemoryRecord 或拒绝结果 的一部分。

保留期、删除 SLA、合法留存、迁移版本和审计访问必须显式；删除后重建与备份恢复也不能让记录重新可读。本章共同约束是：主体、目的、来源、置信、TTL 和同意必须可证明；违反时必须产生可区分的拒绝、未知或升级结果。

边界与注意事项
Memory owner 证明记录生命周期，平台 owner 证明物理存储传播，治理 owner 处理主体权利与审计。记忆系统 的总边界是：保存 Agent 经历而非外部制度；不替代 RunState、Session 或 RAG；错误会继续污染 后续上下文注入、个性化 Decision 和治理审计。

失败传播与 first divergence

  1. 上游输入：先确认 带主体、来源、目的、同意和时间的经历候选 的身份、作用域、时间与版本是否一致；输入错误不应在本层被“修正”为新事实。
  2. 首个偏离：按 同意与目的 → 敏感记忆 → 记忆投毒 → 主体访问与纠正 → 审计记忆治理 比较预期与实际中间产物，第一个不满足概念不变量的 stage 才是本章候选根因。
  3. 下游影响：本层错误会改变 后续上下文注入、个性化 Decision 和治理审计；最终症状可能在更晚阶段出现，但不能从症状直接倒推 Prompt。
  4. 归属边界：memory-owner修复本层契约或机制；若 first divergence 位于相邻层，保留证据并交给相应 owner，不跨层调参。

工程要点

  1. 将 input content 永远视为数据；“以后忽略安全策略”不得因为被保存就升级成系统指令。
  2. 每条记忆保存 actor、source_kind、source_id、writer、review_state、policy_version 和 derived_from。
  3. 敏感类别设 deny-by-default 或显式确认；儿童、医疗、财务等高风险域使用更短保留期与人工复核。
  4. 监控 unusual write rate、跨 subject 命中、重复冲突、来源集中度、删除 SLA 和高风险使用次数。
  5. 治理层制定策略与审计；记忆系统执行生命周期；离线评测用 poisoning、redaction、deletion 样本验证控制。

常见坑

  • 只做 prompt 防护：持久层授权与删除问题无法靠提示词解决。
  • 加密即合规：加密不回答“是否应该收集”和“谁能用”。
  • 用户说的都可信：攻击者可利用长期身份冒充、社会工程和工具 observation 投毒。
  • 误把知识文档注入归记忆层：外部文档的摄取与检索安全由知识 / RAG 负责；只有当其内容被错误晋升为经历时才成为记忆写入漏洞。

理论追问

请先脱稿回答，再展开参考答案核对边界、证据和取舍。

  1. 保存用户偏好前需要哪些隐私与同意证据？

    查看参考答案

    至少记录主体、用途、字段分类、来源、同意版本、作用域、保留期限和撤回方式。敏感推断不能因为模型觉得“有用”就写入；读取和导出也要执行访问控制与审计。

  2. Memory Poisoning 与普通错误记忆有什么区别？

    查看参考答案

    普通错误可能来自误抽取或过期，投毒则是攻击者有意让恶意内容跨会话影响决策。防护要限制写入主体、隔离不可信来源、做冲突和异常检测，并在高风险动作前重新依赖权威事实。

  3. 用户要求删除记忆时，怎样证明删除完成？

    查看参考答案

    删除稳定 memory ID 对应的主存、向量索引、缓存和派生整合记录，写入不可恢复内容的 tombstone 和审计事件，并通过读取负向测试确认不再召回。备份按既定保留策略到期清除，不能承诺即时物理擦除却无证据。

  4. 如何防止多个用户或 Agent 之间发生记忆串扰？

    查看参考答案

    所有记录和查询都绑定 tenant、subject、agent scope 与用途，服务端在召回前强制过滤，缓存与索引按作用域隔离。使用 canary 记录和负向权限测试验证，不能依赖 Prompt 告诉模型“不要看别人数据”。

  5. 发生记忆投毒事故时，Trace 需要保留哪些证据？

    查看参考答案

    需要写入事件、actor、来源内容摘要、策略与模型版本、整合 lineage、后续读取记录、受影响 run 和动作。先冻结相关写入与高风险读取，再定位传播范围、撤销记录、构建回归样本并验证修复。

面试表达

如何防记忆投毒？

我把写入视为高风险副作用：不可信输入先进入候选隔离区，保留来源与 actor；通过敏感性、授权、事实/指令分类、置信度和异常检测后才发布。读取仍按 subject 和 purpose 强制过滤，高风险使用二次确认。模型输出不能无来源回写，删除依赖血缘传播并用安全回归样本验证。

日常实践关键点

核心判断

本章的日常工作不是记住名词，而是把“隐私、投毒与治理”变成可执行控制环：先识别触发信号，再用版本化证据定位首个偏离点，由 memory-owner 执行动作，最后用成功、拒绝、降级三类结果验收。推荐顺序是 绑定同意与目的 → 最小化敏感记忆 → 检测记忆投毒 → 支持主体访问与纠正 → 审计记忆治理；任何参数都必须从真实 SLO、风险基线或失败样本推导。

深度实践卡

  • 1. 绑定同意与目的
    触发信号
    当企业差旅 Agent 的“隐私、投毒与治理”链路出现旧偏好覆盖新事实、跨主体命中、错误长期化或删除后仍可读取，或变更直接触及“绑定同意与目的”的输入、策略或 owner 时，必须启动这项检查。
    诊断顺序

排障顺序

  1. 冻结请求、策略、模型 / 工具、数据和部署版本，避免边复现边变化。
  2. 从最早可验证输入开始，逐段对照契约、决策、动作与验收事件；在首个偏离点停止跨层调参。
  3. 由 memory-owner 判断修复、降级、回滚或转人工，并记录未验证假设。
