03 身份、权限与审计

学习目标

  • 区分认证（authentication）、授权（authorization）、委托（delegation）与审批（approval）。
  • 以 subject、action、resource、context 形成最小权限判定。
  • 让审批绑定不可变动作摘要，并在恢复时重新授权。

核心概念

链路总览与现有讲解

身份、授权与审计属于治理与可观测层：输入是identity context、资源动作、策略版本、风险与关联事件，输出是类型化 PolicyOutcome、审计事件、脱敏遥测或事件升级。学习时先沿现有图、表或主题章节理解链路，再用后续 stage 检查每个中间产物的不变量。

字段问题示例

action做什么能力与副作用？invoice.read、refund.create
resource对哪个对象与范围？tenant/order/123
context在何风险与条件下？渠道、金额、设备、目的、时间
obligations允许后必须遵守什么？mask、limit、approval、retention

核心环节：从输入到可验证输出

1. 身份链

身份链必须绑定当前 actor、资源、动作、目的和版本，并在真正使用能力或数据的时刻仍然有效。

职责与 I/O
governance-owner 在“身份链”环节接收 identity context、资源动作、策略版本、风险与关联事件，产出 类型化 PolicyOutcome、审计事件、脱敏遥测或事件升级；输出必须能由下游按版本消费。
工作机制
身份链先把渠道主体映射为内部 actor，再由策略决策与短期能力令牌约束数据、工具、文件和网络使用。本环节把 identity context、资源动作、策略版本、风险与关联事件 中与“身份链”相关的事实转成 类型化 PolicyOutcome、审计事件、脱敏遥测或事件升级 的一部分。
必须成立的约束
默认最小权限与 deny-by-default；撤回、角色变更、资源版本变化必须使缓存授权和派生访问失效。本章共同约束是：在线控制必须 fail-safe，证据必须可关联且敏感值最小化；违反时必须产生可区分的拒绝、未知或升级结果。
难点与取舍
频繁再授权更安全但增加延迟；缓存更快却可能陈旧，TTL 应由权限变化频率和风险窗口推导。本层总体取舍是 控制强度与用户摩擦、可观测深度与隐私 / 基数成本，参数必须引用版本化数据、SLO 或风险基线。
边界与注意事项
身份认证不等于授权，脱敏不等于匿名化，网络隔离也不能替代业务策略判断。治理与可观测层 的总边界是：在线治理不替代离线评测，执行隔离仍由基础设施沙箱负责；错误会继续污染 编排门禁、工具授权、事件响应和离线评测输入。

2. Policy Decision

Policy Decision必须在动作发生前或事件使用点给出可执行、可审计的类型化结果；unknown 不能被静默当作 allow。

职责与 I/O
governance-owner 在“ Policy Decision”环节接收 identity context、资源动作、策略版本、风险与关联事件，产出 类型化 PolicyOutcome、审计事件、脱敏遥测或事件升级；输出必须能由下游按版本消费。
工作机制
控制点把 identity、resource、action、purpose、风险和 policy_version 求值为 allow、deny、require_approval 或 escalate。本环节把 identity context、资源动作、策略版本、风险与关联事件 中与“ Policy Decision”相关的事实转成 类型化 PolicyOutcome、审计事件、脱敏遥测或事件升级 的一部分。
必须成立的约束
策略依赖不可用、字段缺失或证据过期时采用 fail-safe；审批只绑定所见对象、版本、范围和有效期。本章共同约束是：在线控制必须 fail-safe，证据必须可关联且敏感值最小化；违反时必须产生可区分的拒绝、未知或升级结果。
难点与取舍
更严格会增加用户摩擦和人工负载，更宽松会扩大风险；阈值依据风险等级、误拦成本和事件数据校准。本层总体取舍是 控制强度与用户摩擦、可观测深度与隐私 / 基数成本，参数必须引用版本化数据、SLO 或风险基线。
边界与注意事项
治理决定是否允许和如何留证，不执行工具动作；HITL 是控制结果，不是让人工修补任意不完整上下文。治理与可观测层 的总边界是：在线治理不替代离线评测，执行隔离仍由基础设施沙箱负责；错误会继续污染 编排门禁、工具授权、事件响应和离线评测输入。

5. 审计回放

审计回放必须能把结论追溯到同一请求的输入、版本、决定、动作和验收事件，而不是依赖不可检索的日志文本。

职责与 I/O
governance-owner 在“审计回放”环节接收 identity context、资源动作、策略版本、风险与关联事件，产出 类型化 PolicyOutcome、审计事件、脱敏遥测或事件升级；输出必须能由下游按版本消费。
工作机制
correlation_id / trace_id 串联 span 与结构化事件，引用保留原始证据位置、policy / model / tool 版本和 owner。本环节把 identity context、资源动作、策略版本、风险与关联事件 中与“审计回放”相关的事实转成 类型化 PolicyOutcome、审计事件、脱敏遥测或事件升级 的一部分。
必须成立的约束
时间、因果顺序、采样与脱敏策略必须明确；高基数字段不能无界进入 Metric，敏感原文不能直接写 Log。本章共同约束是：在线控制必须 fail-safe，证据必须可关联且敏感值最小化；违反时必须产生可区分的拒绝、未知或升级结果。
难点与取舍
全量遥测利于归因却增加成本与隐私面；采样按风险与失败类型分层，关键审计事件不得随机丢弃。本层总体取舍是 控制强度与用户摩擦、可观测深度与隐私 / 基数成本，参数必须引用版本化数据、SLO 或风险基线。
边界与注意事项
Trace 解释系统事件与可见证据，不保存模型私有推理；Metric 告警不能替代逐请求审计。治理与可观测层 的总边界是：在线治理不替代离线评测，执行隔离仍由基础设施沙箱负责；错误会继续污染 编排门禁、工具授权、事件响应和离线评测输入。

难点与取舍

冲突目标参数依据决策边界
控制强度与用户摩擦、可观测深度与隐私 / 基数成本使用版本化回放、失败切片、SLO 分位数和风险等级，不把教学值当生产默认。governance-owner只调整本层机制；跨层变化需要对应 owner 与独立验收。
自动化收益 vs. 失败放大比较开启 / 关闭“身份链”与“审计回放”的消融结果、人工负载和恢复时间。证据不足返回 pending / unknown / escalate，不能靠更长 Prompt 隐藏边界。

失败传播与 first divergence

  1. 上游输入：先确认 identity context、资源动作、策略版本、风险与关联事件 的身份、作用域、时间与版本是否一致；输入错误不应在本层被“修正”为新事实。
  2. 首个偏离：按 身份链 → Policy Decision → 最小权限 → 在使用点再授权 → 审计回放 比较预期与实际中间产物，第一个不满足概念不变量的 stage 才是本章候选根因。
  3. 下游影响：本层错误会改变 编排门禁、工具授权、事件响应和离线评测输入；最终症状可能在更晚阶段出现，但不能从症状直接倒推 Prompt。
  4. 归属边界：governance-owner修复本层契约或机制；若 first divergence 位于相邻层，保留证据并交给相应 owner，不跨层调参。

工程要点

  1. 不要继承编排服务的超集权限；下游凭据使用用户委托或单次短期 capability，限定 audience、scope 和 expiry。
  2. 审批证据绑定 actor、action_digest、resource、amount、policy_version、expires_at；参数、资源、身份或版本变化立即失效。
  3. 恢复前重新认证、重新授权并检查 TOCTOU；工具运行时只接受已验证 invocation，不信任模型自报权限。
  4. 审计事件 append-only，至少包含 event_id、time、actor、delegator、decision_id、policy_version、resource reference、outcome 与 trace_id。
  5. 日志服务故障不应使高风险动作静默失去审计；设计本地耐久缓冲、拒绝或降级策略。

常见坑

  • 认证后全放行：身份成立不代表拥有每个资源动作权限。
  • Agent 当最终主体：会丢失用户、服务和审批者的委托链。
  • 审批只存 yes/no：无法证明批准的是哪个参数与资源。
  • Audit 等于 debug log：调试日志可采样和修改，审计必须完整、受限、可验证。

理论追问

请先脱稿回答，再展开参考答案核对边界、证据和取舍。

  1. 用户、Agent 和服务账号在审计中分别是什么角色？

    查看参考答案

    用户通常是 subject 或委托者，Agent 是代表用户规划和发起动作的 actor，服务账号是执行调用的技术身份。审计必须同时记录 delegation chain，不能只看到共享服务账号而失去最终责任主体。

  2. 一次授权决策最少需要哪些输入？

    查看参考答案

    需要 subject、actor、delegation、tenant、resource、action、关键参数、当前对象版本、环境条件、策略版本和审批证据。仅凭工具名称或“用户已登录”无法判断具体资源和动作是否允许。

  3. 权限变更后，进行中的 Run 为什么要重新授权？

    查看参考答案

    旧 allow 决策绑定当时的策略、角色、资源状态和时间。暂停恢复、动作参数变化或凭据撤销后继续沿用会造成 TOCTOU 越权；高风险动作执行前必须重新求值并记录新 decision。

  4. 审计日志与普通应用日志有什么不同？

    查看参考答案

    审计日志面向可追责性，要求主体、动作、资源、决策、策略版本、时间和 receipt 完整，具备防篡改、受控访问与保留策略。普通日志可用于调试且允许采样，但不能替代授权证据。

  5. 多个 Agent 共用一个高权限服务账号有什么问题，如何改进？

    查看参考答案

    资源侧只能看到共享身份，无法最小授权、撤销单个委托或归因越权。应使用短期下游凭据或 capability token，携带用户与 Agent 委托范围，并在网关和资源侧校验、审计。

面试表达

如何避免 Agent 成为 confused deputy？

我不让 Agent 继承服务的宽权限。每个动作都用原始主体、委托链、动作、资源和上下文重新授权；需要人工审批时绑定不可变动作摘要和有效期，恢复时处理 TOCTOU 并重新验证。工具只接收受限短期凭据，审计把策略决定、审批和实际执行通过 decision_id 与 trace_id 串起来。

日常实践关键点

核心判断

本章的日常工作不是记住名词，而是把“身份、授权与审计”变成可执行控制环：先识别触发信号，再用版本化证据定位首个偏离点，由 governance-owner 执行动作，最后用成功、拒绝、降级三类结果验收。推荐顺序是 验证身份链 → 记录 Policy Decision → 执行最小权限 → 在使用点再授权 → 支持审计回放；任何参数都必须从真实 SLO、风险基线或失败样本推导。

深度实践卡

  • 1. 验证身份链
    触发信号
    当企业差旅 Agent 的“身份、授权与审计”链路出现策略结果缺失、日志不可关联、授权在使用点失效、敏感信息泄漏或事件无人接管，或变更直接触及“验证身份链”的输入、策略或 owner 时，必须启动这项检查。
    诊断顺序
    诊断“验证身份链”时先冻结 tenant、actor、policy_version、decision、trace_id、风险等级与审计事件。重点确认决策主体、资源、动作、目的和策略版本是否在“使用点”仍一致，不能只看入口鉴权。
    执行动作
    落实“验证身份链”时，把允许、拒绝和需审批做成类型化策略结果，默认拒绝缺字段或版本不明的请求。由 governance-owner 记录执行前后的版本、决定与接管条件。
    验证方法
    验收“验证身份链”时，运行拒绝路径负测、审计回放、脱敏扫描和故障演练，确认 fail-safe 与 owner 接管生效；同时比较正确、拒绝和降级结果，阈值只从真实基线与风险容忍度推导。
    止损与取舍
    一旦主体或目的无法证明，立即撤销能力令牌并转人工确认，不用事后过滤替代前置授权；切换到更严格策略或只读降级，冻结高风险动作，并创建带时间线的事件记录。对“验证身份链”先保证可恢复和可审计，再优化自动化率。
    面试记忆点
    围绕“验证身份链”准备一条“症状—首个偏离—动作—证据—止损”的真实或个人实验 Trace，并明确自己只承担的 governance-owner 职责。

排障顺序

  1. 冻结请求、策略、模型 / 工具、数据和部署版本，避免边复现边变化。
  2. 从最早可验证输入开始，逐段对照契约、决策、动作与验收事件；在首个偏离点停止跨层调参。
  3. 由 governance-owner 判断修复、降级、回滚或转人工，并记录未验证假设。
贯穿案例｜企业差旅 Agent：员工请求预订超预算航班时，只把制度条款当知识证据，把员工靠窗偏好当记忆，把预订动作交给工具层；本章只处理 身份、授权与审计 的控制与证据。若发现策略结果缺失、日志不可关联、授权在使用点失效、敏感信息泄漏或事件无人接管，先保留 trace_id 与版本，再在本层首个偏离点修复，避免把问题推给 Prompt。

应留证据

最小证据集应能回答谁在何时依据哪个版本、对什么输入做了什么决定，以及结果如何被验证。敏感值使用教学占位或受控引用，不把日志全文复制进面试材料。

代表性工程产物｜artifact-gov-03-policy-audit
case: enterprise-travel-agent
component: governance-observability
unit: 03
owner: governance-owner
input_version: teaching-v1
policy_or_contract_version: teaching-v3
decision: allow | deny | degrade | escalate
first_divergence: stage-name
evidence_refs: [trace-demo, eval-slice-demo]
validation: passed | failed | unknown

以上字段是学习用数据形状，不是生产默认值。实际产物还应记录时间、租户 / 主体作用域、变更来源和保留策略，并能从 evidence_refs 回放本章结论。

实践型面试要点

表达时必须标记经历属于生产实战、个人实验或方案设计，只主张自己实际完成的职责，并用 Trace、评测或变更记录举证。企业差旅案例、示例版本和指标均为教学占位；没有验证线上规模、事故协作或长期效果时要主动说明。

实践面试 1

关联实践：验证身份链、记录 Policy Decision

常见问法

在生产或可复现实验中，你如何把“验证身份链”和“记录 Policy Decision”落到企业差旅 Agent 的 身份、授权与审计 链路？

60–90 秒回答骨架

先交代 身份、授权与审计 的业务风险和 governance-owner 边界；再说触发信号与首个偏离点；随后说明契约、动作和参数依据；最后给出验证、止损和复盘结果。回答要同时覆盖正常路径与拒绝 / 降级路径。

可信证据

围绕“验证身份链 / 记录 Policy Decision”展示脱敏后的 trace_id、tenant、actor、policy_version、decision、trace_id、风险等级与审计事件、前后对照和失败切片；个人实验还应给出固定输入、复现步骤和观察结果，不虚构线上规模。

高频追问

为什么问题属于 治理与可观测层 而不是相邻层？参数如何从 SLO 或风险基线推导？结果未知时怎样避免扩大副作用？governance-owner 不在线时，谁依据什么证据接管？

边界与取舍

“验证身份链 / 记录 Policy Decision”会增加契约、存储或人工成本，只有错误成本与复现收益覆盖这些成本时才扩大自动化。未处理过该类生产事故时，应明确这是方案设计或个人实验结论。
