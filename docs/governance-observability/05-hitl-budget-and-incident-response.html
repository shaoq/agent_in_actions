05 HITL、预算与事件响应

学习目标

  • 定义高风险、不确定、异常和预算条件下的 HITL 触发与超时。
  • 用分层预算控制 token、模型、工具、时间、重试和并发。
  • 设计检测、遏制、取证、修复、恢复和复盘的事件响应闭环。

核心概念

机制触发安全动作恢复条件
HITL高影响动作、低置信度、策略要求checkpoint、最小审阅包、超时审批绑定仍有效且重新授权
预算护栏token、金额、延迟、重试、并发阈值小模型、缩短上下文、限流、终止新预算或明确降级契约
事件响应越权、外泄、异常成本、审计缺口撤权、隔离、停用工具、保全证据根因修复、回归通过、风险签字

工程要点

  1. 审批包只包含决定所需的风险摘要、资源、参数 diff、证据引用和截止时间；避免让人工重新阅读整个 transcript。
  2. 暂停点持久化 checkpoint、action_digest、policy_version 和等待原因；重复通知、批准和恢复都必须幂等。
  3. 预算按 request、user/tenant、workflow 与全局四层累计，预留完成/补偿成本；达到软阈值先降级，硬阈值终止。
  4. 告警必须映射 owner、severity、runbook 与 SLO；用 kill switch 快速禁用模型、工具、策略或租户范围。
  5. 事件取证保留不可变审计和最小必要 trace；修复后由评测层运行离线安全回归，再由治理层受控恢复在线流量。

常见坑

  • 所有操作都 HITL：人工疲劳造成形式化批准；应按影响和不确定性分级。
  • 预算只看 token：忽略工具金额、重试风暴、墙钟时间和人工成本。
  • 审批后直接恢复：身份、资源、参数和策略可能已变化，必须重新授权。
  • 先清日志再处理：会破坏证据；遏制与取证需按 runbook 协同。

面试表达

如何设计 Agent 的 HITL 和事件响应？

我把 HITL 建模为可持久化暂停协议：风险触发后保存 checkpoint 和动作摘要，审批绑定主体、范围、版本与有效期，恢复时重新授权并保证幂等。预算覆盖 token、延迟、重试、工具金额和并发，软阈值降级、硬阈值终止。若检测到安全事件，先按 kill switch 遏制并保全审计，再修复、离线回归、风险签字后逐步恢复。
