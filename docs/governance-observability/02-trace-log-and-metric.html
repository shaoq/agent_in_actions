02 Trace、Log 与 Metric

学习目标

  • 区分 Trace、Log、Metric 与 Audit 的问题域和粒度。
  • 设计跨模型、检索、工具、策略和人工节点的关联字段。
  • 从可观测事实形成可证伪假设，并定位正确修复层。

核心概念

链路总览与现有讲解

Trace、Log 与 Metric属于治理与可观测层：输入是identity context、资源动作、策略版本、风险与关联事件，输出是类型化 PolicyOutcome、审计事件、脱敏遥测或事件升级。学习时先沿现有图、表或主题章节理解链路，再用后续 stage 检查每个中间产物的不变量。

信号回答Agent 例子反模式
Trace一次请求经过什么因果路径？run → retrieval → model → policy → toolspan 没有父子关系或状态版本
Log某时刻发生了什么离散事件？policy_denied、tool_timeout只写自由文本和完整 prompt
Metric一段时间影响多大、是否越过 SLO？拒绝率、p95 延迟、token 成本高基数字段作为标签
Audit谁基于何策略批准并执行了什么？审批、授权、数据访问、策略变更可覆盖、缺少 actor 与 decision_id

核心环节：从输入到可验证输出

1. 贯穿关联上下文

贯穿关联上下文必须能把结论追溯到同一请求的输入、版本、决定、动作和验收事件，而不是依赖不可检索的日志文本。

职责与 I/O
governance-owner 在“贯穿关联上下文”环节接收 identity context、资源动作、策略版本、风险与关联事件，产出 类型化 PolicyOutcome、审计事件、脱敏遥测或事件升级；输出必须能由下游按版本消费。
工作机制
correlation_id / trace_id 串联 span 与结构化事件，引用保留原始证据位置、policy / model / tool 版本和 owner。本环节把 identity context、资源动作、策略版本、风险与关联事件 中与“贯穿关联上下文”相关的事实转成 类型化 PolicyOutcome、审计事件、脱敏遥测或事件升级 的一部分。
必须成立的约束
时间、因果顺序、采样与脱敏策略必须明确；高基数字段不能无界进入 Metric，敏感原文不能直接写 Log。本章共同约束是：在线控制必须 fail-safe，证据必须可关联且敏感值最小化；违反时必须产生可区分的拒绝、未知或升级结果。
难点与取舍
全量遥测利于归因却增加成本与隐私面；采样按风险与失败类型分层，关键审计事件不得随机丢弃。本层总体取舍是 控制强度与用户摩擦、可观测深度与隐私 / 基数成本，参数必须引用版本化数据、SLO 或风险基线。
边界与注意事项
Trace 解释系统事件与可见证据，不保存模型私有推理；Metric 告警不能替代逐请求审计。治理与可观测层 的总边界是：在线治理不替代离线评测，执行隔离仍由基础设施沙箱负责；错误会继续污染 编排门禁、工具授权、事件响应和离线评测输入。

3. 日志脱敏

日志脱敏必须绑定当前 actor、资源、动作、目的和版本，并在真正使用能力或数据的时刻仍然有效。

职责与 I/O
governance-owner 在“日志脱敏”环节接收 identity context、资源动作、策略版本、风险与关联事件，产出 类型化 PolicyOutcome、审计事件、脱敏遥测或事件升级；输出必须能由下游按版本消费。
工作机制
身份链先把渠道主体映射为内部 actor，再由策略决策与短期能力令牌约束数据、工具、文件和网络使用。本环节把 identity context、资源动作、策略版本、风险与关联事件 中与“日志脱敏”相关的事实转成 类型化 PolicyOutcome、审计事件、脱敏遥测或事件升级 的一部分。
必须成立的约束
默认最小权限与 deny-by-default；撤回、角色变更、资源版本变化必须使缓存授权和派生访问失效。本章共同约束是：在线控制必须 fail-safe，证据必须可关联且敏感值最小化；违反时必须产生可区分的拒绝、未知或升级结果。
难点与取舍
频繁再授权更安全但增加延迟；缓存更快却可能陈旧，TTL 应由权限变化频率和风险窗口推导。本层总体取舍是 控制强度与用户摩擦、可观测深度与隐私 / 基数成本，参数必须引用版本化数据、SLO 或风险基线。
边界与注意事项
身份认证不等于授权，脱敏不等于匿名化，网络隔离也不能替代业务策略判断。治理与可观测层 的总边界是：在线治理不替代离线评测，执行隔离仍由基础设施沙箱负责；错误会继续污染 编排门禁、工具授权、事件响应和离线评测输入。

4. 指标基数

指标基数必须以版本化分布、业务切片和风险容忍度为依据；单个平均值或脱离基线的固定阈值不能代表系统边界。

职责与 I/O
governance-owner 在“指标基数”环节接收 identity context、资源动作、策略版本、风险与关联事件，产出 类型化 PolicyOutcome、审计事件、脱敏遥测或事件升级；输出必须能由下游按版本消费。
工作机制
端到端 SLO 被分配到阶段 deadline、token / 调用预算、并发和容量水位，再由 admission、限流或降级保护。本环节把 identity context、资源动作、策略版本、风险与关联事件 中与“指标基数”相关的事实转成 类型化 PolicyOutcome、审计事件、脱敏遥测或事件升级 的一部分。
必须成立的约束
同时看 p50 / p95 / p99、单位任务成本、错误与拒绝率；预算跨层累计且不能靠放宽安全门禁抵消。本章共同约束是：在线控制必须 fail-safe，证据必须可关联且敏感值最小化；违反时必须产生可区分的拒绝、未知或升级结果。
难点与取舍
质量、尾延迟、成本与吞吐互相制约；参数由压测、流量预测、风险切片和业务优先级联合校准。本层总体取舍是 控制强度与用户摩擦、可观测深度与隐私 / 基数成本，参数必须引用版本化数据、SLO 或风险基线。
边界与注意事项
平台提供资源与模型预算，编排决定任务内如何消费；评测阈值与线上容量阈值不能混为一个数字。治理与可观测层 的总边界是：在线治理不替代离线评测，执行隔离仍由基础设施沙箱负责；错误会继续污染 编排门禁、工具授权、事件响应和离线评测输入。

5. 事件 Pivot

事件 Pivot必须在动作发生前或事件使用点给出可执行、可审计的类型化结果；unknown 不能被静默当作 allow。

职责与 I/O
governance-owner 在“事件 Pivot”环节接收 identity context、资源动作、策略版本、风险与关联事件，产出 类型化 PolicyOutcome、审计事件、脱敏遥测或事件升级；输出必须能由下游按版本消费。
工作机制
控制点把 identity、resource、action、purpose、风险和 policy_version 求值为 allow、deny、require_approval 或 escalate。本环节把 identity context、资源动作、策略版本、风险与关联事件 中与“事件 Pivot”相关的事实转成 类型化 PolicyOutcome、审计事件、脱敏遥测或事件升级 的一部分。
必须成立的约束
策略依赖不可用、字段缺失或证据过期时采用 fail-safe；审批只绑定所见对象、版本、范围和有效期。本章共同约束是：在线控制必须 fail-safe，证据必须可关联且敏感值最小化；违反时必须产生可区分的拒绝、未知或升级结果。
难点与取舍
更严格会增加用户摩擦和人工负载，更宽松会扩大风险；阈值依据风险等级、误拦成本和事件数据校准。本层总体取舍是 控制强度与用户摩擦、可观测深度与隐私 / 基数成本，参数必须引用版本化数据、SLO 或风险基线。
边界与注意事项
治理决定是否允许和如何留证，不执行工具动作；HITL 是控制结果，不是让人工修补任意不完整上下文。治理与可观测层 的总边界是：在线治理不替代离线评测，执行隔离仍由基础设施沙箱负责；错误会继续污染 编排门禁、工具授权、事件响应和离线评测输入。

难点与取舍

冲突目标参数依据决策边界
控制强度与用户摩擦、可观测深度与隐私 / 基数成本使用版本化回放、失败切片、SLO 分位数和风险等级，不把教学值当生产默认。governance-owner只调整本层机制；跨层变化需要对应 owner 与独立验收。
自动化收益 vs. 失败放大比较开启 / 关闭“贯穿关联上下文”与“事件 Pivot”的消融结果、人工负载和恢复时间。证据不足返回 pending / unknown / escalate，不能靠更长 Prompt 隐藏边界。

失败传播与 first divergence

  1. 上游输入：先确认 identity context、资源动作、策略版本、风险与关联事件 的身份、作用域、时间与版本是否一致；输入错误不应在本层被“修正”为新事实。
  2. 首个偏离：按 贯穿关联上下文 → 定义 Trace 语义 → 日志脱敏 → 指标基数 → 事件 Pivot 比较预期与实际中间产物，第一个不满足概念不变量的 stage 才是本章候选根因。
  3. 下游影响：本层错误会改变 编排门禁、工具授权、事件响应和离线评测输入；最终症状可能在更晚阶段出现，但不能从症状直接倒推 Prompt。
  4. 归属边界：governance-owner修复本层契约或机制；若 first divergence 位于相邻层，保留证据并交给相应 owner，不跨层调参。

工程要点

  1. 稳定传播 trace_id、span_id、run_id、tenant_id、actor_id、decision_id、tool_call_id 和 policy_version；敏感值使用引用或不可逆摘要。
  2. 模型 span 记录 model、prompt/template version、token、latency、outcome 和安全分类，不默认存储明文 prompt/response。
  3. 定义 RED/USE 类聚合指标与 SLO；用 exemplar 从异常指标跳回代表 trace。
  4. 采样策略按风险分层：错误、拒绝、高成本和高风险动作保留完整结构化证据，正常低风险流量可采样。
  5. 诊断写成“事实 → 假设 → 验证 → 修复 owner”，避免把相关性当因果。

常见坑

  • 日志等于可观测：没有关联和结构，无法重建跨组件因果。
  • 全量记录上下文：制造 PII 泄露、访问扩散和高成本。
  • 指标标签塞 user_id：导致 cardinality 爆炸；身份应在 trace 或受控审计查询中。
  • 只看模型成功率：HTTP 200 可能伴随未授权动作、错误引用或业务验收失败。

理论追问

请先脱稿回答，再展开参考答案核对边界、证据和取舍。

  1. Trace、Log 和 Metric 各回答什么问题？

    查看参考答案

    Trace 关联一次 run 的跨组件因果链，Log 记录具体事件细节，Metric 聚合趋势与 SLO。三者应通过 run_id、span_id、版本和错误分类连接；只有文本日志无法重建状态迁移，只有指标又看不到单次原因。

  2. 跨模型、RAG 和工具调用时，span 父子关系应如何设计？

    查看参考答案

    以 run 为根，编排 step 为主要子 span，step 下挂 retrieval、model、policy、tool 与 acceptance；异步动作通过 link 关联。父子关系表达控制与等待，不能仅按时间相邻推断因果。

  3. 哪些 Trace 不应被普通采样丢弃？

    查看参考答案

    越权与安全事件、effect unknown、补偿失败、人工升级、业务验收失败、预算硬终止和罕见关键错误应保留或提高采样。正常高频路径可按策略采样，但仍需保留聚合指标和版本分布。

  4. 如何控制高基数标签而不失去诊断能力？

    查看参考答案

    Metric 标签只保留有限枚举，如模型族、工具类型、错误分类和租户层级；run_id、用户 ID、Prompt 哈希等高基数值放入 Trace 或 Log，通过示例链接下钻。敏感标识还需 tokenization。

  5. 完整 Prompt 和工具结果是否应该写入 Trace？

    查看参考答案

    默认不应无差别保存。应做字段最小化、脱敏、内容摘要与引用 ID，只在受控存储保留确有调试必要的原文，并设置短 TTL、细粒度权限和 break-glass 审计。可观测性不能成为新的泄露面。

Agent 系统如何设计可观测性？

我先定义 run 主链路和 span 语义，再让 trace、结构化 log、metric 与 audit 共享稳定关联字段。Metric 用于发现 SLO 和成本异常，trace 定位单次因果，log 补充类型化事件，audit 回答授权责任。敏感上下文默认不落盘，高风险失败提高采样率；诊断结果必须落到模型、检索、编排、工具或治理的具体 owner。

日常实践关键点

核心判断

本章的日常工作不是记住名词，而是把“Trace、Log 与 Metric”变成可执行控制环：先识别触发信号，再用版本化证据定位首个偏离点，由 governance-owner 执行动作，最后用成功、拒绝、降级三类结果验收。推荐顺序是 贯穿关联上下文 → 定义 Trace 语义 → 执行日志脱敏 → 控制指标基数 → 支持事件 Pivot；任何参数都必须从真实 SLO、风险基线或失败样本推导。

    诊断顺序
    诊断“支持事件 Pivot”时先冻结 tenant、actor、policy_version、decision、trace_id、风险等级与审计事件。定位请求在哪个控制点应被允许、拒绝、审批或升级，核对策略版本、风险等级、审批上下文和事件 owner 是否完整。
    执行动作
    落实“支持事件 Pivot”时，把策略结果、HITL packet、事件等级和接管时限结构化；依赖不可用或结论 unknown 时采用 fail-safe 行为。由 governance-owner 记录执行前后的版本、决定与接管条件。

    控制面证据不完整时切到拒绝、只读或人工路径；事故后用时间线和行动项验证修复真正关闭；切换到更严格策略或只读降级，冻结高风险动作，并创建带时间线的事件记录。对“支持事件 Pivot”先保证可恢复和可审计，再优化自动化率。

排障顺序

  1. 冻结请求、策略、模型 / 工具、数据和部署版本，避免边复现边变化。
  2. 从最早可验证输入开始，逐段对照契约、决策、动作与验收事件；在首个偏离点停止跨层调参。
  3. 由 governance-owner 判断修复、降级、回滚或转人工，并记录未验证假设。
贯穿案例｜企业差旅 Agent：员工请求预订超预算航班时，只把制度条款当知识证据，把员工靠窗偏好当记忆，把预订动作交给工具层；本章只处理 Trace、Log 与 Metric 的控制与证据。若发现策略结果缺失、日志不可关联、授权在使用点失效、敏感信息泄漏或事件无人接管，先保留 trace_id 与版本，再在本层首个偏离点修复，避免把问题推给 Prompt。

应留证据

最小证据集应能回答谁在何时依据哪个版本、对什么输入做了什么决定，以及结果如何被验证。敏感值使用教学占位或受控引用，不把日志全文复制进面试材料。

input_version: teaching-v1

实践型面试要点

表达时必须标记经历属于生产实战、个人实验或方案设计，只主张自己实际完成的职责，并用 Trace、评测或变更记录举证。企业差旅案例、示例版本和指标均为教学占位；没有验证线上规模、事故协作或长期效果时要主动说明。
