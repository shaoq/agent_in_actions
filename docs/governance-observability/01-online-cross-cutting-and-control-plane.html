01 在线横切职责与控制面

学习目标

  • 区分 data plane 的业务执行与 control plane 的策略判定。
  • 在输入、检索、动作和输出边界放置 fail-closed 或降级门禁。
  • 定义治理决定如何通过 obligations 约束编排、工具和输出。

核心概念

链路总览与现有讲解

在线横切职责与控制面属于治理与可观测层：输入是identity context、资源动作、策略版本、风险与关联事件，输出是类型化 PolicyOutcome、审计事件、脱敏遥测或事件升级。学习时先沿现有图、表或主题章节理解链路，再用后续 stage 检查每个中间产物的不变量。

核心环节：从输入到可验证输出

1. 在线控制点

在线控制点必须在动作发生前或事件使用点给出可执行、可审计的类型化结果；unknown 不能被静默当作 allow。

职责与 I/O
governance-owner 在“在线控制点”环节接收 identity context、资源动作、策略版本、风险与关联事件，产出 类型化 PolicyOutcome、审计事件、脱敏遥测或事件升级；输出必须能由下游按版本消费。
工作机制
控制点把 identity、resource、action、purpose、风险和 policy_version 求值为 allow、deny、require_approval 或 escalate。本环节把 identity context、资源动作、策略版本、风险与关联事件 中与“在线控制点”相关的事实转成 类型化 PolicyOutcome、审计事件、脱敏遥测或事件升级 的一部分。
必须成立的约束
策略依赖不可用、字段缺失或证据过期时采用 fail-safe；审批只绑定所见对象、版本、范围和有效期。本章共同约束是：在线控制必须 fail-safe，证据必须可关联且敏感值最小化；违反时必须产生可区分的拒绝、未知或升级结果。
难点与取舍
更严格会增加用户摩擦和人工负载，更宽松会扩大风险；阈值依据风险等级、误拦成本和事件数据校准。本层总体取舍是 控制强度与用户摩擦、可观测深度与隐私 / 基数成本，参数必须引用版本化数据、SLO 或风险基线。
边界与注意事项
治理决定是否允许和如何留证，不执行工具动作；HITL 是控制结果，不是让人工修补任意不完整上下文。治理与可观测层 的总边界是：在线治理不替代离线评测，执行隔离仍由基础设施沙箱负责；错误会继续污染 编排门禁、工具授权、事件响应和离线评测输入。

5. 明确跨层 Owner

明确跨层 Owner成立的前提是每个控制决定只有一个最终 owner，交接不转移未声明的权限与验收责任。

职责与 I/O
governance-owner 在“明确跨层 Owner”环节接收 identity context、资源动作、策略版本、风险与关联事件，产出 类型化 PolicyOutcome、审计事件、脱敏遥测或事件升级；输出必须能由下游按版本消费。
工作机制
控制点以类型化输入 / 输出连接 owner；交出的是受约束任务或事实，不是共享的隐式控制权。本环节把 identity context、资源动作、策略版本、风险与关联事件 中与“明确跨层 Owner”相关的事实转成 类型化 PolicyOutcome、审计事件、脱敏遥测或事件升级 的一部分。
必须成立的约束
决策、推进、动作、验收和事件升级必须分别可归属，交接双方不能同时写同一控制状态。本章共同约束是：在线控制必须 fail-safe，证据必须可关联且敏感值最小化；违反时必须产生可区分的拒绝、未知或升级结果。
难点与取舍
拆分能提高并行度，却增加协调、上下文复制和冲突成本；是否拆分要由失败归因与吞吐基线决定。本层总体取舍是 控制强度与用户摩擦、可观测深度与隐私 / 基数成本，参数必须引用版本化数据、SLO 或风险基线。
边界与注意事项
推理可以建议，工具可以执行，治理可以允许或拒绝，但只有编排 owner 推进状态，业务 owner 做最终验收。治理与可观测层 的总边界是：在线治理不替代离线评测，执行隔离仍由基础设施沙箱负责；错误会继续污染 编排门禁、工具授权、事件响应和离线评测输入。

难点与取舍

冲突目标参数依据决策边界
控制强度与用户摩擦、可观测深度与隐私 / 基数成本使用版本化回放、失败切片、SLO 分位数和风险等级，不把教学值当生产默认。governance-owner只调整本层机制；跨层变化需要对应 owner 与独立验收。
自动化收益 vs. 失败放大比较开启 / 关闭“在线控制点”与“明确跨层 Owner”的消融结果、人工负载和恢复时间。证据不足返回 pending / unknown / escalate，不能靠更长 Prompt 隐藏边界。

失败传播与 first divergence

  1. 上游输入：先确认 identity context、资源动作、策略版本、风险与关联事件 的身份、作用域、时间与版本是否一致；输入错误不应在本层被“修正”为新事实。
  2. 首个偏离：按 在线控制点 → 类型化策略结果 → Fail-safe 默认 → 策略版本 → 明确跨层 Owner 比较预期与实际中间产物，第一个不满足概念不变量的 stage 才是本章候选根因。
  3. 下游影响：本层错误会改变 编排门禁、工具授权、事件响应和离线评测输入；最终症状可能在更晚阶段出现，但不能从症状直接倒推 Prompt。
  4. 归属边界：governance-owner修复本层契约或机制；若 first divergence 位于相邻层，保留证据并交给相应 owner，不跨层调参。

工程要点

  1. 策略判定输入至少含 subject、action、resource、context、risk、policy_version；输出是结构化 decision、reason、obligations 与有效期。
  2. 关键策略服务不可用时明确 fail-closed、只读降级或人工升级，禁止悄悄放行。
  3. 编排负责在正确生命周期点调用治理并执行暂停；治理不直接篡改 run state。
  4. 把“是否允许”和“允许后必须做什么”分开，例如允许发送但必须脱敏、限额和保留审计。

常见坑

  • 只在入口鉴权：登录不等于对具体资源和副作用持续授权。
  • 把 guardrail 写进 system prompt：模型可建议，不能成为唯一强制点。
  • 治理层执行工具：会模糊授权与执行 owner，破坏审计因果。
  • 所有失败都拒绝：低风险读请求可设计可观测降级，高风险写操作应 fail-closed。

理论追问

请先脱稿回答，再展开参考答案核对边界、证据和取舍。

  1. 在线治理为什么不能只负责记录 Trace？

    查看参考答案

    治理必须在请求、检索、模型决策、工具动作和输出处执行授权、拦截、脱敏、预算或审批；Trace 是这些决策的证据。若只观察不干预，越权和危险副作用会在离线分析前已经发生。

  2. 治理控制面与编排控制层如何分工？

    查看参考答案

    编排决定任务如何推进并持有 Run State；治理根据身份、资源、动作、内容和策略给出 allow、deny、redact、pause 等确定性裁决。编排必须遵守裁决并选择后续分支，治理不替代业务状态机。

  3. 为什么输入检查通过后还要检查检索、动作和输出？

    查看参考答案

    风险会在链路中变化：检索可带入恶意指令，模型可生成越权动作，工具参数可能扩大范围，输出可能泄露 PII。单点入口过滤无法覆盖组合攻击，必须在信任边界逐点校验。

  4. 治理策略应 fail-open 还是 fail-closed？

    查看参考答案

    按风险分级。高风险写操作、敏感数据和权限决策在策略服务不可用时应 fail-closed 或转人工；低风险只读问答可采用降级与最小能力。选择要写入 SLO、告警和演练，不能由调用方临时猜测。

  5. 在线治理 Trace 如何反馈离线评测？

    查看参考答案

    评测层消费脱敏的 policy decision、命中规则、上下文、动作与结果，分析误报、漏报、绕过和业务影响，形成回归集与策略建议。评测不直接修改在线 run，变更需版本化并通过门禁后发布。

面试表达

为什么治理与可观测是横切层？

因为风险在输入、检索、模型决策、工具动作和输出多个信任边界出现。我的设计让治理对每个边界返回可审计的 allow、deny、require-approval 与 obligations，编排负责执行状态转换；同时用同一 trace 关联策略决定和实际结果。离线评测验证这些策略是否有效，基础设施沙箱提供真实隔离，但二者不替代在线门禁。

日常实践关键点

核心判断

本章的日常工作不是记住名词，而是把“在线横切职责与控制面”变成可执行控制环：先识别触发信号，再用版本化证据定位首个偏离点，由 governance-owner 执行动作，最后用成功、拒绝、降级三类结果验收。推荐顺序是 维护在线控制点 → 返回类型化策略结果 → 设置 Fail-safe 默认 → 固定策略版本 → 明确跨层 Owner；任何参数都必须从真实 SLO、风险基线或失败样本推导。

深度实践卡

  • 1. 维护在线控制点
    触发信号
    当企业差旅 Agent 的“在线横切职责与控制面”链路出现策略结果缺失、日志不可关联、授权在使用点失效、敏感信息泄漏或事件无人接管，或变更直接触及“维护在线控制点”的输入、策略或 owner 时，必须启动这项检查。
    诊断顺序
    诊断“维护在线控制点”时先冻结 tenant、actor、policy_version、decision、trace_id、风险等级与审计事件。定位请求在哪个控制点应被允许、拒绝、审批或升级，核对策略版本、风险等级、审批上下文和事件 owner 是否完整。
    执行动作
    落实“维护在线控制点”时，把策略结果、HITL packet、事件等级和接管时限结构化；依赖不可用或结论 unknown 时采用 fail-safe 行为。由 governance-owner 记录执行前后的版本、决定与接管条件。
    验证方法
    验收“维护在线控制点”时，运行拒绝路径负测、审计回放、脱敏扫描和故障演练，确认 fail-safe 与 owner 接管生效；同时比较正确、拒绝和降级结果，阈值只从真实基线与风险容忍度推导。
    止损与取舍
    控制面证据不完整时切到拒绝、只读或人工路径；事故后用时间线和行动项验证修复真正关闭；切换到更严格策略或只读降级，冻结高风险动作，并创建带时间线的事件记录。对“维护在线控制点”先保证可恢复和可审计，再优化自动化率。
    面试记忆点
    围绕“维护在线控制点”准备一条“症状—首个偏离—动作—证据—止损”的真实或个人实验 Trace，并明确自己只承担的 governance-owner 职责。

    执行动作
    落实“明确跨层 Owner”时，用 RACI / control-point 矩阵固定输入、输出、允许动作、拒绝结果和接管时限，交接双方只通过类型化结果协作。由 governance-owner 记录执行前后的版本、决定与接管条件。

    止损与取舍
    责任或验收边界不清时暂停自动推进，把控制权收敛到单一 owner，并保留待裁决事实清单；切换到更严格策略或只读降级，冻结高风险动作，并创建带时间线的事件记录。对“明确跨层 Owner”先保证可恢复和可审计，再优化自动化率。

排障顺序

  1. 冻结请求、策略、模型 / 工具、数据和部署版本，避免边复现边变化。
  2. 从最早可验证输入开始，逐段对照契约、决策、动作与验收事件；在首个偏离点停止跨层调参。
  3. 由 governance-owner 判断修复、降级、回滚或转人工，并记录未验证假设。
贯穿案例｜企业差旅 Agent：员工请求预订超预算航班时，只把制度条款当知识证据，把员工靠窗偏好当记忆，把预订动作交给工具层；本章只处理 在线横切职责与控制面 的控制与证据。若发现策略结果缺失、日志不可关联、授权在使用点失效、敏感信息泄漏或事件无人接管，先保留 trace_id 与版本，再在本层首个偏离点修复，避免把问题推给 Prompt。

应留证据

最小证据集应能回答谁在何时依据哪个版本、对什么输入做了什么决定，以及结果如何被验证。敏感值使用教学占位或受控引用，不把日志全文复制进面试材料。

代表性工程产物｜artifact-gov-01-control-point-matrix
case: enterprise-travel-agent
component: governance-observability
unit: 01
owner: governance-owner
input_version: teaching-v1
policy_or_contract_version: teaching-v3
decision: allow | deny | degrade | escalate
first_divergence: stage-name
evidence_refs: [trace-demo, eval-slice-demo]
validation: passed | failed | unknown
rollback_or_handoff: action-and-owner

以上字段是学习用数据形状，不是生产默认值。实际产物还应记录时间、租户 / 主体作用域、变更来源和保留策略，并能从 evidence_refs 回放本章结论。

实践型面试要点

表达时必须标记经历属于生产实战、个人实验或方案设计，只主张自己实际完成的职责，并用 Trace、评测或变更记录举证。企业差旅案例、示例版本和指标均为教学占位；没有验证线上规模、事故协作或长期效果时要主动说明。

实践面试 1

关联实践：维护在线控制点、返回类型化策略结果

常见问法

在生产或可复现实验中，你如何把“维护在线控制点”和“返回类型化策略结果”落到企业差旅 Agent 的 在线横切职责与控制面 链路？

60–90 秒回答骨架

先交代 在线横切职责与控制面 的业务风险和 governance-owner 边界；再说触发信号与首个偏离点；随后说明契约、动作和参数依据；最后给出验证、止损和复盘结果。回答要同时覆盖正常路径与拒绝 / 降级路径。

可信证据

围绕“维护在线控制点 / 返回类型化策略结果”展示脱敏后的 trace_id、tenant、actor、policy_version、decision、trace_id、风险等级与审计事件、前后对照和失败切片；个人实验还应给出固定输入、复现步骤和观察结果，不虚构线上规模。

高频追问

为什么问题属于 治理与可观测层 而不是相邻层？参数如何从 SLO 或风险基线推导？结果未知时怎样避免扩大副作用？governance-owner 不在线时，谁依据什么证据接管？

边界与取舍

“维护在线控制点 / 返回类型化策略结果”会增加契约、存储或人工成本，只有错误成本与复现收益覆盖这些成本时才扩大自动化。未处理过该类生产事故时，应明确这是方案设计或个人实验结论。
