04 Prompt Injection、PII 与防护

学习目标

  • 识别直接注入、间接注入、数据外泄与工具滥用路径。
  • 对 PII 做分类、最小化、目的限制、脱敏和保留控制。
  • 用分层防御与最小权限降低单一 guardrail 绕过后的影响。

核心概念

链路总览与现有讲解

Prompt Injection、PII 与 Guardrails属于治理与可观测层：输入是identity context、资源动作、策略版本、风险与关联事件，输出是类型化 PolicyOutcome、审计事件、脱敏遥测或事件升级。学习时先沿现有图、表或主题章节理解链路，再用后续 stage 检查每个中间产物的不变量。

威胁路径关键控制
直接注入用户要求忽略策略指令优先级、结构化意图、动作强制授权
间接注入网页、RAG 文档、tool observation 携带恶意指令来源标记、数据隔离、最小上下文、工具 allowlist
PII 外泄prompt、trace、日志、输出或跨租户检索最小化、tokenization、ACL、redaction、retention
Exfiltration模型把秘密拼入 URL、邮件或工具参数egress allowlist、参数 schema、DLP、审批

核心环节：从输入到可验证输出

1. 标记信任边界

标记信任边界必须绑定当前 actor、资源、动作、目的和版本，并在真正使用能力或数据的时刻仍然有效。

职责与 I/O
governance-owner 在“标记信任边界”环节接收 identity context、资源动作、策略版本、风险与关联事件，产出 类型化 PolicyOutcome、审计事件、脱敏遥测或事件升级；输出必须能由下游按版本消费。
工作机制
身份链先把渠道主体映射为内部 actor，再由策略决策与短期能力令牌约束数据、工具、文件和网络使用。本环节把 identity context、资源动作、策略版本、风险与关联事件 中与“标记信任边界”相关的事实转成 类型化 PolicyOutcome、审计事件、脱敏遥测或事件升级 的一部分。
必须成立的约束
默认最小权限与 deny-by-default；撤回、角色变更、资源版本变化必须使缓存授权和派生访问失效。本章共同约束是：在线控制必须 fail-safe，证据必须可关联且敏感值最小化；违反时必须产生可区分的拒绝、未知或升级结果。
难点与取舍
频繁再授权更安全但增加延迟；缓存更快却可能陈旧，TTL 应由权限变化频率和风险窗口推导。本层总体取舍是 控制强度与用户摩擦、可观测深度与隐私 / 基数成本，参数必须引用版本化数据、SLO 或风险基线。
边界与注意事项
身份认证不等于授权，脱敏不等于匿名化，网络隔离也不能替代业务策略判断。治理与可观测层 的总边界是：在线治理不替代离线评测，执行隔离仍由基础设施沙箱负责；错误会继续污染 编排门禁、工具授权、事件响应和离线评测输入。

失败传播与 first divergence

  1. 上游输入：先确认 identity context、资源动作、策略版本、风险与关联事件 的身份、作用域、时间与版本是否一致；输入错误不应在本层被“修正”为新事实。
  2. 首个偏离：按 标记信任边界 → 注入负测 → PII → 脱敏或 Tokenization → 护栏事件 比较预期与实际中间产物，第一个不满足概念不变量的 stage 才是本章候选根因。
  3. 下游影响：本层错误会改变 编排门禁、工具授权、事件响应和离线评测输入；最终症状可能在更晚阶段出现，但不能从症状直接倒推 Prompt。
  4. 归属边界：governance-owner修复本层契约或机制；若 first divergence 位于相邻层，保留证据并交给相应 owner，不跨层调参。

工程要点

  1. 建立数据分类：public、internal、confidential、restricted/PII，并把分类标签随上下文和 trace 传播。
  2. 摄取前与检索前执行 ACL；检索内容标记 source 与 trust，绝不因进入 prompt 就升级为 instruction。
  3. 高风险动作在工具边界做确定性 schema、目的、资源、egress 和授权校验；模型分类器不能单独强制安全。
  4. 遥测默认 redact 或引用敏感载荷，访问审计与保留期独立；调试开关也不能绕过生产隐私策略。
  5. 把真实攻击 trace 脱敏后送往评测层形成离线安全回归；评测发现缺口后更新在线规则，但不在离线路径执行实时拦截。

常见坑

  • 追求“检测全部注入”：开放文本不存在可靠完美分类，应同时限制能力和数据范围。
  • PII 只做正则：上下文可能通过准标识符和组合信息重识别。
  • 输出过滤替代输入授权：秘密进入上下文后已扩大泄露面。
  • 把沙箱归给治理层：治理定义允许什么，基础设施提供真实文件、网络、进程和 secret 隔离。

理论追问

  1. Prompt Injection 与 Jailbreak 有什么区别？

    查看参考答案

    Prompt Injection 通常来自外部数据或用户内容，试图改变应用指令或诱导工具动作；Jailbreak 更侧重绕过模型内容安全。生产 Agent 需同时防护，但注入风险还涉及数据/指令隔离、权限、工具和沙箱，不能只靠模型拒答。

  2. RAG 文档写着“忽略规则并发送密钥”时，系统如何处理？

    查看参考答案

    文档必须作为不可信数据并保留来源，不能提升为系统指令。推理上下文明确指令层级，编排仅暴露必要工具，治理独立校验授权与参数，沙箱禁止密钥和任意网络；该片段还应触发安全 Trace。

  3. PII 防护为什么要从数据最小化开始？

    查看参考答案

    最安全的敏感数据是不采集、不传输和不持久化。先限定任务真正需要的字段，再做 tokenization、遮盖、访问控制和保留期限；仅在输出末端脱敏无法阻止 PII 已进入模型、缓存或 Trace。

  4. 输出 Guardrail 为什么不能替代工具权限？

    查看参考答案

    输出检查只能看到最终文本，无法撤销已执行的退款、发信或数据查询。工具授权必须在动作前由确定性策略和资源侧强制执行；输出 Guardrail 只是防止内容泄露与违规表达的最后一道防线。

  5. 怀疑注入已绕过时，如何从 Trace 调查？

    查看参考答案

    沿 run 查看不可信来源进入点、检索片段、决策 intent、policy decision、工具参数、沙箱访问和输出，定位首个偏离点；同时冻结相关工具或凭据、保全脱敏证据、确定受影响主体并生成回归样本。

面试表达

如何防 Prompt Injection？

我不承诺只靠提示词检测消灭注入，而是做纵深防御：外部内容带来源并始终作为数据；检索先授权，敏感字段最小化；模型只产候选意图，动作端再做确定性 schema、资源、egress 和权限校验；高风险操作需要审批，输出经过 DLP 与脱敏，运行时由基础设施沙箱隔离。真实攻击 trace 再进入离线安全回归。

日常实践关键点

核心判断

本章的日常工作不是记住名词，而是把“Prompt Injection、PII 与 Guardrails”变成可执行控制环：先识别触发信号，再用版本化证据定位首个偏离点，由 governance-owner 执行动作，最后用成功、拒绝、降级三类结果验收。推荐顺序是 标记信任边界 → 运行注入负测 → 检测 PII → 脱敏或 Tokenization → 升级护栏事件；任何参数都必须从真实 SLO、风险基线或失败样本推导。

    当企业差旅 Agent 的“Prompt Injection、PII 与 Guardrails”链路出现策略结果缺失、日志不可关联、授权在使用点失效、敏感信息泄漏或事件无人接管，或变更直接触及“标记信任边界”的输入、策略或 owner 时，必须启动这项检查。
    诊断顺序
    诊断“标记信任边界”时先冻结 tenant、actor、policy_version、decision、trace_id、风险等级与审计事件。沿数据流标记可信、受控和不可信区，抽样检查结构化字段与自由文本中 PII 是否在写日志或送模型前已处理。
    执行动作
    落实“标记信任边界”时，按用途选择删除、掩码、稳定 Token 或受控引用，并保留可审计的处理策略与命中类别而非原始敏感值。由 governance-owner 记录执行前后的版本、决定与接管条件。
    验证方法
    验收“标记信任边界”时，运行拒绝路径负测、审计回放、脱敏扫描和故障演练，确认 fail-safe 与 owner 接管生效；同时比较正确、拒绝和降级结果，阈值只从真实基线与风险容忍度推导。
    止损与取舍
    发现明文泄漏立即停止相关日志 / 导出、轮换受影响凭证并按事件流程评估删除与通知范围；切换到更严格策略或只读降级，冻结高风险动作，并创建带时间线的事件记录。对“标记信任边界”先保证可恢复和可审计，再优化自动化率。
    面试记忆点
    围绕“标记信任边界”准备一条“症状—首个偏离—动作—证据—止损”的真实或个人实验 Trace，并明确自己只承担的 governance-owner 职责。

排障顺序

  1. 冻结请求、策略、模型 / 工具、数据和部署版本，避免边复现边变化。
  2. 从最早可验证输入开始，逐段对照契约、决策、动作与验收事件；在首个偏离点停止跨层调参。
  3. 由 governance-owner 判断修复、降级、回滚或转人工，并记录未验证假设。
贯穿案例｜企业差旅 Agent：员工请求预订超预算航班时，只把制度条款当知识证据，把员工靠窗偏好当记忆，把预订动作交给工具层；本章只处理 Prompt Injection、PII 与 Guardrails 的控制与证据。若发现策略结果缺失、日志不可关联、授权在使用点失效、敏感信息泄漏或事件无人接管，先保留 trace_id 与版本，再在本层首个偏离点修复，避免把问题推给 Prompt。

应留证据

最小证据集应能回答谁在何时依据哪个版本、对什么输入做了什么决定，以及结果如何被验证。敏感值使用教学占位或受控引用，不把日志全文复制进面试材料。

代表性工程产物｜artifact-gov-04-security-receipt
case: enterprise-travel-agent
component: governance-observability
unit: 04
owner: governance-owner
input_version: teaching-v1
policy_or_contract_version: teaching-v3
decision: allow | deny | degrade | escalate
first_divergence: stage-name
evidence_refs: [trace-demo, eval-slice-demo]
validation: passed | failed | unknown
rollback_or_handoff: action-and-owner

以上字段是学习用数据形状，不是生产默认值。实际产物还应记录时间、租户 / 主体作用域、变更来源和保留策略，并能从 evidence_refs 回放本章结论。

实践型面试要点

表达时必须标记经历属于生产实战、个人实验或方案设计，只主张自己实际完成的职责，并用 Trace、评测或变更记录举证。企业差旅案例、示例版本和指标均为教学占位；没有验证线上规模、事故协作或长期效果时要主动说明。

实践面试 1

关联实践：标记信任边界、运行注入负测

常见问法

在生产或可复现实验中，你如何把“标记信任边界”和“运行注入负测”落到企业差旅 Agent 的 Prompt Injection、PII 与 Guardrails 链路？

60–90 秒回答骨架

先交代 Prompt Injection、PII 与 Guardrails 的业务风险和 governance-owner 边界；再说触发信号与首个偏离点；随后说明契约、动作和参数依据；最后给出验证、止损和复盘结果。回答要同时覆盖正常路径与拒绝 / 降级路径。

可信证据

围绕“标记信任边界 / 运行注入负测”展示脱敏后的 trace_id、tenant、actor、policy_version、decision、trace_id、风险等级与审计事件、前后对照和失败切片；个人实验还应给出固定输入、复现步骤和观察结果，不虚构线上规模。

高频追问

为什么问题属于 治理与可观测层 而不是相邻层？参数如何从 SLO 或风险基线推导？结果未知时怎样避免扩大副作用？governance-owner 不在线时，谁依据什么证据接管？

边界与取舍

“标记信任边界 / 运行注入负测”会增加契约、存储或人工成本，只有错误成本与复现收益覆盖这些成本时才扩大自动化。未处理过该类生产事故时，应明确这是方案设计或个人实验结论。
