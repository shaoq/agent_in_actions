04 运行时、存储与网络

学习目标

  • 区分无状态服务、durable run state、队列、对象存储、缓存与审计存储。
  • 理解进程丢失、消息重复、存储暂时不可用和网络分区下的恢复边界。
  • 设计服务间身份、egress allowlist、TLS、DNS/SSRF 防护和 secret 注入。
  • 不把编排状态机或业务数据所有权塞进基础设施层。

核心概念

资源保存什么关键保证
无状态 runtime当前请求计算与短暂缓冲可替换、健康检查、优雅终止
durable state storerun checkpoint、版本、幂等键一致更新、租户范围、备份恢复
queue / event log待处理工作和事件至少一次投递 + consumer 幂等、DLQ
object / artifact store大 trace、文件、评测和构建产物版本、校验和、保留与访问策略
cache可重建的派生数据TTL、失效、不能成为唯一真相

工程要点

  1. 计算节点不把关键 run 状态只放内存；编排在阶段边界持久化业务 checkpoint，基础设施提供一致性、可用性和恢复能力但不解释状态语义。
  2. 队列默认按至少一次投递设计，消息含 tenant、run/step、attempt、deadline、schema version 与 idempotency key；consumer 先查提交证据再执行副作用。
  3. 模型网关 egress 只允许审批的 provider/region；工具沙箱使用独立默认拒绝策略，防 DNS rebinding、私网/metadata 访问和危险重定向。
  4. 服务间使用 workload identity 与双向 TLS；secret broker 按调用短期注入，secret 不进入 prompt、镜像、普通环境快照或明文日志。
  5. 对数据库、队列、对象存储和网络执行故障演练：超时、分区、重复消息、主从切换、备份恢复；记录 RPO/RTO 与恢复证据。

常见坑

  • 容器等于无状态：关键 checkpoint 仍留在本地磁盘，重调度后无法恢复。
  • Exactly-once 口号：队列确认与外部副作用无法原子提交，却没有幂等键和对账。
  • 所有流量共用 egress：模型访问与不可信代码拥有相同网络权限。
  • 基础设施接管业务状态：平台自行推断 run 是否完成，破坏编排边界。

理论追问

请先脱稿回答，再展开参考答案核对边界、证据和取舍。

  1. 基础设施为什么不应理解 Memory 或 Run State 的业务语义？

    查看参考答案

    基础设施提供计算、存储、网络和可靠性原语，上层组件定义状态字段、生命周期、权限与一致性。若基础设施直接推断“哪些消息是记忆”，会造成职责耦合；它只按明确契约持久化和隔离数据。

  2. Run State 存储需要哪些可靠性能力？

    查看参考答案

    需要原子版本更新、幂等写、并发控制、checkpoint、事件关联、备份和恢复，并按风险设置一致性与持久性。状态 schema 和迁移由编排定义，存储层提供 CAS、事务或日志等机制。

  3. 为什么默认禁止工具运行时任意出网？

    查看参考答案

    任意出网会让提示注入或恶意依赖泄露数据、下载载荷和绕过审计。应使用目标白名单、DNS/代理控制、协议和端口限制，凭据按目标绑定，并记录 egress；确需开放时按任务临时授权。

  4. 凭据应如何交给模型和工具？

    查看参考答案

    模型上下文不应包含原始密钥。运行时通过短期、最小作用域凭据或代理代为调用，工具只在受控进程获得必要句柄，日志和 Trace 进行遮盖；支持轮换、撤销和使用审计。

  5. 单区网络故障时，哪些状态决定能否安全恢复？

    查看参考答案

    需要持久 Run State、已提交副作用与 idempotency/effect receipt、审批和授权版本、队列消费位置。仅复制模型消息不足以判断外部动作是否发生；恢复前还要重新验证依赖和权限。

面试表达

怎样部署可恢复的 Agent 系统？

我让编排和网关计算节点保持可替换，业务 checkpoint 由 owner 组件写入 durable store；队列按至少一次投递设计，用 run/step/idempotency key 去重，文件和大 trace 进版本化对象存储。模型出口与工具沙箱分开做默认拒绝网络策略，服务间用 workload identity 和 mTLS，secret 按需短期注入。基础设施提供持久、网络和恢复原语，不接管业务状态机。
