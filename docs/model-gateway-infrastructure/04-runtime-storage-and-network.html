04 运行时、存储与网络

学习目标

  • 区分无状态服务、durable run state、队列、对象存储、缓存与审计存储。
  • 理解进程丢失、消息重复、存储暂时不可用和网络分区下的恢复边界。
  • 设计服务间身份、egress allowlist、TLS、DNS/SSRF 防护和 secret 注入。
  • 不把编排状态机或业务数据所有权塞进基础设施层。

核心概念

链路总览与现有讲解

运行时、存储与网络属于模型网关与基础设施：输入是能力请求、模型参数、tenant、deadline、预算和运行依赖，输出是规范化响应、RoutingDecision、资源遥测或类型化基础设施错误。学习时先沿现有图、表或主题章节理解链路，再用后续 stage 检查每个中间产物的不变量。

1. 定义 Runtime SLO

定义 Runtime SLO必须以版本化分布、业务切片和风险容忍度为依据；单个平均值或脱离基线的固定阈值不能代表系统边界。

职责与 I/O
platform-owner 在“定义 Runtime SLO”环节接收 能力请求、模型参数、tenant、deadline、预算和运行依赖，产出 规范化响应、RoutingDecision、资源遥测或类型化基础设施错误；输出必须能由下游按版本消费。
工作机制
端到端 SLO 被分配到阶段 deadline、token / 调用预算、并发和容量水位，再由 admission、限流或降级保护。本环节把 能力请求、模型参数、tenant、deadline、预算和运行依赖 中与“定义 Runtime SLO”相关的事实转成 规范化响应、RoutingDecision、资源遥测或类型化基础设施错误 的一部分。
必须成立的约束
同时看 p50 / p95 / p99、单位任务成本、错误与拒绝率；预算跨层累计且不能靠放宽安全门禁抵消。本章共同约束是：模型只能经网关访问，租户 / 凭证 / 网络与执行环境必须隔离；违反时必须产生可区分的拒绝、未知或升级结果。
难点与取舍
质量、尾延迟、成本与吞吐互相制约；参数由压测、流量预测、风险切片和业务优先级联合校准。本层总体取舍是 质量、延迟、成本、容量和隔离强度之间的多目标权衡，参数必须引用版本化数据、SLO 或风险基线。
边界与注意事项
平台提供资源与模型预算，编排决定任务内如何消费；评测阈值与线上容量阈值不能混为一个数字。模型网关与基础设施 的总边界是：不决定业务意图或在线授权，也不替代工具动作契约；错误会继续污染 推理结果质量、端到端 SLO、成本和沙箱安全证据。

失败传播与 first divergence

  1. 上游输入：先确认 能力请求、模型参数、tenant、deadline、预算和运行依赖 的身份、作用域、时间与版本是否一致；输入错误不应在本层被“修正”为新事实。
  2. 首个偏离：按 定义 Runtime SLO → 治理连接池 → 存储契约 → 网络超时 → 依赖演练 比较预期与实际中间产物，第一个不满足概念不变量的 stage 才是本章候选根因。
  3. 下游影响：本层错误会改变 推理结果质量、端到端 SLO、成本和沙箱安全证据；最终症状可能在更晚阶段出现，但不能从症状直接倒推 Prompt。
  4. 归属边界：platform-owner修复本层契约或机制；若 first divergence 位于相邻层，保留证据并交给相应 owner，不跨层调参。

工程要点

  1. 计算节点不把关键 run 状态只放内存；编排在阶段边界持久化业务 checkpoint，基础设施提供一致性、可用性和恢复能力但不解释状态语义。
  2. 队列默认按至少一次投递设计，消息含 tenant、run/step、attempt、deadline、schema version 与 idempotency key；consumer 先查提交证据再执行副作用。
  3. 模型网关 egress 只允许审批的 provider/region；工具沙箱使用独立默认拒绝策略，防 DNS rebinding、私网/metadata 访问和危险重定向。
  4. 服务间使用 workload identity 与双向 TLS；secret broker 按调用短期注入，secret 不进入 prompt、镜像、普通环境快照或明文日志。
  5. 对数据库、队列、对象存储和网络执行故障演练：超时、分区、重复消息、主从切换、备份恢复；记录 RPO/RTO 与恢复证据。

常见坑

  • 容器等于无状态：关键 checkpoint 仍留在本地磁盘，重调度后无法恢复。
  • Exactly-once 口号：队列确认与外部副作用无法原子提交，却没有幂等键和对账。
  • 所有流量共用 egress：模型访问与不可信代码拥有相同网络权限。
  • 基础设施接管业务状态：平台自行推断 run 是否完成，破坏编排边界。

理论追问

请先脱稿回答，再展开参考答案核对边界、证据和取舍。

  1. 基础设施为什么不应理解 Memory 或 Run State 的业务语义？

    查看参考答案

    基础设施提供计算、存储、网络和可靠性原语，上层组件定义状态字段、生命周期、权限与一致性。若基础设施直接推断“哪些消息是记忆”，会造成职责耦合；它只按明确契约持久化和隔离数据。

  2. Run State 存储需要哪些可靠性能力？

    查看参考答案

    需要原子版本更新、幂等写、并发控制、checkpoint、事件关联、备份和恢复，并按风险设置一致性与持久性。状态 schema 和迁移由编排定义，存储层提供 CAS、事务或日志等机制。

  3. 为什么默认禁止工具运行时任意出网？

    查看参考答案

    任意出网会让提示注入或恶意依赖泄露数据、下载载荷和绕过审计。应使用目标白名单、DNS/代理控制、协议和端口限制，凭据按目标绑定，并记录 egress；确需开放时按任务临时授权。

  4. 凭据应如何交给模型和工具？

    查看参考答案

    模型上下文不应包含原始密钥。运行时通过短期、最小作用域凭据或代理代为调用，工具只在受控进程获得必要句柄，日志和 Trace 进行遮盖；支持轮换、撤销和使用审计。

  5. 单区网络故障时，哪些状态决定能否安全恢复？

    查看参考答案

    需要持久 Run State、已提交副作用与 idempotency/effect receipt、审批和授权版本、队列消费位置。仅复制模型消息不足以判断外部动作是否发生；恢复前还要重新验证依赖和权限。

面试表达

怎样部署可恢复的 Agent 系统？

我让编排和网关计算节点保持可替换，业务 checkpoint 由 owner 组件写入 durable store；队列按至少一次投递设计，用 run/step/idempotency key 去重，文件和大 trace 进版本化对象存储。模型出口与工具沙箱分开做默认拒绝网络策略，服务间用 workload identity 和 mTLS，secret 按需短期注入。基础设施提供持久、网络和恢复原语，不接管业务状态机。

日常实践关键点

核心判断

本章的日常工作不是记住名词，而是把“运行时、存储与网络”变成可执行控制环：先识别触发信号，再用版本化证据定位首个偏离点，由 platform-owner 执行动作，最后用成功、拒绝、降级三类结果验收。推荐顺序是 定义 Runtime SLO → 治理连接池 → 固化存储契约 → 分配网络超时 → 执行依赖演练；任何参数都必须从真实 SLO、风险基线或失败样本推导。

深度实践卡

  • 1. 定义 Runtime SLO
    触发信号
    当企业差旅 Agent 的“运行时、存储与网络”链路出现路由能力不匹配、级联重试、尾延迟恶化、预算失控、跨租户泄漏或依赖抖动，或变更直接触及“定义 Runtime SLO”的输入、策略或 owner 时，必须启动这项检查。
    诊断顺序
    诊断“定义 Runtime SLO”时先冻结 model_profile、route_reason、预算、deadline、tenant、运行环境与依赖健康度。按阶段和业务切片拆解分布，区分平均值正常但尾部或高风险切片越界的情况。
    执行动作
    落实“定义 Runtime SLO”时，从用户 SLO 和风险容忍度反推预算与阈值，记录样本窗口、分位数和参数依据。由 platform-owner 记录执行前后的版本、决定与接管条件。
    验证方法
    验收“定义 Runtime SLO”时，比较路由 Trace、p50/p95/p99、单位任务成本、容量水位与隔离负测，并执行受控降级演练；同时比较正确、拒绝和降级结果，阈值只从真实基线与风险容忍度推导。
    止损与取舍
    触线后按预定顺序限流、降级或拒绝，不能临时放宽安全与质量门槛换吞吐；固定到已知稳定模型或环境，关闭高风险 fallback / cache，按租户和依赖缩小影响面。对“定义 Runtime SLO”先保证可恢复和可审计，再优化自动化率。
    面试记忆点
    围绕“定义 Runtime SLO”准备一条“症状—首个偏离—动作—证据—止损”的真实或个人实验 Trace，并明确自己只承担的 platform-owner 职责。
  • 2. 治理连接池
    触发信号
    当企业差旅 Agent 的“运行时、存储与网络”链路出现路由能力不匹配、级联重试、尾延迟恶化、预算失控、跨租户泄漏或依赖抖动，或变更直接触及“治理连接池”的输入、策略或 owner 时，必须启动这项检查。
    诊断顺序
    诊断“治理连接池”时先冻结 model_profile、route_reason、预算、deadline、tenant、运行环境与依赖健康度。按依赖分解连接建立、排队、执行、传输和持久化耗时，同时检查过期、连接耗尽和局部失败是否被重试放大。
    执行动作
    落实“治理连接池”时，为每个依赖定义 deadline、连接池上限、缓存键 / TTL、健康判定和熔断降级，避免多个层级独立重试。由 platform-owner 记录执行前后的版本、决定与接管条件。

    止损与取舍
    依赖不稳定时关闭非必要缓存或新连接策略，切到只读 / 排队 / 拒绝路径并保护核心租户容量；固定到已知稳定模型或环境，关闭高风险 fallback / cache，按租户和依赖缩小影响面。对“治理连接池”先保证可恢复和可审计，再优化自动化率。

排障顺序

  1. 冻结请求、策略、模型 / 工具、数据和部署版本，避免边复现边变化。
  2. 从最早可验证输入开始，逐段对照契约、决策、动作与验收事件；在首个偏离点停止跨层调参。
  3. 由 platform-owner 判断修复、降级、回滚或转人工，并记录未验证假设。
贯穿案例｜企业差旅 Agent：员工请求预订超预算航班时，只把制度条款当知识证据，把员工靠窗偏好当记忆，把预订动作交给工具层；本章只处理 运行时、存储与网络 的控制与证据。若发现路由能力不匹配、级联重试、尾延迟恶化、预算失控、跨租户泄漏或依赖抖动，先保留 trace_id 与版本，再在本层首个偏离点修复，避免把问题推给 Prompt。

应留证据

最小证据集应能回答谁在何时依据哪个版本、对什么输入做了什么决定，以及结果如何被验证。敏感值使用教学占位或受控引用，不把日志全文复制进面试材料。

实践型面试要点

表达时必须标记经历属于生产实战、个人实验或方案设计，只主张自己实际完成的职责，并用 Trace、评测或变更记录举证。企业差旅案例、示例版本和指标均为教学占位；没有验证线上规模、事故协作或长期效果时要主动说明。

实践面试 1

关联实践：定义 Runtime SLO、治理连接池

常见问法

在生产或可复现实验中，你如何把“定义 Runtime SLO”和“治理连接池”落到企业差旅 Agent 的 运行时、存储与网络 链路？

60–90 秒回答骨架

先交代 运行时、存储与网络 的业务风险和 platform-owner 边界；再说触发信号与首个偏离点；随后说明契约、动作和参数依据；最后给出验证、止损和复盘结果。回答要同时覆盖正常路径与拒绝 / 降级路径。

可信证据

围绕“定义 Runtime SLO / 治理连接池”展示脱敏后的 trace_id、model_profile、route_reason、预算、deadline、tenant、运行环境与依赖健康度、前后对照和失败切片；个人实验还应给出固定输入、复现步骤和观察结果，不虚构线上规模。

高频追问

为什么问题属于 模型网关与基础设施 而不是相邻层？参数如何从 SLO 或风险基线推导？结果未知时怎样避免扩大副作用？platform-owner 不在线时，谁依据什么证据接管？

边界与取舍

“定义 Runtime SLO / 治理连接池”会增加契约、存储或人工成本，只有错误成本与复现收益覆盖这些成本时才扩大自动化。未处理过该类生产事故时，应明确这是方案设计或个人实验结论。
