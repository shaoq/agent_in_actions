03 成本、延迟与容量

学习目标

  • 把端到端延迟拆成排队、网关、提供商首 token、生成和后处理。
  • 按租户、能力、模型和 run 归因输入/输出 token、缓存与基础设施成本。
  • 用到达率、服务时间、并发、token throughput 和配额估算容量。
  • 区分治理层定义预算与基础设施执行配额、扩缩和过载保护。

核心概念

链路总览与现有讲解

成本、延迟与容量属于模型网关与基础设施：输入是能力请求、模型参数、tenant、deadline、预算和运行依赖，输出是规范化响应、RoutingDecision、资源遥测或类型化基础设施错误。学习时先沿现有图、表或主题章节理解链路，再用后续 stage 检查每个中间产物的不变量。

失败传播与 first divergence

  1. 上游输入：先确认 能力请求、模型参数、tenant、deadline、预算和运行依赖 的身份、作用域、时间与版本是否一致；输入错误不应在本层被“修正”为新事实。
  2. 首个偏离：按 调用成本 → 延迟预算 → 容量 → Admission Control → 成本质量门禁 比较预期与实际中间产物，第一个不满足概念不变量的 stage 才是本章候选根因。
  3. 下游影响：本层错误会改变 推理结果质量、端到端 SLO、成本和沙箱安全证据；最终症状可能在更晚阶段出现，但不能从症状直接倒推 Prompt。
  4. 归属边界：platform-owner修复本层契约或机制；若 first divergence 位于相邻层，保留证据并交给相应 owner，不跨层调参。

工程要点

  1. SLO 至少区分 p50/p95/p99 端到端延迟、TTFT、完成率和取消率；流式首屏快不能掩盖最终完成过慢。
  2. 每次调用记录 input/output/cached/reasoning token（若可得）、单价版本、币种、网关开销、租户、run、route 和成功质量标签；价格变更版本化。
  3. 容量以峰值和突发窗口计算：到达率、请求 token 分布、输出长度、服务时间、并发上限、provider token/min 与 request/min 配额共同约束。
  4. 治理层下发每 run/租户的质量、成本、时间预算；网关执行预估、预留、记账与超限信号，编排决定缩短输出、换策略、等待、询问或终止。
  5. 扩缩结合队列等待、并发和 token throughput；保留 headroom 并做 load shedding，不能等 CPU 高才处理外部模型配额耗尽。

常见坑

  • 只看平均延迟：长尾和队头阻塞被平均值隐藏。
  • 以请求数估容量：100 个分类请求与 100 个长上下文生成的资源完全不同。
  • 只优化单价：更弱模型引发重试、人工接管或业务失败，单位成功成本更高。
  • 自动扩容万能论：provider 账户配额、区域容量和冷启动可能无法靠本地副本解决。

理论追问

请先脱稿回答，再展开参考答案核对边界、证据和取舍。

  1. 为什么只看每千 Token 单价无法判断模型成本？

    查看参考答案

    总成本还取决于输入膨胀、输出长度、重试、候选数、缓存命中、工具循环和人工接管。应按成功任务计算 cost per successful task，并结合质量和失败切片，而不是只比较标价。

  2. RPM、TPM 和并发限制分别影响什么？

    查看参考答案

    RPM 限制请求频率，TPM 限制 token 吞吐，并发限制同时占用的连接或推理槽。长上下文可能在 RPM 很低时耗尽 TPM；容量规划必须同时建模到达率、token 分布和服务时间。

  3. 队列积压时为什么不能无限等待？

    查看参考答案

    等待会违反端到端 SLO，并让上游继续堆积、状态过期和用户重试。应设置优先级、队列上限、deadline、背压与拒绝策略；接近超时时选择降级、异步通知或失败，而不是晚到的无用结果。

  4. 如何在成本下降和质量底线之间做路由？

    查看参考答案

    先定义任务切片的最低质量与安全约束，再在合格模型中优化成本和延迟。用离线评测和线上灰度验证路由，关键任务保留强模型或升级条件；不能让预算策略覆盖 hard fail。

  5. 容量测试应模拟哪些 Agent 特有负载？

    查看参考答案

    除平均请求外，要模拟长上下文、工具等待后的突发恢复、多步重试、并行 worker、流式连接和供应商故障转移。报告 p95/p99 延迟、排队、token 吞吐、降级率、成本和成功任务率。

日常实践关键点

核心判断

本章的日常工作不是记住名词，而是把“成本、延迟与容量”变成可执行控制环：先识别触发信号，再用版本化证据定位首个偏离点，由 platform-owner 执行动作，最后用成功、拒绝、降级三类结果验收。推荐顺序是 归集调用成本 → 分配延迟预算 → 预测容量 → 执行 Admission Control → 设置成本质量门禁；任何参数都必须从真实 SLO、风险基线或失败样本推导。

深度实践卡

  • 1. 归集调用成本
    触发信号
    当企业差旅 Agent 的“成本、延迟与容量”链路出现路由能力不匹配、级联重试、尾延迟恶化、预算失控、跨租户泄漏或依赖抖动，或变更直接触及“归集调用成本”的输入、策略或 owner 时，必须启动这项检查。
    诊断顺序
    诊断“归集调用成本”时先冻结 model_profile、route_reason、预算、deadline、tenant、运行环境与依赖健康度。按阶段和业务切片拆解分布，区分平均值正常但尾部或高风险切片越界的情况。
    执行动作
    落实“归集调用成本”时，从用户 SLO 和风险容忍度反推预算与阈值，记录样本窗口、分位数和参数依据。由 platform-owner 记录执行前后的版本、决定与接管条件。
    验证方法
    验收“归集调用成本”时，比较路由 Trace、p50/p95/p99、单位任务成本、容量水位与隔离负测，并执行受控降级演练；同时比较正确、拒绝和降级结果，阈值只从真实基线与风险容忍度推导。
    止损与取舍
    触线后按预定顺序限流、降级或拒绝，不能临时放宽安全与质量门槛换吞吐；固定到已知稳定模型或环境，关闭高风险 fallback / cache，按租户和依赖缩小影响面。对“归集调用成本”先保证可恢复和可审计，再优化自动化率。
    面试记忆点
    围绕“归集调用成本”准备一条“症状—首个偏离—动作—证据—止损”的真实或个人实验 Trace，并明确自己只承担的 platform-owner 职责。

排障顺序

  1. 冻结请求、策略、模型 / 工具、数据和部署版本，避免边复现边变化。
  2. 从最早可验证输入开始，逐段对照契约、决策、动作与验收事件；在首个偏离点停止跨层调参。
  3. 由 platform-owner 判断修复、降级、回滚或转人工，并记录未验证假设。
贯穿案例｜企业差旅 Agent：员工请求预订超预算航班时，只把制度条款当知识证据，把员工靠窗偏好当记忆，把预订动作交给工具层；本章只处理 成本、延迟与容量 的控制与证据。若发现路由能力不匹配、级联重试、尾延迟恶化、预算失控、跨租户泄漏或依赖抖动，先保留 trace_id 与版本，再在本层首个偏离点修复，避免把问题推给 Prompt。

应留证据

最小证据集应能回答谁在何时依据哪个版本、对什么输入做了什么决定，以及结果如何被验证。敏感值使用教学占位或受控引用，不把日志全文复制进面试材料。

实践型面试要点

表达时必须标记经历属于生产实战、个人实验或方案设计，只主张自己实际完成的职责，并用 Trace、评测或变更记录举证。企业差旅案例、示例版本和指标均为教学占位；没有验证线上规模、事故协作或长期效果时要主动说明。

实践面试 1

关联实践：归集调用成本、分配延迟预算

常见问法

在生产或可复现实验中，你如何把“归集调用成本”和“分配延迟预算”落到企业差旅 Agent 的 成本、延迟与容量 链路？

60–90 秒回答骨架

先交代 成本、延迟与容量 的业务风险和 platform-owner 边界；再说触发信号与首个偏离点；随后说明契约、动作和参数依据；最后给出验证、止损和复盘结果。回答要同时覆盖正常路径与拒绝 / 降级路径。

可信证据

围绕“归集调用成本 / 分配延迟预算”展示脱敏后的 trace_id、model_profile、route_reason、预算、deadline、tenant、运行环境与依赖健康度、前后对照和失败切片；个人实验还应给出固定输入、复现步骤和观察结果，不虚构线上规模。

高频追问

为什么问题属于 模型网关与基础设施 而不是相邻层？参数如何从 SLO 或风险基线推导？结果未知时怎样避免扩大副作用？platform-owner 不在线时，谁依据什么证据接管？

边界与取舍

“归集调用成本 / 分配延迟预算”会增加契约、存储或人工成本，只有错误成本与复现收益覆盖这些成本时才扩大自动化。未处理过该类生产事故时，应明确这是方案设计或个人实验结论。
