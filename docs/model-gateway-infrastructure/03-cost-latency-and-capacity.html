03 成本、延迟与容量

学习目标

  • 把端到端延迟拆成排队、网关、提供商首 token、生成和后处理。
  • 按租户、能力、模型和 run 归因输入/输出 token、缓存与基础设施成本。
  • 用到达率、服务时间、并发、token throughput 和配额估算容量。
  • 区分治理层定义预算与基础设施执行配额、扩缩和过载保护。

核心概念

信号定义工程用途
TTFT接收请求到首 token区分排队、连接和提供商计算
TPOT / tokens/s后续 token 间隔 / 生成速率估计流式体验和长输出占用
并发占用到达率 × 平均服务时间（近似）估算 worker、连接与配额需求
单位成功成本总模型与平台成本 / 合格任务数避免“便宜调用”造成更多失败重试
饱和度队列、并发、token/min 与连接池利用率提前过载保护和扩缩

工程要点

  1. SLO 至少区分 p50/p95/p99 端到端延迟、TTFT、完成率和取消率；流式首屏快不能掩盖最终完成过慢。
  2. 每次调用记录 input/output/cached/reasoning token（若可得）、单价版本、币种、网关开销、租户、run、route 和成功质量标签；价格变更版本化。
  3. 容量以峰值和突发窗口计算：到达率、请求 token 分布、输出长度、服务时间、并发上限、provider token/min 与 request/min 配额共同约束。
  4. 治理层下发每 run/租户的质量、成本、时间预算；网关执行预估、预留、记账与超限信号，编排决定缩短输出、换策略、等待、询问或终止。
  5. 扩缩结合队列等待、并发和 token throughput；保留 headroom 并做 load shedding，不能等 CPU 高才处理外部模型配额耗尽。

常见坑

  • 只看平均延迟：长尾和队头阻塞被平均值隐藏。
  • 以请求数估容量：100 个分类请求与 100 个长上下文生成的资源完全不同。
  • 只优化单价：更弱模型引发重试、人工接管或业务失败，单位成功成本更高。
  • 自动扩容万能论：provider 账户配额、区域容量和冷启动可能无法靠本地副本解决。

理论追问

请先脱稿回答，再展开参考答案核对边界、证据和取舍。

  1. 为什么只看每千 Token 单价无法判断模型成本？

    查看参考答案

    总成本还取决于输入膨胀、输出长度、重试、候选数、缓存命中、工具循环和人工接管。应按成功任务计算 cost per successful task，并结合质量和失败切片，而不是只比较标价。

  2. RPM、TPM 和并发限制分别影响什么？

    查看参考答案

    RPM 限制请求频率，TPM 限制 token 吞吐，并发限制同时占用的连接或推理槽。长上下文可能在 RPM 很低时耗尽 TPM；容量规划必须同时建模到达率、token 分布和服务时间。

  3. 队列积压时为什么不能无限等待？

    查看参考答案

    等待会违反端到端 SLO，并让上游继续堆积、状态过期和用户重试。应设置优先级、队列上限、deadline、背压与拒绝策略；接近超时时选择降级、异步通知或失败，而不是晚到的无用结果。

  4. 如何在成本下降和质量底线之间做路由？

    查看参考答案

    先定义任务切片的最低质量与安全约束，再在合格模型中优化成本和延迟。用离线评测和线上灰度验证路由，关键任务保留强模型或升级条件；不能让预算策略覆盖 hard fail。

  5. 容量测试应模拟哪些 Agent 特有负载？

    查看参考答案

    除平均请求外，要模拟长上下文、工具等待后的突发恢复、多步重试、并行 worker、流式连接和供应商故障转移。报告 p95/p99 延迟、排队、token 吞吐、降级率、成本和成功任务率。

面试表达

如何做 Agent 模型成本和容量治理？

我把预算拆为每 run、租户和模型池的 token、金额、deadline 与并发，并记录到模型 revision、价格版本和路由原因。延迟拆成排队、TTFT、生成和后处理，按 p95/p99 看长尾。容量同时估算到达率、服务时间、token 分布和 provider RPM/TPM 配额，用队列与饱和度扩缩并预留 headroom；最终看单位成功成本，而不只看每 token 单价。
