01 模型抽象、路由与能力匹配

学习目标

  • 设计不泄露厂商细节的统一请求、流式事件、错误与用量契约。
  • 建立版本化模型能力目录，区分声明能力、实测能力与允许能力。
  • 按硬约束过滤、软目标排序、粘性选择和策略版本形成可解释路由。
  • 坚持推理内核只经模型网关访问模型。

核心概念

链路总览与现有讲解

模型抽象、路由与能力匹配属于模型网关与基础设施：输入是能力请求、模型参数、tenant、deadline、预算和运行依赖，输出是规范化响应、RoutingDecision、资源遥测或类型化基础设施错误。学习时先沿现有图、表或主题章节理解链路，再用后续 stage 检查每个中间产物的不变量。

契约面统一字段不可抹平的差异
请求messages、tools、response_schema、deadline、tenant、trace上下文窗口、模态、工具并行、区域支持
响应content、tool_intents、finish_reason、usage、model_revision结构化输出保证、拒绝语义、token 计数
流式start、delta、tool_delta、usage、stop、error首 token 时机、取消和用量结算
错误auth、quota、rate_limit、timeout、invalid、provider可否重试、retry-after、账单影响

核心环节：从输入到可验证输出

1. 能力画像

能力画像必须先满足能力、风险和兼容硬约束，再比较质量、延迟与成本；名称或模型自报不能作为路由证据。

职责与 I/O
platform-owner 在“能力画像”环节接收 能力请求、模型参数、tenant、deadline、预算和运行依赖，产出 规范化响应、RoutingDecision、资源遥测或类型化基础设施错误；输出必须能由下游按版本消费。
工作机制
能力画像裁剪不可用候选，版本化策略对剩余候选评分，adapter 规范输入输出并记录 route_reason 与替代项。本环节把 能力请求、模型参数、tenant、deadline、预算和运行依赖 中与“能力画像”相关的事实转成 规范化响应、RoutingDecision、资源遥测或类型化基础设施错误 的一部分。
必须成立的约束
结构化输出、工具支持、上下文、区域和数据策略等硬约束不可被低价覆盖；fallback 必须兼容安全与语义。本章共同约束是：模型只能经网关访问，租户 / 凭证 / 网络与执行环境必须隔离；违反时必须产生可区分的拒绝、未知或升级结果。
难点与取舍
动态路由提升效率却增加不可复现性；先 shadow 再 canary，用关键切片与 SLO 决定放量和回退。本层总体取舍是 质量、延迟、成本、容量和隔离强度之间的多目标权衡，参数必须引用版本化数据、SLO 或风险基线。
边界与注意事项
网关选择模型与规范协议，不选择业务工具、不拥有推理意图，也不授予工具执行权限。模型网关与基础设施 的总边界是：不决定业务意图或在线授权，也不替代工具动作契约；错误会继续污染 推理结果质量、端到端 SLO、成本和沙箱安全证据。

5. 路由 Trace

路由 Trace必须能把结论追溯到同一请求的输入、版本、决定、动作和验收事件，而不是依赖不可检索的日志文本。

职责与 I/O
platform-owner 在“路由 Trace”环节接收 能力请求、模型参数、tenant、deadline、预算和运行依赖，产出 规范化响应、RoutingDecision、资源遥测或类型化基础设施错误；输出必须能由下游按版本消费。
工作机制
correlation_id / trace_id 串联 span 与结构化事件，引用保留原始证据位置、policy / model / tool 版本和 owner。本环节把 能力请求、模型参数、tenant、deadline、预算和运行依赖 中与“路由 Trace”相关的事实转成 规范化响应、RoutingDecision、资源遥测或类型化基础设施错误 的一部分。
必须成立的约束
时间、因果顺序、采样与脱敏策略必须明确；高基数字段不能无界进入 Metric，敏感原文不能直接写 Log。本章共同约束是：模型只能经网关访问，租户 / 凭证 / 网络与执行环境必须隔离；违反时必须产生可区分的拒绝、未知或升级结果。

全量遥测利于归因却增加成本与隐私面；采样按风险与失败类型分层，关键审计事件不得随机丢弃。本层总体取舍是 质量、延迟、成本、容量和隔离强度之间的多目标权衡，参数必须引用版本化数据、SLO 或风险基线。

难点与取舍

冲突目标参数依据决策边界
质量、延迟、成本、容量和隔离强度之间的多目标权衡使用版本化回放、失败切片、SLO 分位数和风险等级，不把教学值当生产默认。platform-owner只调整本层机制；跨层变化需要对应 owner 与独立验收。
自动化收益 vs. 失败放大比较开启 / 关闭“能力画像”与“路由 Trace”的消融结果、人工负载和恢复时间。证据不足返回 pending / unknown / escalate，不能靠更长 Prompt 隐藏边界。

失败传播与 first divergence

  1. 上游输入：先确认 能力请求、模型参数、tenant、deadline、预算和运行依赖 的身份、作用域、时间与版本是否一致；输入错误不应在本层被“修正”为新事实。
  2. 首个偏离：按 能力画像 → 路由策略 → 模型请求 → Provider Adapter → 路由 Trace 比较预期与实际中间产物，第一个不满足概念不变量的 stage 才是本章候选根因。
  3. 下游影响：本层错误会改变 推理结果质量、端到端 SLO、成本和沙箱安全证据；最终症状可能在更晚阶段出现，但不能从症状直接倒推 Prompt。
  4. 归属边界：platform-owner修复本层契约或机制；若 first divergence 位于相邻层，保留证据并交给相应 owner，不跨层调参。

工程要点

  1. 由推理内核提交 capability_requirements、风险级别、允许区域、最大成本和 deadline；它不指定厂商密钥或绕过网关直连。
  2. 能力目录记录 model_alias、provider/revision、context、modalities、tool/schema 能力、区域、价格、SLO、评测版本和准入状态；厂商宣称只作输入，线上准入依赖实测。
  3. 路由分两阶段：硬过滤数据驻留、能力、合规、上下文与预算；再按质量下限、预计延迟、边际成本和健康度排序。
  4. 同一 run 对需要一致性的步骤使用 route key 与 model revision 粘性；变更策略保存 route_policy_version，便于重放与归因。
  5. 响应必须附 endpoint、model revision、route reason、token、cache、latency 与 trace id；密钥只由网关的 secret broker 获取。

常见坑

  • 最低公共分母：统一接口把 schema、并行工具或取消能力全部抹掉，导致高级能力不可表达。
  • 推理节点直连厂商：路由、审计、预算和密钥边界立刻碎片化。
  • 只看模型名：同名 alias 背后 revision 变化会破坏可重复性。
  • 把路由当业务决策：网关选择满足契约的模型端点，不决定 Agent 下一步行动。

理论追问

请先脱稿回答，再展开参考答案核对边界、证据和取舍。

  1. 为什么推理内核不应直接绑定模型厂商 API？

    查看参考答案

    直接绑定会把请求格式、鉴权、重试、配额和模型能力散落到业务代码，难以切换和审计。模型网关提供统一契约与版本化路由，推理内核只表达任务能力、质量和预算需求。

  2. 模型路由中的硬约束和软偏好有什么区别？

    查看参考答案

    数据驻留、上下文长度、模态、工具能力和合规属于硬约束，不满足就不能入选；成本、延迟和历史质量是合法候选间的软优化。先过滤硬约束再评分，避免便宜模型绕过安全要求。

  3. 为什么不能完全让另一个 LLM 决定模型路由？

    查看参考答案

    语义路由可辅助理解任务，但会引入额外成本、漂移和不可解释错误。硬约束、租户策略与预算必须确定性执行；LLM 只在允许候选中提供信号，并用离线评测验证收益。

  4. 模型网关与在线治理的安全职责如何分工？

    查看参考答案

    网关负责模型访问边界、供应商配置、数据区域、配额和调用证据；治理层负责用户、内容、资源和动作策略。网关可执行治理下发的模型限制，但不能替代 Prompt Injection、工具授权或业务审批。

  5. 一次路由决策需要留下哪些 Trace？

    查看参考答案

    记录请求能力标签、硬约束、候选集合、排除原因、最终模型与版本、路由策略版本、fallback 链、估算和实际 token、延迟、成本及错误。这样才能区分路由错误、供应商故障和推理质量问题。

面试表达

怎样设计多模型网关？

我让推理内核只提交厂商无关的请求、能力要求、风险、区域、预算与 deadline。网关先用能力、合规和上下文做硬过滤，再按经评测的质量、健康度、延迟和成本排序，并对同一 run 保持必要粘性。响应统一规范化，保留模型 revision、路由原因和用量证据；厂商 SDK 与密钥只存在于 adapter 和 secret broker 边界。

日常实践关键点

核心判断

本章的日常工作不是记住名词，而是把“模型抽象、路由与能力匹配”变成可执行控制环：先识别触发信号，再用版本化证据定位首个偏离点，由 platform-owner 执行动作，最后用成功、拒绝、降级三类结果验收。推荐顺序是 维护能力画像 → 制定路由策略 → 规范化模型请求 → 隔离 Provider Adapter → 保留路由 Trace；任何参数都必须从真实 SLO、风险基线或失败样本推导。

    诊断顺序
    诊断“维护能力画像”时先冻结 model_profile、route_reason、预算、deadline、tenant、运行环境与依赖健康度。比较候选能力、风险、延迟和成本约束，确认选择依据来自声明能力而非名称或模型自报。
    执行动作
    落实“维护能力画像”时，先裁剪不满足硬约束的候选，再对剩余候选打分，并把选择理由与替代项写入决策记录。由 platform-owner 记录执行前后的版本、决定与接管条件。
    验证方法
    验收“维护能力画像”时，比较路由 Trace、p50/p95/p99、单位任务成本、容量水位与隔离负测，并执行受控降级演练；同时比较正确、拒绝和降级结果，阈值只从真实基线与风险容忍度推导。
    止损与取舍
    路由信号不全时回到确定性默认路径；候选不可替代时明确失败而不是随意降级；固定到已知稳定模型或环境，关闭高风险 fallback / cache，按租户和依赖缩小影响面。对“维护能力画像”先保证可恢复和可审计，再优化自动化率。
    面试记忆点
    围绕“维护能力画像”准备一条“症状—首个偏离—动作—证据—止损”的真实或个人实验 Trace，并明确自己只承担的 platform-owner 职责。

排障顺序

  1. 冻结请求、策略、模型 / 工具、数据和部署版本，避免边复现边变化。
  2. 从最早可验证输入开始，逐段对照契约、决策、动作与验收事件；在首个偏离点停止跨层调参。
  3. 由 platform-owner 判断修复、降级、回滚或转人工，并记录未验证假设。
贯穿案例｜企业差旅 Agent：员工请求预订超预算航班时，只把制度条款当知识证据，把员工靠窗偏好当记忆，把预订动作交给工具层；本章只处理 模型抽象、路由与能力匹配 的控制与证据。若发现路由能力不匹配、级联重试、尾延迟恶化、预算失控、跨租户泄漏或依赖抖动，先保留 trace_id 与版本，再在本层首个偏离点修复，避免把问题推给 Prompt。

表达时必须标记经历属于生产实战、个人实验或方案设计，只主张自己实际完成的职责，并用 Trace、评测或变更记录举证。企业差旅案例、示例版本和指标均为教学占位；没有验证线上规模、事故协作或长期效果时要主动说明。

常见问法

在生产或可复现实验中，你如何把“维护能力画像”和“制定路由策略”落到企业差旅 Agent 的 模型抽象、路由与能力匹配 链路？
