01 模型抽象、路由与能力匹配

学习目标

  • 设计不泄露厂商细节的统一请求、流式事件、错误与用量契约。
  • 建立版本化模型能力目录，区分声明能力、实测能力与允许能力。
  • 按硬约束过滤、软目标排序、粘性选择和策略版本形成可解释路由。
  • 坚持推理内核只经模型网关访问模型。

核心概念

契约面统一字段不可抹平的差异
请求messages、tools、response_schema、deadline、tenant、trace上下文窗口、模态、工具并行、区域支持
响应content、tool_intents、finish_reason、usage、model_revision结构化输出保证、拒绝语义、token 计数
流式start、delta、tool_delta、usage、stop、error首 token 时机、取消和用量结算
错误auth、quota、rate_limit、timeout、invalid、provider可否重试、retry-after、账单影响

工程要点

  1. 由推理内核提交 capability_requirements、风险级别、允许区域、最大成本和 deadline；它不指定厂商密钥或绕过网关直连。
  2. 能力目录记录 model_alias、provider/revision、context、modalities、tool/schema 能力、区域、价格、SLO、评测版本和准入状态；厂商宣称只作输入，线上准入依赖实测。
  3. 路由分两阶段：硬过滤数据驻留、能力、合规、上下文与预算；再按质量下限、预计延迟、边际成本和健康度排序。
  4. 同一 run 对需要一致性的步骤使用 route key 与 model revision 粘性；变更策略保存 route_policy_version，便于重放与归因。
  5. 响应必须附 endpoint、model revision、route reason、token、cache、latency 与 trace id；密钥只由网关的 secret broker 获取。

常见坑

  • 最低公共分母：统一接口把 schema、并行工具或取消能力全部抹掉，导致高级能力不可表达。
  • 推理节点直连厂商：路由、审计、预算和密钥边界立刻碎片化。
  • 只看模型名：同名 alias 背后 revision 变化会破坏可重复性。
  • 把路由当业务决策：网关选择满足契约的模型端点，不决定 Agent 下一步行动。

理论追问

请先脱稿回答，再展开参考答案核对边界、证据和取舍。

  1. 为什么推理内核不应直接绑定模型厂商 API？

    查看参考答案

    直接绑定会把请求格式、鉴权、重试、配额和模型能力散落到业务代码，难以切换和审计。模型网关提供统一契约与版本化路由，推理内核只表达任务能力、质量和预算需求。

  2. 模型路由中的硬约束和软偏好有什么区别？

    查看参考答案

    数据驻留、上下文长度、模态、工具能力和合规属于硬约束，不满足就不能入选；成本、延迟和历史质量是合法候选间的软优化。先过滤硬约束再评分，避免便宜模型绕过安全要求。

  3. 为什么不能完全让另一个 LLM 决定模型路由？

    查看参考答案

    语义路由可辅助理解任务，但会引入额外成本、漂移和不可解释错误。硬约束、租户策略与预算必须确定性执行；LLM 只在允许候选中提供信号，并用离线评测验证收益。

  4. 模型网关与在线治理的安全职责如何分工？

    查看参考答案

    网关负责模型访问边界、供应商配置、数据区域、配额和调用证据；治理层负责用户、内容、资源和动作策略。网关可执行治理下发的模型限制，但不能替代 Prompt Injection、工具授权或业务审批。

  5. 一次路由决策需要留下哪些 Trace？

    查看参考答案

    记录请求能力标签、硬约束、候选集合、排除原因、最终模型与版本、路由策略版本、fallback 链、估算和实际 token、延迟、成本及错误。这样才能区分路由错误、供应商故障和推理质量问题。

面试表达

怎样设计多模型网关？

我让推理内核只提交厂商无关的请求、能力要求、风险、区域、预算与 deadline。网关先用能力、合规和上下文做硬过滤，再按经评测的质量、健康度、延迟和成本排序，并对同一 run 保持必要粘性。响应统一规范化，保留模型 revision、路由原因和用量证据；厂商 SDK 与密钥只存在于 adapter 和 secret broker 边界。
