02 降级、限流与缓存

学习目标

  • 区分调用前拥塞、429、超时、部分流式响应与语义失败。
  • 用 deadline、retry budget、熔断和隔舱控制故障放大。
  • 设计分层公平限流以及租户/优先级保护。
  • 只在身份、策略、模型和输入语义可安全复用时缓存。

核心概念

链路总览与现有讲解

Fallback、限流与缓存属于模型网关与基础设施：输入是能力请求、模型参数、tenant、deadline、预算和运行依赖，输出是规范化响应、RoutingDecision、资源遥测或类型化基础设施错误。学习时先沿现有图、表或主题章节理解链路，再用后续 stage 检查每个中间产物的不变量。

故障默认处理不能做
本地队列拥塞 / 未派发按优先级排队、拒绝或换健康端点无限排队越过 deadline
429 / 503 且无输出遵守 retry-after、消耗重试预算、熔断每层各自重试形成风暴
流式已输出部分内容标记 incomplete，由编排决定重新生成或解释静默拼接另一模型的续写
安全/能力不满足fail closed 或显式要求放宽契约为了可用性切到不合规模型

核心环节：从输入到可验证输出

1. Fallback 矩阵

Fallback 矩阵只能把系统带回已知状态或安全终态；未知结果在查明副作用前不得被当成可重试失败。

职责与 I/O
platform-owner 在“ Fallback 矩阵”环节接收 能力请求、模型参数、tenant、deadline、预算和运行依赖，产出 规范化响应、RoutingDecision、资源遥测或类型化基础设施错误；输出必须能由下游按版本消费。
工作机制
错误先分瞬时、永久、冲突与 unknown outcome；幂等键、ledger、checkpoint 和补偿记录共同决定 retry、query、resume 或人工核对。本环节把 能力请求、模型参数、tenant、deadline、预算和运行依赖 中与“ Fallback 矩阵”相关的事实转成 规范化响应、RoutingDecision、资源遥测或类型化基础设施错误 的一部分。
必须成立的约束
尝试次数、累计 deadline、退避、副作用作用域和补偿 owner 都必须有界，幂等键需覆盖业务动作而非单次请求。本章共同约束是：模型只能经网关访问，租户 / 凭证 / 网络与执行环境必须隔离；违反时必须产生可区分的拒绝、未知或升级结果。
难点与取舍
重试提高瞬时可用性也会放大负载与副作用；预算由下游容量、用户 deadline 和动作不可逆性决定。本层总体取舍是 质量、延迟、成本、容量和隔离强度之间的多目标权衡，参数必须引用版本化数据、SLO 或风险基线。
边界与注意事项
补偿是新的业务动作而非时间回滚；网关 fallback 不应掩盖工具副作用或编排非法状态。模型网关与基础设施 的总边界是：不决定业务意图或在线授权，也不替代工具动作契约；错误会继续污染 推理结果质量、端到端 SLO、成本和沙箱安全证据。

3. 按作用域限流

按作用域限流必须绑定当前 actor、资源、动作、目的和版本，并在真正使用能力或数据的时刻仍然有效。

职责与 I/O
platform-owner 在“按作用域限流”环节接收 能力请求、模型参数、tenant、deadline、预算和运行依赖，产出 规范化响应、RoutingDecision、资源遥测或类型化基础设施错误；输出必须能由下游按版本消费。
工作机制
身份链先把渠道主体映射为内部 actor，再由策略决策与短期能力令牌约束数据、工具、文件和网络使用。本环节把 能力请求、模型参数、tenant、deadline、预算和运行依赖 中与“按作用域限流”相关的事实转成 规范化响应、RoutingDecision、资源遥测或类型化基础设施错误 的一部分。
必须成立的约束
默认最小权限与 deny-by-default；撤回、角色变更、资源版本变化必须使缓存授权和派生访问失效。本章共同约束是：模型只能经网关访问，租户 / 凭证 / 网络与执行环境必须隔离；违反时必须产生可区分的拒绝、未知或升级结果。
难点与取舍
频繁再授权更安全但增加延迟；缓存更快却可能陈旧，TTL 应由权限变化频率和风险窗口推导。本层总体取舍是 质量、延迟、成本、容量和隔离强度之间的多目标权衡，参数必须引用版本化数据、SLO 或风险基线。
边界与注意事项
身份认证不等于授权，脱敏不等于匿名化，网络隔离也不能替代业务策略判断。模型网关与基础设施 的总边界是：不决定业务意图或在线授权，也不替代工具动作契约；错误会继续污染 推理结果质量、端到端 SLO、成本和沙箱安全证据。

4. 缓存边界

缓存边界必须把资源耗尽和依赖故障限制在声明作用域内，并在上游 deadline 前返回可区分的结果。

职责与 I/O
platform-owner 在“缓存边界”环节接收 能力请求、模型参数、tenant、deadline、预算和运行依赖，产出 规范化响应、RoutingDecision、资源遥测或类型化基础设施错误；输出必须能由下游按版本消费。
工作机制
连接池、分阶段 timeout、健康探测、熔断、缓存键 / TTL 和存储契约共同管理依赖生命周期。本环节把 能力请求、模型参数、tenant、deadline、预算和运行依赖 中与“缓存边界”相关的事实转成 规范化响应、RoutingDecision、资源遥测或类型化基础设施错误 的一部分。
必须成立的约束
多个层级不能各自无界重试；缓存包含 tenant、版本与授权作用域，连接和队列有硬上限。本章共同约束是：模型只能经网关访问，租户 / 凭证 / 网络与执行环境必须隔离；违反时必须产生可区分的拒绝、未知或升级结果。
难点与取舍
复用提高吞吐也增加陈旧与争用；池大小、TTL 和 timeout 由压测分布、依赖 SLO 与容量预测确定。本层总体取舍是 质量、延迟、成本、容量和隔离强度之间的多目标权衡，参数必须引用版本化数据、SLO 或风险基线。
边界与注意事项
基础设施保证资源与故障隔离，不判断内容事实、用户权限或业务是否值得执行。模型网关与基础设施 的总边界是：不决定业务意图或在线授权，也不替代工具动作契约；错误会继续污染 推理结果质量、端到端 SLO、成本和沙箱安全证据。

失败传播与 first divergence

  1. 上游输入：先确认 能力请求、模型参数、tenant、deadline、预算和运行依赖 的身份、作用域、时间与版本是否一致；输入错误不应在本层被“修正”为新事实。
  2. 首个偏离：按 Fallback 矩阵 → 网关重试 → 按作用域限流 → 缓存边界 → 受控降级 比较预期与实际中间产物，第一个不满足概念不变量的 stage 才是本章候选根因。
  3. 下游影响：本层错误会改变 推理结果质量、端到端 SLO、成本和沙箱安全证据；最终症状可能在更晚阶段出现，但不能从症状直接倒推 Prompt。
  4. 归属边界：platform-owner修复本层契约或机制；若 first divergence 位于相邻层，保留证据并交给相应 owner，不跨层调参。

工程要点

  1. 端到端 deadline 由编排传入，网关只分配连接、排队、首 token 与生成子预算；每次尝试共享一个 retry budget，禁止 adapter 和调用方重复重试。
  2. 限流至少按 provider/account、region/model、tenant、user/run 和优先级分层；同时限制请求数、token、并发与队列长度，防止一个大请求吃光容量。
  3. 使用 bulkhead 隔离租户与模型池；熔断基于足够窗口和错误类型，half-open 仅允许少量探针，健康恢复不等于质量通过。
  4. fallback 必须重新执行能力、区域、风险、上下文和预算硬约束，并记录原端点、失败类型、候选差异和策略版本。
  5. 精确/语义缓存键包含租户与授权范围、规范化输入、系统提示/工具/schema 版本、模型 alias/revision、温度、数据/策略版本；敏感响应默认不跨用户复用，TTL 与删除策略显式。

常见坑

  • 重试叠加：编排、网关、SDK 都重试三次，最坏放大 27 倍。
  • 全局限流：单个 noisy neighbor 拖垮所有租户。
  • 缓存键只含 prompt：把其他租户、旧权限或旧工具版本的响应复用回来。
  • 错误降级：从支持结构化工具调用的模型切到纯文本模型，却仍把文本当合法动作。

理论追问

请先脱稿回答，再展开参考答案核对边界、证据和取舍。

  1. 模型 fallback 为什么不能只按供应商顺序依次尝试？

    查看参考答案

    备用模型必须满足原任务的能力、数据区域、工具 schema、质量和上下文约束；否则“可用”会变成静默质量或合规降级。Fallback 还要受总超时和成本预算约束，并记录降级原因。

  2. 限流应该按请求、用户、租户还是模型设置？

    查看参考答案

    通常需要多层：供应商/模型保护共享容量，租户和用户保证公平，单 Run 防止循环耗尽。采用 token、request、concurrency 和 cost 等不同桶，拒绝或排队策略按业务优先级决定。

  3. 模型响应缓存为什么有隐私风险？

    查看参考答案

    缓存键或内容若未包含租户、权限、Prompt 版本和敏感数据边界，可能跨用户复用；原文缓存也扩大 PII 保留。应限制可缓存任务、隔离命名空间、脱敏、TTL、加密，并明确哪些请求永不缓存。

  4. 语义缓存命中为什么仍需质量门禁？

    查看参考答案

    语义相似不代表约束、主体、时间和权限相同。命中前要比较结构化关键字段、知识版本和安全作用域，高风险或时效性问题禁用；命中结果仍需输出治理和引用校验。

  5. Circuit Breaker 打开后，系统应如何降级？

    查看参考答案

    停止向故障依赖继续放大流量，按任务契约切到合格备用模型、排队、返回部分结果或安全失败。编排接收明确的 degraded Observation；不能把网关降级伪装成正常结果。

面试表达

模型服务波动时如何保证可用性？

我先让请求携带端到端 deadline 和统一重试预算，在网关按账号、模型、租户和优先级做 token/并发隔舱。失败按是否派发、是否已有部分输出和是否可安全重试分类；fallback 每次都重验能力、合规与预算。缓存只有在租户、授权、模型、prompt、工具和数据版本都纳入键时才启用。不能满足安全或能力下限就显式失败，而不是伪装成功。

日常实践关键点

核心判断

本章的日常工作不是记住名词，而是把“Fallback、限流与缓存”变成可执行控制环：先识别触发信号，再用版本化证据定位首个偏离点，由 platform-owner 执行动作，最后用成功、拒绝、降级三类结果验收。推荐顺序是 维护 Fallback 矩阵 → 限制网关重试 → 按作用域限流 → 保护缓存边界 → 演练受控降级；任何参数都必须从真实 SLO、风险基线或失败样本推导。

深度实践卡

  • 1. 维护 Fallback 矩阵
    触发信号
    当企业差旅 Agent 的“Fallback、限流与缓存”链路出现路由能力不匹配、级联重试、尾延迟恶化、预算失控、跨租户泄漏或依赖抖动，或变更直接触及“维护 Fallback 矩阵”的输入、策略或 owner 时，必须启动这项检查。
    诊断顺序
    诊断“维护 Fallback 矩阵”时先冻结 model_profile、route_reason、预算、deadline、tenant、运行环境与依赖健康度。先区分可重试瞬时错误、确定性拒绝和结果未知，再检查副作用是否已有完成凭证。
    执行动作
    落实“维护 Fallback 矩阵”时，以业务动作和作用域生成幂等键，设置尝试次数、累计 deadline、退避和补偿 owner。由 platform-owner 记录执行前后的版本、决定与接管条件。
    验证方法
    验收“维护 Fallback 矩阵”时，比较路由 Trace、p50/p95/p99、单位任务成本、容量水位与隔离负测，并执行受控降级演练；同时比较正确、拒绝和降级结果，阈值只从真实基线与风险容忍度推导。
    止损与取舍
    结果未知时先查 ledger 而不是盲目重放；无法补偿的动作必须停在人工验收点；固定到已知稳定模型或环境，关闭高风险 fallback / cache，按租户和依赖缩小影响面。对“维护 Fallback 矩阵”先保证可恢复和可审计，再优化自动化率。
    面试记忆点
    围绕“维护 Fallback 矩阵”准备一条“症状—首个偏离—动作—证据—止损”的真实或个人实验 Trace，并明确自己只承担的 platform-owner 职责。

    执行动作
    落实“保护缓存边界”时，为每个依赖定义 deadline、连接池上限、缓存键 / TTL、健康判定和熔断降级，避免多个层级独立重试。由 platform-owner 记录执行前后的版本、决定与接管条件。

    依赖不稳定时关闭非必要缓存或新连接策略，切到只读 / 排队 / 拒绝路径并保护核心租户容量；固定到已知稳定模型或环境，关闭高风险 fallback / cache，按租户和依赖缩小影响面。对“保护缓存边界”先保证可恢复和可审计，再优化自动化率。

排障顺序

  1. 冻结请求、策略、模型 / 工具、数据和部署版本，避免边复现边变化。
  2. 从最早可验证输入开始，逐段对照契约、决策、动作与验收事件；在首个偏离点停止跨层调参。
  3. 由 platform-owner 判断修复、降级、回滚或转人工，并记录未验证假设。
贯穿案例｜企业差旅 Agent：员工请求预订超预算航班时，只把制度条款当知识证据，把员工靠窗偏好当记忆，把预订动作交给工具层；本章只处理 Fallback、限流与缓存 的控制与证据。若发现路由能力不匹配、级联重试、尾延迟恶化、预算失控、跨租户泄漏或依赖抖动，先保留 trace_id 与版本，再在本层首个偏离点修复，避免把问题推给 Prompt。

应留证据

最小证据集应能回答谁在何时依据哪个版本、对什么输入做了什么决定，以及结果如何被验证。敏感值使用教学占位或受控引用，不把日志全文复制进面试材料。

实践型面试要点

表达时必须标记经历属于生产实战、个人实验或方案设计，只主张自己实际完成的职责，并用 Trace、评测或变更记录举证。企业差旅案例、示例版本和指标均为教学占位；没有验证线上规模、事故协作或长期效果时要主动说明。

实践面试 1

关联实践：维护 Fallback 矩阵、限制网关重试

常见问法

在生产或可复现实验中，你如何把“维护 Fallback 矩阵”和“限制网关重试”落到企业差旅 Agent 的 Fallback、限流与缓存 链路？

60–90 秒回答骨架

先交代 Fallback、限流与缓存 的业务风险和 platform-owner 边界；再说触发信号与首个偏离点；随后说明契约、动作和参数依据；最后给出验证、止损和复盘结果。回答要同时覆盖正常路径与拒绝 / 降级路径。

可信证据

围绕“维护 Fallback 矩阵 / 限制网关重试”展示脱敏后的 trace_id、model_profile、route_reason、预算、deadline、tenant、运行环境与依赖健康度、前后对照和失败切片；个人实验还应给出固定输入、复现步骤和观察结果，不虚构线上规模。

高频追问

为什么问题属于 模型网关与基础设施 而不是相邻层？参数如何从 SLO 或风险基线推导？结果未知时怎样避免扩大副作用？platform-owner 不在线时，谁依据什么证据接管？

边界与取舍

“维护 Fallback 矩阵 / 限制网关重试”会增加契约、存储或人工成本，只有错误成本与复现收益覆盖这些成本时才扩大自动化。未处理过该类生产事故时，应明确这是方案设计或个人实验结论。
