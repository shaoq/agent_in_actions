02 降级、限流与缓存

学习目标

  • 区分调用前拥塞、429、超时、部分流式响应与语义失败。
  • 用 deadline、retry budget、熔断和隔舱控制故障放大。
  • 设计分层公平限流以及租户/优先级保护。
  • 只在身份、策略、模型和输入语义可安全复用时缓存。

核心概念

故障默认处理不能做
本地队列拥塞 / 未派发按优先级排队、拒绝或换健康端点无限排队越过 deadline
429 / 503 且无输出遵守 retry-after、消耗重试预算、熔断每层各自重试形成风暴
流式已输出部分内容标记 incomplete，由编排决定重新生成或解释静默拼接另一模型的续写
安全/能力不满足fail closed 或显式要求放宽契约为了可用性切到不合规模型

工程要点

  1. 端到端 deadline 由编排传入，网关只分配连接、排队、首 token 与生成子预算；每次尝试共享一个 retry budget，禁止 adapter 和调用方重复重试。
  2. 限流至少按 provider/account、region/model、tenant、user/run 和优先级分层；同时限制请求数、token、并发与队列长度，防止一个大请求吃光容量。
  3. 使用 bulkhead 隔离租户与模型池；熔断基于足够窗口和错误类型，half-open 仅允许少量探针，健康恢复不等于质量通过。
  4. fallback 必须重新执行能力、区域、风险、上下文和预算硬约束，并记录原端点、失败类型、候选差异和策略版本。
  5. 精确/语义缓存键包含租户与授权范围、规范化输入、系统提示/工具/schema 版本、模型 alias/revision、温度、数据/策略版本；敏感响应默认不跨用户复用，TTL 与删除策略显式。

常见坑

  • 重试叠加：编排、网关、SDK 都重试三次，最坏放大 27 倍。
  • 全局限流：单个 noisy neighbor 拖垮所有租户。
  • 缓存键只含 prompt：把其他租户、旧权限或旧工具版本的响应复用回来。
  • 错误降级：从支持结构化工具调用的模型切到纯文本模型，却仍把文本当合法动作。

面试表达

模型服务波动时如何保证可用性？

我先让请求携带端到端 deadline 和统一重试预算，在网关按账号、模型、租户和优先级做 token/并发隔舱。失败按是否派发、是否已有部分输出和是否可安全重试分类；fallback 每次都重验能力、合规与预算。缓存只有在租户、授权、模型、prompt、工具和数据版本都纳入键时才启用。不能满足安全或能力下限就显式失败，而不是伪装成功。
