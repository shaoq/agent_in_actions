05 沙箱、租户与执行隔离

学习目标

  • 区分治理授权、工具契约与基础设施实际隔离的责任。
  • 按威胁和风险选择进程、容器、microVM 或远程受管执行边界。
  • 设计贯穿身份、计算、文件、网络、secret、数据、资源与销毁的租户隔离。
  • 用负向测试、审计和残留检查证明隔离，而不是只写配置。

核心概念

链路总览与现有讲解

沙箱、租户与执行隔离属于模型网关与基础设施：输入是能力请求、模型参数、tenant、deadline、预算和运行依赖，输出是规范化响应、RoutingDecision、资源遥测或类型化基础设施错误。学习时先沿现有图、表或主题章节理解链路，再用后续 stage 检查每个中间产物的不变量。

隔离维度最低控制验证证据
身份 / 租户tenant-scoped workload identity，无共享万能账号跨租户访问负测、策略与 audit principal
计算非特权、capability/syscall 限制、独立命名空间逃逸用例、镜像/内核版本、阻断日志
文件 / 数据只读根、独立临时卷、输入最小挂载路径穿越/符号链接负测、销毁后残留检查
网络默认拒绝、域名/IP 双校验、出站代理私网、metadata、DNS rebinding 与重定向负测
secret按动作短期注入、不可导出、立即撤销日志/prompt/文件扫描和撤销测试
资源 / 生命周期CPU、内存、进程、磁盘、网络、时间配额耗尽测试、超时终止、进程树清理与销毁 receipt

核心环节：从输入到可验证输出

1. 租户边界

租户边界必须绑定当前 actor、资源、动作、目的和版本，并在真正使用能力或数据的时刻仍然有效。

职责与 I/O
platform-owner 在“租户边界”环节接收 能力请求、模型参数、tenant、deadline、预算和运行依赖，产出 规范化响应、RoutingDecision、资源遥测或类型化基础设施错误；输出必须能由下游按版本消费。
工作机制
身份链先把渠道主体映射为内部 actor，再由策略决策与短期能力令牌约束数据、工具、文件和网络使用。本环节把 能力请求、模型参数、tenant、deadline、预算和运行依赖 中与“租户边界”相关的事实转成 规范化响应、RoutingDecision、资源遥测或类型化基础设施错误 的一部分。
必须成立的约束
默认最小权限与 deny-by-default；撤回、角色变更、资源版本变化必须使缓存授权和派生访问失效。本章共同约束是：模型只能经网关访问，租户 / 凭证 / 网络与执行环境必须隔离；违反时必须产生可区分的拒绝、未知或升级结果。
难点与取舍
频繁再授权更安全但增加延迟；缓存更快却可能陈旧，TTL 应由权限变化频率和风险窗口推导。本层总体取舍是 质量、延迟、成本、容量和隔离强度之间的多目标权衡，参数必须引用版本化数据、SLO 或风险基线。
边界与注意事项
身份认证不等于授权，脱敏不等于匿名化，网络隔离也不能替代业务策略判断。模型网关与基础设施 的总边界是：不决定业务意图或在线授权，也不替代工具动作契约；错误会继续污染 推理结果质量、端到端 SLO、成本和沙箱安全证据。

难点与取舍

冲突目标参数依据决策边界

失败传播与 first divergence

  1. 上游输入：先确认 能力请求、模型参数、tenant、deadline、预算和运行依赖 的身份、作用域、时间与版本是否一致；输入错误不应在本层被“修正”为新事实。
  2. 首个偏离：按 租户边界 → 凭证 → 沙箱策略 → 网络 Egress → 隔离负测 比较预期与实际中间产物，第一个不满足概念不变量的 stage 才是本章候选根因。
  3. 下游影响：本层错误会改变 推理结果质量、端到端 SLO、成本和沙箱安全证据；最终症状可能在更晚阶段出现，但不能从症状直接倒推 Prompt。
  4. 归属边界：platform-owner修复本层契约或机制；若 first divergence 位于相邻层，保留证据并交给相应 owner，不跨层调参。

工程要点

  1. 治理层输出带 subject、tenant、动作摘要、资源、策略版本和有效期的决定；工具层规范化参数；基础设施据此创建隔离环境，但不能扩大权限。
  2. 按威胁选边界：受信任轻任务可用受限进程，第三方依赖或不可信代码使用强化容器/microVM；浏览器和代码执行默认视为高风险。
  3. 每次 execution_id 绑定 tenant、run/step、image digest、policy、network profile、mount、secret lease、quota 与 deadline；禁止复用其他租户的暖实例，除非隔离证明成立。
  4. 防逃逸之外还防数据越界和 DoS：cgroup/配额、进程数、输出大小、磁盘、网络带宽和 wall-clock timeout；终止整个进程树。
  5. 执行后先规范化 stdout/stderr/artifact/effect receipt，扫描敏感信息，再撤销 secret、卸载卷、销毁环境并记录残留检查；未知结果返回编排对账。

常见坑

  • “用了 Docker 就安全”：特权容器、宿主 socket、共享卷和任意出站仍可突破边界。
  • 只做身份隔离：对象存储前缀、缓存键、队列、日志和指标标签仍可能跨租户。
  • 审批替代沙箱：用户同意执行不代表代码不会越权、泄密或耗尽宿主。
  • 复用暖池无清理：文件、进程、浏览器 profile 或连接可把前一租户数据留给下一租户。

理论追问

请先脱稿回答，再展开参考答案核对边界、证据和取舍。

  1. 沙箱隔离与工具授权为什么缺一不可？

    查看参考答案

    授权限制主体能请求什么，沙箱限制获准代码实际能影响什么。只有授权，工具漏洞仍可越界；只有沙箱，用户仍可能在允许边界内访问不属于自己的数据。两者与审计共同构成纵深防御。

  2. 多租户隔离应覆盖哪些层面？

    查看参考答案

    覆盖身份与策略、进程/容器、文件系统、存储键空间、缓存、队列、网络、凭据和 Trace。仅在数据库加 tenant_id 不足以防止共享缓存、临时文件或日志串扰。

  3. 沙箱网络出口怎样兼顾可用性和安全？

    查看参考答案

    按工具和任务声明目标白名单，使用受审计代理注入短期凭据，限制协议、端口、带宽和 DNS；默认拒绝未知目标。对动态浏览任务可使用隔离浏览器和内容下载扫描，而不是开放主机网络。

  4. CPU、内存、时间和进程数限制如何影响 Agent 恢复？

    查看参考答案

    资源超限应产生类型化 Observation，而不是只杀进程。编排据此决定缩小任务、降级、重试或失败；限制值按工具风险和历史分布设置，防止单 Run 抢占租户或平台容量。

  5. 怀疑沙箱逃逸时，第一步应该做什么？

    查看参考答案

    立即隔离相关 worker、撤销凭据、阻断镜像和网络路径，保全审计与运行证据并评估租户影响。不要在可疑环境中继续调查；随后轮换密钥、重建可信节点、修复并运行逃逸回归。

面试表达

Agent 的代码工具如何做安全隔离？

我先把责任拆开：治理层按主体、租户、动作和风险授权，工具层提交规范化请求，基础设施才创建真实沙箱。每个 execution 绑定镜像 digest、网络/挂载/secret/资源策略和 deadline；不可信代码使用强化容器或 microVM，默认拒绝网络、只读根、临时卷、短期凭据并限制 CPU/内存/进程/磁盘。完成后撤权销毁，并用跨租户、SSRF、路径穿越、逃逸和资源耗尽负测证明边界。

日常实践关键点

核心判断

本章的日常工作不是记住名词，而是把“沙箱、租户与执行隔离”变成可执行控制环：先识别触发信号，再用版本化证据定位首个偏离点，由 platform-owner 执行动作，最后用成功、拒绝、降级三类结果验收。推荐顺序是 隔离租户边界 → 隔离凭证 → 版本化沙箱策略 → 控制网络 Egress → 运行隔离负测；任何参数都必须从真实 SLO、风险基线或失败样本推导。

深度实践卡

  • 1. 隔离租户边界
    触发信号
    当企业差旅 Agent 的“沙箱、租户与执行隔离”链路出现路由能力不匹配、级联重试、尾延迟恶化、预算失控、跨租户泄漏或依赖抖动，或变更直接触及“隔离租户边界”的输入、策略或 owner 时，必须启动这项检查。
    诊断顺序
    诊断“隔离租户边界”时先冻结 model_profile、route_reason、预算、deadline、tenant、运行环境与依赖健康度。重点确认决策主体、资源、动作、目的和策略版本是否在“使用点”仍一致，不能只看入口鉴权。
    执行动作
    落实“隔离租户边界”时，把允许、拒绝和需审批做成类型化策略结果，默认拒绝缺字段或版本不明的请求。由 platform-owner 记录执行前后的版本、决定与接管条件。
    验证方法
    验收“隔离租户边界”时，比较路由 Trace、p50/p95/p99、单位任务成本、容量水位与隔离负测，并执行受控降级演练；同时比较正确、拒绝和降级结果，阈值只从真实基线与风险容忍度推导。
    止损与取舍
    一旦主体或目的无法证明，立即撤销能力令牌并转人工确认，不用事后过滤替代前置授权；固定到已知稳定模型或环境，关闭高风险 fallback / cache，按租户和依赖缩小影响面。对“隔离租户边界”先保证可恢复和可审计，再优化自动化率。
    面试记忆点
    围绕“隔离租户边界”准备一条“症状—首个偏离—动作—证据—止损”的真实或个人实验 Trace，并明确自己只承担的 platform-owner 职责。

排障顺序

  1. 冻结请求、策略、模型 / 工具、数据和部署版本，避免边复现边变化。
  2. 从最早可验证输入开始，逐段对照契约、决策、动作与验收事件；在首个偏离点停止跨层调参。
  3. 由 platform-owner 判断修复、降级、回滚或转人工，并记录未验证假设。
贯穿案例｜企业差旅 Agent：员工请求预订超预算航班时，只把制度条款当知识证据，把员工靠窗偏好当记忆，把预订动作交给工具层；本章只处理 沙箱、租户与执行隔离 的控制与证据。若发现路由能力不匹配、级联重试、尾延迟恶化、预算失控、跨租户泄漏或依赖抖动，先保留 trace_id 与版本，再在本层首个偏离点修复，避免把问题推给 Prompt。

应留证据

最小证据集应能回答谁在何时依据哪个版本、对什么输入做了什么决定，以及结果如何被验证。敏感值使用教学占位或受控引用，不把日志全文复制进面试材料。

代表性工程产物｜artifact-gw-05-isolation-test
case: enterprise-travel-agent
component: model-gateway-infrastructure
unit: 05
owner: platform-owner
input_version: teaching-v1

first_divergence: stage-name
evidence_refs: [trace-demo, eval-slice-demo]

以上字段是学习用数据形状，不是生产默认值。实际产物还应记录时间、租户 / 主体作用域、变更来源和保留策略，并能从 evidence_refs 回放本章结论。

实践型面试要点

表达时必须标记经历属于生产实战、个人实验或方案设计，只主张自己实际完成的职责，并用 Trace、评测或变更记录举证。企业差旅案例、示例版本和指标均为教学占位；没有验证线上规模、事故协作或长期效果时要主动说明。

实践面试 1

关联实践：隔离租户边界、隔离凭证

常见问法

在生产或可复现实验中，你如何把“隔离租户边界”和“隔离凭证”落到企业差旅 Agent 的 沙箱、租户与执行隔离 链路？

60–90 秒回答骨架

先交代 沙箱、租户与执行隔离 的业务风险和 platform-owner 边界；再说触发信号与首个偏离点；随后说明契约、动作和参数依据；最后给出验证、止损和复盘结果。回答要同时覆盖正常路径与拒绝 / 降级路径。

可信证据

围绕“隔离租户边界 / 隔离凭证”展示脱敏后的 trace_id、model_profile、route_reason、预算、deadline、tenant、运行环境与依赖健康度、前后对照和失败切片；个人实验还应给出固定输入、复现步骤和观察结果，不虚构线上规模。

高频追问

为什么问题属于 模型网关与基础设施 而不是相邻层？参数如何从 SLO 或风险基线推导？结果未知时怎样避免扩大副作用？platform-owner 不在线时，谁依据什么证据接管？

边界与取舍

“隔离租户边界 / 隔离凭证”会增加契约、存储或人工成本，只有错误成本与复现收益覆盖这些成本时才扩大自动化。未处理过该类生产事故时，应明确这是方案设计或个人实验结论。
