05 沙箱、租户与执行隔离

学习目标

  • 区分治理授权、工具契约与基础设施实际隔离的责任。
  • 按威胁和风险选择进程、容器、microVM 或远程受管执行边界。
  • 设计贯穿身份、计算、文件、网络、secret、数据、资源与销毁的租户隔离。
  • 用负向测试、审计和残留检查证明隔离，而不是只写配置。

核心概念

隔离维度最低控制验证证据
身份 / 租户tenant-scoped workload identity，无共享万能账号跨租户访问负测、策略与 audit principal
计算非特权、capability/syscall 限制、独立命名空间逃逸用例、镜像/内核版本、阻断日志
文件 / 数据只读根、独立临时卷、输入最小挂载路径穿越/符号链接负测、销毁后残留检查
网络默认拒绝、域名/IP 双校验、出站代理私网、metadata、DNS rebinding 与重定向负测
secret按动作短期注入、不可导出、立即撤销日志/prompt/文件扫描和撤销测试
资源 / 生命周期CPU、内存、进程、磁盘、网络、时间配额耗尽测试、超时终止、进程树清理与销毁 receipt

工程要点

  1. 治理层输出带 subject、tenant、动作摘要、资源、策略版本和有效期的决定；工具层规范化参数；基础设施据此创建隔离环境，但不能扩大权限。
  2. 按威胁选边界：受信任轻任务可用受限进程，第三方依赖或不可信代码使用强化容器/microVM；浏览器和代码执行默认视为高风险。
  3. 每次 execution_id 绑定 tenant、run/step、image digest、policy、network profile、mount、secret lease、quota 与 deadline；禁止复用其他租户的暖实例，除非隔离证明成立。
  4. 防逃逸之外还防数据越界和 DoS：cgroup/配额、进程数、输出大小、磁盘、网络带宽和 wall-clock timeout；终止整个进程树。
  5. 执行后先规范化 stdout/stderr/artifact/effect receipt，扫描敏感信息，再撤销 secret、卸载卷、销毁环境并记录残留检查；未知结果返回编排对账。

常见坑

  • “用了 Docker 就安全”：特权容器、宿主 socket、共享卷和任意出站仍可突破边界。
  • 只做身份隔离：对象存储前缀、缓存键、队列、日志和指标标签仍可能跨租户。
  • 审批替代沙箱：用户同意执行不代表代码不会越权、泄密或耗尽宿主。
  • 复用暖池无清理：文件、进程、浏览器 profile 或连接可把前一租户数据留给下一租户。

面试表达

Agent 的代码工具如何做安全隔离？

我先把责任拆开：治理层按主体、租户、动作和风险授权，工具层提交规范化请求，基础设施才创建真实沙箱。每个 execution 绑定镜像 digest、网络/挂载/secret/资源策略和 deadline；不可信代码使用强化容器或 microVM，默认拒绝网络、只读根、临时卷、短期凭据并限制 CPU/内存/进程/磁盘。完成后撤权销毁，并用跨租户、SSRF、路径穿越、逃逸和资源耗尽负测证明边界。
