UNIT.02

架构位置与边界

职责边界 · 推理内核提出动作意图，编排负责调用时机与状态，治理层决定能否执行，基础设施提供沙箱和短期凭据；工具层拥有动作适配、结果规范化和副作用语义。
契约面组件级定义
输入tool name、版本化参数、actor、risk、idempotency key
输出result / typed error、外部引用、耗时、副作用状态和审计证据
上游编排提交通过结构校验的动作意图，治理附带授权决定
下游API、数据库、文件、浏览器或隔离运行时

UNIT.03

模式与取舍

MODE 1

函数调用

结构化 API 且边界清晰；可靠、易审计。

MODE 2

MCP

跨客户端发现资源与工具；统一协议但仍需本地授权。

MODE 3

Computer Use

缺少 API 的视觉界面；覆盖广但脆弱、风险高。

MODE 4

异步作业

长任务或外部排队；需状态查询、回调和幂等。

UNIT.05

生产工程约束

  • 工具描述、参数、错误码和版本必须可测试。
  • 高风险动作在副作用前授权，恢复后重新校验。
  • 区分未提交、已提交、未知和补偿完成状态。
  • 凭据最小权限、短生命周期，不进入模型上下文或日志。
UNIT.06

面试表达

30 秒回答

工具层把结构化意图变成真实环境动作。生产重点不是函数能否调用，而是契约、授权、隔离、幂等、副作用状态和 Observation 证据；编排控制生命周期，治理决定权限，基础设施限制执行边界。

3 分钟回答结构

从 tool schema 和版本开始，讲清授权与沙箱，再描述 submitted、unknown、committed 的副作用生命周期；比较 Function Calling、MCP 和 Computer Use；最后说明如何用调用 ID、幂等键、对账和类型化错误定位失败。

表达检查 · 不要只报框架或产品名；必须讲清输入输出、owner、证据、失败归因和为什么选择该模式。
