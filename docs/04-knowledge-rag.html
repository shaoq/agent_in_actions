UNIT.02

架构位置与边界

职责边界 · RAG 的事实来源是外部资料；记忆的来源是 Agent 与用户的经历。检索层返回证据包，不保证最终答案正确；推理负责使用证据，治理负责访问控制，评测负责离线质量验证。
契约面组件级定义
输入查询、actor、租户、权限、时效与过滤条件
输出片段、来源、版本、分数、权限标签和证据不足标记
离线链资料摄取 → 清洗切分 → embedding/index → 版本发布
在线链查询理解 → 召回 → 过滤/重排 → Grounding → 引用

UNIT.03

模式与取舍

MODE 1

稠密检索

语义改写丰富；召回强但精确实体可能弱。

MODE 2

稀疏检索

关键词、编号与专名；可解释但语义泛化弱。

MODE 3

混合 + 重排

生产默认候选；质量更高但增加延迟成本。

MODE 4

结构化查询

数据库或知识图谱；约束强但建模成本高。

UNIT.04

失败归因矩阵

表现定位阶段常见原因优先修复
目标证据未召回retrieval recall切分、查询或索引覆盖不足检查 gold evidence recall 与过滤前候选
召回正确却排序靠后rerank特征或模型不适配测 MRR/nDCG 并分析 hard negatives
有证据但答案无引用grounding提示或输出契约未绑定来源强制 claim-citation 对齐和不足拒答
越权检索access filter召回后才做权限过滤在检索前绑定主体和资源标签
UNIT.05

生产工程约束

  • 索引、文档版本和权限变更必须可追溯。
  • 先定义证据单位，再选择 chunk 大小和 overlap。
  • 评测同时覆盖检索、重排、Grounding 和端到端任务。
  • 证据冲突、过期或不足必须显式返回，不靠模型掩盖。
UNIT.06

面试表达

30 秒回答

知识/RAG 层负责从外部资料检索证据，与记录 Agent 经历的记忆系统不同。它要管理摄取、索引、召回、重排、权限、时效和引用，并把证据不足作为一等结果交给推理与编排。

3 分钟回答结构

先画离线和在线双流水线，再讲混合检索与重排；强调访问控制前置、来源版本和引用对齐；用 recall、MRR、faithfulness 和任务成功率分层诊断；最后说明为什么错误不能都归咎于模型。

表达检查 · 不要只报框架或产品名；必须讲清输入输出、owner、证据、失败归因和为什么选择该模式。
