学习目标

完成本单元后,你应能:

  2. 解释 goal、environment、state、action、observation、autonomy 和 feedback loop。
  3. 从控制权而不是产品名称判断一个系统是否具有 Agent 特征。
  4. 将核心概念映射到项目九层架构。
  5. 识别常见错误架构和“为了 Agent 而 Agent”的设计。

1. 为什么 Agent 很难用一句话定义

“Agent”没有唯一、稳定且被所有团队共同接受的定义。不同资料强调的维度不同:

  • 有的强调代表用户自主完成目标。
  • 有的强调 LLM 动态控制流程和工具选择。
  • 有的把预定义 Workflow 也归入广义 agentic system。
  • 有的要求跨多轮观察环境、修正行为和判断完成。

因此,工程讨论不能只问“它是不是 Agent”,还要问:

  1. 谁控制下一步流程?
  2. 系统能否改变外部环境?
  3. 是否根据观察反馈调整后续行为?
  4. 是否维护跨步骤状态?
  5. 自治范围和停止条件是什么?

通用事实:Agent 通常围绕目标执行多步任务,能够根据状态和环境反馈选择下一步动作。

项目参考架构:本项目使用九层架构拆解生产级 Agent 的职责,不把单次模型调用视为完整 Agent。

实现偏好:优先使用能解决问题的最简单方案;只有固定流程不足时才增加动态自治。

2. 四类系统的边界

2.1 Chatbot

Chatbot 描述的是交互形态,不必然描述内部架构。它可能只是消息输入和文本回复,也可能背后连接复杂 Agent。

典型特征:

  • 主要交互是对话。
  • 通常由用户发起每一轮。
  • 不一定具有工具、状态或自主任务推进。

“聊天界面”不能证明系统是 Agent。

2.2 LLM Application

LLM Application 是使用 LLM 完成某项能力的应用,例如摘要、分类、抽取、改写或单轮问答。

典型特征:

  • 应用代码决定何时调用模型。
  • 模型通常只完成一个明确变换。
  • 没有由模型动态控制的多步执行循环。

所有 Agent 都是 LLM Application 的一种可能形态,但不是所有 LLM Application 都是 Agent。

2.3 Workflow

Workflow 将任务拆为预定义步骤。步骤可以包含模型、规则、人工和工具,但路径主要由代码或流程配置决定。

典型特征:

  • 节点和主要边在设计时已知。
  • 易于预测、审计和测试。
  • 适合稳定流程、合规要求和明确例外分支。

Anthropic 的工程分类把“预定义代码路径”称为 workflow,把由 LLM 动态控制过程和工具使用的系统称为 agent。这是一种有用的工程划分,但不是行业标准的唯一口径。

2.4 Agent

Agent 接收目标和约束,维护运行状态,基于当前上下文决定下一步,通过动作与环境交互,读取观察结果并继续、完成、失败或请求人工帮助。

典型特征:

  • LLM 或决策模块拥有一定流程控制权。
  • 所需步骤难以在设计时全部写死。
  • 动作结果会改变下一步决策。
  • 有明确自治边界、停止条件和治理约束。

3. 这不是四个互斥产品类别

现实系统通常位于连续谱上:

确定性代码
  -> 单次 LLM 调用
  -> 固定 LLM Workflow
  -> 局部动态路由
  -> 单 Agent 自治循环
  -> Multi-Agent 协作

同一个产品也可能混合多种方式。例如客服系统可以:

  • 用固定规则校验身份。
  • 用分类模型路由意图。
  • 用 RAG 回答知识问题。
  • 在复杂工单中启动 Agent 动态调用内部系统。
  • 在退款动作前暂停并等待人工批准。

正确问题不是“全系统是不是 Agent”,而是“哪一段流程需要多少动态控制权”。

4. Agent 的核心原语

4.1 Goal:目标

Goal 描述希望达到的结果,不是单条模型指令。一个可工程化的目标还需要:

  • 成功条件:怎样证明完成?
  • 约束:哪些动作禁止或需要审批?
  • 预算:最多允许多少时间、步骤和成本?
  • 失败出口:何时停止、降级或转人工?

目标模糊时,Agent 可能执行很多合理动作,却没有完成用户真正需要的结果。

4.2 Environment:环境

Environment 是 Agent 所观察和影响的外部世界,例如文件系统、浏览器、数据库、工单系统、代码仓库或用户本身。

环境提供 ground truth。模型认为“退款成功”不是事实,只有支付系统返回并被校验的结果才是观察证据。

4.3 State:状态

State 是做出下一步决策所需的显式信息。至少可能包含:

  • 原始目标和约束。
  • 当前步骤和运行状态。
  • 已执行动作与观察。
  • 计划、待办和中间产物。
  • 权限、预算和审批状态。
  • 失败原因与恢复位置。

状态不等于完整聊天历史,也不等于长期记忆。不同生命周期的信息应分别管理。

4.4 Action:动作

Action 是 Agent 影响环境或取得新信息的操作。动作可以是:

  • 读取:搜索、查询数据库、读取文件。
  • 写入:更新工单、发送消息、修改代码。
  • 交互:向用户澄清、请求批准或转人工。
  • 委派:将子任务交给其他执行单元。

动作必须有明确契约、权限、风险等级和可观察结果。高风险副作用不能只依赖模型判断。

4.5 Observation:观察

Observation 是动作执行后的环境反馈,例如查询结果、错误码、测试输出、用户回复或审批结果。

观察必须回写状态。只把工具结果打印到日志,却不进入下一轮上下文,Agent 就无法根据环境修正行为。

4.6 Autonomy:自治

Autonomy 不是开或关,而是权限范围内的控制程度。可以从以下维度衡量:

  • 是否自主选择下一步?
  • 是否自主选择工具和参数?
  • 是否能改变计划?
  • 是否能执行有副作用动作?
  • 是否能决定任务完成?
  • 何时必须请示人类?

更高自治通常带来更强适应性,同时增加成本、不可预测性和复合错误风险。

4.7 Feedback Loop:反馈循环

反馈循环是 Agent 区别于单次生成的关键机制:

  1. 根据当前状态作出决策。
  2. 执行动作。
  3. 从环境取得观察。
  4. 更新状态和进度判断。
  5. 继续、完成、失败或升级。

没有可靠 observation 的循环容易变成模型对自身输出的反复加工,而不是对真实任务进展的修正。

5. 核心原语如何映射到九层架构

核心原语 主要承载层 说明
Goal / user constraints 交互与接入层 + 编排层 接入层接收和规范化,编排层在整个 run 中维护
Decide 推理内核 根据上下文提出结构化的下一步判断
Control flow 编排控制层 推进状态、路由、调用能力并决定停止
Action 工具 / 动作层 声明可执行能力,实际执行受权限和沙箱约束
Run state 编排控制层 服务于一次任务的步骤、观察、预算和状态
Experience 记忆系统 沉淀 Agent 经历、用户偏好和可复用经验
External facts 知识 / 检索 从外部资料取得事实依据
Guardrails / trace 治理与可观测层 在线校验、审计、预算和过程证据
Quality feedback 评测层 离线消费轨迹,判断效果、退化和安全风险
Model capability 模型网关 + 基础设施 提供路由、fallback、缓存、运行时和执行隔离

6. 场景分类

场景 更合适的起点 判断理由
固定格式合同字段抽取 单次 LLM Application 输入输出明确,无动态多步路径
发票审批:校验、金额阈值、人工签字 Workflow 路径稳定且需要强审计
开放式竞品研究并根据新线索继续搜索 Agent 步骤难预先确定,观察会改变计划
FAQ 对话但不访问外部系统 Chatbot / RAG Application 有对话形态,未必需要自治动作
代码修复并运行测试直到通过 Agent + deterministic gates 动态探索适合 Agent,测试和提交门禁应确定化
大额退款 Workflow + 局部 Agent + HITL 理解和调查可动态,最终副作用需权限与人工控制

分类不是追求标签正确,而是判断哪里需要模型控制,哪里必须保持确定性。

7. 常见误区

误区 1:调用了模型 API 就是 Agent

单次摘要或分类没有多步控制、环境动作和反馈循环。它是 LLM Application,但不因此成为 Agent。

误区 2:用了 Agent 框架就拥有完整架构

框架提供执行抽象,不会自动解决权限、状态语义、评测、trace、成本和业务完成条件。框架名不是架构说明。

误区 3:所有业务流程都应该 Agent 化

固定规则、强合规或可完全枚举的流程通常更适合确定性代码。动态自治会增加延迟、成本和不可预测性。

误区 4:Memory 和 RAG 都是向量库,所以是一层

存储技术相同不代表语义相同。RAG 获取外部既有知识;Memory 沉淀 Agent 经历或用户上下文,写入来源和治理要求不同。

误区 5:自治意味着不需要人类

生产自治必须有权限边界、预算、停止条件和高风险动作审批。人在回路是可靠性机制,不是 Agent 不够智能的补丁。

误区 6:模型输出 finish 就说明任务成功

模型只能声明完成意图。业务成功应由外部状态、测试、规则或用户确认等可验证证据判断。

误区 7:Agent 越多能力越强

Multi-Agent 增加通信、状态同步、冲突、成本和归因难度。单 Agent 问题若来自工具差或目标模糊,增加 Agent 不会自动解决。

8. 反例评审

考虑下面的设计:

“我们做了一个生产级 Agent。前端把用户问题发送给 LLM,LLM 可以调用数据库查询工具。所有聊天和公司文档都写入同一个向量库。模型认为完成时直接返回结果。调用失败就重试三次。”

至少存在以下问题:

  1. 没有明确运行状态、预算和终止分类。
  2. 用户身份、工具权限和审计边界缺失。
  3. 对话经历与公司知识混存,写入来源和污染治理不清。
  4. “模型认为完成”缺少业务验收证据。
  5. 所有失败统一重试,可能重复副作用。
  6. 没有 trace、离线评测和回归门禁。
  7. 无法判断它是单轮 tool calling 还是可持续任务循环。

9. 如何判断是否需要 Agent

先回答五个问题:

  1. 任务步骤是否无法在设计时可靠枚举?
  2. 环境反馈是否经常改变下一步?
  3. 结果是否存在可验证的成功条件?
  4. 失败是否可恢复、可停止或可转人工?
  5. 动态自治带来的收益是否覆盖延迟、成本和风险?

若前两项均为否,优先考虑普通代码或 Workflow。若缺少可验证成功条件和停止机制,即使任务复杂也不应直接提高自治程度。

10. 无代码练习

  1. 将“文档摘要、客服问答、自动退款、竞品研究、代码修复”分别放到连续谱上,说明依据。
  2. 画出一次带工具 Agent 请求的控制流、数据流和反馈流。
  3. 为“企业工单助手”标出哪些环节确定化、哪些环节可由模型动态决定。
  4. 评审一个“所有状态都放 messages”的设计,指出生命周期和恢复问题。
  5. 用 3 分钟解释为什么 Agent 不等于“LLM + tool calling”。
