单元 2:状态与生命周期

前置与目标

前置:职责与边界。

完成后能够:

  1. 区分 run state、session state、checkpoint 和 long-term memory。
  2. 说明一次 run 从创建到终止的状态变化。
  3. 分析暂停、恢复、重放、版本迁移和重复副作用。
  4. 设计“最小但足够”的显式状态。

链路总览与现有讲解

状态与生命周期属于编排控制层：输入是用户事件、Decision、PolicyOutcome、Observation 与当前 RunState，输出是合法 transition、checkpoint、委派和验收状态。学习时先沿现有图、表或主题章节理解链路，再用后续 stage 检查每个中间产物的不变量。

1. 为什么状态是编排的中心

决策只有结合状态才有意义。同一个工具错误在第一次出现时可能值得重试,连续第三次出现时可能应转人工。若系统只保留最新 prompt,就无法判断历史动作、预算、权限和恢复位置。

通用事实:可恢复的多步系统必须持有足以重建控制决策的状态。

项目参考架构:一次任务的 run state 归编排层;跨会话经历归记忆系统;外部知识归 RAG。

实现偏好:状态存储可以是内存、数据库、事件日志或框架 checkpointer,但语义应先于存储选型。

2. 四类容易混淆的状态

类型 生命周期 典型内容 主要用途 不应承载
Run state 一次任务运行 goal、step、plan、observations、budget、status 推进和恢复任务 无关历史偏好
Session state 一段交互会话 message history、channel、identity、locale 保持对话连续 所有长期经验
Checkpoint run 在特定时刻的快照 state values、next node、metadata、version 暂停、恢复、回放 业务真相的唯一副本
Long-term memory 跨 run / 跨 session 用户偏好、经验证事实、可复用经验 个性化和经验复用 未经治理的所有 trace

Checkpoint 是保存状态的机制或产物;Memory 是跨任务复用信息的业务语义。某个框架可以用 checkpoint 提供线程记忆,但架构讨论仍需区分“恢复控制流”和“长期记住什么”。

3. Run state 最小模型

RunState
├── identity: run_id, user_id, tenant_id
├── goal: task, success_criteria, constraints
├── control: status, current_step, next_action, retry_count
├── context: observations, selected_knowledge, working_notes
├── governance: permissions, approvals, risk_level
├── budget: max_steps, deadline, token/cost limit
├── result: final_output, acceptance_status
└── diagnosis: error_type, trace_id, state_version

不是所有系统都需要全部字段。选择标准是:

  1. 它是否影响下一步决策?
  2. 它是否需要在失败后恢复?
  3. 它是否用于审计或业务验收?
  4. 它是否有独立的安全和保留期限?

4. 生命周期

关键原则:

  • completed、failed、cancelled、exhausted 是不同终态。
  • waiting 不是失败,它表示需要外部输入。
  • 模型产生 finish 只是完成候选,业务 acceptance 通过后才是 completed。
  • 恢复不等于从头重跑,必须知道哪些副作用已经发生。

5. Checkpoint 边界

Checkpoint 至少需要回答:

  • 保存的是哪个 run / thread?
  • 发生在动作前还是动作后?
  • 对应什么代码和 state schema 版本?
  • 下一个节点或动作是什么?
  • 哪些结果已提交,哪些仍是 pending?
  • 恢复时使用原权限还是重新授权?

LangGraph 的官方文档把 checkpoint 描述为图状态在特定 super-step 的快照,可支持 HITL、time travel 和 fault tolerance。这是一个框架实现示例,不是所有系统都必须按 super-step 保存。

6. 暂停与恢复时序

恢复前至少重新检查:

  1. 用户和租户身份是否仍有效。
  2. 权限和审批是否过期。
  3. 外部资源状态是否已变化。
  4. action 是否曾经成功但响应丢失。
  5. 当前代码能否读取旧 state version。

7. 重放与副作用

Checkpoint 可以恢复计算,但不能自动撤销外部副作用。典型危险:

  1. Agent 发送邮件成功。
  2. 写 checkpoint 前进程崩溃。
  3. 恢复后再次发送同一邮件。

应对方式:

  • 为动作生成稳定 idempotency key。
  • 记录 action intent、submitted、committed 等状态。
  • 恢复前查询外部系统的真实执行状态。
  • 将不可逆动作放在清晰 checkpoint 边界后。
  • 对无法幂等的动作要求人工确认或补偿流程。

8. 状态版本迁移

长任务可能跨越代码发布。若旧 checkpoint 的字段或状态名已变化,恢复会失败或产生错误转移。

建议记录:

  • state_schema_version
  • workflow_version
  • tool_contract_version
  • created_at / expires_at

迁移策略可以是:

  • 向后兼容读取。
  • 显式 state migration。
  • 旧版本运行时继续处理旧 run。
  • 无法安全迁移时终止并转人工。

9. 失败矩阵

失败 证据 风险 修复
状态字段语义混乱 同一字段混合消息和业务状态 路由不可预测 按生命周期拆分 schema
checkpoint 太少 崩溃后重复大量工作 成本和副作用增加 在稳定边界保存
checkpoint 太频繁 写放大、延迟升高 成本和吞吐下降 按风险与恢复代价选择粒度
恢复不重新授权 旧审批继续生效 越权动作 恢复时重验身份与策略
schema 漂移 旧 run 无法加载 长任务中断 版本字段和迁移策略
observation 未写回 重复同一动作 循环不收敛 原子记录结果与状态转移

10. 案例:可暂停退款调查

调查阶段允许 Agent 查询订单、物流和聊天记录。退款动作前进入 waiting,保存:

  • 用户和订单标识。
  • 调查证据和推理结论摘要。
  • 建议退款金额与政策条款。
  • 已调用动作和当前预算。
  • 待审批动作的 idempotency key。
  • policy 和 state schema 版本。

人工批准后不能直接复用旧判断;需要重新查询订单是否已退款、审批是否仍有效、金额是否变化,再执行受控动作。

核心环节：从输入到可验证输出

1. Run State

Run State是生产者与消费者共同承认的版本化解释边界；结构合法不代表语义、授权或业务结果成立。

职责与 I/O
orchestration-owner 在“ Run State”环节接收 用户事件、Decision、PolicyOutcome、Observation 与当前 RunState，产出 合法 transition、checkpoint、委派和验收状态；输出必须能由下游按版本消费。
工作机制
显式 schema 定义字段、枚举、状态转移和错误联合，消费者先做语法校验，再按领域不变量做语义校验。本环节把 用户事件、Decision、PolicyOutcome、Observation 与当前 RunState 中与“ Run State”相关的事实转成 合法 transition、checkpoint、委派和验收状态 的一部分。
必须成立的约束
必填字段、版本兼容、非法状态、未知枚举和错误语义必须确定；不能用自由文本默认值掩盖缺失事实。本章共同约束是：状态转移必须合法、可恢复且预算有界；违反时必须产生可区分的拒绝、未知或升级结果。
难点与取舍
严格契约提升可恢复性但增加演进成本；兼容窗口由调用方升级节奏和旧数据回放覆盖决定。本层总体取舍是 确定性与适应性、自动化率与可恢复性，参数必须引用版本化数据、SLO 或风险基线。
边界与注意事项
契约约束信息形状和解释，不授予权限、不选择业务策略，也不证明下游验收成功。编排控制层 的总边界是：不生成业务推理，也不伪造工具成功或治理许可；错误会继续污染 推理上下文、工具动作、用户状态和离线评测轨迹。

3. Checkpoint 就绪性

Checkpoint 就绪性只有在主体、来源、时间、目的、置信和可撤回状态明确时才是可用记忆，不等于聊天文本或外部知识副本。

职责与 I/O
orchestration-owner 在“ Checkpoint 就绪性”环节接收 用户事件、Decision、PolicyOutcome、Observation 与当前 RunState，产出 合法 transition、checkpoint、委派和验收状态；输出必须能由下游按版本消费。
工作机制
经历先成为 candidate，经验证与同意晋升；读取按作用域、时效、类型和置信排序，冲突保留版本而非覆盖历史。本环节把 用户事件、Decision、PolicyOutcome、Observation 与当前 RunState 中与“ Checkpoint 就绪性”相关的事实转成 合法 transition、checkpoint、委派和验收状态 的一部分。
必须成立的约束
MemoryRecord 必须带 subject / tenant、provenance、consent、policy_version 与 TTL，删除要传播到索引、缓存和派生摘要。本章共同约束是：状态转移必须合法、可恢复且预算有界；违反时必须产生可区分的拒绝、未知或升级结果。
难点与取舍
更高召回能增强个性化，也会增加旧事实、投毒和隐私风险；读取与保留参数由时间切片评测和用户控制要求决定。本层总体取舍是 确定性与适应性、自动化率与可恢复性，参数必须引用版本化数据、SLO 或风险基线。
边界与注意事项
RunState 管当前执行，Session 管交互连续性，RAG 管外部资料；Memory 只沉淀经治理的经历。编排控制层 的总边界是：不生成业务推理，也不伪造工具成功或治理许可；错误会继续污染 推理上下文、工具动作、用户状态和离线评测轨迹。

4. 恢复前重新授权

恢复前重新授权必须绑定当前 actor、资源、动作、目的和版本，并在真正使用能力或数据的时刻仍然有效。

职责与 I/O
orchestration-owner 在“恢复前重新授权”环节接收 用户事件、Decision、PolicyOutcome、Observation 与当前 RunState，产出 合法 transition、checkpoint、委派和验收状态；输出必须能由下游按版本消费。
工作机制
身份链先把渠道主体映射为内部 actor，再由策略决策与短期能力令牌约束数据、工具、文件和网络使用。本环节把 用户事件、Decision、PolicyOutcome、Observation 与当前 RunState 中与“恢复前重新授权”相关的事实转成 合法 transition、checkpoint、委派和验收状态 的一部分。
必须成立的约束
默认最小权限与 deny-by-default；撤回、角色变更、资源版本变化必须使缓存授权和派生访问失效。本章共同约束是：状态转移必须合法、可恢复且预算有界；违反时必须产生可区分的拒绝、未知或升级结果。
难点与取舍
频繁再授权更安全但增加延迟；缓存更快却可能陈旧，TTL 应由权限变化频率和风险窗口推导。本层总体取舍是 确定性与适应性、自动化率与可恢复性，参数必须引用版本化数据、SLO 或风险基线。
边界与注意事项
身份认证不等于授权，脱敏不等于匿名化，网络隔离也不能替代业务策略判断。编排控制层 的总边界是：不生成业务推理，也不伪造工具成功或治理许可；错误会继续污染 推理上下文、工具动作、用户状态和离线评测轨迹。

难点与取舍

冲突目标参数依据决策边界
确定性与适应性、自动化率与可恢复性使用版本化回放、失败切片、SLO 分位数和风险等级，不把教学值当生产默认。orchestration-owner只调整本层机制；跨层变化需要对应 owner 与独立验收。

失败传播与 first divergence

  1. 上游输入：先确认 用户事件、Decision、PolicyOutcome、Observation 与当前 RunState 的身份、作用域、时间与版本是否一致；输入错误不应在本层被“修正”为新事实。
  2. 首个偏离：按 Run State → 非法状态转移 → Checkpoint 就绪性 → 恢复前重新授权 → 状态版本迁移 比较预期与实际中间产物，第一个不满足概念不变量的 stage 才是本章候选根因。
  3. 下游影响：本层错误会改变 推理上下文、工具动作、用户状态和离线评测轨迹；最终症状可能在更晚阶段出现，但不能从症状直接倒推 Prompt。
  4. 归属边界：orchestration-owner修复本层契约或机制；若 first divergence 位于相邻层，保留证据并交给相应 owner，不跨层调参。

概念—实践—证据回扣

核心区回答“是什么、为何成立、怎样传播”，实践区回答“何时触发、如何处置、怎样验收”。每个 stage 与 practice 使用稳定锚点双向连接：

  • Run State的不变量或约束失效时，进入对应日常实践；用本页工程产物、Trace / 评测切片和拒绝 / 降级结果证明是否恢复。
  • 非法状态转移的不变量或约束失效时，进入对应日常实践；用本页工程产物、Trace / 评测切片和拒绝 / 降级结果证明是否恢复。
  • Checkpoint 就绪性的不变量或约束失效时，进入对应日常实践；用本页工程产物、Trace / 评测切片和拒绝 / 降级结果证明是否恢复。
  • 恢复前重新授权的不变量或约束失效时，进入对应日常实践；用本页工程产物、Trace / 评测切片和拒绝 / 降级结果证明是否恢复。
  • 状态版本迁移的不变量或约束失效时，进入对应日常实践；用本页工程产物、Trace / 评测切片和拒绝 / 降级结果证明是否恢复。

11. 常见误区

  1. 把 messages 当成所有状态:消息不适合承载审批、预算和动作提交状态。
  2. 把 checkpoint 当长期记忆:恢复位置和用户偏好有不同生命周期。
  3. 认为恢复就是重跑:有副作用的节点不能盲目重放。
  4. 只保存模型输出:工具真实结果、授权和业务验收同样关键。

理论追问

请先脱稿回答，再展开参考答案核对边界、证据和取舍。

  1. 什么状态必须持久化,什么可以重算?

    查看参考答案

    目标、外部副作用、授权与审批证据、已消费预算、checkpoint、幂等键和不可重复 Observation 必须持久化。可由版本化输入确定重建的派生摘要或缓存可以重算；判断依据是丢失后是否会重复副作用、改变权限语义或无法恢复。

  2. checkpoint 应该在动作前还是动作后?

    查看参考答案

    两侧通常都需要：动作前保存 intent、授权快照和幂等键，动作后保存 result、effect ID 与新 State。只在动作后保存会在崩溃时不知道是否提交，只在动作前保存又无法证明结果；关键是用状态机表达 submitted、unknown、committed。

  3. 为什么 checkpoint 不能解决所有幂等问题?

    查看参考答案

    checkpoint 只能记录本地进度，无法原子地覆盖外部系统副作用。若请求超时，checkpoint 可能停在动作前而外部已成功。仍需稳定幂等键、外部查询接口、effect receipt 或事务 outbox 来识别重复和 effect unknown。

  4. session history 和 run observations 能否共用一个表?

    查看参考答案

    物理上可以共用存储，但必须有明确类型、主体、生命周期、权限和索引边界。Session history 服务交互连续性，Run Observation 服务当前执行与恢复；若不做逻辑隔离，会把展示消息当执行事实，也会让保留和删除策略相互冲突。

  5. 恢复旧 run 为什么要重新授权?

    查看参考答案

    授权依赖主体、资源、动作、参数、策略版本和时间，这些条件在暂停期间可能变化。恢复时必须重查权限、审批有效期、业务对象版本和动作参数，不能把旧的 allow 决策当永久通行证。

  6. 如何在恢复能力和存储成本之间取舍?

    查看参考答案

    按风险和重算代价分层：高风险副作用与人工审批保存细粒度事件和 checkpoint，低风险纯推理可只存关键状态并允许重算。结合 TTL、增量快照、压缩和归档，同时用恢复 SLO 与每 run 存储成本验证策略。

  7. code deploy 时长任务如何迁移?

    查看参考答案

    状态必须带 schema、workflow 和代码版本。部署时让旧 run 继续由兼容 worker 消费，或通过显式迁移器升级 State；不能让新代码直接解释未知旧状态。迁移前要做回放测试，并保留失败回滚与人工接管路径。

日常实践关键点

核心判断

本章的日常工作不是记住名词，而是把“状态与生命周期”变成可执行控制环：先识别触发信号，再用版本化证据定位首个偏离点，由 orchestration-owner 执行动作，最后用成功、拒绝、降级三类结果验收。推荐顺序是 版本化 Run State → 拒绝非法状态转移 → 验证 Checkpoint 就绪性 → 恢复前重新授权 → 演练状态版本迁移；任何参数都必须从真实 SLO、风险基线或失败样本推导。

关联实践：恢复前重新授权、演练状态版本迁移

常见问法

在生产或可复现实验中，你如何把“恢复前重新授权”和“演练状态版本迁移”落到企业差旅 Agent 的 状态与生命周期 链路？

60–90 秒回答骨架

先交代 状态与生命周期 的业务风险和 orchestration-owner 边界；再说触发信号与首个偏离点；随后说明契约、动作和参数依据；最后给出验证、止损和复盘结果。回答要同时覆盖正常路径与拒绝 / 降级路径。

可信证据

围绕“恢复前重新授权 / 演练状态版本迁移”展示脱敏后的 trace_id、run_state、transition、checkpoint、预算、授权快照、工具结果与完成条件、前后对照和失败切片；个人实验还应给出固定输入、复现步骤和观察结果，不虚构线上规模。

高频追问

为什么问题属于 编排控制层 而不是相邻层？参数如何从 SLO 或风险基线推导？结果未知时怎样避免扩大副作用？orchestration-owner 不在线时，谁依据什么证据接管？

边界与取舍

“恢复前重新授权 / 演练状态版本迁移”会增加契约、存储或人工成本，只有错误成本与复现收益覆盖这些成本时才扩大自动化。未处理过该类生产事故时，应明确这是方案设计或个人实验结论。

本单元产出

  • 画出 run/session/checkpoint/memory 四类生命周期。
  • 为退款案例设计最小 RunState。
  • 写出一次恢复前的检查清单。
