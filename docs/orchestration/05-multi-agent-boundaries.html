单元 5:Multi-Agent 的适用边界

前置与目标

前置:可靠性与恢复。

完成后能够:

  1. 先定位单 Agent 瓶颈,再判断是否需要 Multi-Agent。
  2. 比较 orchestrator-workers、并行分工和层级协作。
  3. 说明共享状态、冲突仲裁、权限和评测归因问题。
  4. 用收益指标证明升级值得。

链路总览与现有讲解

Multi-Agent 边界属于编排控制层：输入是用户事件、Decision、PolicyOutcome、Observation 与当前 RunState，输出是合法 transition、checkpoint、委派和验收状态。学习时先沿现有图、表或主题章节理解链路，再用后续 stage 检查每个中间产物的不变量。

1. Multi-Agent 不是默认高级形态

多个 Agent 可以增加并行性、上下文隔离和专业分工,也会增加:

  • 任务分解错误。
  • 通信和 token 成本。
  • 重复搜索和重复动作。
  • 共享状态冲突。
  • 端到端失败归因难度。
  • 权限扩散和审计复杂度。

通用事实:增加独立决策主体会增加协调状态和失败组合。

项目参考架构:Multi-Agent 协作仍由编排控制层统一表达,治理和 trace 必须贯穿每个子 run。

实现偏好:先建立单 Agent 基线,只有可测收益超过协调成本时才升级。

2. 先诊断单 Agent 为什么失败

表现 可能根因 Multi-Agent 是否优先解法
工具经常调用错 工具描述或 schema 差 否,先修工具接口
长上下文丢失重点 上下文管理或任务过大 可能,也可先摘要/分段
独立子任务可并行 单 Agent 串行造成延迟 可能
需要不同专业权限 单一身份权限过宽 可能,但治理要先设计
目标模糊 需求没有澄清 否,更多 Agent 会放大分歧
评测标准不清 不知道何为成功 否,无法证明升级收益

3. 常见协作模式

Orchestrator-Workers

主 Agent 动态拆任务,多个 worker 独立执行,主 Agent 汇总。

适合:

  • 子任务数量和内容依输入而变化。
  • 子任务有较强独立性。
  • 主 Agent 能定义清晰交付契约。

风险:错误分解、遗漏、重复工作、汇总损失证据。

固定并行分工

代码预先定义多个角色或检查维度并行执行,最后确定性聚合或投票。

适合:

  • 分工稳定。
  • 需要多视角审查。
  • 结果可以独立评分或聚合。

它更接近 parallel workflow,不一定属于动态 Multi-Agent。

层级协作

上层负责目标和资源,下层继续委派。适合大型任务,但深度增加后状态、预算和责任链迅速复杂。

Peer-to-Peer / Handoff

Agent 根据能力把控制权移交给另一个 Agent。适合职责清晰的专业接力,需要明确当前 owner、handoff contract 和循环检测。

4. Orchestrator-Workers 时序

每个 worker 应有独立 sub-run ID、预算、工具权限和输出契约。Lead 不能只收到自然语言总结而丢失来源和失败状态。

5. 上下文隔离与共享状态

应隔离的信息

  • 与子任务无关的大量历史。
  • 其他 worker 的中间噪声。
  • 不属于该角色的敏感数据和工具权限。
  • 主 Agent 的未验证假设。

应共享的信息

  • 顶层目标和成功条件。
  • 子任务范围、依赖和预算。
  • 已确认事实与来源。
  • 公共约束和禁止动作。
  • 输出 schema 和截止条件。

共享状态不能让多个 Agent 任意覆盖同一字段。常见策略:

  • 单写者、多读者。
  • 每个 worker 写独立 namespace。
  • append-only 事件与最终聚合。
  • 乐观版本检查或显式锁。
  • 冲突由确定性规则或指定 owner 仲裁。

6. 权限边界

不要让所有子 Agent 继承 Lead 的全部权限。应按任务最小授权:

  • Research worker 只读搜索和资料。
  • Writer 只能写草稿区。
  • Reviewer 只读产物并输出建议。
  • Committer 在人工批准后才有提交权限。

委派需要记录“谁代表谁、在什么范围、有效多久”。子 Agent 进一步委派时必须受深度、预算和权限约束。

7. 冲突与仲裁

多个 Agent 可能产生冲突事实、计划或编辑。仲裁不能默认“再让一个模型投票”。先判断:

  1. 是否存在可查询的外部事实?
  2. 是否有确定性业务规则?
  3. 是否能按来源质量或时间排序?
  4. 是否属于需要人类价值判断的问题?

模型评审适合主观质量比较,不应覆盖数据库真实状态或权限策略。

8. 评测归因

至少评测三层:

层级 指标 关键问题
Worker 子任务成功率、证据质量、成本 worker 是否完成契约?
Coordination 分解覆盖、重复率、冲突率、等待时间 协作是否增加损耗?
End-to-end 最终成功、总延迟、总成本、安全 整体是否优于单 Agent 基线?

仅看最终答案无法判断哪个 worker 有贡献或造成失败。trace 需要保留父子 run、委派原因、输入上下文摘要和产物引用。

9. 升级决策表

问题 是 否

10. 案例:深度研究系统

Anthropic 公布的研究系统使用 lead agent 协调并行 subagents,适合开放式研究中动态拆分多个搜索方向。这个案例成立的前提包括:

  • 搜索方向相对独立,可以并行。
  • 子 Agent 的工具范围明确。
  • Lead 负责计划、继续研究和综合。
  • 系统可以评估来源与最终研究质量。

不能从这个案例推出“所有研究都应 Multi-Agent”。对于范围很小的问题,单 Agent 或固定并行搜索可能成本更低。

核心环节：从输入到可验证输出

1. 拆分收益

拆分收益成立的前提是每个控制决定只有一个最终 owner，交接不转移未声明的权限与验收责任。

职责与 I/O
orchestration-owner 在“拆分收益”环节接收 用户事件、Decision、PolicyOutcome、Observation 与当前 RunState，产出 合法 transition、checkpoint、委派和验收状态；输出必须能由下游按版本消费。
工作机制
控制点以类型化输入 / 输出连接 owner；交出的是受约束任务或事实，不是共享的隐式控制权。本环节把 用户事件、Decision、PolicyOutcome、Observation 与当前 RunState 中与“拆分收益”相关的事实转成 合法 transition、checkpoint、委派和验收状态 的一部分。
必须成立的约束
决策、推进、动作、验收和事件升级必须分别可归属，交接双方不能同时写同一控制状态。本章共同约束是：状态转移必须合法、可恢复且预算有界；违反时必须产生可区分的拒绝、未知或升级结果。
难点与取舍
拆分能提高并行度，却增加协调、上下文复制和冲突成本；是否拆分要由失败归因与吞吐基线决定。本层总体取舍是 确定性与适应性、自动化率与可恢复性，参数必须引用版本化数据、SLO 或风险基线。
边界与注意事项
推理可以建议，工具可以执行，治理可以允许或拒绝，但只有编排 owner 推进状态，业务 owner 做最终验收。编排控制层 的总边界是：不生成业务推理，也不伪造工具成功或治理许可；错误会继续污染 推理上下文、工具动作、用户状态和离线评测轨迹。

3. 上下文与权限

上下文与权限必须让不可信输入即使成功诱导模型，也无法越过身份、能力、文件、网络或租户边界。

职责与 I/O
orchestration-owner 在“上下文与权限”环节接收 用户事件、Decision、PolicyOutcome、Observation 与当前 RunState，产出 合法 transition、checkpoint、委派和验收状态；输出必须能由下游按版本消费。
工作机制
信任标签、最小能力、短期凭证、进程 / 文件 / 网络隔离与输出检查构成分层防御，负测验证可利用链。本环节把 用户事件、Decision、PolicyOutcome、Observation 与当前 RunState 中与“上下文与权限”相关的事实转成 合法 transition、checkpoint、委派和验收状态 的一部分。
必须成立的约束
deny-by-default、凭证不进入模型上下文、租户资源不可共享、egress 采用 allowlist，PII 在离开受控域前处理。本章共同约束是：状态转移必须合法、可恢复且预算有界；违反时必须产生可区分的拒绝、未知或升级结果。
难点与取舍
更强隔离会降低复用和性能；profile 依据副作用、数据敏感度和攻击面分级，而不是所有任务使用同一沙箱。本层总体取舍是 确定性与适应性、自动化率与可恢复性，参数必须引用版本化数据、SLO 或风险基线。
边界与注意事项
Prompt guardrail 降低内容风险但不是安全边界；真正的执行隔离由基础设施提供，业务授权由治理提供。编排控制层 的总边界是：不生成业务推理，也不伪造工具成功或治理许可；错误会继续污染 推理上下文、工具动作、用户状态和离线评测轨迹。

5. 多 Agent 归因

多 Agent 归因必须能把结论追溯到同一请求的输入、版本、决定、动作和验收事件，而不是依赖不可检索的日志文本。

职责与 I/O
orchestration-owner 在“多 Agent 归因”环节接收 用户事件、Decision、PolicyOutcome、Observation 与当前 RunState，产出 合法 transition、checkpoint、委派和验收状态；输出必须能由下游按版本消费。
工作机制
correlation_id / trace_id 串联 span 与结构化事件，引用保留原始证据位置、policy / model / tool 版本和 owner。本环节把 用户事件、Decision、PolicyOutcome、Observation 与当前 RunState 中与“多 Agent 归因”相关的事实转成 合法 transition、checkpoint、委派和验收状态 的一部分。
必须成立的约束
时间、因果顺序、采样与脱敏策略必须明确；高基数字段不能无界进入 Metric，敏感原文不能直接写 Log。本章共同约束是：状态转移必须合法、可恢复且预算有界；违反时必须产生可区分的拒绝、未知或升级结果。
难点与取舍
全量遥测利于归因却增加成本与隐私面；采样按风险与失败类型分层，关键审计事件不得随机丢弃。本层总体取舍是 确定性与适应性、自动化率与可恢复性，参数必须引用版本化数据、SLO 或风险基线。
边界与注意事项
Trace 解释系统事件与可见证据，不保存模型私有推理；Metric 告警不能替代逐请求审计。编排控制层 的总边界是：不生成业务推理，也不伪造工具成功或治理许可；错误会继续污染 推理上下文、工具动作、用户状态和离线评测轨迹。

难点与取舍

冲突目标参数依据决策边界
确定性与适应性、自动化率与可恢复性使用版本化回放、失败切片、SLO 分位数和风险等级，不把教学值当生产默认。orchestration-owner只调整本层机制；跨层变化需要对应 owner 与独立验收。
自动化收益 vs. 失败放大比较开启 / 关闭“拆分收益”与“多 Agent 归因”的消融结果、人工负载和恢复时间。证据不足返回 pending / unknown / escalate，不能靠更长 Prompt 隐藏边界。

失败传播与 first divergence

  1. 上游输入：先确认 用户事件、Decision、PolicyOutcome、Observation 与当前 RunState 的身份、作用域、时间与版本是否一致；输入错误不应在本层被“修正”为新事实。
  2. 首个偏离：按 拆分收益 → 委派契约 → 上下文与权限 → 冲突仲裁 → 多 Agent 归因 比较预期与实际中间产物，第一个不满足概念不变量的 stage 才是本章候选根因。
  3. 下游影响：本层错误会改变 推理上下文、工具动作、用户状态和离线评测轨迹；最终症状可能在更晚阶段出现，但不能从症状直接倒推 Prompt。
  4. 归属边界：orchestration-owner修复本层契约或机制；若 first divergence 位于相邻层，保留证据并交给相应 owner，不跨层调参。

概念—实践—证据回扣

核心区回答“是什么、为何成立、怎样传播”，实践区回答“何时触发、如何处置、怎样验收”。每个 stage 与 practice 使用稳定锚点双向连接：

常见误区

  1. 多 Agent 天然比单 Agent准确。
  2. 给不同 prompt 就等于专业分工。
  3. 共享全部上下文可以避免信息损失。
  4. Lead 能自动识别所有 worker 错误。
  5. 投票可以解决事实冲突和权限问题。

理论追问

请先脱稿回答，再展开参考答案核对边界、证据和取舍。

  1. 什么情况下普通并行任务优于 Multi-Agent?

    查看参考答案

    当子任务可预先枚举、输入输出契约稳定且无需独立上下文决策时，普通并行 worker 更便宜、更快、更易恢复。只有角色需要自主选择步骤、隔离上下文或独立专业工具时，Multi-Agent 才可能带来额外收益。

  2. 如何量化协调成本?

    查看参考答案

    比较单 Agent baseline 与多 Agent 的模型调用数、Token、延迟、handoff 次数、状态冲突、失败恢复时间和额外错误率。协调成本不仅是费用，还包括上下文复制、责任归因和权限管理复杂度。

  3. Lead 和 worker 应共享哪些状态?

    查看参考答案

    共享稳定目标、任务 ID、输入契约、约束、必要证据引用、预算和可见的完成状态；worker 私有草稿和无关上下文不应全量回灌。Lead 通过结构化结果与 provenance 汇总，而不是共享一份无限增长的消息历史。

  4. 子 Agent 能否拥有比 Lead 更高的权限?

    查看参考答案

    可以在受控委托场景中拥有不同专业权限，但不能隐式继承或自行扩大权限。必须由治理策略绑定 subject、delegator、资源、动作、期限和审批，Lead 只能请求委托，资源侧仍执行最终授权。

  5. 如何避免 handoff 循环?

    查看参考答案

    为任务设置 owner、handoff reason、已访问角色集合、最大移交次数和重复状态指纹。编排检测相同目标在角色间往返且无新证据时，应停止、重新分解或转人工，而不是让 Agent 继续互相转派。

  6. 最终答案正确时,为什么仍要评测 worker?

    查看参考答案

    正确结果可能来自偶然抵消，worker 中仍可能存在越权调用、错误证据或高成本循环。轨迹评测要检查各 worker 的契约满足、证据来源、权限和失败传播，才能判断系统是否可复现和安全。

  7. 如何做单 Agent 与 Multi-Agent 的公平 A/B?

    查看参考答案

    固定模型能力、工具、数据集、预算口径和验收器，只改变协作拓扑；同时比较质量、关键切片、安全、延迟、成本和恢复率。做角色消融以确认收益来自分工，而不是更多 Token 或更强模型。

日常实践关键点

核心判断

本章的日常工作不是记住名词，而是把“Multi-Agent 边界”变成可执行控制环：先识别触发信号，再用版本化证据定位首个偏离点，由 orchestration-owner 执行动作，最后用成功、拒绝、降级三类结果验收。推荐顺序是 验证拆分收益 → 固化委派契约 → 隔离上下文与权限 → 设计冲突仲裁 → 保留多 Agent 归因；任何参数都必须从真实 SLO、风险基线或失败样本推导。

深度实践卡

  • 1. 验证拆分收益
    触发信号
    当企业差旅 Agent 的“Multi-Agent 边界”链路出现状态非法跳转、循环不终止、恢复后重复副作用、职责漂移或候选完成被当成业务验收，或变更直接触及“验证拆分收益”的输入、策略或 owner 时，必须启动这项检查。
    诊断顺序
    诊断“验证拆分收益”时先冻结 run_state、transition、checkpoint、预算、授权快照、工具结果与完成条件。列出决策、推进、动作、验收和事件升级五类控制权，检查是否存在两个 owner、无人 owner 或交接后仍能越界写状态。
    执行动作
    落实“验证拆分收益”时，用 RACI / control-point 矩阵固定输入、输出、允许动作、拒绝结果和接管时限，交接双方只通过类型化结果协作。由 orchestration-owner 记录执行前后的版本、决定与接管条件。
    验证方法
    验收“验证拆分收益”时，回放状态转移、故障注入、暂停恢复、幂等与补偿测试，并核对最终验收证据；同时比较正确、拒绝和降级结果，阈值只从真实基线与风险容忍度推导。
    止损与取舍
    责任或验收边界不清时暂停自动推进，把控制权收敛到单一 owner，并保留待裁决事实清单；暂停 run、恢复到已验证 checkpoint，撤销未验收副作用并把控制权交给明确 owner。对“验证拆分收益”先保证可恢复和可审计，再优化自动化率。

    围绕“验证拆分收益”准备一条“症状—首个偏离—动作—证据—止损”的真实或个人实验 Trace，并明确自己只承担的 orchestration-owner 职责。

排障顺序

  1. 冻结请求、策略、模型 / 工具、数据和部署版本，避免边复现边变化。
  2. 从最早可验证输入开始，逐段对照契约、决策、动作与验收事件；在首个偏离点停止跨层调参。
  3. 由 orchestration-owner 判断修复、降级、回滚或转人工，并记录未验证假设。

常见问法

在生产或可复现实验中，你如何把“设计冲突仲裁”和“保留多 Agent 归因”落到企业差旅 Agent 的 Multi-Agent 边界 链路？

60–90 秒回答骨架

先交代 Multi-Agent 边界 的业务风险和 orchestration-owner 边界；再说触发信号与首个偏离点；随后说明契约、动作和参数依据；最后给出验证、止损和复盘结果。回答要同时覆盖正常路径与拒绝 / 降级路径。

可信证据

围绕“设计冲突仲裁 / 保留多 Agent 归因”展示脱敏后的 trace_id、run_state、transition、checkpoint、预算、授权快照、工具结果与完成条件、前后对照和失败切片；个人实验还应给出固定输入、复现步骤和观察结果，不虚构线上规模。
