单元 5:Multi-Agent 的适用边界

前置与目标

前置:可靠性与恢复。

完成后能够:

  1. 先定位单 Agent 瓶颈,再判断是否需要 Multi-Agent。
  2. 比较 orchestrator-workers、并行分工和层级协作。
  3. 说明共享状态、冲突仲裁、权限和评测归因问题。
  4. 用收益指标证明升级值得。

1. Multi-Agent 不是默认高级形态

多个 Agent 可以增加并行性、上下文隔离和专业分工,也会增加:

  • 任务分解错误。
  • 通信和 token 成本。
  • 重复搜索和重复动作。
  • 共享状态冲突。
  • 端到端失败归因难度。
  • 权限扩散和审计复杂度。

通用事实:增加独立决策主体会增加协调状态和失败组合。

项目参考架构:Multi-Agent 协作仍由编排控制层统一表达,治理和 trace 必须贯穿每个子 run。

实现偏好:先建立单 Agent 基线,只有可测收益超过协调成本时才升级。

2. 先诊断单 Agent 为什么失败

表现 可能根因 Multi-Agent 是否优先解法
工具经常调用错 工具描述或 schema 差 否,先修工具接口
长上下文丢失重点 上下文管理或任务过大 可能,也可先摘要/分段
独立子任务可并行 单 Agent 串行造成延迟 可能
需要不同专业权限 单一身份权限过宽 可能,但治理要先设计
目标模糊 需求没有澄清 否,更多 Agent 会放大分歧
评测标准不清 不知道何为成功 否,无法证明升级收益

3. 常见协作模式

Orchestrator-Workers

主 Agent 动态拆任务,多个 worker 独立执行,主 Agent 汇总。

适合:

  • 子任务数量和内容依输入而变化。
  • 子任务有较强独立性。
  • 主 Agent 能定义清晰交付契约。

风险:错误分解、遗漏、重复工作、汇总损失证据。

固定并行分工

代码预先定义多个角色或检查维度并行执行,最后确定性聚合或投票。

适合:

  • 分工稳定。
  • 需要多视角审查。
  • 结果可以独立评分或聚合。

它更接近 parallel workflow,不一定属于动态 Multi-Agent。

层级协作

上层负责目标和资源,下层继续委派。适合大型任务,但深度增加后状态、预算和责任链迅速复杂。

Peer-to-Peer / Handoff

Agent 根据能力把控制权移交给另一个 Agent。适合职责清晰的专业接力,需要明确当前 owner、handoff contract 和循环检测。

4. Orchestrator-Workers 时序

每个 worker 应有独立 sub-run ID、预算、工具权限和输出契约。Lead 不能只收到自然语言总结而丢失来源和失败状态。

5. 上下文隔离与共享状态

应隔离的信息

  • 与子任务无关的大量历史。
  • 其他 worker 的中间噪声。
  • 不属于该角色的敏感数据和工具权限。
  • 主 Agent 的未验证假设。

应共享的信息

  • 顶层目标和成功条件。
  • 子任务范围、依赖和预算。
  • 已确认事实与来源。
  • 公共约束和禁止动作。
  • 输出 schema 和截止条件。

共享状态不能让多个 Agent 任意覆盖同一字段。常见策略:

  • 单写者、多读者。
  • 每个 worker 写独立 namespace。
  • append-only 事件与最终聚合。
  • 乐观版本检查或显式锁。
  • 冲突由确定性规则或指定 owner 仲裁。

6. 权限边界

不要让所有子 Agent 继承 Lead 的全部权限。应按任务最小授权:

  • Research worker 只读搜索和资料。
  • Writer 只能写草稿区。
  • Reviewer 只读产物并输出建议。
  • Committer 在人工批准后才有提交权限。

委派需要记录“谁代表谁、在什么范围、有效多久”。子 Agent 进一步委派时必须受深度、预算和权限约束。

7. 冲突与仲裁

多个 Agent 可能产生冲突事实、计划或编辑。仲裁不能默认“再让一个模型投票”。先判断:

  1. 是否存在可查询的外部事实?
  2. 是否有确定性业务规则?
  3. 是否能按来源质量或时间排序?
  4. 是否属于需要人类价值判断的问题?

模型评审适合主观质量比较,不应覆盖数据库真实状态或权限策略。

8. 评测归因

至少评测三层:

层级 指标 关键问题
Worker 子任务成功率、证据质量、成本 worker 是否完成契约?
Coordination 分解覆盖、重复率、冲突率、等待时间 协作是否增加损耗?
End-to-end 最终成功、总延迟、总成本、安全 整体是否优于单 Agent 基线?

仅看最终答案无法判断哪个 worker 有贡献或造成失败。trace 需要保留父子 run、委派原因、输入上下文摘要和产物引用。

9. 升级决策表

问题 是 否
单 Agent 基线已经测量? 继续 先建立基线
主要瓶颈来自可分离子任务? 继续 修目标、工具或状态
子任务可并行或需上下文隔离? 继续 保持单 Agent
子任务交付契约可验证? 继续 先定义契约
有父子 trace 和预算控制? 继续 先补治理
成功/延迟收益覆盖协调成本? 尝试 Multi-Agent 保持简单方案

10. 案例:深度研究系统

Anthropic 公布的研究系统使用 lead agent 协调并行 subagents,适合开放式研究中动态拆分多个搜索方向。这个案例成立的前提包括:

  • 搜索方向相对独立,可以并行。
  • 子 Agent 的工具范围明确。
  • Lead 负责计划、继续研究和综合。
  • 系统可以评估来源与最终研究质量。

不能从这个案例推出“所有研究都应 Multi-Agent”。对于范围很小的问题,单 Agent 或固定并行搜索可能成本更低。

11. 失败矩阵

失败 Trace 证据 风险 修复
错误分解 多个 worker 均缺关键子问题 最终答案系统性遗漏 分解覆盖检查和动态补任务
重复工作 相同查询/来源跨 worker 重复 成本高无收益 共享已覆盖范围或去重
上下文污染 worker 接收无关历史 偏离子任务 最小上下文和 namespace
写冲突 多 worker 更新同一状态版本 结果丢失 单写者、事件合并或版本检查
无限委派 父子深度持续增长 成本失控 深度、数量和预算上限
无法归因 只有最终汇总 不能定位失败 父子 trace 和交付契约

常见误区

  1. 多 Agent 天然比单 Agent准确。
  2. 给不同 prompt 就等于专业分工。
  3. 共享全部上下文可以避免信息损失。
  4. Lead 能自动识别所有 worker 错误。
  5. 投票可以解决事实冲突和权限问题。

理论追问

  1. 什么情况下普通并行任务优于 Multi-Agent?
  2. 如何量化协调成本?
  3. Lead 和 worker 应共享哪些状态?
  4. 子 Agent 能否拥有比 Lead 更高的权限?
  5. 如何避免 handoff 循环?
  6. 最终答案正确时,为什么仍要评测 worker?
