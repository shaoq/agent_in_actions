单元 1:编排控制的职责与边界

前置与目标

前置:Agent 基础理论。

完成后能够:

  1. 解释编排层为什么不是“模型的大脑”。
  2. 区分编排、推理、Workflow、状态机、工具执行和治理。
  3. 画出一次单 Agent 请求由谁推进、由谁决策、由谁执行。
  4. 识别职责重复、控制权不清和隐式状态问题。

链路总览与现有讲解

职责与边界属于编排控制层：输入是用户事件、Decision、PolicyOutcome、Observation 与当前 RunState，输出是合法 transition、checkpoint、委派和验收状态。学习时先沿现有图、表或主题章节理解链路，再用后续 stage 检查每个中间产物的不变量。

1. 编排解决的核心问题

Agent 不是一次纯函数调用。一次任务可能需要多轮决策、环境动作、观察反馈、暂停和失败恢复。编排回答的是:

  • 当前运行处于什么状态?
  • 下一步需要调用哪个能力?
  • 输出是否符合控制契约?
  • 动作前需要哪些治理检查?
  • 结果写回哪里?
  • 什么时候继续、完成、失败或转人工?

通用事实:多步系统需要某种控制机制推进状态并决定转移。

项目参考架构:本项目把这组职责集中称为“编排控制层”,并让推理、工具、记忆和检索成为被协调的能力源。

实现偏好:简单系统可以是一段显式循环;复杂系统可以使用状态机、图运行时或任务队列。框架不是编排概念本身。

2. 编排与推理

问题 编排控制 推理内核
核心职责 推进运行状态和控制流 根据上下文提出判断或候选决策
输入 run state、事件、预算、策略 prompt/context、观察、约束
输出 状态转移、能力调用、终止 结构化 decision、计划或回答
是否执行副作用 不直接执行,负责协调和门禁 不应直接执行
主要失败 路由错、状态丢失、无法停止 判断错、输出不合法、计划不可行
确定性 尽量显式和可验证 本质上可能非确定

“谁决定下一步”常有两层含义:

  1. 推理内核可以提出“下一步调用 search”。
  2. 编排层验证这个 decision、检查权限和预算、执行路由并更新状态。

不同系统可以把部分路由写在模型 prompt、代码规则或图边中。职责位置可变,但必须能回答控制权在哪里、如何验证和如何观测。

3. 编排与 Workflow

Workflow 是控制路径的组织方式,编排是执行和推进控制的职责。

  • 固定 Workflow:路径主要由代码预定义。
  • 动态 Agent loop:模型根据观察提出路径。
  • 混合系统:外层 Workflow 固定关键阶段,局部节点允许 Agent 动态探索。

编排系统通常可以运行 Workflow 和 Agent loop,不能把两者当作互斥的产品架构。

4. 编排与状态机

状态机是一种表达控制流的工具:

  • 状态集合是显式的。
  • 事件触发状态转移。
  • 转移可以带 guard 和 action。
  • 非法转移可被拒绝。

Agent 编排可以用状态机实现,但“使用状态机”不自动解决:

  • 状态字段语义。
  • 模型输出校验。
  • checkpoint 和恢复。
  • 副作用幂等。
  • 权限、成本和评测。

状态机是编排的实现模式之一,不是编排全部职责。

5. 编排与工具执行

编排层协调工具调用,工具层负责能力契约和执行适配,治理层负责授权和审计。实际系统可能合并服务,但设计讨论要保持语义分离。

6. 编排与治理

治理不是编排器里的一个 if 语句。在线治理至少可能包含:

  • 输入和输出护栏。
  • 用户、Agent、工具三级身份与授权。
  • 高风险动作审批。
  • 成本和速率预算。
  • 审计和 trace。

编排决定在什么控制点调用治理能力;治理策略决定允许、拒绝或升级。把策略硬编码在模型提示词中会降低可验证性和可追溯性。

7. 职责边界表

8. 案例:企业工单 Agent

目标:阅读用户问题,查询知识库,必要时创建工单,高风险操作转人工。

合理分工:

  1. 接入层确认用户身份和会话。
  2. 编排层创建 run state,决定进入问答还是处理流程。
  3. 推理内核判断是否缺少信息、是否需要知识或动作。
  4. RAG 返回外部知识,不是写入长期记忆。
  5. 工具层查询和创建工单。
  6. 治理层在创建、修改和关闭动作前检查权限。
  7. 编排层把 observation 写回并判断是否满足完成条件。
  8. trace 进入离线评测,用于分析错误路由和未解决工单。

核心环节：从输入到可验证输出

1. 控制权边界

控制权边界成立的前提是每个控制决定只有一个最终 owner，交接不转移未声明的权限与验收责任。

职责与 I/O
orchestration-owner 在“控制权边界”环节接收 用户事件、Decision、PolicyOutcome、Observation 与当前 RunState，产出 合法 transition、checkpoint、委派和验收状态；输出必须能由下游按版本消费。
工作机制
控制点以类型化输入 / 输出连接 owner；交出的是受约束任务或事实，不是共享的隐式控制权。本环节把 用户事件、Decision、PolicyOutcome、Observation 与当前 RunState 中与“控制权边界”相关的事实转成 合法 transition、checkpoint、委派和验收状态 的一部分。
必须成立的约束
决策、推进、动作、验收和事件升级必须分别可归属，交接双方不能同时写同一控制状态。本章共同约束是：状态转移必须合法、可恢复且预算有界；违反时必须产生可区分的拒绝、未知或升级结果。
难点与取舍
拆分能提高并行度，却增加协调、上下文复制和冲突成本；是否拆分要由失败归因与吞吐基线决定。本层总体取舍是 确定性与适应性、自动化率与可恢复性，参数必须引用版本化数据、SLO 或风险基线。
边界与注意事项
推理可以建议，工具可以执行，治理可以允许或拒绝，但只有编排 owner 推进状态，业务 owner 做最终验收。编排控制层 的总边界是：不生成业务推理，也不伪造工具成功或治理许可；错误会继续污染 推理上下文、工具动作、用户状态和离线评测轨迹。

3. 治理门禁

治理门禁必须区分结构有效、动作完成、策略合规和业务验收；缺证据时只能是 pending 或 unknown。

职责与 I/O
orchestration-owner 在“治理门禁”环节接收 用户事件、Decision、PolicyOutcome、Observation 与当前 RunState，产出 合法 transition、checkpoint、委派和验收状态；输出必须能由下游按版本消费。
工作机制
每类成功由不同 owner 和证据判定，hard fail 先于综合评分，发布 / 完成只消费版本化验收结果。本环节把 用户事件、Decision、PolicyOutcome、Observation 与当前 RunState 中与“治理门禁”相关的事实转成 合法 transition、checkpoint、委派和验收状态 的一部分。
必须成立的约束
完成条件、拒绝条件、证据引用和验收 owner 必须可机器判断；模型自报或 HTTP 2xx 不能单独关单。本章共同约束是：状态转移必须合法、可恢复且预算有界；违反时必须产生可区分的拒绝、未知或升级结果。
难点与取舍
严格门禁降低错误发布却可能延长流程；软指标观察窗口和 hard fail 范围由风险与历史波动确定。本层总体取舍是 确定性与适应性、自动化率与可恢复性，参数必须引用版本化数据、SLO 或风险基线。
边界与注意事项
工具证明动作结果，治理证明允许，业务 owner 证明价值；编排只聚合这些证据推进终态。编排控制层 的总边界是：不生成业务推理，也不伪造工具成功或治理许可；错误会继续污染 推理上下文、工具动作、用户状态和离线评测轨迹。

难点与取舍

冲突目标参数依据决策边界
确定性与适应性、自动化率与可恢复性使用版本化回放、失败切片、SLO 分位数和风险等级，不把教学值当生产默认。orchestration-owner只调整本层机制；跨层变化需要对应 owner 与独立验收。

失败传播与 first divergence

  1. 上游输入：先确认 用户事件、Decision、PolicyOutcome、Observation 与当前 RunState 的身份、作用域、时间与版本是否一致；输入错误不应在本层被“修正”为新事实。
  2. 首个偏离：按 控制权边界 → 类型化交接 → 治理门禁 → 失败 Owner 矩阵 → 候选完成与业务验收 比较预期与实际中间产物，第一个不满足概念不变量的 stage 才是本章候选根因。
  3. 下游影响：本层错误会改变 推理上下文、工具动作、用户状态和离线评测轨迹；最终症状可能在更晚阶段出现，但不能从症状直接倒推 Prompt。
  4. 归属边界：orchestration-owner修复本层契约或机制；若 first divergence 位于相邻层，保留证据并交给相应 owner，不跨层调参。

10. 常见误区

  1. “编排层就是 LLM”:模型可以参与决策,但不应成为唯一状态和治理机制。
  2. “用了图框架就有可靠编排”:图解决表达,不自动解决状态语义和副作用。
  3. “所有路由都让模型判断更灵活”:确定性规则能稳定解决的问题不必交给模型。
  4. “工具失败是工具层自己的事”:工具分类错误,编排仍需决定重试、补偿或升级。

理论追问

请先脱稿回答，再展开参考答案核对边界、证据和取舍。

  1. 编排层和推理内核能否部署在同一进程?为什么语义仍要分开?

    查看参考答案

    可以部署在同一进程，但语义必须分开。编排拥有 Run State、状态迁移、重试、权限检查和终止权；推理内核只根据受控上下文提出下一步候选判断。进程边界是部署选择，职责边界决定可测试性、可替换性和故障归因。

  2. 哪些路由应该写成代码,哪些可以交给模型?

    查看参考答案

    身份授权、金额阈值、合规门禁、状态合法迁移和确定性业务规则应写成代码；意图模糊、方案比较或开放式任务分解可让模型提出候选。判断标准是规则能否稳定枚举、错误代价和是否需要可复现审计，模型路由仍需由编排校验。

  3. 一个固定 Workflow 是否需要编排器?

    查看参考答案

    需要控制执行顺序、状态和失败恢复的运行机制，但不一定需要复杂的 Agent 编排器。简单固定流程可由普通代码或状态机承担；只有出现持久状态、暂停恢复、并发、补偿或动态分支时，才值得引入更完整的编排基础。

  4. 模型直接调用工具的 SDK 为什么仍然存在编排逻辑?

    查看参考答案

    SDK 只是把模型输出解析成工具调用，调用前后的参数校验、授权、执行、Observation 回写、重试预算和终止判断仍构成编排。若这些逻辑隐含在回调中，它们并没有消失，只是变得更难观察和测试。

  5. 如何证明某次 finish 是业务成功而非模型自我声明?

    查看参考答案

    编排应调用独立验收器读取外部事实，例如按工单 ID 回查状态、运行测试或核对业务约束，并把验收结果作为 Observation 写入 State。Trace 至少要关联 finish 候选、验收规则版本、外部证据和最终 terminal reason。

  6. 如果治理策略在工具服务中,编排层还需要做什么?

    查看参考答案

    工具服务可执行资源侧的最终授权，但编排仍需在调用前维护主体、目标、风险等级、审批状态和预算，决定是否允许进入该工具，并处理拒绝后的分支。两层形成纵深防御：编排负责流程控制，工具服务负责资源边界的强制执行。

日常实践关键点

核心判断

本章的日常工作不是记住名词，而是把“职责与边界”变成可执行控制环：先识别触发信号，再用版本化证据定位首个偏离点，由 orchestration-owner 执行动作，最后用成功、拒绝、降级三类结果验收。推荐顺序是 检查控制权边界 → 固化类型化交接 → 校验治理门禁 → 维护失败 Owner 矩阵 → 分离候选完成与业务验收；任何参数都必须从真实 SLO、风险基线或失败样本推导。

  • 1. 检查控制权边界
    触发信号
    当企业差旅 Agent 的“职责与边界”链路出现状态非法跳转、循环不终止、恢复后重复副作用、职责漂移或候选完成被当成业务验收，或变更直接触及“检查控制权边界”的输入、策略或 owner 时，必须启动这项检查。
    诊断顺序
    诊断“检查控制权边界”时先冻结 run_state、transition、checkpoint、预算、授权快照、工具结果与完成条件。列出决策、推进、动作、验收和事件升级五类控制权，检查是否存在两个 owner、无人 owner 或交接后仍能越界写状态。
    执行动作
    落实“检查控制权边界”时，用 RACI / control-point 矩阵固定输入、输出、允许动作、拒绝结果和接管时限，交接双方只通过类型化结果协作。由 orchestration-owner 记录执行前后的版本、决定与接管条件。

    责任或验收边界不清时暂停自动推进，把控制权收敛到单一 owner，并保留待裁决事实清单；暂停 run、恢复到已验证 checkpoint，撤销未验收副作用并把控制权交给明确 owner。对“检查控制权边界”先保证可恢复和可审计，再优化自动化率。

常见问法

在生产或可复现实验中，你如何把“维护失败 Owner 矩阵”和“分离候选完成与业务验收”落到企业差旅 Agent 的 职责与边界 链路？

60–90 秒回答骨架

先交代 职责与边界 的业务风险和 orchestration-owner 边界；再说触发信号与首个偏离点；随后说明契约、动作和参数依据；最后给出验证、止损和复盘结果。回答要同时覆盖正常路径与拒绝 / 降级路径。

可信证据

围绕“维护失败 Owner 矩阵 / 分离候选完成与业务验收”展示脱敏后的 trace_id、run_state、transition、checkpoint、预算、授权快照、工具结果与完成条件、前后对照和失败切片；个人实验还应给出固定输入、复现步骤和观察结果，不虚构线上规模。

高频追问

为什么问题属于 编排控制层 而不是相邻层？参数如何从 SLO 或风险基线推导？结果未知时怎样避免扩大副作用？orchestration-owner 不在线时，谁依据什么证据接管？

边界与取舍

“维护失败 Owner 矩阵 / 分离候选完成与业务验收”会增加契约、存储或人工成本，只有错误成本与复现收益覆盖这些成本时才扩大自动化。未处理过该类生产事故时，应明确这是方案设计或个人实验结论。
