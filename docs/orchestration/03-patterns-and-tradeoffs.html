单元 3:编排模式与工程取舍

前置与目标

前置:状态与生命周期。

完成后能够:

  1. 区分 deterministic workflow、router、ReAct、plan-and-execute 和混合模式。
  2. 根据任务不确定性、审计、延迟、成本、恢复和风险选择模式。
  3. 说明拒绝其他模式的理由与升级触发条件。

链路总览与现有讲解

模式与取舍属于编排控制层：输入是用户事件、Decision、PolicyOutcome、Observation 与当前 RunState，输出是合法 transition、checkpoint、委派和验收状态。学习时先沿现有图、表或主题章节理解链路，再用后续 stage 检查每个中间产物的不变量。

1. 先从最简单方案开始

模式选择不是自治程度竞赛。更动态的模式提高适应性,同时扩大状态空间、调用次数和失败组合。

通用事实:路径越动态,行为预测、回归覆盖和成本控制越困难。

项目参考架构:编排层可以同时承载确定性路由和模型驱动决策。

实现偏好:先建立最简单可测基线,只有结果证明有必要时才提高自治程度。

2. Deterministic Workflow

主要路径由代码预先定义,模型只在局部节点执行分类、生成或判断。

适合:

  • 步骤与例外可以枚举。
  • 合规和可审计性要求高。
  • 失败处理需要确定性。
  • 延迟和成本预算严格。

代价:对开放式问题适应性弱,规则可能膨胀。

3. Router

Router 先分类输入,再进入专门流程、模型或工具集。分类器可以是规则、传统模型或 LLM。

适合:

  • 输入存在稳定类别。
  • 不同类别需要不同 prompt、模型、权限或 SLA。
  • 路由准确率可以独立评测。

主要风险:

  • 错误路由让后续流程再正确也无法完成任务。
  • 类别边界漂移。
  • 置信度不足时没有 fallback。

Router 不一定是 Agent;如果所有下游路径固定,它仍然是 Workflow。

4. ReAct

ReAct 将 reasoning 和 action 交错:模型根据当前观察提出动作,环境返回结果,模型再更新下一步。

适合:

  • 下一步高度依赖刚取得的信息。
  • 无法提前知道需要多少次检索或工具调用。
  • 环境提供可验证反馈。

优势:适应变化,可根据观察修正路径。

代价:

  • 调用次数和延迟难预测。
  • 容易重复动作或不收敛。
  • 需要严格停止条件、工具契约和 trace。
  • reasoning 文本不应被当作唯一审计证据。

ReAct 论文展示的是 reasoning 与 action 交错范式。生产系统仍需外部状态、授权、预算和完成校验。

5. Plan-and-Execute

先生成计划,再按计划执行;过程中根据观察决定继续、修改或重新规划。

适合:

  • 任务可以分解为相对稳定阶段。
  • 计划需要展示、审批或并行。
  • 希望将规划模型和执行模型分离。

风险:

  • 初始计划基于不完整信息。
  • 计划与现实漂移。
  • 重新规划过频会退化为昂贵的 ReAct。
  • 执行器可能机械完成步骤却没有达成目标。

计划需要版本、假设、依赖和失效条件。计划是控制输入,不是事实。

6. 混合模式

生产系统常用外层确定性 Workflow 包围局部 Agent loop:

混合模式的价值是把不确定性限制在真正需要它的局部,关键边界仍然确定化。

8. 选择流程

这不是机械决策树。业务成功条件、工具可靠性和评测能力不足时,即使任务开放也不应直接使用高自治模式。

9. 三个案例

固定发票审批

选择 Workflow。金额阈值、字段校验、审批角色和异常路径稳定。模型可用于字段抽取,但不控制付款路径。

升级条件:异常类型无法维护且需要综合大量非结构化证据时,可增加局部调查 Agent,付款仍走确定门禁。

开放式竞品研究

选择 bounded ReAct 或 plan-and-execute。研究线索会改变后续搜索;若研究维度可预先定义,先计划再执行更利于覆盖审计。

停止条件:来源覆盖、证据质量、时间和调用预算。

企业工单处理

选择混合模式。意图路由和权限用确定性控制;诊断过程允许 Agent 动态检索和询问;创建、修改和关闭工单前走治理与验收。

核心环节：从输入到可验证输出

1. 确定性流程起步

确定性流程起步必须以可观测信息增益和有界预算推进；没有新证据的重复思考不构成进展。

职责与 I/O
orchestration-owner 在“确定性流程起步”环节接收 用户事件、Decision、PolicyOutcome、Observation 与当前 RunState，产出 合法 transition、checkpoint、委派和验收状态；输出必须能由下游按版本消费。
工作机制
入口条件选择最简单可行模式，每轮比较证据 / 状态变化，触线后返回 clarify、abstain、degrade 或 escalate。本环节把 用户事件、Decision、PolicyOutcome、Observation 与当前 RunState 中与“确定性流程起步”相关的事实转成 合法 transition、checkpoint、委派和验收状态 的一部分。
必须成立的约束
最大步数、时间、成本、重规划次数和退出类型必须显式；自检不能无限递归，也不能绕过外部验证。本章共同约束是：状态转移必须合法、可恢复且预算有界；违反时必须产生可区分的拒绝、未知或升级结果。
难点与取舍
更长推理可能提高复杂任务质量，也会累积延迟和错误；预算由任务价值、失败成本和消融结果决定。本层总体取舍是 确定性与适应性、自动化率与可恢复性，参数必须引用版本化数据、SLO 或风险基线。
边界与注意事项
推理可以发现不确定性并提出澄清，编排负责实际暂停、预算执行和人工接管。编排控制层 的总边界是：不生成业务推理，也不伪造工具成功或治理许可；错误会继续污染 推理上下文、工具动作、用户状态和离线评测轨迹。

2. 模式路由信号

模式路由信号必须先满足能力、风险和兼容硬约束，再比较质量、延迟与成本；名称或模型自报不能作为路由证据。

职责与 I/O
orchestration-owner 在“模式路由信号”环节接收 用户事件、Decision、PolicyOutcome、Observation 与当前 RunState，产出 合法 transition、checkpoint、委派和验收状态；输出必须能由下游按版本消费。
工作机制
能力画像裁剪不可用候选，版本化策略对剩余候选评分，adapter 规范输入输出并记录 route_reason 与替代项。本环节把 用户事件、Decision、PolicyOutcome、Observation 与当前 RunState 中与“模式路由信号”相关的事实转成 合法 transition、checkpoint、委派和验收状态 的一部分。
必须成立的约束
结构化输出、工具支持、上下文、区域和数据策略等硬约束不可被低价覆盖；fallback 必须兼容安全与语义。本章共同约束是：状态转移必须合法、可恢复且预算有界；违反时必须产生可区分的拒绝、未知或升级结果。
难点与取舍
动态路由提升效率却增加不可复现性；先 shadow 再 canary，用关键切片与 SLO 决定放量和回退。本层总体取舍是 确定性与适应性、自动化率与可恢复性，参数必须引用版本化数据、SLO 或风险基线。
边界与注意事项
网关选择模型与规范协议，不选择业务工具、不拥有推理意图，也不授予工具执行权限。编排控制层 的总边界是：不生成业务推理，也不伪造工具成功或治理许可；错误会继续污染 推理上下文、工具动作、用户状态和离线评测轨迹。

难点与取舍

冲突目标参数依据决策边界
确定性与适应性、自动化率与可恢复性使用版本化回放、失败切片、SLO 分位数和风险等级，不把教学值当生产默认。orchestration-owner只调整本层机制；跨层变化需要对应 owner 与独立验收。
自动化收益 vs. 失败放大比较开启 / 关闭“确定性流程起步”与“模式选择 ADR”的消融结果、人工负载和恢复时间。证据不足返回 pending / unknown / escalate，不能靠更长 Prompt 隐藏边界。

失败传播与 first divergence

  1. 上游输入：先确认 用户事件、Decision、PolicyOutcome、Observation 与当前 RunState 的身份、作用域、时间与版本是否一致；输入错误不应在本层被“修正”为新事实。
  2. 首个偏离：按 确定性流程起步 → 模式路由信号 → ReAct 循环 → Replan 门禁 → 模式选择 ADR 比较预期与实际中间产物，第一个不满足概念不变量的 stage 才是本章候选根因。
  3. 下游影响：本层错误会改变 推理上下文、工具动作、用户状态和离线评测轨迹；最终症状可能在更晚阶段出现，但不能从症状直接倒推 Prompt。
  4. 归属边界：orchestration-owner修复本层契约或机制；若 first divergence 位于相邻层，保留证据并交给相应 owner，不跨层调参。

概念—实践—证据回扣

核心区回答“是什么、为何成立、怎样传播”，实践区回答“何时触发、如何处置、怎样验收”。每个 stage 与 practice 使用稳定锚点双向连接：

  • 确定性流程起步的不变量或约束失效时，进入对应日常实践；用本页工程产物、Trace / 评测切片和拒绝 / 降级结果证明是否恢复。
  • 模式路由信号的不变量或约束失效时，进入对应日常实践；用本页工程产物、Trace / 评测切片和拒绝 / 降级结果证明是否恢复。
  • ReAct 循环的不变量或约束失效时，进入对应日常实践；用本页工程产物、Trace / 评测切片和拒绝 / 降级结果证明是否恢复。
  • Replan 门禁的不变量或约束失效时，进入对应日常实践；用本页工程产物、Trace / 评测切片和拒绝 / 降级结果证明是否恢复。
  • 模式选择 ADR的不变量或约束失效时，进入对应日常实践；用本页工程产物、Trace / 评测切片和拒绝 / 降级结果证明是否恢复。

常见误区

  1. ReAct 就等于把完整思维链写入日志。
  2. 计划越详细越可靠;过早细化会放大错误假设。
  3. Router 使用 LLM 后就自动成为 Agent。
  4. 混合模式只是把多种框架拼在一起;关键是控制边界。

理论追问

请先脱稿回答，再展开参考答案核对边界、证据和取舍。

  1. 如何判断任务步骤是否“可枚举”?

    查看参考答案

    看输入类别、分支条件和验收路径能否在设计时稳定列出，并用历史样本验证未知分支率。步骤多不等于不可枚举；若环境新证据会频繁改变目标或产生未预见动作，才需要更动态的 Agent 控制。

  2. Router 的置信度低时有哪些 fallback?

    查看参考答案

    可以请求澄清、使用确定性默认路由、扩大候选后由规则校验、转人工或安全拒绝。不能只把同一请求无限重试给模型；fallback 应记录候选、置信证据、阈值、最终路由和升级原因。

  3. ReAct 如何检测 observation 没有信息增益?

    查看参考答案

    为每步记录目标子问题、证据集合和 Observation 指纹，比较是否新增可验证事实、缩小候选或改变计划。连续重复查询、相同错误和无新来源可触发停机、改策略或转人工，而不是依赖模型自报“没有进展”。

  4. Plan-and-Execute 的重规划触发器应该由谁判断?

    查看参考答案

    编排层根据结构化条件判断，例如依赖失败、前提失效、预算变化、审批拒绝或计划版本过期；模型可以提出重规划建议，但不能自行覆盖已批准计划。触发事件与新旧计划差异必须进入 Trace。

  5. 为什么高风险动作适合放在确定性边界?

    查看参考答案

    因为授权、金额阈值、不可逆性和审批条件需要可复现、可审计且默认拒绝。模型适合解释上下文或提出动作候选，真正执行前应由确定性策略校验主体、资源、参数和审批快照。

  6. 如何通过 trace 把成功轨迹沉淀为 Workflow?

    查看参考答案

    先按目标切片聚合成功轨迹，识别高频稳定步骤、必要不变量和少量动态节点，再用回放与消融验证固定化不会降低成功率或安全性。只把稳定部分编码为 Workflow，保留不确定节点的受控决策能力。

日常实践关键点

核心判断

本章的日常工作不是记住名词，而是把“模式与取舍”变成可执行控制环：先识别触发信号，再用版本化证据定位首个偏离点，由 orchestration-owner 执行动作，最后用成功、拒绝、降级三类结果验收。推荐顺序是 从确定性流程起步 → 记录模式路由信号 → 约束 ReAct 循环 → 设置 Replan 门禁 → 维护模式选择 ADR；任何参数都必须从真实 SLO、风险基线或失败样本推导。

    触发信号

    诊断顺序
    诊断“从确定性流程起步”时先冻结 run_state、transition、checkpoint、预算、授权快照、工具结果与完成条件。检查循环入口、继续条件、累计预算、信息增益和退出结果，确认系统不是用重复推理掩盖缺证据或缺授权。
    执行动作

排障顺序

  1. 冻结请求、策略、模型 / 工具、数据和部署版本，避免边复现边变化。
  2. 从最早可验证输入开始，逐段对照契约、决策、动作与验收事件；在首个偏离点停止跨层调参。
  3. 由 orchestration-owner 判断修复、降级、回滚或转人工，并记录未验证假设。
贯穿案例｜企业差旅 Agent：员工请求预订超预算航班时，只把制度条款当知识证据，把员工靠窗偏好当记忆，把预订动作交给工具层；本章只处理 模式与取舍 的控制与证据。若发现状态非法跳转、循环不终止、恢复后重复副作用、职责漂移或候选完成被当成业务验收，先保留 trace_id 与版本，再在本层首个偏离点修复，避免把问题推给 Prompt。
