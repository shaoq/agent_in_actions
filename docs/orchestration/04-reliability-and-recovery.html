单元 4:可靠性、失败与恢复

前置与目标

前置:模式与取舍。

完成后能够:

  1. 区分模型 finish、编排终止和业务成功。
  2. 为步骤、时间、成本和失败设置有界控制。
  3. 按错误类型选择重试、重新决策、补偿或人工接管。
  4. 用 trace 区分 decide、act、observe 和 terminate 失败。

1. 可靠性不等于“最终返回了答案”

一次可靠运行至少要同时满足:

  • 任务结果通过业务成功条件。
  • 没有越权或未审计副作用。
  • 成本、时间和步骤在预算内。
  • 失败时状态可诊断、可停止或可恢复。
  • 结果可由外部证据验证。

通用事实:非确定模型的自我声明不能替代外部业务验收。

项目参考架构:编排维护控制状态,治理在线门禁,评测离线判断行为质量。

实现偏好:已知失败优先转为 typed state 和 trace,而不是只抛出无上下文异常。

2. 完成判定

完成可以分三层:

  1. Model finish:模型不再请求动作并产生最终回答。
  2. Run termination:编排器停止循环并记录终态。
  3. Business acceptance:测试、外部状态、规则或用户确认任务真的成功。

例如代码 Agent 输出补丁并 finish,但测试失败,只能说明模型结束了,不能标记任务成功。

3. 有界终止

预算 防止的问题 终止后的选择
max steps 无限循环和重复动作 总结已完成工作、转人工或失败
deadline / timeout 外部服务或长推理阻塞 取消、恢复点或异步等待
token / cost 非受控费用 降级模型、缩短上下文或停止
retry limit 暂时错误变成永久循环 重新决策、fallback 或人工
risk budget 多次低风险动作累积成高风险 审批或禁止继续

预算必须进入状态和 trace,否则无法解释“为什么停止”。

4. 重复轨迹与无信息增益

仅有最大步数仍然浪费资源。编排器可以检测:

  • 相同 action + 参数重复。
  • 连续 observation 相同。
  • 计划没有减少未完成项。
  • 错误类型在多轮中不变。
  • 置信度或验收结果没有改善。

检测后可注入失败摘要要求重新规划,切换工具,降低目标,或转人工。不能用“再思考一次”作为唯一修复。

5. 错误分类先于重试

错误类型 示例 默认策略
Transient 限流、短暂网络失败 有界退避重试
Validation 参数缺失、schema 不合法 返回决策器修正,不原样重试
Authorization 权限不足、审批过期 拒绝或重新授权
Business conflict 工单已关闭、余额变化 重新读取环境并决策
Model contract 输出无法解析 结构化修复或模型 fallback
Irreversible uncertainty 支付响应丢失 查询真实状态,禁止盲重试
Permanent 工具不存在、资源删除 失败、降级或人工

工具层可处理传输级短重试;编排层负责跨步骤业务决策。两层需要共享错误分类和 action ID。

6. 幂等和动作状态

对有副作用动作,至少跟踪:

intent_created -> authorized -> submitted -> committed
                         \-> rejected
submitted -> unknown -> reconciled(committed | not_committed)

幂等键用于让重复请求返回同一业务结果,但前提是下游支持该契约。若下游不支持,需要查询、锁、去重表、补偿或人工处理。

7. 补偿不是回滚

分布式副作用通常无法真正回滚。补偿是新的业务动作,例如:

  • 发送更正邮件,而不是“撤销已读邮件”。
  • 创建冲正记录,而不是删除支付历史。
  • 恢复工单状态并保留审计,而不是抹去更新。

补偿也会失败,因此需要独立状态、权限和告警。

8. Checkpoint 与 HITL

适合暂停的控制点:

  • 高风险、不可逆或大额动作前。
  • 模型置信度不足且需要业务判断时。
  • 用户目标存在歧义时。
  • 预算接近阈值但已有部分成果时。

人在回路请求应包含:

  • 目标和当前状态摘要。
  • 已验证事实与来源。
  • 拟执行动作、参数和风险。
  • 可选决策及其后果。
  • 审批有效期和恢复标识。

恢复时重新验证身份、权限和环境状态,不能把人类的一次批准当永久授权。

9. Trace 最小语义

TraceEvent
├── run_id / parent_id / step
├── phase: orchestrate | decide | act | observe | terminate
├── event_type
├── state_version
├── model/tool identity and version
├── input/output summary or reference
├── duration / token / cost
├── permission and approval result
└── error_type / retryability / terminal_reason

Trace 不应直接记录密钥、完整 PII 或未经策略允许的内部 reasoning。应记录足以复现控制决策的结构化证据。

10. 失败归因矩阵

Phase 现象 已知证据 可能原因 修复层次
decide decision schema invalid validation error + raw response ref 指令、模型或 schema 推理契约 / gateway
act action timeout action ID + timeout 下游慢或网络 工具重试 / fallback
observe 结果未写入 state tool success 但 state version 未变 原子性或映射错误 编排状态更新
terminate max steps 重复轨迹 目标、工具或策略不收敛 编排策略 / HITL
finish acceptance failed 模型 finish + 测试失败 完成判断过早 验收器 / 重新决策
resume duplicate side effect 同一业务键多次提交 checkpoint 边界或幂等缺失 动作协议 / 恢复流程

分析时要分开:

  • 事实:trace 已证明的事件。
  • 假设:还需要更多证据验证的原因。
  • 修复:应落在哪个组件。
  • 回归:如何证明同类问题不再发生。

11. 案例:工单 Agent 卡在重复检索

Trace 显示连续四步调用同一搜索动作,query 略有改写,返回文档集合相同。

事实:

  • action 和 observation 高度重复。
  • 未完成项没有变化。
  • 达到 max steps 后停止。

不能直接断言“模型太差”。可能原因还包括:

  • 检索工具没有返回缺失信息提示。
  • 目标缺少可接受的降级条件。
  • 状态没有把已读文档和未解决问题传回模型。
  • 编排没有重复检测。

修复需要分别验证工具、状态和策略,并增加重复轨迹回归用例。

常见误区

  1. 所有异常重试三次。
  2. 有 checkpoint 就不会重复副作用。
  3. 人工批准后不需要重新授权。
  4. Trace 越详细越好,忽略隐私和成本。
  5. run_completed 就是任务成功率分子。

理论追问

  1. 如何确定最大步数而不是拍脑袋?
  2. 工具层重试和编排层重试如何分工?
  3. 响应超时如何判断动作是否实际成功?
  4. 补偿失败时系统进入什么状态?
  5. 如何定义 observation 的信息增益?
  6. 哪些 trace 字段必须脱敏?
  7. 在线监控和离线 eval 如何消费同一 trace?
