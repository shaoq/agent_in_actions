单元 4:可靠性、失败与恢复

前置与目标

前置:模式与取舍。

完成后能够:

  1. 区分模型 finish、编排终止和业务成功。
  2. 为步骤、时间、成本和失败设置有界控制。
  3. 按错误类型选择重试、重新决策、补偿或人工接管。
  4. 用 trace 区分 decide、act、observe 和 terminate 失败。

链路总览与现有讲解

可靠性与恢复属于编排控制层：输入是用户事件、Decision、PolicyOutcome、Observation 与当前 RunState，输出是合法 transition、checkpoint、委派和验收状态。学习时先沿现有图、表或主题章节理解链路，再用后续 stage 检查每个中间产物的不变量。

1. 可靠性不等于“最终返回了答案”

一次可靠运行至少要同时满足:

  • 任务结果通过业务成功条件。
  • 没有越权或未审计副作用。
  • 成本、时间和步骤在预算内。
  • 失败时状态可诊断、可停止或可恢复。
  • 结果可由外部证据验证。

通用事实:非确定模型的自我声明不能替代外部业务验收。

项目参考架构:编排维护控制状态,治理在线门禁,评测离线判断行为质量。

实现偏好:已知失败优先转为 typed state 和 trace,而不是只抛出无上下文异常。

2. 完成判定

完成可以分三层:

  1. Model finish:模型不再请求动作并产生最终回答。
  2. Run termination:编排器停止循环并记录终态。
  3. Business acceptance:测试、外部状态、规则或用户确认任务真的成功。

例如代码 Agent 输出补丁并 finish,但测试失败,只能说明模型结束了,不能标记任务成功。

3. 有界终止

预算 防止的问题 终止后的选择
max steps 无限循环和重复动作 总结已完成工作、转人工或失败
deadline / timeout 外部服务或长推理阻塞 取消、恢复点或异步等待
token / cost 非受控费用 降级模型、缩短上下文或停止
retry limit 暂时错误变成永久循环 重新决策、fallback 或人工
risk budget 多次低风险动作累积成高风险 审批或禁止继续

预算必须进入状态和 trace,否则无法解释“为什么停止”。

4. 重复轨迹与无信息增益

仅有最大步数仍然浪费资源。编排器可以检测:

  • 相同 action + 参数重复。
  • 连续 observation 相同。
  • 计划没有减少未完成项。
  • 错误类型在多轮中不变。
  • 置信度或验收结果没有改善。

检测后可注入失败摘要要求重新规划,切换工具,降低目标,或转人工。不能用“再思考一次”作为唯一修复。

5. 错误分类先于重试

错误类型 示例 默认策略
Transient 限流、短暂网络失败 有界退避重试
Validation 参数缺失、schema 不合法 返回决策器修正,不原样重试
Authorization 权限不足、审批过期 拒绝或重新授权
Business conflict 工单已关闭、余额变化 重新读取环境并决策
Model contract 输出无法解析 结构化修复或模型 fallback
Irreversible uncertainty 支付响应丢失 查询真实状态,禁止盲重试
Permanent 工具不存在、资源删除 失败、降级或人工

工具层可处理传输级短重试;编排层负责跨步骤业务决策。两层需要共享错误分类和 action ID。

6. 幂等和动作状态

对有副作用动作,至少跟踪:

幂等键用于让重复请求返回同一业务结果,但前提是下游支持该契约。若下游不支持,需要查询、锁、去重表、补偿或人工处理。

7. 补偿不是回滚

分布式副作用通常无法真正回滚。补偿是新的业务动作,例如:

  • 发送更正邮件,而不是“撤销已读邮件”。
  • 创建冲正记录,而不是删除支付历史。
  • 恢复工单状态并保留审计,而不是抹去更新。

补偿也会失败,因此需要独立状态、权限和告警。

8. Checkpoint 与 HITL

适合暂停的控制点:

  • 高风险、不可逆或大额动作前。
  • 模型置信度不足且需要业务判断时。
  • 用户目标存在歧义时。
  • 预算接近阈值但已有部分成果时。

人在回路请求应包含:

  • 目标和当前状态摘要。
  • 已验证事实与来源。
  • 拟执行动作、参数和风险。
  • 可选决策及其后果。
  • 审批有效期和恢复标识。

恢复时重新验证身份、权限和环境状态,不能把人类的一次批准当永久授权。

9. Trace 最小语义

TraceEvent
├── run_id / parent_id / step
├── phase: orchestrate | decide | act | observe | terminate
├── event_type
├── state_version
├── model/tool identity and version
├── input/output summary or reference
├── duration / token / cost
├── permission and approval result
└── error_type / retryability / terminal_reason

Trace 不应直接记录密钥、完整 PII 或未经策略允许的内部 reasoning。应记录足以复现控制决策的结构化证据。

10. 失败归因矩阵

Phase 现象 已知证据 可能原因 修复层次
decide decision schema invalid validation error + raw response ref 指令、模型或 schema 推理契约 / gateway
act action timeout action ID + timeout 下游慢或网络 工具重试 / fallback
observe 结果未写入 state tool success 但 state version 未变 原子性或映射错误 编排状态更新
terminate max steps 重复轨迹 目标、工具或策略不收敛 编排策略 / HITL
finish acceptance failed 模型 finish + 测试失败 完成判断过早 验收器 / 重新决策
resume duplicate side effect 同一业务键多次提交 checkpoint 边界或幂等缺失 动作协议 / 恢复流程

分析时要分开:

  • 事实:trace 已证明的事件。
  • 假设:还需要更多证据验证的原因。
  • 修复:应落在哪个组件。
  • 回归:如何证明同类问题不再发生。

11. 案例:工单 Agent 卡在重复检索

Trace 显示连续四步调用同一搜索动作,query 略有改写,返回文档集合相同。

事实:

  • action 和 observation 高度重复。
  • 未完成项没有变化。
  • 达到 max steps 后停止。

不能直接断言“模型太差”。可能原因还包括:

  • 检索工具没有返回缺失信息提示。
  • 目标缺少可接受的降级条件。
  • 状态没有把已读文档和未解决问题传回模型。
  • 编排没有重复检测。

修复需要分别验证工具、状态和策略,并增加重复轨迹回归用例。

核心环节：从输入到可验证输出

1. 完成条件

完成条件成立的前提是每个控制决定只有一个最终 owner，交接不转移未声明的权限与验收责任。

职责与 I/O
orchestration-owner 在“完成条件”环节接收 用户事件、Decision、PolicyOutcome、Observation 与当前 RunState，产出 合法 transition、checkpoint、委派和验收状态；输出必须能由下游按版本消费。
工作机制
控制点以类型化输入 / 输出连接 owner；交出的是受约束任务或事实，不是共享的隐式控制权。本环节把 用户事件、Decision、PolicyOutcome、Observation 与当前 RunState 中与“完成条件”相关的事实转成 合法 transition、checkpoint、委派和验收状态 的一部分。
必须成立的约束
决策、推进、动作、验收和事件升级必须分别可归属，交接双方不能同时写同一控制状态。本章共同约束是：状态转移必须合法、可恢复且预算有界；违反时必须产生可区分的拒绝、未知或升级结果。
难点与取舍
拆分能提高并行度，却增加协调、上下文复制和冲突成本；是否拆分要由失败归因与吞吐基线决定。本层总体取舍是 确定性与适应性、自动化率与可恢复性，参数必须引用版本化数据、SLO 或风险基线。
边界与注意事项
推理可以建议，工具可以执行，治理可以允许或拒绝，但只有编排 owner 推进状态，业务 owner 做最终验收。编排控制层 的总边界是：不生成业务推理，也不伪造工具成功或治理许可；错误会继续污染 推理上下文、工具动作、用户状态和离线评测轨迹。

2. 有界终止

有界终止必须以可观测信息增益和有界预算推进；没有新证据的重复思考不构成进展。

职责与 I/O
orchestration-owner 在“有界终止”环节接收 用户事件、Decision、PolicyOutcome、Observation 与当前 RunState，产出 合法 transition、checkpoint、委派和验收状态；输出必须能由下游按版本消费。
工作机制
入口条件选择最简单可行模式，每轮比较证据 / 状态变化，触线后返回 clarify、abstain、degrade 或 escalate。本环节把 用户事件、Decision、PolicyOutcome、Observation 与当前 RunState 中与“有界终止”相关的事实转成 合法 transition、checkpoint、委派和验收状态 的一部分。
必须成立的约束
最大步数、时间、成本、重规划次数和退出类型必须显式；自检不能无限递归，也不能绕过外部验证。本章共同约束是：状态转移必须合法、可恢复且预算有界；违反时必须产生可区分的拒绝、未知或升级结果。
难点与取舍
更长推理可能提高复杂任务质量，也会累积延迟和错误；预算由任务价值、失败成本和消融结果决定。本层总体取舍是 确定性与适应性、自动化率与可恢复性，参数必须引用版本化数据、SLO 或风险基线。
边界与注意事项
推理可以发现不确定性并提出澄清，编排负责实际暂停、预算执行和人工接管。编排控制层 的总边界是：不生成业务推理，也不伪造工具成功或治理许可；错误会继续污染 推理上下文、工具动作、用户状态和离线评测轨迹。

3. 按错误类型重试

按错误类型重试只有在主体、来源、时间、目的、置信和可撤回状态明确时才是可用记忆，不等于聊天文本或外部知识副本。

职责与 I/O
orchestration-owner 在“按错误类型重试”环节接收 用户事件、Decision、PolicyOutcome、Observation 与当前 RunState，产出 合法 transition、checkpoint、委派和验收状态；输出必须能由下游按版本消费。
工作机制
经历先成为 candidate，经验证与同意晋升；读取按作用域、时效、类型和置信排序，冲突保留版本而非覆盖历史。本环节把 用户事件、Decision、PolicyOutcome、Observation 与当前 RunState 中与“按错误类型重试”相关的事实转成 合法 transition、checkpoint、委派和验收状态 的一部分。
必须成立的约束
MemoryRecord 必须带 subject / tenant、provenance、consent、policy_version 与 TTL，删除要传播到索引、缓存和派生摘要。本章共同约束是：状态转移必须合法、可恢复且预算有界；违反时必须产生可区分的拒绝、未知或升级结果。
难点与取舍
更高召回能增强个性化，也会增加旧事实、投毒和隐私风险；读取与保留参数由时间切片评测和用户控制要求决定。本层总体取舍是 确定性与适应性、自动化率与可恢复性，参数必须引用版本化数据、SLO 或风险基线。
边界与注意事项
RunState 管当前执行，Session 管交互连续性，RAG 管外部资料；Memory 只沉淀经治理的经历。编排控制层 的总边界是：不生成业务推理，也不伪造工具成功或治理许可；错误会继续污染 推理上下文、工具动作、用户状态和离线评测轨迹。

4. 动作幂等

动作幂等只能把系统带回已知状态或安全终态；未知结果在查明副作用前不得被当成可重试失败。

职责与 I/O
orchestration-owner 在“动作幂等”环节接收 用户事件、Decision、PolicyOutcome、Observation 与当前 RunState，产出 合法 transition、checkpoint、委派和验收状态；输出必须能由下游按版本消费。
工作机制
错误先分瞬时、永久、冲突与 unknown outcome；幂等键、ledger、checkpoint 和补偿记录共同决定 retry、query、resume 或人工核对。本环节把 用户事件、Decision、PolicyOutcome、Observation 与当前 RunState 中与“动作幂等”相关的事实转成 合法 transition、checkpoint、委派和验收状态 的一部分。
必须成立的约束
尝试次数、累计 deadline、退避、副作用作用域和补偿 owner 都必须有界，幂等键需覆盖业务动作而非单次请求。本章共同约束是：状态转移必须合法、可恢复且预算有界；违反时必须产生可区分的拒绝、未知或升级结果。
难点与取舍
重试提高瞬时可用性也会放大负载与副作用；预算由下游容量、用户 deadline 和动作不可逆性决定。本层总体取舍是 确定性与适应性、自动化率与可恢复性，参数必须引用版本化数据、SLO 或风险基线。
边界与注意事项
补偿是新的业务动作而非时间回滚；网关 fallback 不应掩盖工具副作用或编排非法状态。编排控制层 的总边界是：不生成业务推理，也不伪造工具成功或治理许可；错误会继续污染 推理上下文、工具动作、用户状态和离线评测轨迹。

5. 补偿与恢复

补偿与恢复只能把系统带回已知状态或安全终态；未知结果在查明副作用前不得被当成可重试失败。

职责与 I/O
orchestration-owner 在“补偿与恢复”环节接收 用户事件、Decision、PolicyOutcome、Observation 与当前 RunState，产出 合法 transition、checkpoint、委派和验收状态；输出必须能由下游按版本消费。

难点与取舍

冲突目标参数依据决策边界
确定性与适应性、自动化率与可恢复性使用版本化回放、失败切片、SLO 分位数和风险等级，不把教学值当生产默认。orchestration-owner只调整本层机制；跨层变化需要对应 owner 与独立验收。
自动化收益 vs. 失败放大比较开启 / 关闭“完成条件”与“补偿与恢复”的消融结果、人工负载和恢复时间。证据不足返回 pending / unknown / escalate，不能靠更长 Prompt 隐藏边界。

失败传播与 first divergence

  1. 上游输入：先确认 用户事件、Decision、PolicyOutcome、Observation 与当前 RunState 的身份、作用域、时间与版本是否一致；输入错误不应在本层被“修正”为新事实。
  2. 首个偏离：按 完成条件 → 有界终止 → 按错误类型重试 → 动作幂等 → 补偿与恢复 比较预期与实际中间产物，第一个不满足概念不变量的 stage 才是本章候选根因。
  3. 下游影响：本层错误会改变 推理上下文、工具动作、用户状态和离线评测轨迹；最终症状可能在更晚阶段出现，但不能从症状直接倒推 Prompt。
  4. 归属边界：orchestration-owner修复本层契约或机制；若 first divergence 位于相邻层，保留证据并交给相应 owner，不跨层调参。

概念—实践—证据回扣

核心区回答“是什么、为何成立、怎样传播”，实践区回答“何时触发、如何处置、怎样验收”。每个 stage 与 practice 使用稳定锚点双向连接：

  • 完成条件的不变量或约束失效时，进入对应日常实践；用本页工程产物、Trace / 评测切片和拒绝 / 降级结果证明是否恢复。
  • 有界终止的不变量或约束失效时，进入对应日常实践；用本页工程产物、Trace / 评测切片和拒绝 / 降级结果证明是否恢复。
  • 按错误类型重试的不变量或约束失效时，进入对应日常实践；用本页工程产物、Trace / 评测切片和拒绝 / 降级结果证明是否恢复。
  • 动作幂等的不变量或约束失效时，进入对应日常实践；用本页工程产物、Trace / 评测切片和拒绝 / 降级结果证明是否恢复。
  • 补偿与恢复的不变量或约束失效时，进入对应日常实践；用本页工程产物、Trace / 评测切片和拒绝 / 降级结果证明是否恢复。

常见误区

  1. 所有异常重试三次。
  2. 有 checkpoint 就不会重复副作用。
  3. 人工批准后不需要重新授权。
  4. Trace 越详细越好,忽略隐私和成本。
  5. run_completed 就是任务成功率分子。

理论追问

请先脱稿回答，再展开参考答案核对边界、证据和取舍。

  1. 如何确定最大步数而不是拍脑袋?

    查看参考答案

    基于任务分解的必要步骤、历史成功轨迹分布、延迟和成本 SLO 设置软硬预算，并按任务类型分层。门槛应通过评测验证：过低造成截断，过高掩盖循环；运行中还要结合信息增益和重复检测提前停止。

  2. 工具层重试和编排层重试如何分工?

    查看参考答案

    工具层处理连接抖动等可安全透明重试，并遵守短超时和幂等条件；编排层根据业务错误类别、已发生副作用、全局预算和替代路径决定是否再次尝试、补偿或升级。两层必须共享 attempt、call ID 和 effect 状态。

  3. 响应超时如何判断动作是否实际成功?

    查看参考答案

    把它标记为 effect unknown，禁止直接重放。使用幂等键或原调用 ID 查询外部系统，取得 receipt、资源版本或业务状态；确认未提交才可重试，确认已提交则回写成功，无法确认时转人工对账。

  4. 补偿失败时系统进入什么状态?

    查看参考答案

    进入显式的 compensation_failed 或 needs_manual_repair，而不是继续宣称失败已回滚。State 要保留原动作、补偿尝试、剩余副作用和责任主体，暂停后续相关动作并触发告警与人工修复。

  5. 如何定义 observation 的信息增益?

    查看参考答案

    信息增益可以表现为新增可信证据、减少候选空间、更新关键状态或解除下一步前置条件。应比较规范化内容、来源与业务字段，而不只比较文本；若连续多步没有增益，就触发改策略、停止或升级。

  6. 哪些 trace 字段必须脱敏?

    查看参考答案

    Prompt、工具参数、检索片段、Observation、用户标识、凭据和业务载荷都可能含 PII 或密钥。应按字段分级做采集前最小化、tokenization 或遮盖，并记录策略版本；调试所需原文只能通过受审计的 break-glass 访问。

  7. 在线监控和离线 eval 如何消费同一 trace?

    查看参考答案

    在线监控读取低延迟事件和聚合指标，用于超时、越权、预算和错误告警；离线评测读取脱敏、版本化的完整轨迹，分析结果质量、首个偏离点和回归。二者共享 run、step、版本和事件语义，但保留不同延迟与数据权限。

日常实践关键点

核心判断

本章的日常工作不是记住名词，而是把“可靠性与恢复”变成可执行控制环：先识别触发信号，再用版本化证据定位首个偏离点，由 orchestration-owner 执行动作，最后用成功、拒绝、降级三类结果验收。推荐顺序是 验证完成条件 → 设置有界终止 → 按错误类型重试 → 保护动作幂等 → 记录补偿与恢复；任何参数都必须从真实 SLO、风险基线或失败样本推导。

深度实践卡

  • 1. 验证完成条件
    触发信号
    当企业差旅 Agent 的“可靠性与恢复”链路出现状态非法跳转、循环不终止、恢复后重复副作用、职责漂移或候选完成被当成业务验收，或变更直接触及“验证完成条件”的输入、策略或 owner 时，必须启动这项检查。
    诊断顺序
    诊断“验证完成条件”时先冻结 run_state、transition、checkpoint、预算、授权快照、工具结果与完成条件。列出决策、推进、动作、验收和事件升级五类控制权，检查是否存在两个 owner、无人 owner 或交接后仍能越界写状态。
    执行动作
    落实“验证完成条件”时，用 RACI / control-point 矩阵固定输入、输出、允许动作、拒绝结果和接管时限，交接双方只通过类型化结果协作。由 orchestration-owner 记录执行前后的版本、决定与接管条件。
    验证方法
    验收“验证完成条件”时，回放状态转移、故障注入、暂停恢复、幂等与补偿测试，并核对最终验收证据；同时比较正确、拒绝和降级结果，阈值只从真实基线与风险容忍度推导。

    责任或验收边界不清时暂停自动推进，把控制权收敛到单一 owner，并保留待裁决事实清单；暂停 run、恢复到已验证 checkpoint，撤销未验收副作用并把控制权交给明确 owner。对“验证完成条件”先保证可恢复和可审计，再优化自动化率。

排障顺序

  1. 冻结请求、策略、模型 / 工具、数据和部署版本，避免边复现边变化。

关联实践：保护动作幂等、记录补偿与恢复

常见问法

在生产或可复现实验中，你如何把“保护动作幂等”和“记录补偿与恢复”落到企业差旅 Agent 的 可靠性与恢复 链路？

60–90 秒回答骨架

先交代 可靠性与恢复 的业务风险和 orchestration-owner 边界；再说触发信号与首个偏离点；随后说明契约、动作和参数依据；最后给出验证、止损和复盘结果。回答要同时覆盖正常路径与拒绝 / 降级路径。

可信证据

围绕“保护动作幂等 / 记录补偿与恢复”展示脱敏后的 trace_id、run_state、transition、checkpoint、预算、授权快照、工具结果与完成条件、前后对照和失败切片；个人实验还应给出固定输入、复现步骤和观察结果，不虚构线上规模。

高频追问

为什么问题属于 编排控制层 而不是相邻层？参数如何从 SLO 或风险基线推导？结果未知时怎样避免扩大副作用？orchestration-owner 不在线时，谁依据什么证据接管？

边界与取舍

“保护动作幂等 / 记录补偿与恢复”会增加契约、存储或人工成本，只有错误成本与复现收益覆盖这些成本时才扩大自动化。未处理过该类生产事故时，应明确这是方案设计或个人实验结论。

本单元产出

  • 为工单 Agent 写出错误分类和重试策略。
  • 从给定重复 trace 区分事实、假设、修复和回归。
  • 设计高风险动作前的 HITL 信息包。
