UNIT.02

架构位置与边界

职责边界 · 输入是编排整理的决策上下文，输出是结构化意图。编排决定何时调用、是否接受和如何推进；模型调用统一经过模型网关；工具层只执行已校验动作。
契约面组件级定义
输入目标、当前状态、Observation、可用能力、预算与治理约束
输出act / finish / clarify / escalate，以及参数、证据、置信与理由摘要
上游编排控制层准备上下文并持有运行状态
下游模型网关提供模型访问；编排把意图路由到工具、RAG 或交互层

UNIT.03

模式与取舍

MODE 1

直接决策

短任务、上下文明确；延迟低，但复杂任务容易遗漏约束。

MODE 2

ReAct

观察会改变下一步；适应性高，但需循环预算与重复检测。

MODE 3

Plan-and-Execute

依赖可提前分解；便于审计，但计划过时需要重规划。

MODE 4

受约束解码

高风险或强 schema；可验证性高，但不能替代业务验收。

UNIT.05

生产工程约束

  • 决策对象必须版本化并经过 schema 校验。
  • 上下文只携带完成决策所需的最小充分证据。
  • 不确定性要转成澄清、检索或升级，而不是隐藏在措辞中。
  • 记录模型、prompt、输入摘要、决策与校验错误以支持回放。
UNIT.06

面试表达

30 秒回答

推理内核是 Agent 的决策面：它根据编排提供的目标、状态、观察和约束，形成结构化的下一步意图。它不维护生命周期、不直接执行工具，也不直接调用模型厂商；模型访问走网关，意图由编排校验和路由。

3 分钟回答结构

先定义决策契约，再讲输入来自编排状态；随后说明模型网关、结构化输出和业务验收边界；用直接决策、ReAct、计划执行说明模式取舍；最后落到证据不足、重复轨迹、预算和降级的 trace 诊断。

表达检查 · 不要只报框架或产品名；必须讲清输入输出、owner、证据、失败归因和为什么选择该模式。
