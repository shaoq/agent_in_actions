编排控制层无代码理论检查点

目的

验证学习者是否能够在不阅读实验代码的情况下解释编排边界、状态、模式和可靠性。通过后才推荐进入最小 Agent Loop 实验。

前置:

  1. 通过Agent 基础理论检查点。
  2. 完成编排五单元理论课程。

规则

  • 总分:100 分。
  • 建议时限:120 分钟,不作为自动推进条件。
  • 通过线:总分不低于 80 分。
  • 单项门槛:五项任务均不低于 12 分。
  • 必须提交全部理论证据。
  • 不允许用框架 API 或代码替代架构解释。

任务一:请求链路与职责边界

分值:20 分。

不看资料画出一次带工具的单 Agent 请求,至少包含:

  • 接入、编排、推理、治理、工具、观察和模型网关。
  • 控制流和数据流方向。
  • 动作执行前的权限点。
  • finish 和业务 acceptance 的区别。

再回答:编排和推理可以在同一进程中,为什么架构语义仍需区分?

评分锚点:

分数 标准
0-7 只有组件名或将模型视为全部控制。
8-11 主链路基本正确,但治理、观察或完成边界缺失。
12-16 职责和方向正确,能区分决策意图与控制执行。
17-20 能补充替代实现、参考架构前提和错误边界。

任务二:状态与生命周期

分值:20 分。

为“高风险退款调查 Agent”设计状态模型和生命周期,要求:

  1. 区分 run、session、checkpoint 和 long-term memory。
  2. 给出 running、waiting、completed、failed、cancelled、exhausted 转移。
  3. 说明 checkpoint 保存时机。
  4. 列出恢复前检查。
  5. 解释如何避免重复退款。

评分锚点:

分数 标准
0-7 所有信息放在 messages 或同一“memory”中。
8-11 能区分主要状态,但恢复和副作用不完整。
12-16 生命周期、checkpoint、重新授权和幂等基本完整。
17-20 能讨论 schema version、外部状态核对和无法安全迁移的出口。

任务三:模式选择

分值:20 分。

分别为以下场景选择 Workflow、Router、ReAct、Plan-and-Execute 或混合模式:

  1. 固定发票审批。
  2. 多类别客服分流。
  3. 开放式竞品研究。
  4. 可先审批计划的批量数据修复。
  5. 既要动态调查又有高风险动作的企业工单。

每题必须说明:

  • 选择依据。
  • 不选两个替代模式的原因。
  • 模式升级或降级的观测信号。

评分锚点:

分数 标准
0-7 按“复杂就用 Agent”选择。
8-11 能区分固定和动态路径,取舍维度不足。
12-16 同时考虑不确定性、审计、延迟、成本和风险。
17-20 能给出混合边界、切换信号和评测方案。

任务四:失败 Trace 归因

分值:20 分。

分析以下摘要:

step=1 phase=decide event=decision_made kind=act action=search
step=1 phase=observe event=result docs=[A,B]
step=2 phase=decide event=decision_made kind=act action=search
step=2 phase=observe event=result docs=[A,B]
step=3 phase=decide event=decision_made kind=act action=search
step=3 phase=observe event=result docs=[A,B]
step=3 phase=terminate event=max_steps_reached

要求分别写出:

  • Trace 已证明的事实。
  • 尚未证明的原因假设。
  • 需要补充的观测字段。
  • 可能涉及的组件。
  • 修复方案和回归验证。

评分锚点:

分数 标准
0-7 直接断言“模型能力不足”。
8-11 能识别重复循环,但事实和推测混合。
12-16 能分层检查状态、工具、目标和策略。
17-20 能设计补充证据、重复检测和有针对性的回归用例。

任务五:无代码系统设计与口述

分值:20 分。

设计一个“可暂停、可恢复的企业工单 Agent”,8 分钟内说明:

  1. 问题和成功条件。
  2. 控制流和状态模型。
  3. 模式选择及拒绝其他模式的原因。
  4. 动作权限、重试、幂等和 HITL。
  5. Trace、业务验收和离线评测。
  6. 什么时候考虑 Multi-Agent。

随后接受至少两轮追问:

  • 模型说完成但工单未解决怎么办?
  • 恢复后如何避免重复动作?
  • 为什么不直接用 Multi-Agent?
  • 哪些失败应重试,哪些应转人工?

评分锚点:

分数 标准
0-7 主要讲框架或模型,没有控制和失败设计。
8-11 主流程可理解,状态、治理或验收不完整。
12-16 架构、状态、可靠性和取舍形成闭环。
17-20 能说明假设、替代方案、证据边界和演进条件。

补强规则

  • 边界不足:重画“谁提出决策、谁验证、谁执行、谁记录”。
  • 状态不足:重新按生命周期拆分,补恢复与副作用场景。
  • 模式不足:每题增加“不选其他方案”的理由和切换信号。
  • 归因不足:强制分开事实、假设、修复和回归。
  • 表达不足:录制 3 分钟与 8 分钟两个版本,重练追问不稳定处。

通过后的动作

  1. 在进度跟踪登记五类证据。
  2. 进入最小 Agent Loop 实验,把理论映射到代码。
  3. 完成实验后再执行编排工程模块考核。
