统一场景

为一家多租户企业设计“制度与产品支持 Agent”。资料来自员工制度、产品手册、工单知识库；员工只能访问本租户和自身职级允许的内容，经理可访问团队范围。文档每天变化，存在撤回、冲突版本和文档内 prompt injection。系统要求回答逐条引用，证据不足时拒答，并能从 trace 定位失败。

深化交付契约

在既有 A–E 任务中继续使用同一场景，并补充以下可审计约束：

  • A：标出 document/version/content/chunk 四层身份如何贯穿离线、在线与 trace，并解释 namespace+hash 的边界。
  • B：给出 registry 状态机、幂等键、解析置信/HITL、staging→publish、删除/撤权恢复演练；比较 child 召回与 parent context。
  • C：按 query bucket 选择 transformation，展示约束保真、各路候选预算、Context Budget 与高级检索采用条件。
  • D：分别验证 citation validity/correctness/completeness 和 semantic support，说明 Judge 校准、冲突与 abstention。
  • E：提供固定版本矩阵、oracle/ablation、切片样本、质量—延迟—成本 Pareto、hard fail 与灰度回滚。

任何 top-k、阈值、chunk size、延迟或成本数字必须标注为“本场景实验结果”或“结构示例”，并说明校准数据，不能宣称通用默认值。

证据要求

  1. 图：至少一张离线/在线链路图，节点标 owner，边标主要字段，包含“知识 / RAG ≠ Agent 经历记忆”。
  2. 契约：给出 RetrievalRequest、EvidenceChunk 或等价 schema，必须包含身份/租户、来源定位、版本、ACL 和 trace 关联。
  3. 矩阵：比较至少两种 chunk、两种检索信号和是否使用 rerank 的取舍。
  4. 安全：至少覆盖跨租户泄露、缓存污染、撤权延迟与间接 prompt injection。
  5. 诊断：提供一次错误回答的候选 ids、重排前后、最终 context、claims/citations、索引与策略版本。
