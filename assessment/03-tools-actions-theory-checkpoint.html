工具 / 动作层理论检查点

通过规则 · 满分 100；总分至少 80，且任务 1—5 每项至少 12/20。任一单项未达到 60%，即使总分达到 80 也不通过。通过表示 L2 理论达标，不表示 L3 工程实现完成。

作答场景

你要为企业售后 Agent 设计三项能力：查询订单（只读）、发起退款（资金副作用）、向客户发送确认邮件（不可收回的外部传播）。系统可能通过内部 API 或审核后的 MCP server 接入；客服后台还保留 Computer Use 兜底。一次退款曾在下游处理后客户端超时，无法确认是否已扣减可退余额。

所有回答必须基于同一场景，明确哪些事实来自权威系统，哪些是模型提议，并保持推理内核、编排控制、治理、工具层和基础设施边界一致。

任务与评分

任务 1 · 能力契约与 Schema（20 分）

为退款能力定义 capability_id，给出 input、output、error、effect 四类契约；至少包含 4 个输入约束、4 种类型化错误、幂等与审批声明，并说明一次不兼容变更如何迁移。

评分点分值达标证据
四类契约完整且字段语义清楚8schema/表格可确定性验证，不用自由文本代替状态
权威字段与边界6身份、凭据、endpoint、risk 不由模型提供；工具不规划任务
版本与契约测试6识别 breaking change、迁移/回滚与 schema 指纹

任务 2 · 生命周期与失败 Trace（20 分）

画出或表述退款从 ActionIntent 到 observation 的 prepare、authorize、dispatch、observe、commit 链路。给出一条含关键关联字段的失败 trace，并区分 transport success、tool success、business acceptance。

评分点分值达标证据
五阶段顺序与责任8每阶段有输入、输出、负责人和失败
Observation / trace 可诊断7run/step/decision/call、version、attempt、policy/effect refs 齐全
三种成功边界5工具成功不直接等于 run 完成

任务 3 · 权限、审批与沙箱（20 分）

为查询、退款、发信三项能力设计权限矩阵；给出退款 approval 的绑定字段和失效条件；说明治理层、工具层、基础设施分别如何防 confused deputy、TOCTOU、secret exposure 与间接 prompt injection。

评分点分值达标证据
最小权限与授权输入7subject/action/resource/context；读写与资源范围分离
审批绑定与恢复重验7digest、身份、有效期、版本；参数/资源变化使审批失效
沙箱与威胁边界6治理定策略、工具强制、基础设施提供真实隔离

任务 4 · 重试、幂等与副作用恢复（20 分）

处理“退款已派发但超时”的未知结果：给出状态查询、幂等记录、重试/补偿/人工核对的决策表；解释相同 key 不同参数的处理，并设计统一重试预算。

评分点分值达标证据
错误与未知结果分类6不把 timeout 直接当未执行；区分 dispatch boundary
幂等协议8稳定 key、请求摘要、原子 claim、结果保存、conflict/TTL
恢复与预算6查询优先；受限 retry、compensation、manual reconcile 和防放大

任务 5 · 接口选型与系统设计（20 分）

比较 Function Calling、MCP、Computer Use，分别说明层次、信任边界、适用场景和风险。为查询、退款、发信选择接入方式，并说明 API/MCP 不可用时如何降级而不扩大权限。

评分点分值达标证据
三种方式辨析准确7结构化模型接口、连接协议、环境交互不混为一层
选型有工程依据7稳定性、可观测性、权限、幂等、维护成本与风险
降级与端到端边界6API 优先；Computer Use 受限/HITL；不静默扩大权限
