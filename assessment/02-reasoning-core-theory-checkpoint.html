推理内核理论检查点

目的与前置

本检查点验证你能否把模型能力放进清晰的决策契约，正确选择推理模式，处理不确定性，并依据证据归因失败。它不要求代码，也不以框架 API 代替架构解释。

开始前应完成推理内核四单元课程。建议闭卷完成初稿，再回看资料补充引用；把两种颜色或两个版本分开，保留真实掌握证据。

通过规则

  • 总分 100 分；五项任务各 20 分。
  • 总分必须不低于 80 分。
  • 每项任务必须不低于 12 分（该项满分的 60%）。总分达标但任一单项低于 12，仍不通过。
  • 建议时限 120 分钟；时限只用于复盘，不替代能力门禁。
  • 必须提交作答正文和证据链接；口述任务需保留提纲、录音或同伴追问记录。
评分快照 · “available”只表示检查点资产已经可用；只有实际提交、评分并达到双门槛，才能在 progress tracker 记录为理论通过。

任务一：边界与决策契约（20 分）

为“企业工单 Agent”设计一次 decide 调用：

  1. 画出编排层、推理内核、治理层、工具层和模型网关的数据流与控制边界。
  2. 定义 DecisionContext 的至少六组字段及来源。
  3. 定义 act、respond、clarify、escalate、finish 五类 Decision 及共同字段。
  4. 列出至少六条不变量，必须覆盖状态、授权、执行、完成和模型访问。
  5. 回答：为什么这些模块同进程部署时仍需语义分层？
区间评分锚点
0–7把推理内核等同于整个 Agent 或一次 LLM 调用；控制与副作用混在模型输出中。
8–11主链路基本正确，但 finish/acceptance、授权或模型网关边界缺失。
12–16输入、判别联合和五层边界正确；不变量可验证。
17–20补充版本、证据引用、上下文最小化、失败类型与替代部署说明。

任务二：结构化输出与工具选择（20 分）

审查以下候选决策：

{
  "kind": "act",
  "tool": "refund_order",
  "arguments": {"order_id": "42", "amount": 800},
  "message": "用户很着急，我已退款"
}

已知：当前只验证了订单号；金额和支付状态未知；用户身份是客服访客；退款超过 500 元需财务审批。要求：

  1. 指出至少八个契约、证据、语义、权限或状态问题。
  2. 给出正确的 Decision 判别联合和一个合法的下一步候选。
  3. 按“能力裁剪—意图选择—结构校验—语义校验—授权—执行”说明负责人。
  4. 设计解析失败、未知工具、证据不足、策略拒绝四类错误的修复或停止策略。
区间评分锚点
0–7只把字段改成合法 JSON，仍直接执行退款。
8–11发现缺参和审批，但没有分开选择、授权与执行。
12–16四层验证、能力裁剪、合法查询或澄清意图和有限修复基本完整。
17–20能处理 evidence refs、稳定 capability ID、敏感值注入、错误码、修复预算和回归指标。

任务三：推理模式选择（20 分）

为以下五个场景分别选择 Direct/Router、Decomposition、Plan-and-Execute、有限 ReAct、候选比较，或明确交给确定性 Workflow：

  1. 客服单意图分类。
  2. 规则稳定的发票字段校验。
  3. 需要多来源证据的竞品研究。
  4. 先审批计划再批量修复生产数据。
  5. 证据冲突且影响重大的合规判断。

每题必须写：选择理由、不选至少两个替代方案的理由、结构化产物、预算/停止条件、升级或降级信号。另解释“推理模式”和“编排模式”的区别。

区间评分锚点
0–7以“复杂就用 ReAct/Multi-Agent”作答，没有停止条件。
8–11大体选对，但缺少替代取舍、风险或成本。
12–16按可枚举性、观察依赖、计划稳定性、风险与预算做出可执行选择。
17–20给出简单基线、动态路由、分桶评测和上线回退条件。

任务四：不确定性与 Trace 归因（20 分）

run=R7 strategy=limited-react contract=v3
step=1 decision=act capability=search_orders args_hash=a9 confidence=high
step=1 observation=result result_hash=z2 docs=[order-42]
step=2 decision=act capability=search_orders args_hash=a9 confidence=high
step=2 observation=result result_hash=z2 docs=[order-42]
step=3 decision=act capability=search_orders args_hash=a9 confidence=high
step=3 terminate=max_steps

要求：

  1. 列出 trace 已证明的事实，不得夹带原因断言。
  2. 提出至少四个跨组件假设，并指出验证每个假设需要的字段。
  3. 识别这里的置信问题和不确定性类型。
  4. 给出推理、编排、工具/知识和网关各层可能的修复边界。
  5. 设计三个回归用例，至少一个验证无进展停止。
区间评分锚点
0–7直接断言模型能力不足并建议换更大模型。
8–11识别重复，但事实、假设和修复混在一起。
12–16能分层补证，设计重复检测、显式失败和针对性回归。
17–20进一步讨论上下文指纹、观察 provenance、校准、恢复核对和隐私安全 trace。

任务五：系统设计与口述（20 分）

设计一个“高风险退款调查 Agent”的推理内核，提交架构图和 8 分钟口述：

  1. 定义成功条件、DecisionContext、Decision schema 与不变量。
  2. 说明能力裁剪、结构/语义校验、权限和执行链路。
  3. 为调查、澄清、审批和完成选择推理模式及停止条件。
  4. 处理目标歧义、证据冲突、模型不可用、策略拒绝和恢复后的外部状态变化。
  5. 设计 trace、离线评测指标、上线门槛和回退方案。

至少接受两轮追问：为什么不让模型直接退款？为什么不记录完整思维过程？换模型时如何验证契约？模型说完成但业务未验收怎么办？

区间评分锚点
0–7主要讲 prompt 或模型，没有契约、失败和权限。
8–11正常路径可理解，但边界、恢复或评测不完整。
12–16契约、模式、证据、失败、安全和评测形成闭环。
17–20能说明假设、替代方案、演进条件、校准和成本/延迟取舍，并稳定应对追问。
