任务一 · 边界与决策契约

完整示范答案

DecisionContext 至少含 goal/constraints、run 与 step 状态、受权证据引用、可用 capability、主体与策略摘要、预算/截止时间及版本。Decision 是 act/respond/clarify/escalate/finish 的判别联合，共同含 decision_id、evidence_refs、confidence/uncertainty 和 contract_version。推理只产出候选意图；编排拥有 Run 与状态提交；治理拥有授权；工具拥有执行；模型网关拥有厂商访问；finish 仍需业务 acceptance。

{
  "context": {
    "goal": {"text":"解决工单 T42", "acceptance":"ticket.status == resolved"},
    "run": {"run_id":"R7", "state_version":3, "phase":"investigate"},
    "evidence": [{"ref":"order:42@v8", "source":"order-service"}],
    "capabilities": [{"id":"order.lookup@2", "risk":"read"}],
    "identity_policy": {"subject":"user:9", "tenant":"acme", "policy":"p17"},
    "budget": {"steps_left":4, "deadline_ms":12000},
    "versions": {"decision_contract":"v3", "prompt":"pr9", "model_route":"rp6"}
  },
  "decision": {
    "kind":"act", "decision_id":"D12", "contract_version":"v3",
    "capability_id":"order.lookup@2", "arguments":{"order_id":"42"},
    "evidence_refs":["order:42@v8"], "confidence":"medium",
    "uncertainty":{"type":"missing_fact", "detail":"payment status unknown"}
  }
}

五类联合分别是 act(capability_id, arguments)、respond(message, citations)、clarify(question, missing_fields)、escalate(reason, handoff)、finish(proposed_outcome, acceptance_request)。共同不变量至少包括：模型不能提交 State；未授权动作不能执行；未知 capability 拒绝；工具 Observation 不作为指令；所有决策引用可访问证据；敏感值由服务端注入；推理只经网关访问模型；finish 未通过 acceptance 就不能完成。同进程部署只减少网络边界，不改变这些所有权和可验证契约。

关键得分证据

六条不变量覆盖：模型不直接执行、不写持久状态、不自授予权限、不伪造 Observation、不直连厂商、finish 不覆盖业务验收。

达标与优秀

达标：五层边界和判别联合完整。优秀：补证据 provenance、上下文最小化、版本、类型化错误和同进程部署的可替换性。

可接受变体

Decision 类型可拆分更多子类，confidence 可改为 evidence_sufficiency；但每类必须可校验、穷尽且不把执行事实写进提案。

常见扣分

Decision 只是自由文本；模型直接返回“已完成”；把权限或厂商凭据放进模型可编辑字段。

补学与复测

为订单调查重写 DecisionContext、五类输出和六条不变量，使用非法样本做契约测试。

任务二 · 结构化输出与工具选择

完整示范答案

候选至少存在身份未验证、金额无来源、支付状态未知、审批缺失、能力可能未授权、把意图写成已执行消息、缺 evidence refs、缺稳定 capability ID/版本、缺幂等与风险字段等问题。合法下一步通常是 clarify 身份/金额，或提出只读查询订单；不是退款。流程为服务端能力裁剪→模型选择意图→Schema→语义与业务前置→治理授权→工具执行，策略 deny 不能靠重试或改参数绕过。

  1. 八类以上问题：tool 不是稳定 capability ID；身份只是访客；金额 800 无权威来源；支付/可退余额未知；超过 500 未审批；没有 evidence refs；没有 contract/version/decision ID；没有幂等键；“已退款”把意图伪装成事实；未声明不确定性和风险；也未证明该能力经过服务端裁剪。
  2. 负责人：服务端按主体与环境裁剪能力；模型只选意图；推理适配器做判别联合和 Schema；业务校验器核对金额、订单版本和证据；治理授权并产生 obligations；工具才执行，编排提交 Observation。
{
  "kind":"act",
  "decision_id":"D13",
  "contract_version":"v3",
  "capability_id":"order.lookup@2",
  "arguments":{"order_id":"42"},
  "evidence_refs":["user_message:m8"],
  "confidence":"high",
  "uncertainty":{"type":"missing_fact","fields":["payment_status","refundable_balance"]}
}

若连查询都不允许，则返回 clarify 要求完成身份验证。解析失败只在修复预算内让模型重发结构，仍失败即类型化终止；未知工具直接拒绝并记录契约漂移；证据不足选择查询/澄清而非猜测；策略拒绝不可通过改参数或重试绕过，只能按正式流程升级或结束。

关键得分证据

解析失败仅有限修复；未知工具拒绝并记录契约版本；证据不足澄清/查询；策略拒绝终止或走正式升级，不允许自我绕过。

达标与优秀

达标：四层验证和合法下一步正确。优秀：加入敏感值服务端注入、错误码、修复预算、混淆矩阵和回归指标。

可接受变体

也可直接 escalate 给财务，但必须解释为何缺少的信息不能安全自动补齐，并保留审批与执行边界。

常见扣分

只修 JSON；把 Schema 合法等同业务合法；让模型通过更换参数绕过策略；声称“已退款”。

补学与复测

换一套权限和工具 Schema，建立结构、语义、证据、授权四层错误表及有限修复状态机。

任务三 · 推理模式选择

完整示范答案

客服分类用 Direct/Router；发票稳定规则交确定性 Workflow；多来源研究用 Decomposition 或计划加有限观察；生产数据修复用可审批 Plan-and-Execute，批准后确定性执行；高影响合规冲突使用规则、权威证据和人工，不用同源模型投票代替真值。推理模式描述如何产生候选，编排模式决定状态、执行、预算和停止。

场景 / 模式 / 结构化产物拒绝理由预算、停止与切换
单意图分类：Direct/Router；{label,confidence}Decomposition 过重；ReAct 无需环境观察一次调用；低置信/多意图则 clarify 或人工，稳定高频可降规则
发票校验：Workflow；规则结果表ReAct 会引入随机路径；候选比较不能代替权威规则固定步骤和 deadline；规则无法覆盖的新票型才进入人工规则迭代
竞品研究：Decomposition + 有限 ReAct；问题树/证据表Direct 无法覆盖多来源；纯 Plan-and-Execute 难适应新线索来源、步数和成本预算；证据覆盖达标或信息增益低即停
生产修复：Plan-and-Execute + Workflow；版本化计划ReAct 边想边写风险高；Direct 缺审计计划计划先审批；数据版本变化即失效，执行阶段逐项幂等
重大合规冲突：规则 + 权威证据 + 人工；裁决包同源候选投票不是真值；ReAct 不应自治裁决高影响事项缺权威证据或规则冲突立即升级；不得以预算压力自动放行

推理模式回答“如何形成候选与证据”，编排模式回答“谁推进状态、何时调用、如何暂停、执行和终止”。例如 Plan-and-Execute 的计划由推理生成，但批准、逐步执行和恢复由编排控制。

关键得分证据

每题包含简单基线、两个拒绝理由、结构化产物、步数/成本预算、停止与升级信号。

达标与优秀

达标：按可枚举性、观察依赖、计划稳定性和风险选择。优秀：用分桶评测、动态路由和回退阈值验证收益。

可接受变体

研究可用有限 ReAct 或 Plan-and-Execute，关键是证据变化是否驱动下一步、计划是否稳定以及预算能否受控。

常见扣分

复杂就 ReAct/Multi-Agent；候选投票替代事实和权限；没有停止条件或最简基线。

补学与复测

用五个新场景填写选择矩阵，并为每个选择写一个触发降级或转人工的信号。

任务四 · 不确定性与 Trace 归因

完整示范答案

已证明相同 capability、args_hash 和 result_hash 被重复使用并耗尽步数；未证明是模型、上下文写回、搜索、缓存或 progress 规则中的哪层。应提出：Observation 未进入 DecisionContext、目标条件缺失、模型忽略证据、搜索返回不完整、编排重复检测缺失等假设，并分别查看 state_diff、context_fingerprint、evidence_refs、Prompt/model revision、query/result provenance 和 progress reason。

假设验证字段 / 可能修复边界
Observation 未写回state_version/diff、下一轮 context_fingerprint；编排修复原子提交
模型看见结果仍重复evidence refs、Prompt/model revision、decision rationale code；推理增加重复候选约束和显式不确定性
搜索结果缺支付状态result schema、provenance、coverage、tool version；工具/RAG 修复契约或返回 missing_field
缓存返回陈旧上下文gateway cache key、route revision、request hash；网关修复隔离键或禁用不安全缓存
编排无进展检测缺失action/args/result hash 历史、progress score；两次相同无增益后停止/澄清

事实仅有：R7 使用 limited-react/v3；三次相同 search_orders 和参数；前两次得到同一结果；第三步因 max_steps 终止。confidence=high 与持续缺少新信息矛盾，属于校准不足；同时存在 epistemic（支付事实未知）和 system（写回/缓存是否正确）不确定性。回归用例：①同结果两次后必须停止或澄清；②第二次返回新支付证据时允许转入下一步；③网关缓存/上下文缺失时必须产生类型化失败而非继续搜索。

关键得分证据

置信度“high”与无进展冲突，表明需校准 evidence sufficiency；修复按 owner 分层，至少回归正常重复、重复无进展和缺证据升级三类。

达标与优秀

达标：事实与假设分开且补证可执行。优秀：加入上下文指纹、恢复核对、隐私最小化 Trace 与校准指标。

可接受变体

可先从编排或模型侧验证，排序由检查成本与信息增益决定；不能在无证据时宣判单一根因。

常见扣分

直接换大模型；把 confidence 当真实正确率；修复只加步数；记录完整私有思维链代替可观察契约。

补学与复测

对新失败 Trace 写事实、四个假设、所需字段、最便宜验证和三条定向回归。

任务五 · 系统设计与口述

完整示范答案

高风险退款调查的推理内核只负责基于受权证据产生 clarify/search/escalate/finish 或退款提案；能力先按主体、风险和环境裁剪。调查可用有限 ReAct，稳定核验用 Workflow，退款必须规则校验、审批和幂等执行。目标歧义先澄清，证据冲突保留来源并升级，模型不可用走安全降级，策略拒绝不可绕过，恢复后外部状态变化使旧提案失效。Trace 记录输入/证据/决策/校验版本，不保存私有思维链。

  1. 成功与契约：成功是权威退款账本确认且用户工单更新。DecisionContext 包含主体/租户、订单快照、支付与可退余额证据、政策版本、审批状态、预算和能力目录；Decision 使用五类判别联合及前述不变量。
  2. 链路：编排裁剪为只读查询→推理决定澄清或查询→Schema/语义验证→治理授权→工具执行→Observation 回写。只有证据齐全才生成 refund.propose，超过阈值写 checkpoint 等审批；执行使用幂等键，finish 再过业务 acceptance。
  3. 模式与停止：调查用有限 ReAct，稳定资格核验用 Workflow，冲突证据做候选比较后升级人工；相同动作结果两次、无信息增益、deadline/成本耗尽或策略拒绝即停止。
  4. 失败：目标歧义问最小问题；来源冲突保留 provenance 并转人工；模型不可用时只展示已验证状态或安全失败；策略 deny 不重试；恢复时订单/金额/政策版本变化使旧审批和提案失效。
  5. 评测：Trace 记录 contract、context fingerprint、evidence refs、decision、验证/策略/工具版本、receipt 和 State diff；离线比较成功率、严重越权、无进展循环、校准、成本、延迟和恢复正确率。安全 hard fail 为零容忍，候选不达门槛则回退旧模型/Prompt/路由。

不让模型直接退款，是因为模型不拥有授权、业务前置和副作用执行；不记录完整思维过程，是为了最小化敏感信息并避免把不可验证内部过程当证据，只保存结构化决策理由和引用；换模型时用固定 contract 的非法输出、边界和配对回归验证兼容性；模型说完成但 acceptance 失败时，编排把失败 Observation 写回并继续、失败或转人工，绝不返回成功。

关键得分证据

成功条件由业务 acceptance 定义；离线指标覆盖决策合法率、证据充分率、循环率、严重越权、成本/延迟和人工升级质量。

达标与优秀

达标：契约、模式、证据、安全、失败和评测闭环。优秀：补替代方案、校准、SLO、灰度门禁和回退。

可接受变体

可把调查拆成多个专长模型或单模型多阶段，但工具执行、授权、状态提交和验收不能交给推理内核。

常见扣分

模型直接退款；用 CoT 日志做审计；模型 finish 即成功；更换模型却不做契约与回归验证。

补学与复测

换成“生产数据修复”重做八分钟设计，回答权限、模型切换、恢复和业务验收四个追问。
