任务一 · 九层职责与边界

完整示范答案

交互层规范化渠道、会话与身份；编排持有 Run State 和下一步控制；推理产出结构化决策意图；工具执行受控动作；Memory 管理经历；RAG 管理外部证据；治理与可观测做在线策略、Trace 和审计；评测离线消费结果与轨迹；模型网关统一厂商访问，基础设施提供运行、存储、网络与隔离。每层还需写“不负责项”。

组件职责 / 不负责输入 → 输出错放风险
交互接入规范化渠道、会话、身份与多模态；不推进 Run渠道事件 → normalized request + identity context信任客户端 user_id 导致冒用
编排控制持有 State、路由、预算、暂停恢复与终止；不自由生成业务判断请求/Observation → 下一步控制与 State commit状态只在消息中导致无法恢复
推理内核产生结构化决策意图；不授权、不执行、不提交状态目标/上下文/能力 → decide result模型直接执行造成越权
工具 / 动作声明并执行业务能力；不决定主体是否获权已授权调用 → receipt/Observation失败盲重试造成重复副作用
Memory治理 Agent 经历；不充当外部知识库可写经历/身份 → 有来源的记忆旧偏好污染高风险决策
知识 / RAG摄取检索外部资料；不保存个人经历查询/ACL → chunks/citations跨租户召回或过期政策
治理与可观测在线策略、授权、预算、Trace、审计；不做离线总体质量结论上下文/事件 → decision/obligations/telemetry只做末端过滤无法阻止动作
评测离线评价结果与轨迹并反馈发布；不在实时链路代替治理数据集/Trace → 指标/归因/门禁平均分掩盖安全 hard fail
模型网关 + 基础设施统一模型访问、路由、运行、存储、网络与隔离；不拥有业务目标模型请求/运行规格 → 模型事件/受控环境推理直连厂商或工具任意 egress

这是逻辑职责表，不要求九层各自部署成九个服务；同进程可以承载编排与推理，但 State 的提交权、模型输出的可信级别和失败 owner 仍必须分开。

关键得分证据

九层均有输入、输出、上游、下游和职责错放风险，例如推理不持久提交 State，评测不在实时链路拦截请求。

达标与优秀

达标：九层 owner 与方向稳定。优秀：区分逻辑分层和物理部署，说明可替代位置的约束与代价。

可接受变体

模型网关与基础设施可拆成两个服务，治理可横切部署；只要语义边界、证据归属和故障 owner 不变即可。

常见扣分

用产品名替代职责；缺少不负责项；让所有层直接调用模型；把安全单独堆成无边界大层。

补学与复测

只重做错放的层，为每层补“输入—输出—不负责—错放风险”，再用新请求串联。

任务二 · 在线主链路与离线旁路

完整示范答案

渠道带身份与目标进入编排；编排建立 Run State，先经治理约束，再按需读 Memory 和带 ACL 的 RAG。推理通过模型网关提出澄清、回答或创建工单意图；创建前做授权，超阈值写 checkpoint 并 waiting，审批恢复后重验主体、参数与外部状态，再由工具在受控环境执行。Observation 回写 State，业务 acceptance 决定完成。Trace 脱敏后进入离线评测，反馈发布门禁和编排策略。

控制流由编排发起，数据流携带身份、证据和结果，反馈流由 Observation 改变 State。政策 chunk 属于外部知识，过去偏好属于 Memory，当前金额、审批和 receipt 属于 Run State，三者不能合并。

关键得分证据

图区分控制、数据、反馈、在线和离线五种关系；政策证据、用户经历和当前 Run State 使用不同对象与生命周期。

达标与优秀

达标：链路、权限、暂停恢复和验收正确。优秀：补失败出口、幂等键、策略版本、审批摘要绑定和参考架构假设。

可接受变体

只读政策回答无需沙箱执行；创建工单工具可在业务服务内执行，但仍需授权、幂等和 Observation 契约。

常见扣分

让 LLM 持有全部状态；把 Trace 当控制流；把评测画在实时阻断路径；审批恢复后不重验。

补学与复测

用“修改地址并触发高风险支付”重画五类箭头，逐条解释谁拥有事实和控制权。

任务三 · 七条关系与边界

完整示范答案

七条关系的核心方向是：编排协调而推理建议；治理横切在线；推理经网关访问模型；Memory 保存经历而 RAG 提供外部知识；工具声明/执行与沙箱隔离分开；评测离线消费 Trace 并反馈；安全分别由在线控制、离线验证和执行隔离承担。四组易混边界都应以 owner、时间尺度、输入输出和失败后果解释。

  1. 编排→推理：编排把当前 State 和允许能力交给推理，推理只回决策意图；让模型直接提交 State 会使重试、恢复和审计失控。
  2. 在线治理横切：输入、检索、动作、输出各有门禁；只在输出过滤无法阻止越权工具执行。
  3. 推理→模型网关：网关统一能力过滤、路由和成本证据；组件直连厂商会绕过策略并破坏重放。
  4. Memory≠RAG：前者是经治理的经历，后者是版本化外部资料；把政策直接写个人记忆会失去来源与更新。
  5. 工具≠沙箱：工具定义业务能力和 Observation，沙箱限制实际文件/网络/资源；只有授权没有隔离仍可能被恶意输入利用。
  6. Trace→离线评测→编排：评测发现失败切片后反馈策略与发布门禁；把 Judge 放实时阻断路径会增加不确定性和延迟。
  7. 安全分布式承担：治理实时阻断、评测离线发现系统性风险、沙箱限制爆炸半径；合成“安全层”会隐藏责任和证据。
边界各自负责场景与混合风险
编排 / 推理状态控制 / 决策建议退款恢复时由编排核对 receipt；交给推理会重复执行
Memory / RAG经历 / 外部知识用户偏好可记忆，公司政策需按版本检索；混合会引用过期规则
治理 / 评测在线控制 / 离线评价治理拒绝当前越权，评测统计版本严重错误；混合会让实时决策依赖不稳定 Judge
工具 / 沙箱业务契约 / 执行隔离Python 工具定义输入输出，沙箱禁止公网；混合会把“获准分析”误当“可访问任何资源”

关键得分证据

每条给出反例，例如“RAG 结果自动写长期记忆”会丢失来源、版本与权限；“授权通过即无须沙箱”无法限制恶意或失控执行。

达标与优秀

达标：方向和风险准确。优秀：说明同库、同进程或同团队实现仍可保持语义分离，并讨论替代部署代价。

可接受变体

工具执行器可以就是沙箱内进程，但能力契约、授权策略和资源隔离仍需不同 owner/证据。

常见扣分

只复述标题；用“都是安全”合并治理、评测、沙箱；把 Multi-Agent 当新增基础设施层。

补学与复测

为四组边界各写一个正例、反例和首个可观测失败信号，再随机互换场景复述。

任务四 · 错误架构归因

完整示范答案

独立问题至少包括：请求绕过接入规范化与治理；LLM 直接持久化历史；共享向量库混合 Memory/RAG 且无 ACL；推理直接执行数据库；模型自由重试导致重复副作用；缺持久 State 和 checkpoint；finish 无业务验收；日志不可关联且无权限证据；多 Agent 无协议、仲裁和贡献评测；缺离线评测。修复先处理越权和副作用，再恢复可诊断性与质量门禁。

事实与主要 owner风险修复、补证与优先级
请求直达 LLM / 接入、治理身份未规范化、输入策略被绕过P0：服务端身份链和输入门禁；查认证/策略日志
LLM 保存全部历史 / 编排、MemoryState 不可恢复，敏感信息无生命周期结构化 Run State 与受治理 Memory；查快照和删除记录
共享向量库 / RAG、Memory跨部门/租户泄露与来源混乱P0：ACL 预过滤、逻辑隔离；做 canary 负测
LLM 直接执行数据库 / 推理、工具、治理越权和不可控副作用P0：结构化提议→授权→工具；核对 audit/receipt
模型自由重试 / 编排、工具重复扣款或重试风暴P0：类型化错误、统一预算、幂等和 effect_unknown 对账
无持久 State/checkpoint / 编排审批或故障后只能重跑保存阶段、参数摘要、版本和 receipt；演练恢复
模型 finish 即成功 / 编排业务目标未达成仍报成功权威系统 acceptance；保存验收证据
只有文本日志 / 可观测无法关联首个偏离点和权限trace_id/decision_id、span、策略版本和审计事件
三个 Agent 即免治理 / 编排、评测协作冲突且风险放大先与单 Agent 配对评测，再定义协议/仲裁
无离线评测 / 评测升级后回归不可见版本化数据集、hard fail 与发布门禁

“系统可能已经泄露或重复扣款”只是原因假设，需要 canary 查询、数据库审计和幂等 receipt 验证。修复顺序先封堵越权与副作用，再恢复 State/Trace，最后用评测决定 Multi-Agent 是否保留。

关键得分证据

明确事实是架构描述本身；“已泄露”“已重复扣款”是待验证后果，需要授权日志、工具 receipt、State diff 和 Trace。

达标与优秀

达标：八个独立问题都有层级、风险和修复。优秀：给出优先级、验证顺序、残余风险，并质疑 Multi-Agent 的必要性。

可接受变体

问题可按信任边界或请求阶段组织，不必严格按九层顺序，但每项必须有唯一主要 owner 和跨层协作者。

常见扣分

把多个现象重复归为“LLM 幻觉”；修复只写换模型或加 Prompt；事实与原因假设混写。

补学与复测

对新 Trace 建立“事实—假设—补证—owner—回归”表，先验证最早偏离点。

任务五 · 连续口述与追问

完整示范答案

口述按“接入身份—编排状态—知识/记忆—推理/网关—治理授权—工具/沙箱—Observation/验收—Trace/离线评测”推进。实时路径负责当前请求，离线评测负责基于版本化 Trace 做回归与发布反馈。模型只提出 finish，编排调用业务验收器；Multi-Agent 是编排与推理模式组合，不是新基础设施层。

示范口述：“用户请求先经交互接入层完成渠道、会话和服务端身份规范化，编排据此创建 Run State。编排按需读取带 ACL 的外部知识和经治理的用户经历，把目标、证据和允许能力交给推理内核；推理只通过模型网关获得模型能力并返回结构化意图。涉及动作时，在线治理核对 subject、action、resource、context 和审批义务，工具再在受控运行时或沙箱执行。结果作为 Observation 回到编排，编排更新状态并用业务验收器决定继续、暂停、失败或完成。全程 Trace 脱敏后进入离线评测，结果反馈发布门禁和各组件 owner。这是逻辑参考架构，组件可以同进程部署，但责任与证据不能合并。”

  1. 编排为何不能和推理合并理解？部署可合并，语义不能：模型输出是不可信建议，State commit、重试和终止必须确定可审计。
  2. 为什么不是所有组件都调网关？只有需要模型能力的推理/相关离线任务调用；工具和存储有自己的确定性接口。
  3. RAG 何时写 Memory？不是直接写；只有经来源验证、用户同意、用途和生命周期审查的经历摘要才可能晋升。
  4. 授权后为何还要沙箱？授权限制业务权利，沙箱限制进程实际能力和爆炸半径。
  5. 在线治理后为何还要评测？治理处理当前请求，评测从版本化样本发现系统性回归和策略盲点。
  6. 谁做业务验收？编排调用权威业务验收器，不能相信模型自报完成。
  7. Multi-Agent 放哪里？它是编排拓扑与推理分工的组合，需要共享基础设施，而不是第十层。

关键得分证据

五分钟内能说明九层方向，并在三轮追问下保持 Memory/RAG、编排/推理、治理/评测和工具/沙箱边界一致。

达标与优秀

达标：链路和边界稳定。优秀：主动声明参考架构假设、失败出口、替代实现和验证证据。

可接受变体

可以从一次失败 Trace 倒推九层，只要最终覆盖在线主链路、离线旁路和四组关键边界。

常见扣分

仅背层名；所有箭头都指向 LLM；把权限通过、工具成功、模型 finish 与业务完成混为一谈。

补学与复测

更换为“文档问答后提交高风险申请”，录制新的五分钟回答并让同伴随机追问三个边界。
