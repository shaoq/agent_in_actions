任务一 · 术语辨析

完整示范答案

Agent 是围绕 Goal，在 Environment 中读取 Observation、维护 State、选择 Action，并依据反馈继续或停止的受控系统。Workflow 的路径主要由开发者预先定义；Agent 允许模型在约束内决定部分下一步。Chatbot 是交互形态，LLM Application 是更宽的实现集合，两者都不必然拥有持续状态、环境动作和反馈循环。Autonomy 是可配置连续谱，不是“完全放权”。

  • Goal：希望达到且可验收的目标，例如“在预算内解决工单”，不是一句模糊指令。
  • Environment：Agent 可观察并可能改变的外部世界，包括用户、文档、API 和业务系统。
  • State：当前 run 已知事实、阶段、预算、历史动作和未决事项；它由系统持有，不等于聊天记录。
  • Observation：环境返回的证据，例如查询结果、错误码或审批决定；Action 是系统对环境提出并经授权执行的操作。
  • Feedback loop：Action 产生 Observation，Observation 更新 State 并影响下一步；Autonomy 表示其中有多少下一步由模型在白名单、预算、审批和停止条件内选择。

因此，一个客服聊天界面可能只是 Chatbot；一个使用 LLM 分类后走固定分支的系统仍是 Workflow；只有当系统围绕目标持续维护状态、依据新观察动态选行动并受明确终止条件约束时，才更适合称为 Agent。Tool calling 只是 Action 的一种表达接口，本身不足以改变系统控制权。

关键得分证据

必须讲出控制权、状态、观察—动作闭环、成功条件和终止条件，并能解释 tool calling 只是一种动作接口。

达标与优秀

达标：定义与关系稳定。优秀：能说明同一系统可在不同节点使用代码、Workflow 与局部 Agent，以及学术定义和工程定义的差异。

可接受变体

可以把 Agent 定义得更窄或更宽，但必须声明采用的自治阈值，并保持对 Workflow、Chatbot 和 LLM Application 的判别一致。

常见扣分

把“用了大模型”“能聊天”或“调用工具”直接等同于 Agent；忽略环境反馈、预算和业务验收。

补学与复测

回到基础理论，用一个审批 Workflow 和一个研究 Agent 重写十一组术语；复测时口述两者的控制权差异。

任务二 · 场景分类

完整示范答案

固定字段简历抽取从单次 LLM + Schema 校验开始；大额退款审批用确定性 Workflow，LLM 只做材料理解；行业研究适合有限 Agent；代码修复适合局部 Agent，在测试反馈下迭代；企业 FAQ 从 RAG + 单次 LLM 开始，证据不足拒答。每项都应先选最简单基线，再写升级条件。

场景与起点两个主要理由主要风险与升级条件
简历抽取：单次 LLM一次输入即可完成；结果可用固定 Schema 和字段规则验证风险是漏字段/编造；只有需要跨来源补证并反复纠错时才升级为局部 Agent
大额退款：Workflow审批路径可枚举；副作用和合规要求必须确定风险是越权退款；LLM 仅提取材料，动态调查可局部 Agent 化但无执行权
行业研究：有限 Agent新线索会改变查询；完成标准依赖多轮外部证据风险是无限搜索；设置来源范围、步数/成本预算和证据覆盖终止条件
代码修复：局部 Agent测试 Observation 可驱动下一轮；修改路径难预枚举风险是破坏代码；限定沙箱分支、文件范围和测试门禁，模板化修复可降级 Workflow
企业 FAQ：RAG + 单次 LLM主要任务是检索并组织已有知识；成功可由引用和拒答规则验证风险是错引/越权；只有问题需要多系统调查和连续动作时才升级

这五项都不以“任务听起来复杂”为依据，而以路径是否可枚举、是否需要环境反馈、成功能否确定性验证以及动作副作用来决定最简起点。

关键得分证据

理由同时覆盖路径可枚举性、外部反馈、不确定性、副作用、可验证成功条件和成本；风险分别指出解析错、越权审批、研究循环、破坏性修改和错误引用。

达标与优秀

达标：五项起点合理且有风险。优秀：给出混合边界，例如代码 Agent 只可改沙箱分支，退款 Agent 无权执行审批。

可接受变体

FAQ 可在问题高度固定时退化为普通检索；代码修复可在补丁模板固定时使用 Workflow。变体必须给出适用条件和回退信号。

常见扣分

按“复杂度高就用 Agent”作答；不说明为何不用更简单方案；忽略高风险动作的确定性门禁。

补学与复测

为五个新场景填写“最简基线—升级信号—不可接受风险—验收器”四列表后复测。

任务三 · 九层概念图

完整示范答案

在线主链路应从交互接入进入编排；编排构造状态并调用推理，推理统一经模型网关获得模型能力；动作提案经治理授权后交给工具，在沙箱或受控运行时执行，Observation 回写编排。RAG 提供带来源的外部知识，Memory 保存经治理的 Agent 经历。在线治理横切请求，离线评测消费 Trace 后反馈编排和发布门禁。

权限回答“这个主体能否做该动作”，沙箱回答“即使获准执行，进程实际能访问哪些文件、网络和资源”。两者不能互相替代。模型输出 finish 也只是决策意图，编排仍需调用业务验收器确认目标已达成。

关键得分证据

图中同时标出控制流、数据流、反馈流、权限点、业务验收、离线旁路，并区分授权决定与沙箱隔离。

达标与优秀

达标：九层方向和关键边界正确。优秀：标注这是参考架构，说明同进程部署不改变语义 owner，并给出异常、暂停与恢复出口。

可接受变体

治理可以通过 sidecar、middleware 或服务实现，网关也可与平台层合并部署；只要决策责任、证据和不可绕过性保持清楚即可。

常见扣分

只列组件不画方向；让推理直接调用厂商或工具；把 Memory 与向量库、在线治理与离线评测合并。

补学与复测

参考九层总览重画“查询政策后创建工单”链路，遮住组件名只看箭头仍应能解释每个 owner。

任务四 · 反例评审

完整示范答案

至少识别：框架不等于生产能力；三个 Agent 未证明分工收益；对话、工具结果和公司文档混入同库破坏 Memory/RAG 边界；缺少租户与 ACL；统一重试可能放大副作用；未知执行结果不能直接重试；模型 finish 不等于业务完成；缺少结构化 State、Trace、治理、评测和沙箱。每项都要写风险、owner 与验证证据。

错误假设 / 层级风险修复与证据
用了框架即生产就绪 / 全局可靠性、安全和验收能力缺失建立 SLO、失败矩阵、权限和回归证据
三个 Agent 必然更好 / 编排协作错误、成本与延迟上升先做单 Agent 基线，用质量/成本/延迟贡献实验决定是否拆分
所有内容进同一向量空间 / Memory 与 RAG来源、生命周期和 ACL 混乱逻辑隔离 schema/索引/权限；用跨租户 canary 检索验证
所有失败统一重试 / 工具与编排重复退款、重复邮件类型化错误、幂等键、重试预算；effect_unknown 先查 receipt/对账
模型 finish 等于成功 / 编排工单未解决却返回成功独立业务 acceptance，记录验收输入输出
无需治理、Trace 与沙箱 / 治理和基础设施越权、不可诊断、恶意执行扩散在线门禁、结构化 Trace、离线评测和最小隔离负测

描述明确证明“共享向量库、统一重试、模型直接判完成”存在；它没有证明已经泄露或重复扣款。后两者仍需访问日志、工具 receipt、幂等记录和 canary 测试验证，这也是事实与原因假设的边界。

关键得分证据

把明确事实与待验证假设分开，例如“存在共享库”是事实，“已经跨租户泄露”仍需 canary 查询和访问日志证明。

达标与优秀

达标：六个独立问题分层修复。优秀：按越权与重复副作用优先，先建立单 Agent 基线，再以质量、成本和延迟证明 Multi-Agent 收益。

可接受变体

同一物理存储可以承载多类数据，但必须逻辑隔离 schema、ACL、生命周期和检索入口；不能因“同库”就自动判错。

常见扣分

反复写“会幻觉”“不安全”而没有独立因果链；提出换更强模型，却不修权限、状态和验收。

补学与复测

用新的错误架构逐项填写“事实—层级—风险—修复—证据”，至少覆盖六个不同 owner。

任务五 · 连续口述追问

完整示范答案

三分钟回答可采用“定义—是否需要—约束—验收”：Agent 在目标和约束下利用状态与环境反馈动态选择行动；当路径固定、结果可确定性计算、风险高且不需探索时优先代码或 Workflow。Tool calling 只是接口；Workflow 使用 LLM 仍可保持确定性控制；自治用能力白名单、预算、审批和停止条件收缩；模型 finish 必须经过业务验收。

示范口述：“我把 Agent 定义为：围绕可验收目标维护状态，读取环境观察，并在约束内动态选择下一步行动的系统。是否使用 Agent 不取决于有没有 LLM，而取决于路径是否能预先枚举、是否需要连续反馈，以及错误副作用是否可控。固定审批、规则计算和高风险确定性流程优先普通代码或 Workflow；开放研究、测试驱动修复这类需要多轮观察的任务才考虑有限 Agent。生产中我会用能力白名单、总步数和成本预算、审批 checkpoint、幂等和业务验收收缩自治。”

  1. Tool calling 为什么不等于 Agent？它只是结构化动作接口；若调用时机和顺序都由代码固定，系统仍是 Workflow。
  2. Workflow 用了 LLM 会变成 Agent 吗？不会。LLM 可以只是某个固定节点，只要控制路径仍由开发者定义。
  3. 如何控制自治？限制可用能力、参数范围、预算、数据权限、审批点和终止条件，并记录每次决策证据。
  4. 为何还要业务验收？模型只能声明意图完成，权威系统才知道退款是否入账、工单是否解决。
  5. 如何证明 Multi-Agent 更好？与单 Agent 做配对评测，比较质量、严重错误率、成本、延迟和可恢复性；不能只看 Agent 数量。

关键得分证据

能连续回答至少两轮追问，结论前后一致，并用一个真实场景解释不选 Agent 的理由。

达标与优秀

达标：结构清楚且边界稳定。优秀：主动说明定义连续谱、证据限制，以及 Multi-Agent 需要与单 Agent 基线做贡献评测。

可接受变体

可用“控制权—不确定性—反馈—风险”或“目标—状态—动作—观察”组织，只要回答覆盖替代方案和生产约束。

常见扣分

背框架名、将自治理解为无限循环、把模型自评当业务成功、以 Agent 数量替代效果证据。

补学与复测

录制新场景的 30 秒结论和 3 分钟展开，请同伴随机追问“为什么不用 Workflow”和“谁验收”。
