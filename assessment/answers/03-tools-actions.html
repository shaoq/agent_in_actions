任务 1 · 能力契约与 Schema

建议作答

能力契约至少分 capability、input、output、error 四类：稳定 ID/版本、用途与前置；类型、必填、枚举和敏感字段；结构化结果与 effect receipt；validation/policy/business/transient/effect_unknown 错误。身份、凭据、endpoint 和 risk 由服务端注入，模型不得声明。

关键得分证据

提供 Schema、正反契约测试、breaking change 迁移与回滚，并证明工具不负责规划任务。

达标与优秀

达标：四类契约、边界和版本清楚。优秀：补 schema 指纹、兼容窗口、废弃指标和重放证据。

可接受变体

错误可用 discriminated union 或状态码映射；必须保留可机器判断的类别和 retryability。

常见扣分

返回自由文本；模型提供权限字段；只写自然语言描述而无可验证 Schema。

补学与复测

重写退款工具契约，增加三个非法输入、一个 breaking change 和回滚方案。

任务 2 · 生命周期与失败 Trace

建议作答

生命周期为 propose→validate/authorize→dispatch→execute→observe/commit。Trace 关联 run/step/decision/call、capability/schema/policy version、attempt、args hash、dispatch boundary、effect receipt 和结果摘要。工具成功只说明执行器返回成功，业务验收和 Run 完成由上层决定。

关键得分证据

能够从 Trace 区分“未派发、已派发失败、执行成功但提交失败、结果未知”。

达标与优秀

达标：五阶段和三种成功边界正确。优秀：补因果 ID、状态 diff、敏感最小化和重放限制。

可接受变体

远端 API 可合并 execute/observe，但 dispatch boundary 与 receipt 仍必须可定位。

常见扣分

HTTP 200 等于业务成功；日志无关联 ID；timeout 一律解释为未执行。

补学与复测

重画一条提交失败 Trace，标出第一处事实不确定和安全恢复动作。

任务 3 · 权限、审批与沙箱

建议作答

授权输入为 subject/action/resource/context，先最小化能力暴露，执行前再次强制校验。审批 receipt 绑定动作摘要、主体、资源、参数、策略版本和有效期；任何关键变化都使审批失效。治理定义策略，工具服务强制，基础设施沙箱限制计算、文件、网络、secret 和资源。

关键得分证据

展示参数漂移后的拒绝、恢复重验和沙箱越权负测；说明审批不等于隔离。

达标与优秀

达标：最小权限、摘要绑定、重验、隔离边界完整。优秀：覆盖 TOCTOU、短期凭据、egress 和销毁证据。

可接受变体

审批可由规则或人工产生，但高影响动作不能由同一模型自批。

常见扣分

认证等于授权；按钮确认不绑定参数；容器名称替代真实隔离证明。

补学与复测

用“审批后金额变化”场景重写 receipt、重验与沙箱网络策略。

任务 4 · 重试、幂等与副作用恢复

建议作答

先按 validation、policy、business、transient、effect_unknown 分类。幂等协议使用稳定 key + 请求摘要、原子 claim、结果/receipt 持久化和冲突检测。timeout 越过 dispatch boundary 后先查询 effect，不盲重试；必要时受限 retry、补偿或人工对账，并由统一预算防重试放大。

关键得分证据

给出重复请求返回同一结果、摘要冲突拒绝、未知结果查询和 TTL 规则。

达标与优秀

达标：分类、幂等、查询优先、预算完整。优秀：讨论原子性、补偿不可逆性和灾难恢复。

可接受变体

天然幂等的 PUT 可简化协议，但跨服务副作用仍需 receipt 与业务键。

常见扣分

所有失败指数退避；客户端随机新 key；把补偿当完美回滚。

补学与复测

为支付 timeout 建立状态机，覆盖未派发、已执行、未知和冲突四条路径。

任务 5 · 接口选型与系统设计

建议作答

Function Calling 是模型结构化表达动作意图的接口；MCP 是连接上下文与工具能力的协议；Computer Use 是通过界面操作环境。稳定业务 API 优先 Function Calling/MCP 封装；只有无 API 的遗留界面才用受限 Computer Use，并配截图证据、状态校验、HITL 和回退。

关键得分证据

选型比较稳定性、可观测、权限、幂等、延迟、维护成本和副作用风险，且不静默扩大权限。

达标与优秀

达标：三种方式分层正确且有降级。优秀：给接口演进、能力发现、熔断和端到端验收。

可接受变体

MCP server 可封装 API 或本地能力；协议选择不改变资源侧授权和执行隔离。

常见扣分

把三者当同类框架；有 API 仍优先点 UI；界面变化后继续盲执行。

补学与复测

换一个遗留系统做选型矩阵，写清 API 优先、Computer Use 限制和退出条件。
