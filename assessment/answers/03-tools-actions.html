任务 1 · 能力契约与 Schema

完整示范答案

能力契约至少分 capability、input、output、error 四类：稳定 ID/版本、用途与前置；类型、必填、枚举和敏感字段；结构化结果与 effect receipt；validation/policy/business/transient/effect_unknown 错误。身份、凭据、endpoint 和 risk 由服务端注入，模型不得声明。

{
  "capability_id":"refund.create@2", "effect":"financial_write",
  "input_schema":{"required":["order_id","amount_minor","currency","reason_code","idempotency_key"],
    "properties":{"order_id":{"pattern":"^ord_[0-9]+$"},"amount_minor":{"type":"integer","minimum":1},
      "currency":{"enum":["CNY"]},"reason_code":{"enum":["damaged","duplicate","service_failure"]},
      "idempotency_key":{"type":"string","minLength":20}}},
  "output":{"refund_id":"rf_88","status":"accepted","effect_receipt":"ledger:771@v1"},
  "errors":["validation_failed","policy_denied","business_conflict","transient_unavailable","effect_unknown"],
  "approval":{"required_if":"amount_minor > 50000","bind":["subject","order_id","amount_minor","currency","reason_code","policy_version"]}
}

租户、subject、服务凭据、endpoint、risk class 和最大可退额由服务端根据身份及订单注入，模型只能提出业务参数。若 v3 把 amount_minor 拆成 items[]，这是 breaking change：新增 refund.create@3 并双读影子验证，调用方显式迁移，保留 v2 回滚窗口；契约测试固定 Schema 指纹、错误码和幂等行为，不能静默复用原 ID。

关键得分证据

提供 Schema、正反契约测试、breaking change 迁移与回滚，并证明工具不负责规划任务。

达标与优秀

达标：四类契约、边界和版本清楚。优秀：补 schema 指纹、兼容窗口、废弃指标和重放证据。

可接受变体

错误可用 discriminated union 或状态码映射；必须保留可机器判断的类别和 retryability。

常见扣分

返回自由文本；模型提供权限字段；只写自然语言描述而无可验证 Schema。

补学与复测

重写退款工具契约，增加三个非法输入、一个 breaking change 和回滚方案。

任务 2 · 生命周期与失败 Trace

完整示范答案

生命周期为 propose→validate/authorize→dispatch→execute→observe/commit。Trace 关联 run/step/decision/call、capability/schema/policy version、attempt、args hash、dispatch boundary、effect receipt 和结果摘要。工具成功只说明执行器返回成功，业务验收和 Run 完成由上层决定。

对应 Trace：下面保留逐事件事实，用于定位 dispatch boundary、effect unknown、receipt 与业务验收边界。

run=R8 step=4 decision=D9 event=action_proposed capability=refund.create@2 args_hash=h1
run=R8 call=C5 event=prepared schema=v2 idempotency_key=k77
run=R8 call=C5 event=authorized policy=p17 approval=ap4 action_digest=h1
run=R8 call=C5 event=dispatched attempt=1 dispatch_boundary=crossed
run=R8 call=C5 event=transport_timeout effect=unknown
run=R8 call=C5 event=effect_lookup key=k77 receipt=ledger:771 status=accepted
run=R8 step=4 event=observation_committed state_version=12 tool_success=true
run=R8 event=business_acceptance refund_visible=true ticket_updated=false
run=R8 event=continue next=update_ticket

prepare 由工具适配器验证 Schema 和幂等记录；authorize 由治理判定并由工具服务强制；dispatch 跨越副作用边界；observe 把 transport/tool 状态转成类型化 Observation；commit 由编排原子写 State。HTTP/transport 成功只代表收到了响应，tool success 表示退款服务接受，business acceptance 还要核对退款账本与工单目标，三者不能合并。

关键得分证据

能够从 Trace 区分“未派发、已派发失败、执行成功但提交失败、结果未知”。

达标与优秀

达标：五阶段和三种成功边界正确。优秀：补因果 ID、状态 diff、敏感最小化和重放限制。

可接受变体

远端 API 可合并 execute/observe，但 dispatch boundary 与 receipt 仍必须可定位。

常见扣分

HTTP 200 等于业务成功；日志无关联 ID；timeout 一律解释为未执行。

补学与复测

重画一条提交失败 Trace，标出第一处事实不确定和安全恢复动作。

任务 3 · 权限、审批与沙箱

完整示范答案

授权输入为 subject/action/resource/context，先最小化能力暴露，执行前再次强制校验。审批 receipt 绑定动作摘要、主体、资源、参数、策略版本和有效期；任何关键变化都使审批失效。治理定义策略，工具服务强制，基础设施沙箱限制计算、文件、网络、secret 和资源。

能力subject/action/resource/context控制
order.lookup客服 S / read / tenant=T, order=42 / assigned_case=true只读字段投影，不能读取其他租户或完整支付凭据
refund.create客服 S / financial_write / order=42 / amount=800, policy=p17需要财务审批、稳定幂等键和服务端可退额校验
email.send客服 S / external_send / customer=42 / template=v6, recipient verified收件人和模板白名单；发送前预览，高敏内容 HITL

审批 receipt 为 {approval_id, actor, subject, action_digest, resource_version, policy_version, issued_at, expiry}；金额、收件人、订单版本、主体或策略变化都使其失效，恢复后重新授权，防 TOCTOU。治理产生 decision/obligations，工具服务以短期 workload identity 强制执行，基础设施把文件、网络、secret 和资源限制在 execution；服务端注入凭据防 confused deputy，文档内容永远作为数据，不能要求扩大 capability 或读取 secret，从而抵御间接注入。

关键得分证据

展示参数漂移后的拒绝、恢复重验和沙箱越权负测；说明审批不等于隔离。

达标与优秀

达标：最小权限、摘要绑定、重验、隔离边界完整。优秀：覆盖 TOCTOU、短期凭据、egress 和销毁证据。

可接受变体

审批可由规则或人工产生，但高影响动作不能由同一模型自批。

常见扣分

认证等于授权；按钮确认不绑定参数；容器名称替代真实隔离证明。

补学与复测

用“审批后金额变化”场景重写 receipt、重验与沙箱网络策略。

任务 4 · 重试、幂等与副作用恢复

完整示范答案

先按 validation、policy、business、transient、effect_unknown 分类。幂等协议使用稳定 key + 请求摘要、原子 claim、结果/receipt 持久化和冲突检测。timeout 越过 dispatch boundary 后先查询 effect，不盲重试；必要时受限 retry、补偿或人工对账，并由统一预算防重试放大。

观察动作理由
dispatch 前校验失败不重试，返回 validation_failed参数不变不会自愈
策略拒绝终止或正式升级重试不能绕过权限
dispatch 前 429在全局 deadline/retry budget 内退避明确未产生副作用
dispatch 后 timeout先按 key 查询 effect结果未知，不等于未执行
查询为 succeeded复用 receipt 并提交 Observation禁止重复退款
查询明确 not_found取得剩余预算后同 key 重试下游承诺 key 唯一
查询仍 unknown暂停并人工对账；必要时受控补偿无法安全自动判定

幂等记录状态为 claimed→dispatched→succeeded|failed|unknown，原子保存 request digest 和 receipt。相同 key + 相同摘要返回原结果；相同 key + 不同金额/订单返回 idempotency_conflict，绝不覆盖。统一预算按 Run 计算，例如最多 3 次外部尝试、累计 8 秒、单 provider 2 次；SDK、编排和工具不能各自叠加重试。

关键得分证据

给出重复请求返回同一结果、摘要冲突拒绝、未知结果查询和 TTL 规则。

达标与优秀

达标：分类、幂等、查询优先、预算完整。优秀：讨论原子性、补偿不可逆性和灾难恢复。

可接受变体

天然幂等的 PUT 可简化协议，但跨服务副作用仍需 receipt 与业务键。

常见扣分

所有失败指数退避；客户端随机新 key；把补偿当完美回滚。

补学与复测

为支付 timeout 建立状态机，覆盖未派发、已执行、未知和冲突四条路径。

任务 5 · 接口选型与系统设计

完整示范答案

Function Calling 是模型结构化表达动作意图的接口；MCP 是连接上下文与工具能力的协议；Computer Use 是通过界面操作环境。稳定业务 API 优先 Function Calling/MCP 封装；只有无 API 的遗留界面才用受限 Computer Use，并配截图证据、状态校验、HITL 和回退。

方式层次与信任边界主要风险
Function Calling模型输出结构化意图；服务端仍负责验证、授权和执行把 Schema 合法误当业务合法
MCP能力发现/调用协议；MCP server 是受审服务边界能力投毒、版本漂移、宽凭据
Computer Use通过像素/DOM 操作环境，执行面最不稳定界面漂移、误点、注入和副作用难确认

订单查询和退款都走内部 API，由 Function Calling 表达候选，或通过审核后的 MCP server 暴露同一受控 API；邮件使用模板化 API/MCP，收件人服务端解析。API/MCP 不可用时，只读查询可降级到受限 Computer Use：专用账号、只读页面、域名白名单、截图和字段二次校验；退款和发信默认 fail closed，除非人工明确批准一次性 UI 操作。降级不继承更宽凭据，不新增 capability，不因“兜底”跳过幂等、HITL 或业务验收。

关键得分证据

选型比较稳定性、可观测、权限、幂等、延迟、维护成本和副作用风险，且不静默扩大权限。

达标与优秀

达标：三种方式分层正确且有降级。优秀：给接口演进、能力发现、熔断和端到端验收。

可接受变体

MCP server 可封装 API 或本地能力；协议选择不改变资源侧授权和执行隔离。

常见扣分

把三者当同类框架；有 API 仍优先点 UI；界面变化后继续盲执行。

补学与复测

换一个遗留系统做选型矩阵，写清 API 优先、Computer Use 限制和退出条件。
