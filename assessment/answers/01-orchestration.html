任务一 · 请求链路与职责边界

完整示范答案

接入层把身份、目标和渠道上下文交给编排；编排持有 Run State、预算和当前阶段，构造受权 DecisionContext；推理经模型网关提出 act/respond/clarify/finish 意图；治理验证主体、动作、资源和上下文；工具执行后返回类型化 Observation；编排原子提交状态并决定下一步。model_finish 仍需 business_acceptance，不能直接返回成功。

控制权始终在编排：推理提出“查订单”，治理决定是否允许，工具拥有执行语义，Observation 是数据而不是下一步指令。编排与推理即使在同一进程，也必须通过可验证契约分界，否则模型重试、状态提交和业务终止会混成不可重放的隐式循环。

关键得分证据

图上标出谁提议、谁校验、谁执行、谁提交、谁验收；同进程部署仍以契约、可替换性和故障归因保持语义分层。

达标与优秀

达标：链路和方向正确。优秀：补充版本、超时、取消、错误类型、状态提交失败和降级出口。

可接受变体

治理可以在编排 middleware 内实现，工具可由远端服务执行；只要模型不能绕过策略、执行事实只由 Observation 建立即可。

常见扣分

让推理直接执行动作或写 State；把工具返回当 Run 成功；遗漏模型网关或授权点。

补学与复测

重画一条失败请求，仅用“提议—校验—执行—观察—提交—验收”六个动词解释 owner。

任务二 · 状态与生命周期

完整示范答案

Run 是一次目标执行，Session 是交互容器，Checkpoint 是可恢复快照，长期 Memory 是经治理的跨 Run 经历。状态机包含 running、waiting、completed、failed、cancelled、exhausted；在外部动作前记录 intent 与幂等键，动作 receipt 后原子提交 Observation。恢复前核对 schema、主体、审批摘要、预算、外部资源版本和未知副作用，不能把 waiting 直接改回 running。

关键守卫：waiting → running 必须收到新输入并重验身份、动作摘要、有效期与订单版本；running → completed 必须通过退款权威系统验收；不可恢复错误、策略拒绝、预算耗尽和取消进入不同终态，终态复测必须新建 Run。

退款动作前保存 action_digest、idempotency_key=tenant/order/refund_version 和审批要求，调用后保存 receipt。若超时，状态记为 effect_unknown，先用幂等键查询退款账本：已成功则回写 Observation，明确未执行才允许在剩余预算内重试，无法确认则转人工对账。Session 只承载对话，Memory 只保存经同意且不含本次临时审批的长期经历。

关键得分证据

退款用稳定 idempotency_key 和请求摘要；timeout 后先查询 effect receipt，不能假定未执行。完成与失败均有终态原因。

达标与优秀

达标：对象、状态转移、checkpoint、重授权和幂等完整。优秀：讨论 schema migration、不可安全恢复出口、补偿和人工对账。

可接受变体

状态名可以不同，Checkpoint 可用事件溯源重建；但终态不可含糊、恢复必须重验、外部副作用必须可关联。

常见扣分

所有信息塞 messages；把 Memory 当运行状态；恢复后直接重放写操作；completed 没有业务验收证据。

补学与复测

为“审批后恢复但订单已取消”画状态转移和恢复检查表，加入 effect_unknown 分支。

任务三 · 模式选择

完整示范答案

固定发票审批选 Workflow；客服分类选 Router；开放研究选有限 ReAct 或计划+观察；可先审批的批量修复选 Plan-and-Execute，批准计划后确定性执行；动态调查加高风险动作选混合模式，调查可适应，写动作回到 Workflow、审批和幂等门禁。选择基于路径可枚举性、Observation 依赖、计划稳定性、风险、成本和审计。

场景 / 选择不选两个替代方案升降级信号
固定发票审批 / WorkflowReAct 增加不可审计分支；Plan-and-Execute 对固定规则过重规则覆盖率下降且出现需调查异常时，只把调查局部升级
客服分流 / RouterWorkflow 会产生大量分支；ReAct 对单次分类成本高低置信或多意图率升高则澄清/人工；类别稳定可降为规则
竞品研究 / 有限 ReActWorkflow 无法预枚举线索；纯 Router 不能连续取证重复搜索、信息增益低或预算逼近时停止；任务稳定可先规划
批量数据修复 / Plan-and-Execute + WorkflowReAct 边想边写风险高；纯 Workflow 难生成一次性计划计划参数或数据版本漂移则审批失效；重复任务模板化后降级 Workflow
高风险企业工单 / 混合纯 Workflow 无法适应调查；纯 ReAct 不能安全执行写动作调查可动态，所有写动作进入确定性授权、HITL 和幂等链

模式不是整套系统的永久标签，而是不同阶段的控制方式；切换必须由可观测信号触发并进入评测，而不是由模型临时自我升级权限。

关键得分证据

每题写两个不选理由，并给出升级/降级信号，例如重复无进展、预算逼近、计划假设失效、风险升高或人工批准。

达标与优秀

达标：模式与场景匹配。优秀：给简单基线、动态切换、分桶评测和回退阈值。

可接受变体

行业研究也可用 Decomposition + 并行检索；前提是合并协议、证据冲突处理和预算优于单循环基线。

常见扣分

复杂就用 ReAct/Multi-Agent；没有停止条件；把推理模式和编排模式视为同一层概念。

补学与复测

用五个未见场景完成“基线—模式—拒绝理由—切换信号—评测”矩阵。

任务四 · 失败 Trace 归因

完整示范答案

Trace 只证明连续三次 search 得到相同文档并因 max_steps 终止；尚未证明模型能力差。假设包括 Observation 未写回/未进入下轮上下文、决策缓存错误、progress detector 缺失、搜索结果缺必要字段、目标或完成条件不可满足。应补 state_version/diff、context_fingerprint、decision_id、args_hash、result_hash、provenance、progress_score、termination_reason 和模型/Prompt 版本。

类别本 Trace 的具体结论
已证事实step 1–3 都选择 search；三次结果均为 A、B；没有其他动作；最终由 max_steps 终止
未证假设模型没看见结果、结果内容不足、上下文缓存陈旧、目标不可满足、策略缺 progress 规则，当前日志不能区分
补充字段query/args_hash、result_hash 与摘要、state_version/diff、context_fingerprint、evidence refs、progress_score、Prompt/模型/工具版本
可能 owner编排的 Observation 写回与重复检测、推理的决策策略、工具的结果契约、RAG 的证据质量、目标/验收定义
修复与回归相同 action+args+result 连续两次即澄清/换策略/停止；构造“有新证据”和“无信息增益”两组回归，断言不再第三次重复

先补证后修复：若 context fingerprint 已包含 A/B，才检查推理为何重复；若未包含，首个偏离点是编排写回。直接更换模型会掩盖真正 owner，也无法证明问题不会复发。

关键得分证据

为每个假设给可证伪检查；修复加入相同动作+参数+结果的重复检测、无信息增益停止和类型化失败，而非只换模型。

达标与优秀

达标：事实、假设、补证、owner、修复分开。优秀：设计 context 指纹、隐私最小化 Trace 和针对性回归。

可接受变体

重复阈值可为两次或三次，关键是结合动作、参数、结果与目标进展，避免把合法分页或轮询误判为循环。

常见扣分

直接断言模型弱；把所有失败归编排；提出无限重试；没有验证修复是否误杀正常重复。

补学与复测

使用一个新 Trace，先写“已知/未知”，再按最低验证成本排列四个假设和三条回归。

任务五 · 无代码系统设计与口述

完整示范答案

企业工单 Agent 以“问题解决且业务验收通过”为成功条件。编排持久化目标、阶段、证据、预算、审批和动作 receipt；调查使用有限 ReAct，稳定步骤用 Workflow，高风险写动作暂停审批。恢复时重验身份、摘要、外部版本和幂等状态；错误按验证失败、策略拒绝、暂时故障、业务冲突和 effect_unknown 分类。Trace 支持首个偏离点诊断，离线评测覆盖结果、轨迹、安全、成本和恢复。

  1. 问题与成功：输入工单目标、租户和主体；成功条件是权威工单状态 resolved、用户确认或规则验收通过，而非模型说 finished。
  2. 控制与状态：编排保存 phase、facts、evidence_refs、pending_action、approval、budget、receipts 和 state_version；调查阶段有限 ReAct，信息齐全后进入确定性解决 Workflow。
  3. 权限与 HITL：读操作按 tenant ACL；退款/改配置等写动作绑定 action digest、policy version 和 expiry 后暂停。恢复时重验主体、参数和外部版本。
  4. 可靠性：校验失败不重试；429/短暂网络错在 deadline 和统一预算内退避；策略拒绝终止或正式升级；业务冲突重新规划；effect_unknown 先用幂等键查 receipt，不能盲重放。
  5. 验收与评测：每步 Trace 记录 decision、policy、tool receipt 和 State diff；离线评测结果、轨迹、安全、成本、延迟、暂停恢复和首偏离点。

追问回答：模型说完成但工单未解决时，acceptance 失败并将 Observation 回写，预算允许则继续，否则转人工；恢复防重复依靠 action digest、幂等键、receipt 和外部状态重验；默认不用 Multi-Agent，因为单 Agent 加确定性工具更易审计，只有角色隔离或并行取证带来经评测的净收益才拆分；验证错误和策略拒绝不重试，暂时故障有限重试，业务冲突重新规划，未知副作用与不可安全迁移状态转人工。

关键得分证据

明确模型 finish 与 acceptance、暂停/恢复协议、重试预算、HITL、补偿或对账，以及 Multi-Agent 只有在分工收益可测时才引入。

达标与优秀

达标：控制、状态、模式、安全、恢复、验收闭环完整。优秀：说明 SLO、版本、发布门禁、回滚和替代方案演进条件。

可接受变体

可采用事件溯源或快照状态、单 Agent 或 orchestrator-workers；必须保持状态 owner、预算、停止条件和贡献评测清楚。

常见扣分

主要讲模型和 Prompt；恢复等于重跑；所有错误都重试；默认 Multi-Agent 更可靠。

补学与复测

换成“高风险数据修复”重新录制八分钟方案，接受“未知副作用如何恢复”和“为什么不用多 Agent”追问。
