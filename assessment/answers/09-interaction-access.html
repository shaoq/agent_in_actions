A · 渠道、会话与身份

完整示范答案

connection 是传输连接，conversation 是用户可见对话，session 是连续交互上下文，subject 是服务端确认的授权主体，run 是一次目标执行。接入层规范化消息、locale、附件引用、客户端事件 ID 和 schema version；身份来自服务端链路，匿名转登录、群聊代理和跨设备都需显式重绑。

IDOwner / 生命周期售后实例
connection_id渠道适配器；一次 WebSocket/电话连接断线重连生成 C2，不代表新用户
conversation_id交互层；用户可见线程App 与 IM 可映射到同一售后对话
session_id接入/会话服务；连续上下文窗口跨设备恢复 S8，受 TTL 和身份约束
subject_id身份系统；服务端认证主体匿名 A1 登录后绑定 user:42，不能信客户端 user_id
run_id编排；一次“处理订单 88 破损退款”目标执行重复 webhook 复用 client_event_id 去重，不创建第二 Run
{"schema_version":"input-v3","channel":"enterprise-im","client_event_id":"evt-77",
 "conversation_id":"conv-9","session_id":"sess-8",
 "identity":{"subject":"user:42","tenant":"acme","authn_level":"mfa"},
 "intent":{"type":"after_sale","text":"订单88破损，申请退款"},
 "artifacts":[{"artifact_id":"img-6","purpose":"damage_evidence"}],"locale":"zh-CN"}

匿名转登录时只在服务端证明所有权后迁移允许的对话，旧匿名权限不会继承；群聊代理需声明 on_behalf_of 和委托范围；跨设备恢复重新认证并绑定 session。接入层输出规范化契约后交给编排，不执行退款、推进状态或宣告完成。

关键得分证据

五类 ID owner 图、去重键、身份上下文和重复/乱序事件 Trace。

达标与优秀

达标：五类边界、身份和规范化正确。优秀：匿名迁移、代理人、跨设备和版本兼容。

可接受变体

conversation/session 可在简单产品合并，但 subject 与 run 仍不能由客户端自报。

常见扣分

session 代表一切；信任 user_id；接入层直接执行退款。

补学与复测

重放匿名登录后的重复消息，证明身份和 Run 没有错误合并。

B · 流式与多模态

完整示范答案

流式协议包含 event_id、sequence、cursor、run/turn、类型、时间和 terminal status，区分 token、结构化进展、可撤回草稿和业务终态；断线用 cursor 重放或 snapshot，取消向下游传播并处理迟到事件。附件先上传、扫描、类型验证、ACL、OCR/转码和 provenance，再以 artifact 引用进入编排。

事件 envelope：时序图描述恢复关系，下面保留可逐字段检查的事件类型、cursor、终态与错误实例。

{seq:1,cursor:"c1",type:"accepted",run:"R8"}
{seq:2,cursor:"c2",type:"progress",stage:"checking_order"}
{seq:3,cursor:"c3",type:"message.delta",draft:true,text:"正在核对"}
{seq:4,cursor:"c4",type:"clarification",question_id:"Q2",text:"是否更换收货地址？"}
-- 断线；客户端以 cursor=c4 重连，服务端重放 seq>4 或返回 snapshot --
{seq:5,cursor:"c5",type:"error",code:"approval_expired",recoverable:true}
{seq:6,cursor:"c6",type:"run.completed",terminal:true,acceptance:"ticket-resolved"}

图片流程：短期预签名上传→服务端 MIME/魔数校验→恶意文件扫描→tenant/subject ACL→OCR（confidence=0.61）→保存 source hash、OCR version 和 locator→只把 artifact reference 交编排；低置信字段要求用户确认，删除/撤回沿原图、缩略图、OCR 文本、缓存和模型临时引用传播。取消事件写 state，由编排终止下游；迟到 delta 丢弃或标记已撤回，HTTP 200/消息完成都不等于业务 Run 完成。

关键得分证据

背压、重放、取消、部分输出撤回、低置信 OCR 和删除传播 Trace。

达标与优秀

达标：事件协议与附件治理完整。优秀：背压、snapshot、跨设备恢复和低置信降级。

可接受变体

SSE、WebSocket 或轮询均可，只要顺序、恢复、取消和终态语义明确。

常见扣分

只流 token；HTTP 200=业务完成；附件原样直送模型。

补学与复测

模拟断线、取消和 OCR 低置信，验证 cursor、撤回与安全降级。

C · 澄清与 HITL

完整示范答案

澄清只询问阻塞决策的最小信息，记录 question_id、缺口、可选值和到期；HITL 暂停持久化 checkpoint。审批展示人可理解的动作摘要，并把 decision receipt 绑定 subject、action、resource、参数摘要、策略版本、有效期和 checkpoint；恢复前重验，参数漂移则失效。

场景化 receipt：下面保留退款案例中的 question ID、checkpoint、动作摘要、策略版本、有效期与恢复重验字段。

编排发现 address_id 缺失 → interaction.clarify(Q7, choices=[原址,新址], expiry=10m)
用户选新址 → State 更新 → 退款金额 1,200 元触发 HITL
checkpoint CP9 {run:R8,state_v:12,pending:refund.create,digest:h1}
展示：“向订单88退款 ¥1,200 至原支付方式；不会修改地址”
主管决定 → receipt {subject:manager7,action:approve,resource:order88,
                     action_digest:h1,policy:p17,expiry:15:30,checkpoint:CP9}
恢复 → 重验身份/expiry/order version/amount/digest → execute 或重新审批

迟到答复引用过期 question_id 时返回 decision_expired，不能应用到新 checkpoint；用户在等待期取消时状态转 cancelled 并撤销 receipt；金额、退款方式、收件地址或政策变化均使旧批准失效。按钮只提交 decision receipt，不直调工具；语音/读屏需完整播报动作、金额、对象、有效期和取消方式。

关键得分证据

澄清/审批/恢复时序、迟到答复、取消、重复决定和可访问呈现。

达标与优秀

达标：问题—决定—checkpoint 关联和摘要绑定。优秀：并发回答、过期、参数漂移和多渠道等价。

可接受变体

低风险缺省值可自动采用，但必须向用户可见、可撤销且不扩大权限。

常见扣分

循环追问；“确认吗”无动作摘要；按钮点击直调工具。

补学与复测

用审批后金额变化场景验证旧 receipt 失效和安全恢复。

D · 人工移交

完整示范答案

handoff package 只包含原因、目标、已确认事实与来源、推断、未决项、已执行 receipt、effect_unknown、权限范围、checkpoint/resume token 和队列 SLA。状态机支持 requested/queued/accepted/in-human/resolved/returned/cancelled；人工结果结构化回写，Agent 恢复前处理版本和并发冲突。

{
 "handoff_id":"H5","reason":"damage dispute + refund effect_unknown","goal":"resolve order88",
 "facts":[{"value":"photo img-6 received","source":"artifact:img-6"},{"value":"ledger query inconclusive","source":"tool:C8"}],
 "inferences":["package may be transit-damaged"],"open_questions":["did refund settle?"],
 "receipts":["email:E2"],"unknown_effects":["refund key:k77"],
 "scope":{"tenant":"acme","order":"88","pii":"minimal"},
 "checkpoint":"CP10","resume_token":"one-time-r5","queue":"after-sale-p1","sla":"10m"
}

状态为 requested→queued→accepted→in-human→resolved，人工也可 returned 给 Agent 或取消。队列超 SLA 时告知用户并提供回电/稍后处理，不能丢 Run。人工不可用则保留 checkpoint 安全失败；人工写回结构化 outcome、receipts 和 unresolved fields，编排以 state_version 做并发控制，版本冲突先合并/仲裁。handoff 只提供解决问题所需最小事实，不复制全量 transcript 或其他租户数据。

关键得分证据

最小权限包、人工不可用降级、双向接力、超时和未知副作用记录。

达标与优秀

达标：最小 handoff 与双向状态完整。优秀：并发控制、SLA 分级、回交和审计。

可接受变体

人工可直接结束 Run 或交回 Agent；两种路径都需业务终态和证据。

常见扣分

转人工即丢状态；复制全部 transcript；人工修改后 Agent 用旧状态继续。

补学与复测

演练人工队列不可用和人工交回 Agent 两条路径。

E · 同意、恢复与可访问性

完整示范答案

Consent receipt 记录目的、数据/动作范围、主体、版本、时间、有效期和撤回方式，撤回沿缓存与派生传播。错误按 validation/policy/business/transient/partial/effect_unknown 映射为重试、澄清、查询、转人工或停止。可访问性覆盖键盘、焦点、读屏标签、状态播报、非颜色提示、触控目标和多模态等价。

{"consent_id":"CS9","subject":"user:42","purpose":"damage_assessment",
 "data":["image:img-6","ocr:text"],"actions":["analyze","retain:30d"],
 "policy_version":"consent-v4","issued_at":"2026-07-19T10:00Z","expiry":"2026-08-18",
 "withdraw":"settings://privacy/CS9"}
错误用户呈现与恢复
validation：地址缺失聚焦到字段并澄清，不重试工具
policy_denied：无退款权限说明可行替代/申诉，不让模型改参数绕过
transient：查询 429显示可恢复进度，在预算内重试并允许取消
partial：邮件已发、工单未更新明确已完成/未完成部分，重试幂等工单更新
effect_unknown：退款超时先查 receipt/账本，等待或转人工，绝不重复退款

撤回 CS9 后立即阻止新读取，并删除原图、OCR、缩略图和缓存，记录传播 receipt。验收在 390px 无整体横向溢出；Tab 顺序与可见焦点正确，菜单支持 Enter/Escape 且关闭后焦点返回；进度用节制的 live region，错误不只靠颜色；照片有文本替代/语音等价上传，审批与取消均可键盘和读屏完成，并统计辅助技术任务完成率。

关键得分证据

consent receipt、错误矩阵、撤回传播、跨设备恢复和辅助技术完成率。

达标与优秀

达标：同意、类型化恢复和 WCAG 核心路径完整。优秀：partial/effect_unknown、跨设备、真实辅助技术和可追溯验收。

可接受变体

某些必要处理依据合同/法规而非 consent，但仍需告知、最小化和可审计依据。

常见扣分

consent=true；所有错误都重试；只测颜色对比；流式更新不播报状态。

补学与复测

用键盘和读屏完成一次 effect_unknown 恢复，并验证撤回传播。
