进入条件

开始考核前，应完成并保存以下案例的实现、测试、调试和表达证据：

  1. E0 最小 Agent Loop
  2. E1 结构化决策契约
  3. E2 工具调用与 Schema
  4. E3 权限与可靠副作用
  5. E4 Checkpoint 与安全恢复

案例资产可用或基准测试通过不等于满足进入条件。个人状态以进度跟踪中的证据为准。

总体规则

  • 总分：100 分；建议总时限：140 分钟。
  • 通过线：总分不低于 80 分。
  • 单项门槛：理解、实现、调试、表达每项不低于该项满分的 60%。
  • 必须提交四类证据；缺少任一类证据均不判定为阶段通过。
  • 代码任务必须在独立分支或临时副本完成，不直接覆盖 E0～E4 基准案例。
  • 首次未通过时保存原始结果，补强后使用新动作或新故障组合复测。

任务一：架构理解

分值与时限：20 分，20 分钟。

任务：不查看课程与案例，画出“企业工单 Agent 请求分派处理人”的链路，并回答：

  1. 原始模型输出在哪个边界转换成类型化 Decision？
  2. Tool Registry、Schema、授权、审批、派发和 Observation 分别由谁负责？
  3. 派发后响应丢失时，已知事实、未知事实和下一步查询分别是什么？
  4. 等待审批时 Checkpoint 至少保存什么，恢复时为什么必须重验？
  5. 哪些 Trace 能证明“没有越权执行”，哪些 Trace 仍不足以证明业务成功？

评分锚点：

分数标准
0～7只能描述模型调用或框架 API，无法区分控制与副作用边界。
8～12能画主流程，但权限、派发、未知结果或恢复语义混乱。
13～16边界、状态、证据和失败出口基本正确。
17～20能主动说明替代方案、生产缺口、幂等与恢复竞态。

通过证据：架构图或 Markdown 答案、完成时间、自评盲点。

任务二：迁移性工程实现

分值与时限：35 分，55 分钟。

新场景：实现“将工单分派给指定处理人”，不得复制案例中的同名动作。

  1. 为控制协议增加类型化 escalate 决策，用于把无法自动处理的请求转人工队列；严格拒绝缺字段和多余字段。
  2. 注册写工具 assign_ticket，Schema 要求 ticket_id、assignee_id 和 reason，并禁止额外字段。
  3. 动作必须检查 ticket:assign 权限；跨团队分派需要审批，审批绑定主体、目标工单、处理人和原因摘要。
  4. 使用稳定幂等键和 Effect Receipt；派发后响应丢失时先查询效果，不能直接换键重试。
  5. 等待审批时保存版本化 Checkpoint；恢复时重验主体、环境、动作摘要、权限和工单版本。
  6. 至少提供正常完成、非法决策、未授权、unknown outcome、恢复参数变化和重复恢复测试。

评分锚点：

分数标准
0～13代码不可运行，或把模型输出、权限和真实执行混在一起。
14～21主流程可运行，但错误分类、幂等、Checkpoint 或测试不完整。
22～28契约、策略、Receipt、恢复和关键回归测试完整。
29～35还能解释崩溃窗口、并发 claim、审计字段和生产替换方案。

通过证据：代码差异、完整测试输出、状态/协议设计说明和未解决风险。

任务三：Trace 调试

分值与时限：25 分，35 分钟。

运行以下故障场景，并对每条 Trace 按“现象、第一处事实偏离、已知、未知、责任层、修复、回归”记录：

conda run -n agent_in_actions python labs/decision-contract-minimal/demo.py --scenario extra-field
conda run -n agent_in_actions python labs/tool-calling-minimal/demo.py --scenario business-rejected
conda run -n agent_in_actions python labs/tool-reliability-minimal/demo.py --scenario unknown-outcome
conda run -n agent_in_actions python labs/checkpoint-resume-minimal/demo.py --scenario action-changed

必须额外回答：

  1. 哪两个场景在派发前停止，证据是什么？
  2. 哪个场景已经产生效果但客户端最初不知道？为什么最终没有重复写入？
  3. 动作摘要变化时，应该复用旧审批、重新审批、重新规划还是直接失败？依据是什么？
  4. 如果最终出现 run_completed，还需要什么业务验收信号才能证明工单确实分派正确？

评分锚点：0～9 分主要靠错误文案猜测；10～15 分能定位但混淆事实和推测；16～20 分能按边界归因并提出回归；21～25 分能识别证据上限和跨层修复风险。

通过证据：四份结构化调试记录和一份业务验收信号设计。

任务四：工程表达

分值与时限：20 分，30 分钟。

  1. 30 秒回答：“为什么 Function Calling 不等于安全工具执行？”
  2. 3 分钟回答：“一次工单写动作超时后应该如何处理？”
  3. 8 分钟设计：“支持暂停、审批、恢复和人工移交的工单 Agent”。
  4. 连续追问：Schema 版本变化、权限撤销、Checkpoint 过期、重复恢复、Trace 脱敏分别如何处理？

评分锚点：0～7 分以术语和框架名为主；8～12 分能解释主流程；13～16 分按问题、边界、链路、失败、证据和取舍组织；17～20 分能引用真实实验失败并主动说明未解决问题。

通过证据：录音、逐字稿或评分表，以及一个回答不佳追问的改进版本。
