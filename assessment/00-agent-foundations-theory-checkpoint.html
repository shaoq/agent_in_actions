Agent 基础理论检查点

目的

验证你是否建立了 Agent 应用工程的共同语义。此检查点不写代码,也不检查背诵数量。通过后才进入九层架构和编排理论深潜。

前置阅读:Agent 基础理论。

规则

  • 总分:100 分。
  • 建议时限:75 分钟,仅用于训练节奏。
  • 通过线:总分不低于 80 分。
  • 单项门槛:五项任务均不低于该项满分的 60%。
  • 必须提交全部证据;只写“已阅读”不计分。
  • 首次未通过时只补强薄弱项,使用新案例复测。

任务一:术语辨析

分值:20 分。

不看文档解释以下概念,并说明它们之间的关系:

  • Agent
  • Workflow
  • Chatbot
  • LLM Application
  • Goal
  • Environment
  • State
  • Action
  • Observation
  • Autonomy
  • Feedback loop

评分锚点:

分数 标准
0-7 主要复述名词,无法说明控制权和反馈。
8-12 基本定义正确,但将界面形态、架构和实现混合。
13-16 能用控制权、状态、环境和反馈解释边界。
17-20 能说明灰色地带、连续谱和不同资料的定义差异。

任务二:场景分类

分值:20 分。

分别为以下场景选择“普通代码、单次 LLM、Workflow、局部 Agent、Agent”中的起点,每题说明至少两个理由和一个风险:

  1. 固定字段的简历信息抽取。
  2. 大额退款审批。
  3. 根据新线索不断扩展的行业研究。
  4. 代码修复、测试和继续修改。
  5. 企业知识库 FAQ。

评分锚点:

分数 标准
0-7 只按任务复杂程度判断,所有复杂任务都选 Agent。
8-12 能区分固定和动态路径,但忽略验证或风险。
13-16 同时考虑不确定性、反馈、成功条件和副作用。
17-20 能提出混合方案、升级条件和不选其他方案的理由。

任务三:九层概念图

分值:25 分。

在空白纸或 Markdown 中画出:

  1. 用户请求的在线主链路。
  2. 编排、推理、动作、观察和模型网关的关系。
  3. Memory 与 RAG 的边界。
  4. 在线治理与离线评测的关系。
  5. 沙箱和权限各自解决的问题。

评分锚点:

分数 标准
0-9 只有组件列表,没有方向和数据流。
10-15 主链路基本正确,但在线/离线或边界混乱。
16-20 能画出主要控制流、数据流和反馈流。
21-25 能标出参考架构假设、替代位置和关键治理点。

任务四:反例评审

分值:20 分。

评审以下描述:

“系统使用 Agent 框架和三个 Agent,所以具备生产能力。所有对话、工具结果和公司文档都进入一个向量库。动作失败统一重试,模型输出完成后直接返回成功。”

要求指出至少六个问题,每个问题包含:

  • 错误假设。
  • 对应架构层。
  • 潜在风险。
  • 修复方向。

评分锚点:

分数 标准
0-7 只指出“可能幻觉”等泛化问题。
8-12 能识别 Memory/RAG、重试或完成判定问题。
13-16 能做分层归因并提出针对性修复。
17-20 能区分事实和推测,并质疑是否需要 Multi-Agent。

任务五:连续口述追问

分值:15 分。

先用 3 分钟回答“什么是 Agent,什么时候不应该使用 Agent”,再回答至少两轮追问:

  1. Tool calling 为什么不必然等于 Agent?
  2. Workflow 中使用 LLM 后是否变成 Agent?
  3. 自治程度如何控制?
  4. 模型说完成时为什么还需要业务验收?
  5. 如何证明 Multi-Agent 比单 Agent 更好?

评分锚点:

分数 标准
0-5 回答以框架名和术语堆砌为主。
6-9 能解释主概念,追问下边界不稳定。
10-12 结构清晰,能结合场景说明取舍。
13-15 能主动说明定义差异、证据边界和替代方案。

补强规则

  • 术语不足:重新用自己的案例解释,不能抄定义。
  • 场景选择不足:每题补充“不选其他方案”的理由。
  • 架构图不足:从一次真实请求重新画控制流、数据流和反馈流。
  • 反例不足:按九层逐项检查职责、失败和治理。
  • 口述不足:录制 3 分钟回答,只重练表达不稳定部分。
