L1 九层 Agent 架构知识地图检查点

目的

验证你是否已经从“知道九层名称”进阶到“能判断职责位置、上下游方向和错误流向”。本检查点不写代码,也不考查某个框架 API;通过后再进入 L2 编排控制层理论深潜。

前置条件:通过L0 Agent 基础理论检查点,并完成九层 Agent 架构总览。

L0、L1、L2 的考核边界

阶段核心问题本阶段不深入
L0 Agent 基础什么是 Agent,什么时候需要或不需要 Agent?九层完整覆盖与组件内部机制
L1 九层知识地图系统职责分别放在哪里,一次请求如何在层间流动?单个组件的算法、框架实现和代码
L2 组件理论深潜某个组件内部如何工作、取舍、失败和恢复?工程实现细节,留到 L3 验证

因此,L0 中的九层概念图只是基础关系预检;本检查点要求覆盖全部九层、七条关键关系、精确方向和错误归因。

规则

  • 总分:100 分。
  • 建议时限:90 分钟,仅用于训练节奏。
  • 通过线:总分不低于 80 分。
  • 单项门槛:五项任务均不低于该项满分的 60%。
  • 必须提交全部证据;缺失任务不得用其他任务表现推定得分。
  • 评分依据是架构语义和证据,不依据绘图美观、框架名数量或术语密度。
  • 首次未通过时只补强薄弱项,并使用新场景复测。

任务一:九层职责与边界

分值:20 分。

不看文档,为九个核心组件分别填写:

  1. 核心职责。
  2. 明确不负责什么。
  3. 主要输入。
  4. 主要输出。
  5. 至少一个职责错放的风险。

九个组件:

  1. 交互与接入层。
  2. 编排控制层。
  3. 推理内核。
  4. 工具 / 动作层。
  5. 记忆系统。
  6. 知识 / 检索。
  7. 治理与可观测层。
  8. 评测层。
  9. 模型网关 + 基础设施。

评分锚点:

分数标准
0-7主要列出组件名称或框架名,无法说明职责方向。
8-12大部分职责基本正确,但不负责事项、输入输出或在线/离线位置模糊。
13-16能说明九层主要输入、输出、上下游和常见职责错放。
17-20能区分通用事实、项目参考架构与合理替代位置,并说明不同选择的代价。

任务二:在线主链路与离线旁路

分值:25 分。

用户通过企业客服询问退款政策。系统需要结合用户身份检索其可见的公司制度;若用户决定申请退款,系统创建退款工单;金额超过阈值时必须暂停并等待人工审批。

画出该请求从进入到结束的系统图。可以使用纸笔、Markdown、Mermaid 或其他形式,但必须标注方向并覆盖:

  1. 用户/渠道、身份和目标如何进入系统。
  2. 编排如何维护 State,调用推理、RAG、Memory 和工具。
  3. 推理内核如何经过模型网关取得模型能力。
  4. 政策检索结果、用户经历和当前 Run State 如何区分。
  5. 创建工单前后的权限、审批、沙箱和业务验收。
  6. 工具结果如何作为 Observation 回写 State,以及系统如何继续、完成、失败或转人工。
  7. 在线治理在哪些位置检查请求、检索、动作和输出。
  8. 结构化 Trace 如何进入离线评测,评测结果反馈到哪里。

使用不同线型、颜色或文字标签区分:

  • 控制流:谁决定下一步。
  • 数据流:身份、目标、上下文和结果传到哪里。
  • 反馈流:Observation 如何改变 State 与下一步。
  • 在线路径:一次实时请求经过的组件。
  • 离线旁路:请求结束后消费 Trace 的评测链路。

评分锚点:

分数标准
0-9只有组件列表或无方向连线,无法识别谁持有控制权。
10-15在线主链路基本正确,但控制流、数据流、Observation 或在线/离线关系混合。
16-20能区分四类流向,正确放置模型网关、RAG、Memory、工具、沙箱和离线评测。
21-25进一步标出关键治理点、暂停与审批、业务验收、失败出口和参考架构假设。

任务三:七条关键关系与边界对照

分值:20 分。

分别解释七条关系,每条给出一个错误设计及其风险:

  1. 编排层协调推理、动作、Memory 和 RAG。
  2. 在线治理横切实时请求。
  3. 推理内核统一经过模型网关。
  4. Memory 与 RAG 分离。
  5. 工具能力声明与沙箱执行分离。
  6. 离线评测消费在线 Trace 并反馈编排。
  7. 安全分布在在线治理、离线安全评测和沙箱三处。

再用“各自负责什么 / 为什么容易混淆 / 混合后的风险 / 一个场景”深入对照:

  • 编排控制层 vs 推理内核。
  • Memory vs RAG。
  • 在线治理 vs 离线评测。
  • 工具 / 动作层 vs 沙箱执行环境。

评分锚点:

分数标准
0-7只能复述关系标题,无法解释方向或风险。
8-12能解释部分关系,但把编排/推理、Memory/RAG 或治理/评测混合。
13-16七条关系方向基本稳定,四组边界均能结合场景说明。
17-20能主动说明架构假设、合理替代位置、治理条件和替代方案代价。

任务四:错误架构归因

分值:20 分。

“用户请求直接发送给 LLM。LLM 保存全部历史,直接查询包含所有部门文档和用户对话的共享向量库,再直接执行数据库工具。工具失败后由模型自由决定是否反复重试。模型声明完成就返回成功。系统只打印文本日志,不保留结构化 State、权限证据或 Trace。团队使用三个 Agent,所以认为不再需要额外治理和评测。”

指出至少八个相互独立的问题。每个问题必须包含:

  • 错误假设或错误流向。
  • 对应的九层组件。
  • 控制流或数据流为什么错误。
  • 潜在生产风险。
  • 修复方向和必要证据。

回答中必须区分“描述中明确存在的事实”“基于事实的原因假设”和“需要进一步验证的信息”。

评分锚点:

分数标准
0-7主要回答“会幻觉”“不安全”等泛化风险,没有层级和流向。
8-12能识别若干独立问题,但层级归因重复或修复方向笼统。
13-16至少八个问题基本独立,能准确归因并给出针对性修复和证据。
17-20还能识别事实与推测、质疑 Multi-Agent 必要性,并说明修复顺序和残余风险。

任务五:连续口述与追问

分值:15 分。

不看文档,用不超过 5 分钟回答:

请从用户请求开始,完整讲解一次 Agent 请求如何经过九层架构,并说明哪些组件位于实时路径、哪些属于离线旁路。

然后随机回答至少三轮追问:

  1. 为什么编排层不能和推理内核合并理解?
  2. 为什么不是所有组件都直接调用模型网关?
  3. RAG 检索结果在什么条件下才能写入 Memory?
  4. 权限检查通过后为什么还需要沙箱?
  5. 在线治理已经检查请求,为什么还需要离线评测?
  6. 模型声明完成后,谁负责业务验收?
  7. Multi-Agent 应当位于哪一层,为什么它不是一个独立基础设施层?

评分锚点:

分数标准
0-5主要背九层名称,无法形成请求链路或回答方向追问。
6-9主链路基本正确,但追问下至少两组关键边界发生变化。
10-12表达结构清晰,实时/离线和四组边界稳定,能结合场景说明。
13-15能主动声明参考架构假设、替代方案、证据边界和工程取舍。

补强规则

  • 九层职责不足:只重做错放层级,补齐输入、输出和“不负责事项”。
  • 链路图不足:更换业务场景,重新标注控制流、数据流、反馈流与离线旁路。
  • 关键关系不足:为不稳定边界各增加一个正例和反例,不得抄原定义。
  • 错误归因不足:先区分事实、假设和未知,再按九层逐项定位。
  • 口述不足:录制新的 5 分钟主回答,只重练追问下不稳定的边界。

通过后更新进度跟踪,再进入L2 编排控制层。
